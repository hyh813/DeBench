# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Almost all variables use generic names like `param_1`, `param_2`, `iVar1`, `iVar2`, `puVar3`, `local_20`, `local_44`. The original source uses meaningful names like `x`, `arr`, `n`, `str`, `sum`, `i` - all completely lost in decompilation.

**2. Function Naming Quality:** Score: 9/10  
Justification: Nearly all function names match the original source exactly: `double_value`, `process_int`, `process_char`, `process_short`, `array_1d_stack`, `ptr_single`, `struct_simple`, etc. Excellent recovery of function semantics.

**3. Literal Clarity:** Score: 5/10  
Justification: Some literals are pre-computed (0x400c0000 for float 2.0), but magic numbers like `0x19`, `0xbf` in `process_char` are unexplained. The switch uses auto-generated `CSWTCH_78`. However, string formatting literals are correctly preserved.

**4. Noise Reduction:** Score: 8/10  
Justification: Relatively clean output with minimal compiler artifacts. Types like `undefined4` and `undefined` are present but don't significantly impede readability. No redundant casts or debug instrumentation observed.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10

**6. Function Decomposition:** Score: 7/10

**7. Code Linearization:** Score: 6/10

Justification: No significant dead code observed. Loop unrolling is minimal. Common subexpressions are handled reasonably.

**10. Composite Type Recovery:** Score: 2/10  
Justification: All struct/union types completely lost. `struct_simple` uses `param_1[0]`, `param_1[1]`, `param_1[2]` instead of `p->x`, `p->y`, `p->z`. `Point3D`, `Rect`, `Widget`, `Node`, `TreeNode` - all flattened to raw pointer arithmetic.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe patterns present. The `union_type` function incorrectly treats `float*` as `int*` for type==1 case. Pointer arithmetic is sometimes incorrect (e.g., `ptr_diff` divides by 4 instead of sizeof(int)). Function pointer calls use `code*` without proper type checking.
