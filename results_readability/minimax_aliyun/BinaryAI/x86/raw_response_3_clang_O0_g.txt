# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `local_30`, `local_14`, `param_1`, `uVar1`, `aiStack_2c`. The original source had meaningful names like `arr`, `p`, `head`, `temp` - these are completely lost. Even stack arrays are named `acStack_804` instead of descriptive names.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions are correctly named (`local_vars`, `heap_basic`, `double_value`, `memop_memset`), but many internal functions use generic patterns like `FUN_00011030`, `FUN_00011130`. External library functions are properly prefixed with `<EXTERNAL>::`.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `0x2a` (42), `0x14` (20), `0x800` (2048) appear without explanation. String literals are replaced with `&DAT_00014008` references. Some constants like `0xffffffff` (error return) are consistent but unexplained.

**4. Noise Reduction:** Score: 8/10  
Justification: Minimal compiler artifacts present. No debug instrumentation, register spilling code, or redundant casts observed. The decompiled code is relatively clean from noise artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation - `undefined4` replaces `int`, `byte*` replaces `char*`, `void*` replaces specific pointer types. Original `int`, `size_t`, `char*` types are lost. Function pointers use `code*` instead of proper signatures.

**10. Composite Type Recovery:** Score: 2/10  
Justification: The `Point` struct is completely lost - `heap_struct` uses raw pointer arithmetic `*__ptr` and `__ptr[1]` instead of `p->x` and `p->y`. The `Node` struct for linked lists is similarly reduced to `int*` with manual offset calculations. No struct field names recovered.

**11. Type Safety:** Score: 5/10  
Justification: Significant unsafe pointer arithmetic throughout (e.g., `(int)__ptr + local_14 * 4`). Many casts between `void*` and typed pointers. No bounds checking on array accesses. However, no obviously dangerous operations like buffer overflows are introduced by decompilation.

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.3/10

**Critical Defects:**
- Variable naming is nearly unusable - `local_14`, `param_1`, `uVar1` provide no semantic meaning
- Composite types (structs Point, Node, TreeNode) completely lost - replaced with raw pointer arithmetic
- No comments whatsoever - all source code explanations and test case identifiers are missing
- Type precision severely degraded - `undefined4`, `void*`, `byte*` everywhere instead of `int`, `char*`
- No anomaly indication for dangerous patterns like dangling pointers, memory leaks, double-free

**Best Quality Aspect:** Noise Reduction (scored 8/10)  
The decompiled code is clean from compiler artifacts - no debug instrumentation, register spilling, or redundant casts clutter the output.

**Most Needs Improvement:** Comment Quality & Presence (scored 1/10)  
The complete absence of comments is devastating for readability. The original source had extensive documentation explaining each test case's purpose, scene classification, and expected behavior - all completely lost.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Recover variable names from usage patterns** - `local_14` used in loop `local_14 < param_1` should be inferred as loop counter `i`
2. **Reconstruct struct types from pointer arithmetic** - When seeing `*(int*)((int*)ptr + offset)`, recognize this as struct field access and attempt struct layout recovery
3. **Add semantic comments for test functions** - Detect test patterns (expected values in printf) and annotate with test case identifiers from string literals
