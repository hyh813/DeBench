# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use purely mechanical names like `local_30`, `iVar1`, `param_1`, `puVar3`. The decompiler makes no attempt to infer semantic meaning - `local_30` in `test_cpp_member_func` is actually the `SimpleClass obj` but carries no indication of its purpose. Compare to source where `obj`, `r1`, `r2`, `r3` clearly convey meaning.

**2. Function Naming Quality:** Score: 6/10  
Justification: Member functions are correctly identified (`SimpleClass::getValue`, `LifecycleClass::getData`). Template instantiations are reasonably named (`template_max_int_`, `template_swap_int_`). However, operators are problematic: `Point::operator_` and `Point::operator__` don't distinguish between `operator+`, `operator==`, and `operator++`. Virtual function dispatch shows as indirect calls without semantic recovery.

**3. Literal Clarity:** Score: 7/10  
Justification: Most numeric literals match source (0x1234 for class ID, 5 for array size, 10/20/30 for test values). String "Test" preserved. Minor issue: `0x14` appears as decimal 20 without indicating it's `sizeof(name)-1`. At -O0, literals are generally preserved well.

**4. Noise Reduction:** Score: 8/10  
Justification: Minimal compiler artifacts. No debug register spills, redundant casts are minimal, no __asm__ blocks. The code is clean except for some verbose type prefixes like `undefined4`, `undefined8` which are Ghidra conventions rather than true noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: At -O0, control flow closely matches source. Loops use `for` construct (`for (local_20 = 0; local_20 < param_1; local_20 = local_20 + 1)`), conditionals are straightforward. Exception handling shows as `__cxa_throw`/`__cxa_begin_catch` which is appropriate for C++. No spurious goto statements.

**6. Function Decomposition:** Score: 7/10  
Justification: Main test functions (12-30 lines) are reasonable. The decompiler correctly separates member functions. However, it generates many small standard library helper functions (tuple, unique_ptr internals) that obscure the main logic - this is a consequence of template instantiation, not decompiler failure.

**7. Code Linearization:** Score: 8/10  
Justification: Code reads top-to-bottom without excessive jumps. Early returns used appropriately. No deep nesting (>4 levels) observed. The exception handling in `test_cpp_exception` shows proper try-catch structure through library calls.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No dead code. Some repetition in template instantiations (e.g., multiple `std::get_0ul_int__` functions) but this reflects actual code generation. Loop unrolling is minimal at -O0.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses generic types like `undefined4` (32-bit), `undefined8` (64-bit), `ulong` instead of specific types. `int` is used generically. Pointers are typed (`int*`, `char*`) but the `undefined` prefix adds noise. Compare to source using `int`, `size_t`, `bool` precisely. Const correctness is not preserved.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Class layouts are correctly reconstructed with proper offsets. `SimpleClass` shows value at offset 0, name[32] at offset 4. `LifecycleClass` correctly identifies data pointer and size fields. However, field names are generic - the decompiler doesn't recover `value`, `name`, `size`, `data` from the source.

**11. Type Safety:** Score: 7/10  
Justification: Explicit casts used appropriately for type conversions. Pointer arithmetic is correct. No obvious buffer overflow patterns. The main issue is the prevalence of `undefined*` types requiring casts. Function pointer types are sometimes generic `code*` rather than specific signatures.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments present. No explanation of what functions do, no correlation to source code. The decompiler provides only function signatures. Compare to source which has extensive comments explaining each test case's purpose.

**13. Idiom Recognition:** Score: 4/10  
Justification: C++ idioms like RAII (unique_ptr destructor), virtual dispatch (vtable calls), and template instantiation are visible in the code structure but not explicitly identified or named. The decompiler doesn't annotate "This implements RAII pattern" or "Virtual dispatch via vtable".

**14. Anomaly Indication:** Score: 2/10  
Justification: No flags for uncertain or anomalous code. Indirect virtual calls are shown without explanation that they couldn't be statically resolved. No warnings about potential undefined behavior. The decompiler presents everything with equal confidence.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants are documented. The source has implicit contracts (e.g., `getData` requires idx < size) that aren't recovered. No indication of error handling expectations.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are explicit within each function. However, no documentation of call graph relationships - which functions call which, which are callbacks, virtual dispatch targets. The relationship between test functions and the classes they test isn't indicated.

**17. Module Organization:** Score: 5/10  
Justification: Functions are organized by address (Ghidra standard), not by logical grouping. Related class methods are grouped together (all SimpleClass methods, all LifecycleClass methods) which is good. But the file structure doesn't reflect the original source organization with sections.

**18. Build System Compatibility:** Score: 7/10  
Justification: External functions are declared with appropriate signatures. The code would compile with standard C++ library headers. No obvious syntax errors. However, no build instructions or dependency documentation is provided.

### Critical Defects (Bullet List)
- **Variable naming is completely generic**: `local_30`, `iVar1`, `param_1` provide no semantic information about their purpose (e.g., should infer `obj`, `result`, `size` from usage)
- **No comments or documentation**: Zero explanation of what functions do, how they relate to source, or any algorithm descriptions
- **No anomaly indication**: Indirect virtual calls presented without explanation that resolution was uncertain
- **Field names not recovered**: Struct/class fields use generic names (`field0`, `field1`) instead of recovered names (`value`, `name`, `data`)
- **Operator overloading not distinguished**: `Point::operator_` and `Point::operator__` don't indicate which operator (+, ==, or ++)

### Best Quality Aspect (Single Dimension)
- **Control Flow Naturalness (8/10)**  
- At -O0, the decompiler preserves natural loop and conditional structures that closely match the source code's intent

### Most Needs Improvement (Single Dimension)
- **Semantic Transparency (2.5/10 avg)**  
- The decompiler provides almost no semantic context: no comments, no pattern recognition, no anomaly flags, no contract recovery. This makes the code technically accurate but practically unusable for understanding the original intent.

### Top 3 Recommendations for Decompiler Improvement
1. **Infer variable names from usage patterns**: Track how variables are used (e.g., `local_c += iVar1` suggests accumulator → name it `result` or `sum`)
2. **Add semantic comments**: Include brief descriptions of function purpose, algorithm notes, and correlation to source patterns
3. **Recover operator names**: Distinguish between different operators (operator+, operator==, operator++) rather than generic `operator_`
