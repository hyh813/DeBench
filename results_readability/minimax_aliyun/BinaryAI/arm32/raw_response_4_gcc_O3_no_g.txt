# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names like `param_1`, `param_2`, `param_3`. Local variables use cryptic names like `iVar1`, `uVar1`, `uVar2`, `ppbVar2`, `ppbVar3`. The original source used meaningful names like `a`, `b`, `count`, `args`, `str`, `x`, `y` - all completely lost in decompilation.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are correctly preserved from symbols: `cdecl_func`, `stdcall_func`, `varargs_func`, `param_by_value_int`, `ret_pointer`, etc. However, some show compiler artifacts like `.constprop.0` suffix, and some call wrapper functions are inlined away entirely.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers appear throughout without explanation: `0xf` (15), `0x32` (50), `0x15` (21), `0x2a` (42), `0xad` (173). While some values like `0x2a` can be inferred as ASCII, most are unexplained. The original source had no magic numbers in the logic itself.

**4. Noise Reduction:** Score: 5/10  
Justification: Moderate noise present including `.constprop.0` suffixes on optimized functions, `undefined4`/`undefined8` types for unresolved values, and spurious `code*` type for function pointers. However, no debug artifacts or register spilling clutter visible.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control structures are natural - `if-else` statements, `do-while` loops (e.g., in `varargs_func`), and switch-like constructs (e.g., in `ret_multi_branch`). At -O0, the control flow closely matches source. Minor issues with some computed jumps but generally readable.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are appropriately sized (20-50 lines typical). However, some trivial wrappers like `call_cdecl()` returning constant `0xf` are inlined, while some complex functions like `func_struct_byval` are fragmented into many parameters. Mixed granularity.

**7. Code Linearization:** Score: 8/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately. Some forward jumps exist but minimal. The code flows naturally without spaghetti control flow.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No dead code observed. Some loop unrolling present (e.g., `ret_large_struct` manually unrolls the loop), which is expected at any optimization level. Minor redundant expressions but manageable.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Most parameters use generic `int` or `undefined4`. Function pointers use `code*` instead of proper function signatures. The original had specific types like `int*`, `char*`, `double`, `long long` - all eroded to generic types. Const correctness completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct recovery. `func_struct_byval` receives 21 separate parameters (the struct was flattened into individual values). `func_mixed_args` has `double` parameter decomposed into four `undefined4` values. Original had `LargeStruct`, `SmallStruct`, `TestIntPair`, `BigStruct` - all completely lost.

**11. Type Safety:** Score: 5/10  
Justification: Several unsafe operations: casting to `byte*` for string access (`param_string`), using `code*` for function pointers (`param_func_ptr`), arithmetic on `undefined4` types. No alignment or bounds checking. Moderate risk of undefined behavior.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no useful comments. Only string data references like `&DAT_0001143c` for printf format strings. No explanation of algorithm logic, no uncertainty indicators, no API contracts. Original had extensive comments explaining each test case.

**13. Idiom Recognition:** Score: 1/10  
Justification: No design patterns, algorithms, or idioms identified or named. The varargs handling, struct passing conventions, and calling convention tests are all presented as generic code without recognizing the patterns.

**14. Anomaly Indication:** Score: 1/10  
Justification: No flags for suspicious code. No warnings about indirect calls, unusual stack layouts, or potential undefined behavior. The decompiler presents all code with equal confidence regardless of reliability.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. Original source had implicit contracts (e.g., `func_struct_byptr` requires non-null pointer) that are not indicated in the decompiled code.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are present and traceable. However, no documentation of who calls whom, no call graph information, no indication of callback relationships. Each function is isolated.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in address order rather than logical grouping. Original organized by test category (calling conventions, parameter passing, return values) - this organization is completely lost. Arbitrary ordering makes understanding the test structure difficult.

**18. Build System Compatibility:** Score: 6/10  
Justification: Code uses external declarations (`puts`, `strlen`, `__printf_chk`) but lacks proper headers. The `undefined4` type would need definition. Basic compilation possible with stubs but not production-ready.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.2/10

**Critical Defects:**
- Variable naming is completely generic (`param_1`, `iVar1`, `uVar2`) - original meaningful names completely lost
- No struct/union recovery - `LargeStruct` (128 bytes) flattened to 21 individual parameters
- No comments explaining any logic - cannot understand test purpose without source
- Type precision severely degraded - `double`, `long long`, function pointers all become generic types
- No anomaly indication for uncertain decompilations

**Best Quality Aspect:** Control Flow Naturalness (8/10)  
- At -O0, the decompiler correctly recovers loops, conditionals, and switch statements that closely match the original source structure

**Most Needs Improvement:** Semantic Transparency (1.25/4)  
- Zero useful comments, no pattern recognition, no anomaly indication, no contract recovery - the decompiled code provides no semantic context to help analysts understand the code's purpose

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns**: `param_1 + param_2` in a function returning their sum should be named `a + b`, not `param_1 + param_2`
2. **Recover composite types**: Detect when multiple consecutive parameters form a struct (especially for large structs like `LargeStruct` with 16 `long long` fields) and reconstruct the struct with field names
3. **Add semantic comments**: Recognize test harness patterns (like this calling convention test suite) and add explanatory comments about what each function tests
