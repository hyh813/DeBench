# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use completely generic names like `local_28`, `uStack_24`, `pvVar1`, `iVar2`, `iVar3`. The original source uses meaningful names like `value`, `name`, `data`, `size`, but these are completely lost. For example, in `test_cpp_member_func`, the string "Test" is stored in `local_28` with no semantic meaning.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions are correctly named (`test_cpp_member_func`, `test_cpp_constructor`), but many class member functions are missing or incorrectly named. Template instantiations use awkward suffixes like `template_max_int_` instead of proper C++ names. Some functions like `<EXTERNAL>::operator.new` are just stubs.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers appear throughout without explanation. For example, `0x125c` (4704 in decimal, the expected return value) appears in multiple functions without being named as a constant. The string "Test" is encoded as `0x74736554` in memory layout. No enum or constant recovery is evident.

**4. Noise Reduction:** Score: 3/10  
Justification: Heavy noise pollution from external function stubs (operator_new, printf, etc.) that just call themselves. Type declarations use `undefined4`, `undefined2`, `undefined` which are decompiler artifacts. Multiple duplicate destructor definitions exist (e.g., three versions of `Base::~Base`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic control flow is preserved, but `test_cpp_constructor` uses a confusing do-while loop with negative counter (`iVar3 = iVar3 + -1`) instead of the original forward loop. The virtual function call in `call_virtual_func` uses triple-indirect call `***(code ***)param_1` which is hard to follow.

**6. Function Decomposition:** Score: 4/10  
Justification: Many class member functions from the original source are missing or merged. For example, `SimpleClass::setValue()` and `SimpleClass::compute()` are not recovered as separate functions. The `test_cpp_oo_features` function is a monolith combining multiple test cases. Template class methods are partially recovered but with incorrect signatures.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with minimal forward jumps. However, the `test_cpp_oo_features` function has complex nesting and jumps that make tracing difficult. Some computed jumps in virtual inheritance functions disrupt linear reading.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Multiple duplicate destructor definitions exist (e.g., three `MultiDerived::~MultiDerived` variants at different addresses). The virtual inheritance functions have repeated code patterns for accessing `shared_data`. No dead code elimination is apparent.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Severe type degradation throughout. All integers use `undefined4` or `undefined` instead of `int`, `size_t`, etc. Pointers use generic `undefined4*` or `void*`. The original `int`, `char`, `size_t` types are completely lost. Const correctness is not preserved.

**10. Composite Type Recovery:** Score: 1/10  
Justification: No struct recovery. The original classes are completely lost. Access to class members uses raw pointer arithmetic like `*(int *)((int)pvVar1 + 8)` instead of `obj->data`. The `SimpleClass` with `int value` and `char name[32]` is completely unrecognizable.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe operations: raw pointer arithmetic throughout, incorrect virtual function call syntax `(***(code ***)param_1)()`, mixing of `int` and pointer types. The virtual inheritance access patterns use complex pointer arithmetic that could easily cause undefined behavior.

### Critical Defects (Bullet List)
- **Variable naming**: All variables use generic decompiler-generated names (`local_28`, `iVar1`, `pvVar1`) losing all semantic meaning from original source
- **Type precision**: Complete type loss - all types are `undefined4`, `undefined`, or `void*` instead of `int`, `char*`, `size_t`
- **Composite type recovery**: Zero struct/class recovery - all class members accessed via raw pointer arithmetic instead of struct field access
- **Anomaly indication**: Missing functions and simplified logic presented as correct without any warnings (e.g., lambda, operator overload, template functions all reduced to constants)
- **Comment quality**: No useful comments explaining algorithm, purpose, or indicating decompilation confidence

### Best Quality Aspect (Single Dimension)
- **Function Decomposition (scored 4/10)** - While imperfect, the test functions are reasonably separated and the overall function boundaries are preserved, making it the least degraded aspect

### Most Needs Improvement (Single Dimension)
- **Composite Type Recovery (scored 1/10)** - The complete failure to recover any class/struct definitions is the most severe issue, making the code essentially unusable for analysis

### Top 3 Recommendations for Decompiler Improvement
1. **Recover class/struct definitions**: Use RTTI data and vtable analysis to reconstruct class layouts with proper field names and types
2. **Improve variable naming**: Infer names from usage patterns (e.g., string length → `name_len`, object creation → `obj`, counter → `i` or `count`)
3. **Add anomaly flags**: Indicate when functions have been simplified, constants substituted for logic, or when indirect calls could not be resolved
