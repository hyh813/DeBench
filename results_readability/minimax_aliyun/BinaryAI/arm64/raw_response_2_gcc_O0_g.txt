# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All parameters use generic names (`param_1`, `param_2`) and locals use numeric patterns (`local_4`, `local_8`, `local_c`). The original source used meaningful names like `x`, `c`, `arr`, `n`, `p`. This makes it extremely difficult to track variable purpose across the 2219 lines of decompiled code.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are correctly preserved from source: `double_value`, `process_char`, `process_int`, `array_1d_stack`, `ptr_single`, etc. However, `process_ld` has wrong return type (void instead of long double), and `ptr_triple` uses `undefined8 *` instead of proper `int ***` type.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervade the code: `0x40`, `0x5b` for ASCII bounds checking (should be 'A', 'Z'); `0x28` (40 bytes) for 2D array row stride; `0xc` (12 bytes) for 3-field struct. The original source had readable constants like `'A'`, `'Z'`, `32`, `10` (array size). No named constants recovered.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack check guards (`___stack_chk_guard`) add noise; library functions `__addtf3` and `__multf3` are inlined (should be external calls); some redundant type casts present. However, no debug instrumentation or assembly artifacts pollute the main functions.

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.6/10

**Critical Defects:**
- **Struct/union types completely lost**: All composite types reduced to raw pointer arithmetic (`param_1[1]`, `param_1[2]` instead of `p->y`, `p->z`), losing all semantic meaning of fields
- **Variable naming is unusable**: `param_1`, `param_2`, `local_4`, `local_8` provide no semantic information about variable purpose
- **Type precision severely degraded**: `process_ld` returns `void` instead of `long double`; pointers use `long` instead of proper pointer types; `process_char` returns `byte` not `char`
- **No comments or contracts**: Zero documentation of algorithm intent, parameter meanings, or expected behavior
- **Magic numbers everywhere**: `0x28`, `0xc`, `0x40`, `0x5b` without explanation; original had readable constants like `'A'`, `'Z'`, `10`, `32`

**Best Quality Aspect:** Function Decomposition (scored 8/10)  
The decompiler preserved the original function granularity - all 40+ small functions remain separate with appropriate responsibilities, matching the source structure.

**Most Needs Improvement:** Composite Type Recovery (scored 3/10)  
The most severe readability loss is the complete absence of struct/union type recovery. The original had clear typedefs (`Point3D`, `Rect`, `Node`, `UnionData`) with named fields - all reduced to opaque pointer arithmetic.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Recover struct/union types**: Analyze stack frame layout and pointer access patterns to reconstruct composite types with field names (e.g., detect `*(int*)(param_1+4)` as field access and name it `y` based on offset patterns)
2. **Infer variable names from usage**: Use data flow analysis to name parameters (e.g., `param_1` used as array subscript → `arr`; `param_2` compared to array length → `n`)
3. **Add semantic comments**: Recognize common patterns (array traversal, linked list walk, struct field access) and add explanatory comments like "// Traversing linked list via next pointer"
