# Decompilation Readability Assessment

## Target Function: `param_fake_branch`

### Analysis Overview

The original source code implements a fake branch obfuscation technique with two "always false" conditions that will never execute:
1. `x * x < 0` (impossible for real numbers)
2. `strlen(str) < 0` (string length can't be negative)

The function simply returns the input `x` unchanged.

The decompiled version shows a **critical failure**: the function signature is completely lost - it shows `void param_fake_branch(void)` instead of `int param_fake_branch(int x)`.

### Top 3 Recommendations for Decompiler Improvement

1. **Recover function calling conventions**: At `-O0`, function signatures should be preserved. The decompiler should analyze call sites to infer missing parameters - `call_fake_branch` passes `10` to `param_fake_branch`, clearly indicating a parameter was lost.

2. **Flag incomplete decompilation**: When a function body appears empty or the signature doesn't match call sites, the decompiler should add an anomaly comment like `// WARNING: Function signature may be incorrect` or `// NOTE: Possible dead code elimination`.

3. **Recognize obfuscation patterns**: The fake branch idiom (永假分支) is a known obfuscation technique. The decompiler should recognize patterns like `x*x < 0` as always-false conditions and either preserve them with comments or document the elimination.
