# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names (`param_1`, `param_2`, `param_3`) with no semantic meaning. Local variables use cryptic names like `local_8`, `iVar1`, `uVar2`, `piVar3`. The original source uses meaningful names like `a`, `b`, `count`, `sum`, but these are completely lost in decompilation.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most test functions correctly recovered their source names (`cdecl_func`, `stdcall_func`, `varargs_func`, `call_cdecl`). However, some internal functions use address-based names like `FUN_00100640`, and external library stubs have placeholder names.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere - `0xf` (15), `0x32` (50), `0x2a` (42) without explanation. String literals referenced as `&DAT_00101752` with no readable content. The decompiler shows hex values instead of decimal, making it harder to verify expected outputs.

**4. Noise Reduction:** Score: 8/10  
Justification: Relatively clean output for -O0 compilation. No obvious compiler artifacts like register spilling code or debug instrumentation. Minor noise from auto-generated function stubs for external symbols.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Most simple functions have natural control flow. However, `varargs_func` and `param_varargs` use complex goto-like patterns with manual stack manipulation (`local_8`, `&stack0xffffffffffffffe0`) that obscure the simple loop structure. The ARM64 varargs handling is particularly convoluted.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions are properly separated with good granularity. Each test function is correctly isolated. No 500-line monoliths. Parameter lists are reasonable, though some expose implementation details (e.g., `int *param_1` for struct parameters).

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom. The varargs functions have multiple forward jumps and complex nesting that disrupts linear reading. Early returns present but appropriate.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code observed. Loop unrolling is minimal. Common subexpression elimination artifacts are not significant. The code is relatively clean in this regard.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Generic `int` used throughout instead of specific types. `long` vs `int` mismatches (e.g., `func_mixed_args` has `long param_4` but source uses `long long`). Function pointers use `code *` instead of proper function signatures. No `const` correctness preserved.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct types completely lost. `func_struct_byval` treats the 128-byte `LargeStruct` as `int *param_1` with manual array indexing (`param_1[0x1e]`, `param_1[0x1c]`). `SmallStruct` not recovered at all - treated as `int *`. No field names recovered.

**11. Type Safety:** Score: 4/10  
Justification: Significant unsafe operations. Struct fields accessed via pointer arithmetic (`param_1[1] * *param_1`). Function pointer cast to `code *` loses type safety. Mixed signed/unsigned in varargs handling (`uint` vs `int`). No bounds checking.

### Critical Defects (Bullet List)

- **Variable naming completely generic**: All parameters use `param_N` and locals use `var_N`/`iVar_N` patterns, making it impossible to understand variable purposes without cross-referencing source
- **Struct types completely lost**: `LargeStruct` (128 bytes) and `SmallStruct` reduced to `int *` with manual array indexing, losing all semantic meaning of struct fields
- **No comments or documentation**: Zero explanatory text for algorithms, parameter meanings, return value contracts, or non-obvious logic
- **Magic numbers everywhere**: Hex values like `0xf`, `0x32`, `0x2a` without explanation; string literals as `&DAT_00101752`
- **Type precision severely degraded**: All specific types (int32_t, uint64_t, long long) collapsed to generic `int` or `long`

### Top 3 Recommendations for Decompiler Improvement

1. **Infer variable names from usage patterns**: When `param_1` is used as a loop counter or accumulated in a sum, name it `count` or `sum`. When `param_1` is dereferenced for field access, recognize it's a struct pointer and recover the struct type.

2. **Recover struct/union types from stack layout analysis**: The 128-byte LargeStruct is passed by value but decompiled as `int *`. Analyze the stack frame layout to identify composite types and generate proper struct declarations with field names.

3. **Add explanatory comments for non-trivial patterns**: Document varargs handling, explain why hex values are used, flag uncertain indirect calls, and describe algorithm patterns (e.g., "// Varargs: iterates through variadic arguments using va_list").
