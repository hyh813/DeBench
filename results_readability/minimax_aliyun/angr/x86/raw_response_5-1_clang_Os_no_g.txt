# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `v2`, `ptr`, `i`. The original source has meaningful names like `value`, `name`, `size`, `data`, `multiplier`, but these are completely lost. Even loop counters are just `i` rather than descriptive names.

**2. Function Naming Quality:** Score: 4/10  
Justification: Test functions like `test_cpp_member_func`, `test_cpp_constructor` are correctly named using mangled C++ symbols. However, most utility functions use address-based names (`sub_401030`, `sub_401160`) with no semantic meaning. The original source has clear function names like `getValue`, `setValue`, `compute` that are not recovered.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals like "Test" are preserved inline. Magic numbers such as `10`, `50`, `4` appear without explanation. The value `10` in `test_cpp_member_func` should be named as it's the expected value after `setValue(10)`. Some constants like `0x1234` from source are lost.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe VEX IR artifacts pollute the code, particularly in functions like `Container<double>::push` and `Container<double>::get` where lines like `[D] PutI(72:F64x8)[t3,0] = t6()` appear. Multiple functions show "Failed to decompile" messages. This level of noise makes the code practically unusable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Functions like `register_tm_clones` use complex goto-based control flow with computed jumps. The source uses natural C++ constructs (if-else, for loops, try-catch), but these are transformed into spaghetti code. At -O0, such transformation is excessive and unexpected.

**6. Function Decomposition:** Score: 5/10  
Justification: Function sizes are reasonable, but many functions are empty stubs (e.g., `test_cpp_virtual_func`, `test_cpp_multiple_inheritance` return immediately without implementation). The original has meaningful member functions that are not properly decompiled.

**7. Code Linearization:** Score: 5/10  
Justification: Most functions read reasonably top-to-bottom, but some have forward jumps (e.g., `goto ptr[2]` in `sub_401030`). The exception handling code shows non-linear flow with re-throw patterns.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some duplicate patterns exist but not severe. The main issue is incomplete decompilation leaving functions with just `return;` statements rather than redundant code.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types are generic `unsigned int` or `int`. The original uses `size_t`, `int*`, `bool`, but these are not recovered. `void*` is overused. The source has precise types like `size_t idx` in `getData`, but decompiled version uses generic types.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct definitions are attempted (`struct_0`, `struct_1`) but field names are generic (`field_0`, `field_4`, `padding_0`). The original has `class SimpleClass` with `int value; char name[32];` - these class members are not recovered with meaningful names.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe operations present: pointer arithmetic like `ptr + 4 * index`, casting through `void*`, and indirect function calls without type safety. The virtual function call `(*((int *)*((int *)arg_0)))(arg_0, arg_1)` is dangerous and unclear.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no useful comments. The VEX IR annotations are noise, not helpful. No explanation of algorithms, no recovery of preconditions/postconditions. The original has extensive comments explaining each test case.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or algorithms identified. The original demonstrates C++ idioms (RAII with smart pointers, virtual function dispatch, exception handling) but these are not recognized or named in the decompiled output.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some "[D]" markers indicate unsupported operations, and "Failed to decompile" messages appear. However, many incomplete functions (empty bodies) are not flagged as anomalies. The quality of anomaly indication is inconsistent.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No contracts, preconditions, or postconditions recovered. The original has implicit contracts (e.g., `idx < size` check in `getData`) but these are not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Some `extern` declarations show dependencies. Function call relationships are partially visible but not documented. The original shows clear caller/callee relationships (e.g., `call_virtual_func` calls `obj->virtual_func`).

**17. Module Organization:** Score: 4/10  
Justification: Functions are somewhat grouped by class but not well organized. The original has clear sections (CPP-L3-01, CPP-L3-02, etc.) which are lost. Related functions are scattered.

**18. Build System Compatibility:** Score: 2/10  
Justification: Multiple issues prevent compilation: "Failed to decompile" for `template_max<double>`, incomplete function bodies, missing extern definitions. The code would require significant patching to compile.

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.4/10

### Critical Defects:
- **Variable naming**: All meaningful variable names lost (`value`, `name`, `size`, `data` â†’ `v0`, `v1`, `ptr`)
- **Function naming**: Utility functions use address-based names (`sub_401030`) instead of semantic names
- **Noise**: VEX IR artifacts (`[D] PutI...`) pollute multiple functions making them unreadable
- **Incomplete decompilation**: Many test functions (`test_cpp_virtual_func`, `test_cpp_multiple_inheritance`) are empty stubs
- **Type loss**: Class members not recovered; struct fields are generic (`field_0`, `padding_0`)

### Best Quality Aspect: Literal Clarity (5/10)  
String literals like "Test" are preserved inline, and some numeric constants can be inferred from context. This is the least degraded dimension.

### Most Needs Improvement: Variable Naming Quality (2/10)  
The complete loss of meaningful variable names is the most severe readability issue. At -O0, variables should retain their semantic names from the source.

### Top 3 Recommendations:
1. **Infer variable names from usage patterns**: `v0 = 10` followed by `strlen` suggests `v0` should be related to string handling; analyze data flow to recover names like `name` or `value`
2. **Remove VEX IR artifacts**: The `[D] PutI...` lines are internal decompiler noise that should never appear in output
3. **Recover class member names**: Use RTTI and vtable analysis to recover class member names like `value`, `name`, `multiplier` instead of generic offsets
