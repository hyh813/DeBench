# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `v0`, `v1`, `v2`, `ptr`, `a0`, `a1`, `a2` throughout. The original source had meaningful names like `value`, `name`, `data`, `size`, `x`, `y`, `capture_by_value`, `capture_by_ref`. The decompiled code provides no semantic hints about variable purpose.

**2. Function Naming Quality:** Score: 4/10  
Justification: Mixed results - some C++ mangled names are preserved (e.g., `_Z20test_cpp_member_funcv`, `template_max<int>`), but many functions use address-based generic names like `sub_401020`, `sub_40102c`. The original function names like `getValue`, `compute`, `call_virtual_func` are not consistently recovered.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers like `4`, `8`, `40`, `80` appear without explanation. String literals are referenced via global variables (`&g_4021d4`, `&g_40207d`) rather than inlined, making them unreadable. The value `42` in exception throwing lacks context about being an exception payload.

**4. Noise Reduction:** Score: 7/10  
Justification: Minimal compiler artifacts present. Many functions are empty (`return;`), which is expected from -O0 but creates noise. No redundant casts or debug instrumentation observed. The code is relatively clean aside from trivial wrapper functions.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: The virtual function call uses `goto *((long long *)*((long long *)&arg_0));` which is a computed goto through vtable - unusual and hard to follow. Most other functions are simple with minimal control flow. At -O0, more natural flow is expected.

**6. Function Decomposition:** Score: 6/10  
Justification: Template functions are properly separated (`template_max<int>`, `template_max<double>`, `template_swap<int>`). However, many trivial wrapper functions exist (e.g., `sub_401239` just calls `test_cpp_member_func`). Some functions like `test_cpp_member_func` contain partial logic but don't return the computed value.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom with minimal jumps. The main disruption is the virtual function dispatch pattern. No excessive forward jumps or deep nesting observed. Generally good linearization.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No obvious dead code or duplicate expressions. Loop unrolling is minimal (expected at -O0). Code is relatively clean without redundant computations.

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.9/10

### Critical Defects:
- **Variable names** are all generic (`v0`, `v1`, `ptr`, `a0`) with no semantic meaning recovered from source
- **Function return types incorrect** - functions like `test_cpp_member_func()`, `test_cpp_constructor()` should return `int` but are declared `void`, losing computed results
- **No comments** - original source had extensive documentation explaining each test case and expected values, completely lost
- **Struct/class recovery poor** - `struct struct_0`, `struct struct_1` with generic fields instead of meaningful class members
- **String literals not inlined** - referenced via `&g_4021d4` etc. making code unreadable

### Best Quality Aspect: Redundancy Elimination (8/10)
The decompiled code has minimal redundant computations and dead code. Template functions are properly separated, and the basic control flow structure is preserved without excessive duplication.

### Most Needs Improvement: Semantic Transparency (1.75/4)
The decompiled code lacks any comments, pattern identification, anomaly detection, or contract recovery. This dimension scored lowest because the original source's extensive documentation and expected value annotations are completely lost, making it impossible to understand what each function is supposed to compute without referring to the source.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover variable names from usage patterns** - `v0` used as `10` (the value being set) should be named `value`; `ptr` used for object allocation should indicate object type
2. **Fix return type inference** - `test_cpp_member_func()` clearly computes `r1 + r2 + r3` and should return `int`, not `void`
3. **Inline string literals and add expected value comments** - Replace `&g_40207d` with actual strings like "CPP-L3-01: %d (期望: 4704)" and add comments showing expected return values from source
