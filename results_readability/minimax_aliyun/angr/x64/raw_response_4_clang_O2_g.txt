# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `a0`, `a1`, `v0`, `v1`, `v2`, `v3`, `v19`, `v20`, `v29`-`v37`. The parameter `a0` in `cdecl_func` should be named `a` or `param_a` based on usage. Only a few exceptions like `len` in `func_mixed_args` have meaningful names.

**2. Function Naming Quality:** Score: 6/10  
Justification: About 40% of functions have correct semantic names (`cdecl_func`, `stdcall_func`, `test_calling_conventions`, `param_by_value_int`), but the majority use address-based generic names like `sub_401020`, `sub_40102c`. The test functions and core logic functions are well-named, but wrapper/thunk functions are not.

**3. Literal Clarity:** Score: 5/10  
Justification: Return values like `15`, `50`, `6` are correct but lack explanation. Magic number `4294967295` (0xFFFFFFFF) appears frequently without comment. The value `206158430216` in varargs functions is unexplained. Some literals like `42` in `func_no_args` are self-evident.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise in varargs functions with 30+ variables (v19-v37, v0-v18) from register mapping. Type annotations like `uint128_t`, `unsigned long`, `unsigned long long` are mixed inconsistently. Many unused variable declarations clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most functions have natural if/else and return statements. The varargs functions (`varargs_func`, `param_varargs`) have complex control flow with do-while loops that are somewhat readable. No excessive goto usage. At -O0, control flow should be straightforward.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized (typically 5-30 lines). Core logic functions like `cdecl_func`, `param_by_value_int` are concise. Wrapper functions exist but are minimal. Some larger functions like varargs processing could be split but are acceptable.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom without forward jumps. The varargs loop is the main complexity but remains traceable. Early returns used appropriately in functions like `func_struct_byptr`.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No obvious dead code. Loop unrolling is minimal. Some redundant register spills in varargs functions but not severe. The code is generally clean of obvious redundancy.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Significant type degradation - `int` becomes `unsigned int` or `unsigned long` inconsistently. Parameters like `a0` in `cdecl_func` should be `int` not `unsigned int`. Function return types vary incorrectly (e.g., `call_cdecl` returns `unsigned long long` instead of `int`). `uint128_t` appears for no apparent reason.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Only minimal struct recovery attempted (`struct_0` in `param_double_ptr` with `field_0`). Most structs lost - `func_struct_byval` returns empty function. `LargeStruct` and `SmallStruct` not recovered. `SmallPoint` not recovered. Union types completely absent.

**11. Type Safety:** Score: 4/10  
Justification: Several unsafe patterns: `param_func_ptr` uses `unsigned long long *a0` instead of proper function pointer type. `ret_pointer` returns `long long` instead of `int*`. Pointer arithmetic in `param_ptr_array` uses raw offsets. Some incorrect pointer casts in `ret_large_struct`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments in the decompiled output. Only auto-generated function signatures present. No explanation of algorithm logic, no @param/@return documentation, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: Varargs pattern is present but not named or commented. Function pointer usage in `param_func_ptr` and `ret_func_ptr` is not identified as a pattern. No recognition of calling convention patterns (cdecl, stdcall, etc.).

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags. `func_struct_byval` returns empty function (severe issue) with no warning. `ret_opaque_handle` has incorrect return (`"d"` instead of pointer). No indication of these problems to the analyst.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. No error condition explanations. The `func_struct_byptr` null-check is not documented as a precondition.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are clear within test functions. Some external references documented (e.g., `extern char g_4020e0`). But no clear call graph documentation, no indication of which functions call which. Callback relationships not documented.

**17. Module Organization:** Score: 3/10  
Justification: Functions grouped by address, not by logical purpose. The test functions should be grouped together (they are), but the implementation functions are scattered. No static function scoping indicated. No logical grouping like "calling_conventions.c" or "parameter_passing.c".

**18. Build System Compatibility:** Score: 2/10  
Justification: Code has multiple compilation failures: empty returns in `func_struct_byval`, incorrect string literal in `ret_opaque_handle` (`"d"`), missing `#include <stdio.h>`, type mismatches throughout. Would require significant patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.9/10

**Critical Defects:**
- Variable naming is nearly unusable - all parameters are `a0`, `a1`, etc.; all locals are `v0`, `v1`, etc.
- `func_struct_byval` returns empty function - completely loses struct-by-value semantics
- `ret_opaque_handle` returns string `"d"` instead of pointer to static variable - semantically incorrect
- No comments whatsoever - cannot understand algorithm intent
- Type precision severely degraded - `int` becomes `unsigned int`/`unsigned long` inconsistently
- Struct recovery almost completely failed - only one generic `struct_0` recovered

**Best Quality Aspect:** Control Flow Naturalness (scored 7/10)  
The decompiled code maintains natural if/else structures and readable loops. At -O0, the control flow should closely match source and it mostly does.

**Most Needs Improvement:** Semantic Transparency (scored 2.25/10)  
The combination of no comments, no idiom recognition, no anomaly indication, and no contract recovery makes the code nearly impossible to understand without referring back to the original source.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Variable Name Inference**: Analyze usage patterns to infer semantic names. For example, `a0 + a1` in `cdecl_func` suggests parameters should be named `a` and `b`, not `a0` and `a1`.
2. **Struct Recovery**: Implement proper struct reconstruction for by-value parameters. `func_struct_byval` returning empty is a critical failure that loses all semantic meaning.
3. **Comment Generation**: Add basic documentation explaining what each function does, parameter purposes, and return value meaning. Even minimal comments would dramatically improve readability.
