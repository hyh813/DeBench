# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `v0`, `v1`, `v2`, `ptr`, `p`, `a0`, `a1`, `a2` throughout. For example, in `test_cpp_member_func()`, `v0` should be `value` based on assignment `v0 = 10`. No semantic meaning recovered.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some functions have proper C++ mangled names like `_Z20test_cpp_member_funcv`, `template_max<int>`, `Container<int>::push`. However, many functions use generic names like `sub_401020`, `sub_40102c`, `sub_401175` - no semantic meaning.

**3. Literal Clarity:** Score: 3/10  
Justification: String literals are not decoded - appear as `g_40207d`, `g_40209b`, etc. Magic numbers like `42` in exception throwing are present but unexplained. Expected at -O0 to have clearer literal recovery.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant noise present - unused variables like `flag1`, `result` in `test_cpp_member_func()` that serve no purpose. Many functions return values that aren't used. Some dead code from register spilling visible.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `call_virtual_func()` uses `goto *((long long *)*((long long *)&arg_0));` - a computed goto for vtable dispatch that's unreadable. Some functions have empty returns that don't match source logic.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are mostly preserved and match source granularity. Template functions like `template_max<int>` and `template_swap<int>` are properly extracted. However, some functions like `test_cpp_constructor()` are empty stubs losing all logic.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but has unusual patterns. The `call_virtual_func()` computed goto disrupts linear flow. Some forward jumps present in exception handling code.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundant code present - `strlen(&v1)` called but result unused in `test_cpp_member_func()`. Dead code from optimization at -O0 shouldn't be this prominent.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Types are generic - `unsigned long`, `unsigned int`, `void*` everywhere. In `template_swap<int>`, parameters are `int &ptr, int &p` but passed as `unsigned long` in caller. Const correctness mostly lost.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Some struct recovery attempted - `struct struct_0` with `field_0`, `field_8`. However, field names are generic. The `Base` class is declared but empty: `class Base { } Base;` - no members recovered.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe operations: `*((char **)ptr) = &g_403d78;` in RTTI code uses raw pointer arithmetic. In `Container<int>::push`, `ptr[40]` used for index without bounds verification. Many implicit casts between pointer types.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Minimal comments present. Only auto-generated header like "Angr Decompilation" at top. No explanation of vtable mechanism, exception handling, or other C++ features. No uncertainty indicators for unresolved calls.

**13. Idiom Recognition:** Score: 4/10  
Justification: Some C++ idioms partially recognized - vtable mechanism visible in `call_virtual_func()` computed goto, exception handling shows `__cxa_throw`, `__cxa_begin_catch`. However, not explicitly named or documented as patterns.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some anomalies present but not flagged. The `goto *((long long *)*((long long *)&arg_0));` in `call_virtual_func()` is highly unusual but no comment explaining it. Empty functions like `test_cpp_constructor()` not flagged as suspicious.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants recovered. No documentation of expected behavior. Empty functions like `test_cpp_constructor()` return without any indication of missing logic.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships mostly visible through code structure. `main` calls `test_cpp_oo_features`. Template instantiations properly linked. However, virtual dispatch relationships not clearly documented.

**17. Module Organization:** Score: 4/10  
Justification: Functions grouped by address rather than logically. Template functions scattered. C++ runtime functions mixed with user code. No namespace or class structure organization visible.

**18. Build System Compatibility:** Score: 2/10  
Justification: Code has many compilation issues - undefined extern variables like `g_403fe8`, `g_404008`. Empty function implementations. Missing class member definitions. Would require extensive patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.7/10

**Critical Defects:**
- Variable names are entirely generic (`v0`, `v1`, `ptr`, `p`) - no semantic meaning recovered
- `test_cpp_constructor()` is completely empty, losing all logic (should initialize `LifecycleClass` with data array)
- `call_virtual_func()` uses unreadable computed goto for vtable dispatch without explanation
- String literals not decoded - appear as `g_40207d`, `g_40209b` etc.
- `Base` class declared as empty `class Base { } Base;` - no data members recovered

**Best Quality Aspect:** Function Decomposition (scored 6/10)  
Functions mostly preserved at appropriate granularity. Template functions properly extracted as separate entities.

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
Every variable uses generic names with no semantic connection to source code intent.

**Top 3 Recommendations for Decompiler Improvement:**
1. Infer variable names from usage patterns - `v0 = 10` in SimpleClass context should yield `value` or `m_value`
2. Recover string literals from binary data instead of using placeholder names like `g_40207d`
3. Add comments explaining vtable dispatch patterns and flag empty/stub functions as suspicious
4. Recover class data members from constructor logic - `test_cpp_constructor()` allocates `data = new int[s]` showing `int* data` member
