# Decompilation Readability Assessment

## Target Function Analysis: `param_strcpy`

The original source shows a simple string copy function that calls `strcpy(dst, src)` and returns `strlen(dst)`. The decompiled version shows significant quality issues that I'll document across all 18 dimensions.

---

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: The decompiled code uses exclusively generic names like `ptr`, `a0`, `a1`, `v0`, `v1`, `v2`, `v3`, `len`, `count`, `result`, `i`, `j`. The original uses semantic names like `dst`, `src`, `buffer`, `input`, `filename`. Not a single variable name conveys meaningful purpose.

**2. Function Naming Quality:** Score: 7/10  
Justification: The target function `param_strcpy` is correctly named, matching the source. However, many supporting functions have cryptic names like `sub_10ca8`, `sub_11298`, `sub_112a8`. The function name preservation for the target function is good, but inconsistent across the codebase.

**3. Literal Clarity:** Score: 3/10  
Justification: The code contains unexplained magic numbers: `0x24120` (mutex address), `0x2413c`, `0x24158`, `0x24190` (atomic counter), and `4294967295` (0xFFFFFFFF for error returns). The original source has no such literals - they're compiler/runtime artifacts.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe noise pollution includes: stack canary checks (`GLIBC_2.4::__stack_chk_guard`), redundant `strlen()` calls (called twice in param_strcpy), spurious `armg_calculate_condition()` wrapper functions, and `__stack_chk_fail()` artifacts. The code is cluttered with compiler-generated artifacts.

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: The basic control flow (strcpy â†’ strlen â†’ return) is preserved, but the decompiler introduces `armg_calculate_condition()` wrapper calls that obscure simple comparisons. At -O0, this complexity is unnecessary. The while loops use awkward patterns like `while(true) { ... if(ptr != v2) goto LABEL_117c0; }`.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are reasonably separated. The target function `param_strcpy` is 6 lines, appropriately sized. However, some functions like `param_pthread_join` are 80+ lines with complex nested loops. The decomposition is acceptable but not optimal.

**7. Code Linearization:** Score: 4/10  
Justification: The code has multiple forward jumps (LABEL_11664, LABEL_116f4, LABEL_1170a) that disrupt linear reading. The param_strcpy function itself is relatively linear, but many other functions have complex goto patterns that make tracing execution difficult.

**8. Redundancy Elimination:** Score: 3/10  
Justification: Critical redundancy in param_strcpy: `strlen(ptr)` is called twice - once to assign to `len`, then again in the return statement. The original source calls it once and stores the result. This is a clear decompilation artifact.

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: The parameter `a1` should be `const char *src` but is declared as `char *a1` (missing const qualifier). The return type `int` is correct. Many parameters are generic `unsigned int` when more specific types (size_t, pid_t) would be appropriate. The original uses `char *dst, const char *src`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct types are mostly lost. The decompiler uses generic `struct_0`, `struct_1` with generic field names like `field_0`, `field_78`. The original defines `ThreadData` with meaningful fields (`start`, `end`, `result`). No struct recovery is evident.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe patterns exist: `(char)armg_calculate_condition(...)` casts, pointer arithmetic without bounds checking, and mixing signed/unsigned (returning `4294967295` for errors). The basic string operations (strcpy, strlen) are type-safe but the surrounding infrastructure has issues.

### === FINAL SUMMARY ===

**Overall Readability Score:** 3.6/10

**Critical Defects:**
- Variable names are entirely generic (`a0`, `a1`, `v0`, `v1`, `ptr`, `len`) - zero semantic meaning
- Redundant `strlen()` call in param_strcpy (called twice instead of storing result)
- Missing `const` qualifier on source parameter (`char *a1` should be `const char *a1`)
- No comments or documentation recovered from original source
- Unexplained magic numbers (0x24120, 0x24190, etc.) for runtime addresses
- Spurious `armg_calculate_condition()` wrapper obscuring simple comparisons

**Best Quality Aspect:** Idiom Recognition (7/10)  
The decompiler correctly identifies standard library functions (strcpy, strlen, memcpy, pthread_*). This is the highest-scoring area because the core libc function calls are properly named and recognizable.

**Most Needs Improvement:** Variable Naming Quality (2/10)  
The complete absence of semantic variable names is the most severe issue. Every variable uses generic register/stack names that require reverse-engineering their purpose through careful analysis. This single issue makes the code nearly unusable for practical analysis.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns**: Track how variables are used (e.g., `strlen(ptr)` â†’ `str`, `strcpy(dst, src)` â†’ `dst`, `src`) to generate semantic names
2. **Eliminate redundant function calls**: Detect when a function result is computed but discarded or computed twice (like the double strlen)
3. **Recover const correctness**: Analyze pointer usage to determine which parameters should be `const` - if a parameter is never written to, mark it const
