# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Parameters use generic names like `a0`, `a1`, `a2`, `a3` instead of semantic names. Local variables are `v0`, `v1`, `v2` etc. with no indication of purpose. For example, in `cdecl_func(unsigned int a0, unsigned int a1)`, the parameters should be `a` and `b` to match source.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some functions correctly named: `cdecl_func`, `stdcall_func`, `func_a`, `func_b`. However, many have auto-generated names like `sub_103c4`, `sub_1047c`, and compiler suffixes like `.constprop.0`. The test functions are properly named.

**3. Literal Clarity:** Score: 2/10  
Justification: String literals are replaced with addresses like `puts(0x1143c)` instead of actual strings. Magic numbers like `4294967295` (for -1) appear without explanation. The format strings are completely lost, showing only hex addresses.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe compiler artifact pollution. Stack protection code (`__stack_chk_guard`, `__stack_chk_fail`) pollutes many functions. Register annotations like `// r0`, `// r2`, `// [bp-0x4]` clutter every function. The `.constprop.0` suffix indicates optimized variants but adds noise.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Most functions have natural control flow, but some show awkward patterns. The `param_ptr_array` uses goto-based loops instead of for/while. The `ret_multi_branch` has deeply nested conditionals with multiple `armg_calculate_condition` calls that obscure the switch logic.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are reasonably sized and not monolithic. The decompilation correctly separates test functions (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`). Some inlining occurred (call_* functions return constants).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with minimal forward jumps. Some functions like `varargs_func` have do-while loops that are readable. The main disruption is from goto statements in `param_ptr_array`.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy in test functions - `__printf_chk` called twice with same arguments in `test_calling_conventions`. Loop unrolling present in `varargs_func` but not commented. Some dead code patterns visible.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation. All parameters are `unsigned int` regardless of actual type. Pointers shown as `unsigned int *`. No distinction between `int`, `char*`, `long long`. The source had `int` parameters but decompilation uses `unsigned int` throughout.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Partial struct recovery with generic names. `param_ptr_array` shows `struct_0` with `field_0`. `param_double_ptr` shows another `struct_0` with `field_0`. Original struct names like `SmallStruct`, `LargeStruct` are lost. Field names are meaningless.

**11. Type Safety:** Score: 3/10  
Justification: Multiple type safety issues. `param_by_value_ptr` has `*(ptr) = *(ptr) * 2 & 1` which is incorrect (should be `*ptr * 2`). The `ret_opaque_handle` shows `v1 = "d"` which is nonsensical. Pointer/integer confusion throughout.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: No meaningful comments. Only low-level register annotations like `// r0`, `// r2` and stack frame comments like `// [bp-0x4]`. No explanation of what functions do, no parameter descriptions, no uncertainty indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or algorithms identified. The varargs handling, function pointers, and complex casting patterns are not recognized or named. The original source's calling convention tests are not identified as such.

**14. Anomaly Indication:** Score: 2/10  
Justification: No flagging of suspicious code. The `param_complex_cast` has broken logic with `if ((vvar_1 != 0x1<32>)) { Goto None }` but no indication of decompilation failure. Unusual patterns like `armg_calculate_condition` are not explained.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original source had clear function contracts (e.g., "requires: ptr != NULL") but none are recovered. Error conditions not documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are present but relationships not documented. The caller-callee relationships between `call_cdecl` → `cdecl_func` are visible but not explained. No indication of which functions are test helpers vs. actual test targets.

**17. Module Organization:** Score: 5/10  
Justification: Some logical grouping exists - test functions grouped together, library functions at end. However, the clean separation in source (calling conventions section, parameter passing section, return values section) is not reflected in the decompiled output ordering.

**18. Build System Compatibility:** Score: 2/10  
Justification: Code would not compile. Many undefined references: `__stack_chk_fail`, `__printf_chk`, `armg_calculate_condition`, `__aeabi_dadd`. Undefined globals: `g_22008`, `g_22028`, `__TMC_END__`. No headers or dependencies documented.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.3/10

**Critical Defects:**
- Variable naming is unusable: all parameters are `a0`, `a1`, `a2`, `a3`; all locals are `v0`, `v1`, `v2`
- String literals completely lost - all replaced with hex addresses like `0x1143c`
- No comments explaining any function purpose or logic
- Severe type degradation: all parameters are `unsigned int` regardless of actual type
- Struct field names are generic: `struct_0`, `field_0` lose all semantic meaning
- Compiler artifacts pollute every function: `__stack_chk_guard`, register annotations, `.constprop.0` suffixes

**Best Quality Aspect:** Function Decomposition (7/10)  
Functions are reasonably sized and not monolithic. The test entry points (`test_calling_conventions`, etc.) are correctly identified and separated.

**Most Needs Improvement:** Comment Quality & Presence (1/10)  
The decompilation has zero meaningful comments. Every function needs explanation of purpose, parameters, return values, and any decompilation uncertainties.

**Top 3 Recommendations:**
1. **Recover variable names from usage patterns**: `a0` used in arithmetic should be named based on semantic role (e.g., `param1`, `count`, `index`)
2. **Recover string literals from .rodata**: The format strings like "CALL-L1-01: %d\n" should be extracted and used instead of hex addresses
3. **Add structural comments**: Each function should have a header comment explaining purpose, parameters, and any decompilation confidence level
