# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `a0`, `a1`, `v0`, `v1`, `v2`, `v3`, `v4`, `v5`, `v6`, `v7`, `v8`, `v9`. The original source uses meaningful names like `count`, `args`, `sum`, `i`, `ptr`, `val` - all completely lost in decompilation. This makes the code nearly impossible to understand without constant reference to the source.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most test functions retain their original names: `cdecl_func`, `call_cdecl`, `varargs_func`, `func_no_args`, `func_many_args`, `param_by_value_int`, etc. However, runtime initialization functions use generic names like `sub_10408`, `sub_104c0`. The core test functions are well-named.

**3. Literal Clarity:** Score: 3/10  
Justification: String literals are replaced with addresses like `0x12400`, `0x124c4`, `0x12540` without any string content recovery. Magic number `4294967295` appears frequently (representing -1 as unsigned) without explanation. The constant `42` in `func_no_args` is correctly preserved. Most numeric constants are unexplained.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe noise pollution throughout. Every variable has register annotations like `// r0`, `// r3`, `// lr` and stack frame annotations like `// [bp-0x4]`, `// [bp-0x10]`, `// [bp-0x14]`. These annotations appear on nearly every line, making the code extremely cluttered and difficult to read.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic control flow structures (if/else, for loops, switch) are preserved and recognizable. At -O0 optimization, the control flow should closely match source, and it does for most functions. Some minor awkwardness in complex functions like `__aeabi_dadd` but overall acceptable.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions are appropriately sized and match the source decomposition. Each test function is separate as in the original. No over-fragmentation or monoliths observed. Function boundaries align well with source intent.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns and breaks used appropriately. Some forward jumps exist but are manageable. The code flows reasonably well for most functions.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy present - many functions call themselves redundantly. Example: `v0 = cdecl_func(5, 10); return cdecl_func(5, 10);` - the function is called twice with the same result. This pattern appears in `call_cdecl`, `call_stdcall`, `call_fastcall`, and many others. Dead code from register spills adds to the redundancy.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Parameters use `unsigned int` instead of `int` throughout (losing sign information). Function pointers use `unsigned int *` instead of proper function pointer types like `int (*)(int)`. The `char*` type is correctly identified for string parameters. Overall significant type degradation - original `int` parameters should be signed.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Some struct recovery attempted (e.g., `struct_0 { char field_0[4]; unsigned int field_4; }`) but field names are generic (`field_0`, `field_4`). Original struct names like `LargeStruct`, `SmallStruct`, `TestIntPair` are completely lost. The code mostly uses raw pointer arithmetic instead of struct access.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe operations present. Function pointer type confusion - `param_func_ptr` takes `unsigned int *a0` but should be `int (*callback)(int)`. Pointer arithmetic visible but generally type-safe. No buffer overflow patterns that are obvious. Mixed signed/unsigned operations could cause issues.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Comments are almost entirely auto-generated register/stack annotations (e.g., `// r0`, `// [bp-0x4]`). No algorithm explanations, no decompilation confidence indicators, no API contracts. The comments add noise rather than clarity. Zero useful semantic comments about what the code does.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns or algorithms explicitly identified. Basic C idioms like varargs usage (`va_list`, `va_start`, `va_arg`, `va_end`) are present but not named or explained. The varargs pattern is visible in the code but not recognized as a standard idiom.

**14. Anomaly Indication:** Score: 1/10  
Justification: No flags for indirect calls or unusual patterns. The code silently presents potentially incorrect information without any warnings. No indication of uncertain decompilation decisions. The `call_thiscall` function returns constant `15` without explaining why (the C fallback path).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No pre/postcondition documentation. No invariant recognition. The function `func_struct_byptr` has an implicit null-check contract (`if (!p) return -1`) but this is not documented. Error return values like `-1` for failure cases are not explained.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are visible in the code. However, relationships between functions are not documented. No call graph information, no indication of which functions call which. The relationship between `call_cdecl` and `cdecl_func` is obvious from naming but not explicitly documented.

**17. Module Organization:** Score: 4/10  
Justification: Functions are organized by address (memory location) rather than by functionality. The original source groups functions by category (calling conventions, parameter passing, return values) but this organization is lost. All functions appear in a single flat list sorted by address.

**18. Build System Compatibility:** Score: 2/10  
Justification: Code has syntax issues - e.g., `unsigned int [2] ret_small_struct` is invalid C syntax. Missing `#include` statements. External function declarations present but incomplete. The code would require significant patching to compile. The decompilation output includes runtime library functions (`__aeabi_dadd`, etc.) that are not properly declared.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.7/10

**Critical Defects:**
- Variable names are entirely generic (`a0`, `a1`, `v0`, `v1`) - completely losing the semantic meaning from source (`count`, `args`, `sum`, `i`)
- Severe noise from register/stack annotations on every line (`// r0`, `// [bp-0x4]`)
- Redundant function calls throughout (`v0 = func(); return func();`)
- String literals replaced with addresses (`0x12400`) losing all string content
- Type precision severely degraded (`unsigned int` instead of `int`, function pointers as `unsigned int *`)
- No useful comments - only auto-generated register annotations

**Best Quality Aspect:** Function Decomposition (scored 8/10)  
Function boundaries match the source well, with appropriate granularity and no over-fragmentation or monoliths.

**Most Needs Improvement:** Variable Naming Quality (scored 2/10) and Noise Reduction (scored 2/10)  
The combination of generic variable names and pervasive register/stack annotations makes the code nearly unreadable. Every line is cluttered with meaningless annotations while semantic variable names are completely lost.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns**: Variables used as loop counters should be named `i`, `index`, `idx`; variables used as sizes/lengths should be `size`, `len`, `count`; pointers should indicate their target type
2. **Eliminate register/stack annotations from output**: These are useful for low-level debugging but destroy readability for semantic analysis
