# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `v0`, `v1`, `v2`, `v3`, `v4`, `v5`, `v6` with only register hints (e.g., `// r0`, `// r3`) or stack frame locations (e.g., `// [bp-0x4]`). No semantic meaning is conveyed - for example, in `array_1d_stack`, the variables `v0`, `v1`, `idx` should indicate `sum`, `array_element` based on their usage.

**2. Function Naming Quality:** Score: 4/10  
Justification: Some functions correctly preserve original names (`double_value`, `process_int`, `process_char`, `main`), but many use auto-generated names like `sub_103e8`, `sub_104b0`, `sub_104d4`, `call_weak_fn`, `deregister_tm_clones`. The mix of preserved and generated names is inconsistent.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers appear without explanation - e.g., `0x11ed4`, `0x1203f`, `0x1215b` for string pointers, `65` and `32` for ASCII conversion in `process_char`, `25` for range check, `65535` for short masking. Some are inferable from context but not documented.

**4. Noise Reduction:** Score: 3/10  
Justification: Heavy noise from register annotations (`// r0`, `// r3`, `// lr`), stack frame comments (`// [bp-0x4]`), pseudo-operations like `armg_calculate_condition` and `armg_calculate_flag_v`. Extensive use of `goto` for control flow creates additional visual clutter.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Extensive use of `goto` for loop constructs (e.g., `array_1d_stack` uses `goto LABEL_107f0` for iteration). While some control flow is readable, the goto-based loops are awkward. The `process_char` function has convoluted conditional logic with multiple goto paths.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are reasonably sized - most are under 50 lines. However, some are large: `test_array_types` is ~150 lines, and `__aeabi_dadd` is ~400 lines (library function). Overall decomposition is acceptable but could be improved.

**7. Code Linearization:** Score: 3/10  
Justification: Poor linearization due to extensive goto usage. Forward jumps into loop bodies (e.g., `goto LABEL_107f0`), computed jumps, and deep nesting in complex functions like `__aeabi_dadd` make the code difficult to read top-to-bottom.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No obvious dead code. Some loop unrolling present but not explicitly marked as such. Common subexpression elimination appears reasonable. Overall, the code is relatively clean of obvious redundancy.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Heavy use of generic `unsigned int` for most variables. Some type precision exists: `unsigned short` for `process_short`, `char*` for string operations. Function pointers are generic `unsigned int`. Integer types are not distinguished (int vs long vs long long all become unsigned int).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Some struct recovery with generic names: `struct struct_0 { unsigned int field_0; }` and `struct struct_1 { struct struct_0 *field_0; }`. Field names are generic (`field_0`, `field_1`, `field_4`). Union types not clearly distinguished from structs. Nested structures partially recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts present (e.g., `*((int *)&v0)`). Pointer arithmetic used extensively without bounds checking. Function pointer type confusion - `ptr_func_simple` uses `goto a0` which is dangerous. Some void pointer casts without validation.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments only indicate register allocation (`// r0`, `// r3`) and stack frame locations. No algorithm explanations, no documentation of non-obvious logic, no decompilation confidence indicators. Comments are purely technical, not semantic.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or algorithms are identified or named. The code doesn't recognize common patterns like state machines, callbacks, or data structure patterns. Everything is presented as raw decompiled logic.

**14. Anomaly Indication:** Score: 2/10  
Justification: No flagging of suspicious code or potential issues. No indication of uncertainty in indirect calls or unusual patterns. The decompiler presents all code with equal confidence, even when analysis may be uncertain.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. No error condition documentation. The code provides no information about expected input constraints or guaranteed output properties.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Function call relationships are somewhat inferable from the code but not explicitly documented. Global variable usage is not clearly indicated. No call graph or usage context provided.

**17. Module Organization:** Score: 2/10  
Justification: All code appears as a single monolithic block. No logical grouping of related functions. No separation between application code, library functions, and runtime support. No header-like declarations to show API boundaries.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code would require significant patching to compile. External function declarations are incomplete (many marked as `extern`). Dependencies on library functions (`__printf_chk`, `__stack_chk_fail`) not fully resolved. No build instructions or dependency documentation.

**Overall Readability Score:** 3.66/10

### Critical Defects
- All variables use generic `v*` names with no semantic meaning (e.g., `v0`, `v1`, `idx` should be `sum`, `element`)
- Extensive goto usage for loops instead of while/for constructs obscures control flow
- No struct field name recovery - all fields are generic (`field_0`, `field_1`)
- No comment content beyond register allocation hints
- No preconditions, postconditions, or invariant documentation

### Best Quality Aspect: Function Decomposition (6/10)
- Functions are reasonably sized with acceptable granularity
- Most functions stay under 50 lines
- Logical separation between different test categories maintained

### Most Needs Improvement: Assertion & Contract Recovery (1/10)
- Zero documentation of function contracts
- No indication of input constraints or expected behavior
- No error condition documentation
- Makes it impossible to understand function requirements without external reference

### Top 3 Recommendations for Decompiler Improvement
1. **Infer variable names from usage patterns**: Variables used as array indices should be named `index` or `i`, accumulators should be `sum` or `total`, based on how they're used in loops and operations.
2. **Recover struct field names from context**: Analyze how fields are accessed (e.g., `field_4` used as next pointer in linked list should be named `next`).
3. **Replace goto-based loops with while/for constructs**: Recognize backward jump patterns as loops and emit idiomatic loop structures instead of goto labels.
4. **Add contract documentation**: Detect null checks, bounds checks, and range validations to document preconditions.
5. **Flag anomalies and uncertainties**: Indicate when indirect calls couldn't be resolved or when analysis confidence is low.
