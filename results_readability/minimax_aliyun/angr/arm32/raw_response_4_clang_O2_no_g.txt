# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `a0`, `a1`, `v0`, `v1`, `v2`, `v3` throughout. For example, `cdecl_func(unsigned int a0, unsigned int a1)` and `varargs_func` uses `v5`, `v6`, `v7`, `ptr` - none convey semantic meaning like "parameter_a" or "sum".

**2. Function Naming Quality:** Score: 5/10  
Justification: Some functions correctly match source names (`cdecl_func`, `stdcall_func`, `test_calling_conventions`), but many use auto-generated patterns like `sub_4003f0`, `sub_400490`. The test functions are well-named but helper functions lack semantic clarity.

**3. Literal Clarity:** Score: 6/10  
Justification: Return values like `15`, `50`, `100` are computed correctly but unexplained. Magic number `4294967295` (0xFFFFFFFF) appears without constant naming. String literals referenced via external addresses (`&g_40170a`) are not human-readable.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe noise pollution includes register annotations (`// r0`, `// r1`, `// [bp-0x4]`), complex condition calculations (`armg_calculate_condition`), and massive artifact functions like `__adddf3` with 100+ meaningless variables. The floating-point library functions are essentially unreadable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Basic functions use natural if/else and loops, but the `__adddf3` function (lines 400f14-401324) contains spaghetti code with nested conditionals and computed jumps. Some goto statements present in library functions.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity matching the source. Each test function (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) is properly separated. No 500-line monoliths; decomposition closely follows original source organization.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom with minimal forward jumps. However, `param_double_ptr` has complex branching with multiple labels (`LABEL_400b05`, `LABEL_400b06`) that disrupt linear reading.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Some loop unrolling present in `varargs_func` and `param_varargs` without comments. The `call_varargs_param` calls `param_varargs` twice redundantly. Library functions contain significant dead code paths.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: All parameters typed as `unsigned int` instead of original `int`. Function pointers like in `param_func_ptr` use `unsigned int *a0` instead of proper function signature. Some pointer types correct (`char *a1` in `param_string`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct recovery - all struct parameters flattened to individual fields. `func_struct_byval` takes 30 `unsigned int` parameters instead of `LargeStruct`. `param_ptr_array` defines a dummy `struct_0` with only `field_0`. All semantic struct information lost.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe operations: `param_func_ptr` treats `unsigned int *a0` as function pointer (`a0(a1)`), `func_mixed_args` uses incorrect floating-point operations (`__adddf3`, `__floatdidf`), pointer arithmetic in struct functions without bounds.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only register location comments (`// r0`, `// [bp-0x4]`) present. No algorithm explanations, no API contracts, no decompilation confidence indicators. Comments are purely mechanical, not informative.

**13. Idiom Recognition:** Score: 4/10  
Justification: Varargs pattern is recognizable in `varargs_func` but not explicitly named as such. Function pointer callbacks visible but not identified as "callback pattern". No design pattern identification.

**14. Anomaly Indication:** Score: 1/10  
Justification: No anomaly flags whatsoever. Indirect calls, unusual stack layouts, and potential undefined behavior are silently presented as correct code. The `param_func_ptr` function pointer call is presented without warning.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition/postcondition documentation. Null checks in `func_struct_byptr` and `param_double_ptr` not explained as contracts. Error return values not documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: External functions declared with `extern` keyword. Global variables like `g_401460`, `ret_opaque_handle.handle1` referenced. However, call relationships not documented - no "Called by:" or "Calls:" comments.

**17. Module Organization:** Score: 6/10  
Justification: Functions logically grouped by test category (calling conventions, parameter passing, return values). However, runtime support functions (`__adddf3`, `__aeabi_drsub`) mixed in. Some arbitrary ordering present.

**18. Build System Compatibility:** Score: 4/10  
Justification: `extern` declarations present for external dependencies. However, no compilation attempted. Missing standard includes (`<stdio.h>`, `<stdarg.h>`, `<string.h>`). Undefined references to library functions would prevent compilation.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.8/10

**Critical Defects:**
- Variable names are entirely generic (`a0`, `a1`, `v0`, `v1`) - no semantic meaning recovered
- Struct types completely lost - `LargeStruct` flattened to 30 individual `unsigned int` parameters
- Function pointer type safety violated - `param_func_ptr` treats `unsigned int*` as function pointer without warning
- Severe noise in floating-point library functions makes them unreadable (100+ variables, complex conditionals)
- No comments explaining algorithm logic, only register location annotations

**Best Quality Aspect:** Function Decomposition (scored 8/10)  
- Function granularity closely matches original source; test functions properly separated

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
- All parameters and local variables use generic names like `a0`, `v1`, `v2` with no semantic recovery

**Top 3 Recommendations for Decompiler Improvement:**
1. Infer variable names from usage patterns (e.g., `a0 + a1` in `cdecl_func` suggests "addend1" and "addend2")
2. Recover struct types from parameter patterns - 30 consecutive `unsigned int` parameters should reconstruct `LargeStruct`
3. Add type safety warnings for function pointer calls through `void*` or `unsigned int*` - current output is dangerous
