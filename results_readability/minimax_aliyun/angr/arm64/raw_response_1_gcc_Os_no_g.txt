# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Parameters use generic names like `a0`, `a1`, `a2` instead of original names (`a`, `b`, `c`). Local variables are `v0`, `v1`, `v2`, `i` - completely generic. For example, `sequential_ops(unsigned int a0, unsigned int a1, unsigned int a2)` should be `sequential_ops(int a, int b, int c)`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Many functions correctly retain original names (`sequential_ops`, `single_if`, `if_else`, `recursion_factorial`, `double_value`, `triple_value`). However, some utility functions have cryptic addresses (`sub_400954`, `sub_400a20`). Test functions are properly named.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic number `4294967295` (0xFFFFFFFF) appears frequently without explanation. The expression `100 + 100 * a0` in `if_elseif_long` obscures the original `100*x` pattern. Some constants like array sizes are embedded in computations (e.g., `v1 = 85899345930` for `{10, 20, 30, 40, 50}`).

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__stack_chk_guard`) add noise. Redundant duplicate returns like `return register_tm_clones();` appear twice in `frame_dummy`. Some goto statements for control flow. However, no debug instrumentation or frame pointer manipulation clutter.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control flow is natural - loops use `for`/`while`, if-else is clear, switch statements are properly structured. However, `call_func_ptr` uses `goto a0` which is unusual. At -O0, control flow should be nearly perfect, but some goto patterns remain.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions maintain appropriate granularity (mostly under 30 lines). Test functions are large (~50 lines) but cohesive. Parameter lists are reasonable. No obvious outline artifacts - functions map to source functions well.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately. Some forward jumps in switch statements but manageable. The `goto` statements in `call_func_ptr` and `call_func_ptr_array` disrupt linearity slightly.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Minimal dead code observed. Some duplicate expressions like `return register_tm_clones();` appearing twice. Loop unrolling is minimal at -O0. Overall clean.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Parameters default to `unsigned int` or `unsigned long` when original was `int`. Function pointers use `unsigned long` instead of proper function pointer types. For example, `int (*f)(int)` becomes `unsigned long a0`. Signed/unsigned mismatches exist (original `int x` becomes `unsigned int a0`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: No struct recovery. Arrays are reconstructed as raw pointer arithmetic: `*((int *)&v1 + 4 * v6)` in `loop_break` should be `arr[i]`. The matrix in `loop_multi_exit` uses cryptic uint128_t values instead of proper 2D array.

**11. Type Safety:** Score: 5/10  
Justification: Several unsafe operations: pointer arithmetic like `a0 + v4 * 4` without bounds, function pointer treated as `unsigned long` for goto. Casts from unsigned to signed could cause issues. However, no obvious buffer overflow patterns.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only register annotations (`// x1`, `// [bp-0x10]`) and stack layout comments. No algorithm explanations, no decompilation confidence indicators, no API contracts. Comments are purely mechanical, not semantic.

**13. Idiom Recognition:** Score: 5/10  
Justification: Basic patterns recognized (loops, if-else, switch). However, no explicit naming of design patterns like "state machine" for `state_machine` function. The Duff's device in `duffs_device` is not recognized as such.

**14. Anomaly Indication:** Score: 2/10  
Justification: No flagging of suspicious code. Indirect calls like `goto *((long long *)((char *)&v1 + 8 * a0))` in `large_jump_table` are presented without warning. No indication of uncertain decompilation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. The `duffs_device` function has no comment explaining its unusual structure. Error conditions not documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: No documentation of which functions call which. No indication of callbacks or global usage. The relationship between `indirect_recursion_a` and `indirect_recursion_b` is not explained.

**17. Module Organization:** Score: 5/10  
Justification: Functions grouped by address but not logically organized. Static functions not properly scoped (all in global scope). No header file structure or API declarations.

**18. Build System Compatibility:** Score: 4/10  
Justification: External functions declared with `extern` (e.g., `extern char g_401c70;`). Would require significant patching to compile - missing definitions for `puts`, `__printf_chk`, etc. No dependency documentation.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.6/10

**Critical Defects:**
- Variable naming is completely generic (`a0`, `a1`, `v0`, `v1`) - should infer names from usage like `temp1`, `temp2` from source
- No struct/array recovery - raw pointer arithmetic `*((int *)&v1 + 4 * v6)` instead of `arr[i]`
- Type precision lost - `int` becomes `unsigned int`, function pointers become `unsigned long`
- No comments explaining algorithms or indicating decompilation confidence
- Function pointers handled with `goto` instead of proper call syntax

**Best Quality Aspect:** Control Flow Naturalness (7/10)  
- At -O0, loops and conditionals are correctly structured; the decompiler preserved the original control flow patterns well

**Most Needs Improvement:** Variable Naming Quality (3/10)  
- Generic single-letter names with numbers (`a0`, `v1`) make code nearly impossible to trace; should use semantic names based on usage patterns

**Top 3 Recommendations:**
1. Infer variable names from usage patterns - `a0 + a1` in first operation suggests `a` and `b`; `v1 += i` in loop suggests `sum`
2. Recover composite types - detect array access patterns like `base + index * element_size` and reconstruct as arrays
3. Add semantic comments - at minimum, document what each function does, parameter purposes, and flag uncertain decompilations (indirect calls, unusual constructs)
