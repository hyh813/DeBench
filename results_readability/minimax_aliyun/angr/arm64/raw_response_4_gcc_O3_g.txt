# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `v2`, `a0`, `a1`, `ptr`, `cur`. The original source has meaningful names like `count`, `args`, `sum`, but these are completely lost. Even loop indices use `v18`, `i` instead of descriptive names.

**2. Function Naming Quality:** Score: 5/10  
Justification: Core test functions like `cdecl_func`, `func_a`, `func_b`, `varargs_func` are correctly named. However, wrapper functions use auto-generated addresses (e.g., `sub_4007e0`, `sub_400a58`) with no semantic meaning. The original source's clear naming convention is only partially recovered.

**3. Literal Clarity:** Score: 6/10  
Justification: Most return values are correctly inlined (15, 50, 6, 100, 21, etc.). However, magic numbers like `4294967294` (0xFFFFFFFE) appear without explanation, and the bit manipulation in `param_by_value_int` (`(a0 * 2 | a0 >> 31) & 4294967294`) is obscure - likely a signed-to-unsigned fix that obscures the original `x * 2`.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: stack canary checks (`__stack_chk_guard`), redundant assignments (e.g., `v0 = frame_dummy(); return frame_dummy();`), and unusual bit operations that appear to be optimization artifacts. The code is cluttered with these non-semantic elements.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Basic control flow is preserved - loops use `while` constructs, conditionals use `if`. At `-O0` (debug), the flow should be nearly identical to source. Minor issues include some convoluted loop structures in varargs handling that could be cleaner.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized (20-80 lines). The original source's logical separation is mostly maintained. Some small wrapper functions exist (e.g., `sub_400a58` calling `call_cdecl`) but these are reasonable for thunk functions.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately. Some forward jumps exist in complex functions like `varargs_func` but are manageable. The overall flow is not severely disrupted.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Several redundant patterns observed: duplicate function calls (`v0 = frame_dummy(); return frame_dummy();`), dead assignments, and the unusual bit-masking in arithmetic operations that appears to be compiler-generated but obscures original intent.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation. Original `int` parameters become `unsigned int` or `unsigned long`. Pointers are often `unsigned long` (e.g., `param_double_ptr(unsigned int *ptr)` correctly uses pointer, but `func_mixed_args` uses `unsigned long` for what should be `double`). The original type information is significantly lost.

**10. Composite Type Recovery:** Score: 3/10  
Justification: The original `LargeStruct` (128-byte with `long long data[16]`) is completely lost - `func_struct_byval` returns `[D] unsupported_Iop_ZeroHI64ofV128()`. Small structs like `SmallStruct` are partially recovered but with generic field names. The semantic meaning of composite types is largely lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe patterns: unusual bit operations that could overflow, mixing of signed/unsigned in arithmetic, and the `ret_opaque_handle` returning `"d"` (a string literal) when it should return a pointer to static data. These create potential undefined behavior.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no useful comments. Only auto-generated notes like `/* do not return */` for abort calls. The original source has extensive comments explaining calling conventions, test scenarios, and expected results - all completely lost. No documentation of what each function tests or why.

**13. Idiom Recognition:** Score: 3/10  
Justification: Varargs handling shows some pattern recognition but is overly complex. The original `va_list`, `va_start`, `va_arg` idiom is replaced with manual pointer arithmetic. No recognition of calling convention patterns (cdecl, stdcall, etc.) beyond function names.

**14. Anomaly Indication:** Score: 4/10  
Justification: The `func_struct_byval` shows "unsupported" marker, but most anomalies are silently presented as normal code. No warnings about uncertain indirect calls, unusual stack layouts, or potential undefined behavior from the original source's intentional UB tests.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original source has clear contracts (e.g., "requires: ptr != NULL", "ensures: return is heap-allocated"). The decompiled code shows null checks but doesn't explain the contract they fulfill.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function call relationships are somewhat visible through the code structure. External dependencies like `__printf_chk`, `__libc_start_main` are marked as `extern`. However, no documentation of call graphs, caller/callee relationships, or data flow between functions.

**17. Module Organization:** Score: 4/10  
Justification: Functions are grouped by address proximity (0x400xxx) rather than semantic relatedness. The original source clearly groups by: calling conventions, parameter passing, return values. This logical organization is completely lost in the flat address-based layout.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code would not compile standalone. Missing definitions for many extern symbols (`__stack_chk_guard`, `handle1.1`, string data). Unusual constructs like `[D] unsupported_Iop_ZeroHI64ofV128()` are not valid C. Requires significant patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.0/10

**Critical Defects:**
- Variable naming is entirely generic (`v0`, `v1`, `a0`, `a1`) - no semantic meaning recovered
- Type precision severely degraded: `int` → `unsigned int`/`unsigned long`, pointers lost
- Composite types not recovered: `LargeStruct` shows "unsupported" marker
- No comments explaining calling convention tests, expected values, or test scenarios
- Unusual bit operations (`& 4294967294`) obscure original arithmetic intent

**Best Quality Aspect:** Control Flow Naturalness (7/10)  
- Basic loop and conditional structures are preserved and readable; at -O0, the flow closely matches source

**Most Needs Improvement:** Variable Naming Quality (2/10)  
- Every local variable uses generic names; the original source's meaningful names (count, args, sum, val, ptr) are completely lost

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns**: Track how variables are used (e.g., `strlen(v0)` → `v0` should be `str`; `*ptr * 2` → `ptr` should be `value_ptr`)
2. **Recover type information more precisely**: Use debug symbols and stack frame analysis to distinguish `int` from `unsigned int`, and preserve pointer types instead of converting to `unsigned long`
3. **Add semantic comments for test functions**: When decompiling test harnesses, include the expected values and test purpose (e.g., "// Tests cdecl calling convention, expects: 15")
