# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names like `a1`, `a2`, `a3` and local variables use `v1`, `v2`, `v3`, `result`. Compare to source where parameters have meaningful names like `count`, `args`, `str`, `x`, `y`. The decompilation loses all semantic meaning from variable names.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most test functions correctly preserve original names (`cdecl_func`, `varargs_func`, `param_by_value_int`). However, some utility functions remain as `function_1040`, `function_1050` with only address-based naming. Main test functions are well-named.

**3. Literal Clarity:** Score: 3/10  
Justification: Global variables initialized with unexplained hex values (`g1 = 0x3a9b9ce6`). String format pointers use complex offset calculations like `(char *)(v1 + (int32_t)&g7 - 0x1cae)` without explaining what strings they reference. Magic number `4` for pointer size is unexplained.

**4. Noise Reduction:** Score: 7/10  
Justification: Address range comments (`// Address range: 0x1000 - 0x102e`) add clutter but don't impede understanding. Otherwise clean with minimal compiler artifacts. Some redundant pointer casts present but manageable.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops use while constructs appropriately. Some complex ternary expressions like `v1 < 0 == ((v1 ^ a1) & (a2 ^ a1)) < 0 == (v1 != 0)` are harder to follow than source's simple ternary `x > y ? (z * 2) : (z + 10)`. At -O0, control flow should be more direct.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity matching source. Each test function (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) remains cohesive. No over-fragmentation or monoliths observed.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom. However, complex offset calculations for string pointers (`v1 + (int32_t)&g7 - 0x1cae`) create non-linear mental jumps. Loop constructs are clear but nested structure access uses pointer arithmetic disrupting flow.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code observed. Some loop unrolling in varargs functions is present but reasonable. No significant redundant expressions that would impede understanding.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `int32_t` consistently which is appropriate for x86. However, loses distinction between `int` vs `long long` from source. Function parameters like `func_mixed_args` show `int32_t a5, a6` where source has `long long ll`. Pointer types often just `int32_t` without specific type.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct types completely lost. Source has `SmallStruct`, `LargeStruct`, `BigStruct` with clear fields. Decompiled code uses raw pointer arithmetic: `*(int32_t *)(a1 + 4)` instead of `p->y`. The `func_struct_byval` uses `*(int32_t *)(8 * i + (int32_t)&v1)` losing all struct semantics.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe pointer-to-int casts throughout. Function pointers treated as `int32_t` (e.g., `param_func_ptr(int32_t a1)` should be `int32_t (*callback)(int32_t)`). Pointer arithmetic like `v2 += 4` for varargs lacks bounds checking. No `const` correctness preserved.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only address range comments present (`// Address range: 0x1210 - 0x1219`). No algorithm explanations, no decompilation confidence indicators, no API contracts. Compare to source which has extensive comments explaining each test case's purpose.

**13. Idiom Recognition:** Score: 5/10  
Justification: Varargs handling pattern is recognizable (va_list simulation via stack pointer). Struct passing by value shows characteristic stack layout. However, no explicit naming or documentation of these patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags. The architecture mismatch (source says arm64, decompiled is x86) isn't flagged. Complex offset calculations for string access aren't marked as uncertain. Code presents everything as confident decompilation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: ptr != NULL"). No postconditions (e.g., "ensures: return is heap-allocated"). No invariant recognition. Compare to source which has explicit null checks like `if (!p) return -1` that could document contracts.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships visible through direct calls. However, no documentation of who calls whom, no callback relationships documented. The `callback_func` static function in source isn't clearly marked as callback in decompiled output.

**17. Module Organization:** Score: 6/10  
Justification: All functions in single file. Related functions grouped together (calling convention tests, parameter passing tests, return value tests). Static functions like `func_a`, `func_b` properly scoped. But no header file structure recovered.

**18. Build System Compatibility:** Score: 7/10  
Justification: Includes standard headers (`<stdint.h>`, `<stdio.h>`, `<stdlib.h>`, `<string.h>`). Function prototypes declared. Should compile with minor adjustments. Some x86-specific functions like `__x86_get_pc_thunk_bx` would need stubs.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.5/10

**Critical Defects:**
- Variable naming uses only generic names (`a1`, `a2`, `v1`, `v2`) losing all semantic meaning from source
- Composite types completely lost - structs become raw pointer arithmetic (`*(int32_t *)(a1 + 4)`)
- No comments explaining algorithms, test purposes, or decompilation confidence
- Type safety severely compromised - function pointers treated as `int32_t`, no const correctness
- String literals replaced with complex offset calculations (`v1 + (int32_t)&g7 - 0x1cae`) without explanation

**Best Quality Aspect:** Function Decomposition (scored 8/10)  
- Functions maintain appropriate granularity matching source structure; test functions remain cohesive and well-organized

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
- All parameters and locals use generic names (`a1`, `v1`, `result`) making it nearly impossible to understand variable purposes without cross-referencing source

**Top 3 Recommendations for Decompiler Improvement:**
1. Infer variable names from usage patterns - `a1` used as count in loop should be named `count` or `num_elements`
2. Recover struct types from stack layout patterns - detect sequential field accesses and reconstruct struct definitions
3. Add semantic comments for complex operations - explain offset calculations, flag uncertain decompilations, document calling relationships
