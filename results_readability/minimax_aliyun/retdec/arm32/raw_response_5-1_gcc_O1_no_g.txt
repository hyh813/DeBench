# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v1, v2, v3, v4, str, result) with no semantic meaning. For example, `int32_t v1 = _Znwj(4);` should indicate it's an RTTI pointer, and `int32_t str` in `test_cpp_member_func` should be named `name` buffer based on its usage with `strncpy` and `strlen`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some functions are properly demangled (e.g., `_Z20test_cpp_member_funcv` → `test_cpp_member_func()`), but others use cryptic addresses like `function_10a9c`. The demangling is inconsistent - some functions have both mangled and demangled forms while others lack proper naming entirely.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers appear without explanation (e.g., `0x125c`, `0x4004000000000000` for 2.5). The value `71` returned directly from `test_cpp_multiple_inheritance` hides the computation. Some string literals are preserved but encoded (Chinese characters as hex).

**4. Noise Reduction:** Score: 7/10  
Justification: Relatively clean output with minimal compiler artifacts. Some unnecessary casts present (e.g., `(int32_t)v1` conversions), but no debug instrumentation or frame pointer manipulation clutter. The code is mostly readable without excessive noise.

**9. Type Precision:** Score: 6/10  
Justification: Uses `int32_t` consistently which is appropriate for ARM64. However, pointers are often treated as `int32_t` losing the pointer semantics. The `float64_t` return type for `template_max<double>` is correct. Mixed results.

**10. Composite Type Recovery:** Score: 4/10  
Justification: VTable structures are recovered (`struct vtable_1187c_type`) but with generic field names (e0, e1, e2, e3). Class member data is not reconstructed - instead raw pointer arithmetic is used (e.g., `*(int32_t *)(v2 + 4)`). No meaningful struct field names.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe operations present. The `call_virtual_func` has completely wrong return type semantics. Pointer arithmetic like `*(int32_t *)(v1 + 4 + *(int32_t *)(v1 - 12))` is error-prone. No bounds checking. Some casts are questionable.

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.5/10

### Critical Defects:
- **`call_virtual_func` is semantically incorrect**: Returns the pointer value instead of calling the virtual function through vtable dispatch - this is a complete functional failure
- **All variables use generic names (v1, v2, v3)**: No semantic meaning conveyed, making code flow extremely difficult to trace
- **Composite types not recovered**: Class member data accessed via raw pointer arithmetic (e.g., `*(int32_t *)(v2 + 4)`) instead of named struct fields
- **No idiom recognition**: Virtual function dispatch, exception handling, and templates shown as low-level runtime calls without high-level explanation
- **Magic numbers unexplained**: Values like `0x125c`, `71`, `1000` appear without context or named constants

### Best Quality Aspect: **Noise Reduction (7/10)**
The decompiled code is relatively clean without excessive compiler artifacts, debug instrumentation, or frame pointer manipulation. While not excellent, the code doesn't have significant noise pollution.

### Most Needs Improvement: **Variable Naming Quality (2/10)**
Every variable uses generic names (v1, v2, v3, v4, str, result) with no semantic meaning. This makes it extremely difficult to understand what each variable represents and track data flow through functions. At -O0 with debug symbols, the decompiler should be able to infer meaningful names from usage patterns.

### Top 3 Recommendations:
1. **Infer variable names from usage patterns**: Analyze how variables are used (e.g., `strncpy(&str, "Test", 31)` → name buffer, `strlen(&str)` → name length) to generate semantic names like `name_buffer`, `name_length`
2. **Fix virtual function call resolution**: The `call_virtual_func` must call through the vtable (e.g., `result = (*(vtable->e0))(obj, x)`) not just return the pointer
3. **Recover class member fields**: Instead of `*(int32_t *)(v2 + 4)`, generate `obj->value` or `obj->multiplier` based on class layout analysis
