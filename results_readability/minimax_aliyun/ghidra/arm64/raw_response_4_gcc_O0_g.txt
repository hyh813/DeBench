# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic names like `iVar1`, `iVar2`, `piVar2` with unnecessary `_local` suffixes (e.g., `a_local`, `b_local`). While some names like `sum`, `count` match source, the pattern of adding `_local` to all parameters creates noise without adding semantic value.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match the original source exactly (`cdecl_func`, `stdcall_func`, `varargs_func`, `func_no_args`). Some internal symbols like `FUN_00100750` are generic, but the core test functions are well-named.

**3. Literal Clarity:** Score: 5/10  
Justification: Inconsistent literal presentation - `0x2a` for 42, `0x14`/`0x1e`/`0x28` for 20/30/40 in hex when decimal would be clearer. The source uses decimal literals consistently; the decompiler introduces hex artifacts.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive warnings clutter output: "WARNING: Unknown calling convention", "WARNING: Variable defined which should be unmapped", "WARNING: Removing unreachable block". These 50+ warning lines obscure the actual code and make reading difficult.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control flow is natural - loops use `for` constructs, conditionals are clear. Minor issue: `param_by_value_int` shows `x << 1` instead of `x * 2`, which is an optimization artifact but semantically equivalent.

**6. Function Decomposition:** Score: 9/10  
Justification: Functions maintain appropriate granularity - each has single responsibility. The original 665 LOC expanded to 1548 LOC partly due to stack protector code, but core functions are well-decomposed.

**7. Code Linearization:** Score: 8/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately. The `varargs_func` has complex pointer manipulation but remains traceable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Element-by-element struct copying in `test_calling_conventions` (16 separate assignments for `large.data[i]`) is redundant - should use `memcpy` or aggregate initialization. This is a decompilation artifact.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Generic `int` used throughout. Function pointer type `_func_int_int` is non-standard (should be `int (*)(int)`). The `longlong` type lacks proper spacing. At -O0, more precise types like `int32_t` could be inferred.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Struct types recovered: `LargeStruct`, `SmallStruct`, `SmallPoint`, `BigStruct`, `TestIntPair`. Field names are generic (`data[0]`, `data[1]`) but layout is correct. The source uses meaningful field names like `x`, `y` which are preserved in `SmallPoint`.

**11. Type Safety:** Score: 5/10  
Justification: Several unsafe casts present: `param_complex_cast` uses raw `(int *)p` and `(char *)int_ptr` without validation. The `varargs_func` has complex pointer arithmetic with `args.__gr_offs` that could lead to undefined behavior. No `restrict` or safety annotations.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Only auto-generated warnings present (e.g., "WARNING: Unknown calling convention"). No meaningful comments explaining algorithm intent, parameter contracts, or non-obvious logic. The source has extensive comments that are completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: No design patterns or algorithm patterns explicitly identified. The varargs handling, though complex, is not labeled as such. Callback patterns not named.

**14. Anomaly Indication:** Score: 6/10  
Justification: Warnings present for unknown calling conventions and unmapped variables. However, the warning "WARNING: Unknown calling convention -- yet parameter storage is locked" is generic. Could better indicate which calling convention was expected.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants recovered. The source has no explicit assertions, but implicit contracts (e.g., "ptr != NULL" in `func_struct_byptr`) are not documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function call relationships are clear from code structure. However, no explicit documentation of who calls whom or callback relationships. The `callback_func` → `param_func_ptr` relationship is visible but not annotated.

**17. Module Organization:** Score: 7/10  
Justification: Functions logically grouped by test category (calling conventions, parameter passing, return values). Static functions properly scoped. However, all in single output rather than matching source file organization.

**18. Build System Compatibility:** Score: 5/10  
Justification: Code has compilation issues: `va_list` internal structure (`__gr_offs`, `__stack`) is architecture-specific. Stack protector references (`___stack_chk_guard`) require linking libgcc. Some type names like `longlong` are non-standard.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 5.7/10

### Critical Defects:
- **Excessive warning noise**: 50+ warning lines ("Unknown calling convention", "Removing unreachable block") obscure actual code
- **Generic variable names with `_local` suffix**: `a_local`, `b_local`, `count_local` add no value and create visual clutter
- **Inconsistent literal presentation**: Hex literals like `0x2a`, `0x14` instead of decimal 42, 20 when source uses decimal
- **Element-by-element struct copying**: 16 separate assignments for array copy instead of meaningful aggregate operation
- **No semantic comments**: All source comments lost; only auto-generated warnings remain

### Best Quality Aspect: Function Decomposition (8/10)
Functions maintain appropriate granularity and single responsibility. Core test functions like `cdecl_func`, `varargs_func` are well-separated and cohesive.

### Most Needs Improvement: Noise Reduction (3/10)
The decompiled output is buried under dozens of warning messages that make reading the actual code difficult. This is the most severe readability issue.

### Top 3 Recommendations:
1. **Suppress generic warnings**: Filter out "Unknown calling convention" and "Removing unreachable block" warnings that don't aid understanding
2. **Remove `_local` suffix pattern**: Don't add `_local` to all parameters - this is mechanical and adds no semantic value
3. **Preserve literal formatting**: Match source literal style (decimal vs hex) rather than introducing arbitrary hex representations
