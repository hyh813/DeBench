# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Most variables use generic names like `local_30`, `param_1`, `aiStack_2c`, `local_14`. Only a few like `global_counter` are semantically meaningful. The decompiler fails to infer purpose from usage patterns (e.g., `local_2c` should be `i` in loops).

**2. Function Naming Quality:** Score: 7/10  
Justification: Test functions correctly match source names (`local_vars`, `heap_basic`, `global_var_access`). Static helper functions like `static_helper` are properly identified. However, some initialization functions retain generic names like `FUN_00100920`.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x2a` (42) appear without explanation. String references use raw addresses like `&DAT_00102714`. Some literals like `0x800` (2048) are interpretable from context but not documented.

**4. Noise Reduction:** Score: 6/10  
Justification: Ghidra warnings about unreachable blocks and dead code elimination are present but minimal. Type casts are relatively contained. No significant debug instrumentation clutter. Some spurious warnings like "Heritage AFTER dead removal" add noise.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops are correctly recognized as `for` constructs. Conditionals are clear. However, `stack_alias` function has confusing control flow with a strange stack address comparison (`&stack0x00000000 == (undefined1 *)0x1c`) that doesn't match source logic.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity matching source. Test functions are logically grouped. No over-fragmentation or monolithic functions observed. Parameter counts generally match source.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately in error-checking code. Minor forward jumps present but manageable. No severe disruption to reading flow.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code from register spilling observed. Loop unrolling is minimal at -O0. Common subexpression elimination artifacts are limited. Clean code overall.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of `undefined4`, `undefined1`, `undefined8` instead of proper types (`int`, `char`, `long`). Integer types are generic `int` or `undefined4`. Pointers often generic `void*` or `undefined4*`. No `size_t` recovery. Signed/unsigned often lost.

**10. Composite Type Recovery:** Score: 3/10  
Justification: The `Point` struct is completely lost - treated as two separate int allocations in `heap_struct`. The `Node` struct appears as raw `malloc(0x10)` with integer field access instead of proper struct. `TreeNode` partially recovered but fields accessed via integer offsets.

**11. Type Safety:** Score: 5/10  
Justification: Pointer arithmetic is generally correct. Some unsafe operations present: `*(undefined4 *)(param_1 + 1)` in `memop_unaligned_access` bypasses proper alignment. Function pointer types are generic `code*` instead of proper signatures. No `restrict` or `_Atomic` annotations.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Minimal comments - mostly Ghidra auto-generated warnings like "WARNING: Removing unreachable block". No algorithm explanations. No documentation of non-obvious logic. No decompilation confidence indicators.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library functions (`malloc`, `free`, `memset`, `memcpy`) are correctly identified. Some patterns like linked list traversal are present but not explicitly named or commented. No design pattern recognition.

**14. Anomaly Indication:** Score: 4/10  
Justification: Ghidra provides some warnings about decompilation issues (unreachable blocks, dead removal). However, the `stack_alias` function's completely broken logic (comparing stack addresses) is not flagged as anomalous. No indication of undefined behavior in `dangling_pointer`.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: ptr != NULL"). No postconditions (e.g., "ensures: return is heap-allocated"). No invariant recognition. Error conditions not documented. Completely absent.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are correctly resolved. Global variable usage is transparent (`global_counter`, `global_array`). However, callback relationships (like `set_global_callback` → `call_global_callback`) are not documented. No call graph information.

**17. Module Organization:** Score: 6/10  
Justification: Functions are grouped logically by category (stack, heap, static, memory ops). Static functions properly scoped. However, no header file structure recovered. All functions in single output.

**18. Build System Compatibility:** Score: 5/10  
Justification: Code structure allows compilation with stubs. External dependencies (`extern_global_var`, `extern_function`) declared but not documented. Some type mismatches may require fixes. No build instructions.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 5.0/10

**Critical Defects:**
- Variable naming is almost entirely generic (`local_30`, `param_1`, `aiStack_2c`) - should infer from usage patterns
- `Point` struct completely lost - treated as two separate int pointers in `heap_struct`
- `stack_alias` function has completely broken logic (comparing stack addresses) not flagged as anomaly
- Type precision severely degraded - heavy use of `undefined4`, `undefined1`, `undefined8`
- No struct field recovery - `Node` and `TreeNode` accessed via raw pointer arithmetic

**Best Quality Aspect:** Function Decomposition (scored 8/10)  
Functions maintain appropriate granularity matching source structure. Test functions are logically grouped and parameter counts align with original signatures.

**Most Needs Improvement:** Variable Naming Quality (scored 3/10)  
Nearly all variables use generic names like `local_X` or `param_Y`. The decompiler should infer meaningful names from usage patterns (e.g., loop counters → `i`, `j`; size parameters → `n`, `size`).

**Top 3 Recommendations:**
1. Infer variable names from usage patterns - loop counters should be `i`, `j`; array indices should reflect their purpose
2. Recover struct types from allocation sizes and field access patterns - `malloc(8)` with two int fields should be `Point*`
3. Add anomaly warnings for suspicious logic like stack address comparisons that don't match source semantics
