# Decompilation Readability Assessment

## LEVEL 1: LEXICAL CLARITY

### 1. Variable Naming Quality
**Score: 2/10**

The decompiled code uses extremely generic variable names throughout. Most variables are marked as `Unresolved local var: int result@[???]` or similar. In functions like `param_asm_simd`, variables are named `iVar1` through `iVar6` with no semantic meaning. The original source used meaningful names like `arr`, `n`, `sum`, but these were not recovered.

### 2. Function Naming Quality
**Score: 7/10**

The core test functions (`param_macro_constants`, `call_macro_constants`, `param_conditional_compile`, etc.) have been correctly recovered with their original names. However, some internal functions like `FUN_00011030` and startup functions have generic names. The main functions under test are properly named, which is the critical aspect.

### 3. Literal Clarity
**Score: 5/10**

Magic numbers are present throughout without explanation. For example, `0x200` (512), `0x40` (64), `0xdef0` appear without context. Some values like `0x1e` for `_SC_PAGESIZE` are not explained. The decompiler did not recover the original macro names (MAX_SIZE, BUFFER_COUNT) and left bare numeric constants.

### 4. Noise Reduction
**Score: 4/10**

Significant noise pollutes the output. There are numerous "WARNING" comments about unknown calling conventions, unresolved local variables marked with `@[???]`, and auto-generated markers like `/* Unresolved local var: int sum@[???] */`. This makes the code cluttered and harder to read.

## LEVEL 2: STRUCTURAL INTELLIGIBILITY

### 5. Control Flow Naturalness
**Score: 7/10**

Control flow is generally natural - loops are rendered as `for` loops, conditionals as `if` statements. The assembly in `param_asm_clobber` was correctly converted to a C loop. At -O0, the control flow should be straightforward, and the decompiler largely achieved this.

### 6. Function Decomposition
**Score: 8/10**

Function granularity is appropriate. Each test function (`param_macro_constants`, `param_asm_basic`, etc.) is separate. The original source had good decomposition, and the decompiler preserved this. No monolithic functions or over-fragmentation issues observed.

### 7. Code Linearization
**Score: 8/10**

Code reads mostly top-to-bottom without excessive jumps. Early returns are used appropriately in functions like `param_dynamic_code`. The flow is linear and easy to follow.

### 8. Redundancy Elimination
**Score: 8/10**

No significant redundancy observed. Dead code from optimization is minimal at -O0. Loop unrolling is not present (as expected at -O0). The code is clean in this regard.

## LEVEL 3: TYPE SYSTEM FIDELITY

### 9. Type Precision
**Score: 4/10**

Types are generic throughout. The original source used `size_t` for page sizes, but the decompiler uses `size_t` in some places but `int` in others. Parameters like `int *arr` are correct, but many local variables are unresolved. The original `int` types are preserved but without the semantic precision of the source.

### 10. Composite Type Recovery
**Score: 3/10**

No struct/union recovery is evident. The code uses raw pointer arithmetic throughout. For example, in `param_asm_simd`, the decompiler uses individual array element accesses (`a[1]`, `a[2]`, etc.) rather than recovering the `__m128i` vector types from the intrinsics version.

### 11. Type Safety
**Score: 6/10**

Some unsafe operations present. The atomic operations use `LOCK()` and `UNLOCK()` macros which are compiler-generated. Pointer arithmetic is explicit (e.g., `(int)dst + 1`). Overall manageable but not ideal.

## LEVEL 4: SEMANTIC TRANSPARENCY

### 12. Comment Quality & Presence
**Score: 3/10**

Most comments are auto-generated warnings like "WARNING: Unknown calling convention" or "Unresolved local var". There is no explanation of what the functions do, no recovery of the original comments from the source. The useful content is minimal.

### 13. Idiom Recognition
**Score: 4/10**

Some patterns are recognized - the atomic operations show `LOCK()` and `UNLOCK()`. The SIMD operations are decompiled to scalar code. However, the original patterns (SSE intrinsics, inline assembly) are not identified or named as such.

### 14. Anomaly Indication
**Score: 5/10**

The decompiler does flag some anomalies with "WARNING" comments, particularly about unresolved local variables and unknown calling conventions. However, these are generic warnings without specific guidance on how to resolve them.

### 15. Assertion & Contract Recovery
**Score: 2/10**

No preconditions, postconditions, or invariants are documented. The original source had no explicit assertions, but the decompiler could have inferred some (e.g., "requires: counter != NULL" for atomic operations).

### Overall Readability Score: 5.3/10

### Critical Defects
- **Variable naming**: All local variables use generic names (`iVar1`, `result@[???]`) instead of semantic names like `sum`, `counter`, `page_size`
- **Type precision**: No recovery of composite types - `__m128i` vectors become individual element accesses; no struct reconstruction
- **Literal clarity**: Magic numbers like `0x200`, `0xdef0` appear without recovering original macro names (MAX_SIZE, etc.)
- **Comment quality**: Only auto-generated warnings present; no semantic explanation of what functions do
- **Cross-reference clarity**: No documentation of function call relationships or global variable usage

### Best Quality Aspect: Function Decomposition (8/10)
Function granularity is excellent - each test function is properly separated, matching the original source structure. The decompiler correctly preserved the function boundaries from the source.

### Most Needs Improvement: Variable Naming Quality (2/10)
The decompiler failed to recover any meaningful variable names. At -O0 with debug symbols (-g), variable names should be recoverable. The generic `iVar1`/`iVar2` pattern and `Unresolved local var` markers severely impede readability.

### Top 3 Recommendations
1. **Infer variable names from debug symbols**: At -O0 with -g, DWARF debug info contains variable names. The decompiler should use this to name variables like `sum`, `arr`, `n` instead of generic names.
2. **Recover macro constants**: Detect magic numbers that match common patterns (e.g., `0x200` = 512 = MAX_SIZE/2) and suggest or recover original macro names.
3. **Add semantic comments**: Instead of just "WARNING: Unknown calling convention", explain what the function does based on its implementation (e.g., "Implements atomic fetch-and-add operation").
