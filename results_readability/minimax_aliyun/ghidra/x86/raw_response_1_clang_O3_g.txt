# Decompilation Readability Assessment

## LEVEL 1: LEXICAL CLARITY

### 1. Variable Naming Quality
**Score: 2/10**

Justification: The decompiler failed to recover meaningful variable names for most local variables. Functions like `sequential_ops` show "Unresolved local var: int temp1@[???]", `loop_for_fixed` shows "Unresolved local var: int sum@[???]" and "int i@[???]", and `loop_while` shows "Unresolved local var: int count@[???]". The original source had clear names like `temp1`, `temp2`, `temp3`, `sum`, `i`, `count` - these should have been recoverable at -O0.

### 2. Function Naming Quality
**Score: 9/10**

Justification: All 20+ test functions retain their original names from the source: `sequential_ops`, `single_if`, `if_else`, `nested_if_2`, `loop_for_fixed`, `loop_while`, `goto_forward`, `goto_backward`, etc. The naming is semantically accurate and matches the source exactly. Minor deduction for some internal helper functions that could be better named.

### 3. Literal Clarity
**Score: 6/10**

Justification: Some literals are clear (e.g., `0x15` = 21 for sequential_ops test, `0x14` = 20 for single_if test), but many magic numbers remain unexplained. In `loop_for_fixed`, the complex formula `(n-1)*(n-2)` is presented without explaining it's the sum of 0 to n-1. The `0x12` (18) in loop_while/dowhile is unexplained - it's related to the digit counting logic.

### 4. Noise Reduction
**Score: 3/10**

Justification: Severe noise pollution from "Unresolved local var" annotations appearing throughout. Functions like `switch_small`, `loop_break`, `loop_continue`, `loop_nested`, `goto_forward`, `goto_backward` all have multiple unresolved variable markers. This creates visual clutter that significantly impedes readability. The "WARNING" comments about unreachable blocks add more noise.

## LEVEL 2: STRUCTURAL INTELLIGIBILITY

### 5. Control Flow Naturalness
**Score: 5/10**

Justification: Some functions maintain natural control flow (e.g., `sequential_ops` as single return statement, `if_else` as ternary), but others are problematic. `single_if` uses `x << (0 < x)` which is clever but obscure - the original `if (x > 0) x = x * 2` is much clearer. `loop_for_fixed` has an unrecognizable formula instead of a for loop. `loop_while` and `loop_dowhile` use unusual `while (0x12 < uVar2)` conditions instead of `while (x != 0)`.

### 6. Function Decomposition
**Score: 8/10**

Justification: Functions are appropriately sized - each test function is separate and focused. The decompiler correctly kept `test_control_flow_l1`, `test_control_flow_l2`, `test_control_flow_l3`, and `main` as separate functions. No over-fragmentation or monolith issues observed. The decomposition matches the source organization well.

### 7. Code Linearization
**Score: 7/10**

Justification: Most functions read top-to-bottom without excessive jumps. Early returns are used appropriately in functions like `if_else`, `multi_return`. The `goto_forward` function has been linearized into a single return statement. Minor issues with some complex conditions but overall good linearization.

### 8. Redundancy Elimination
**Score: 6/10**

Justification: `sequential_ops` is perfectly optimized to `(b + a) * 2 - c`. However, `loop_for_fixed` shows unrolled computation that could be cleaner. `loop_continue` has complex loop unrolling that obscures the simple "sum odd numbers" logic. Some dead code elimination is present but not comprehensive.

## LEVEL 3: TYPE SYSTEM FIDELITY

### 9. Type Precision
**Score: 8/10**

Justification: Most parameters correctly typed as `int` matching the source. Function pointer types are properly declared (e.g., `_func_int_int *f` in `call_func_ptr`). Minor issues with some unsigned/signed mixing (e.g., `(uint)x` in conditions). Overall type precision is good for -O0 code.

### 10. Composite Type Recovery
**Score: 3/10**

Justification: The decompiler failed to recover array and matrix structures. `loop_break` shows "Unresolved local var: int[5] arr@[???]" when the original had `int arr[] = {10, 20, 30, 40, 50}`. `loop_multi_exit` shows "Unresolved local var: int[3][4] matrix@[???]" when original had `int matrix[3][4]`. The array contents are not accessible. This is a significant degradation from the source.

### 11. Type Safety
**Score: 7/10**

Justification: Generally type-safe code. Pointer arithmetic in `duffs_device` is handled correctly with proper `src = src + 1` and `dst = dst + 1`. Function pointers are called correctly. Some minor issues with unsigned comparisons on signed values but no severe safety violations that would cause undefined behavior.

## LEVEL 4: SEMANTIC TRANSPARENCY

### 12. Comment Quality & Presence
**Score: 4/10**

Justification: Function-level comments identify the decompiler version and binary path, which is useful. However, the pervasive "Unresolved local var" annotations are not helpful comments - they're error messages. No explanation of algorithm intent (e.g., "counts digits in number" for loop_while). The test functions have hardcoded expected values instead of function calls, losing the ability to verify behavior.

### 13. Idiom Recognition
**Score: 6/10**

Justification: Some patterns recognized - `duffs_device` shows switch-case with goto labels for fall-through, which is correct. `switch_small` correctly identifies jump table pattern via `*(int *)(&DAT_00013700 + op * 4)`. However, `loop_for_fixed` doesn't recognize the "sum of 0 to n-1" pattern, and `loop_continue` doesn't identify the "sum of odd numbers" pattern. Could be better at naming common algorithms.

### 14. Anomaly Indication
**Score: 5/10**

Justification: Ghidra provides "WARNING" comments for unknown calling conventions and unreachable blocks, which is helpful. However, the "Unresolved local var" messages don't explain WHY variables couldn't be resolved or what would be needed to resolve them. No indication of potential decompilation confidence issues for complex functions like `goto_backward`.

### 15. Assertion & Contract Recovery
**Score: 3/10**

Justification: No preconditions, postconditions, or invariants are documented. For functions like `duffs_device` where `n <= 0` returns -1, this precondition is not documented. The contract that `loop_while` returns at least 1 for digit count is not indicated. The decompiler could infer more about expected input ranges.

## LEVEL 5: CONTEXTUAL COHERENCE

### 16. Cross-Reference Clarity
**Score: 5/10**

Justification: Function calls within test functions are visible (e.g., `sequential_ops(5, 7, 3)`). However, the test functions have hardcoded return values instead of actual function calls, losing the relationship. For example, `test_control_flow_l1` shows `printf("CF-L1-01 (sequential_ops): %d\n",0x15)` instead of calling `sequential_ops(5,7,3)`. This breaks the ability to understand call relationships.

### 17. Module Organization
**Score: 7/10**

Justification: Functions are grouped logically - all CF-L1 functions together, CF-L2 functions together, CF-L3 functions together, matching the source file structure. Static helper functions (op_add, op_sub, etc.) are placed near their usage. The organization reflects the original source reasonably well.

### 18. Build System Compatibility
**Score: 4/10**

Justification: The code would not compile as-is. Many functions reference unresolved symbols like `DAT_00013700`, `PTR_double_value_00015ec8`, `jump_buffer`. The test functions have hardcoded values instead of function calls, so even if symbols were defined, the logic wouldn't execute correctly. Requires significant patching to compile.

### Critical Defects

1. **Variable names completely unresolved**: All local variables in most functions show "Unresolved local var: int X@[???]" - the decompiler failed to recover any meaningful variable names despite -O0 compilation.

2. **Composite types lost**: Arrays like `int arr[] = {10,20,30,40,50}` and `int matrix[3][4]` are unrecoverable, showing only "Unresolved local var" markers.

3. **Test functions use hardcoded values**: `test_control_flow_l1` prints `0x15` instead of calling `sequential_ops(5,7,3)`, losing the ability to verify behavior and understand call relationships.

4. **Control flow transformations obscure logic**: `single_if` uses `x << (0 < x)` instead of if-statement; `loop_for_fixed` uses complex formula instead of for-loop; `loop_while` uses `while (0x12 < uVar2)` instead of `while (x != 0)`.

5. **Cannot compile**: References to `DAT_00013700`, `PTR_*` symbols, and `jump_buffer` are undefined, making the code non-compilable.

---

### Best Quality Aspect

**Function Naming Quality (9/10)** - All test function names match the original source exactly: `sequential_ops`, `single_if`, `if_else`, `nested_if_2`, `loop_for_fixed`, etc. The decompiler preserved the semantic intent in function names.

---

### Most Needs Improvement

**Variable Naming Quality (2/10)** - The decompiler failed to recover any local variable names from debug information. At -O0 with debug symbols, variable names like `temp1`, `temp2`, `sum`, `i`, `count` should be recoverable but instead show as "Unresolved local var".

---

### Top 3 Recommendations

1. **Infer variable names from debug symbols at -O0**: The original was compiled with `-g` debug symbols. Ghidra should extract `temp1`, `temp2`, `temp3`, `sum`, `i`, `count` from DWARF debug info. Ensure debug symbol parsing is enabled and working.

2. **Recover array/struct types from debug info**: Use DWARF to reconstruct `int arr[5]` and `int matrix[3][4]` with their initializers, not just show "Unresolved local var".

3. **Preserve function calls in test code**: Instead of hardcoding `printf("...: %d\n",0x15)`, emit `printf("...: %d\n", sequential_ops(5, 7, 3))` to maintain semantic relationships and enable compilation testing.
