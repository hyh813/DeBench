# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `param_1`, `param_2`, `local_d8`, `local_b8`, `uVar1`, `iVar1`, `piVar2`. These provide no semantic meaning - the decompiler should infer names like `count`, `sum`, `args` from usage patterns in functions like `varargs_func`.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are excellently preserved: `cdecl_func`, `stdcall_func`, `varargs_func`, `func_no_args`, `func_many_args`, `func_mixed_args` all match the original source exactly. The decompiler correctly recovered the intended naming.

**3. Literal Clarity:** Score: 5/10  
Justification: Return values like `0xf` (15), `0x32` (50), `0x2a` (42) are computed constants - acceptable. However, magic numbers like `0x29` (41) in varargs handling and `0xfffffffe` lack explanation. The hex format obscures the actual values being used.

**4. Noise Reduction:** Score: 6/10  
Justification: Ghidra-specific types (`undefined8`, `undefined4`, `code *`) add noise. Stack frame setup `int local_b8 [46];` and `int *local_d0;` create clutter. However, no debug instrumentation or register spilling artifacts pollute the logic significantly.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most functions have natural control flow. The `varargs_func` and `param_varargs` show complex loop structures due to stack/spill handling, but the logic is traceable. At -O0, the flow should be more straightforward.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity - each has a single responsibility. The caller functions (e.g., `call_cdecl`) are correctly inlined to return constants. Function sizes are reasonable (typically 5-20 lines).

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. The varargs functions have some forward jumps in the loop structure, but it's manageable. No excessive nesting observed.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No significant dead code. The `varargs_func` and `param_varargs` are duplicated (same logic appears twice) - this is because they're separate functions in source but could be recognized as identical patterns. Minor redundancy in stack offset calculations.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Generic types dominate: `int`, `long`, `undefined8`, `undefined4`. The original `int` parameters are preserved, but `uint` is used inconsistently (should be `unsigned int`). Function pointer type `code *` is uninformative - should be `int (*)(int)`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct recovery is poor. `func_struct_byval` uses raw stack offsets (`in_stack_00000008`, `in_stack_00000010`, etc.) instead of struct field access. `LargeStruct` and `SmallStruct` are completely lost. The decompiler shows the structure passing mechanism but doesn't reconstruct the type.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe patterns exist. In `param_double_ptr`, the cast `*(undefined4 *)*param_1` is type-unsafe. The `ret_small_struct` uses bit manipulation `param_1 | param_2 << 0x20` to return a struct via registers - this is correct ABI behavior but not type-safe in the decompiled form. Mixed results.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated comments present: "WARNING: Removing unreachable block", "WARNING: Do nothing block with infinite loop". No meaningful comments explaining calling convention behavior, varargs handling, or struct passing semantics. The source had extensive comments that are completely lost.

**13. Idiom Recognition:** Score: 4/10  
Justification: The varargs pattern (`va_list`, `va_start`, `va_arg`, `va_end`) is partially visible in the stack manipulation but not recognized as a standard idiom. No explicit naming of patterns like "varargs" or "AAPCS calling convention". The `call_*` functions returning constants could be recognized as "constant propagation after inlining".

**14. Anomaly Indication:** Score: 5/10  
Justification: Ghidra correctly flags unreachable blocks with warnings. However, the unusual struct return mechanism in `ret_small_struct` (bit packing into 64-bit) isn't flagged as anomalous. The indirect function pointer call in `param_func_ptr` isn't marked as potentially unsafe.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No pre/postconditions recovered. The original source had no explicit assertions, but the decompiler could infer: "param_1 != NULL" for pointer parameters, "count >= 0" for varargs, etc. None of this is documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are explicit and traceable. However, the relationships between test functions and their targets aren't documented. The `callback_func` static function isn't labeled as a callback. Virtual dispatch isn't distinguished from direct calls.

**17. Module Organization:** Score: 5/10  
Justification: Functions are listed in address order, not logically grouped. The original had clear sections: calling conventions, parameter passing, return values. This organization is lost. Static functions like `callback_func`, `func_a`, `func_b` are properly scoped but not grouped with their users.

**18. Build System Compatibility:** Score: 3/10  
Justification: The code uses Ghidra types (`undefined8`, `undefined4`, `code *`) that won't compile. External dependencies (`puts`, `printf`, `strlen`, `__libc_start_main`) are not declared. Would require significant patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.8/10

**Critical Defects:**
- Variable naming is completely generic (`param_1`, `local_d8`, `uVar1`) - should infer semantic names from usage
- Composite types (structs) are not reconstructed - `func_struct_byval` uses raw stack offsets instead of struct field access
- No comments explaining calling convention behavior, varargs handling, or struct passing
- Type precision is poor - uses `undefined8`, `code *` instead of proper types
- Ghidra-specific types prevent compilation without significant patching

**Best Quality Aspect:** Function Naming Quality (scored 9/10)  
Function names like `cdecl_func`, `varargs_func`, `func_many_args` perfectly match the original source, making it easy to identify the intended functionality.

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
All parameters and locals use generic indexed names. The decompiler should leverage usage patterns (e.g., `param_1` used as count in loop → `count`; `local_b8` used for argument storage → `args` or `varargs`) to provide meaningful names.

**Top 3 Recommendations:**
1. **Infer variable names from usage**: In `varargs_func`, `param_1` is used as loop bound → name it `count`; `local_b8` stores arguments → name it `args` or `varargs_buffer`
2. **Recover struct types**: Instead of `in_stack_00000008`, reconstruct `LargeStruct` with field names `data[0]`, `data[1]`, etc.
3. **Replace Ghidra types with standard C**: Convert `undefined8` → `uint64_t` or `unsigned long`, `code *` → proper function pointer signature, add `#include` statements for external functions
