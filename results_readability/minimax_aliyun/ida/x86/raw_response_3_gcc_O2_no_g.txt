# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (a1, v0, v1, v2, v3, v4, v5, v6, v7, v8, v9). No semantic meaning is conveyed - e.g., `a1` should be `x` or `input_value`, `v4` should be `result` or `sum`. This is a severe degradation from the source where variables like `arr`, `i`, `n`, `p` have clear meanings.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match source exactly: `double_value`, `local_vars`, `local_array`, `heap_basic`, `heap_calloc`, `memory_leak`, `dangling_pointer`. Internal functions like `sub_1030` are boilerplate. Main test functions correctly named.

**3. Literal Clarity:** Score: 7/10  
Justification: Magic number `0x14u` (stack canary) appears without explanation. Some hardcoded test values like `20`, `10`, `15` in test functions are acceptable since they match expected outputs. Constants like `42` in `address_of_local` are clear from context.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: `__readgsdword(0x14u)` stack canary checks in `local_array`, `large_stack_frame`, `vla_stack`, `alloca_usage`, and test functions. These add noise but don't completely obscure logic. The `__usercall` and `__cdecl` annotations are also noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control flow is natural - loops use `for`/`while` constructs, conditionals use `if`/`else`. At `-O0` (debug), minimal optimization artifacts expected. Some complex stack allocation code in `vla_stack` has awkward pointer arithmetic but remains readable.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions maintain reasonable granularity (20-80 lines). Test functions call actual implementations rather than being monolithic. Some internal functions like `sub_1140` are minimal. Parameter lists generally appropriate.

**7. Code Linearization:** Score: 8/10  
Justification: Code reads top-to-bottom with minimal forward jumps. Early returns used appropriately in error-checking code (e.g., `if (!v1) return -1;`). No spaghetti code or excessive goto usage.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No significant dead code. Some loop unrolling in test functions (hardcoded values printed directly). Minor redundancy in pointer assignments (v1 = v0; v0 = malloc...) but acceptable.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of generic types: `_DWORD*` instead of `int*`, `void*` for all heap allocations, `_BYTE` for char buffers. Original `int`, `size_t`, `Point*`, `Node*` types lost. Function pointers use generic signatures. At `-O0`, types should be more precise.

**10. Composite Type Recovery:** Score: 3/10  
Justification: No struct recovery - all structures accessed via raw pointer arithmetic. The `Point` struct (2 ints) is completely lost - accessed as `*v1 = 10; v1[1] = 20;` instead of `p->x = x; p->y = x*2;`. The `Node` struct similarly lost. Should show `struct Node { int data; struct Node *next; };`.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe operations: `(char*)&v8[-2] + v4` pointer arithmetic in VLA code, `*(_DWORD *)ptr` after free in `dangling_pointer` (intentional bug). Most casts are necessary for memory operations. No severe violations that would cause immediate harm.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No comments explaining algorithm logic, no decompilation confidence indicators, no @param/@return documentation. Only auto-generated function headers. Source had extensive comments (MEM-L1-01, MEM-L2-03, etc.) completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Some patterns recognizable: linked list traversal (`while (v4)`), tree creation/cleanup, memory allocation/free pairs. But no explicit naming like "linked list" or "binary tree" - just generic `_DWORD*` chains.

**14. Anomaly Indication:** Score: 3/10  
Justification: No indication of suspicious code. The `dangling_pointer` function accesses memory after free (undefined behavior) but no warning. The `double_free` function has the bug but no comment. No flags for potential issues.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: n > 0" for `alloca_usage`). No postconditions (e.g., "ensures: return is heap-allocated and must be freed"). Error return values not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are clear (`malloc`, `free`, `memcpy`). Global variables like `global_counter`, `global_array` visible. Callback relationships in `set_global_callback`/`call_global_callback` partially clear but function pointer type lost.

**17. Module Organization:** Score: 7/10  
Justification: Functions grouped logically: stack memory functions together, heap functions together, static/global functions together. Test functions separated. Matches source organization reasonably well.

**18. Build System Compatibility:** Score: 5/10  
Justification: Missing includes (`<stdio.h>`, `<stdlib.h>`, `<string.h>`, `<unistd.h>`, `<sys/wait.h>`). External declarations needed (`extern_global_var`, `extern_function`). Would require moderate patching to compile. Some functions have wrong return types (e.g., `set_file_static` returns `Elf32_Dyn**`).

2. **Recover struct types**: Detect allocation size (8 bytes = Point, 8 bytes = Node) and reconstruct struct layouts with field names
3. **Add semantic comments**: Include test scenario tags (MEM-L1-01), algorithm descriptions, and flag suspicious patterns (use-after-free, double-free)
