# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `a1`, `a2`, `result`) with no semantic meaning. For example, in `varargs_func`, `v3` holds the count parameter but isn't named `count`, and `v7` is a pointer to va_list arguments without indication.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are excellently preserved from source (`cdecl_func`, `varargs_func`, `test_calling_conventions`, `param_by_value_int`). This is the decompiler's strongest aspect, maintaining original semantic intent.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation (`0xffffffff`, `0x100000000`, `41` in varargs loop). String literals stored in globals (`g4`, `g5`) with encoded UTF-8 bytes obscuring their content.

**4. Noise Reduction:** Score: 4/10  
Justification: Assembly intrinsics remain (`__asm_movaps`, `__asm_movdqa`, `__asm_paddq`), redundant casts throughout, and frame pointer manipulation visible. Compiler artifacts not cleaned up.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Complex functions use excessive `goto` labels (`lab_0x1396`, `lab_0x13ab`) instead of structured loops. The `varargs_func` has nested while loops with goto breaks that obscure the original for-loop structure.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions properly separated, but many caller functions are constant-folded (`call_cdecl` returns 15 directly instead of calling `cdecl_func(5, 10)`), losing call relationship semantics.

**7. Code Linearization:** Score: 4/10  
Justification: Heavy use of forward jumps and goto labels disrupts top-to-bottom reading. The `varargs_func` has multiple exit points via goto that make execution flow difficult to trace.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Constant folding removed actual function calls (semantic loss), but some redundancy remains in type casting. Loop unrolling not present but control flow is unnecessarily complex.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type erosion - everything is `int64_t` or `int32_t`. Original `char*`, `double`, `va_list`, and struct types lost. `func_mixed_args` takes `int64_t str` instead of `char*`, and `double d` becomes `int64_t a3`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `LargeStruct` (128 bytes) became `int128_t` with no field access. `SmallStruct` pointer access is raw arithmetic (`*(int32_t *)(a1 + 4) * (int32_t)a1`). No struct definitions recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(int32_t *)`, `(int64_t *)`), pointer arithmetic without bounds checking. `param_by_value_ptr` has undefined behavior with uninitialized `v1` before dereference.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled code. No explanation of algorithm logic, no decompilation confidence indicators, no API documentation. Original source had extensive comments that are completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern not recognized (should show `va_start`/`va_arg`/`va_end` semantics). Struct passing conventions not identified. No pattern naming for common C idioms.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about uncertain decompilation. The complex varargs handling should flag uncertainty, but presents confidently. Assembly intrinsics left without explanation of why they couldn't be lifted.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `func_struct_byptr` requires non-NULL pointer). No postconditions or error condition documentation. Null checks present but not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible in code but relationships not documented. Global variables (`g4`-`g26`) used without indicating their purpose as string constants. Call graph inferable but not explicit.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with no logical grouping. Related functions (calling convention tests, parameter passing tests) not organized into sections. No header file separation.

**18. Build System Compatibility:** Score: 7/10  
Justification: Includes standard headers (`stdio.h`, `stdlib.h`, `string.h`), has function prototypes. Should compile with minor fixes. External dependencies (`__libc_start_main`, `__cxa_finalize`) declared.

**Overall Readability Score:** 4.1/10

### Critical Defects:
- **Variable `a1`, `a2`, `v1` should be named semantically** (e.g., `count`, `args`, `sum` in varargs functions)
- **Zero comments** - original source had extensive documentation completely lost
- **Type erosion** - `char*`, `double`, `va_list`, and struct types all became `int64_t`
- **Control flow uses goto** instead of structured loops in complex functions like `varargs_func`
- **Struct definitions lost** - `LargeStruct` became `int128_t`, `SmallStruct` access is raw pointer arithmetic

### Best Quality Aspect:
- **Function Naming Quality (9/10)** - Original function names preserved perfectly, making it easy to identify purpose of each function

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation or decompilation confidence indicators

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement variable name inference** from usage patterns (e.g., variables used in loop bounds should be named `count`, `limit`, `index`)
2. **Add decompilation confidence annotations** to indicate uncertain regions (especially for varargs handling and assembly intrinsics)
3. **Recover high-level C idioms** like `va_start`/`va_arg`/`va_end` patterns instead of raw stack manipulation code
