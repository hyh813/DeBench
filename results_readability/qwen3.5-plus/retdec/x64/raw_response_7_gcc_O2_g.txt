# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`a1`, `a2`, `v1`, `v2`, `v3`) with no semantic meaning. Original `x`, `result`, `str` are lost. Only occasional names like `dest_str` and `len` provide hints.

**2. Function Naming Quality:** Score: 6/10  
Justification: Original function names preserved (`param_fake_branch`, `call_opaque_predicate`), but several wrapper functions use generic names (`function_10d0`, `function_1100`). Critical issue: `param_opaque_predicate` signature shows 5 parameters instead of 1.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x100000000`, `90` (XOR key), and `0xffffffff` appear without explanation. String literals stored as `g1`, `g2` with UTF-8 encoded Chinese text. Original `0xDEADBEEF` eliminated by optimization.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__readfsqword(40)` stack canary checks throughout, excessive type casts (`(int64_t)`, `(int32_t)`, `(char *)`), inline assembly (`__asm_pxor`, `__asm_movaps`), and verbose struct initializations for `jmp_buf`.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Original `param_fake_branch` false conditions completely eliminated (over-optimization). `param_opaque_predicate` GCD loop recovered as while-loop but with complex bit manipulation. Some functions constant-folded (`call_fake_branch` returns `10` directly).

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved, but `param_opaque_predicate` has incorrect signature (5 params vs 1). Many single-line wrapper functions (`function_1100`, `function_1110`) for library calls add fragmentation.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with while loops recovered appropriately. Stack canary checks at function ends create minor non-linear patterns. No excessive goto usage.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Dead code from false conditions properly eliminated. Some redundant type casts remain. Constant folding applied appropriately to caller functions. No obvious loop unrolling artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `int64_t`, `int32_t`, `uint32_t` but with incorrect signatures (`param_fake_branch` takes `int64_t` vs original `int`). Excessive explicit casts throughout. `param_opaque_predicate` has 5 `int64_t` params instead of 1 `int`.

**10. Composite Type Recovery:** Score: 5/10  
Justification: `struct __jmp_buf_tag` recovered with fields `e0[8]`, `e1`, `e2` but generic names lose semantic meaning. `struct _TYPEDEF___sigset_t` recovered. No field names indicate purpose (should be `__jmpbuf`, `__mask_was_saved`, `__saved_mask`).

**11. Type Safety:** Score: 5/10  
Justification: Many unsafe casts: `(char *)`, `(int32_t *)`, `(int64_t *)`. Pointer arithmetic with casts throughout. `__longjmp_chk` takes `int64_t *` but used with struct pointers. No buffer overflow warnings despite original code containing them.
