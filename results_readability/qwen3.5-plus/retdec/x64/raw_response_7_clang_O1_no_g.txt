# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`a1`, `a2`, `v1`, `v2`, `v3`) with zero semantic meaning. Original source had descriptive names like `result`, `encrypted`, `decrypted`, `canary` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Original function names preserved accurately (`param_fake_branch`, `call_opaque_predicate`, `param_string_encryption`). Only helper functions like `function_1030` use generic naming.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere (`0xffffffff`, `0x100000000`, `90`, `0x4060`). String literals stored as hex-encoded globals (`g3`, `g4`) with UTF-8 Chinese characters unreadable without decoding.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive `& 0xffffffff` masking throughout, unnecessary type casts (`(int32_t)a1`, `(int64_t)v1`), and inline assembly artifacts (`__asm_xorps`, `__asm_movaps`) pollute the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Dead code elimination removed the always-false branches correctly, but `param_opaque_predicate` has convoluted nested conditions. While loops recovered but with awkward variable tracking (`v3`, `v4`, `v5`, `v6`).

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes are reasonable, but `param_fake_branch` collapsed to one line losing original structure. Library wrappers (`function_1030` for `strncpy`) are appropriately separated.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably. Some while loops have awkward variable reassignment patterns that disrupt flow (e.g., `v5 = v6; v6 = v5 + 1` patterns).

**8. Redundancy Elimination:** Score: 5/10  
Justification: Dead code eliminated correctly, but redundant masking operations appear throughout. The GCD loop in `param_opaque_predicate` has unnecessary variable shuffling.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Original `int` types degraded to `int64_t`/`int32_t` inconsistently. Pointer types often become `int64_t` (e.g., `param_null_pointer_deref` parameter `a1` should be `int*`). Lost signed/unsigned distinctions.

**10. Composite Type Recovery:** Score: 4/10  
Justification: `__jmp_buf_tag` struct recovered but with generic field names (`e0[8]`, `e1`, `e2`). No user-defined struct recovery. Bit-field struct in `param_implementation_defined` completely lost.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout (`(char *)str`, `(int32_t)a1`). Pointer arithmetic on `int64_t` types. `param_null_pointer_deref` returns uninitialized `v2` in the success path.
