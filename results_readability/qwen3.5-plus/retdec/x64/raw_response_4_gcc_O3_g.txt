# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (a1, a2, v1, v2, temp) with zero semantic meaning. In `func_mixed_args`, parameters should be `x`, `s`, `d`, `ll` based on source usage, but appear as `a1`, `str`, `a3`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Original function names preserved well (cdecl_func, call_cdecl, test_calling_conventions). However, helper functions like `function_1070`, `function_1080` are generic, and optimization artifacts appear in names like `varargs_func_constprop_0`.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere (0xffffffff, 0x100000000, 48, 56) without explanation. String literals stored as hex-encoded globals (g1-g18) making Chinese text unreadable. Return values pre-computed (15, 50, 6) instead of showing operations.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler artifacts: `__readfsqword(40)` stack canaries in every function, `__asm_*` intrinsics throughout, `& 0xffffffff` masking everywhere, redundant type casts. At -O0, this noise is unacceptable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `varargs_func` uses goto labels (lab_0x14e0) instead of proper for/while loops. Complex bit operations obscure simple comparisons. At -O0 optimization level, this control flow distortion is a decompiler failure.

**6. Function Decomposition:** Score: 5/10  
Justification: Many functions constant-folded to single returns (call_cdecl returns 15 directly). Functions split into _cold and _constprop variants showing optimization artifacts. Original boundaries preserved but internal structure lost.

**7. Code Linearization:** Score: 5/10  
Justification: Multiple goto statements disrupt reading flow (varargs_func, param_varargs). Some functions read linearly but others have backward jumps that should be loops. Inconsistent patterns throughout.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Repeated `& 0xffffffff` masking throughout all arithmetic. Duplicate stack canary checks in every function. Constant folding created redundancy where computation should occur at runtime.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 3/10  
Justification: Everything collapsed to int64_t/int32_t, losing original distinctions. Pointers are int64_t, structs are int128_t, function pointers are int64_t. No const correctness, no unsigned types. Original `char *`, `double`, `long long` distinctions completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: LargeStruct (128 bytes) becomes int128_t parameter. SmallStruct not recovered—func_struct_byptr takes int64_t instead of SmallStruct*. No struct field names recovered. All typedef structures completely lost.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts implicit in int64_t usage throughout. Pointer arithmetic on int64_t without proper typing. Function pointers treated as integers. The int128_t typedef for struct data is non-standard and problematic.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero meaningful comments in decompiled code. Only auto-generated header about compiler detection. No algorithm explanations, no parameter documentation, no anomaly warnings anywhere.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern not recognized (complex stack access instead of va_list). Loop patterns not recognized (goto instead of for/while). Struct access patterns lost. No design pattern identification whatsoever.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about constant folding, no indication that varargs handling is uncertain. Decompiler presents incorrect code confidently (call_cdecl returns 15 directly without showing the actual call to cdecl_func).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (NULL checks not highlighted as contracts). No postcondition inference. Stack canary checks present but not explained as safety mechanisms.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships visible through names, but actual calls often constant-folded away. Global variables (g1-g26) not clearly related to usage. Callback relationships (param_func_ptr) obscured by type erosion.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped somewhat logically (test functions together). However, helper functions (function_1070-10b0) interspersed randomly. Global variables listed separately. No header file organization.

**18. Build System Compatibility:** Score: 4/10  
Justification: External dependencies (__libc_start_main, __printf_chk) need stubs. Assembly intrinsics may not compile on all platforms. int128_t typedef is non-standard. Requires significant modification to compile standalone.

**Overall Readability Score: 3.6/10**

### Critical Defects:
- **Constant folding destroyed semantics**: Functions like `call_cdecl()` return pre-computed `15` instead of showing `cdecl_func(5, 10)` call
- **Complete type erosion**: All structs lost (LargeStruct→int128_t, SmallStruct→int64_t), pointers become integers
- **Varargs completely broken**: Complex stack manipulation instead of recognizable va_list pattern
- **Zero semantic comments**: No explanation of assembly intrinsics, optimization artifacts, or uncertain recoveries
- **Generic variable naming**: All parameters are `a1`, `a2`, `a3` with no indication of purpose or type

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Original function names mostly preserved, making it possible to correlate decompiled code with source

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any explanatory comments, confidence indicators, or anomaly warnings

### Top 3 Recommendations for Decompiler Improvement:
1. **Disable aggressive constant folding for -O0 binaries** - Preserve actual function calls and computations instead of pre-computing results
2. **Recover struct definitions from stack layouts** - Analyze memory access patterns to reconstruct LargeStruct, SmallStruct with field names
3. **Add confidence annotations** - Mark uncertain recoveries (varargs, function pointers) with comments like `// UNCERTAIN: varargs handling may be incorrect`
