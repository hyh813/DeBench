# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`a1`, `a2`, `v1`, `v2`, `temp_1`) with zero semantic meaning. For example, `varargs_func` uses `v1` through `v31` instead of meaningful names like `arg_count`, `sum`, or `va_list_ptr`.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are preserved from source (`cdecl_func`, `varargs_func`, `test_calling_conventions`). This is the decompiler's strongest aspect, maintaining original intent.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0xffffffff`, `0x100000000`, and `41` appear without explanation. String literals are preserved as globals (`g4`, `g5`) but with encoded UTF-8 bytes obscuring content.

**4. Noise Reduction:** Score: 4/10  
Justification: Assembly intrinsics remain (`__asm_movaps`, `__asm_movdqa`, `__asm_paddq`). Unnecessary type casts and bit-shift operations for 32-bit truncation clutter simple arithmetic.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: `varargs_func` and `param_varargs` use excessive goto labels (`lab_0x1356`, `lab_0x1386`, `lab_0x134f`) instead of recognizable for/while loops. This makes the va_list iteration logic nearly unreadable.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries are preserved, but many caller functions are constant-folded (`call_cdecl` returns 15 directly instead of calling `cdecl_func(5, 10)`), losing the call relationship.

**7. Code Linearization:** Score: 4/10  
Justification: Forward jumps into loop bodies (`goto lab_0x134f`) disrupt top-to-bottom reading. The varargs functions have 6+ label targets making execution path tracing difficult.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated assembly intrinsic sequences (`__asm_movaps` called 8 times in varargs functions) suggest unoptimized register spill recovery that wasn't cleaned up.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: All parameters degraded to `int64_t` or `int32_t`. Original `char*`, `double`, `va_list` types lost. `func_mixed_args` shows `str` as `int64_t` instead of `char*`, and `double` parameter `a3` has no floating-point type.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `LargeStruct` (128 bytes) became `int128_t`. `SmallStruct` with `x`/`y` fields is accessed via pointer arithmetic (`*(int32_t *)(a1 + 4) * (int32_t)a1`). No struct definitions recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(int32_t *)`, `(int128_t *)`). `func_mixed_args` uses assembly conversions (`__asm_cvtsi2sd`, `__asm_cvttsd2si`) indicating lost type information for floating-point handling.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments recovered. Original source had extensive documentation (CALL-L1-01, PARAM-L2-01, etc.) explaining each test case's purpose. None preserved.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern (`va_start`, `va_arg`, `va_end`) not recognized - appears as complex pointer arithmetic. Function pointer callback pattern in `param_func_ptr` reduced to simple addition.

**14. Anomaly Indication:** Score: 2/10  
Justification: Assembly intrinsics present without explanation. No warnings about uncertain decompilation (e.g., varargs handling, struct passing conventions). Code presents incorrect information confidently.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `func_struct_byptr` requires non-NULL pointer). Null checks exist but aren't annotated as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call graph visible through preserved names, but constant-folding hides actual call relationships. Global variables (`g4`-`g26`) have no semantic connection to their usage.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file without logical grouping. Original source had clear sections (4.1 Calling Conventions, 4.2 Parameter Passing, 4.3 Return Values) - all lost.

**18. Build System Compatibility:** Score: 6/10  
Justification: Includes standard headers and should compile, but assembly intrinsics (`__asm_movaps`, `__asm_hlt`) are compiler-specific and may not port. Some function prototypes mismatch implementations.

### Critical Defects:
- **Variable `a1`, `a2`, `v1`-`v31`** should be named based on usage (e.g., `arg_count`, `sum`, `va_list_ptr` in varargs functions)
- **Control flow uses goto labels** (`lab_0x1356`, `lab_0x1386`) instead of recognizable for/while loops in varargs handling
- **Struct types completely lost** - `LargeStruct` became `int128_t`, `SmallStruct` accessed via pointer arithmetic
- **Zero comments recovered** from extensive original documentation explaining each test case
- **Assembly intrinsics remain** (`__asm_movaps`, `__asm_movdqa`) obscuring simple operations

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Original function names preserved (`cdecl_func`, `varargs_func`, `test_calling_conventions`), maintaining semantic intent despite other degradations.

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete loss of all source documentation, making it impossible to understand test purposes without referring to original source.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement varargs pattern recognition** - Detect `va_start`/`va_arg`/`va_end` sequences and recover as readable loop constructs with proper variable names
2. **Recover struct definitions from memory access patterns** - Analyze pointer arithmetic to reconstruct field names and types (e.g., `a1+4` and `a1` access suggests `x`/`y` fields)
3. **Preserve source comments in debug builds** - When `-g` flag is present, extract and embed DWARF debug information as comments in decompiled output
