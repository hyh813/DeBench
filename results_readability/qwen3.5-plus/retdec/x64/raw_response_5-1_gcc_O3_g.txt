# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v1, v2, v3, str, result) with zero semantic meaning. For example, in `_Z13test_cpp_rttiv`, variables v4-v14 should be named based on their RTTI usage (type_info_ptr, cast_result, etc.).

**2. Function Naming Quality:** Score: 3/10  
Justification: Functions use mangled C++ names (`_Z20test_cpp_member_funcv`) instead of readable names. Some have demangled comments but the actual function names remain unreadable to humans.

**3. Literal Clarity:** Score: 4/10  
Justification: String literals like g1-g13 are hex-encoded (`"\xe6\xb5\x8b\xe8\xaf\x95..."`). Magic numbers like `40` in `__readfsqword(40)` and `8`, `16`, `32` in destructor calls lack explanation.

**4. Noise Reduction:** Score: 3/10  
Justification: Contains significant compiler artifacts: `__readfsqword(40)` stack canary checks, `__asm_movsd`, `__asm_cvttsd2si`, and redundant type casts throughout. These obscure the actual logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Where code exists (e.g., `_Z13test_cpp_rttiv`), control flow is understandable but uses low-level patterns. Many functions are empty stubs with no control flow to evaluate.

**6. Function Decomposition:** Score: 2/10  
Justification: Critical failure - 10+ test functions are empty stubs (`return;` only). Functions like `_Z20test_cpp_member_funcv`, `_Z20test_cpp_constructorv` lost all original logic despite being 20-50 lines in source.

**7. Code Linearization:** Score: 5/10  
Justification: Code that exists reads relatively linearly, but the prevalence of empty functions and early returns disrupts overall flow. Stack canary checks create artificial branching.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundancy from compiler artifacts (stack canary checks in every function), but no obvious loop unrolling or dead code duplication observed.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation - all C++ class instances reduced to `int64_t*`. Template functions partially recovered (`_Z12template_maxIiET_S0_S0_`) but parameter types often generic.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Vtable structures partially recovered (`vtable_3c70_type`, `vtable_3c98_type`) with function pointer fields. However, class layouts (SimpleClass, LifecycleClass, etc.) completely lost.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts: `(int64_t *)v1`, `(int32_t)(float32_t)`, `(void*)pa == (void*)pb` patterns. Pointer arithmetic on class objects without type safety.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated metadata comments (address ranges, line numbers, demangled names). No semantic explanations of what the code actually does or why.

**13. Idiom Recognition:** Score: 3/10  
Justification: C++ patterns (vtables, thunks, RTTI) partially recognized in function names (`_ZThn16_`, `_ZTv0_n24_`) but not explained. No design pattern identification.

**14. Anomaly Indication:** Score: 2/10  
Justification: No indication of decompilation uncertainty. Empty functions presented confidently without warning that logic was lost. No flags for unresolved calls.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Exception handling semantics (`__cxa_throw`, `__cxa_begin_catch`) present but contracts not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function call graph partially visible through calls, but class inheritance relationships not documented. Virtual dispatch through vtables not clearly explained.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file with some grouping (prototypes, globals, functions). No logical separation between test functions and class implementations.

**18. Build System Compatibility:** Score: 4/10  
Justification: Has includes and declarations, but many functions are stubs that won't produce correct behavior. External dependencies (C++ runtime) not clearly documented.
