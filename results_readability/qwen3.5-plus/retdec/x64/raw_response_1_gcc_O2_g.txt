# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`a1`, `a2`, `v1`, `v2`, `v3`) with no semantic meaning. Original source had meaningful names like `temp1`, `sum`, `count`, `result` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`sequential_ops`, `single_if`, `nested_if_2`), but helper functions use address-based names (`function_1080`, `function_1090`) and some are generic (`call_func_ptr`).

**3. Literal Clarity:** Score: 4/10  
Justification: Pervasive magic numbers (`0xffffffff`, `0x100000000`, `0x66666667`) from type casting and division optimization. String literals stored as hex-encoded globals (`g1`, `g2`, `g3`) without explanation.

**4. Noise Reduction:** Score: 4/10  
Justification: Stack canary checks (`__readfsqword(40)`, `__stack_chk_fail()`) add significant noise. Excessive bit masking (`& 0xffffffff`) throughout. Compiler runtime functions (`_init`, `_start`, `deregister_tm_clones`) clutter output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic if/else recovered reasonably, but loops often converted to while with complex conditions. `switch_fallthrough` loses fall-through semantics. Some functions use goto labels (`lab_0x1640_2`) where structured flow existed.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved correctly. Helper functions (`op_add`, `op_sub`) properly separated. However, test functions are monolithic (e.g., `test_control_flow_l1` spans 0x1700-0x1a40 with ~200 lines).

**7. Code Linearization:** Score: 5/10  
Justification: Forward jump labels (`lab_0x191a_2`, `lab_0x196d_2`) disrupt top-to-bottom reading. Stack canary checks create artificial jumps at function exits. Most code reads linearly but with notable interruptions.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Repetitive bit masking (`& 0xffffffff`) on nearly every arithmetic operation. Type conversion patterns duplicated throughout expressions. No dead code elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Original `int` parameters inflated to `int64_t` universally. Signed/unsigned distinctions lost (e.g., `op_div` uses `uint32_t` inconsistently). Function signatures don't match source (e.g., `loop_for_fixed` takes 5 parameters instead of 1).

**10. Composite Type Recovery:** Score: 6/10  
Justification: `jmp_buf` struct partially recovered with generic field names (`e0[8]`, `e1`, `e2`). No complex structs needed for this code, but recovered structures lack semantic field names.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts throughout (`(int32_t)a1`, `(int64_t)v1`). Pointer arithmetic obscured by type conversions. Function pointer types degraded (e.g., `call_func_ptr` parameters don't match original signature).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments explaining logic. Only auto-generated file headers present. Original source had extensive documentation for each function's purpose and test cases—completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: Control flow patterns not identified (no "factorial calculation" comments). Duff's device completely unrecognizable. State machine patterns in `state_machine` not annotated.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about uncertain decompilation. `computed_goto` returns address `0x2490` instead of implementing jump—critical failure not flagged. `duffs_device` broken (just returns dst pointer) without warning.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for loops). No postconditions inferred. Error conditions not explained. Original test case expectations completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`g1`, `g2`, `g3`) referenced without explanation. No call graph or dependency information provided.

**17. Module Organization:** Score: 6/10  
Justification: All functions in single file. Some logical grouping by test function (`test_control_flow_l1`, `l2`, `l3`). No header file separation or namespace structure.

**18. Build System Compatibility:** Score: 4/10  
Justification: Includes standard headers but references undefined symbols (`__readfsqword`, `__printf_chk`). Would require significant patching and stub implementations to compile standalone.

**Overall Readability Score:** 4.5/10

### Critical Defects:
- **Variable names completely generic** (`a1`, `a2`, `v1`) losing all semantic meaning from original (`temp1`, `sum`, `count`)
- **Type inflation** - all `int` parameters converted to `int64_t`, breaking type precision
- **Pervasive bit masking** (`& 0xffffffff`) on every arithmetic operation obscuring simple math
- **Critical functions broken** - `duffs_device` returns pointer instead of copying, `computed_goto` returns address instead of jumping
- **Zero documentation** - no comments explaining logic, test cases, or function purposes

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Most original function names preserved (`sequential_ops`, `nested_if_deep`, `switch_fallthrough`), providing semantic anchors for understanding

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Complete absence of explanatory comments; original source had extensive documentation for each function's purpose, test cases, and difficulty level—all lost

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement variable name inference** from usage patterns (e.g., loop counters should be `i`, `j`; accumulators should be `sum`, `count`)
2. **Eliminate redundant type casting artifacts** - remove unnecessary `& 0xffffffff` masks and normalize integer types to match original source width
3. **Add decompilation confidence annotations** - flag uncertain reconstructions (like `computed_goto` and `duffs_device`) with warnings for manual review
