# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `v3`, `a1`, `a2`) with no semantic meaning. Original source used descriptive names like `result`, `str`, `buffer`, `decrypted` which are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names are preserved from source (`param_opaque_predicate`, `call_tail_call_optimized`), but several functions are missing entirely (`param_fake_branch`, `param_instruction_substitution`, `param_integer_overflow`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound (`0x16d2bff0`, `0x4d3c2b1a`, `90`) without explanation. String literals are stored as hex-encoded globals (`g2`, `g3`) instead of readable strings, obscuring their content.

**4. Noise Reduction:** Score: 5/10  
Justification: Verbose `setjmp` initialization with struct literals creates significant noise. Redundant type casts (`(char *)str`, `(int32_t)&g18`) and global variable indirection for strings add unnecessary complexity.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loop structures are recovered but awkwardly expressed (e.g., `param_vectorized_loop` uses pointer arithmetic instead of array indexing). The opaque predicate logic in `param_opaque_predicate` is mangled with incorrect condition chaining.

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes are reasonable, but 6+ functions from original source are completely missing. Parameter lists sometimes expose implementation details (`int32_t * str2, int32_t * str` instead of `const char* encrypted, char* buffer`).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but pointer arithmetic loops (`v4 += 4; v5 += 4;`) disrupt natural reading flow. Early returns are present but not optimally placed.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Pointer increment patterns repeat unnecessarily (`v4 += 4; v5 += 4; v6 += 4;` in loop). The `decrypt_string` function has redundant variable assignments that could be simplified.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 6/10  
Justification: Uses `int32_t` consistently but loses semantic type information (everything becomes `int32_t` or `int32_t *`). Original `const char *`, `size_t`, `unsigned int` distinctions are erased.

**10. Composite Type Recovery:** Score: 7/10  
Justification: `jmp_buf` handling is present but verbose with struct literals. No complex structs in this code sample limits evaluation opportunity, but `setjmp` recovery shows reasonable type handling.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts present (`(char *)str`, `(int32_t)&g18`, `(void (*)(int32_t))&g1`). Function pointer `g1` is cast without type safety verification.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled output. Original source had extensive Chinese documentation explaining obfuscation techniques, expected values, and test scenarios—all lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition (GCD algorithm in opaque predicate, XOR decryption, tail recursion). The decompiler treats all code uniformly without identifying algorithmic patterns.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about signal handling, `setjmp/longjmp` usage, or potential undefined behavior. The division by zero and null pointer dereference functions should be flagged as exceptional.

**15. Assertion & Contract Recovery:** Score: 2/10

3. **Add decompilation confidence annotations** for signal handling, setjmp/longjmp, and missing functions to alert analysts to potentially unreliable recovered code.
