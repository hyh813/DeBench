# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Most variables use generic names (`v1`, `v2`, `v3`, `v4`) instead of semantic names like `i`, `index`, or `counter`. Some original names preserved (`large_buf`, `arr`, `mem`) but loop counters consistently degraded to `v2`, `v3`, `v4`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`local_array`, `heap_basic`, `vla_stack`, `test_stack_memory`). However, helper function `function_ffff0fa0` is cryptic and string globals renamed to `g1`, `g2`, `g3`.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `4 * n`, `8`, `12`, `20`, `40` for malloc sizes lack explanation. String literals encoded as hex escapes in globals (`g1`, `g2`). Constant `65` for 'A' not explained.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive redundant casts (`(int32_t)`, `(char *)`, `(uint32_t)`). Explicit pointer arithmetic (`4 * v2 + v1`) instead of array indexing. Frame pointer manipulation visible throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: For-loops converted to while with manual increment. `linked_list_heap` uses goto labels (`lab_0x10cf4`) instead of structured loops. `test_heap_memory` has multiple goto disruptions.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved matching source. However, `linked_list_heap` is severely fragmented with goto-based logic. Test functions preserved but with structural issues.

**7. Code Linearization:** Score: 5/10  
Justification: Forward jumps in `linked_list_heap` and `test_heap_memory` disrupt top-to-bottom reading. Multiple label targets (`lab_0x11060`, `lab_0x1107c`) create non-linear flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: No aggressive loop unrolling (appropriate for -O0). Some redundant pointer arithmetic calculations visible. Duplicate type casts throughout functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `int32_t` consistently but `size_t` degraded to `int32_t`. Pointer types sometimes wrong (`memop_bcopy` has `int32_t * src, char * dst` vs original `const void *src, void *dst`). Function pointer type preserved.

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - `Point` struct completely lost (`heap_struct` uses `malloc(8)` and `3 * x`). `Node` struct lost (raw pointer arithmetic `*(int32_t *)(v1 + 4)`). `TreeNode` struct not recovered. No struct definitions in output.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts (`(int32_t)mem`, `(char *)v1`). `dangling_pointer` returns `*mem` after free (preserved from source but dangerous). Pointer arithmetic without bounds checking throughout.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: **No comments whatsoever** in decompiled code. Original source had extensive comments explaining each test case (MEM-L1-01, HEAP-L2-01, etc.). No decompiler confidence indicators or algorithm explanations.

**13. Idiom Recognition:** Score: 3/10  
Justification: malloc/free patterns visible but not identified as memory management idioms. Loop patterns not recognized. No design pattern identification. String operations not labeled as such.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about `dangling_pointer` undefined behavior. No indication of potential `double_free` issues. Silent on dangerous memory patterns that should be flagged.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: Null checks preserved but not documented as preconditions. No postcondition inference (e.g., "returns heap-allocated memory"). Error conditions not documented.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but not documented. Global variable usage not explained. No call graph information. External symbols (`extern_function`, `extern_global_var`) declared but relationships unclear.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped somewhat logically (test functions together). Global variables at top. Test functions separate from implementation. Reasonable but not optimal organization.

**18. Build System Compatibility:** Score: 7/10  
Justification: Includes present. External function declarations exist. Metadata indicates compilation succeeds. Some adjustments needed for string literal handling.
