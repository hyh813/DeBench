# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `v3`, `a1`, `a2`, `result`) with no semantic meaning. For example, in `array_1d_stack`, `a1` should be `arr` and `a2` should be `n` based on usage pattern.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user function names preserved correctly (`process_char`, `array_1d_stack`, `struct_simple`), but helper functions have cryptic names (`function_3f8`, `function_404`, `_24_a`).

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation (4, 40, 100, 0x100000, 0x800000). String literals stored in globals with hex encoding (`g13 = "=== \xe6\xb5\x8b..."`) instead of readable text.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler runtime pollution (__aeabi_dmul, __mulsf3, __adddf3, register_tm_clones, _start, _init, _fini). Test functions contain hardcoded expected values instead of actual function calls.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: For loops converted to while loops with manual counter management (`while (v2 != 1) { v2 = v3; ... v3 = v2 - 1; }`). Floating point runtime functions use excessive goto labels (lab_0x15c4, lab_0x149c).

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved, but runtime functions mixed with user code. Some functions are trivial wrappers (`ptr_cast` just returns result), while floating point functions are excessively large (200+ lines).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom in simple functions, but complex functions have forward jumps disrupting flow. Test functions have interleaved initialization code that obscures logic.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions print hardcoded expected values instead of computing them (e.g., `printf("DT-L1-01...: %c\n", 97)` instead of calling `process_char('A')`). Array processing functions have duplicated loop patterns.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Severe type erosion - everything becomes `int32_t`. Original `char`, `short`, `long`, `long long`, `float`, `double` all lost. `process_float` takes `int32_t a1` instead of `float f`. Pointer types all degraded to `int32_t`.

**10. Composite Type Recovery:** Score: 1/10  
Justification: Complete failure - all struct/union/enum types lost. `Point3D` accessed as `*(int32_t *)(a1 + 4) + a1 + *(int32_t *)(a1 + 8)`. `struct_bitfields` has no struct definition, just raw bit manipulation. No field names recovered.

**11. Type Safety:** Score: 2/10  
Justification: Numerous unsafe casts throughout. Pointer arithmetic without type information (`a1 + 4`, `v1 += 40`). `ptr_void` loses type discrimination logic. Function pointers reduced to `int32_t` parameters.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments recovered from original source. No decompiler-generated comments explaining uncertainty or complex logic. Original Chinese comments completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Linked list traversal (`linked_list`), tree recursion (`binary_tree_sum`), and graph traversal appear as raw code without identification.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about incorrect decompilation. Critical errors silent: `process_double` returns constant `0.1` instead of `d/2.0+0.1`, `process_ll` ignores second parameter.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `arr != NULL`, `n > 0`). No postconditions. Null checks present in code but not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls preserved but without call graph documentation. Global variables (`g13`, `g14`) referenced without explanation of purpose. Callback relationships lost.

**17. Module Organization:** Score: 4/10  
Justification: All 87 functions in single file with no logical grouping. Runtime functions mixed with user code. No header file or API separation.

**18. Build System Compatibility:** Score: 5/10  
Justification: Includes present but many external functions undefined (__libc_start_main, __cxa_finalize). Would require significant stubbing to compile standalone.

**Overall Readability Score:** 3.7/10

### Critical Defects:
- **Complete type system collapse**: All types degraded to `int32_t`, losing char/short/long/float/double distinctions
- **Zero composite type recovery**: Structs (Point3D, Rect, Widget), unions, and enums completely lost - accessed via raw pointer arithmetic
- **Silent semantic errors**: `process_double` returns constant `0.1` instead of `d/2.0+0.1`; `process_ll` ignores second parameter
- **Test functions useless**: Hardcoded expected values instead of actual function calls, defeating test purpose
- **No comments recovered**: Original documentation and Chinese comments completely lost

### Best Quality Aspect: Function Naming Quality (scored 7/10)
- Most user-defined function names preserved accurately from original source

### Most Needs Improvement: Composite Type Recovery (scored 1/10)
- Complete failure to recover any struct/union/enum definitions, forcing all access through pointer arithmetic

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement struct/union type inference** from access patterns (e.g., `a1+4`, `a1+8` at consistent offsets indicates struct fields)
2. **Add semantic validation passes** to detect and flag incorrect decompilation (e.g., functions returning constants when source has computations)
3. **Preserve and translate source comments** including non-ASCII characters, with decompiler confidence annotations for uncertain regions
