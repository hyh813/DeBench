# Decompilation Readability Assessment

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names (`v1`, `v2`, `v3`, `temp`, `str`) with no semantic meaning. Only occasional hints like `derived`, `b`, `len` provide minimal context. Original source had meaningful names like `capture_by_value`, `instance_count`, `multiplier` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Functions retain mangled names (`_Z20test_cpp_member_funcv`) with demangling hints in comments, but wrapper functions use meaningless names (`function_1170`, `function_1180`). Original function names like `test_cpp_virtual_func` are recoverable but not presented cleanly.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive throughout (`0x125c`, `0x378c`, `0x3b5d`, `0x37f7`) with no explanation. String literals exist but aren't identified as format strings or messages. Original constants like `0x1234` for `getClassID()` are obscured.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__x86_get_pc_thunk_*` functions (10+ instances), stack canary checks (`__readgsdword(20)`), and register spilling variables. At `-O0` these should be minimal, indicating decompiler introduced noise.

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Heavy `goto` usage especially in `test_cpp_exception` (labels `lab_0x1740`, `lab_0x1745`). Loop constructs not recovered - original `for` loops become manual pointer arithmetic. Exception handling is convoluted with manual stack manipulation.

**6. Function Decomposition:** Score: 4/10  
Justification: Several critical functions just `return;` (`test_cpp_operator_overload`, `test_cpp_template_class`, `test_cpp_lambda`). Virtual dispatch fragmented into 20+ thunk functions (`_ZThn8_*`, `_ZTv0_n*`). Original cohesive class methods are scattered.

**7. Code Linearization:** Score: 3/10  
Justification: Poor linearity with frequent forward jumps disrupting flow. `test_cpp_exception` has 6+ goto labels. Exception handling code jumps backward and forward making execution path nearly impossible to follow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundancy from compiler artifacts but not severe. Multiple `__x86_get_pc_thunk_*` calls with identical patterns. No obvious dead code, but recovery of optimized patterns is inconsistent.

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `int32_t`, `void*`, `float64_t`. C++ classes become `int32_t*` pointers. No const-correctness recovery. Original types like `std::unique_ptr<int>`, `Container<int>`, `Base*` are completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Only one struct recovered (`_Unwind_Exception` with `e0` field). All C++ class structures lost - `SimpleClass`, `LifecycleClass`, `Point` become raw pointer arithmetic. No field names, sizes, or layouts preserved.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout (`(int32_t *)v2`, `(char *)v4`). Pointer arithmetic on `int32_t*` for object access. Virtual function calls use raw integer offsets. Original type-safe C++ code is now unsafe C.

### === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Source line ranges provided (`Line range: 41-50`) but no semantic explanations. No comments explaining algorithm logic, C++ features, or decompiler confidence. Original extensive comments (500+ lines) completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition for RAII, virtual dispatch, exception handling, or smart pointers. Template instantiations not identified as such. Diamond inheritance thunks present but not explained as virtual inheritance artifacts.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings for broken functions (`test_cpp_lambda` returns void instead of 85). No flags for unresolved indirect calls or stack layout issues. Code presents incorrect results confidently.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original expected values (e.g., "期望: 4704") not preserved as assertions. Exception contracts not recovered.

### === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but relationships undocumented. Virtual dispatch through thunks not explained. No call graph or dependency documentation. Original class inheritance hierarchy completely opaque.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file with function prototypes section. Some grouping by source module (`From module: /work/5-1.cpp`). No logical separation of classes, tests, or utilities.

**18. Build System Compatibility:** Score: 4/10  
Justification: Has includes and prototypes but many functions broken (return void when should return int). Would not produce correct results. External dependencies (`__cxa_*`, `_Znwj`) declared but not implementable.
