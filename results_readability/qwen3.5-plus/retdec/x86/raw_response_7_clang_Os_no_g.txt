# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `v3`, `a1`, `a2`, `result`) with no semantic meaning. Original source had meaningful names like `encrypted`, `decrypted`, `canary`, `heap_buffer` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Main function names preserved from source (`param_fake_branch`, `call_opaque_predicate`), but helper functions are generically named (`function_1040`, `function_1050`). Core functionality names are accurate.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation (`0x100000000`, `0x2d03`, `0x2b8c`, `90` for XOR key). Original had comments explaining `0xDEADBEEF` and encryption keys; decompiled code has none.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: `__x86_get_pc_thunk_*` functions (wrong architecture - x86 on ARM64 binary), `deregister_tm_clones`, `register_tm_clones`, uninitialized variables returned (`int32_t result; return result;`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Critical control flow lost - `param_fake_branch` collapses to `return result;` eliminating the fake branch logic entirely. `param_opaque_predicate` uses convoluted modulo arithmetic instead of clear conditionals. At `-O0`, this degradation is unacceptable.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved, but internal logic severely simplified. `param_fake_branch` (original: 15 lines with conditionals) becomes 1 line. Signal handlers (`div_zero_handler`) have malformed structure.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom with reconstructed while loops. However, awkward variable initialization patterns exist (`int32_t v1; // 0x10d0` then used without clear initialization).

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundant calculations visible in opaque predicate GCD reconstruction. `test_obf_opt_edge` has duplicate printf pattern code. No dead code removal of compiler boilerplate.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `int32_t`, `uint32_t` consistently but function signatures wrong (`param_fake_branch(int32_t result)` instead of `int x`). Pointer types often degraded to `int32_t`. Architecture mismatch (x86 thunks on ARM64).

**10. Composite Type Recovery:** Score: 5/10  
Justification: `jmp_buf` struct reconstructed with generic fields (`e0[8]`, `e1`, `e2`) instead of semantic names. No meaningful field recovery. Basic struct layout present but loses all semantic meaning.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts visible (`(void (*)(int32_t))(v3 - 50)` for signal handler). Pointer arithmetic on `int32_t` values throughout. Signal handler type confusion patterns not flagged.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled code. Original had extensive documentation explaining obfuscation techniques, expected values, and test scenarios. No decompiler confidence indicators or uncertainty warnings.

**13. Idiom Recognition:** Score: 3/10  
Justification: GCD algorithm in opaque predicate not identified. XOR decryption pattern not named. Signal handling with `setjmp`/`longjmp` pattern not documented. No algorithm recognition visible.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly warnings despite severe issues. `param_fake_branch` incorrectly simplified to identity function presented confidently. Architecture mismatch (x86 on ARM64) not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `x != 0` for division). No postconditions inferred. Original had expected return values in comments (`// 期望：10`); all lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but global variable usage unclear (`g1`, `g2` without context). No call graph documentation. Callback relationships (signal handlers) not clearly linked.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single file with no logical grouping. Original had section comments (7.1 Obfuscation, 7.2 Optimization, 7.3 Edge Cases); decompiled code has none. Functions listed somewhat randomly.

**18. Build System Compatibility:** Score: 6/10  
Justification: Includes standard headers and has function prototypes. However, references undefined globals (`g1`, `g2`). Metadata claims "Compilation Result: Yes" but code has architecture inconsistencies.

### Critical Defects:
- **`param_fake_branch` completely broken**: Original had two fake branch conditionals; decompiled version is just `return result;` - semantic meaning entirely lost
- **Architecture mismatch**: x86 thunk functions (`__x86_get_pc_thunk_*`) in ARM64 binary indicates serious decompilation error
- **Zero semantic comments**: All original documentation about obfuscation techniques, expected values, and test scenarios completely absent
- **Generic variable naming**: All variables are `v1`, `v2`, `a1`, `result` - no inference from usage patterns (e.g., XOR key `90` should be `decrypt_key`)
- **Signal handler corruption**: `div_zero_handler` and `segv_handler` have malformed structure with incorrect `longjmp` usage

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Core function names preserved from source (`param_fake_branch`, `call_opaque_predicate`, `param_vectorized_loop`), making basic navigation possible despite other issues.

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Complete absence of any documentation, confidence indicators, or semantic explanations makes understanding the decompiler's intent impossible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable inference**: Analyze usage patterns to name variables (e.g., variable used in XOR operation with string → `decrypt_key`; variable compared to 0 in division → `divisor`)
2. **Add decompilation confidence annotations**: Flag uncertain reconstructions with comments like `// DECOMPILER_UNCERTAIN: Control flow may be incorrect` especially for optimized-away branches
3. **Preserve architecture metadata**: Detect and report architecture mismatches; don't emit x86-specific constructs for ARM64 binaries; add architecture validation pass
