# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Almost all variables use generic names (`v1`, `v2`, `a1`, `a2`, `result`) with no semantic meaning. Original `temp1`, `sum`, `count`, `i`, `j` are completely lost. Example: `loop_while` uses `v1`, `v2`, `v3`, `v4` instead of meaningful `count` and working variable.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions preserved (`sequential_ops`, `single_if`, `nested_if_2`), but many helpers degraded to `function_1040`, `function_20c1`, `function_20d5`. Original helper names like `state_idle`, `double_value` are preserved.

**3. Literal Clarity:** Score: 4/10  
Justification: Compiler optimization artifacts appear as magic numbers: `0x66666667` for division, `0x100000000`, `0x8000000000000000`. Original code had no magic numbers. Constants like `10`, `100` remain clear in context.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__x86_get_pc_thunk_*` stub functions, redundant casts `(uint32_t)(int32_t)(bool)`, frame pointer manipulation. Many stub functions (`function_1090`, `function_10cc`) add noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `single_if` degraded to `return a1 << (uint32_t)(int32_t)(bool)(a1 >= 0 == (a1 != 0))` instead of simple if. `switch_fallthrough` logic is completely broken. `loop_while` uses complex division recovery instead of simple `/10`.

**6. Function Decomposition:** Score: 4/10  
Justification: Function boundaries preserved but signatures corrupted. `recursion_factorial` has extra `uint32_t a2` parameter. `call_func_ptr` and `conditional_func_ptr` are broken stubs returning uninitialized values.

**7. Code Linearization:** Score: 5/10  
Justification: Most code reads top-to-bottom, but `loop_multi_exit` and test functions use excessive goto labels (`lab_0x17b0`, `lab_0x1822`, `lab_0x1832`) disrupting flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Optimization recovery creates redundancy: `loop_while` has complex bit-shift division instead of `/10`. Multiple unnecessary type casts throughout. Some dead code from register spilling.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Uses `int32_t`/`uint32_t` but with mismatches. `switch_small` takes `uint32_t a2` instead of `int32_t op`. Function pointer types completely lost (`call_func_ptr` takes `int32_t` instead of function pointer).

**10. Composite Type Recovery:** Score: 5/10  
Justification: `struct __jmp_buf_tag` recovered but with generic fields `e0`, `e1`, `e2`. No meaningful field names. Array handling degrades to pointer arithmetic.

**11. Type Safety:** Score: 3/10  
Justification: Many unsafe casts `(uint32_t)(int32_t)(bool)`. Function pointer calls lose type information. `call_func_ptr` cannot properly invoke function pointers. Pointer arithmetic without proper typing.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: ZERO comments in decompiled code. Original had extensive documentation for each function. No decompiler confidence indicators or algorithm explanations.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. Factorial and digit-counting algorithms not recognized. Switch statements sometimes recovered but not named as dispatch patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: Silent failures throughout. `computed_goto` always returns -1 with no warning. `large_jump_table` returns -1 always. `call_func_ptr` broken with no indication.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or error documentation. Original had test validation comments; none recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships undocumented. Global variables `g1`, `g2`, `g3`, `g4` unexplained. Callback relationships lost (function pointers).

**17. Module Organization:** Score: 6/10  
Justification: All functions in one file matching original. Function prototypes at top, globals section present. Cluttered with stub functions and compiler artifacts.

**18. Build System Compatibility:** Score: 4/10  
Justification: Standard library includes present. Many undefined globals (`g1`-`g4`). Function pointer functions won't compile correctly. Requires significant patching.

**Overall Readability Score:** 3.9/10

### Critical Defects:
- **Zero comments** - Original had extensive documentation; decompiled has none
- **Generic variable names** - `v1`, `v2`, `a1`, `a2` obscure all semantic meaning
- **Function pointer handling broken** - `call_func_ptr`, `conditional_func_ptr` return uninitialized values
- **Silent failures** - `computed_goto`, `large_jump_table` always return -1 with no anomaly warning
- **Compiler optimization artifacts** - Magic numbers like `0x66666667` appear without explanation

### Best Quality Aspect: Module Organization (scored 6/10)
- Functions grouped in single file with prototypes at top, matching original structure reasonably well

### Most Needs Improvement: Semantic Transparency (averaged 1.75/10)
- Complete absence of comments, no idiom recognition, no anomaly warnings, no contract recovery

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover semantic variable names** from usage patterns (e.g., loop counters should be `i`, `j`; accumulators should be `sum`, `count`)
2. **Add confidence indicators** for uncertain decompilations (mark broken function pointer calls, unresolvable jumps with warnings)
3. **Recognize common algorithms** and add explanatory comments (factorial, digit counting, switch dispatch tables)
