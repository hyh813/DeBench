# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v1`, `v2`, `v3`, `result`, `this`) with zero semantic meaning. For example, in `_Z20test_cpp_member_funcv`, the `SimpleClass` object is just `v3` instead of `obj`, and return values are `v4`, `v5` instead of `r1`, `r2`.

**2. Function Naming Quality:** Score: 4/10  
Justification: Functions retain mangled C++ names (`_Z20test_cpp_member_funcv`) with demangled comments. Test functions are identifiable but not human-readable; helper functions like `function_1040` have no semantic meaning.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation: `0x1234` for class ID is preserved, but offsets like `0x7e50`, `0x7dc8` are unexplained. String literals like `" ,"` in `_Z20test_cpp_constructorv` appear nonsensical.

**4. Noise Reduction:** Score: 3/10  
Justification: Massive noise from C++ runtime: 257 detected functions including vtable thunks (`_ZThn8_N12MultiDerived5funcBEv`), template instantiations, and STL internals (`_ZNSt15__uniq_ptr_impl...`). Only ~20 functions are user code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic control flow preserved but C++ ABI artifacts create awkward patterns. Exception handling in `_Z18test_cpp_exceptionv` is reduced to direct `__cxa_throw` calls without try/catch structure.

**6. Function Decomposition:** Score: 5/10  
Justification: User test functions are appropriately sized, but drowned in 200+ STL/compiler helper functions. Lambda closures become separate functions (`_ZZ15test_cpp_lambdavENK3_24__1clEi`) losing closure semantics.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably well. Some forward references to globals (`&g37`) disrupt flow but don't prevent understanding.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundancy from C++ ABI (multiple destructor variants: `D0Ev`, `D1Ev`, `D2Ev`), but not excessive. Template instantiations create duplicate code for `Container<int>` vs `Container<double>`.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Complete type erasure - all C++ classes become `int32_t *`. `SimpleClass`, `Base`, `Derived` all indistinguishable. `std::unique_ptr` internals exposed as raw pointer arithmetic. No const-correctness preserved.

**10. Composite Type Recovery:** Score: 1/10  
Justification: Zero class/struct definitions recovered. `SimpleClass` with `int value` and `char name[32]` becomes pointer offsets: `*(char *)(result2 + 4)`, `*(char *)(result2 + 35)`. Field names completely lost.

**11. Type Safety:** Score: 3/10  
Justification: Pervasive unsafe casts: `(int32_t *)v1`, `(int32_t **)result2`. Pointer arithmetic on class objects (`result + 4`, `result + 8`) without type safety. Virtual calls through raw vtable pointers.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only demangled function names as comments (`// Demangled: test_cpp_member_func()`). No algorithm explanations, no parameter documentation, no uncertainty indicators for ambiguous decompilation.

**13. Idiom Recognition:** Score: 4/10  
Justification: C++ patterns visible but unexplained: vtable calls (`*a2` in `_Z17call_virtual_funcP4Basei`), inheritance thunks, RAII patterns in destructors. No pattern naming or explanation.

**14. Anomaly Indication:** Score: 2/10  
Justification: Zero anomaly flags. Uncertain decompilations (like lambda capture semantics, virtual dispatch) presented confidently. No warnings about potentially incorrect recovery.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `idx < size` checks in `getData` not flagged as contracts). Exception safety, null pointer checks not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but inheritance relationships obscured. `Derived` calling `Base` constructor not obvious. Virtual dispatch through `call_virtual_func` loses polymorphism semantics.

**17. Module Organization:** Score: 4/10  
Justification: Single monolithic file with 257 functions. No logical grouping - user code mixed with STL internals, runtime helpers, and thunks. No header file separation.

**18. Build System Compatibility:** Score: 3/10  
Justification: Would not compile standalone. Missing C++ standard library headers, undefined globals (`g37`, `g25`, etc.), architecture mismatch (x86 code for arm64 metadata). Requires extensive stubbing.

**Overall Readability Score:** 3.5/10

### Critical Defects:
- **Complete type erasure**: All C++ classes reduced to `int32_t *` with offset arithmetic, losing all semantic type information
- **No struct/class recovery**: `SimpleClass { int value; char name[32]; }` becomes pointer math like `*(char *)(result2 + 4)`
- **Generic variable naming**: Every variable is `v1`, `v2`, `v3`, `result` - zero semantic naming recovered
- **Massive noise pollution**: 257 functions where only ~20 are user code; STL internals overwhelm analysis
- **No anomaly indication**: Uncertain decompilations (lambda captures, virtual dispatch) presented without warnings

### Best Quality Aspect: **Code Linearization (6/10)**
- Functions generally read top-to-bottom without excessive jumps, making basic tracing possible despite other issues

### Most Needs Improvement: **Composite Type Recovery (1/10)**
- Complete failure to recover any class/struct definitions; all object-oriented structure lost to pointer arithmetic

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement C++ class structure recovery**: Analyze constructor/destructor patterns to infer class layouts and field names from usage patterns
2. **Add semantic variable naming**: Use data flow analysis to name variables based on usage (e.g., variables passed to `strlen` should be `string` or `name`)
3. **Filter STL/runtime noise**: Separate user code from compiler-generated code; collapse template instantiations into generic representations with type parameters noted
