# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v1, v2, v3, result) with zero semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `len` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original function names preserved (`cdecl_func`, `call_cdecl`, `varargs_func`), but many helpers are generic (`function_770`, `function_780`). Function signatures lost all parameter information.

**3. Literal Clarity:** Score: 4/10  
Justification: String literals stored as hex-encoded globals (g7-g24). Magic numbers like `0xffffffff`, `0xfffffffe`, `0x100000000` appear throughout without explanation or named constants.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive bit masking (`& 0xffffffff`), stack canary checks (`__stack_chk_fail()`), and redundant type casts throughout. Compiler artifacts obscure actual logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: While loops recovered but often awkward (e.g., `varargs_func` has complex loop with break). Many functions constant-folded to single returns losing original structure.

**6. Function Decomposition:** Score: 5/10  
Justification: Original function boundaries mostly preserved, but many functions lost parameters (`cdecl_func(void)` instead of `cdecl_func(int a, int b)`). Some are just constant returns.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with while loops. Some break statements used appropriately. Forward jumps minimal but present in varargs handling.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy from repeated bit masking operations. Stack canary checks duplicated across functions. No dead code elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything collapsed to `int64_t` or `int32_t`. Original types (int, char*, double, struct) completely lost. Function parameters all `void` or `int64_t` regardless of original signature.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct definitions (`LargeStruct`, `SmallStruct`, `SmallPoint`) completely lost. All struct access is raw pointer arithmetic (`*(int32_t *)(v2 + 4)`).

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout. Pointer arithmetic without type safety. Varargs handling uses raw stack manipulation without type information.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments. Only auto-generated header ("This file was generated by the Retargetable Decompiler"). Zero documentation of logic or intent.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Varargs handling not identified as such. Calling convention differences not documented. No algorithm identification.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomaly flags present. Code presented confidently even when clearly incorrect (e.g., `func_a` and `func_b` have wrong logic). No uncertainty indicators.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., null checks not explained). No postconditions. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but relationships not documented. Global variables (g1-g25) used without explanation of purpose. No call graph documentation.

**17. Module Organization:** Score: 5/10  
Justification: Functions listed but not logically grouped by category. No header file structure. Original test organization (calling conventions, parameter passing, return values) not preserved.

**18. Build System Compatibility:** Score: 5/10  
Justification: Has includes but external dependencies unclear. Some functions reference undefined globals. May not compile standalone without significant modification.

### Critical Defects:
- **Complete type erasure**: All variables are `int64_t`/`int32_t`; original types (struct, char*, double) lost
- **Zero variable semantics**: Every variable named v1, v2, v3 with no indication of purpose
- **Struct definitions lost**: `LargeStruct`, `SmallStruct` accessed via raw pointer arithmetic

- **No documentation**: Zero comments explaining logic, anomalies, or contracts

### Best Quality Aspect:

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions; all composite types reduced to pointer arithmetic

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover type information from usage patterns** - Analyze how variables are used (arithmetic vs. pointer dereference vs. struct access) to infer original types
2. **Preserve function signatures from debug symbols** - When `-g` flag is present, extract parameter names and types from DWARF information
3. **Add confidence annotations** - Mark uncertain recoveries (e.g., "// DECOMPILE UNCERTAINTY: varargs handling may be incorrect") to guide manual review
