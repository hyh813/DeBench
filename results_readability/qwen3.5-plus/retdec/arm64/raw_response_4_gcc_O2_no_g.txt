# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v1`, `v2`, `v3`, `result`) with zero semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `large`, `s` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Test functions retained names (`call_cdecl`, `test_calling_conventions`) but helper functions lost names (`function_730`, `function_740`, `function_1004`). Critical: all function signatures show `void` parameters when they should have parameters.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere (`0xffffffff`, `0xfffffffe`, `0x12345678`) without explanation. String literals stored as generic globals (`g7`, `g8`, `g9`) with no semantic meaning.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive type masking (`& 0xffffffff`, `& 0xfffffffe`) throughout. Stack canary checks (`__stack_chk_fail()`) and unnecessary casts pollute the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Heavy use of goto labels (`lab_0xad0`, `lab_0xaf0`, `lab_0xadc`) instead of natural loop constructs. Varargs functions use complex goto chains where simple for-loops existed in source.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but parameter lists are wrong (e.g., `cdecl_func(void)` instead of `cdecl_func(int a, int b)`). Many functions constant-folded losing actual logic.

**7. Code Linearization:** Score: 3/10  
Justification: Frequent forward jumps and goto labels disrupt top-to-bottom reading. Varargs functions have particularly convoluted jump patterns that obscure the simple loop logic.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated masking operations across functions. Similar patterns in `func_a`, `func_b`, `cdecl_func` all show `& 0xffffffff` without consolidation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `int64_t`/`int32_t`. Lost distinction between `int`, `char*`, `double`, `struct`. Function signatures show `void` parameters when they should have typed parameters.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct definitions completely lost (`LargeStruct`, `SmallStruct`, `TestIntPair`). All struct access through pointer arithmetic (`*(int32_t *)(v2 + 4)`) with no field names.

**11. Type Safety:** Score: 3/10  
Justification: Many unsafe casts visible. Pointer arithmetic without type safety. Function pointers degraded to `int64_t` (`ret_func_ptr` returns addresses as integers).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled code. Original had extensive comments explaining each test case (CALL-L1-01, PARAM-L2-01, etc.) - all lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition visible. Varargs handling not identified as such. Calling convention differences not noted despite being the core test purpose.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about uncertain decompilation. Parameter loss not flagged. Type degradation not noted. Code presents incorrect information confidently.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `p != NULL` checks). No postconditions inferred. No invariants recognized from original assertions.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. Global variables named generically (`g1`, `g2`, `g7`) with no indication of purpose (strings, constants, etc.).

**17. Module Organization:** Score: 5/10  
Justification: All functions in one file with prototypes at top. Original had clear section organization (4.1 Calling Conventions, 4.2 Parameter Passing, 4.3 Return Values) - all lost.

**18. Build System Compatibility:** Score: 4/10  
Justification: Includes standard headers and has prototypes, but many functions have wrong signatures (missing parameters). Would not compile correctly due to parameter mismatches.

### Critical Defects:
- **Function signatures lost all parameter information** - `cdecl_func(void)` should be `cdecl_func(int a, int b)`
- **All variable names are generic** (`v1`, `v2`, `v3`) with zero semantic meaning
- **Struct definitions completely lost** - `LargeStruct`, `SmallStruct` accessed via raw pointer arithmetic
- **Zero comments** - original had extensive documentation explaining each test case
- **Excessive goto usage** - simple for-loops converted to complex label-based control flow

### Best Quality Aspect:
- **Module Organization (5/10)** - Functions are grouped with prototypes at top, providing basic structural coherence despite other issues

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation makes understanding the decompiled code's purpose nearly impossible

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover function parameters from calling convention analysis** - Analyze register/stack usage at function entry to reconstruct parameter types and counts
2. **Infer variable names from usage patterns** - Variables used in loop counters should be named `i`, `count`; accumulators should be `sum`, `total`
3. **Recover struct definitions from memory access patterns** - Analyze consistent offset patterns (`+0`, `+4`, `+8`) to reconstruct struct layouts with field names
