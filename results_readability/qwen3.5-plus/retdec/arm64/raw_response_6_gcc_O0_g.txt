# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Variables like `v1`, `v2`, `v3` dominate the code obscuring their semantic roles (e.g., `int64_t v1` in `param_fread_fwrite` should be `items_written`). Some parameters retain good names (`dst`, `src`, `filename`), but locals are generic.

**2. Function Naming Quality:** Score: 7/10  
Justification: Main functions preserved well (`param_strcpy`, `call_strcpy`, `thread_compute`), but helper wrappers use mechanical names (`function_13f0`, `function_1400`) losing semantic meaning for library call wrappers.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals stored in opaque globals (`g9`, `g10`...) without context. Magic numbers like `0x140000000a`, `0x6ffe400ebffffdb` appear unexplained. Constants like `42`, `100` lack semantic naming.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`*(int64_t *)g2 != *(int64_t *)g2`) repeated in nearly every function. Architecture-specific intrinsics (`__aarch64_cas4_acq_rel`) present but expected for atomics.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Extensive goto usage (`lab_0x1f88`, `lab_0x2e64`) replaces natural loop constructs. Awkward conditionals like `signal_received == 0 == (int32_t)&g30 > 0` obscure logic intent.

**6. Function Decomposition:** Score: 7/10  
Justification: Main functions appropriately sized and cohesive. Library wrappers separated (`function_13f0` for memcpy). Thread functions properly extracted. Some monolithic functions could be split.

**7. Code Linearization:** Score: 5/10  
Justification: Forward jumps and labels disrupt top-to-bottom reading. Multiple exit points via goto in `param_fread_fwrite` and `param_thread_local_storage` create non-linear flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Stack canary checks duplicated across 40+ functions. Some redundant type casts (`(int64_t *)dst` where `dst` is already typed). No dead code removal observed.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Uses `int32_t`, `int64_t` appropriately but degrades some types (e.g., `char *` for thread function return instead of `void *`). Pointer arithmetic on `int64_t *` loses original intent.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs defined (`struct _IO_FILE`, `struct ipc_perm`) but fields named generically (`e0`, `e1`, `e2`...). Original field semantics completely lost (e.g., `sin_family`, `sin_port` become `e0`, `e1`).

**11. Type Safety:** Score: 5/10  
Justification: Numerous unsafe casts: `(char *)0xffffffff`, `(int64_t *)mem`. Pointer arithmetic on void pointers. Thread function signatures use `char *` instead of `void *` causing type confusion.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated header comments present. Zero algorithm explanations, no API contracts, no decompilation confidence indicators. Original source comments completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (`strcpy`, `strlen`, `pthread_create`). pthread patterns partially identified. No design pattern naming or algorithm identification present.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some `UNREACHABLE` markers present after `_exit()`. No warnings for suspicious indirect calls or unresolved symbols. Stack canary checks present but unexplained.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `dst != NULL` for strcpy). Error return values not explained. No postcondition or invariant documentation anywhere.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships undocumented. Global variable usage (`g1`, `g2`...) not explained. No call graph or callback relationship documentation.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped by section (standard library, system calls, threads) matching source structure. All code in single file with no header separation. Reasonable but monolithic.

**18. Build System Compatibility:** Score: 6/10  
Justification: Standard includes present. Some architecture-specific functions need stubs (`__aarch64_cas4_acq_rel`). Would compile with minor modifications and platform-specific adjustments.
