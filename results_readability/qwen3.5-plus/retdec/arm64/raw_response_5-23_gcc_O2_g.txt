# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `v1`, `v2`, `v3` with no semantic meaning. For example, `param_macro_constants` uses `v1`, `v2`, `v3` instead of `size`, `result`, or `threshold`.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core functions retained names like `param_macro_constants`, `call_asm_basic`, but many helper functions are generic (`function_830`, `function_840`). Original function names preserved where possible.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere (`0xdef0`, `0xffffffff`, `0x200000001`) without explanation. String literals stored as hex-encoded globals (`g8`, `g9`) instead of readable strings.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains compiler artifacts like stack canary checks (`__stack_chk_fail()`), TM clone tables, and unnecessary type casts. Some cleanup performed but significant noise remains.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Simple conditionals transformed into complex expressions like `v3 == 0 | v3 < 0 != ((v3 ^ v2) & (v2 ^ (int32_t)&g27)) < 0` instead of `if (size > MAX_SIZE)`.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but many functions lost parameters (e.g., `param_macro_constants(void)` should be `param_macro_constants(int size)`). Some functions constant-folded to single return values.

**7. Code Linearization:** Score: 4/10  
Justification: While loops recovered in some cases (`param_asm_clobber`), many functions use complex expression chains that disrupt top-to-bottom reading flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Constant folding resulted in functions like `call_macro_constants` returning literal `64` instead of showing computation. Some redundancy from optimization artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Excessive use of `int64_t` where `int32_t` suffices. Parameter types lost (functions show `void` instead of `int size`). Frequent `& 0xffffffff` masking indicates type confusion.

**10. Composite Type Recovery:** Score: 5/10  
Justification: No structs in tested functions, but global variables all generic (`g1`, `g2`, `g3`). Pointer arithmetic visible instead of array access patterns.

**11. Type Safety:** Score: 5/10  
Justification: Multiple implicit casts and `& 0xffffffff` masking throughout. Stack canary checks present but type-unsafe patterns like `(int64_t *)v1` without validation.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10

**13. Idiom Recognition:** Score: 2/10

**14. Anomaly Indication:** Score: 2/10

**15. Assertion & Contract Recovery:** Score: 1/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10

**17. Module Organization:** Score: 4/10
