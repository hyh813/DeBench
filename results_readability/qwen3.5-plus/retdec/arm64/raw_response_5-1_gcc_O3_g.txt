# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables are generic (`v1`, `v2`, `v3`, `str`, `result`) with no semantic meaning. Original source had meaningful names like `capture_by_value`, `instance_count`, `multiplier` that are completely lost.

**2. Function Naming Quality:** Score: 3/10  
Justification: Functions use mangled C++ names (`_Z18test_cpp_exceptionv`) instead of readable names like `test_cpp_exception`. Demangled comments exist but the actual function names remain unreadable.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers (4, 42, 8, 16, 24, 32, 48) appear without explanation. String literals stored in globals (`g16`, `g17`) with no semantic labels. Original had clear expected values in comments.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present including `__stack_chk_fail()` calls, unnecessary type casts, and generic global variables (`g1` through `g30`). Debug mode (-O0) should have eliminated most of these.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Extensive goto usage (`lab_0x10e0`, `lab_0x10e4`) instead of structured control flow. Exception handling logic is particularly broken with confusing jump patterns that don't match try/catch semantics.

**6. Function Decomposition:** Score: 2/10  
Justification: Critical functions are empty stubs (`test_cpp_constructor`, `test_cpp_virtual_func`, `test_cpp_lambda` all just `return;`). Logic completely lost despite -O0 compilation that should preserve it.

**7. Code Linearization:** Score: 4/10  
Justification: Excessive forward jumps disrupt reading flow. Exception handling in `_Z18test_cpp_exceptionv` has multiple label jumps that make execution path nearly impossible to follow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Similar vtable thunk functions repeated without abstraction. Exception handling code shows duplicated patterns that could be consolidated.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `int64_t`, `int32_t`, `void*`. Lost all C++ class type information, const-correctness, and specific types like `size_t`. Original had precise type declarations throughout.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Only basic vtable structures recovered (`vtable_12c28_type`). All class definitions (`SimpleClass`, `LifecycleClass`, `Base`, `Derived`) completely lost with no field names or structure.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(int64_t *)result`, `(int32_t *)v1`). Pointer arithmetic without type safety. No const correctness preserved from original source.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**14. Anomaly Indication:** Score: 2/10

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions/postconditions documented. Exception contracts not recovered. Class invariants completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===
