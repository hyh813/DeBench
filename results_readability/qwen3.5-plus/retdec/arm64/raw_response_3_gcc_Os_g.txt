# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (v1, v2, v3, temp, result) with zero semantic meaning. Original source had meaningful names like `large_buf`, `arr`, `p`, `head` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `heap_basic`, `vla_stack`) but many functions have generic names (`function_9d0`, `function_9e0`). Several function signatures are broken (e.g., `local_vars(void)` should be `local_vars(int x)`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere (0xfffffffe, 0x10000, 0xfc00) without explanation. String literals stored as hex-encoded globals (g9, g10) instead of readable strings. No constant recovery.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive compiler artifacts: stack canary checks (`__stack_chk_fail`), generic globals (g1-g19), assembly intrinsics (`__asm_sbfiz`, `__asm_ubfx`), and redundant type casts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: All loops converted to while+goto patterns instead of natural for/while constructs. Labels like `lab_0x129c` and `lab_0xea8` disrupt readability. Original for-loops completely unrecognizable.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved but signatures corrupted. Helper wrappers created for library calls (`function_9d0` for memcpy). Some functions missing parameters entirely.

**7. Code Linearization:** Score: 4/10  
Justification: Heavy goto usage with forward jumps into loop bodies. Stack check failures create non-linear exit paths. Cannot read top-to-bottom without tracking label targets.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated type cast patterns (`(int64_t)(uint32_t)`), duplicated stack canary checks in every function, and redundant bit-masking operations (e.g., `& 0xfffffffe`).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: int32_t/int64_t used but often incorrectly. Pointer types degraded to int64_t. Excessive unnecessary casts like `(int64_t)(uint32_t)local_array(2)`. Original type information partially lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - Point struct not recovered (accessed as `*(int32_t *)mem` and `*(int64_t *)(v1 + 8)`). Node/TreeNode structs completely lost. All struct access via raw pointer arithmetic.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts throughout. Pointer arithmetic without type safety (`(int32_t *)(4 * v3 + v2)`). Function pointer type partially preserved (`global_func_ptr`) but usage broken.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero meaningful comments recovered. Only auto-generated metadata (address ranges, line ranges). No algorithm explanations, no contract documentation, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns (malloc+free, linked list traversal, tree creation) not recognized or named. No design pattern identification. Memory operation idioms obscured by pointer arithmetic.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about decompilation uncertainty. Stack canary checks present but unexplained. No indication of potential undefined behavior (dangling pointer, double free cases).

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks present but not documented as preconditions. No postconditions inferred. Error conditions not documented (e.g., return -1 meanings unexplained).

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls preserved but relationships not documented. Global variables have cryptic names (g1-g19). No call graph or usage information. Callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file. Some grouping by test category visible in function names. Static functions not properly scoped. Original file structure lost.

**18. Build System Compatibility:** Score: 4/10  
Justification: Includes present but some incorrect. Function prototypes declared but many signatures wrong. Would require significant patching to compile (wrong parameter counts, missing types).

**Overall Readability Score:** 3.9/10

### Critical Defects:
- **Struct types completely lost** - Point, Node, TreeNode accessed via raw pointer arithmetic (`*(int64_t *)(v1 + 8)`)
- **Function signatures corrupted** - `local_vars(void)` should be `local_vars(int32_t x)`, `local_struct(void)` missing parameter
- **All variables generic** - v1, v2, v3 throughout with zero semantic meaning
- **Control flow uses goto** - All loops converted to while+goto with labels like `lab_0x129c`
- **Zero meaningful comments** - No documentation of algorithm intent, contracts, or decompilation uncertainty

### Best Quality Aspect:
- **Function Naming Quality (5/10)** - Some original function names preserved (`double_value`, `heap_basic`, `vla_stack`, `linked_list_heap`) allowing partial semantic recovery

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Zero meaningful comments recovered; only auto-generated metadata with no algorithm explanations or confidence indicators

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct/union types from access patterns** - Detect consistent offset patterns (e.g., +8, +16) and reconstruct composite types with field names
2. **Infer variable names from usage context** - Variables used in size checks should be named `size`/`count`, loop indices should be `i`/`j`, pointers should indicate target type
3. **Recover high-level control structures** - Convert goto-based loops back to for/while constructs and add comments explaining optimization artifacts at -O0
