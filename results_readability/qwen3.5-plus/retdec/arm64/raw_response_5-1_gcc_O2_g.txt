# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (v1, v2, v3, str, result) with no semantic meaning. For example, in `_Z18test_cpp_exceptionv()`, variables v1-v11 obscure the exception handling logic that should show catch blocks and rethrow operations.

**2. Function Naming Quality:** Score: 3/10  
Justification: Functions retain mangled C++ names (_Z20test_cpp_member_funcv, _ZN4BaseD2Ev) instead of readable names. While demangled comments exist ("Demangled: test_cpp_member_func()"), the actual function names remain unreadable without external tools.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals are present but UTF-8 encoded (g17 = "=== \xe6\xb5\x8b\xe8\xaf\x95..."). Magic numbers like 0x12c28 for vtables lack explanation. Some constants (42, 100, 200) are recoverable from context.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains compiler artifacts including stack canary checks (__stack_chk_fail), initialization functions (_GLOBAL__sub_I__), and runtime support (deregister_tm_clones, register_tm_clones) that clutter the output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Uses explicit goto labels (lab_0x10e0, lab_0x10e4) instead of structured control flow. Exception handling in `_Z18test_cpp_exceptionv()` uses jumps instead of try/catch blocks, making the logic nearly impossible to follow.

**6. Function Decomposition:** Score: 4/10  
Justification: Critical functions are stubbed with just "return;" (_Z21test_cpp_virtual_funcv, _Z26test_cpp_operator_overloadv, _Z15test_cpp_lambdav). No proper decomposition of C++ class methods into coherent units.

**7. Code Linearization:** Score: 4/10  
Justification: Goto statements disrupt natural top-to-bottom reading. Exception handling code jumps between labels (0x10e0, 0x10e4) creating non-linear execution paths that obscure the try/catch/rethrow semantics.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundancy from compiler initialization code, but the primary issue is missing implementation rather than duplication. Stack canary checks appear repeatedly without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Loses all C++ class types - everything becomes int64_t, int32_t, or void*. The SimpleClass, LifecycleClass, Base, Derived classes are completely erased. Template instantiations lose type parameters.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Only vtable structures recovered (vtable_12c28_type, vtable_12c50_type) with generic field names (e0, e1, e2). No class structure recovery - data members like `value`, `name[32]`, `multiplier` are completely lost.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout (int64_t to int32_t, pointer arithmetic on void*). Example: `*(int32_t *)v1 = 42` and `(int64_t)(2 * *(int32_t *)v9 + *(int32_t *)v7)` show type confusion.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated metadata comments (address ranges, line numbers, demangled names). No semantic explanations of what the code does or why. No algorithm documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: C++ idioms completely unrecognized - virtual dispatch, RAII, exception handling, templates, lambdas all appear as raw pointer operations. No pattern identification or naming.

**14. Anomaly Indication:** Score: 3/10  
Justification: No indication of decompilation uncertainty. Functions that are stubbed (return;) give no warning that implementation was lost. No confidence indicators for recovered code.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Exception specifications, null checks, and bounds validation are not identified as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are present but virtual dispatch relationships are opaque. Vtable structures exist but the connection between Base::virtual_func and Derived::virtual_func is not documented.

**17. Module Organization:** Score: 4/10  
Justification: All code in single file with no logical grouping. Related class methods scattered throughout. No namespace or package structure recovery from C++ code.

**18. Build System Compatibility:** Score: 5/10  
Justification: Has includes and function declarations but would require significant work to compile. Missing C++ standard library headers, runtime support functions need stubs.

### Critical Defects:
- **All variables use generic names** (v1, v2, v3) making data flow impossible to trace without constant reference to original source
- **C++ class structures completely lost** - no recovery of SimpleClass, LifecycleClass, Base, Derived, or any template classes
- **Critical functions stubbed** - test_cpp_virtual_func, test_cpp_operator_overload, test_cpp_lambda return immediately with no implementation
- **Exception handling uses goto** instead of try/catch, obscuring the nested exception logic in test_cpp_exception()
- **Mangled function names** remain in output (_Z20test_cpp_member_funcv) requiring external demangling tools

### Best Quality Aspect:
- **Cross-Reference Clarity (5/10)** - Function call relationships are at least preserved, and vtable structures show some virtual dispatch recovery, providing minimal context for understanding call chains.

### Most Needs Improvement:
- **Idiom Recognition (2/10)** - C++ patterns (virtual functions, inheritance, templates, lambdas, exceptions, RAII) are completely unrecognized, appearing as raw pointer arithmetic and function calls with no semantic meaning.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover C++ class structures** from vtable layouts and object memory patterns to restore class definitions with field names (value, name, multiplier, data, size)
2. **Recognize exception handling patterns** from __cxa_* function calls and reconstruct try/catch/throw blocks instead of goto-based control flow
3. **Apply semantic variable naming** by analyzing usage patterns (e.g., variables used in size comparisons should be named size/length, variables passed to strlen should be named string/buffer)
