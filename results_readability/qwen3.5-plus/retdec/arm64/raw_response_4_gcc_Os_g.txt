# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v1`, `v2`, `v3`, `result`) with zero semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `len` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Most original function names preserved (`cdecl_func`, `call_cdecl`), but many helper functions named generically (`function_770`, `function_780`). Function signatures often missing parameters entirely.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive (`0xffffffff`, `0xfffffffe`, `0x100000000`) without explanation. String literals stored in opaque globals (`g8`, `g9`) with hex-encoded content.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive compiler artifacts including redundant bitwise AND operations (`& 0xffffffff`), stack check failures (`__stack_chk_fail()`) throughout, and incorrect parameter recovery creating noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Varargs functions use `while(true)` with breaks instead of natural `for` loops. Simple functions like `cdecl_func` show broken logic (`2 * v1 & 0xffffffff` instead of `a + b`).

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved but signatures corrupted. Many functions show `void` parameters when original had 2-8 parameters (e.g., `amd64_sysv_func(void)` should have 6 parameters).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom within functions. Some complex functions have awkward loop structures but no severe goto spaghetti.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Repetitive stack check code in nearly every function. Redundant type casts and bitwise masking operations throughout without consolidation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Inconsistent type usage (`int64_t` vs `int32_t`). Many functions return `int64_t` when source returns `int`. Parameter types completely lost in many functions.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `LargeStruct` and `SmallStruct` definitions completely lost. All struct operations converted to raw pointer arithmetic (`*(int32_t *)(v2 + 4)`).

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout. Function pointer types degraded (`int32_t (*callback)(int32_t)` preserved in some places but lost in others). Pointer arithmetic without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated address ranges and module info. Zero explanatory comments for logic. Original source had extensive documentation completely absent.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern not recognized (complex while loop instead of `va_start/va_arg/va_end`). Loop patterns, struct access patterns all unrecognizable.

**14. Anomaly Indication:** Score: 1/10  
Justification: Incorrect logic presented confidently without warnings. `cdecl_func` shows wrong calculation with no indication of uncertainty. No anomaly flags for parameter loss.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `p != NULL` checks not explained). No postconditions or error handling documentation recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but global variables opaque (`g1`, `g2`, `g8`). No call graph documentation. Callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file. Functions grouped somewhat logically but no header separation. Original test organization lost.

**18. Build System Compatibility:** Score: 7/10  
Justification: Code compiles successfully per metadata. Necessary includes present (`stdio.h`, `stdlib.h`, `string.h`). External function declarations provided.

**Overall Readability Score:** 3.6/10

### Critical Defects:
- **Function parameters completely lost**: `cdecl_func(void)` should be `cdecl_func(int a, int b)` - makes logic incomprehensible
- **All variables generic**: `v1`, `v2`, `v3` provide zero semantic information about purpose
- **Struct types destroyed**: `LargeStruct` and `SmallStruct` converted to raw pointer arithmetic
- **Incorrect logic presented confidently**: `cdecl_func` shows `2 * v1 & 0xffffffff` instead of `a + b` with no warning
- **Varargs handling broken**: Complex while-loop instead of recognizable `va_start/va_arg/va_end` pattern

### Best Quality Aspect:
- **Build System Compatibility (7/10)** - Code compiles successfully with proper includes and external declarations, making it at least runnable despite semantic issues.

### Most Needs Improvement:
- **Anomaly Indication (1/10)** - Decompiler presents incorrect logic with full confidence, providing no warnings about parameter loss or uncertain recovery, which is dangerously misleading.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover function signatures from calling conventions**: Analyze register/stack usage at function entry to reconstruct parameter count and types, especially for ARM64 AAPCS.
2. **Implement variable naming from usage patterns**: Infer names like `count`, `sum`, `index` from loop patterns and arithmetic operations rather than generic `v1`, `v2`.
3. **Add uncertainty annotations**: When parameter recovery fails or logic is ambiguous, add comments like `// UNCERTAIN: parameters not recovered` instead of presenting broken code confidently.
