# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `uVar2`, `piVar1`, `local_34`) instead of semantic names from source (`a`, `b`, `c`, `arr`, `p`, `local`). For example, `local_array`'s loop variable `iVar2` should be `i`, and `piVar1` should be `arr_ptr`.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names are well-preserved from debug symbols (`double_value`, `local_vars`, `heap_basic`, `linked_list_heap`). This is the strongest aspect due to `-g` flag preserving symbol table.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound without explanation: `0x2a` (42), `0x14` (20), `0xffffffff` (-1), `0x32` (50). Some are inferable but none are documented as named constants.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__stack_chk_fail`) appear in many functions unnecessarily. Multiple "WARNING: Removing unreachable block" annotations clutter output. Runtime functions (`__libc_start_main`, `__gmon_start__`) included.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: For-loops converted to do-while patterns (e.g., `local_array`, `heap_basic`). Some functions incorrectly return `void` instead of `int` (`local_array`, `vla_stack`, `alloca_usage`), breaking semantic flow.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved but several have wrong signatures (void vs int return). `create_tree_node` returns void but should return `TreeNode*`. No inappropriate inlining observed.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but do-while loops with pointer arithmetic disrupt natural reading. No excessive goto usage, but stack canary epilogues interrupt flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Stack canary prologue/epilogue code repeated across functions. Pointer arithmetic patterns (`(int)__ptr + -4`) could be simplified to array indexing.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of `undefined4` instead of `int`. `size_t` inconsistently represented as `uint` or `size_t`. Function pointers degraded to `code *`. Signed/unsigned distinctions often lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Critical failure - `Point` struct not recovered (fields accessed via `param_1 * 3` arithmetic). `Node` struct in linked list completely lost (fields accessed as `__ptr[1]`). No struct definitions in output.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts: `(int *)((int)__ptr + -4)`, `(undefined4 *)0x0`. Pointer arithmetic instead of array indexing creates potential UB. Function pointer calls use generic `code *` type.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated Ghidra headers present. No algorithm explanations, no decompilation confidence indicators, no API documentation. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns (malloc/free pairs, linked list traversal, tree creation) not identified or named. Memory operation functions (`memset`, `memcpy`) recognized but not annotated as such.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some warnings present ("WARNING: Subroutine does not return", "Removing unreachable block") but critical issues like wrong return types not flagged. Stack canary insertion not explained.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks exist in code but not documented as preconditions. No postcondition inference (e.g., "ensures: returned memory is heap-allocated"). Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`global_counter`, `file_scope_static`) referenced without declaration context. Callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All 112 functions in single output without logical grouping. Static functions not clearly distinguished from external. No header file structure recovered.

**18. Build System Compatibility:** Score: 4/10  
Justification: Will not compile standalone - missing struct definitions, undefined globals (`global_counter`, `extern_global_var`), external function dependencies undocumented. Requires significant patching.

**Overall Readability Score:** 4.6/10

### Critical Defects:
- **Struct types completely lost**: `Point` and `Node` structs not recovered; all field access via pointer arithmetic (`__ptr[1]`, `param_1 * 3`)
- **Incorrect return types**: 10+ functions show `void` return instead of `int` (`local_array`, `vla_stack`, `alloca_usage`, `create_tree_node`, etc.)
- **Generic variable naming**: All variables use `param_N`, `iVarN`, `uVarN` patterns losing all semantic meaning
- **Magic numbers undocumented**: `0x2a`, `0x14`, `0xffffffff` used without constant definitions or comments
- **Stack canary pollution**: `__stack_chk_fail` calls inserted without explanation, cluttering function epilogues

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Debug symbols (`-g` flag) preserved function names accurately, making function identification straightforward.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct definitions; all composite types reduced to pointer arithmetic, severely impacting code understanding.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct/union types from memory access patterns** - Analyze consistent offset patterns (e.g., `__ptr[0]`, `__ptr[1]` with 4/8 byte spacing) to reconstruct struct definitions with field names.
2. **Infer variable names from usage context** - Use data flow analysis to name variables based on their role (loop counters → `i`, array pointers → `arr_ptr`, sizes → `size`).
3. **Document type mismatches and confidence levels** - Add comments when return types are uncertain or when stack canary code is inserted, helping analysts understand decompilation limitations.
