# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `uVar2`, `piVar3` with zero semantic meaning. Original source had meaningful names like `sum`, `len`, `current`, `head` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most application function names preserved (`process_char`, `array_1d_stack`, `struct_simple`), but helper functions use generic names (`FUN_00011718`) and some runtime functions have opaque names (`call_weak_fn`).

**3. Literal Clarity:** Score: 4/10  
Justification: Character literals converted to hex (`0x41` instead of `'A'`, `0x20` instead of `0x20` for space). Float constants shown as IEEE 754 hex (`0x3fc00000` instead of `1.5f`). Magic numbers throughout with no explanation.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains compiler runtime helpers (`__muldf3`, `__aeabi_dadd`, `__printf_chk`), stack canary checks (`__stack_chk_fail()`), and multiple "WARNING: Removing unreachable block" comments that clutter the output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: For-loops converted to do-while with pointer arithmetic (`piVar3 = piVar3 + 1; iVar1 = iVar1 + *piVar3`). At O2 optimization this is expected, but original array indexing (`arr[i]`) is lost to pointer math.

**6. Function Decomposition:** Score: 6/10  
Justification: Test functions became monolithic with inlined code (e.g., `test_array_types` is 200+ lines with all array initialization inlined). Helper functions appropriately separated but application logic overly consolidated.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom with early returns. Stack canary checks at function ends create minor disruption. Some functions have forward jumps for error paths but remain traceable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Loop unrolling visible in `array_3d` (unrolled inner loop with `piVar2[1] + piVar2[2] + piVar2[3] + piVar2[4]`). Some redundant pointer arithmetic patterns from optimization but not severe.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `long long` split into two `uint/int` parameters, `float/double` become `undefined4/undefined8`, function pointers become `code*`. Original type information largely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - all structs shown as `int*` with array indexing (`param_1[1]`, `param_1[2]`). No field names recovered. `Point3D`, `Rect`, `Widget` all reduced to raw pointer arithmetic. Unions and bitfields expanded to bitwise operations.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts (`(int *)*piVar2`, `(undefined4 *)`), generic pointer types everywhere, function pointers as `code*` without signatures. No const-correctness preserved despite original `const int *` usage.
