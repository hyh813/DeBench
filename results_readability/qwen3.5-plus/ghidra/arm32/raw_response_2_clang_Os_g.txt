# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Local variables use generic names like `iVar1`, `uVar2`, `piVar1` obscuring their purpose. Function parameters are better preserved (`c`, `x`, `p`), but many locals marked as "Unresolved local var" with no names at all.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source (`process_char`, `array_1d_stack`, `struct_simple`). Helper functions like `FUN_00011594` are generic, but core logic functions retain semantic names.

**3. Literal Clarity:** Score: 4/10  
Justification: Test functions show computed constants instead of expressions (`0x61` instead of `'a'`, `0xb` instead of `11`). Float operations use magic hex values (`0x3fc00000`, `0x3f000000`) without explanation.

**4. Noise Reduction:** Score: 6/10  
Justification: "Unresolved local var" comments scattered throughout. Compiler helper functions (`__mulsf3`, `__adddf3`) visible. WARNING comments for unreachable blocks add noise but are informative.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: For loops converted to do-while patterns (`do { n = n + -1; ... } while (n != 0)`). At -O0 this is unexpected degradation. Simple functions like `process_short` remain natural.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly. Sizes are reasonable (no monolithic functions). Helper functions for float operations appropriately separated.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom. Test functions have constant folding disrupting flow. Some forward jumps in complex array functions (`array_3d`, `array_complex_index`).

**8. Redundancy Elimination:** Score: 7/10  
Justification: Test functions show aggressive constant folding (good). No obvious dead code. Some redundant patterns in soft-fp helper functions but those are library code.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `char`, `short`, `long`). **Critical:** `volatile` qualifier lost in `volatile_access` (shows `int *p` not `volatile int *p`). Function pointer types partially recovered (`_func_int_int`).

**10. Composite Type Recovery:** Score: 7/10  
Justification: Struct types recovered with field names (`Point3D`, `Rect`, `Widget`). Union types preserved (`UnionData`). **Issue:** Bitfield struct `Flags` decompiled as raw ushort access instead of bitfield operations.

**11. Type Safety:** Score: 7/10  
Justification: Some unsafe casts visible (`(int*)p`, `(void *)size`). Pointer arithmetic present but generally correct. No obvious buffer overflow patterns introduced by decompiler.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: Only decompiler-generated comments present ("Unresolved local var", "WARNING: Could not recover jumptable"). No algorithm explanations or API documentation recovered from source comments.

**13. Idiom Recognition:** Score: 5/10  
Justification: Basic loop patterns recognized but converted to do-while. No design pattern identification. Standard C idioms (array indexing, pointer dereference) preserved but not annotated.

**14. Anomaly Indication:** Score: 6/10  
Justification: Function pointer calls flagged ("WARNING: Could not recover jumptable at 0x00010d08. Too many branches"). Unresolved variables noted. Some anomalies silently handled.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition/postcondition comments recovered. Null checks present in code but not documented as contracts. Source assertions completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible in code. No call graph documentation. Global variable references present (`DAT_00012453`) but not named meaningfully.

**17. Module Organization:** Score: 4/10  
Justification: All 81 functions in single output file. Original source organization (sections for data types, arrays, pointers, composites) not preserved. No header file generation.

**18. Build System Compatibility:** Score: 7/10  
Justification: Metadata indicates "Compilation Result: Yes". External dependencies visible (`__libc_start_main`, `printf`, `puts`). Soft-fp helpers need linking but are standard.

**Overall Readability Score: 5.9/10**

### Critical Defects
- **`volatile` qualifier completely lost** in `volatile_access` function - semantic meaning destroyed
- **Test functions show constant-folded values** instead of actual function calls (`printf("...%d\n",0xf)` instead of calling `array_1d_stack`)
- **Bitfield struct `Flags` decompiled as raw ushort** access losing field semantics (`f->flag1` becomes `(uVar1 & 1)`)
- **All source comments lost** - original documentation about test scenarios and expected values gone
- **Float operations expanded to soft-fp library calls** (`__mulsf3`, `__adddf3`) obscuring simple `f * 1.5f + 0.5f`

### Best Quality Aspect
- **Function Naming Quality (8/10)** - Ghidra successfully preserved original function names from debug symbols, making navigation and identification straightforward.

### Most Needs Improvement
- **Assertion & Contract Recovery (3/10)** - Zero contract documentation recovered; preconditions, postconditions, and expected behaviors from source comments completely absent.

### Top 3 Recommendations for Decompiler Improvement
1. **Preserve type qualifiers** - Detect and retain `volatile`, `const`, `restrict` from debug info or usage patterns to maintain semantic correctness
2. **Recover source comments** - Extract and embed DWARF debug line comments and function documentation into decompiled output
3. **Avoid aggressive constant folding in test code** - Preserve function call structure even when arguments are constants to maintain traceability to source
