# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Most variables use generic names (`iVar1`, `uVar2`, `local_c`, `piVar6`) obscuring semantic purpose. Original parameter names preserved in some functions (`cdecl_func(int a, int b)`) but locals marked as "Unresolved local var: int sum@[???]" lose all meaning.

**2. Function Naming Quality:** Score: 8/10  
Justification: Original function names excellently preserved (`cdecl_func`, `varargs_func`, `test_calling_conventions`). Action-oriented names clearly reflect purpose, matching source intent perfectly.

**3. Literal Clarity:** Score: 5/10  
Justification: Constants folded to hex (`0xf`, `0x32`, `0x2a`, `0x12345678`) instead of decimal values. String literals externalized as `&DAT_0001148c` references rather than inline text, requiring cross-reference lookup.

**4. Noise Reduction:** Score: 4/10  
Justification: Multiple warning artifacts ("WARNING: Unknown calling convention", "WARNING: Removing unreachable block"). Runtime library expansion (`__aeabi_dadd`, `__floatsidf`) bloats simple floating-point operations significantly.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic loops recovered as do-while constructs correctly. However, floating-point operations expand into complex conditional chains that obscure the original simple arithmetic expressions.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries match original source well with appropriate granularity. Issue: many test functions constant-folded to single return statements (`call_cdecl` returns `0xf` directly), losing computational structure.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom naturally. Floating-point library functions (`__aeabi_dadd` at 200+ lines) contain excessive branching that disrupts linear reading flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Floating-point operations show repetitive bit-manipulation patterns across `__aeabi_dadd`, `__subdf3`, `__floatsidf`. Constant folding eliminates some redundancy but library expansion creates new patterns.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Basic types preserved (`int`, `char*`, `longlong`) but function pointers degraded to `code*`. Floating-point uses `undefined8`/`ulonglong` instead of `double`. Struct types named correctly but internal precision lost.

**10. Composite Type Recovery:** Score: 6/10  
Justification: Struct types identified (`LargeStruct`, `SmallStruct`, `BigStruct`) with correct field access (`s.data[i]`, `p->x`, `p->y`). However, field names not recoveredâ€”accessed via array indices rather than named members.

**11. Type Safety:** Score: 6/10  
Justification: Multiple implicit casts in floating-point operations (`__floatsidf`, `__aeabi_l2d`). Pointer casts in `param_complex_cast` present but not flagged as potentially unsafe. Code compiles but type safety concerns unmarked.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated warnings present ("WARNING: Unknown calling convention"). No semantic comments explaining algorithm purpose or non-obvious logic. "Unresolved local var" notes lack contextual meaning.

**13. Idiom Recognition:** Score: 4/10  
Justification: Varargs pattern visible but not identified as such. Struct passing conventions not documented. Floating-point library calls not recognized as standard conversion idioms.

**14. Anomaly Indication:** Score: 5/10  
Justification: Calling convention warnings and unreachable block notices present. However, warnings are generic without explaining implications or suggesting manual review actions.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: Null checks present (`if (p == (SmallStruct *)0x0)`) but no precondition/postcondition documentation. Error conditions not documented with expected behaviors or guarantees.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible and traceable through code. No explicit call graph documentation. Callback relationships (`param_func_ptr`) visible but not annotated as callback patterns.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in logical sequence matching source structure. Test functions grouped together appropriately. No header file separation or API boundary documentation.

**18. Build System Compatibility:** Score: 4/10  
Justification: External dependencies (`__printf_chk`, `strlen`, `memcpy`) undeclared. Runtime library functions (`__aeabi_dadd`, `__floatundidf`) undefined. Requires significant stubbing to compile standalone.
