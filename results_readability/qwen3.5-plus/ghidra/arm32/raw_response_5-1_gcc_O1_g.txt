# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`iVar1`, `iVar2`, `sVar1`, `pvVar1`, `puVar1`) with no semantic meaning. Many locals marked as "Unresolved local var" with unknown addresses. Original names like `result`, `r1`, `r2`, `capture_by_value` are completely lost despite -O0 debug compilation.

**2. Function Naming Quality:** Score: 8/10  
Justification: Test function names preserved well (`test_cpp_member_func`, `test_cpp_constructor`). Class methods recovered with scope (`Base::virtual_func`, `Derived::getName`). Template instantiations named correctly (`template_max<int>`). Only startup helpers use generic names (`FUN_000109bc`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation: `0x125c`, `0x15`, `0x47`, `0x16`, `0x55`, `0x2bf` returned directly from test functions. Some inferable (`0x1f` for strncpy, `0x14` for allocation), but most obscure the expected return values from source comments.

**4. Noise Reduction:** Score: 5/10  
Justification: Multiple "WARNING: Removing unreachable block" annotations clutter output. Runtime initialization functions (`__do_global_dtors_aux`, `register_tm_clones`) included. Stack canary checks (`__stack_chk_fail`) appear in template functions. However, no excessive register spilling artifacts.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Critical failure: `test_cpp_multiple_inheritance`, `test_cpp_operator_overload`, `test_cpp_template_class`, `test_cpp_lambda`, `test_cpp_smart_ptr` all return hardcoded constants instead of actual logic. Exception handling in `test_cpp_exception` doesn't recover try-catch structure at all.

**6. Function Decomposition:** Score: 6/10  
Justification: Class methods properly separated as individual functions. Template instantiations correctly split (`template_max<int>` vs `template_max<double>`). However, many test functions collapsed to single return statements, losing all internal structure and logic.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally. Virtual thunk functions have complex pointer arithmetic but follow consistent patterns. No excessive goto usage. Early returns used appropriately in container methods.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Multiple destructor variants for same class (regular, virtual thunk, non-virtual thunk) create apparent redundancy. Some functions like `test_cpp_diamond_inheritance` have overlapping field warnings. However, no obvious dead code within individual functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Basic types preserved (`int`, `double`, `void*`), but C++ specifics degraded: `std::unique_ptr` becomes generic struct references, lambdas become `anon_struct`, template parameters lost in some contexts. Many `undefined4` types where specific types should exist.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Class structures partially recovered with vtable pointers (`_vptr_Base`), but field names lost to generic offsets (`_20_4_`, `field_0x10`, `_8_4_`). Multiple inheritance creates "ignoring overlapping field" warnings. Virtual inheritance structure not fully resolved.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts: `(undefined4 *)`, `(int)pvVar1`, `(char **)`, `(type_info **)`. Pointer arithmetic on typed objects throughout. Virtual function calls use `(**obj->_vptr_Base)()` pattern losing type safety. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: DWARF prototype information preserved in some function comments ("DWARF original prototype:"). Decompiler warnings present for structural issues. However, zero semantic comments explaining what code does. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: Vtable dispatch partially visible but not identified as such. RTTI operations (`__dynamic_cast`, `type_info::operator==`) present but not labeled. Smart pointer RAII pattern, lambda capture semantics, and exception handling idioms completely unrecognized.

**14. Anomaly Indication:** Score: 6/10  
Justification: Good warning coverage: "Unresolved local var", "Removing unreachable block", "Struct ignoring overlapping field", "Unknown calling convention". However, indirect call resolution issues not explained, and confidence levels not indicated for recovered logic.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition/postcondition documentation. Exception guarantees not indicated. Class invariants not recovered. No documentation of expected return values despite source having explicit "期望" (expected) comments.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible and mostly accurate. Virtual dispatch partially recovered through vtable references. However, no call graph documentation, callback relationships unclear, and global variable usage not explicitly documented.

**17. Module Organization:** Score: 5/10  
Justification: Test functions grouped together, class methods associated with classes. However, no header file structure, static functions not properly scoped, and startup/runtime functions mixed with application logic.

**18. Build System Compatibility:** Score: 3/10  
Justification: Will not compile standalone. Missing standard library declarations (`<iostream>`, `<memory>`), undefined external functions (`operator_new__`, `__cxa_throw`, `__dynamic_cast`), and no dependency documentation provided.

**Overall Readability Score:** 4.8/10

### Critical Defects:
- **Test functions return hardcoded constants** instead of actual computation (`test_cpp_multiple_inheritance` returns `0x47`, `test_cpp_lambda` returns `0x55`)
- **Variable names completely generic** (`iVar1`, `pvVar1`) despite -O0 debug compilation with symbol information
- **Exception handling structure lost** - `test_cpp_exception` shows `__cxa_throw` but no try-catch recovery
- **Composite type fields use offset notation** (`_20_4_`, `field_0x10`) instead of semantic names (`shared_data`, `multiplier`)
- **Magic numbers without explanation** - return values like `0x125c` should be documented as computed results

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Test function names and class method scopes preserved accurately from debug symbols, making navigation possible despite other issues.

### Most Needs Improvement:
- **Assertion & Contract Recovery (2/10)** - Zero documentation of expected behaviors, preconditions, or return value semantics despite source having explicit expected value comments.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover local variable names from debug symbols** - At -O0 with `-g`, DWARF contains variable names that should be used instead of `iVar1`/`pvVar1` patterns.
2. **Detect and reconstruct control flow from basic blocks** - Functions returning constants indicate failed CFG reconstruction; implement better block merging for test functions.
3. **Add semantic annotations for C++ idioms** - Label vtable calls as "virtual dispatch", mark `__cxa_*` calls as exception handling, and document smart pointer patterns.
