# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `param_1`, `param_2`, `iVar1`, `uVar1` with zero semantic meaning. Original source had meaningful names like `arr`, `n`, `sum`, `p`, `current` that indicated purpose.

**2. Function Naming Quality:** Score: 7/10  
Justification: User function names preserved well (`process_char`, `array_1d_stack`, `struct_simple`), but runtime functions have opaque names (`__muldf3`, `FUN_00011608`). Function names match original source intent.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere: `0xbf`, `0x1a`, `0x2c`, `0x1b8`, `0x10000` with no explanation. Test functions print pre-computed hex values (`0x61`, `0xf`, `0x3c`) instead of showing actual computation.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler runtime functions (`__muldf3`, `__adddf3`, `__mulsf3`) bloat output with 200+ lines each. Multiple "WARNING: Removing unreachable block" annotations pollute the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Loops converted to awkward `do-while` with manual decrement (`param_2 = param_2 + -1`). Float functions incorrectly return `void` instead of values. Original `for` loops lost entirely.

**6. Function Decomposition:** Score: 6/10  
Justification: User functions appropriately sized, but test functions optimized away to constant printing. Runtime functions excessively large (200+ lines for float operations).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but complex functions like `__muldf3` have deep nesting and forward jumps. Simple functions maintain reasonable linearity.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions show redundant constant printing instead of actual computation. Runtime float functions have significant bit-level operation redundancy from software emulation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe degradation: `process_short` returns `int` not `short`, `process_ll` returns `undefined8`, float/double functions return `void`. `_Bool` becomes `bool`/`uint`. Pointer types collapsed to `int*`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - all structs accessed as `int*` with offsets (`param_1[1]`, `param_1 + 0x14`). No struct definitions recovered. Unions, bitfields, nested structs all lost. Original had `Point3D`, `Rect`, `Widget`, `Node`, etc.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts throughout: `(int *)*param_1`, `undefined4 ****`. Pointer arithmetic on generic `int*`. Function pointers typed as `code *`. Original type safety completely lost.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated warnings ("WARNING: Type propagation algorithm not settling"). No semantic comments explaining algorithm purpose. Original had extensive Chinese comments explaining each test case.

**13. Idiom Recognition:** Score: 2/10  
Justification: Zero pattern recognition. Linked list traversal (`linked_list`, `doubly_linked_list`) not identified. Tree recursion (`binary_tree_sum`) not labeled. Graph traversal appears as raw pointer chasing.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**14. Anomaly Indication:** Score: 4/10  
Justification: Some warnings present ("Removing unreachable block", "Type propagation algorithm not settling") but no clear uncertainty markers on critical decompilation failures like struct loss.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL`). No postconditions. Original had implicit contracts like "requires valid pointer" completely lost.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls preserved but relationships not documented. No call graph. Test functions don't actually call the functions they test (print constants instead).

**17. Module Organization:** Score: 4/10  
Justification: All 81 functions in single output. Runtime functions mixed with user code. No logical grouping by category (data types, arrays, pointers, composites).

**18. Build System Compatibility:** Score: 3/10  
Justification: Won't compile standalone - missing struct definitions, type definitions (`undefined4`, `undefined8`), runtime function implementations. Requires significant patching.

**Overall Readability Score: 3.9/10**

### Critical Defects:
- **All variables named generically** (`param_1`, `iVar1`) losing all semantic meaning from original source
- **Complete struct/union type loss** - all composite types reduced to `int*` with magic offsets (`param_1 + 0x14`, `param_1[1]`)
- **Test functions optimized away** - print pre-computed constants instead of actually calling functions, hiding real behavior
- **Float/double functions return `void`** instead of proper types, breaking semantic understanding
- **Type precision collapsed** - `short`, `long long`, `_Bool` all degraded to `int`/`uint`/`undefined8`

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - User function names like `process_char`, `array_1d_stack`, `struct_simple` preserved accurately from source, providing anchor points for understanding.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover any struct/union definitions; all field access reduced to pointer arithmetic with magic offsets, making complex data structure code nearly unreadable.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct definitions from access patterns** - Analyze consistent offset usage (`+0`, `+4`, `+8`, `+12`) to reconstruct struct layouts with field names based on usage context.

2. **Preserve actual function calls in test code** - Don't optimize test functions to constant printing; maintain the actual call relationships to show how functions interact.

3. **Infer variable names from usage semantics** - Variables used in loop bounds should be named `count`/`n`, accumulators should be `sum`, pointers should indicate what they point to (`arr_ptr`, `node_ptr`).
