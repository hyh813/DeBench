# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`local_4`, `local_c`, `iVar1`, `local_18`) instead of semantic names from source (`arr`, `sum`, `result`, `head`). Even at `-O0` with debug symbols, Ghidra fails to recover meaningful variable names.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names are well-preserved from source (`local_vars`, `heap_basic`, `vla_stack`, `linked_list_heap`). Action-oriented test functions also retain descriptive names (`test_stack_memory`, `test_heap_memory`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers appear throughout (`0x2a` for 42, `0x14` for 20, `0x100` for 256). Bit shifts replace arithmetic (`x << 1` instead of `x * 2`), obscuring intent.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive "Unresolved local var" comments clutter every function. Compiler runtime functions (`_init`, `_start`, `__sync_*` variants) inflate output without value.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Most loops preserved as `for` constructs, but `stack_alias` function is completely broken (compares stack address to `0x10`). `tls_access` uses incomprehensible `__aeabi_read_tp` calls.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries match source exactly. No inappropriate inlining or fragmentation. Each test function remains separate and cohesive.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but awkward variable reuse (`local_28[1]` as loop counter in `vla_stack`) disrupts flow. Some functions use `while(true)` with breaks instead of natural loops.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Pointer arithmetic repeated throughout (`(int)__ptr + local_18 * 4` instead of `__ptr[local_18]`). No dead code elimination of compiler prologue/epilogue patterns.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `Point*` becomes `int*`, `Node*` sometimes `int*`, many `undefined4` types. `size_t` parameters often shown as `int`. Signed/unsigned distinctions lost.

**10. Composite Type Recovery:** Score: 5/10  
Justification: `Node` struct partially recovered with `data` and `next` fields. `Point` struct completely lost (accessed as `__ptr[0]`, `__ptr[1]`). `TreeNode` struct recovered with field names.

**11. Type Safety:** Score: 4/10  
Justification: Extensive unsafe casts (`(int*)0x0`, `(void*)0x0`). Pointer arithmetic instead of array indexing throughout. Function pointer type `_func_int_int` is non-standard.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments are auto-generated warnings ("Unresolved local var", "WARNING: Unknown calling convention") that indicate problems but provide no helpful explanation. No algorithm documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. `malloc`/`free` pairs not identified as memory management. Linked list traversal not recognized. VLA/allocation patterns appear as raw pointer arithmetic.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies flagged ("Unresolved local var", "Heritage AFTER dead removal"), but critical failures like `stack_alias` logic corruption not warned. TLS access completely garbled without explanation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `n > 0` checks not highlighted). Null pointer checks present but not documented as contracts. No postcondition inference.

---

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls preserved with correct names. Global variables referenced by name (`global_counter`, `global_array`). Callback relationships not documented (`set_global_callback` → `call_global_callback`).

**17. Module Organization:** Score: 5/10  
Justification: All 114 functions in single output file. No logical grouping by memory type (stack/heap/static). Runtime functions mixed with user code.

**18. Build System Compatibility:** Score: 4/10  
Justification: Would not compile standalone. Missing type definitions (`Node`, `TreeNode`, `Point`). External dependencies (`extern_global_var`, `extern_function`) undefined. Non-standard types (`_func_int_int`, `__pid_t`).

### Overall Readability Score: **4.9/10**

### Critical Defects:
- **`stack_alias` function completely broken** - Logic corrupted, compares stack address to `0x10` instead of pointer aliasing check
- **`tls_access` function unreadable** - `__aeabi_read_tp` calls with bit manipulation obscure thread-local storage access
- **Variable names all generic** - `local_4`, `local_c`, `iVar1` instead of `arr`, `sum`, `result` despite `-O0 -g` debug symbols
- **Type information severely degraded** - `Point*` becomes `int*`, struct fields accessed via array indexing
- **VLA/allocation logic obscured** - Complex pointer arithmetic (`iVar1 = -0x28 - (n * 4 + 7U & 0xfffffff8)`) instead of clear array allocation

### Best Quality Aspect: **Function Naming Quality (8/10)**
- Function names preserved accurately from source, making high-level navigation possible despite other issues

### Most Needs Improvement: **Idiom Recognition (2/10)**
- No pattern recognition for common constructs (malloc/free pairs, linked list traversal, memory operations)

### Top 3 Recommendations for Decompiler Improvement:
1. **Leverage debug symbols at -O0** - DWARF information contains variable names that should be recovered instead of `local_X` generics
2. **Recover struct types from field access patterns** - Consistent `ptr[0]`, `ptr[1]` access should suggest struct recovery with field names
3. **Add semantic comments for memory operations** - Identify `malloc`/`free` pairs, array initialization loops, and pointer aliasing patterns with explanatory comments
