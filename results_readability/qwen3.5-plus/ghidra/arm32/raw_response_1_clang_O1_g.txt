# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic names like `iVar1`, `iVar2`, `uVar2` throughout; many locals marked as "Unresolved local var" (e.g., `temp1`, `temp2`, `temp3` in `sequential_ops`). Only function parameters retain original names (a, b, c, x).

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names perfectly preserved from source (`sequential_ops`, `single_if`, `nested_if_2`, `duffs_device`, etc.). Action-oriented names clearly reflect purpose.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x3e9` (1001), `0x65` (101), `0x12` (18) appear without explanation. Some recoverable through context but not documented.

**4. Noise Reduction:** Score: 6/10  
Justification: Multiple "WARNING: Removing unreachable block" comments and "Unresolved local var" annotations pollute output. Runtime functions (`__divsi3`, `__aeabi_idivmod`) included without clear separation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: `single_if` uses bit shift `x << (uint)(0 < x)` instead of clear conditional. `loop_for_fixed` replaced with mathematical formula. `duffs_device` uses goto chains instead of elegant fall-through.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity matches source well. No excessive inlining or fragmentation. Each function maintains single responsibility.

**7. Code Linearization:** Score: 6/10  
Justification: `duffs_device` has multiple goto labels (`switchD_00010d28_caseD_7`) disrupting flow. `nested_if_2` flattened to sequential ifs losing else structure.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Loop optimizations converted to formulas (`loop_for_fixed`, `loop_nested`) create cognitive overhead. No dead code observed but optimization artifacts remain.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Generic `int` used throughout instead of specific types. `longlong` appears in `loop_for_fixed` where `int` sufficed. Function pointers use `_func_int_int *` (non-standard).

**10. Composite Type Recovery:** Score: 6/10  
Justification: Arrays shown as pointer arithmetic `(&DAT_00011974)[iVar1]`. Matrix access in `loop_multi_exit` uses `puVar5[iVar2 * 4]` instead of `matrix[i][j]`.

**11. Type Safety:** Score: 6/10  
Justification: Multiple unsafe casts visible: `(uint)x`, `(ulonglong)`, `(code *)`. Function pointer calls use explicit casts that obscure type safety.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10

**14. Anomaly Indication:** Score: 7/10

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition/postcondition documentation. Error conditions not explicitly documented. Function contracts entirely missing.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10
