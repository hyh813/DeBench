# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `local_30`, `pbVar1`) with zero semantic meaning. No variable indicates its purpose (e.g., `result`, `encrypted`, `buffer`).

**2. Function Naming Quality:** Score: 7/10  
Justification: Original function names preserved (`param_fake_branch`, `param_opaque_predicate`), accurately reflecting purpose. However, helper functions use generic names (`FUN_00011030`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers throughout without explanation (`0x14`, `0xe1`, `0x2b`, `0x7a314`, `0x5a`). Some are constant-folded results but lack context about original meaning.

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler artifacts present: `get_pc_thunk_*` functions, stack canary checks (`in_GS_OFFSET`), and multiple "WARNING: Removing unreachable block" annotations clutter the output.

### Critical Defects

- **Variable `param_1`, `iVar1`, `local_30`** should be named semantically (e.g., `input_value`, `gcd_result`, `decrypted_buffer`) based on usage context
- **Caller functions constant-folded**: `call_fake_branch()`, `call_opaque_predicate()`, etc. return literals instead of calling actual functions, destroying call graph visibility
- **`param_fake_branch` logic eliminated**: The fake branch conditions (`x*x < 0`, `strlen(str) < 0`) completely removed by optimizer, losing the obfuscation demonstration
- **Composite types lost**: Bitfield struct in `param_implementation_defined` optimized to constant `0x2b` with no struct recovery
- **Architecture mismatch**: Original ARM64, decompiled as x86 - indicates potential binary mismatch or re-compilation

---

### Best Quality Aspect

**Control Flow Naturalness (8/10)** - Loop structures recovered accurately from assembly, including tail-call optimization conversion from recursion to iteration in `param_tail_call_optimized`.

---

### Most Needs Improvement

**Composite Type Recovery (2/10)** - Complete failure to recover any struct/union types; the bitfield example was optimized away entirely with no indication of original structure.

---

### Top 3 Recommendations for Decompiler Improvement

1. **Preserve call relationships even with constant folding**: When a caller is constant-folded, add a comment showing the original call (e.g., `// Originally: return param_fake_branch(10)`)

2. **Detect and annotate optimization artifacts**: Flag when dead code elimination removed logic (e.g., "OPTIMIZATION: Always-false branch eliminated at -Os")

3. **Infer variable names from usage patterns**: Variables used in arithmetic contexts should be named by operation (e.g., `iVar2` used in GCD loop → `gcd_temp`), and buffer variables → `buffer`, `decrypted`
