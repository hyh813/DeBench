# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names like `iVar1`, `iVar2`, `uVar1` with no semantic meaning. Original source had meaningful names (`temp1`, `sum`, `count`, `result`) that are completely lost. Multiple "Unresolved local var" comments indicate failed variable recovery.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved correctly (`sequential_ops`, `nested_if_2`, `recursion_factorial`). However, some helpers use generic names (`FUN_00011030`) and architecture mismatch warnings appear throughout.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x3e9` (1001), `0x65` (101), `0x14` (20) appear without explanation. Test functions print expected constants instead of computing values, obscuring actual logic flow.

**4. Noise Reduction:** Score: 5/10  
Justification: Significant compiler artifacts present: `__x86.get_pc_thunk_*` injections, "WARNING: Unknown calling convention", "WARNING: Removing unreachable block", and stack canary checks (`__stack_chk_fail_local`) clutter the output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most loops recovered as do-while correctly, but `loop_multi_exit` uses goto labels (`LAB_00011a9e`) instead of structured loops. `computed_goto` completely fails, returning constant `0x14`. At -O0, structure should be better preserved.

**6. Function Decomposition:** Score: 6/10  
Justification: Function granularity is reasonable, but test functions are over-optimized (printing constants instead of calling functions). Indirect call warnings ("Could not recover jumptable") indicate decomposition issues.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but `duffs_device` has complex label-based flow (`_L185`, `_L187`, etc.). `loop_complex_cond` has deeply nested if-chains (10+ levels) that disrupt linear reading.

**8. Redundancy Elimination:** Score: 6/10  
Justification: `duffs_device` shows unrolled loop code that's hard to follow. `loop_complex_cond` has repetitive if-chain patterns that could be simplified. Some redundancy from optimization not cleaned up.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic `int` types preserved, but function pointers degraded to `_func_int_int*`. Multiple "Unknown calling convention" warnings. Some implicit casts like `(uint)(event == 1)` lose type clarity.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Arrays recovered (`funcs[3]`, `ops[10]`, `state_handlers[4]`), but no struct recovery visible. All composite types remain as raw arrays or pointers without field names.

**11. Type Safety:** Score: 5/10  
Justification: Unsafe casts present (`(byte)b & 0x1f` for shift operations). Function pointer calls have "Treating indirect jump as call" warnings. Stack canary checks indicate potential safety concerns.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments are almost entirely auto-generated warnings ("WARNING:", "Unresolved local var"). No semantic comments explaining algorithm purpose or non-obvious logic. Original source had extensive documentation completely lost.

**13. Idiom Recognition:** Score: 4/10  
Justification: State machine pattern in `state_machine` and `fsm_func_table` not identified. Function pointer dispatch tables not recognized as design patterns. No algorithm naming or pattern hints.

**14. Anomaly Indication:** Score: 6/10  
Justification: Some anomalies flagged ("Could not recover jumptable", "Unresolved local var", "Unknown calling convention"). However, `computed_goto` silently returns wrong value without warning in first occurrence.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Error conditions not explained. Original source had test validation comments completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variable `jump_buffer` used without declaration context. Callback relationships in `process_with_callback` not explained.

**17. Module Organization:** Score: 5/10  
Justification: Functions listed sequentially without logical grouping. No header file structure. Test functions separated from implementation but organization appears arbitrary.

**18. Build System Compatibility:** Score: 4/10  
Justification: Won't compile standalone - undefined types (`_func_int_int*`, `EVP_PKEY_CTX`), missing external declarations (`__printf_chk`, `__do_global_ctors_aux`), architecture mismatch (ARM64 source → x86 decompilation).

**Overall Readability Score:** 5.1/10

### Critical Defects:
- **Variable names completely generic** (`iVar1`, `iVar2`) - original semantic names (`temp1`, `sum`, `count`) entirely lost
- **Test functions print constants** instead of calling actual functions, hiding real computation logic
- **`computed_goto` returns constant `0x14`** - complete decompilation failure without adequate warning
- **Architecture mismatch** - source claims ARM64 but decompilation shows x86 with x86-specific artifacts
- **No semantic comments** - original extensive documentation completely absent, only auto-generated warnings remain

### Best Quality Aspect: **Function Naming Quality (7/10)**
- Most original function names preserved correctly, allowing identification of purpose despite other degradation

### Most Needs Improvement: **Assertion & Contract Recovery (2/10)**
- Zero precondition/postcondition documentation; no error handling contracts; original test validation comments completely lost

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement variable name inference** from usage patterns and debug symbols when available (`-g` flag should preserve `temp1`, `sum`, `count` names)
2. **Detect and flag incomplete decompilation** - `computed_goto` returning constant should trigger high-confidence warning about failed recovery
3. **Preserve source comments and documentation** from debug sections when compiled with `-g`, especially test validation expectations
