# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`iVar1`, `local_24`, `local_30`) with no semantic meaning. "Unresolved local var" comments show original names like `buffer`, `len`, `ret` were lost during decompilation.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core functions retain original names (`param_strcpy`, `param_strcmp`, `thread_compute`) which is excellent. However, initialization functions use generic names (`FUN_00011030`, `deregister_tm_clones`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation: `0x2a` (42), `0x5a` (90), `0xc` (12), `0xf` (15). String literals partially recovered as `DAT_` references instead of inline strings.

**4. Noise Reduction:** Score: 5/10  
Justification: Excessive "Unresolved local var" comments clutter every function. Architecture mismatch warnings (ARM64 source → x86 decompilation) and calling convention warnings add noise throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most control flow is natural (if/else, while loops), but `call_strcpy` shows optimization artifacts with split `builtin_strncpy` calls instead of single `strcpy`. Some do-while patterns could be cleaner.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized and cohesive. Thread functions properly separated (`thread_compute`, `thread_sum`, `thread_increment`). Test functions correctly isolated from implementation.

**7. Code Linearization:** Score: 6/10  
Justification: Code generally reads top-to-bottom, but forward references to `DAT_` constants disrupt flow. Some functions have nested conditionals that could be flattened.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Constant folding occurred aggressively (`call_strcpy`, `call_strlen` return constants directly). Some redundant type casts present. Original computation logic lost in caller functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Significant type degradation: `const char*` becomes `char*`, `size_t` often shown as `int`, `ssize_t` lost. Function `param_strcpy` shows `char *src` instead of `const char *src`.

**10. Composite Type Recovery:** Score: 5/10  
Justification: `stat` struct recovered for file operations, `sockaddr` partially recovered. However, `ThreadData` struct not recovered—shown as raw memory access (`*(int *)((int)arg + 8)`).

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts visible: `(int)buffer`, `(pthread_mutex_t *)&counter_mutex`. No const-correctness preserved. Pointer arithmetic exposed in thread functions.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Only auto-generated "Unresolved local var" comments present. No algorithm explanations, no API contracts, no documentation of non-obvious logic. Original Chinese comments completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (`strcpy`, `strlen`, `pthread_create`). Thread patterns partially identified. No design pattern identification or higher-level semantic recovery.

**14. Anomaly Indication:** Score: 5/10  
Justification: "WARNING" comments for calling conventions and unresolved variables. However, architecture mismatch (ARM64→x86) not clearly flagged. Some anomalies silently ignored.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., "requires: dst != NULL"). No postconditions inferred. Error return values not explained. Original comments about expected values completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped logically (library, syscall, thread sections preserved). Test functions separate from implementation. No header file structure or namespace organization.

**18. Build System Compatibility:** Score: 6/10

**Overall Readability Score: 5.1/10**

### Critical Defects
- **Variable names completely generic** (`iVar1`, `local_24`) losing all semantic meaning from original source
- **Const-correctness lost** - `const char*` parameters degraded to `char*` throughout
- **Magic numbers unexplained** - `0x2a`, `0x5a`, `0xc` appear without connection to original values (42, 90, 12)
- **Caller functions constant-folded** - `call_strcpy`, `call_strlen` return constants directly, losing original computation logic
- **Architecture mismatch unflagged** - Original ARM64 code decompiled as x86 without clear warning

### Best Quality Aspect
- **Function Naming Quality (7/10)** - Core function names (`param_strcpy`, `thread_compute`, `signal_handler`) preserved from original source, maintaining semantic clarity for main entry points.

### Most Needs Improvement
- **Assertion & Contract Recovery (3/10)** - No preconditions, postconditions, or error documentation recovered. Original comments explaining expected values and error conditions completely lost.

### Top 3 Recommendations for Decompiler Improvement
1. **Recover variable names from debug symbols** - The `-g` flag was used during compilation; Ghidra should leverage DWARF debug info to restore original variable names instead of `local_XX` generics.

2. **Preserve const-qualifiers and type precision** - Implement better type inference to maintain `const char*` vs `char*` distinctions and recover `size_t`/`ssize_t` instead of generic `int`.

3. **Detect and annotate constant folding** - When caller functions are optimized to return constants, add comments showing original computation (e.g., `// Original: strlen("HelloLib") = 8`) to maintain semantic transparency.
