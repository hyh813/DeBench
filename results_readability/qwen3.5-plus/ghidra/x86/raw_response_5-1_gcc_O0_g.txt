## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Variables like `iVar1`, `iVar2`, `iVar3`, `iVar4`, `iVar5` dominate test functions, obscuring their semantic roles. Some better names exist (`capture_by_value`, `capture_by_ref`, `obj`, `result`), but class member access (`this->value`, `this->name`) is the primary source of clarity.

**2. Function Naming Quality:** Score: 7/10  
Justification: Test functions retain original names (`test_cpp_member_func`, `test_cpp_constructor`), and class methods use proper `Class::method` format. However, helper functions have cryptic names (`FUN_00011030`, `__uniq_ptr_data`, `anon_struct_8_2_35feab7f`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound without explanation: `0x1234` (class ID), `0x1f` (buffer size-1), `0x14` (stack offset), `0x2a` (42 decimal), `0x96` (150 decimal), `0xfa` (250 decimal). Some are inferable from context but none are named constants.

Justification: Stack canary checks appear in EVERY function (`if (iVar1 != *(int *)(in_GS_OFFSET + 0x14))`), 211 functions include many compiler thunks (`__x86.get_pc_thunk_*`), and internal helper functions bloat the output significantly.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===
