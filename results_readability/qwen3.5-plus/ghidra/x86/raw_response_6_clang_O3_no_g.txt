# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are generically named (`param_1`, `param_2`, `iVar1`, `uVar2`, `local_24`) with no semantic meaning. Original source had meaningful names like `dst`, `src`, `buffer`, `fd`, `shm` that clearly indicated purpose.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`param_strcpy`, `call_strcmp`, `thread_compute`). However, helper functions use generic names (`FUN_00011030`, `FUN_00012525`) and some thread functions lost descriptive context.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive throughout: `0x2a` (42), `0xffffffff` (-1), `0xfffffffe` (-2), `0x1b6` (file permissions). String literals referenced as `DAT_00014xxx` instead of inline. No named constants recovered.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: `LOCK()/UNLOCK()` macros for atomics, `get_pc_thunk_*` injection warnings, redundant type casts (`(undefined4 *)`, `(void **)0x0`). Some unreachable block warnings clutter output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic control flow preserved but ternary operators degraded to verbose patterns (`iVar2 = -(uint)(iVar1 != 0)`). Loop unrolling in `param_malloc_free` despite `-O0` compilation. Some functions use awkward conditional chains.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries correctly identified with appropriate granularity. Thread functions properly separated. However, return type mismatches (void vs int) indicate decomposition errors in signature recovery.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with appropriate early returns. No excessive goto usage. Nested conditionals in `param_signal_handling` and `param_fread_fwrite` are deep but traceable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Unexpected loop unrolling in `param_malloc_free` creates redundant code blocks. `call_*` wrapper functions often duplicate logic from `param_*` functions instead of calling them. Some dead code patterns from optimization.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Significant type degradation: `size_t` becomes `uint` or `int`, `const char*` loses const qualifier, `FILE*` sometimes becomes generic pointer. Return types wrong (`param_strcpy` returns void instead of int). `undefined4` used frequently.

**10. Composite Type Recovery:** Score: 4/10  
Justification: `stat` and `sockaddr` structs partially recovered but fields accessed via array indexing (`local_5c.st_size`). ThreadData struct completely lost (fields accessed as `param_1[0]`, `param_1[1]`, `param_1[2]`). TLS accessed via raw `in_GS_OFFSET` pointer arithmetic.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts: `(pthread_mutex_t *)counter_mutex`, `(volatile int*)&atomic_counter`. Function pointers often `void*`. Pointer arithmetic without proper typing in TLS and atomic operations. Missing const correctness throughout.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments explaining logic. Only auto-generated warnings ("WARNING: Function... replaced", "WARNING: Subroutine does not return"). No algorithm explanations, API contracts, or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (`strcpy`, `strlen`, `printf`, `pthread_create`). Signal handling pattern partially visible. No design pattern identification. Atomic operations shown as `LOCK()/UNLOCK()` without explaining CAS semantics.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some warnings present ("Subroutine does not return" for `_exit`), "Type propagation algorithm not settling" for `param_pthread_join`. No flagging of uncertain decompilation or potential undefined behavior. Inconsistent anomaly reporting.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL`). No postconditions inferred. Error return values (`0xffffffff`, `0xfffffffe`) not explained. No documentation of expected behavior or failure modes.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible but no call graph documentation. Global variables (`signal_received`, `atomic_counter`, `shared_counter`) referenced but not grouped. Thread callback relationships not explicitly documented.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in source order (library → syscall → thread tests). Related test functions grouped together. No header file structure or API declarations. All 75 functions in single output without logical separation.

**18. Build System Compatibility:** Score: 4/10  
Justification: External dependencies not documented (pthread, libc, syscalls). Would require extensive stubs for `LOCK()/UNLOCK()`, `get_pc_thunk_*`, and platform-specific functions. Architecture mismatch (ARM64 source → x86 decompilation) creates portability issues.

- **Magic numbers unexplained**: `0x2a`, `0xffffffff`, `0x1b6` throughout code without constant definitions
- **TLS access via raw pointers**: `in_GS_OFFSET[-9]` instead of named `tls_counter` variable
- **Struct fields lost**: `ThreadData` accessed as array indices `param_1[0]`, `param_1[1]`, `param_1[2]` instead of `start`, `end`, `result`

### Best Quality Aspect:

3. **Recover struct field names from usage patterns**: Analyze field access patterns across functions to reconstruct `ThreadData { start, end, result }` instead of array indexing
