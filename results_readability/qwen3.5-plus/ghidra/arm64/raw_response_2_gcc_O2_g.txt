# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic names like `iVar1`, `piVar1`, `lVar3` instead of semantic names like `sum`, `i`, `len` from source. "Unresolved local var" comments acknowledge lost information but don't recover it.

**2. Function Naming Quality:** Score: 7/10  
Justification: Application functions preserved original names (`process_char`, `array_1d_stack`, `struct_simple`), but runtime functions use generic names (`FUN_001006d0`, `FUN_0010087c`).

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions use hardcoded hex values (`0x61`, `0x62`, `0x400c000000000000`) instead of computed results. Magic numbers like `0xbfU` in `process_char` obscure the `'A'-'Z'` range check intent.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`___stack_chk_guard`), runtime init functions (`_init`, `__do_global_dtors_aux`), and "Unresolved local var" comments add noise but don't completely obscure logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions use natural constructs, but `binary_tree_sum` transformed from clean recursion to 200+ lines of nested do-while loops. Loop unrolling in `array_3d` obscures the original triple-nested for-loop structure.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved, but test functions (`test_data_types_l1`, `test_array_types`) replaced actual function calls with hardcoded expected values, losing the test structure.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `binary_tree_sum` has excessive forward jumps and nested while-true patterns that disrupt linear reading flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `binary_tree_sum` repeats identical traversal patterns 7+ levels deep. Long double operations (`__addtf3`, `__multf3`) expanded to 500+ lines each with duplicated bit-manipulation logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types (int, char, short, long) recovered accurately. However, `long_double` becomes `undefined1[16]` and function pointers use synthetic types (`_func_int_int`) instead of proper signatures.

**10. Composite Type Recovery:** Score: 8/10  
Justification: Excellent struct recovery—`Point3D`, `Rect`, `Widget`, `Node`, `DNode`, `TreeNode`, `Graph` all preserved with field names. Union `UnionData` and enum `State` also recovered correctly.

**11. Type Safety:** Score: 7/10  
Justification: Casts are mostly appropriate (`(int*)p`, `(char*)p`). Some void* conversions lack explicit documentation but don't introduce obvious type confusion or unsafe patterns.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: "Unresolved local var" and WARNING comments indicate decompiler uncertainty, but no algorithm explanations or logic documentation. Comments mark problems without explaining intent.

**13. Idiom Recognition:** Score: 5/10  
Justification: Linked list traversal recognized in `linked_list` function, but tree recursion completely lost (expanded to iterative). No design pattern identification (State machine in `enum_switch` not noted).

**14. Anomaly Indication:** Score: 7/10  
Justification: Good warning coverage—"WARNING: Could not recover jumptable", "WARNING: Unknown calling convention", "WARNING: Subroutine does not return". Clearly flags indirect call resolution failures.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition documentation (e.g., `arr != NULL`, `n > 0`). No postcondition inference. Array bounds checks present but not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible in code but no call graph documentation. Struct field access clear, but no documentation of callback relationships (e.g., `ptr_func_simple`'s function pointer parameter).

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file. Runtime functions (`_init`, `_fini`, `__do_global_dtors_aux`) mixed with application code. No header file separation or logical grouping.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates "Compilation Result: Yes" but external dependencies (`__printf_chk`, `puts`, `__libc_start_main`) not clearly documented. Would need stub implementations.

**Overall Readability Score:** 5.8/10

### Critical Defects
- **Variable names completely generic** (`iVar1`, `piVar1`, `lVar3`) losing all semantic meaning from original (`sum`, `i`, `len`)
- **`binary_tree_sum` recursion destroyed** — transformed from 5-line recursive function to 200+ lines of nested do-while loops
- **Test functions use hardcoded values** instead of actual function calls, losing test verification structure
- **`long double` operations expanded** to 500+ line software emulation functions (`__addtf3`, `__multf3`)
- **No contract documentation** — preconditions, postconditions, and null checks completely absent

### Best Quality Aspect
- **Composite Type Recovery (8/10)** — Struct definitions (`Point3D`, `Widget`, `TreeNode`, `Graph`) recovered with accurate field names and nesting preserved

### Most Needs Improvement
- **Assertion & Contract Recovery (3/10)** — Zero documentation of function preconditions, postconditions, or error conditions despite clear validation patterns in source

### Top 3 Recommendations for Decompiler Improvement
1. **Implement usage-based variable naming** — Analyze variable usage patterns to infer names (e.g., accumulator variables → `sum`, loop counters → `i`)
2. **Preserve recursive structures** — Detect tail-call and tree traversal patterns to recover recursive form instead of expanding to iterative loops
3. **Add contract inference** — Detect null checks, bounds validation, and return value patterns to generate precondition/postcondition documentation
