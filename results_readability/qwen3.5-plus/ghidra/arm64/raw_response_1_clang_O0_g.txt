# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are generically named (`local_4`, `local_8`, `local_c`) instead of semantic names like `temp1`, `sum`, `count`. Multiple "Unresolved local var" comments indicate the decompiler couldn't recover original variable purposes.

**2. Function Naming Quality:** Score: 8/10  
Justification: User-defined function names preserved accurately (`sequential_ops`, `single_if`, `nested_if_deep`). However, some internal functions use generic names (`FUN_00100880`, `FUN_001009fc`).

**3. Literal Clarity:** Score: 5/10  
Justification: Mix of decimal and hex literals (e.g., `0x14` instead of `20`, `0x32` instead of `50`). Constant expressions folded (`a+b=10+5` becomes `0xf`), obscuring original computation intent.

**4. Noise Reduction:** Score: 5/10  
Justification: "Unresolved local var" comments appear throughout. Redundant type casts (`(ulong)uVar1` for printf). Some functions have unreachable block warnings that clutter output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control structures preserved (if-else, for, while). However, `switch_fallthrough` converted to nested if-statements, and `loop_break` uses `while(true)` with break instead of for loop.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity preserved appropriately. No inappropriate inlining or fragmentation. Each function maintains single responsibility as in source.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom. Some functions use `while(true)` patterns with breaks that disrupt natural flow. `duffs_device` has goto labels for fall-through cases.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant variable assignments present (`local_4 = x;` then `local_4 = x << 1;`). Array initialization in `loop_break` shows individual element assignments instead of initializer list.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Generic types dominate (`undefined4`, `undefined8`, `int`). Original `int` types mostly preserved but without size specificity. Function pointer types partially recovered (`_func_int_int *`).

**10. Composite Type Recovery:** Score: 6/10  
Justification: Arrays shown as individual element assignments (`local_20[2] = 0x1e; local_20[3] = 0x28;`). No struct recovery needed for this code, but array semantics degraded.

**11. Type Safety:** Score: 6/10  
Justification: Unsafe casts present (`(ulong)uVar1` for printf arguments). Function pointer calls use generic `code *` type. No buffer overflow protections visible.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Only technical warnings present ("WARNING: Subroutine does not return"). No semantic documentation about algorithm purpose. "Unresolved local var" comments indicate uncertainty but don't explain intent.

**13. Idiom Recognition:** Score: 6/10  
Justification: Basic loop patterns recognized (for, while, do-while). Duff's device partially recognized with goto labels. Complex patterns like fall-through switch not properly identified.

**14. Anomaly Indication:** Score: 7/10  
Justification: Good warning coverage for `longjmp` ("Subroutine does not return") and `computed_goto` ("Could not recover jumptable"). "Unresolved local var" comments indicate decompiler uncertainty appropriately.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition/postcondition documentation recovered. No contract information about parameter validity or return value guarantees. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls preserved and visible. Global variable references present (`jump_buffer`, `completed_0`). No explicit call graph documentation or relationship annotations.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in reasonable order matching source. Test functions grouped together. No header file structure or API declarations recovered.

**18. Build System Compatibility:** Score: 4/10  
Justification: Code dump from single file. External dependencies not clearly documented (printf, memcpy, setjmp). Would require significant work to compile standalone.

### Top 3 Recommendations for Decompiler Improvement:

3. **Recover array initializer syntax** from sequential element assignments to improve readability of static data structures
