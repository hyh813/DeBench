# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`iVar1`, `piVar2`, `puVar3`, `__ptr`) with no semantic meaning. Many locals show as "Unresolved local var: int arr@[???]" losing all naming information from source.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions preserved (`double_value`, `local_vars`, `heap_basic`) but many helpers are `FUN_001009b0`, `FUN_00100bfc`. Source function names recovered where debug info existed.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive (`0x2a` instead of `42`, `0x14` instead of `20`, `0xf` instead of `15`). Hex representation obscures semantic meaning of simple integer constants.

**4. Noise Reduction:** Score: 5/10  
Justification: Repeated "Unresolved local var" comments throughout (e.g., `/* Unresolved local var: int i@[???] */`). Multiple "WARNING: Removing unreachable block" and "Unknown calling convention" warnings clutter output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loops recovered as do-while patterns but with awkward pointer arithmetic (`piVar2 = piVar2 + 1`). Some functions show "WARNING: Removing unreachable block" indicating flow reconstruction issues.

**6. Function Decomposition:** Score: 5/10  
Justification: Test functions (`test_stack_memory`, `test_heap_memory`) have actual function calls replaced with hardcoded constants, losing the call structure. Function sizes reasonable but semantic decomposition lost.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but complex VLA/allocation functions have nested pointer arithmetic that disrupts linear reading. Some functions have multiple exit points without clear structure.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions compute values at compile-time then hardcode results (e.g., `local_vars(5)` becomes `0x14` directly). This eliminates runtime behavior visibility.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Many types degraded to `undefined8`, `undefined4`, `undefined1`. Proper types recovered for `Node*`, `TreeNode*` but `Point` struct lost (shown as `x * 3` calculation instead of struct access).

**10. Composite Type Recovery:** Score: 6/10  
Justification: `Node` and `TreeNode` structs recovered with field names (`data`, `next`, `left`, `right`). `Point` struct completely lost (fields accessed as arithmetic). Array types often shown as pointer arithmetic.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts (`(int *)(puVar3 + lVar7 * 4 + lVar1 + 0x10)`). Pointer arithmetic common without bounds checking. Function pointer types partially recovered (`_func_int_int*`).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated "Unresolved local var" comments present. No semantic explanations of algorithm purpose, no API contracts, no decompilation confidence indicators beyond warnings.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition visible. Memory allocation patterns, linked list traversal, tree operations all appear as raw pointer arithmetic without idiom identification.

**14. Anomaly Indication:** Score: 6/10  
Justification: Does flag issues ("WARNING: Could not recover jumptable", "Unresolved local var"). However, warnings are generic without actionable guidance for manual analysis.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `n > 0` checks not documented as contracts). No postcondition inference. Error conditions not explained semantically.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`global_counter`, `file_scope_static`) referenced but no usage map. Callback relationships partially recovered.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output with no logical grouping. Static functions not clearly distinguished from public API. No header file or module boundaries.

**18. Build System Compatibility:** Score: 7/10  
Justification: Should compile with standard libraries (malloc, free, printf available). External dependencies (`extern_global_var`, `extern_function`) properly declared. Some undefined types need resolution.

**Overall Readability Score:** 4.8/10

### Critical Defects:
- **Variable names completely lost**: `iVar1`, `piVar2`, `puVar3` provide zero semantic information; should infer from usage (e.g., `arr`, `result`, `sum`)
- **Test functions optimized away**: Actual function calls replaced with hardcoded constants (`local_vars(5)` → `0x14`), losing behavioral verification capability
- **Magic numbers in hex**: Simple values like `42` shown as `0x2a`, `20` as `0x14`, obscuring intent
- **Point struct lost**: Structure access `p.x + p.y` decompiled to `x * 3`, losing type semantics entirely
- **No contract documentation**: Null checks present but not documented as preconditions for API understanding

### Best Quality Aspect:
- **Build System Compatibility (7/10)** - External dependencies properly declared and code should compile with standard C libraries without extensive modification.

### Most Needs Improvement:
- **Assertion & Contract Recovery (2/10)** - No precondition/postcondition documentation despite numerous null checks and bounds validations in source code.

### Top 3 Recommendations for Decompiler Improvement:
1. **Infer variable names from usage patterns** - Analyze how variables are used (loop counters, accumulators, pointers) to generate semantic names like `i`, `sum`, `arr` instead of `iVar1`, `iVar5`, `piVar2`
2. **Preserve function call structure in test code** - Avoid constant folding that eliminates actual function calls; maintain call graph for behavioral analysis
3. **Add contract documentation for null checks** - When null pointer checks are detected, generate `@requires: ptr != NULL` comments to document API contracts
