# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `uVar2`, `puVar4`) with zero semantic meaning. Original source had meaningful names like `x`, `arr`, `n`, `sum` that indicated purpose.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions preserved original names (`process_char`, `array_1d_stack`, `struct_simple`), but helper functions use mechanical names (`FUN_00100620`, `FUN_0010077c`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers throughout without explanation (`0x20`, `0x41`, `0x19` in `process_char`). Test functions show constant-folded values (`printf("...%d\n",0xf)`) instead of computed results.

**4. Noise Reduction:** Score: 5/10  
Justification: Moderate compiler artifacts present including redundant casts and intermediate variables, but not overwhelming. Some functions like `process_int` remain clean.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple functions readable, but array functions (`array_1d_stack`, `array_3d`) use heavily unrolled loops with 8+ accumulators. Floating point functions (`__addtf3`) are extremely convoluted.

**6. Function Decomposition:** Score: 5/10  
Justification: Original function boundaries mostly preserved, but some inflated due to inlining. `array_3d` expanded to 100+ lines of pointer arithmetic instead of nested loops.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but complex functions have deep nesting and multiple exit points that disrupt flow.

**8. Redundancy Elimination:** Score: 3/10  
Justification: Severe loop unrolling redundancy in array functions. `array_1d_stack` has 8 separate accumulators (`iVar10` through `iVar17`) for SIMD-like operations that obscure the simple sum logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Basic types mostly preserved but significant degradation exists. `process_double` returns `undefined1[16]` instead of `double`. Pointer types often generic (`undefined8*`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - All struct definitions completely lost. `struct_simple` accesses fields via `param_1[0]`, `param_1[1]`, `param_1[2]` instead of `p->x`, `p->y`, `p->z`. No field names recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts present (`(int *)*param_1`, `(undefined4 *)`). Pointer arithmetic without bounds checking throughout array functions.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments except compiler warnings ("WARNING: Could not recover jumptable"). No algorithm explanations or parameter documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Loop unrolling not identified as optimization. Standard library calls (`printf`, `puts`) not distinguished from user code.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some warnings for indirect calls (`ptr_func_simple`, `struct_func_ptr`) but many issues not flagged. Floating point exception handling noted.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `arr != NULL`). No postconditions inferred. Null checks present but not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. No call graph information. Global variable usage (`completed_0`, `__dso_handle`) not explained.

**17. Module Organization:** Score: 3/10  
Justification: All functions in single output with no logical grouping. Original test structure (data types, arrays, pointers, composites) lost in flat listing.

**18. Build System Compatibility:** Score: 4/10  
Justification: External dependencies (`__libc_start_main`, `printf`, `__addtf3`) need stubs. Some undefined symbols. Would require moderate patching to compile standalone.

**Overall Readability Score:** 3.8/10

### Critical Defects:
- **Struct type information completely lost** - All composite types accessed via raw pointer arithmetic (`param_1[0]`, `param_1[1]`) instead of named fields
- **Variable names convey zero semantics** - `param_1`, `iVar1`, `uVar2` throughout makes tracking data flow difficult
- **Loop unrolling obscures simple logic** - `array_1d_stack` uses 8 accumulators for what was a 5-line sum loop
- **No documentation or comments** - Zero algorithm explanations, only compiler warnings present
- **Test functions constant-folded** - `printf` calls show pre-computed results instead of actual function calls, losing test semantics

### Best Quality Aspect:
- **Function Naming Quality (6/10)** - Core function names preserved from source (`process_char`, `array_1d_stack`, `struct_simple`), providing some semantic anchors

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions; all field access reduced to pointer arithmetic losing all semantic meaning

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement struct pattern recognition** - Detect consistent offset patterns (0, 4, 8 for 3-int struct) and suggest named struct types with field recovery from usage context
2. **Add variable name inference from usage** - Analyze how variables are used (loop counters, accumulators, pointers) to suggest semantic names like `sum`, `index`, `array_ptr`
3. **Recover high-level loop constructs from unrolled code** - Detect SIMD-style unrolling patterns and collapse back to readable for/while loops with comments noting optimization
