# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Simple functions preserve parameter names (`process_char(char c)`, `process_int(int x)`), but complex functions use generic names like `iVar1`, `uVar3`, `piVar1`, `pTVar3`. Many "Unresolved local var" comments indicate lost semantic information (e.g., `sum@[???]`, `i@[???]`).

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match original source exactly (`process_char`, `array_1d_stack`, `struct_simple`). A few runtime functions use generic names (`FUN_001006d0`, `FUN_0010087c`), but core logic functions are well-named.

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions show constant-folded values (`0x61`, `0x62` instead of `'a'`, `'b'`). Floating-point literals use hex representations (`0x400c000000000000`). Simple arithmetic literals are clear (`c + ' '`, `x * 2 + 1`).

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive "Unresolved local var" comments throughout. Compiler artifacts like `___stack_chk_guard`, `__printf_chk`, and stack canary checks pollute test functions. Optimized functions have 50+ temporary variables (`uVar1` through `uVar50` in `array_3d`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions have natural flow, but `-O3` optimization causes severe issues. `binary_tree_sum` converts recursive tree traversal into deeply nested do-while loops. `array_3d` unrolls all 125 iterations into explicit additions.

**6. Function Decomposition:** Score: 5/10  
Justification: `struct_array` completely FAILED to decompile. `binary_tree_sum` becomes a 400+ line monolith. Simple functions maintain appropriate granularity, but optimization destroys function boundaries.

**7. Code Linearization:** Score: 4/10  
Justification: `binary_tree_sum` has extreme nesting (8+ levels) with while loops inside do-while loops. `array_3d` has a single 100+ line return statement. Forward jumps and complex conditionals disrupt reading flow.

**8. Redundancy Elimination:** Score: 3/10  
Justification: Severe loop unrolling in `array_3d` creates 50+ `uVar` temporaries with repetitive patterns. `binary_tree_sum` duplicates traversal logic across multiple nesting levels. No abstraction of common subexpressions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved well (`char`, `short`, `int`, `long`). `long double` degrades to `undefined1 [16]`. Function pointers use generic typedefs (`_func_int_int *`). Pointer arithmetic uses explicit casts (`(undefined8 *)piVar1`).

**10. Composite Type Recovery:** Score: 6/10  
Justification: Struct field names preserved (`p->x`, `r->position.x`, `w->bounds.position.z`). Bitfields recovered via manual masking (`uVar1 & 1`, `uVar1 >> 1 & 3`). `struct_array` failed completely. Union access partially recovered.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts throughout (`(undefined8 *)`, `(long)arr`, `(void *)(long)size`). `volatile_access` loses volatile semantics (two reads collapsed to `*p + *p`). Pointer arithmetic uses explicit size calculations.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: "Unresolved local var" comments provide some debugging info. Warnings like "FAILED to decompile" and "Treating indirect jump as call" are helpful. No algorithm explanations or API contract documentation.

**13. Idiom Recognition:** Score: 4/10  
Justification: Linked list traversal pattern preserved but not identified. Tree recursion converted to iteration without pattern naming. No recognition of common algorithms (sum, traversal, search patterns).

**14. Anomaly Indication:** Score: 6/10  
Justification: Good warning messages for indirect calls ("Could not recover jumptable"). "FAILED to decompile" clearly marks `struct_array`. Stack canary checks flagged with "WARNING: Subroutine does not return".

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `arr != NULL`, `n > 0`). No postcondition inference. Error conditions not documented. Bounds checking logic present but not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible in test functions but constant-folded (showing results, not calls). Struct type references preserved. Callback relationships unclear (`ptr_func_simple` shows `_func_int_int *` without target resolution).

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in source-order grouping. Runtime functions (`_init`, `_start`, `_fini`) mixed with application code. No header file structure or API declarations.

**18. Build System Compatibility:** Score: 4/10  
Justification: Won't compile standalone - missing struct definitions (`Point3D`, `Rect`, `Widget`), undefined external functions (`__printf_chk`, `___stack_chk_guard`), platform-specific types (`long_double`, `undefined8`).

### Critical Defects:

- `volatile_access` loses volatile semantics (two memory reads collapsed to `*p + *p`)

### Best Quality Aspect: **Function Naming Quality (8/10)**
- Core function names preserved exactly from source, making basic navigation straightforward

### Most Needs Improvement: **Assertion & Contract Recovery (2/10)**
- No precondition/postcondition documentation despite many functions having implicit requirements (non-NULL pointers, positive sizes)

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover loop constructs from unrolled code** - Detect sequential array access patterns and reconstruct for/while loops instead of emitting 100+ line expressions
2. **Preserve volatile semantics** - Track volatile-qualified memory accesses and emit separate load statements even when values appear identical
3. **Add contract inference** - Analyze null checks and bounds validation to generate precondition comments (e.g., "requires: arr != NULL && n > 0")
