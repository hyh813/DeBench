# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are generic Ghidra auto-names (`param_1`, `iVar1`, `uVar2`, `lVar2`) with no semantic meaning. For example, `process_int(int param_1)` should use `x` or `value` to match source intent.

**2. Function Naming Quality:** Score: 7/10  
Justification: User functions preserve original names well (`double_value`, `process_char`, `struct_simple`), but runtime functions get generic names (`FUN_001006d0`, `FUN_0010083c`).

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions contain unexplained magic numbers (`0x61`, `0x62`, `0x400c000000000000` for float constants). Some literals like `' '` are readable, but floating-point hex representations obscure meaning.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`___stack_chk_guard`) appear in every test function. Runtime initialization code (`deregister_tm_clones`, `register_tm_clones`) adds noise but is clearly marked.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10

**6. Function Decomposition:** Score: 5/10  
Justification: Basic functions are appropriately sized, but floating-point implementations are monolithic (500+ lines). Test functions lost structure entirelyâ€”replaced with printf calls and hardcoded values.

**7. Code Linearization:** Score: 6/10

**8. Redundancy Elimination:** Score: 5/10  
Justification: Floating-point functions repeat similar bit manipulation patterns extensively. Test functions eliminate redundancy by hardcoding values, losing actual computation logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===
