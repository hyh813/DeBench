# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`iVar1`, `uVar2`, `auVar5`) with "Unresolved local var" comments. Original names like `temp1`, `sum`, `count` are completely lost, making semantic tracking difficult.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names preserved accurately from source (`sequential_ops`, `single_if`, `nested_if_deep`). This is the decompiler's strongest lexical feature, maintaining original intent.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound (`0xfffffff8`, `0x12345678`, `0x3e9`) without explanation. Some constants like return values are clear, but optimization-derived literals obscure meaning.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: vector operations (`auVar5 [16]`), unresolved locals, and multiple "WARNING: Removing unreachable block" comments pollute simple functions.

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Critical failures: `single_if` uses bit shift `x << (0 < x)` instead of conditional, `loop_continue` becomes 100+ lines of vectorized code. At `-O0` this transformation is unacceptable.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved correctly, but test functions (`test_control_flow_l1`) are constant-folded losing actual call structure. Helper functions remain separate appropriately.

**7. Code Linearization:** Score: 5/10  
Justification: Simple functions read linearly, but complex ones (`goto_backward`, `recursion_factorial`) have extreme non-linearity with nested loops and vector operations disrupting flow.

**8. Redundancy Elimination:** Score: 3/10  
Justification: Massive redundancy in optimized functions: `recursion_factorial` has 200+ lines of vector operations for a 5-line recursive function. Loop unrolling not recognized or commented.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: All integers are generic `int`/`uint` without size specification. Function pointers use `_func_int_int` typedef but many show "Could not recover jumptable" warnings indicating type loss.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Arrays shown as pointer arithmetic (`*(int *)(&DAT_00103280 + (long)op * 4)`). No struct recovery visible; matrix operations use raw memory access patterns.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe patterns: function pointer calls with "Treating indirect jump as call" warnings, implicit casts between signed/unsigned, no const-correctness preserved.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated warnings present ("WARNING: Globals starting with '_' overlap"). No algorithm explanations, no parameter documentation, no semantic recovery comments.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition: state machines not identified, loop patterns not labeled, Duff's device shown as raw switch/goto without explanation of the optimization technique.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some warnings present ("Unresolved local var", "Could not recover jumptable") but inconsistent. Critical semantic errors (wrong logic in `nested_if_2`) not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: Zero precondition/postcondition documentation. Functions like `duffs_device` requiring `n>0` have no contract indication. Error conditions not documented.

---

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Callback patterns (`process_with_callback`) show function pointer usage but no call graph documentation.

**17. Module Organization:** Score: 5/10  
Justification: All 72 functions in single output with no logical grouping. Test functions mixed with implementation. No header file or API boundary indication.

**18. Build System Compatibility:** Score: 6/10  
Justification: Code appears syntactically valid but has architecture mismatch (original ARM64, decompiled x86). External dependencies (`__libc_start_main`, `_setjmp`) need stubs.

**Overall Readability Score: 4.6/10**

### Critical Defects
- **Architecture mismatch**: Original compiled for ARM64, decompiled as x86 - fundamental binary analysis error
- **Semantic corruption**: `nested_if_2` returns wrong values (logic transformed incorrectly from nested conditionals)
- **Excessive optimization artifacts**: Simple 5-line `recursion_factorial` becomes 200+ lines of vector operations despite `-O0` compilation
- **Variable name loss**: All local variables generic (`iVar1`, `uVar2`) with "Unresolved local var" comments
- **Test function constant-folding**: `test_control_flow_l1` shows hardcoded return values instead of actual function calls, losing verification capability

### Best Quality Aspect
- **Function Naming Quality (8/10)** - Original function names preserved accurately, making function identification straightforward despite other issues.

### Most Needs Improvement
- **Assertion & Contract Recovery (2/10)** - Zero documentation of preconditions, postconditions, or error conditions, making safe usage impossible without source reference.

### Top 3 Recommendations for Decompiler Improvement
1. **Preserve -O0 compilation semantics**: When debug symbols exist (`-g` flag), avoid applying optimization recovery transformations that corrupt simple control flow patterns.
2. **Recover variable names from debug info**: DWARF debug information contains original variable names that should be extracted rather than showing "Unresolved local var".
3. **Add semantic confidence indicators**: Flag functions where decompiled logic may differ from source (e.g., `nested_if_2` has incorrect conditional transformation) with uncertainty warnings.
