# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`param_1`, `iVar1`, `local_38`, `piVar2`) obscuring semantic meaning. Simple functions preserve original names (`a`, `b`, `x`, `y`) but complex functions like `varargs_func` lose all meaningful variable names.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`cdecl_func`, `varargs_func`, `test_calling_conventions`). However, helper functions use mechanical names (`FUN_00101020`, `varargs_func.cold`) and duplicate definitions exist.

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions are constant-folded showing hex literals (`0xf`, `0x32`, `0x2a`) instead of original computations. Magic numbers replace readable expressions like `cdecl_func(5, 10)` → `return 0xf`.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__stack_chk_fail()`), frame pointer operations (`in_FS_OFFSET`), and "Unresolved local var" comments clutter output. These compiler artifacts obscure actual logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple functions have natural flow, but varargs functions show complex nested loops that don't match source `for` loop structure. At `-O0` this restructuring is unexpected and reduces clarity.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are properly separated but duplicate definitions exist (`varargs_func` appears twice at different addresses). Test functions are over-optimized losing call relationships.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom but varargs functions have complex nested `do-while` loops with forward jumps disrupting flow. Stack check branches add non-linearity.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Duplicate function bodies for `varargs_func` and `param_varargs`. Test functions constant-folded losing original logic entirely, making verification impossible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `char*`, `double`) but varargs handling shows type confusion with `undefined8` parameters. Function pointers use generic `code*` instead of proper signatures.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Structs `LargeStruct`, `SmallStruct`, `BigStruct` recovered with field access (`s.data[0]`, `p->x`). Field names preserved from debug info which is commendable.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe casts in `param_complex_cast` (`*(int *)p`). Stack canary checks add runtime safety but are compiler artifacts not source semantics.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments are auto-generated warnings ("WARNING: Unknown calling convention", "Unresolved local var"). No semantic comments explaining algorithm purpose or logic.

**13. Idiom Recognition:** Score: 4/10  
Justification: Varargs patterns not identified as such despite `va_list` type present. No recognition of calling convention patterns or parameter passing idioms.

**14. Anomaly Indication:** Score: 5/10  
Justification: Warnings present for calling conventions and unresolved variables but generic. No specific guidance on what manual analysis is needed.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or contracts recovered. Null checks present but not documented as API requirements.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but test functions constant-folded losing call relationships. Cannot trace which functions call which from decompiled output alone.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output with no logical grouping. Related test functions not grouped with their implementation counterparts.

**18. Build System Compatibility:** Score: 6/10  
Justification: External dependencies (`__printf_chk`, `puts`, `strlen`) need linking. Some functions reference undefined globals (`handle1_1`, `DAT_00102008`).

**Overall Readability Score:** 5.3/10

### Critical Defects:
- **Test functions constant-folded**: `call_cdecl()` returns `0xf` instead of showing `cdecl_func(5, 10)` call, losing semantic meaning
- **Variable names generic**: `param_1`, `iVar1`, `local_38` obscure purpose in complex functions like `varargs_func`
- **Duplicate function definitions**: `varargs_func` appears twice at different addresses with different implementations
- **Varargs handling broken**: Complex stack manipulation doesn't match simple source `va_arg` loop pattern
- **No semantic comments**: Only auto-generated warnings, no explanation of algorithm purpose or calling conventions

### Best Quality Aspect:
- **Composite Type Recovery (7/10)** - Struct definitions (`LargeStruct`, `SmallStruct`, `BigStruct`) recovered with field names preserved from debug info, enabling readable field access like `s.data[0]` and `p->x`.

### Most Needs Improvement:
- **Assertion & Contract Recovery (2/10)** - No preconditions, postconditions, or API contracts documented; null checks exist but aren't identified as requirements.

### Top 3 Recommendations for Decompiler Improvement:
1. **Preserve call relationships in test functions** - Avoid constant-folding that hides function call patterns; show actual calls even when arguments are constants
2. **Infer variable names from usage context** - Variables used in loops should be named `i`, `count`, `sum` based on pattern recognition
3. **Add semantic comments for calling conventions** - When `__attribute__((cdecl))` or similar is detected, add comments explaining parameter passing mechanism
