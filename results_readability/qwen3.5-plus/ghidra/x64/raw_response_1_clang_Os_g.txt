# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`iVar1`, `uVar2`, `uVar5`-`uVar18` in `loop_continue`); original names like `temp1`, `sum`, `count`, `i`, `j` are lost with "Unresolved local var" comments indicating recovery failure.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names are well-preserved (`sequential_ops`, `single_if`, `nested_if_2`, `duffs_device`); only a few generic names like `FUN_00101020` appear for startup code.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x3e9` (1001), `0x65` (101), `0x80000000` appear without explanation; test functions show hex constants (`0x15`, `0x2d`) instead of decimal values from source.

**4. Noise Reduction:** Score: 5/10  
Justification: Frequent "Unresolved local var" comments, DW_OP_reg references (`DW_OP_reg0(RAX)`), and decompiler warnings ("Could not recover jumptable") clutter the output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions recover well (`sequential_ops`, `ternary_op`), but complex loops become unreadable (`loop_continue` uses 9+ temp variables with bit manipulation); `single_if` uses bit-shift trick `x << (0 < x)` instead of clear conditional.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly; helper functions (`double_value`, `triple_value`) remain separate; however, test functions are constant-folded losing actual call structure.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `duffs_device` uses goto labels for switch fall-through and `loop_break` has explicit `LAB_001012e3` label disrupting flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Loop unrolling visible in `recursion_factorial` and `goto_backward` with vector operations (`auVar13`, `auVar14`); test functions show aggressive constant folding losing runtime behavior.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `uint`, `ulong`) but many generic types; function pointers partially recovered (`_func_int_int *f`); vector types shown as `undefined1 auVar13 [16]` indicating type confusion.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays referenced as unresolved (`int[5] arr@[???]`, `int[3][4] matrix@[???]`); no struct recovery visible; `jmp_buf` referenced but not properly typed as `__jmp_buf_tag`.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe casts present (`(ulong)(n - 2)`, `(int)lVar1`); function pointer calls have warnings ("Treating indirect jump as call"); no obvious buffer overflow patterns but type erasure present.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: "Unresolved local var" comments indicate failures but don't explain semantics; WARNING comments show uncertainty; no algorithm explanations or API contracts documented.

**13. Idiom Recognition:** Score: 4/10  
Justification: Switch fall-through in `duffs_device` recognized with labels; recursion optimized to loop (tail recursion visible); but loop patterns not recognized and no design pattern identification.

**14. Anomaly Indication:** Score: 7/10  
Justification: Good warnings for critical issues ("Could not recover jumptable", "Subroutine does not return" for longjmp); "Unresolved local var" clearly indicates recovery failures.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation; no postcondition inference; error conditions not documented; test expectations lost due to constant folding.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but constant-folded in tests; global variables referenced as `DAT_00102090`; callback relationships partially visible but no call graph documentation.

**17. Module Organization:** Score: 6/10  
Justification: All functions in single output; helper functions grouped together; test functions separate from implementation; no header file structure or namespace organization.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates compilation succeeds; external dependencies visible (`__libc_start_main`, `printf`); architecture mismatch (original ARM64, decompiled x86) is concerning.
