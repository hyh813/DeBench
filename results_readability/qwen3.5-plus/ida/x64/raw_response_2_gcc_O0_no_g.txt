# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a1`, `a2`, `v0`, `v1`, `v17`, `v28`) with zero semantic meaning. Original source had meaningful names like `x`, `c`, `arr`, `n`, `sum`, `i` that indicated purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source (`double_value`, `process_char`, `array_1d_stack`, `struct_simple`). Only runtime helpers use generic names (`sub_1020`, `sub_1030`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `65`, `90`, `32` (ASCII conversions) and array offsets (`40`, `100`, `12`) lack explanation. String literals referenced as opaque labels (`s`, `asc_3180`, `asc_32A2`).

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary code (`__readfsqword(0x28u)`) appears in every test function. Excessive type casts (`(unsigned int)`, `(__int64)`) and stack frame variables (`v17`, `v28`, `v36`, `v49`) add clutter.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loop structures mostly preserved as for/while. However, `enum_switch` uses if-else chains instead of switch statement. Generally readable execution flow.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries correctly preserved with appropriate granularity. No inappropriate inlining or fragmentation. Each function maintains single responsibility.

**7. Code Linearization:** Score: 7/10  
Justification: Code reads mostly top-to-bottom with minimal forward jumps. Loop structures are clear. Some stack variable initialization at function start disrupts flow slightly.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Some redundant type casts present throughout. No significant dead code observed. At -O0, no loop unrolling artifacts to manage.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Integer types degraded to `_DWORD`, `__int64` instead of `int`, `long`. Pointer types use `_DWORD *` instead of `int *`. Const/volatile qualifiers completely lost (e.g., `const_param`, `volatile_access`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - All structs lost. `struct_simple` accesses fields via `a1[1] + *a1 + a1[2]` instead of `p->x + p->y + p->z`. Bitfields accessed via bit operations (`(*a1 >> 3) & 7`). No struct/union definitions recovered.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts (`(__int64)`, `(unsigned int)`) throughout. Pointer arithmetic replaces proper array indexing. `void *` used where specific types expected (`ptr_void`, `opaque_handle_op`).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero meaningful comments. Only auto-generated function headers present. No explanation of logic, algorithms, or decompilation confidence. Original source had extensive documentation.

**13. Idiom Recognition:** Score: 4/10  
Justification: Common C idioms not recognized (string length calculation in `array_string`, array sum patterns). No design pattern identification. Algorithms appear as raw operations.

**14. Anomaly Indication:** Score: 5/10  
Justification: Failed decompilations noted at end (libc functions). No warnings about potential issues in recovered code. Stack canary code present but unexplained.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., null pointer checks). No postconditions inferred. Error conditions not documented. Original had clear test expectations.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls preserved but relationships undocumented. Global variable references unclear (`s`, `asc_3180`, `asc_32A2`, `asc_3452`). No call graph documentation.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in reasonable order matching source. Test functions grouped but not explicitly marked. No clear module boundaries or namespace structure.

**18. Build System Compatibility:** Score: 5/10  
Justification: Six functions failed to decompile (libc functions). Requires stubs for external functions. Some type definitions missing. Would need moderate patching to compile standalone.

**Overall Readability Score: 5.2/10**

### Critical Defects:
- **Struct types completely lost** - All composite types accessed via pointer arithmetic (`a1[1] + *a1 + a1[2]`) instead of field names (`p->x + p->y + p->z`)
- **Variable names entirely generic** - `a1`, `a2`, `v0`, `v1` provide zero semantic information about purpose
- **Const/volatile qualifiers lost** - `const_param` and `volatile_access` functions lose critical type semantics
- **Zero documentation** - No comments explaining logic, despite original source having extensive documentation
- **Magic numbers unexplained** - ASCII values (65, 90, 32) and array offsets (40, 100, 12) lack context

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Most original function names preserved accurately, making it easy to identify function purpose despite other degradation.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions forces all composite type access through opaque pointer arithmetic, severely impacting readability.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct definitions from access patterns** - Analyze consistent offset patterns (e.g., +0, +4, +8 for Point3D) to reconstruct struct layouts with field names
2. **Infer variable names from usage context** - Variables used in loop counters should be named `i`, `j`, `k`; sum accumulators should be named `sum`; size parameters should be named `n` or `size`
3. **Preserve type qualifiers from debug symbols** - Since compiled with `-g`, const/volatile information should be recoverable and applied to parameter types
