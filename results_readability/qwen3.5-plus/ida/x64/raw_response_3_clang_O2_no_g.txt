# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`a1`, `a2`, `v0`, `v1`, `v2`) with zero semantic recovery. For example, `local_vars(int a1)` should have `x` as parameter name, and loop counters like `i` are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most original function names preserved (`local_vars`, `heap_basic`, `vla_stack`, `linked_list_heap`). However, some helpers became `sub_1020` and the metadata incorrectly claims Ghidra when header says IDA Pro.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation: `0x10`, `0x14`, `0x28` for allocations, `0xFFFFFFFFLL` for errors, `4294967294LL` instead of `-2`. No constant recovery attempted.

**4. Noise Reduction:** Score: 3/10  
Justification: Heavy SIMD artifacts (`_mm_load_si128`, `_mm_add_epi32`, `_OWORD`) clutter simple loops. Vectorized operations obscure original scalar logic. Type casts like `(__int64)` and `(_DWORD *)` everywhere.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions over-optimized: `stack_alias()` returns constant `20` instead of showing alias logic. `tree_heap_traversal()` returns `60` directly. Test functions show constant-folded printf values instead of actual function calls.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly. However, `create_tree_node` exists but `tree_heap_traversal` was constant-folded, breaking the logical relationship between helper and caller.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but SIMD loop unrolling creates complex nested patterns. Some functions use `goto LABEL_X` patterns instead of natural loop constructs.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Aggressive loop unrolling creates apparent redundancy. Same SIMD pattern repeated across `heap_basic`, `heap_array`, `memory_leak` without abstraction. Test functions duplicate constant values instead of calling actual functions.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Generic types dominate (`_DWORD`, `_QWORD`, `__int64`). Original `int`, `char*`, `size_t` lost. Function pointers degraded: `set_global_callback(__int64 a1)` should be `int (*f)(int)`. Signedness often incorrect.

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure**: `Point` struct completely lost—`local_struct` shows `3 * a1` instead of `p.x + p.y`. `Node` struct lost—`heap_nested` uses `_QWORD *` and raw offsets (`v2[1]`). `TreeNode` not recovered at all.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts: `(__int64)a1`, `(_DWORD *)v0`, `(char *)&v17`. Pointer arithmetic on void pointers. `dangling_pointer` returns `*v1` after free with no warning about undefined behavior.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: **Zero meaningful comments**. Only auto-generated type annotations. No explanation of SIMD operations, no algorithm descriptions, no decompiler confidence indicators. Original source had extensive comments; none recovered.

**13. Idiom Recognition:** Score: 2/10  
Justification: Common patterns unrecognized: malloc/free pairs not identified as memory management, linked list traversal not named, tree structure not detected. SIMD vectorization not explained as optimization artifact.

**14. Anomaly Indication:** Score: 3/10  
Justification: Some failures noted at bottom ("FAILED to decompile: free"), but critical anomalies silent: `dangling_pointer` use-after-free not flagged, `double_free` logic simplified away, `heap_overflow` returns `0` instead of showing overflow.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions recovered (e.g., `a1 > 0` checks exist but not documented). No postconditions. Error returns (`0xFFFFFFFFLL`) not explained. Null checks present but not annotated as safety contracts.

---

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. `double_value` referenced in `test_static_global` as `global_func_ptr = double_value` but callback mechanism not explained. Global variables (`global_counter`, `static_local_counter`) appear without declaration context.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in reasonable order matching source. However, test functions interleaved with implementation. No file/section organization indicating stack vs. heap vs. static memory test groups.

**18. Build System Compatibility:** Score: 4/10  
Justification: External functions listed as "FAILED to decompile" at bottom but not properly declared. Missing includes (`<stdio.h>`, `<stdlib.h>`). Would require significant stubbing to compile standalone.

**Overall Readability Score:** 4.2/10

### Critical Defects:
- **Struct types completely lost**: `Point`, `Node`, `TreeNode` structures not recovered—all pointer arithmetic
- **Test functions constant-folded**: `printf` statements show computed values (e.g., `20`, `10`, `15`) instead of actual function calls, destroying test semantics
- **Zero semantic comments**: Original source had extensive documentation; decompiled code has none
- **Variable names all generic**: `a1`, `v0`, `v1` provide no semantic information about purpose
- **SIMD optimization artifacts obscure logic**: Vector operations (`_mm_load_si128`, `_mm_add_epi32`) make simple loops unreadable

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Most original function names preserved accurately, allowing identification of test purposes

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any explanatory comments despite complex optimization artifacts requiring explanation

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover composite types from usage patterns**: Analyze struct field access patterns (`v2[0]`, `v2[1]`) to reconstruct `Point {x, y}` and `Node {data, next}` types with field names
2. **Preserve function call semantics in test code**: Avoid constant-folding test functions; maintain actual function calls to show test structure and enable debugging
3. **Add optimization-aware annotations**: When SIMD vectorization is detected, add comments explaining original scalar logic (e.g., "// Vectorized: original loop iterated i=0 to n")
