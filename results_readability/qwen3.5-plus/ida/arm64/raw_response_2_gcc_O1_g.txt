# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Simple functions preserve parameter names (`x`, `p`, `arr`), but complex functions use generic `v1`, `v2`, `v3` patterns. Test functions have variables like `v0`, `v1` instead of descriptive names like `result` or `sum`. Long double functions have 95+ generic variables (`v2` through `v95`).

**2. Function Naming Quality:** Score: 8/10  
Justification: Most original function names preserved accurately (`double_value`, `process_char`, `array_1d_stack`, `struct_simple`). Some helper functions remain generic (`sub_6D0`, `frame_dummy`). External library functions marked as "FAILED to decompile."

**3. Literal Clarity:** Score: 4/10  
Justification: Character literals converted to numbers (`'A'`→`65`, `'a'`→`97`). Magic numbers throughout (`0x19u`, `0x7FFF`, `0x8000000000000uLL`). Test functions show pre-computed results (`97`, `300`, `11`) instead of original expressions, obscuring the computation logic.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: `__fastcall` calling conventions, register comments (`// x0`, `// w0`, `// x1`). One function has warning: "local variable allocation has failed, the output may be wrong!" Long double functions contain excessive temporary variables from register spilling.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions maintain natural `if`/`return` patterns. Complex functions use goto labels (`LABEL_100`, `LABEL_132`, `LABEL_217`). Switch statements converted to nested if-else chains (`enum_switch`). Loops often use do-while instead of for constructs.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately sized for their purpose. Helper functions (`complex_callback`, `opaque_handle_create`) preserved separately. Test functions are large but cohesive. `__addtf3` and `__multf3` are excessively large (300+ lines each).

**7. Code Linearization:** Score: 6/10  
Justification: Simple functions read top-to-bottom naturally. Long double functions have many forward jumps disrupting flow. Test functions have non-linear initialization patterns with scattered variable assignments before function calls.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Test functions show constant folding (results pre-computed instead of function calls). Some redundant type casts present (`(unsigned __int8)`, `(_DWORD)`). Long double functions have significant variable redundancy from optimization artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types well recovered (`int`, `char`, `float`, `double`, `long double`). Platform-specific substitutions: `long`→`__int64`, `short`→`__int16`. Const correctness partially preserved (`const int *p`). Volatile qualifier preserved in signature but implementation simplified.

**10. Composite Type Recovery:** Score: 8/10  
Justification: Excellent struct recovery with field names (`Point3D`, `Rect`, `Widget`, `Node`, `Flags`, `UnionData`). Nested structures work correctly (`w->bounds.position.z`). Bit fields degraded to manual operations: `(*(_DWORD *)f & 1) + ((*(_DWORD *)f >> 1) & 3)`. Enum types preserved with symbolic names.

**11. Type Safety:** Score: 7/10  
Justification: Some unsafe casts present (`(unsigned __int8)`, `(_DWORD)`, `(int)(long)handle`). Pointer arithmetic generally safe. Void pointer casts appropriate for the use case. No obvious buffer overflow patterns introduced by decompiler.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Minimal comments - only auto-generated function headers (`/* Function: double_value @ 0x894 */`). No algorithm explanations. One warning comment about variable allocation failure. Original source comments completely lost.

**13. Idiom Recognition:** Score: 4/10  
Justification: No design patterns identified. Common patterns (linked list traversal, array summation, tree recursion) not explicitly noted. No algorithm naming or pattern hints provided.

**14. Anomaly Indication:** Score: 5/10  
Justification: Failed decompilations noted for external functions (`__libc_start_main`, `__printf_chk`). One warning about variable allocation. No other anomaly indicators for potentially unsafe patterns or unresolved indirect calls.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `arr != NULL`, `n > 0`). No postconditions inferred. No invariants recognized. Array bounds checks present in code but not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls preserved with original names. Callback relationships visible (`ptr_func_simple(double_value, 5)`). No explicit call graph documentation. Global variable references present but not well documented.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file. No header file separation. Static functions marked appropriately. No namespace or logical grouping beyond function order. Original section comments lost.

**18. Build System Compatibility:** Score: 4/10  
Justification: Seven external functions marked "FAILED to decompile" requiring stubs. Compiler-specific types (`__int64`, `__fastcall`, `__n128`). Would need significant work to compile standalone without library dependencies.

**Overall Readability Score: 5.7/10**

### Critical Defects
- **Test functions show constant folding**: Original function calls replaced with pre-computed values (e.g., `process_char('A')` result shown as `97`), losing semantic meaning of the computation
- **Long double functions extremely verbose**: `__addtf3` and `__multf3` have 300+ lines with 95+ generic variables, making them practically unreadable
- **Bit field operations degraded to manual bit manipulation**: `f->flag1` becomes `(*(_DWORD *)f & 1)`, losing type semantics
- **No comments recovered**: Original source had extensive documentation; decompiled code has only auto-generated headers
- **Character literals converted to numeric values**: `'A'`→`65`, obscuring the uppercase-lowercase conversion intent

### Best Quality Aspect
- **Composite Type Recovery (8/10)** - Struct and union types excellently recovered with accurate field names and nested access patterns preserved (`w->bounds.position.z`)

### Most Needs Improvement
- **Assertion & Contract Recovery (2/10)** - No preconditions, postconditions, or invariants documented despite many functions having implicit requirements (null checks, bounds validation)

### Top 3 Recommendations for Decompiler Improvement
1. **Preserve semantic literals**: Keep character literals as `'A'` instead of `65`, and avoid constant-folding test function calls that obscure original computation logic
2. **Add decompilation confidence annotations**: Mark uncertain reconstructions (bit fields, complex floating-point) with comments indicating potential accuracy issues
3. **Recover function contracts**: Analyze null checks and bounds validation to generate precondition documentation (e.g., `// requires: arr != NULL && n > 0`)
