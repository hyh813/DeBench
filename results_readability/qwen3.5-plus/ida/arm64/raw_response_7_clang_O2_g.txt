# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`v0`, `v1`, `v2`, `x`, `result`) with zero semantic recovery. For example, in `param_opaque_predicate`, variables `v1-v6` obscure their roles as predicate calculations (`p1`, `p2`, `p3` from source).

**2. Function Naming Quality:** Score: 9/10  
Justification: Excellent recovery - function names like `param_fake_branch`, `param_opaque_predicate`, `decrypt_string` match source exactly. Action-oriented naming preserved throughout.

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers like `0x5A` (encryption key), `0xDEADBEEF` appear without explanation. However, some constants are appropriately folded (e.g., `call_fake_branch` returns literal `10`).

**4. Noise Reduction:** Score: 7/10  
Justification: Moderate compiler artifacts present (`LOBYTE`, `__int64`, `__fastcall` calling conventions). Some redundant type casts but generally readable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Mixed results - `param_fake_branch` is constant-folded (returns `x` directly, losing the fake branch logic). `param_vectorized_loop` shows complex SIMD recovery with goto labels (`LABEL_9`, `LABEL_11`) instead of clean loops.

**6. Function Decomposition:** Score: 8/10  
Justification: Good granularity - functions are appropriately sized. `decrypt_string` properly separated from `param_string_encryption`. No monolithic functions observed.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom. Some forward jumps in complex functions (`param_vectorized_loop` has multiple `goto LABEL_X` patterns) but mostly manageable.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Aggressive constant folding eliminates some redundancy (e.g., `call_opaque_predicate` returns `20` directly). However, `param_string_encryption` and `call_string_encryption` have duplicated decryption logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 7/10  
Justification: Types mostly preserved (`int`, `char*`, `size_t`, `__int64`). Some degradation with generic `__int64` instead of `intptr_t`. Signed/unsigned distinctions maintained in most cases.

**10. Composite Type Recovery:** Score: 5/10  
Justification: The bitfield struct in `param_implementation_defined` is completely constant-folded to return `48` - no struct recovery visible. Array handling preserved but struct semantics lost.

**11. Type Safety:** Score: 6/10  
Justification: Several unsafe casts present: `(__sighandler_t)div_zero_handler`, `LOBYTE()` macros, and pointer arithmetic without bounds checking. No buffer overflow warnings despite source having intentional overflows.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no meaningful comments. Only auto-generated function headers with register assignments (`// w8`, `// x10`). No algorithm explanations or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition. The GCD algorithm in `param_opaque_predicate` is not identified. XOR decryption pattern in `decrypt_string` not documented. Tail call optimization not noted.

**14. Anomaly Indication:** Score: 5/10  
Justification: Good: "FAILED to decompile" notices for 11 library functions (`strlen`, `signal`, `printf`). Bad: No warnings about the fake branches being eliminated or constant folding changing semantics.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition documentation (e.g., `param_division_by_zero` doesn't note `x=0` triggers handler). No postcondition or return value guarantees documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 7/10  
Justification: Function call relationships clear. Global variables like `jmp_buffer`, `div_zero_caught` referenced correctly. `param_string_encryption_encrypted` external reference visible.

**17. Module Organization:** Score: 6/10  
Justification: Functions listed sequentially but no logical grouping by category (obfuscation vs. optimization vs. edge cases). No header file structure or API documentation.

**18. Build System Compatibility:** Score: 5/10  
Justification: 11 library functions failed to decompile requiring stubs. Would need significant patching to compile standalone. External dependencies not documented.
