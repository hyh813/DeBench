# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`v0`, `v1`, `v2`, `w8`, `x9`) with register hints instead of semantic names like `result`, `encrypted`, or `buffer` from the original source.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user functions preserved original names (`param_fake_branch`, `param_opaque_predicate`), but system functions are generic (`sub_730`, `init_proc`) and 11 library functions failed to decompile.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x5A` (XOR key), `8` (SIGFPE), `11` (SIGSEGV) appear without symbolic names; `0xDEADBEEF` from original is completely eliminated through optimization.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains register type annotations (`w8`, `x9`), stack frame markers (`[xsp+0h] [xbp-20h]`), and casting macros (`LOBYTE`, `HIBYTE`) that clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loop structures recovered (do-while for GCD), but `param_fake_branch` lost its fake branch conditions entirely due to constant folding, changing semantic meaning.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly, but caller functions like `call_fake_branch` are over-simplified to constant returns instead of actual function calls.

**7. Code Linearization:** Score: 7/10  
Justification: Code reads mostly top-to-bottom with appropriate loop constructs; no excessive goto usage, though some do-while patterns are slightly awkward.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `call_string_encryption` duplicates `param_string_encryption` logic instead of calling it; `test_obf_opt_edge` inlines decryption logic that should be a function call.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types preserved (`int`, `char*`, `size_t`) with calling conventions (`__fastcall`, `__cdecl`), but `unsigned __int8` used where `char` would be clearer.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Bitfield struct from `param_implementation_defined` completely lost (returns constant 48); only simple stack arrays recovered (`_BYTE v6[31]`).

**11. Type Safety:** Score: 6/10  
Justification: Multiple unsafe casts present (`(__sighandler_t)`, `(unsigned __int8*)`); pointer arithmetic in decryption loops lacks bounds checking annotations.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10

**13. Idiom Recognition:** Score: 4/10

**14. Anomaly Indication:** Score: 5/10

**15. Assertion & Contract Recovery:** Score: 2/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10

**18. Build System Compatibility:** Score: 3/10
