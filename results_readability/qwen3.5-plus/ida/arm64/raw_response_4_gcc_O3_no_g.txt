# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (a1, a2, v0, v1, v3, etc.) with zero semantic meaning recovered. For example, `varargs_func` parameters should be `count` and `args` based on usage pattern, but appear as `a1, a2...a9`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names preserved (cdecl_func, stdcall_func, varargs_func), but some have optimization suffixes (.constprop.0) and one remains as `sub_710`. Good verb preservation in test functions.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals referenced as `asc_1598`, `aCallL206` instead of inline text. Magic numbers like return values (15, 50, 6) are clear due to constant folding but lack context explanations.

**4. Noise Reduction:** Score: 5/10  
Justification: Excessive type casts throughout (`(unsigned int)`, `(unsigned __int64)`). Vector intrinsics in `func_struct_byval` (vaddq_s64, vdupq_n_s32) and stack canary code (`return v4 - _stack_chk_guard`) add noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple functions have natural flow, but `varargs_func` shows messy while/do-while loops from stack traversal. `ret_complex_expr` uses explicit boolean flags (`v3 = a1 <= a2`) instead of direct conditionals.

**6. Function Decomposition:** Score: 6/10  
Justification: Constant folding makes many `call_*` functions trivial single-return statements losing call relationships. `varargs_func` is ~40 lines of complex stack handling that could be abstracted.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with minimal forward jumps. No excessive goto usage, though varargs stack traversal has some non-linear patterns.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Nested `vaddq_s64` calls in `func_struct_byval` show redundant vector operations. Repeated type casts throughout. Constant-folded functions eliminate redundancy but lose semantic meaning.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `__int64`, `_DWORD` instead of precise `int32_t`/`int64_t`. Pointer types degrade to generic types. Function pointer types partially recovered (`unsigned int (__fastcall *a1)(_QWORD)`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: `LargeStruct` becomes `int64x2_t*` losing original struct definition. `SmallStruct` becomes `_DWORD*` with no field names. `func_struct_byval` parameter should be `LargeStruct s` not `int64x2_t *a1`.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(unsigned int)`, `(int*)`). Pointer arithmetic in varargs handling lacks bounds checking. `func_struct_byptr` null check exists but not documented as contract.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments beyond function signatures. No algorithm explanations, decompilation confidence indicators, or API contract documentation. Only "attributes: thunk" on one function.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns or algorithm patterns identified. Varargs handling isn't recognized as standard C variadic pattern. Vector operations not explained as SIMD optimization.

**14. Anomaly Indication:** Score: 4/10  
Justification: External functions marked "FAILED to decompile" at end, but no anomalies flagged within decompiled code itself. No warnings about potential undefined behavior in `call_order_dep`.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants documented. Null checks exist in code (`if (a1)`) but not documented as contracts. No error condition documentation.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. No call graph information. Global variable references unclear (`handle1_1`, `handle2_0` without context).

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output without logical grouping. No header file structure or namespace organization. Related test functions not grouped together.

**18. Build System Compatibility:** Score: 3/10  
Justification: 8 external functions marked "FAILED to decompile" with no stubs provided. Code won't compile standalone. No dependency documentation for libc functions.
