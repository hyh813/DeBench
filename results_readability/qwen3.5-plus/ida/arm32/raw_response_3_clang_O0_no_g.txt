# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`a1`, `v1`, `v2`, `ptr`) instead of semantic names from source (`x`, `arr`, `local`, `i`). For example, `local_vars(int a1)` should be `local_vars(int x)`, and `heap_basic` uses `ptr` instead of `arr`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user functions preserved original names (`local_vars`, `heap_basic`, `test_stack_memory`), but runtime functions are generic (`sub_588`, `init_proc`). The core test functions are well-named and match source intent.

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers appear without explanation (e.g., `0xFFFFFFF8` for alignment, `0x14u` for 20 bytes). Some constants are clear (`4 * a1` for `sizeof(int) * n`), but alignment masks obscure intent.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains compiler artifacts like `((unsigned int)v6 >> 31)` for sign extension, alignment calculations `((4 * v6 + 7) & 0xFFFFFFF8)`, and unnecessary intermediate variables (`v4 = a1` then use `v4`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: At -O0, control flow is mostly natural with proper for/while loops. However, VLA functions use complex pointer arithmetic instead of clean array indexing, and `stack_alias` has broken parameter handling.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity matches source well - each test function is separate. No inappropriate inlining or fragmentation. Helper functions like `create_tree_node` properly extracted.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom with appropriate early returns. Some functions have excessive stack variable declarations at top (`test_heap_memory` has 10+ temp variables) disrupting flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Sign extension patterns repeat (`(a1 + (a1 >> 31))`), alignment calculations duplicated across VLA/alloca functions. Array index calculations like `((2 * (v6 + ((unsigned int)v6 >> 31))) & 0xFFFFFFFD)` are verbose.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Significant type degradation - `int*` becomes `_DWORD*`, `Point*` becomes `_DWORD*`, `size_t` becomes `unsigned int`. Function `set_file_static(int *result)` should be `set_file_static(int val)`. TLS function shows `GLOBAL_OFFSET_TABLE_ = a1` which is type-confused.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Critical failure - `Point` struct completely lost. `heap_struct` accesses as `*ptr` and `ptr[1]` instead of `p->x` and `p->y`. `Node` struct in `linked_list_heap` accessed as `v5[1]` for `next` pointer. `TreeNode` similarly degraded.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts (`(_DWORD *)`, `(char *)`), pointer arithmetic on void pointers. `stack_alias()` signature broken (no parameters but should have `int *p1, int *p2`). `tls_access(void *a1)` should be `int val`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments explaining logic. Only header comment mentions decompiler. No documentation of VLA handling, alloca behavior, or why complex index calculations exist. Critical anomalies unexplained.

**13. Idiom Recognition:** Score: 4/10  
Justification: Common patterns (malloc/free pairs, linked list traversal) not identified. VLA and alloca patterns present but not labeled. Memory barrier inline assembly not explained.

**14. Anomaly Indication:** Score: 3/10  
Justification: `stack_alias` has broken signature with no warning. `tls_access` shows incorrect `GLOBAL_OFFSET_TABLE_` assignment without flagging. VLA stack pointer manipulation not marked as unusual.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks preserved but not documented as preconditions. No postcondition comments (e.g., "returns heap value at index n/2"). Error returns (-1, -2, -3) not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls preserved correctly. Global variables referenced but some with wrong names (`dword_140A4` instead of `global_point.y`). Callback relationship in `set_global_callback` clear.

**17. Module Organization:** Score: 7/10  
Justification: Functions grouped logically (stack, heap, static, memop tests). Test functions call appropriate sub-functions. However, all in single file without header separation.

**18. Build System Compatibility:** Score: 4/10  
Justification: 18 functions failed to decompile (marked `FAILED`). External dependencies (`extern_global_var`, `extern_function`) declared but undefined. Would require significant stubbing to compile.

**Overall Readability Score:** 5.3/10

### Critical Defects:
- **`stack_alias()` signature completely broken** - takes no parameters but source has `(int *p1, int *p2)`, making function unusable
- **`Point` struct type completely lost** - accessed as `ptr[0]`, `ptr[1]` instead of `p->x`, `p->y` across multiple functions
- **`tls_access` shows incorrect semantics** - `GLOBAL_OFFSET_TABLE_ = a1` is wrong; should be thread-local variable assignment
- **VLA/alloca index calculations obfuscated** - `((2 * (v6 + ((unsigned int)v6 >> 31))) & 0xFFFFFFFD)` instead of `v6/2`
- **18 external functions failed to decompile** - includes critical libc functions (malloc, free, printf, etc.)

### Best Quality Aspect: **Function Decomposition (8/10)**
- Function boundaries preserved correctly with appropriate granularity matching source structure

### Most Needs Improvement: **Comment Quality & Presence (2/10)**
- Virtually no documentation explaining complex patterns, anomalies, or semantic intent

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct types from usage patterns** - Detect consistent field access patterns (`ptr[0]`, `ptr[1]`) and propose struct definitions with field names
2. **Add semantic comments for complex expressions** - Annotate alignment calculations, sign extensions, and VLA stack manipulation with explanations
3. **Flag signature mismatches as anomalies** - When parameter count/types don't match call sites, explicitly warn users (e.g., `stack_alias` called with 2 args but decompiled with 0)
