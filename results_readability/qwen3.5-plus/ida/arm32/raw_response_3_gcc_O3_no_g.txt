# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v0`, `v1`, `a1`, `v2`, `v3`) with zero semantic meaning. In `heap_basic`, variables should be `arr`, `i`, `ptr`, `result` based on usage patterns, but remain as `v2`, `v3`, `v4`, `v5`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `local_vars`, `heap_basic`), but many are corrupted (`alloca_usage.constprop.0`) or generic (`sub_10598`). Test functions kept names but lost actual functionality.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers throughout without explanation (`0x14u`, `0x28u`, `0xCu` for struct sizes). Some values are clear from context (42, 100) but allocation sizes should be named constants.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: register comments (`// r0`, `// r3`), stack frame annotations (`[sp+0h] [bp-120h]`), BYREF markers, and type casts (`(_DWORD *)`, `*(_DWORD *)`) clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Original for-loops transformed into do-while with pointer arithmetic (`v4 += 4; v3 += 2; while (v3 != 2 * a1)`). `large_stack_frame` uses pointer increment instead of array indexing, obscuring the simple loop logic.

**6. Function Decomposition:** Score: 5/10  
Justification: Functions maintain reasonable size but test functions lost structure - `test_stack_memory` prints constants instead of calling actual functions. `stack_alias` returns constant 20 instead of performing aliasing logic.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but complex functions like `linked_list_heap` use goto labels (LABEL_10, LABEL_29) disrupting flow. Most simple functions are adequately linearized.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Aggressive optimization created semantic redundancy - `test_heap_memory` allocates/frees memory just to print pre-computed constants. Test functions compute at compile-time then print results, losing actual logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Types are generic (`int`, `_DWORD *`, `void *`, `__int64`). Signed/unsigned distinction lost. Function pointers degraded to `void *` (`global_func_ptr = (int (*)(void))double_value`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct types completely lost - `Point` and `Node` structures not recovered. `heap_struct` allocates 8 bytes without struct type, `heap_nested` uses `_DWORD *` instead of `Node *`, `create_tree_node` returns `_DWORD *` instead of `TreeNode *`.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe casts throughout (`*(_DWORD *)v0`, `(_DWORD *)v4`), pointer arithmetic without bounds checking, function pointers cast to `void *`. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only compiler-generated comments exist (register assignments, stack locations, "BYREF"). Zero semantic documentation explaining function purpose, algorithm, or non-obvious logic.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or algorithm patterns identified. Memory allocation patterns, linked list operations, and tree traversals not recognized or named. Raw pointer arithmetic obscures data structure intent.

**14. Anomaly Indication:** Score: 3/10  
Justification: Failed decompilations marked ("FAILED to decompile: __imp_malloc") but no warnings about undefined behavior (dangling pointer in `dangling_pointer`, double-free in `double_free`). No uncertainty indicators for indirect calls.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Null checks exist but aren't documented as contracts. No error condition documentation for allocation failures.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. `double_value` referenced by `set_global_callback` without clear callback relationship. Global variables used without usage documentation or call graph information.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single file with no logical grouping. Static functions not properly scoped (no file organization). No header file structure or namespace organization recovered.

**18. Build System Compatibility:** Score: 3/10  
Justification: 18 functions failed to decompile (library imports like `__imp_malloc`, `__imp_printf`). External dependencies unclear. Won't compile standalone without significant stubbing and library linking.

## Additional Observations

**Critical Metadata Issue:** The original source claims `-O0 -g` (debug, no optimization) but the decompiled binary header shows `arm32/3/3_gcc_O3_no_g` (O3 optimization, no debug). This explains the aggressive constant folding and function inlining that destroyed test function logic.

**Tool Attribution Confusion:** Metadata claims "Ghidra (12.0_PUBLIC)" but decompiled code header says "Decompiled by IDA Pro 9.1 with Hex-Rays" - indicates potential metadata inconsistency.

**Semantic Loss Severity:** The test suite's educational purpose is completely lost - functions that should demonstrate memory patterns now just print constants, making the decompilation useless for understanding the original intent.
