# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables use generic names like `v0`, `v1`, `v4`, `v5` throughout (e.g., `param_opaque_predicate`), obscuring semantic purpose compared to original's `result`, `p1`, `p2`, `a`, `b`, `temp`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved accurately (`param_fake_branch`, `param_opaque_predicate`), but startup functions use generic names (`sub_10514`, `init_proc`) and metadata mismatch (Ghidra vs IDA Pro header).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `90` (0x5A encryption key), signal numbers `8`/`11` lack explanation; constant folding in `call_*` functions hides original computations (e.g., `call_fake_branch` returns `10` directly).

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler artifacts present: register annotations (`// r0`, `// r1`), stack comments (`[sp-4h] [bp-4h]`), and confusing warning "positive sp value has been detected, the output may be wrong!" in `_start`.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Dead code elimination transformed `param_fake_branch` from conditional branches to simple `return x`; `param_opaque_predicate` has restructured logic with special `x == -1` case not in original; pointer arithmetic loops harder to follow.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved with reasonable sizes; helper `decrypt_string` recovered well, but some functions became trivial after optimization (`param_fake_branch` just returns `x`).

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `param_opaque_predicate` has multiple early returns disrupting flow; `param_vectorized_loop` uses do-while with pointer increments instead of clean for-loop.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Critical semantic loss: `param_buffer_overflow_heap` overflow loop replaced with meaningless countdown (`v3 = 33; do --v3; while (v3);`); constant folding hides original computation intent.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 6/10  
Justification: Some type changes (`param_string_encryption` returns `size_t` vs original `int`); generic `int` types used; `__fastcall` annotations present but unsigned conversions like `(unsigned __int8)` lack context.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Bitfield struct from `param_implementation_defined` completely optimized away (returns constant `44`); no struct definitions recovered in decompiled output despite original having explicit struct with named fields.

**11. Type Safety:** Score: 5/10  
Justification: Implicit casts throughout without explanation; pointer arithmetic in `param_vectorized_loop` (`v4 = a - 1; ++v4`) is hard to verify; signal handler types preserved but no safety annotations.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no semantic comments; only compiler annotations about registers/stack; original's extensive obfuscation technique explanations (永假条件，不透明谓词) completely absent.

**13. Idiom Recognition:** Score: 3/10  
Justification: Obfuscation patterns (fake branches, opaque predicates, instruction substitution) not identified; decompiler treated `param_fake_branch` as simple return, missing dead code elimination significance.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**14. Anomaly Indication:** Score: 4/10  
Justification: 15 functions marked "FAILED to decompile" (external libs); warning about stack value in `_start`; but semantic anomalies unflagged (buffer overflow becoming no-op, dead code elimination).

**15. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships undocumented; global variables (`jmp_buffer`, `segv_buffer`, `encrypted_0`) appear without context or usage explanation.

**16. Module Organization:** Score: 5/10  
Justification: Functions grouped but original's section comments (7.1 混淆与保护，7.2 编译器特有优化) lost; startup/teardown functions mix with application code without logical separation.

**17. Build System Compatibility:** Score: 4/10  
Justification: 15 external functions failed decompilation requiring stubs; won't compile standalone; no dependency documentation for library functions (signal, malloc, strlen, etc.).

**18. Optimization Awareness:** Score: 5/10  
Justification: Binary compiled at O1 (not O0 as metadata claims); aggressive optimizations eliminated obfuscation code without indication; decompiler doesn't distinguish optimization artifacts from original logic.
