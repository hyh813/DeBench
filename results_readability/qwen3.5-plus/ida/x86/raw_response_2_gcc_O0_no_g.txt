# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a1`, `a2`, `v0`, `v1`, `v2`) with no semantic meaning. Original source had meaningful names like `x`, `c`, `arr`, `n`, `sum`, `i` that indicated purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately from source (`double_value`, `process_char`, `array_1d_stack`, `struct_simple`). Some helper functions use generic names (`sub_1030`, `sub_1040`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers obscure intent: `64`, `90`, `32` in `process_char` instead of `'A'`, `'Z'`, character offset. Array offsets like `40 * i + 4 * i` in `array_2d_stack` lack explanation of stride calculations.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__readgsdword(0x14u)`) in every function, temporary variables for return values (`v0`, `v1`), and `__usercall`/`__cdecl` annotations add noise without semantic value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops recovered as for/while constructs correctly. However, `enum_switch` uses chained if-else instead of switch statement, and some conditionals are inverted from source.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity preserved well; each original function remains separate. No inappropriate inlining or fragmentation observed.

**7. Code Linearization:** Score: 7/10  
Justification: Code reads mostly top-to-bottom with clear loop structures. Some forward variable declarations (`v15`, `v16`, `v17`) disrupt flow slightly.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant type casts throughout (`(_DWORD *)`, `(_BYTE *)`). Stack frame setup code repeated in each function without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `long`→`int` (`process_long`), `long long`→`int` (`process_ll`), `float`/`double`→`long double` return types. `volatile` and `const` qualifiers completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Structs completely lost—`Point3D` accessed as `a1[0]`, `a1[1]`, `a1[2]` instead of `p->x`, `p->y`, `p->z`. No struct definitions recovered. Bitfields accessed via raw bit operations. Unions not recovered.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(_DWORD *)`, `(_BYTE *)`). Pointer arithmetic on generic int types. Function pointer signatures use generic `_DWORD *` instead of proper types.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments. Only auto-generated function headers. No algorithm explanations, API contracts, or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Critical bug: `doubly_linked_list` and `linked_list` have identical decompiled code despite different original implementations. Linked list traversal idiom not identified.

**14. Anomaly Indication:** Score: 4/10  
Justification: Library functions marked "FAILED to decompile". However, critical semantic loss (`volatile_access` optimized to `2 * *a1`) not flagged. Type mismatches not indicated.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `arr != NULL`). No postconditions inferred. No error condition documentation for functions like `array_complex_index` that return -1 on bounds error.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are clear but no call graph documentation. Global variable usage not documented. Callback relationships (e.g., `ptr_func_simple` with `double_value`) not explicitly noted.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in reasonable order matching source. However, single output file with no header declarations. Static functions not properly scoped.

**18. Build System Compatibility:** Score: 5/10  
Justification: External dependencies (printf, puts) declared but 5 functions "FAILED to decompile". Would require stubs and type corrections to compile standalone.

**Overall Readability Score:** 4.8/10

### Critical Defects:
- **Struct types completely lost**: `Point3D`, `Rect`, `Widget` all accessed as `_DWORD *` with array indexing instead of field names
- **Type precision degraded**: `long`→`int`, `long long`→`int`, `float`/`double`→`long double` return types
- **Volatile semantics lost**: `volatile_access` optimized to `2 * *a1`, losing the double-read guarantee
- **Identical code for different functions**: `doubly_linked_list` and `linked_list` have identical decompiled output
- **No semantic variable names**: All parameters are `a1`, `a2`, `v0`, `v1` with no indication of purpose

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Most original function names preserved accurately, making it easy to identify which decompiled function corresponds to which source function.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions; all composite types reduced to raw pointer arithmetic with array indexing.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct definitions from access patterns** - Detect consistent offset patterns (e.g., `a1[0]`, `a1[1]`, `a1[2]` always accessed together) and propose struct definitions with field names.
2. **Preserve type qualifiers** - Track `volatile` and `const` through analysis of memory access patterns and prevent optimization that removes semantic guarantees.
3. **Infer variable semantics from usage** - Analyze how variables are used (loop counters, accumulators, pointers) to suggest meaningful names like `i`, `sum`, `ptr` instead of `v0`, `v1`, `a1`.
