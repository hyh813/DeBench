# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `v2`, `result`, `temp`) with no semantic meaning. For example, `sequential_ops` uses no intermediate variables despite the source having `temp1`, `temp2`, `temp3`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core functions retained original names (`sequential_ops`, `recursion_factorial`, `double_value`), but helper functions are generic (`sub_1030`, `sub_1040`). Good recovery of test function names.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers present without explanation (`0x14u` for stack canary, `100 * (x + 1)` patterns). Some literals like array sizes `[5]`, `[8]` are clear from context.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__readgsdword(0x14u)` stack canary checks, register comments (`v0; // eax`), calling convention markers (`__cdecl`, `__usercall`), and inline assembly blocks.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control structures recovered well (if-else, loops, switches). However, `recursion_factorial` was transformed from recursive to iterative despite `-O0` compilation, and test functions use `goto` with labels.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized with no monolithic structures. Test functions are long due to printf statements but remain cohesive.

**7. Code Linearization:** Score: 5/10  
Justification: Test functions use `goto` with labels (`LABEL_4`, `LABEL_7`) disrupting linear reading. Most other functions read top-to-bottom appropriately.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundant patterns from optimization (repeated printf in test functions). No significant dead code, but some duplicated comparison logic in switch cases.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Only generic `int` types used without precision markers (`int32_t`, `uint64_t`). Function pointer types partially recovered but lack full signature detail.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays recovered (`matrix[3][4]`, `funcs[3]`, `ops[10]`) but no struct definitions visible. All composite types represented as raw arrays.

**11. Type Safety:** Score: 6/10  
Justification: Function pointers work correctly. Some unsafe casts present (`void *obj` to function pointers). No buffer overflow patterns detected.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**13. Idiom Recognition:** Score: 4/10

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies flagged ("FAILED to decompile: 7 functions", "bad sp value at call"). However, many potential issues go unmarked.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or error handling contracts documented. No parameter validation comments.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible in code but relationships not documented. No call graph or dependency information provided.

**17. Module Organization:** Score: 3/10

**18. Build System Compatibility:** Score: 3/10

### Most Needs Improvement:

1. **Implement semantic variable naming** - Analyze usage patterns to infer meaningful names (e.g., variables used in size checks should be named `size`, `count`, `length`)

3. **Recover function contracts** - Detect parameter validation patterns and document preconditions (e.g., `// REQUIRES: n > 0 for factorial calculation`)
