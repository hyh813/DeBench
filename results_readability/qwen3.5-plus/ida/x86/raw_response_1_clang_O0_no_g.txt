# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a1`, `a2`, `a3`, `v2`, `v3`) instead of semantic names like `temp1`, `sum`, `count` from source. For example, `sequential_ops` uses `a1, a2, a3` instead of `a, b, c`.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately (`sequential_ops`, `single_if`, `nested_if_deep`). Some helper functions have generic names (`sub_1030`, `sub_10A0`) but core functions match source.

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers like `0x7FFFFFF8` in `duffs_device` and `1000` in `infinite_loop` lack explanation. Most numeric literals are inferable from context.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack annotations clutter code (`[esp+0h] [ebp-4h]`, `BYREF`). Calling convention markers (`__cdecl`, `__usercall`) add visual noise without semantic value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control structures recovered well (if/else, switch, for/while). `switch_fallthrough` converted to nested if losing fallthrough semantics. `goto_forward/backward` successfully restructured.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries preserved correctly with appropriate granularity. No inappropriate inlining or fragmentation observed.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom naturally. `duffs_device` has complex label jumps (`LABEL_6`, `LABEL_8`) disrupting flow. `computed_goto` has inline assembly breaking linearity.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Temporary variables (`v2`, `v3`) create redundancy. `duffs_device` has significant code duplication from switch case expansion.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: All types degraded to generic `int` instead of `int32_t`, `size_t`. Type errors present: `op_or` takes `__int64` but should take two `int` parameters.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays recovered as raw buffers with `memcpy` (`dest[5]`, `v19[20]`). No struct recovery visible. Pointer arithmetic in `process_with_callback` (`*(_DWORD *)(a1 + 4 * i)`).

**11. Type Safety:** Score: 5/10  
Justification: Unsafe casts present (int to function pointer in `call_func_ptr_array`). Pointer arithmetic without bounds checking. Some signature mismatches (`op_or`, `state_done`).

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments in decompiled code. Only auto-generated function headers. No algorithm explanations or logic documentation compared to source's extensive comments.

**13. Idiom Recognition:** Score: 4/10  
Justification: Basic patterns recognized (loops, switches). Duff's device not identified as such. State machine pattern in `state_machine`/`fsm_func_table` not explicitly named.

**14. Anomaly Indication:** Score: 5/10  
Justification: Library functions marked "FAILED to decompile". `computed_goto` inline assembly noted. No warnings about `switch_fallthrough` semantics loss or type mismatches.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for `duffs_device`). No postconditions or error condition documentation recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls clear through direct invocation. Global variable usage (`jump_buffer`) not well documented. Callback relationships visible but not explicitly annotated.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single flat output. No logical grouping by control flow level (L1/L2/L3). Static functions properly marked but no namespace structure.

**18. Build System Compatibility:** Score: 4/10  
Justification: 7 functions "FAILED to decompile" (printf, memcpy, longjmp, etc.). Architecture mismatch (x86 decompilation vs arm64 source). Requires stubs for library functions.
