# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are generically named (`v1`, `v2`, `v3`, `result`) instead of semantic names like `temp1`, `sum`, `count` from source. For example, `loop_for_fixed` uses no loop variable name at all due to optimization.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most original function names preserved (`sequential_ops`, `single_if`, `recursion_factorial`). Some anonymous functions remain (`sub_1030`, `sub_2152`) but core logic functions are well-named.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x12`, `0x65`, `0x12345678` appear without explanation. Expressions like `10 * x + 10` are inferable but not documented as representing switch case values.

**4. Noise Reduction:** Score: 6/10  
Justification: Register variable annotations (`v1 // eax`, `v2 // ecx`) create visual noise. Some `__asm` blocks present. However, no excessive dead code from register spilling.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Critical issue - loops converted to closed-form formulas (`loop_for_fixed`), recursion converted to iteration (`recursion_factorial`), and `switch_fallthrough` becomes nested if-statements. These transformations obscure original intent despite being semantically equivalent.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries generally preserved. Helper functions created for `computed_goto` labels (`label1`, `label2`, etc.). No excessive inlining or fragmentation observed.

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom. `duffs_device` uses goto labels which is appropriate for this pattern. `goto_backward` shows complex loop reconstruction with multiple exit paths.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `loop_continue` shows duplicated logic blocks for handling remainder cases. `goto_backward` has complex unrolled factorial computation that obscures the simple recursive source.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types preserved correctly (`int`, `void*`, function pointers). Some unsigned/signed ambiguity (`unsigned int` vs `int` in comparisons). Function pointer signatures generally accurate.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Arrays handled well (`int arr[]`, `int matrix[3][4]`). No complex structs in this codebase to evaluate. Function pointer arrays (`funcs_1B6A[]`) correctly typed.

**11. Type Safety:** Score: 6/10  
Justification: Function pointer casts present (`(int (__cdecl *)(int))funcs_1B6A[idx]`). Some implicit conversions between signed/unsigned. No severe buffer overflow patterns visible.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: **Critical deficiency** - Decompiled code has virtually no comments while source had extensive documentation. No algorithm explanations, no parameter descriptions, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 5/10  
Justification: Some patterns recognizable (factorial computation, switch dispatch) but not explicitly identified. `duffs_device` pattern preserved but not labeled as such. State machine patterns not named.

**14. Anomaly Indication:** Score: 4/10  
Justification: "FAILED to decompile" notices for 6 external functions. No warnings about aggressive optimizations that transformed code structure. `computed_goto` handled but not flagged as compiler extension.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `n > 0` for `loop_for_fixed`). No postconditions or error condition documentation. Source test cases not recovered as assertions.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible and traceable. Global variables referenced (`jump_buffer`, `funcs_*`) but not well-documented. Callback relationships inferable but not explicit.

**17. Module Organization:** Score: 7/10  
Justification: Functions appear in logical order matching source. Test functions grouped together (`test_control_flow_l1/2/3`). Static helper functions properly scoped.

**18. Build System Compatibility:** Score: 5/10  
Justification: 6 external functions failed to decompile (`printf`, `puts`, `longjmp`, etc.). Would require stub implementations. Some compiler-specific constructs (`__cdecl`, `__usercall`) need handling.

### Overall Readability Score: **5.5/10**

### Critical Defects:
- **Variable names completely generic** (`v1`, `v2`, `result`) losing all semantic meaning from source (`temp1`, `sum`, `count`)
- **Control flow transformations obscure original intent** - loops became formulas, recursion became iteration without documentation
- **Zero comments** in decompiled output despite extensive source documentation
- **`loop_continue` decompilation is nearly unreadable** - complex optimized code doesn't reflect simple odd-number summation
- **No contract/precondition documentation** for functions with requirements (e.g., `n > 0`)

### Best Quality Aspect: **Function Naming Quality (8/10)**
- Original function names preserved accurately, making it easy to match decompiled code to source functions

### Most Needs Improvement: **Comment Quality & Presence (2/10)**
- Complete absence of documentation makes understanding optimized transformations nearly impossible without source comparison

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover variable names from debug symbols** - The `-g` flag was used during compilation but variable names weren't recovered
2. **Add decompilation confidence annotations** - Flag aggressively optimized code with warnings like "OPTIMIZED: loop converted to closed-form"
3. **Preserve source comments in debug builds** - When `-g` is present, extract and embed original comments in decompiled output
