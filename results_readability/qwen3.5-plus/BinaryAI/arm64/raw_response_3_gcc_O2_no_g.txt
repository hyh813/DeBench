# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`param_1`, `iVar1`, `puVar2`, `aiStack_30`) with no semantic meaning. Original names like `arr`, `local`, `out`, `p` are completely lost, making it difficult to track variable purpose across functions.

**2. Function Naming Quality:** Score: 6/10  
Justification: Some function names preserved correctly (`double_value`, `local_vars`, `heap_basic`, `test_stack_memory`), but many critical functions degraded to `FUN_00100c7c`, `FUN_0010120c`. Test harness functions retained names but implementation functions suffered.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers pervasive throughout (`0x2a` for 42, `0x14` for 20, `0x3c` for 60, `0xffffffff` for -1). Hex representation obscures decimal intent, and no constants are recovered despite `-g` debug symbols being available.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: stack canary checks (`___stack_chk_guard`), excessive type casts (`(undefined4 *)`, `(undefined8 *)`), and redundant pointer arithmetic. Debug compilation (`-O0 -g`) should have produced cleaner output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Original `for` loops transformed into `do-while` with pointer iteration (`piVar2 = piVar2 + 1`). Some functions incorrectly return `void` instead of values (e.g., `local_array`, `vla_stack`). Execution flow requires mental reconstruction.

**6. Function Decomposition:** Score: 5/10  
Justification: Test functions inline expected results instead of calling actual functions (`test_stack_memory` prints `0x14` instead of calling `local_vars(5)`). Some functions merged incorrectly, losing modular structure present in source.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but pointer arithmetic and stack operations create cognitive overhead. No excessive goto usage, but the low-level memory manipulation disrupts natural reading flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant type casts throughout (`(undefined4 *)`, `(long)`). Stack canary code duplicated in multiple functions. No dead code elimination despite debug build should have minimal optimization artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `undefined4`, `undefined8`, `void*` replace specific types. Return types incorrect (`void` instead of `int` for `local_array`, `address_of_local`). Original `Point*`, `Node*` types completely lost.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs (`Point`, `Node`, `TreeNode`) completely lost. Access becomes pointer arithmetic (`*(undefined4 **)(__ptr + 2)`). Field names absent, requiring manual reverse engineering to understand structure layout.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout. Pointer arithmetic without bounds checking. Function pointer types degraded to `undefined8`. Original type safety guarantees from C type system completely erased.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation explaining each test case purpose, expected values, and scenarios. No decompilation confidence indicators or algorithm explanations.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition evident. Memory allocation patterns, linked list traversal, tree operations all appear as raw pointer manipulation. No identification of common C idioms like "allocate-initialize-use-free" patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about potential issues. Functions with undefined behavior (`dangling_pointer`, `double_free`, `heap_overflow`) presented without any caution. No indication of decompilation uncertainty despite obvious type mismatches.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL` checks exist but aren't labeled as contracts). No postconditions or invariants. Error return values (`-1`, `-2`, `0xffffffff`) unexplained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables referenced (`global_counter`, `file_scope_static`) without context. Callback relationships (`global_func_ptr`) not explained.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output without logical grouping. Original source had clear sections (Stack, Heap, Static, Memory Ops) with section headers. No namespace or file organization preserved.

**18. Build System Compatibility:** Score: 5/10  
Justification: External functions declared but many use incorrect signatures. Would require significant patching to compile standalone. Dependencies on libc functions present but not documented.
