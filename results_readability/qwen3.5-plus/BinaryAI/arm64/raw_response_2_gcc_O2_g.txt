# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`param_1`, `param_2`, `iVar1`, `uVar1`, `lVar3`) with no semantic meaning. Original source had meaningful names like `x`, `c`, `arr`, `n`, `sum` that indicated purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved correctly (`process_char`, `process_short`, `test_data_types_l1`). Some helper functions have generic names (`FUN_001006d0`, `FUN_0010087c`) but core functionality is identifiable.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers throughout without explanation (`0xbfU`, `0x1a`, `0x3fc00000` for float constants). Floating-point literals shown as hex (`0x3fe0000000000000`) instead of decimal values. Test expected values shown as hex (`0x61`, `0x62`) instead of character literals.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks visible (`___stack_chk_guard`), redundant variable assignments (`local_8 = ___stack_chk_guard`), and floating-point exception handling (`__sfp_handle_exceptions`) add noise but don't completely obscure logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions have natural flow, but complex functions like `binary_tree_sum` transformed from recursive to deeply nested iterative loops with 20+ variables. Loop constructs use `do-while` with pointer arithmetic instead of clean `for` loops.

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes vary appropriately, but `binary_tree_sum` expanded from 6 lines recursive to 200+ lines iterative. Test functions have logic inlined that should remain as separate calls.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `binary_tree_sum` has extreme nesting (8+ levels) with multiple `while(true)` loops and `break` statements that disrupt linear reading.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Some loop unrolling visible in `array_3d` (unrolled inner loop summing 5 elements at once). No significant dead code, but some redundant variable assignments for stack canary checks.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Integer types mostly preserved, but floating-point types degraded (`undefined4` instead of `float`, `undefined8` instead of `double`). `long double` completely lost (uses `__multf3`/`__addtf3` library calls). Bool type lost (uses `uint`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - All struct types completely lost. `Point3D`, `Rect`, `Widget`, `Node`, `Graph` all accessed as `int*` with raw offsets (`param_1 + 8`, `param_1 + 0x14`). Union types lost (`UnionData` accessed as `float*`). Bitfield operations visible but not as struct bitfields.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts (`undefined4*`, `undefined8*`), function pointer types lost (`code*` instead of proper signatures), some functions have wrong return types (`ptr_const_ptr` returns `void` instead of `int`). Pointer arithmetic on generic `int*` instead of typed pointers.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: **No comments whatsoever** in decompiled output. No algorithm explanations, no decompilation confidence indicators, no documentation of non-obvious logic. Original source had extensive comments explaining each test case.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns identified. Data structure patterns completely obscured (linked list, binary tree, graph all look like generic pointer arithmetic). No algorithm patterns named or explained.

**14. Anomaly Indication:** Score: 4/10  
Justification: One anomaly marked (`UNRECOVERED_JUMPTABLE` in `ptr_func_simple`), but no indication of type uncertainty, struct loss, or potential issues. No warnings about the severe composite type recovery failure.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL`), no postconditions inferred, no invariants recognized. Original source had implicit contracts through type system that are now lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but not documented. No call graph information, global variable usage not documented (`DAT_00102d48`, `completed_0`). Callback relationships not clear (`ptr_func_simple` shows `UNRECOVERED_JUMPTABLE`).

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file with no logical grouping. No header file structure, no separation between implementation and test code. Original source had clear section organization with comments.

**18. Build System Compatibility:** Score: 4/10  
Justification: External functions declared but some missing definitions. Type mismatches would cause compilation errors (`undefined4` vs `float`). Would require significant stubs and type corrections to compile standalone.
