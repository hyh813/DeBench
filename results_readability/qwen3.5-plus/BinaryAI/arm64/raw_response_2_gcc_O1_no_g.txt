# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `uVar2`, `lVar3` with zero semantic meaning. Original source had meaningful names like `sum`, `len`, `current`, `head` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved well (`process_char`, `array_1d_stack`, `ptr_single`, `struct_simple`). However, some helper functions have generic names like `FUN_001006d0`, `FUN_0010083c`.

**3. Literal Clarity:** Score: 4/10  
Justification: Test functions contain hardcoded hex values (`0x61`, `0x62`, `0xb`, `0xf`) instead of actual function calls. Magic numbers like `0xbfU`, `0x1a` in `process_char` obscure the uppercase detection logic.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`___stack_chk_guard`) present in most functions. External function stubs (`__libc_start_main`, `__cxa_finalize`) clutter the output. Floating-point helpers (`__addtf3`, `__multf3`) are excessively verbose.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple loops converted to do-while with manual index management. Complex floating-point functions use excessive goto labels (`LAB_00102110`, `LAB_001022b8`) making flow impossible to follow.

**6. Function Decomposition:** Score: 5/10  
Justification: Individual computation functions are appropriately sized, but test functions are monolithic with all test cases combined. No recovery of helper functions from original source.

**7. Code Linearization:** Score: 5/10  
Justification: Simple functions read top-to-bottom reasonably. Test functions have excessive local variable declarations at top (`local_534`, `local_530`, etc.). Complex functions have many forward jumps disrupting flow.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Test functions have repeated `__printf_chk` call patterns. Floating-point functions (`__addtf3`, `__multf3`) have massive code duplication with nearly identical logic blocks.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Critical errors: `process_ld` returns `void` instead of `long double`, `process_bool` returns `uint` instead of `_Bool`. Pointer types often wrong (`undefined8*` instead of proper types). `volatile` qualifier completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct definitions recovered at all. All struct access done through pointer arithmetic (`param_1[1]`, `param_1[2]`, `param_1 + 8`). Union types, bitfields, and nested structures completely lost.

**11. Type Safety:** Score: 3/10  
Justification: Many unsafe casts present (`(int*)(long)handle`). Pointer arithmetic without type safety throughout. `void*` used extensively where specific types needed. Function pointer signatures degraded to `code*`.
