# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`param_1`, `local_10`, `uVar1`, `iVar1`) obscuring semantic meaning. Original source had meaningful names like `arr`, `head`, `current`, `new_node` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names recovered accurately (`double_value`, `local_vars`, `heap_basic`, `linked_list_heap`). A few generic names exist (`FUN_00101020`) but core test functions are well-preserved.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers prevalent: `0x2a` (42), `0x14` (20), `0x10` (16), `0x1000` appear without explanation. Original source used decimal literals that were more readable.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__stack_chk_fail()`) repeated in most functions. Frame pointer manipulation (`in_FS_OFFSET + 0x28`) adds visual noise. Redundant type casts throughout.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Critical failure - `Point`, `Node`, `TreeNode` structs completely lost. Accessed via pointer arithmetic (`local_38 + 2` for next pointer, `__ptr[1]` for struct fields). Original struct semantics obliterated.

- **Struct types completely lost**: `Point`, `Node`, `TreeNode` accessed via raw pointer arithmetic (`local_38 + 2`, `__ptr[1]`) instead of field names
- **Variable names generic**: `param_1`, `local_10`, `uVar1` obscure semantic meaning throughout all functions
- **No comments recovered**: Original source had extensive documentation explaining test scenarios and expected values; decompiled code has zero comments
- **Dangerous code not flagged**: `dangling_pointer`, `double_free`, `heap_overflow` not marked as undefined behavior or security concerns
- **Type precision degraded**: `undefined4`, `undefined8` used instead of proper C types; function pointers lose signature information

### Top 3 Recommendations for Decompiler Improvement

1. **Recover struct/union types from access patterns**: Analyze consistent field offsets across functions to reconstruct `Point {x, y}`, `Node {data, next}`, `TreeNode {data, left, right}` with meaningful field names

2. **Infer variable names from usage context**: Variables used in size calculations should be named `size`/`count`; pointer iteration variables should be named `ptr`/`current`; loop indices should be `i`/`idx`

3. **Flag dangerous patterns with warnings**: Add annotations for intentional undefined behavior (`// WARNING: use-after-free`), memory leaks (`// WARNING: no free()`), and buffer overflows (`// WARNING: off-by-one write`)
