# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables use generic decompiler names like `param_1`, `local_20`, `local_1c`, `iVar1` throughout. No semantic meaning is conveyed (e.g., `param_1` should be `x`, `local_38` should be `decrypted` buffer).

**2. Function Naming Quality:** Score: 8/10  
Justification: Original function names are preserved (`param_fake_branch`, `param_opaque_predicate`, `decrypt_string`). This is the strongest aspect - action-oriented names clearly reflect purpose.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x5a`, `0x41`, `0x42`, `0x2a` appear without explanation. The original `0xDEADBEEF` constant is completely lost. XOR key `0x5a` should be named `DECRYPTION_KEY`.

**4. Noise Reduction:** Score: 5/10  
Justification: Library stub functions (`free`, `strlen`, `printf`) that just call themselves add no value. Stack canary checks (`in_FS_OFFSET`, `__stack_chk_fail`) clutter every function with stack protection code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: CRITICAL: `param_fake_branch` completely loses both fake branch conditions - returns `param_1` directly instead of preserving the `if (x*x < 0)` and `if (strlen(str) < 0)` checks. Other functions have reasonable control flow.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are appropriately sized and match source structure. However, `param_link_time_optimization` doesn't return the value from `lto_target_func`, and `call_*` functions don't return values when source does.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally. Loop structures recovered as `for`/`while`. Early returns are clear. No excessive goto usage observed.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Library stub functions are completely redundant (each just calls external function). `param_implementation_defined` optimized to constant `0x2f` losing all original logic about char signedness, bitfields, etc.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Uses Ghidra-specific types `undefined4`, `undefined8`, `ulong` instead of standard C types (`int`, `int32_t`, `size_t`). `param_vectorized_loop` uses `long` for array pointers instead of `int*`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: No struct definitions recovered. `__jmp_buf_tag` referenced but not defined. The bitfield struct in `param_implementation_defined` is completely lost (function returns constant). Arrays handled as pointer arithmetic.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe casts: `(long)param_1`, `(undefined *)0x0`, `(undefined4 *)`. `param_null_pointer_deref` takes `undefined4 *` instead of `int *`. Pointer arithmetic without proper typing throughout.

**13. Idiom Recognition:** Score: 2/10

**14. Anomaly Indication:** Score: 2/10

**15. Assertion & Contract Recovery:** Score: 2/10
