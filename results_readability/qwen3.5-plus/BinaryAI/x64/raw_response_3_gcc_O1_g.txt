# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`param_1`, `iVar1`, `uVar2`, `pvVar1`, `plVar1`) with no semantic meaning. Original source had meaningful names like `arr`, `head`, `current`, `sum`, `new_node` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately from source (`double_value`, `local_vars`, `heap_basic`, `linked_list_heap`). This is the decompiler's strongest aspect—original function names survived compilation and were recovered.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers appear as hex without explanation (`0x2a` for 42, `0x14` for 20, `0xf` for 15). String literals referenced as `&DAT_00103004` instead of actual content. Original source had clear decimal constants.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__stack_chk_fail()`) appear in nearly every function. Frame pointer manipulation (`in_FS_OFFSET`, `local_10`) clutters code. External function wrappers add unnecessary indirection.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic loops recovered as do-while constructs, but some functions over-simplified (`local_vars` returns `param_1 * 2 + 10` directly, losing step-by-step logic). `vla_stack` has overly complex loop structure not matching source.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved but semantic loss evident. `create_tree_node` has wrong return type (void instead of `TreeNode*`). Some functions collapsed intermediate computations that source showed explicitly.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom with logical flow. Stack canary checks at function ends create minor disruption. No excessive goto usage; loop structures mostly linear and traceable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant type casts throughout (`(undefined4 *)`, `(long)`). Stack canary checks repeated in every function. External function wrappers (e.g., `malloc` wrapper calling `malloc`) add unnecessary layers.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation—`undefined4`, `undefined8`, `ulong` replace specific types. Original `size_t`, `Point*`, `Node*` lost. Function pointer `global_func_ptr` becomes `undefined8` instead of `int (*)(int)`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure—no struct definitions recovered (Point, Node, TreeNode all missing). All struct access via pointer arithmetic: `*(int **)(__ptr + 2)` instead of `->next`. Critical semantic loss.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts pervasive (`(undefined4 *)`, `(long)`, `(void *)`). Pointer arithmetic on void* throughout. No const-correctness preserved. Function pointer calls lack proper type signatures.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition—linked list creation/traversal not identified, tree operations not recognized, malloc/free pairs not marked as memory management. All idioms appear as raw operations.

**16. Cross-Reference Clarity:** Score: 5/10

**17. Module Organization:** Score: 6/10

**18. Build System Compatibility:** Score: 4/10
