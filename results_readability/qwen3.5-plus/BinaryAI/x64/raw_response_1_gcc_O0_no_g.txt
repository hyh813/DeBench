# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic patterns like `param_1`, `param_2`, `local_c`, `local_10` instead of semantic names. Original source had `temp1`, `temp2`, `sum`, `count` which were more meaningful.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source (`sequential_ops`, `single_if`, `nested_if_deep`). Some external wrappers have generic names (`FUN_00101020`) but core functions are well-named.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers appear in hex (`0xffffffff` for -1, `0x14` for 20, `0x3e9` for 1001). Original source used decimal literals that were clearer for human reading.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`in_FS_OFFSET`, `__stack_chk_fail`) add noise to simple functions like `loop_break`. External function wrappers for `printf`, `puts` add unnecessary indirection.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most loops recovered as proper `for`/`while` constructs. Some switch statements converted to if-else chains (`switch_small`, `switch_default`). Duff's device appropriately uses goto labels for fall-through.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity matches source well. Helper functions (`op_add`, `op_sub`, `double_value`) preserved as separate functions. No excessive inlining or fragmentation observed.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally. `switch_fallthrough` and `duffs_device` use goto labels appropriately for their patterns. Some forward jumps disrupt flow in `loop_break`.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant type casts throughout (`(ulong)`, `(int)`, `(long)`). Stack canary comparisons add redundancy (`if (*(long *)(in_FS_OFFSET + 0x28) == *(long *)(in_FS_OFFSET + 0x28))`).

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Generic types dominate (`undefined8`, `undefined4`, `undefined [16]`). Some return types incorrect (`if_else` returns `bool` instead of `int`). Pointer types sometimes lost (`int*` becomes `int` in `infinite_loop`).

**10. Composite Type Recovery:** Score: 5/10  
Justification: Arrays recovered as local variables (`local_28[4]`, `local_68[10]`). No struct/union definitions visible. Function pointer arrays recovered but with generic types (`code *local_28[3]`).

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(ulong)uVar1`, `(int)local_c`). Function pointer calls use generic `code *` type. Stack canary checks indicate potential safety concerns but no explicit warnings.
