# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are generically named `param_1`, `param_2`, `iVar1`, `uVar1` instead of semantic names like `temp1`, `sum`, `count` from source. `loop_break` uses meaningless `iStack_28`, `local_24` obscuring the array purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source (`sequential_ops`, `nested_if_2`, `duffs_device`). Helper functions like `op_add`, `double_value` are clear. Only a few show generic names like `FUN_00101020`.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic hex numbers appear without explanation (`0x3e9`=1001, `0x65`=101, `0x1e`=30). Original source had decimal values with comments explaining expected results.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`local_10 = *(long *)(in_FS_OFFSET + 0x28)`) repeated in many functions. External wrappers like `puts` show redundant variable assignments (`iVar1 = puts(__s); return iVar1;`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately sized with clear boundaries. Helper functions (`op_add`, `op_sub`) properly separated. Test functions (`test_control_flow_l1`) are overly long but cohesive.

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom but `loop_multi_exit` uses goto labels (`LAB_00101adc`, `LAB_00101aed`). `duffs_device` has complex jump structure disrupting flow.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Stack canary checks repeated across functions. Some redundant assignments (`iVar1 = param_1 * 2; if (param_1 < 1) { iVar1 = param_1; }`). No obvious dead code.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays in `loop_break` shown as stack variables with complex addressing (`(&iStack_28)[uVar1]`). No struct definitions visible. Function pointer arrays recovered as local arrays without type clarity.

**11. Type Safety:** Score: 5/10  
Justification: `UNRECOVERED_JUMPTABLE` marker in `call_func_ptr` indicates unsafe function pointer handling. `call_func_ptr_array` has stack canary but type mismatches. Some parameter type confusion (uint vs int).

**14. Anomaly Indication:** Score: 5/10

**15. Assertion & Contract Recovery:** Score: 2/10

- **Generic variable names** - `param_1`, `iVar1`, `uVar1` obscure semantic meaning (should be `temp1`, `sum`, `count`)

### Best Quality Aspect:

### Most Needs Improvement:

### Top 3 Recommendations for Decompiler Improvement:
1. **Infer variable names from usage patterns** - Analyze how variables are used (e.g., `iVar1` accumulates values → name it `sum`; `iVar1` counts iterations → name it `count`)

3. **Recover original type information from debug symbols** - Since compiled with `-g`, leverage DWARF debug info to restore precise types instead of generic `int`/`undefined`
