# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `iVar1`, `param_1`, `uVar1`, `pbVar1` with no semantic meaning. Original source had meaningful names like `result`, `encrypted`, `decrypted`, `heap_buffer`, `canary` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`param_fake_branch`, `call_opaque_predicate`, `param_tail_call_optimized`). However, runtime functions like `_init`, `_start` add noise, and some names don't reflect actual decompiled behavior.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere: `0x5a` (encryption key), `0xe1` (225), `0x7a314` (500500), `0x14` (20) with no explanation. Original source had comments explaining expected values and purposes.

**4. Noise Reduction:** Score: 5/10

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: `param_tail_call_optimized` correctly converted recursive to iterative loop. However, `param_vectorized_loop` uses awkward do-while with nested conditionals, and `param_opaque_predicate` has confusing while loop for GCD.

**6. Function Decomposition:** Score: 5/10  
Justification: Critical failure: `param_fake_branch` is completely empty (lost dead code branches), `call_fake_branch` just returns constant 10. `__aeabi_idiv` is 200+ lines of compiler runtime mixed with application code.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably. However, `__aeabi_idiv` has 30+ sequential boolean conditionals (`bVar5` through `bVar35`) that disrupt flow significantly.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Aggressive constant folding obscures original intent - `call_fake_branch`, `call_opaque_predicate`, `call_instruction_substitution` all return pre-computed constants. Makes understanding original logic impossible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**10. Composite Type Recovery:** Score: 2/10  
Justification: Bitfield struct in `param_implementation_defined` completely lost (returns constant `0x2c`). No struct definitions visible. `jmp_buf` used but not properly typed as `__jmp_buf_tag *`. Arrays handled as pointer arithmetic.

**11. Type Safety:** Score: 4/10  
Justification: Signal handlers have incorrect signatures (`div_zero_handler(void)` should take `int sig`). Multiple implicit casts visible. `param_division_by_zero` takes `undefined4` instead of `int`. Stack canary checks use wrong parameter patterns.
