# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `uVar2`, `local_c` with no semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `x`, `y` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source symbols (`cdecl_func`, `varargs_func`, `test_calling_conventions`). This is the decompiler's strongest aspect - original symbol table was available.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers in hex obscure meaning (`0xf` instead of `15`, `0x32` instead of `50`, `0x2a` instead of `42`). Original decimal values are more readable.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__stack_chk_fail`) and frame pointer variables (`local_c`) add noise. Floating-point helper functions (`__aeabi_dadd`, `__floatsidf`) are excessively verbose.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple functions have natural flow, but complex functions like `__subdf3` (200+ lines) use confusing conditional chains. Loop structures generally preserved but some converted to do-while patterns.

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes are reasonable, but critical logic loss occurs - `varargs_func` has empty loop body, `call_cdecl` returns constant `0xf` instead of calling `cdecl_func(5, 10)`.

**7. Code Linearization:** Score: 7/10  
Justification: Most code reads top-to-bottom with appropriate early returns. Some forward jumps in complex floating-point helpers disrupt flow slightly.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Floating-point helper functions contain massive redundant bit-manipulation code. Constant folding eliminated actual function call logic in wrapper functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Many types degraded to `undefined4`, `undefined8`, generic `int`. Original had `char*`, `double`, `long long`, `va_list` - all lost. Function pointers shown as `code *`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct types completely lost - `LargeStruct`, `SmallStruct` not recovered. All struct access shown as array indexing (`param_1[0]`, `param_1[1]`). No field names recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts (`(char *)`, `(int *)`, `(undefined4 *)`). `func_mixed_args` shows 6 parameters instead of 4, with double values split across multiple `undefined4` parameters.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no API contract documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: Standard C idioms not recognized - varargs pattern lost, struct passing not identified, function pointer callbacks shown as `code *` without context.

**14. Anomaly Indication:** Score: 1/10  
Justification: Critical bugs not flagged - `varargs_func` has empty loop body (logic completely lost), `func_mixed_args` has wrong parameter count. No warnings issued.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `param_1 != NULL` checks not explained). No postconditions or error condition documentation.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Callback relationships (`callback_func` → `param_func_ptr`) not explicitly indicated.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file. No logical grouping by test category (calling conventions vs. parameter passing vs. return values).

**18. Build System Compatibility:** Score: 4/10  
Justification: References external functions (`__printf_chk`, `strlen`, `__aeabi_*`) without declarations. Type mismatches would prevent compilation without extensive patching.

### Critical Defects:
- **`varargs_func` logic completely lost** - loop body is empty, va_list operations not recovered
- **Struct types not recovered** - `LargeStruct`, `SmallStruct` accessed as raw integer arrays
- **`func_mixed_args` has wrong signature** - shows 6 parameters instead of 4, double split incorrectly
- **Zero semantic comments** - no explanation of purpose, algorithms, or decompilation confidence
- **Constant folding eliminated call logic** - `call_cdecl` returns `0xf` instead of showing `cdecl_func(5, 10)`

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Original symbol names preserved from debug information, making function identification straightforward.

### Most Needs Improvement:
- **Semantic Transparency (1.25/10 average)** - Complete absence of comments, anomaly detection, and contract recovery makes understanding intent nearly impossible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover varargs patterns** - Detect `va_start`/`va_arg`/`va_end` sequences and reconstruct proper variable argument handling instead of empty loops.
2. **Add decompilation confidence annotations** - Flag uncertain reconstructions (e.g., "// DECOMPILER UNCERTAIN: varargs logic not recovered") to guide manual review.
3. **Recover composite types from usage patterns** - Analyze struct access patterns (`param[0]`, `param[1]`) to suggest struct definitions with field names based on usage context.
