# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are entirely generic (`iVar1`, `uVar2`, `param_1`, `local_28`, `pbVar1`) with no semantic meaning. Original source used descriptive names like `result`, `encrypted`, `decrypted`, `buffer` that indicated purpose.

**2. Function Naming Quality:** Score: 7/10  
Justification: Function names preserved from source (`param_fake_branch`, `param_opaque_predicate`, `decrypt_string`), but many functions are empty or transformed beyond recognition, reducing name accuracy.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation (`0x5a`, `0x14`, `0x23`, `0xe1`, `0x7a314`). Original had comments explaining expected values and purposes (e.g., `"Hello" ^ 0x5A`).

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise from compiler runtime (`__do_global_dtors_aux`, `register_tm_clones`), redundant external function wrappers, and massive `__divsi3` (300+ lines) that obscures simple division operations.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `param_opaque_predicate` uses complex bitwise operations instead of clear `if (p1 && p2 && p3)`. `param_tail_call_optimized` lost recursive structure entirely, replaced with arithmetic formulas.

**6. Function Decomposition:** Score: 3/10  
Justification: Critical failure: `param_fake_branch` is empty (logic lost), `call_fake_branch` returns constant 10 (optimized away). `__divsi3` is a 300+ line monolith that should be a library call.

**7. Code Linearization:** Score: 5/10  
Justification: Most functions read top-to-bottom, but `__divsi3` has extreme non-linearity with 35+ boolean variables (`bVar5` through `bVar35`) creating unreadable cascading conditionals.

**8. Redundancy Elimination:** Score: 3/10  
Justification: `__divsi3` and `_divsi3_skip_div0_test` are nearly identical 300+ line functions. External function wrappers (`raise`, `printf`, `strlen`) duplicate themselves unnecessarily.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Uses Ghidra's `undefined4`, mixed `int`/`uint`/`byte` without consistency. Original had `int32_t`, `size_t`, `char*`. `param_1` typed as `uint` when source used `int`, causing signed/unsigned confusion.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No structs recovered. Original `param_implementation_defined` had bitfield struct with named fields (`a:3`, `b:5`, `c:24`); decompiled version just returns constant `0x2c` with all structure lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts: `(byte *)`, `(undefined4 *)`, `(char *)`, `(__sighandler_t)0x0`. Pointer arithmetic without bounds checking visible throughout signal handling functions.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: ZERO comments in decompiled code except file header SHA256. Original had extensive section headers, algorithm explanations, and expected value documentation. No decompilation confidence indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: No patterns identified. GCD algorithm in `param_opaque_predicate` not recognized. XOR decryption pattern in `decrypt_string` not named. Instruction substitution patterns completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about `param_fake_branch` being empty despite source having logic. No indication that dead code elimination removed branches. Silent failures throughout.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != 0` for division). No postconditions. Original had expected return values in comments (e.g., "期望: 10"); none recovered.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`div_zero_caught`, `jmp_buffer`, `segv_buffer`) used without context or initialization visibility.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping. External runtime functions mixed with application code. No separation between obfuscation tests, optimization tests, and edge cases.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates compilation succeeds, but external dependencies (`__aeabi_idivmod`, `__libc_start_main`) not clearly documented. Would need significant stubs for standalone use.

**Overall Readability Score:** 3.6/10

### Critical Defects:
- **`param_fake_branch` is completely empty** - all logic lost despite -O0 compilation (should preserve dead code branches)
- **`call_fake_branch` returns constant 10** - function call optimized away, losing call relationship
- **Zero comments** - no semantic documentation recovered from original source's extensive comments
- **`__divsi3` is 300+ lines** - simple division operation expanded into unreadable boolean cascade
- **All variable names generic** - `iVar1`, `uVar2`, `param_1` provide no semantic information

### Best Quality Aspect: **Contextual Coherence - Cross-Reference Clarity (5/10)**
- Function call relationships are at least visible in the code, allowing manual tracing of execution flow despite other deficiencies.

### Most Needs Improvement: **Semantic Transparency - Comment Quality (1/10)**
- Complete absence of comments is catastrophic for readability; original source had extensive documentation that was entirely lost.

### Top 3 Recommendations for Decompiler Improvement:
1. **Preserve source comments in debug builds** - At -O0 with `-g` flag, DWARF debug info should allow comment/annotation recovery
2. **Detect and flag dead code elimination** - When functions become empty, add warning comments explaining optimization removed the logic
3. **Recognize common library patterns** - `__divsi3` should be identified as compiler runtime and collapsed to a single `divide()` call with comment
