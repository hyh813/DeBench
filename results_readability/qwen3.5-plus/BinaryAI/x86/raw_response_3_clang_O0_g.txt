# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are almost entirely generic (`param_1`, `local_c`, `local_14`, `aiStack_2c`, `pvVar1`) with no semantic meaning. For example, `local_14` in `heap_basic` should be `index` or `i`, and `__ptr` should indicate it's a heap array pointer.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are preserved from source (`local_vars`, `heap_basic`, `test_stack_memory`, `linked_list_heap`). However, some helper functions have generic names like `FUN_00011030` and `FUN_00011130`.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound without explanation: `0x2a` (should be `42`), `0xffffffff` (should be `-1`), `0x800` (should be `2048`). String literals referenced as `&DAT_00014008` instead of actual strings.

**4. Noise Reduction:** Score: 6/10  
Justification: Moderate noise from redundant casts (`(undefined4 *)`, `(void *)0x0`), stack variable declarations (`undefined local_c [4]`), and external function stubs that add clutter without value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control flow is natural (if/else, for loops preserved). However, `linked_list_heap` uses `while(true)` with break instead of a cleaner for loop, and some loops use increment patterns like `local_18 = local_18 + 1` instead of `++`.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions are appropriately sized and match original source structure. No excessive inlining or fragmentation. Each test function remains separate and cohesive.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with clear execution flow. Some forward references to data (`&DAT_00014008`) disrupt reading slightly, but no excessive goto usage.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Minor redundancy from repeated type casts and some duplicate null checks. No significant dead code, but loop structures could be cleaner (e.g., `local_14 = local_14 + 1` repeated throughout).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Significant type degradation: `undefined4` instead of `int`, `void *` instead of specific pointer types. Original types like `size_t`, `pid_t` partially preserved but often lost (e.g., `tls_access` uses `undefined4` instead of `int`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Critical failure - struct types (`Point`, `Node`, `TreeNode`) not recovered. Access through raw pointer arithmetic: `__ptr[1]`, `*(int *)((int)*param_1 + 4)`. No struct definitions visible in output.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts present (`(undefined4 *)`, `(int *)`), pointer arithmetic without type safety (`(int)__ptr + local_14 * 4`). No const-correctness preserved. `dangling_pointer` unsafe access not flagged.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled code. No algorithm explanations, no decompilation confidence indicators, no API contracts. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. Common patterns (linked list creation, tree traversal, memory allocation) not named or explained. `linked_list_heap` doesn't indicate it's building a linked list.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomalies flagged. Critical issues like `dangling_pointer` (use-after-free), `double_free`, `heap_overflow` not warned about. No uncertainty indicators for indirect calls.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `param_1 > 0` for `vla_stack`). No postconditions inferred. Error return values (`-1`, `-2`, `0xffffffff`) not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls are clear and traceable. Global variable usage visible (`global_counter`, `global_array`) but not well documented. No call graph documentation or callback relationship explanation.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in reasonable order matching source. Related test functions grouped together. However, no header file structure or clear separation between implementation and interface.

**18. Build System Compatibility:** Score: 5/10  
Justification: External functions declared as stubs (`<EXTERNAL>::printf`). Would require significant work to compile standalone. Dependencies (libc, pthread for TLS) not clearly documented.
