# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `iVar1`, `iVar2`, `param_1`, `local_10`, `local_20` with zero semantic meaning. Original source had descriptive names like `dst`, `src`, `buffer`, `thread_count`, `shared_counter`.

**2. Function Naming Quality:** Score: 4/10  
Justification: Some original names preserved (`param_strcpy`, `signal_handler`, `main`) but library calls degraded to `FUN_00011xxx` (e.g., `FUN_00011510` should be `strcpy`). Many helper functions use address-based names like `FUN_00011030`.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere: `0x2a` (42), `0x5a` (90), `0xc` (12), `0xf` (15). String literals referenced as `&DAT_00014xxx` instead of actual content. No named constants recovered.

**4. Noise Reduction:** Score: 4/10  
Justification: Compiler artifacts pervasive: `in_GS_OFFSET`, `__stack_chk_fail_local`, `LOCK()/UNLOCK()` macros, stack canary checks in every function. These obscure actual logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Mix of natural if-else and goto patterns (`LAB_00011d63`, `LAB_00012bb4`). Do-while loops partially recovered but many backward jumps remain as explicit gotos.

**6. Function Decomposition:** Score: 5/10  
Justification: Original function structure mostly preserved but compiler created `.part.0` and `.constprop.0` variants (e.g., `param_fork_exec.part.0`, `param_thread_local_storage.constprop.0`) fragmenting logic.

**7. Code Linearization:** Score: 5/10  
Justification: Stack check code at function ends creates non-linear reading. Forward jumps to error handling labels disrupt top-to-bottom flow in many functions.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Stack canary checks (`local_10 == *(int *)(in_GS_OFFSET + 0x14)`) repeated identically in nearly every function. No dead code elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Heavy use of `undefined4`, generic `int`, and `void*`. Original `size_t`, `const char*`, `pthread_t` all degraded. No const-correctness preserved.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `ThreadData` struct completely lost—accessed as array indices (`param_1[0]`, `param_1[1]`, `param_1[2]`). No struct definitions visible in output.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts everywhere: `(int *)`, `(undefined4 *)`, `(char *)`. Pointer arithmetic on generic types. No type safety annotations or bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no API documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: Thread patterns (pthread_create, mutex, condition variables) visible but not identified. No design pattern recognition. Atomic operations reduced to `LOCK()/UNLOCK()`.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags for unresolved indirect calls. `FUN_00011xxx` calls not marked as uncertain. No warnings about potential issues.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., "requires: param_1 != NULL"). No postconditions inferred. Error conditions not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but library functions unnamed. Global variables (`shared_counter`, `atomic_counter`, `signal_received`) referenced but not clearly identified as globals.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping. No header file structure. Related functions not clustered (test functions scattered).

**18. Build System Compatibility:** Score: 2/10  
Justification: References ~50 undefined `FUN_00011xxx` functions requiring extensive stubbing. Dependencies not documented. Would not compile standalone.
