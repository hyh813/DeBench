# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `a0`, `a1`, `v10`, `v12`) with no semantic meaning. Original names like `count`, `sum`, `args`, `large`, `s` are completely lost. Only occasional names like `ptr`, `idx`, `len` provide minimal clarity.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core test functions preserved well (`cdecl_func`, `varargs_func`, `test_calling_conventions`). However, many helper functions degraded to `sub_401020`, `sub_40102d` pattern. Original semantic names like `param_by_value_int`, `ret_small_struct` are preserved.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive: `4294967295` (0xFFFFFFFF), `0x100000000`, `47`, `8` appear without explanation. String literals referenced via opaque globals (`g_402004`, `g_4020c5`) instead of inline. No named constants recovered.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: uninitialized assignments (`v0 = v2` where v2 undefined), redundant stack variables (`[bp-0x58]` through `[bp+0x8]`), unsupported jumpkind errors, and explicit register hints (`// rax`, `// rdx`) clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most loops recovered as do-while/for constructs successfully. However, `sub_401020` uses `goto g_403fb0` pattern, and varargs functions have confusing nested conditionals. At -O0, this restructuring is unexpected.

**6. Function Decomposition:** Score: 6/10  
Justification: Function granularity mostly preserved, but critical failures exist: `param_struct_byval()` returns without value, `func_struct_byval()` loses struct parameter entirely. Initialization functions (`_init`, `_fini`, `deregister_tm_clones`) appropriately separated.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with clear function boundaries. Loop structures are linear. Some forward references in varargs handling but no severe spaghetti code patterns.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Moderate redundancy from stack variable proliferation. Varargs functions (`varargs_func`, `param_varargs`) are nearly identical duplicates. No loop unrolling (expected at -O0), but redundant assignments like `v0 = v2` persist.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `int` becomes `unsigned int`/`unsigned long long`, `char*` often becomes `unsigned long`. `func_mixed_args` loses `double` and `long long` parameters entirely. Return types inconsistent (`unsigned long long` vs `int`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct recovery failed catastrophically. `LargeStruct` and `SmallStruct` definitions lost. `func_struct_byval()` takes no parameters instead of `LargeStruct`. Generic `struct_0 { unsigned long long field_0; }` replaces all composite types.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe operations throughout: `AddV`, `ShlNV` operators suggest type confusion. Pointer arithmetic without type safety (`(&v0)[idx]`). Function pointer `param_func_ptr` has incorrect signature (`unsigned long long *a0` instead of callback). Multiple implicit casts.
