# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `a0`, `a1`, `ptr`, `result`). Original source had semantic names like `result`, `p1`, `p2`, `p3`, `encrypted`, `decrypted` which are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Some original names recovered accurately (`param_fake_branch`, `call_opaque_predicate`, `param_tail_call_optimized`), but many functions have mechanical names (`sub_401030`, `sub_401040`) that provide no semantic meaning.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `90` (XOR key), `4294967295` (-1), and `0x77359400` appear without explanation. Original constants like `0xDEADBEEF` and expected values are not preserved as named constants.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler artifacts present: `_init`, `_start`, `__do_global_dtors_aux`, `register_tm_clones`, `deregister_tm_clones`, `frame_dummy`, `__x86.get_pc_thunk.*` - many stub functions that add no analytical value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Some functions use `goto` instead of loops (`sub_401030`, `sub_4010f0`). The GCD loop in `param_opaque_predicate` is recovered reasonably, but tail recursion conversion to iteration lacks documentation.

**6. Function Decomposition:** Score: 4/10  
Justification: Many tiny wrapper functions that appear to be PLT/GOT stubs (`sub_40103c` calls `sub_401040`, etc.). `test_obf_opt_edge` is monolithic with inline code that should be separate function calls.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but frequent `goto` statements disrupt flow. Some functions like `param_fake_branch` are clean (`return a0;`), but others have confusing jump patterns.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Duplicate stub functions throughout. `test_obf_opt_edge` contains significant redundant code that duplicates logic from individual test functions instead of calling them.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Almost everything is `unsigned int`. Original distinctions (`int`, `char`, `size_t`, `const char*`) are lost. `param_division_by_zero` loses its `int x` parameter entirely in decompiled version.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Poor struct recovery - generic `struct_0`, `struct_1` with `padding_0[N]` and `field_X`. The bitfield struct in `param_implementation_defined` is completely lost, returning constant `43` instead.

**11. Type Safety:** Score: 3/10  
Justification: `param_null_pointer_deref` loses pointer parameter entirely. Function pointer calls through generic types. Implicit casts in pointer arithmetic throughout. Architecture mismatch (ARM64→X86) introduces type confusion.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: No meaningful comments in decompiled code. Original had extensive documentation explaining obfuscation techniques, expected values, and test scenarios - all completely absent.

**13. Idiom Recognition:** Score: 4/10  
Justification: GCD algorithm in `param_opaque_predicate` is recognizable but not named. Tail call optimization converted to loop but not documented. No pattern identification for encryption, opaque predicates, or instruction substitution.

**14. Anomaly Indication:** Score: 3/10  
Justification: Only one anomaly noted: `[D] Unsupported jumpkind Ijk_SigTRAP at address 4198907()`. No indication of decompilation uncertainty, architecture mismatch, or parameter loss in critical functions.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original comments like `// 期望：10` (expected: 10) and `// 永假条件` (always-false condition) are completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships mostly clear through direct calls. Global variables referenced (`encrypted.0`, `jmp_buffer`, `segv_buffer`) but not well documented. Signal handler relationships unclear.

**17. Module Organization:** Score: 4/10  
Justification: All code in single file with no logical grouping. Runtime initialization functions mixed with application logic. No separation between test harness and obfuscation functions.

**18. Build System Compatibility:** Score: 3/10  
Justification: External dependencies undefined (`__stack_chk_fail_local`, `_ccall`, `g_402008`, etc.). Architecture mismatch (X86 decompilation of ARM64 binary). Would require extensive stubbing to compile.

**Overall Readability Score: 3.8/10**

### Critical Defects
- **Architecture mismatch**: Original ARM64 binary decompiled as X86, causing fundamental type and calling convention errors
- **Parameter loss**: `param_division_by_zero` and `param_null_pointer_deref` lose critical pointer/int parameters
- **No comments**: All original documentation about obfuscation techniques, expected values, and test scenarios completely absent
- **Generic variable names**: `v0`, `v1`, `a0`, `a1` throughout obscure semantic meaning (e.g., `encrypted`, `decrypted`, `result`)
- **Compiler artifact pollution**: 40+ stub functions (`sub_401030`, `__x86.get_pc_thunk.*`, etc.) that add no analytical value

### Best Quality Aspect
- **Function Naming Quality (6/10)** - Some original function names were accurately recovered (`param_fake_branch`, `param_opaque_predicate`, `param_tail_call_optimized`), providing semantic anchors in an otherwise opaque codebase.

### Most Needs Improvement
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation means analysts cannot understand the obfuscation techniques, expected behaviors, or test purposes without referring to original source.

### Top 3 Recommendations for Decompiler Improvement
1. **Preserve and recover source comments** - Implement debug symbol parsing to extract and retain original comments, especially for obfuscation test cases where intent is critical
2. **Detect and flag architecture mismatches** - When binary architecture doesn't match decompilation target, prominently warn users and adjust type/calling convention recovery accordingly
3. **Improve parameter recovery for signal/exception handlers** - Functions using `setjmp`/`longjmp` and signal handlers need special handling to preserve parameter types and callback relationships
