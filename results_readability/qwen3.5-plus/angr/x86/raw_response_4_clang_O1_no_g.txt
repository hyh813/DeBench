# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables predominantly use generic names like `v0`, `v1`, `v2`, `ptr` throughout (e.g., `varargs_func` uses `v0`, `v1`, `v3` instead of `sum`, `args`, `count`). Occasional better names like `idx`, `len`, `i` appear but are inconsistent.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core functions retained original names (`cdecl_func`, `varargs_func`, `test_calling_conventions`), but numerous artificial wrapper functions (`sub_401201`, `sub_401219`, `sub_401226`) clutter the output without semantic value.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `4294967295` (unsigned -1) and `305419896` (0x12345678) appear without explanation. String literals replaced with opaque global references (`&g_4020a8`, `&g_402352`) instead of readable text.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant decompiler artifacts including `[D] Unsupported jumpkind Ijk_SigTRAP`, `[D] unsupported_<class 'pyvex.expr.GetI'>()` errors in `func_mixed_args`. Wrapper functions that simply call the real function add unnecessary noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple functions have natural flow (e.g., `cdecl_func` returns `a1 + a0`), but `func_mixed_args` has broken control flow with unsupported operations. Some initialization code uses goto patterns (`_start`, `sub_401030`).

**6. Function Decomposition:** Score: 5/10  
Justification: Artificial wrapper functions fragment the code (e.g., `sub_401201` calls `cdecl_func`). Many test functions are constant-folded (`call_cdecl` returns 15 directly), losing actual computation logic from source.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably well with preserved loop structures. However, unusual variable initialization patterns (e.g., `v0 = &vvar_1` in `varargs_func`) disrupt linear reading.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Wrapper function pairs create redundancy (every function has a `sub_XXXXXX` caller). Constant-folded test functions duplicate expected values rather than showing actual computation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types degraded to `unsigned int` or `int`. Original specific types (`char *`, `double`, `long long`, struct types) lost. `func_mixed_args` shows `double v4` but also lists `unsigned long` as "Other Possible Types".

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered with generic field names only (`field_0`, `padding_0`, `field_2ee7`). Original struct names (`LargeStruct`, `SmallStruct`, `TestIntPair`) and meaningful field names (`data`, `x`, `y`) completely lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe patterns: `func_mixed_args` has unsupported type operations, pointer arithmetic without proper typing (`(&v0)[2 * idx]`), function pointer calls with incorrect signatures (`a0(a1)` where `a0` is `unsigned int *`).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments in decompiled code. Only decompiler error tags (`[D]`) present. No algorithm explanations, API contracts, or semantic documentation recovered from original source comments.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns or algorithm recognition. Standard C idioms (varargs handling, struct passing) treated as generic code without identification. No pattern hints provided.

**14. Anomaly Indication:** Score: 6/10  
Justification: `[D]` tags do flag unsupported operations in `func_mixed_args` and `varargs_func`. However, anomalies lack explanation of what went wrong or suggested manual analysis steps.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `func_struct_byptr` should note `a0 != NULL`). No postconditions inferred. Error conditions like null pointer returns not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. Global variables referenced as `g_4020a8`, `g_402352` without indicating they're format strings. No call graph documentation.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output file with no logical grouping. Wrapper functions scattered throughout. No separation between test functions, implementation functions, and runtime initialization.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code contains compilation errors (`[D] unsupported_` operations). External dependencies not documented. Would require significant manual patching to compile standalone.

1. **Implement variable name inference** from usage patterns (e.g., variables used in loop counters should be named `i`, `idx`, `count`; accumulators should be `sum`, `total`)
