# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`, `ptr`, `idx`) with no semantic meaning. For example, `process_float` uses `v1` instead of `result` and the input parameter is `a0` instead of `f`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `process_char`, `test_data_types_l1`) but many functions are generic (`sub_401030`, `sub_401040`). Helper functions like `__x86.get_pc_thunk.ax` are compiler artifacts that should be hidden.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `65`, `98`, `4294967295` appear without explanation. Some test values are recognizable (100, 200) but character literals should show `'A'` not `65`.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe noise pollution with `[D] unsupported_<class 'pyvex.expr.GetI'>()` markers throughout float functions, `__x86.get_pc_thunk.*` calls in every function, and architecture mismatch artifacts (ARM64 source → x86 decompilation).

---

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic loops and conditionals are recovered (e.g., `array_1d_stack` has proper for-loop), but float functions have broken conditional branches with `[D]` markers disrupting flow.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are properly separated with reasonable granularity. Each test function is distinct, though helper functions like `sub_401030` appear to be compiler stubs that could be eliminated.

**7. Code Linearization:** Score: 6/10  
Justification: Code generally reads top-to-bottom with proper loop structures, but `[D]` marker blocks and thunk calls create visual interruptions that disrupt reading flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated `[D] unsupported` blocks appear multiple times in float functions. Stack check code (`__stack_chk_fail_local`) is duplicated at end of many functions unnecessarily.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 3/10  
Justification: Critical type degradation: `float`/`double`/`long double` all became `unsigned int`. `process_ll` takes two `unsigned int` instead of `long long`. `process_char` returns `int` instead of `char`. Signedness is often wrong.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs are recovered but with generic field names (`field_0`, `field_4`, `padding_0`). `struct_simple` accesses `a0[2] + *(a0) + a0[1]` instead of `p->x + p->y + p->z`. Union type information is lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe patterns: `ptr_void` uses ternary with mixed types, `struct_bitfields` uses bit manipulation on `void*`, pointer arithmetic uses raw integers instead of proper pointer types.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original source had extensive comments explaining each test case (e.g., `// DT-L1-01: 字符类型（char）⭐`), all lost in decompilation.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns not identified: linked list traversal in `linked_list` not labeled, binary tree recursion in `binary_tree_sum` not recognized, enum+switch pattern in `enum_switch` not documented.

**14. Anomaly Indication:** Score: 4/10  
Justification: `[D]` markers indicate problems but provide no explanation. Float functions show `[D] unsupported` but don't explain what operation failed or suggest manual review.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `array_complex_index` should note bounds checking). No postconditions or error condition documentation. Stack check failures not explained.

---

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are visible but relationships aren't documented. `test_data_types_l1` calls `process_char` but no call graph is provided. Global variables (`g_403010`) lack context.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in source order but aren't grouped by category. Test functions and implementation functions are mixed. No header file or API documentation generated.

**18. Build System Compatibility:** Score: 2/10  
Justification: Code will NOT compile due to `[D]` markers, undefined symbols (`_ccall`, `nan`), and architecture-specific artifacts. Requires extensive manual patching.

**Overall Readability Score:** 4.1/10

### Critical Defects:
- **Float/double operations completely broken** - All floating-point functions (`process_float`, `process_double`, `process_ld`) contain `[D] unsupported` markers and return incorrect types
- **Zero comments recovered** - Original source had extensive documentation; decompiled code has none
- **Type degradation for 64-bit integers** - `long long` split into two `unsigned int` parameters (`process_ll`)
- **Generic struct field names** - All struct fields are `field_0`, `field_4`, `padding_0` losing semantic meaning
- **Architecture mismatch artifacts** - ARM64 source decompiled as x86 with thunk calls polluting every function

### Best Quality Aspect: **Function Decomposition (7/10)**
- Functions are properly separated with appropriate granularity; each test case maps to a distinct function that can be analyzed independently.

### Most Needs Improvement: **Comment Quality & Presence (1/10)**
- Complete absence of comments makes understanding intent impossible; original source comments explaining each test scenario were entirely lost.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover floating-point operations properly** - The current pyvex-based analysis fails completely on float/double; integrate better FPU instruction handling or flag these as requiring manual review with clearer error messages.

2. **Preserve or infer semantic variable names** - Use data flow analysis to name variables based on usage (e.g., variables used in loop bounds should be `count`/`limit`, pointer dereferences should indicate the pointed-to type).

3. **Generate decompilation confidence annotations** - Instead of `[D] unsupported` markers, provide structured comments explaining what failed, why, and what manual steps are needed (e.g., `// DECOMPILER: Float operation not recovered - original was likely f * 1.5f + 0.5f`).
