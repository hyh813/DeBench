# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names (`v0`, `v1`, `a0`, `a1`, `ptr`) with no semantic meaning. Original source had descriptive names like `temp1`, `sum`, `count`, `result` that indicated purpose.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions retained original names (`sequential_ops`, `single_if`, `if_else`), but many helper functions use mechanical names (`sub_401030`, `sub_401090`). Runtime functions (`_init`, `_start`, `_fini`) are correctly named.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation: `4294967295` (should be `-1`), `1717986919` (division optimization constant), `0x80000000`. Original had clear test values with expected output comments.

**4. Noise Reduction:** Score: 4/10  
Justification: Compiler artifacts present: `__x86.get_pc_thunk.*` functions, complex struct padding definitions, unsupported jumpkind errors (`[D] Unsupported jumpkind Ijk_SigTRAP`), and unnecessary type casts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple loops transformed into incomprehensible bit manipulation (`loop_for_fixed`, `loop_while` use `>> 32 CONCAT` instead of iteration). Some structures recovered well (`if_else`, `nested_if_2`), but `duffs_device` uses awkward `LABEL_*` goto patterns.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved, but test functions severely broken (`test_control_flow_l1` has `printf` calls without format arguments). Some empty functions (`sub_401096`) suggest over-fragmentation.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but `duffs_device` uses multiple `goto LABEL_*` statements disrupting flow. Loop conditions often use complex expressions instead of clean `while` constructs.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundant variable assignments (`v0 = v1` patterns), but no severe dead code. At `-O0` optimization, expected minimal redundancy, yet optimization artifacts still present in loop functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Original `int` parameters decompiled as `unsigned int` causing signed/unsigned mismatches. Function pointers degraded to `unsigned int *a0` instead of proper signatures (`int (*f)(int)`). No const-correctness preserved.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs have generic field names (`field_0`, `padding_0`, `field_3ed7`). Critical error: `loop_break` treats int array `arr[]` as string `"\n"`. No meaningful field semantics recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts (`(char *)&ptr[4048]`), incorrect function pointer calls (`call_func_ptr_array` missing second argument), and pointer arithmetic without proper typing throughout.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled output. Original had extensive documentation explaining each function's purpose, test cases, and expected outputs. No decompilation confidence indicators present.

**13. Idiom Recognition:** Score: 5/10  
Justification: Basic patterns recognized (if-else chains, switch statements), but loop idioms completely missed (converted to bit manipulation). State machine patterns partially recovered (`state_machine`, `fsm_func_table`).

**14. Anomaly Indication:** Score: 3/10  
Justification: Only one error message (`[D] Unsupported jumpkind Ijk_SigTRAP`). Incorrect code presented confidently (test functions with broken printf calls). No uncertainty warnings for complex decompilations.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or error documentation recovered. Original had explicit test validation comments showing expected inputs/outputs for each function.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls present but relationships undocumented. Global variables referenced (`g_4025d1`, `g_40204c`) without context. Callback relationships unclear in `process_with_callback`.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output file with no logical grouping. Helper functions mixed with main functions. No header file structure or namespace organization.

**18. Build System Compatibility:** Score: 5/10  
Justification: Metadata claims compilation succeeds, but platform mismatch (original ARM64, decompiled X86). External dependencies declared (`_GLOBAL_OFFSET_TABLE_`) but would require significant patching to use portably.
