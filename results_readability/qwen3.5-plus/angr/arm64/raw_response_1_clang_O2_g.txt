# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`, `a2`) with no semantic meaning. Original `temp1`, `sum`, `count` became `v0`, `v1` obscuring their purpose completely.

**2. Function Naming Quality:** Score: 7/10  
Justification: Main function names preserved (`sequential_ops`, `if_else`, `nested_if_deep`) matching source, but helper functions use generic names (`sub_400894`, `sub_400970`) losing context.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere (`4294967295`, `1717986919`, `4294967294`) with no named constants. Original clear values like `10`, `100`, `200` transformed into opaque bit manipulation constants.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__ROL__`, `arm64g_calculate_condition`, `[D] unsupported_Iop_ZeroHI96ofV128()`. Stack frame manipulation (`v0 = &v1`) and runtime init code mixed with application logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple loops transformed into complex bit operations (`loop_for_fixed`, `loop_while`). Goto statements used where structured constructs should exist (`call_func_ptr: goto a0`). Some if-else preserved (`if_else`, `nested_if_deep`).

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but internal complexity increased dramatically. `recursion_factorial` and `tail_recursion` bloated with optimization artifacts. Test functions consolidated output rather than calling individual functions.

**7. Code Linearization:** Score: 5/10  
Justification: Mixed linearity - `if_else` and `ternary_op` read cleanly, but `loop_continue`, `goto_backward`, and `duffs_device` have significant forward jumps disrupting flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant bit masking and shifts throughout (`& 4294967295`, `| x >> 31`). Loop unrolling artifacts present without comments. Dead code not fully eliminated in several functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most `int` types degraded to `unsigned int`/`unsigned long long`. No const-correctness preserved. Function pointer types generic (`unsigned long`). Signed/unsigned confusion in `sequential_ops` (original `int` became `unsigned int`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Arrays in `loop_break` became switch statements. Matrix access in `loop_multi_exit` became complex pointer arithmetic. No struct/union recovery visible. All composite types lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout. Pointer arithmetic without bounds checking. Function pointer calls with generic types (`goto a0` in `call_func_ptr`). No warnings about potential issues.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no comments in decompiled code. Original had extensive documentation explaining each function's purpose and test cases. No algorithm explanations or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 4/10  
Justification: Basic patterns recognized (if-else chains, switch statements) but not explicitly identified. Loop patterns obscured by optimization. State machine in `state_machine` partially recovered but not labeled.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some anomalies flagged (`[D] unsupported_Iop_ZeroHI96ofV128()`) but inconsistent. No warnings about indirect calls or complex transformations. Incorrect transformations not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or error condition documentation. Original test cases showing expected behavior completely lost. No contract information recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls preserved but not documented. Global variables referenced without explanation (`g_412ff8`, `g_401a00`, `g_412fd0`). No call graph or callback relationship documentation.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output file with no logical grouping. Runtime initialization code (`_init`, `_start`, `_fini`) mixed with application code. Test functions not organized by level.

**18. Build System Compatibility:** Score: 3/10  
Justification: Does not compile standalone - contains `unsupported_Iop_ZeroHI96ofV128()` and `arm64g_calculate_condition`. External dependencies not documented. Would require significant patching.

### Critical Defects:
- **Variable names completely generic** - `a0`, `a1`, `v0`, `v1` instead of `a`, `b`, `c`, `sum`, `count`
- **Simple loops transformed to opaque math** - `loop_for_fixed` became `__ROL__((a0-1)*(a0-2), 63) + a0 - 1` instead of recognizable for-loop
- **No comments recovered** - Original had extensive documentation; decompiled has zero comments
- **Unsupported operations present** - `[D] unsupported_Iop_ZeroHI96ofV128()` appears in multiple functions
- **Type degradation** - Signed `int` became `unsigned int` causing potential semantic changes

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Main function names preserved from source, allowing identification of purpose despite internal degradation

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Complete loss of all documentation, making it impossible to understand intent without comparing to original source

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement variable name inference** from usage patterns and debug symbols when available (`-g` flag was present but not utilized)
2. **Recognize and reconstruct common loop patterns** from bit manipulation sequences (e.g., division by 10 via multiplication by magic constant should become `/10`)
3. **Add decompilation confidence annotations** to flag uncertain transformations and unsupported operations with actionable guidance
