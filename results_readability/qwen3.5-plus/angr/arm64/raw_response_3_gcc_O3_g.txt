# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (`v0`, `v1`, `a0`, `ptr`) with no semantic meaning. For example, `double_value(unsigned int a0)` loses the original parameter name `x`, and loop counters become `i` → `v6`/`v7` with no indication of purpose.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names recovered (`double_value`, `local_vars`, `heap_basic`), but many functions have mechanical names (`sub_4009c4`, `sub_400b24`). Wrapper functions like `sub_400c54` that just call `double_value()` add confusion.

**3. Literal Clarity:** Score: 3/10  
Justification: Pervasive magic numbers like `4294967295`, `17179869180`, `0x3000000020000000100000000` with no explanation. Simple `x * 2` becomes `(a0 * 2 | a0 >> 31) & 4294967294` obscuring intent.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe compiler artifacts: `arm64g_calculate_condition()`, `__ROL__()`, `ShlNV()`, `AddV()`, `CONCAT`, `[D] unsupported_Iop_ZeroHI64ofV128()`. Stack variables like `v0: [bp-0x10]` clutter every function.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Heavy `goto` usage replaces natural loops (e.g., `LABEL_400e84`). Complex conditionals with `arm64g_calculate_condition(10, a0 & 3, 0, 0)` obscure simple bounds checks.

**6. Function Decomposition:** Score: 5/10  
Justification: Original functions mostly preserved, but constant propagation created `alloca_usage.constprop.0()` returning hardcoded `15`. Wrapper functions (`sub_400c54`) add unnecessary indirection.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps (`goto LABEL_400fc0`) disrupt top-to-bottom reading. Deep nesting in functions like `vla_stack()` with 20+ local variables makes tracing difficult.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Loop unrolling artifacts present without abstraction. Duplicate expressions like `((unsigned int)v2 * 0x40000000 | (unsigned int)v2 >> 2) & 1073741823` repeated throughout.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Types degraded to generic `unsigned long long`, `unsigned int`. Original `int` becomes `unsigned int` in `double_value()`. Lost signed/unsigned distinctions and const-correctness throughout.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered but with generic field names (`field_0`, `field_8`, `padding_4`). `Node` struct becomes `struct_0 { unsigned int field_0; char padding_4[4]; struct struct_0 *field_8; }` losing semantic meaning.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe pointer arithmetic throughout. Casts like `*((int *)((char *)addr + 4 * ...))` without bounds checking. Function pointers lose signature information (`goto global_func_ptr`).

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero meaningful comments. No algorithm explanations, no decompilation confidence indicators. Only auto-generated type definitions present.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Linked list creation in `linked_list_heap()` not identified as such. Memory allocation patterns not annotated.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some unsupported operations marked `[D] unsupported_Iop_...` but many anomalies silent. `dangling_pointer()` use-after-free not flagged as dangerous.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `a0 > 0` checks not annotated). No postconditions or error condition documentation.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but relationships undocumented. `test_stack_memory()` calls not annotated. Global variable usage (`global_counter`) not contextualized.

**17. Module Organization:** Score: 4/10  
Justification: All 60+ functions in single output. No logical grouping by memory type (stack/heap/static). Original section comments lost.

**18. Build System Compatibility:** Score: 3/10  
Justification: Contains `[D]` unsupported operations that won't compile. External dependencies (`__libc_start_main`, `arm64g_calculate_condition`) not properly declared. Metadata claims "Compilation Result: Yes" but code has unresolvable symbols.

**Overall Readability Score: 3.4/10**

### Critical Defects:
- **Variable `a0` should be named `x`** in `double_value()` based on original source and usage pattern
- **Control flow uses `goto LABEL_400e84`** instead of natural loop/conditional constructs
- **Magic number `4294967294`** used repeatedly without explanation (should be `UINT32_MAX - 1`)
- **Struct fields all generic** (`field_0`, `field_8`, `padding_4`) losing semantic meaning for `Node`, `Point` types
- **Unsupported operations** (`[D] unsupported_Iop_ZeroHI64ofV128()`) make code non-compilable despite metadata claim

### Best Quality Aspect:
- **Function Naming Quality (5/10)** - Some original function names were successfully recovered (`double_value`, `local_vars`, `heap_basic`, `linked_list_heap`), providing anchors for understanding despite other degradation.

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation, algorithm explanations, or decompilation confidence indicators makes understanding intent nearly impossible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable naming** - Analyze usage patterns to infer names (e.g., variables used in loop bounds → `count`, `size`; pointer dereferences → `ptr`, `buffer`)
2. **Recover high-level control structures** - Convert `goto` + condition patterns back to `for`/`while` loops and `if-else` chains with proper nesting
3. **Add decompilation confidence annotations** - Mark uncertain reconstructions with comments like `// DECOMPILER UNCERTAINTY: indirect call target unresolved` and flag dangerous patterns (use-after-free, buffer overflows)
