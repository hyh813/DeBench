# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `v2`, `ptr`, `i`) with no semantic meaning. Original names like `large_buf`, `global_counter`, `head`, `current` are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Core functions retain original names (`double_value`, `local_vars`, `heap_basic`), but many helpers are generic (`sub_4009c4`, `sub_400b24`). Test functions preserved but logic replaced with constants.

**3. Literal Clarity:** Score: 2/10  
Justification: Simple arithmetic `x * 2` becomes `(a0 * 2 | a0 >> 31) & 4294967294`. Magic numbers like `4294967295` (UINT_MAX) appear without explanation. Test functions hardcode expected values instead of computing them.

**4. Noise Reduction:** Score: 4/10  
Justification: Stack canary checks (`__stack_chk_guard`) add verbosity. ARM64-specific `arm64g_calculate_condition` calls obscure simple conditionals. Redundant variable assignments throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loops recovered as do-while but with awkward pointer arithmetic. Some functions use `goto` instead of structured calls (`call_global_callback` uses `goto global_func_ptr`).

**6. Function Decomposition:** Score: 6/10  
Justification: Original function boundaries mostly preserved, but wrapper functions like `sub_400c54` calling `double_value()` are compiler artifacts. Function sizes reasonable.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but complex conditionals with `arm64g_calculate_condition` disrupt flow. Some forward jumps in error handling paths.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Functions like `sub_4009c4` call `memcpy` twice redundantly. Repeated bit manipulation patterns for simple arithmetic throughout codebase.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Generic `unsigned long long` and `char *` dominate. Original types (`int`, `Point`, `Node`) lost. Signed/unsigned often incorrect (e.g., `unsigned int` for signed operations).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs recovered with generic field names (`field_0`, `padding_4`, `field_8`). Original struct names (`Point`, `Node`, `TreeNode`) lost. Multiple conflicting `struct_0` definitions exist.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe casts throughout (`(char *)ptr + 4 * idx`). Function pointer calls use `goto` instead of proper call syntax. Pointer arithmetic on void* common.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments. Only auto-generated function headers. Complex bit manipulation completely unexplained. No decompilation confidence indicators.

**13. Idiom Recognition:** Score: 4/10  
Justification: Basic malloc/free patterns recognizable. No design pattern identification. Algorithm patterns (linked list traversal, tree creation) not named or documented.

**14. Anomaly Indication:** Score: 3/10  
Justification: Dangerous patterns not flagged—`dangling_pointer` use-after-free and `double_free` not highlighted as undefined behavior. Silent on security issues.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks present but not documented as preconditions. No postcondition inference. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables declared as `extern` without context. No call graph information.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output. Original organization (stack/heap/static sections) lost. No logical grouping by test category.

**18. Build System Compatibility:** Score: 5/10  
Justification: External dependencies declared but platform-specific code (`arm64g_calculate_condition`) requires stubs. Would need significant patching to compile.

**Overall Readability Score:** 4.0/10

### Critical Defects:
- **Test functions return hardcoded constants** instead of calling actual functions (e.g., `test_stack_memory` prints "20" instead of calling `local_vars(5)`)
- **Simple arithmetic obscured** by bit manipulation: `x * 2` becomes `(a0 * 2 | a0 >> 31) & 4294967294`
- **All struct field names generic** (`field_0`, `padding_4`, `field_8`) losing semantic meaning of `Point.x`, `Node.data`, etc.
- **No comments explaining complex patterns** or decompilation uncertainty
- **Dangerous patterns unflagged**: use-after-free in `dangling_pointer` and double-free in `double_free` not warned

### Best Quality Aspect:
- **Function Decomposition (5.25/10)** - Original function boundaries mostly preserved with reasonable granularity, though wrapper artifacts exist

### Most Needs Improvement:
- **Semantic Transparency (3.00/10)** - Complete absence of comments, no anomaly warnings, and test functions replaced with constants represents fundamental semantic loss

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover high-level arithmetic patterns** - Detect signed multiplication overflow patterns and restore to simple `x * 2` instead of bit manipulation sequences
2. **Preserve test function semantics** - Do not constant-fold test functions; maintain actual function calls to preserve test logic
3. **Add decompilation confidence annotations** - Flag uncertain reconstructions (indirect calls, complex bit patterns) and warn about dangerous patterns (use-after-free, double-free)
