# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a0`, `a1`, `v0`, `v1`) with no semantic meaning. Original `sequential_ops(int a, int b, int c)` became `sequential_ops(unsigned int a0, unsigned int a1, unsigned int a2)` losing parameter intent entirely.

**2. Function Naming Quality:** Score: 7/10  
Justification: Main function names preserved well (`sequential_ops`, `nested_if_deep`, `switch_fallthrough`), but runtime helpers use generic names (`sub_400894`, `sub_400970`). Good recovery of user-defined function names.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere: `4294967295` (0xFFFFFFFF), `1717986919` (division-by-10 optimization), `4294967294`. Original clear literals (10, 5, 2) replaced with optimization artifacts.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__ROL__` calls, `[D] unsupported_Iop_ZeroHI96ofV128()`, frame pointer manipulation (`v0 = &v1`), redundant casts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple if/else recovered well (`if_else`, `nested_if_deep`), but loops severely degraded (`loop_while` uses multiplication magic instead of `/10`), `duffs_device` has mangled switch/goto structure.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved correctly, but some functions bloated with optimization artifacts. Runtime support functions (`deregister_tm_clones`, `frame_dummy`) mixed with user code.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom, but `duffs_device` uses goto labels (`LABEL_401060`), `conditional_func_ptr` has `goto v4`, `test_control_flow_l2` has complex nested gotos.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant assignments visible (`v0 = v2; v1 = v3;` in `$x` function), duplicate expressions in optimization artifacts, some dead code patterns from register spilling.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation: `int` becomes `unsigned int` inconsistently, return types vary (`int`, `long long`, `unsigned long long`), signed/unsigned correctness often wrong (`sequential_ops` parameters changed).

**10. Composite Type Recovery:** Score: 2/10  
Justification: Arrays become pointer arithmetic (`loop_break`'s `arr[]` becomes `*((int *)(4200812 + 4 * v0))`), no struct definitions visible, matrix in `loop_multi_exit` loses 2D structure.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe patterns: `goto a0` in `call_func_ptr` (function pointer as address), unsupported operations indicate type confusion, pointer arithmetic without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: ZERO comments in decompiled output. Original source had extensive documentation (algorithm explanations, test expectations, scenario tags) - all completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: Basic if/else/switch recognized, but Duff's device mangled, state machine patterns not identified, no algorithm naming or pattern explanation anywhere.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies marked (`[D] unsupported_Iop_*`, `/* do not return */`), but many incorrect recoveries not flagged, no confidence indicators for uncertain decompilation.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or error documentation recovered. Original test expectations (e.g., "输入5,7,3应返回21") completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**17. Module Organization:** Score: 3/10
