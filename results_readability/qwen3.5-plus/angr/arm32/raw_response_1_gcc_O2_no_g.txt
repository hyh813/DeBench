# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`, `cur`) with no semantic meaning. Original source had meaningful names like `temp1`, `sum`, `count`, `total` that were completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user-defined functions retained original names (`sequential_ops`, `recursion_factorial`, `double_value`), but internal functions use generic names (`sub_10454`, `sub_10530`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere: `4294967295` (UINT_MAX/-1), `1717986919` (division constant), `74000` (hardcoded address). No named constants recovered from original source.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant angr artifacts (`armg_calculate_condition`), stack canary checks (`__stack_chk_guard`), redundant variable assignments (`v0 = v1`), and excessive goto labels (`LABEL_10600`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Excessive goto usage instead of structured control flow. `if_elseif_long` is completely garbled with nested labels. `switch_small` uses array indexing instead of switch. Simple functions like `sequential_ops` are acceptable.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved, but parameter lists are often wrong (`switch_small` has 4 params instead of 1, `nested_if_deep` has 4 instead of 5).

**7. Code Linearization:** Score: 4/10  
Justification: Heavy forward jump usage (`LABEL_107b2`, `LABEL_107e5`) disrupts top-to-bottom reading. `if_elseif_long` has particularly bad label chains.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Duplicate `armg_calculate_condition` calls with identical arguments, redundant stack canary checks, and unnecessary variable copying throughout.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Almost everything is `unsigned int` regardless of original signed/unsigned distinction. Original `int` types lost. Function return types often wrong (e.g., `op_div` returns `unsigned int` but should handle signed division).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Arrays shown as individual stack variables (`v0`, `v1`, `v2`, `v3`, `v4`, `v5` in `loop_break`). Matrix in `loop_multi_exit` decomposed to 12 separate variables instead of `int[3][4]`.

**11. Type Safety:** Score: 4/10  
Justification: Function pointer calls use unsafe `goto` syntax (`goto a0` in `call_func_ptr`, `goto v2` in `conditional_func_ptr`). Implicit signed/unsigned conversions throughout.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation explaining each function's purpose, test cases, and difficulty levels. No decompiler confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: Loop patterns not recognized (shown as do-while with goto). Switch statements not recovered. State machine patterns in `state_machine` and `fsm_func_table` not identified.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about uncertain decompilation. Broken functions like `if_else` and `nested_if_deep` presented confidently. `armg_calculate_condition` calls indicate unresolved logic but aren't explained.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `n > 0` for factorial). No postcondition inference. Original test case expectations completely absent from decompiled code.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables shown (`g_23008`, `jump_buffer`) but usage not explained. No call graph documentation.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single flat output. Test functions mixed with implementation. No logical grouping or header file separation.

**18. Build System Compatibility:** Score: 3/10  
Justification: Won't compile without major modifications. angr-specific artifacts (`armg_calculate_condition`) need stubs. External dependencies (`__printf_chk`, `__aeabi_idiv`) undocumented.

**Overall Readability Score:** 4.0/10

### Critical Defects:
- **`armg_calculate_condition` artifacts** throughout code indicate unresolved symbolic execution, making logic incomprehensible
- **Wrong parameter counts** in `nested_if_deep` (4 vs 5) and `switch_small` (4 vs 1) break semantic correctness
- **Function pointers use `goto`** instead of proper call syntax (`goto a0` in `call_func_ptr`)
- **Array structures lost** - shown as individual stack variables instead of arrays/matrices
- **Zero comments** despite original source having extensive documentation for each function

### Best Quality Aspect: **Function Naming Quality (7/10)**
- Most user-defined functions retained their original semantic names, allowing identification of purpose despite other degradation.

### Most Needs Improvement: **Comment Quality & Presence (2/10)**
- Complete absence of any documentation, confidence indicators, or semantic explanations makes reverse engineering extremely difficult.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover array/struct types from stack layout patterns** - detect consecutive stack variables with uniform access patterns as arrays
2. **Replace `armg_calculate_condition` with actual conditional expressions** - this angr artifact obscures all branch logic
3. **Add decompiler confidence annotations** - flag uncertain reconstructions (e.g., `// DECOMPILER UNCERTAIN: indirect call`) to guide manual review
