# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `a0`, `a1`, `idx` with no semantic meaning. Original variables like `sum`, `len`, `i`, `c` are completely lost. Even simple functions like `process_char` use `v0`, `v1` instead of meaningful names like `char_code`, `offset`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `process_int`, `process_char`, `array_1d_stack`), but many are generic (`sub_103e8`, `sub_115a8`). Test functions partially recovered (`test_data_types_l1`, `test_array_types`). No action-oriented naming for runtime functions.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation: `65`, `32`, `255` for character operations; `0x3f000000`, `2576980378` for floating-point constants; `40`, `110` for array strides. No named constants recovered.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive `armg_calculate_condition()` calls (ARM condition code emulation), stack canary checks (`GLIBC_2.4::__stack_chk_guard`), and compiler runtime calls (`__mulsf3`, `__aeabi_dadd`) pollute the code throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Heavy use of `goto LABEL_XXXX` instead of natural loops and conditionals. Simple while loops become `while(true) { if(...) goto LABEL; return; LABEL: ... }`. Even `process_bool` has 6+ goto labels for a simple boolean expression.

**6. Function Decomposition:** Score: 5/10  
Justification: Function granularity is reasonable, but test functions are flattened into sequential printf calls. Runtime functions like `__muldf3` (400+ lines) should be external library calls, not decompiled inline.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps disrupt reading flow. `array_2d_stack` has nested do-while with goto that obscures the simple diagonal sum logic. Multiple exit points via goto instead of return statements.

**8. Redundancy Elimination:** Score: 4/10  
Justification: ARM condition code calculations repeat identically in branches. Floating-point operations expand into hundreds of lines of bit manipulation that could be simplified to `f * 1.5f + 0.5f`.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 2/10  
Justification: Severe type erosion: `char` → `unsigned int`, `short` → `unsigned short` (but masked manually), `float`/`double` → `unsigned int` with bit operations, all pointers → `unsigned int`. No const/volatile preservation (`volatile_access` becomes simple `*p * 2`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: Structs recovered as `struct_0`, `struct_1` with `field_0`, `field_1`, `field_4`, `field_8`. Original `Point3D {x, y, z}`, `Rect {position, size}`, `Node {data, next}` completely lost. Bitfields in `Flags` struct unrecoverable.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe pointer-to-int casts throughout (`(int)(long)handle`). Array indexing via manual offset calculation (`ptr + idx * 4`). Function pointers called via `goto a0->field_4` instead of proper call syntax.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No explanation of ARM condition codes, floating-point bit layouts, or algorithm intent. Original source had extensive test documentation completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: Common patterns unrecognized: character case conversion (`c >= 'A' && c <= 'Z'` → `c + 32`), array summation loops, linked list traversal, string length calculation. All appear as raw arithmetic.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about unresolved indirect calls, suspicious `armg_calculate_condition` patterns, or potential undefined behavior. `array_oob` intentional overflow not flagged.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `ptr != NULL`), no postconditions, no error condition documentation. Function contracts completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships unclear. `double_value` called from `ptr_func_simple` but no indication this is a callback pattern. Global variables (`g_23008`, `g_11ed4`) unnamed and unexplained.

**17. Module Organization:** Score: 4/10  
Justification: Functions appear in arbitrary order. Runtime functions mixed with user code. No logical grouping by test category (data types, arrays, pointers, composites).

**18. Build System Compatibility:** Score: 3/10  
Justification: Won't compile standalone. Missing external declarations for `armg_calculate_condition`, `__mulsf3`, `__aeabi_dadd`. Custom types (`struct_0`) need definition. Stack canary references unresolved.

**Overall Readability Score:** 3.0/10

### Critical Defects:
- **Variable `v0`, `v1`, `a0`** should be named based on usage (e.g., `char_code`, `sum`, `index`)
- **Control flow uses goto** for simple loops instead of while/for constructs
- **All struct fields generic** (`field_0`, `field_1`) losing semantic meaning of `x`, `y`, `data`, `next`
- **Type information lost** - `char`, `float`, `double` all become `unsigned int`
- **Zero comments** explaining ARM condition codes or floating-point bit manipulation

### Best Quality Aspect: **Function Naming Quality (5/10)**
- Some original function names preserved (`double_value`, `process_int`, `array_1d_stack`) providing anchor points for understanding

### Most Needs Improvement: **Comment Quality & Presence (1/10)**
- Complete absence of any documentation makes understanding algorithm intent nearly impossible

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable naming** - Analyze usage patterns to infer names (loop counters → `i`, accumulators → `sum`, character values → `char_code`)
2. **Recover high-level control structures** - Convert `while(true) { if(cond) goto LABEL; ... }` patterns back to natural `while`/`for` loops
3. **Add confidence annotations** - Flag uncertain decompilations (ARM condition codes, floating-point operations) with comments indicating manual review needed
