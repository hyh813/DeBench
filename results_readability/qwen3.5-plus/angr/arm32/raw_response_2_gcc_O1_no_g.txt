# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (`v0`, `v1`, `a0`, `a1`, `ptr`, `i`) with no semantic meaning. For example, in `process_char`, `v0` and `v1` should be `char_offset` and `result` based on usage.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `process_int`, `array_1d_stack`), but many runtime helpers are `sub_103c4`, `sub_11674`. Floating point helpers like `__muldf3` are accurate but the code is unreadable.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation: `0x80000000`, `2047`, `255`, `0x100000`, `0x7f000000`. The value `32` for uppercase-to-lowercase conversion in `process_char` is not documented.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe compiler artifacts including `armg_calculate_condition()`, `armg_calculate_flag_*()` calls throughout. Floating point functions contain hundreds of lines of bit-manipulation noise that obscures simple operations like `f * 1.5f + 0.5f`.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Excessive `goto` statements replace natural loops and conditionals. The `process_bool` function uses 6+ goto labels for a simple `(x > 0 && x % 2 == 0)` expression. Floating point functions have deeply nested conditional chains.

**6. Function Decomposition:** Score: 5/10  
Justification: User functions maintain reasonable size, but runtime helpers like `__muldf3` (400+ lines) and `__aeabi_dadd` (500+ lines) are monolithic. These should be recognized as library calls, not decompiled.

**7. Code Linearization:** Score: 4/10  
Justification: Forward jumps into loop bodies disrupt reading flow. The `array_1d_stack` function has backward jumps that could be recovered as `while` loops but remain as `do-while` with goto patterns.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Repeated `armg_calculate_condition()` patterns with identical parameters appear throughout floating point functions. Dead code from register spilling is present in many functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Severe type erosion - `char` becomes `unsigned int`, `float`/`double` become `unsigned int` with bit manipulation, `short` loses 16-bit semantics. `process_float` returns `void` instead of `float`. `volatile_access` loses volatile semantics entirely.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs become `struct_0`, `struct_1` with generic `field_0`, `field_1`, `field_4` names. Original field names like `x`, `y`, `z` in `Point3D` are lost. Nested structures like `Widget` → `Rect` → `Point3D` are flattened to pointer arithmetic.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe casts: `(int*)(long)handle` in `opaque_handle_op`, void pointer casts throughout. Pointer arithmetic uses raw integers instead of proper pointer types. `ptr_double` uses `struct_0**` instead of `int**`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No explanation of algorithm intent, no decompilation confidence indicators, no API documentation. Original source had extensive comments explaining test scenarios.

**13. Idiom Recognition:** Score: 2/10  
Justification: Common patterns not recognized: character case conversion (`c >= 'A' && c <= 'Z'`), array summation loops, linked list traversal. Floating point operations should be recognized as library calls, not bit manipulation.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about problematic decompilation. The floating point functions are clearly incorrect/incomplete but no anomaly flags are present. Stack canary checks appear without explanation.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `arr != NULL`, `n > 0`). No postconditions or return value guarantees. Error conditions like array bounds checks are present but not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are visible but relationships aren't documented. `test_data_types_l1` calls are clear, but callback relationships like `ptr_func_simple(double_value, 5)` lack context about function pointer usage.

**17. Module Organization:** Score: 4/10  
Justification: Functions appear in arbitrary order mixed with runtime helpers. No logical grouping by test category (data types, arrays, pointers, composites). Static functions not properly scoped.

**18. Build System Compatibility:** Score: 3/10  
Justification: Does not compile standalone - missing external declarations (`GLIBC_2.4::__stack_chk_guard`, `__printf_chk`), undefined symbols (`armg_calculate_condition`), and architecture-specific code (ARM).

**Overall Readability Score:** 3.1/10

### Critical Defects:
- **Type erosion**: All types collapsed to `unsigned int` - `float`, `double`, `char`, `short` semantics completely lost
- **Floating point decompilation failure**: Functions like `__muldf3` (400+ lines) should be recognized as library calls, not decompiled to bit manipulation
- **Variable naming**: Generic names (`v0`, `v1`, `a0`) throughout obscure semantic meaning
- **Control flow degradation**: Simple conditionals converted to complex goto chains (e.g., `process_bool`)
- **Zero documentation**: No comments explaining algorithm intent or decompilation confidence

### Best Quality Aspect: **Function Naming Quality (5/10)**
- Some original function names preserved (`double_value`, `process_int`, `array_1d_stack`) providing basic semantic anchors

### Most Needs Improvement: **Comment Quality & Presence (1/10)**
- Complete absence of any documentation makes understanding decompiler intent impossible

### Top 3 Recommendations for Decompiler Improvement:
1. **Recognize standard library functions** - Floating point helpers (`__muldf3`, `__aeabi_dadd`) should be identified as runtime calls, not decompiled to hundreds of lines of bit manipulation
2. **Recover type information from usage patterns** - Analyze arithmetic operations to infer original types (e.g., `& 65535` suggests `short`, float constants suggest `float`/`double`)
3. **Recover high-level control structures** - Convert goto-based loops back to `for`/`while` constructs and simplify conditional chains to standard if-else patterns
