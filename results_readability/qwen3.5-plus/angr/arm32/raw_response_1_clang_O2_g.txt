# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`v0`, `v1`, `a0`, `a1`, `a2`) with no semantic meaning. Original names like `temp1`, `sum`, `count`, `result` are completely lost, making it difficult to track variable purpose across functions.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved (`sequential_ops`, `single_if`, `nested_if_2`), but helper functions have cryptic names (`sub_4004dc`, `$a.14`, `$a.16`). Main test functions retain original semantic meaning.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `4294967295` (UINT_MAX/-1), `1717986919` (division optimization), and `0x80000000` appear without explanation. Original clear literals (10, 5, 2) are obscured by compiler optimization artifacts.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant noise from `armg_calculate_condition()` calls (Angr REIL artifacts), dead code after returns (`v2 = result;` after `return result;`), and excessive temporary variables from register spilling throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `single_if` uses bit manipulation instead of simple if; `nested_if_deep` uses goto and condition calculations instead of nested if; loops use do-while with goto instead of clean for/while structures. Only `sequential_ops` and `ternary_op` recovered naturally.

**6. Function Decomposition:** Score: 5/10  
Justification: `duffs_device` broken into multiple sub-functions (`$a.14`, `sub_400d5c`, etc.); test functions have hardcoded return values instead of actual function calls; some inlined code not properly recovered.

**7. Code Linearization:** Score: 5/10  
Justification: Excessive forward jumps (`LABEL_400720`, `LABEL_400824`) disrupt reading flow; dead code after returns creates false execution paths; some functions like `sequential_ops` read linearly but most don't.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Dead code after every conditional return; `armg_calculate_condition` calls duplicated in both branches; redundant variable assignments (`v1 = v0;` followed by immediate use) throughout.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: All `int` converted to `unsigned int` (semantic change for negative handling); `volatile int *` becomes `unsigned int *`; function pointers degraded to `unsigned int`; no const-correctness preserved.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Array `int arr[] = {10, 20, 30, 40, 50}` in `loop_break` became switch-case; matrix in `loop_multi_exit` became global array references (`g_401e6c`); no struct recovery visible.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts throughout (`(char)armg_calculate_condition(...)`); `goto a0;` in `call_func_ptr` treats parameter as address; bit manipulation expressions like `a0 >> 31` lack proper type handling.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original had extensive documentation explaining each function's purpose, test cases, and control flow patterns. No decompiler confidence indicators present.

**13. Idiom Recognition:** Score: 3/10  
Justification: Switch fallthrough not recognized (became nested if); loop patterns (for/while/do-while) all degraded to do-while with goto; state machine pattern in `state_machine` logic broken; no algorithm identification.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about `armg_calculate_condition` artifacts; no indication of unresolved indirect calls; `computed_goto` function body empty with no explanation; code presented confidently despite obvious recovery failures.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No precondition documentation; no postcondition inference; test cases not preserved as assertions; error conditions undocumented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls preserved where names recovered; global variables have generic names (`g_401e2c`, `g_413008`); test functions use hardcoded values instead of actual calls; callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output (expected); no logical grouping visible; static functions not properly scoped; helper functions scattered without organization.

**18. Build System Compatibility:** Score: 2/10  
Justification: `goto a0;` is invalid C (goto to variable); `armg_calculate_condition` undefined; multiple undefined global references; would not compile without extensive modification.
