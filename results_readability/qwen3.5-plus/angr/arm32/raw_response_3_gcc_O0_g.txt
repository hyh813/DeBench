## Decompilation Readability Assessment

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `v2`, `ptr`, `idx`) instead of semantic names like `local`, `arr`, `head`, `current` from source. Example: `local_vars` uses `v0`, `v1` instead of `a`, `b`, `c`.

**4. Noise Reduction:** Score: 3/10  
Justification: Stack canary checks (`__stack_chk_guard`) injected into nearly every function, creating repetitive boilerplate. Compiler runtime functions (`__sync_*` variants) bloating output significantly.

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most loops recovered as `for` constructs, but stack canary checks add artificial conditionals at function exits. Some functions use `goto` patterns unnecessarily.

**6. Function Decomposition:** Score: 7/10  
Justification: Function granularity mostly matches source. However, compiler runtime functions (`__sync_fetch_and_*`) are extracted as separate functions, creating fragmentation.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but stack canary checks create exit branches that disrupt flow. Example: `local_array` has canary check interrupting return path.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Stack canary code repeated identically across 20+ functions. No consolidation of common patterns. Type casts repeated unnecessarily.

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Widespread type degradation: `int` → `unsigned int`, `size_t` → `unsigned int`, `char*` → `unsigned int`. Example: `double_value(unsigned int a0)` should be `int`. Pointer types often lost.

**10. Composite Type Recovery:** Score: 5/10

**11. Type Safety:** Score: 4/10
