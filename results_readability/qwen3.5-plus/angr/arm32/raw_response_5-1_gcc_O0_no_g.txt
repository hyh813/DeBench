# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `v2`, `ptr`) with no semantic meaning. Original source had meaningful names like `capture_by_value`, `instance_count`, `multiplier` that are completely lost.

**2. Function Naming Quality:** Score: 5/10

**3. Literal Clarity:** Score: 3/10

**4. Noise Reduction:** Score: 4/10  
Justification: Stack canary checks (`__stack_chk_guard`) in every function. Compiler runtime functions (`_init`, `_start`, `deregister_tm_clones`) included. Redundant type casts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Excessive `goto` statements with labels (`LABEL_10c28`, `LABEL_11035`) instead of structured control flow. Original's clean if/else and loops converted to jump patterns.

**6. Function Decomposition:** Score: 5/10  
Justification: Test functions preserved but STL template instantiations over-fragmented (20+ tiny functions for `unique_ptr` implementation). Original's cohesive class methods split into disconnected pieces.

**7. Code Linearization:** Score: 4/10  
Justification: Forward jumps into middle of functions disrupt top-to-bottom reading. Stack canary checks at function ends create non-linear return patterns.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Stack canary code duplicated in every function. Template instantiations create massive redundancy (similar `tuple`, `_Head_base` functions repeated).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**10. Composite Type Recovery:** Score: 4/10

**11. Type Safety:** Score: 3/10
