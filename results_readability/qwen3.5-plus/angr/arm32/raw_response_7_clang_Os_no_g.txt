# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `v0`, `v1`, `v2` with register hints (`r0`, `lr`) that provide no semantic meaning. Original source had meaningful names like `result`, `encrypted`, `decrypted`, `canary` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Core function names preserved (`param_fake_branch`, `call_opaque_predicate`) but signatures are wrong (e.g., `param_fake_branch()` takes no parameters when source has `int x`). Helper functions use mechanical names (`sub_4004b8`, `sub_4005a0`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere without explanation: `4294967295` (should be `-1` or `UINT_MAX`), `90` (XOR key), `0x77359400`. No constants recovered from original `0xDEADBEEF`, `0x5A`, etc.

**4. Noise Reduction:** Score: 3/10  
Justification: Extensive register spilling artifacts (`v0 = v1`, `v1 = v2`), goto labels (`LABEL_4005c4`), and runtime helper calls (`armg_calculate_condition`, `__aeabi_idivmod`) pollute the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Heavy goto usage instead of structured control flow. Loops use `do-while` with goto labels instead of clean `for/while`. Original `if (x * x < 0)` dead branches not recognized as such.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved but some are broken (`param_fake_branch` loses its parameter entirely). Runtime helpers (`__divsi3`, `__aeabi_idivmod`) incorrectly exposed as separate functions.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps (`goto LABEL_4005c4`) disrupt top-to-bottom reading. Multiple return paths with goto instead of early returns make flow hard to trace.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundant operations from optimization visible (e.g., `call_string_encryption()` calls `param_string_encryption()` twice). Dead code from register allocation not eliminated.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types degraded to `unsigned int` or `int`. Lost distinction between `char*`, `size_t`, `volatile`. Original `const char *str` becomes `char *a0`. Signed/unsigned confusion throughout.

**10. Composite Type Recovery:** Score: 3/10  
Justification: No struct recovery visible. Original `struct { unsigned int a:3; b:5; c:24; }` in `param_implementation_defined` completely lost. Arrays shown as pointer arithmetic (`*((unsigned int *)&(&v0)[4 * idx])`).

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts (`(char)armg_calculate_condition(...)`), pointer-to-int conversions (`unsigned int *a0`), and missing const correctness. `param_null_pointer_deref` takes `unsigned int *` instead of `int *`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments except platform header. No explanation of obfuscation patterns, dead branches, or why certain calculations exist. Original had extensive documentation completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Original's "永假条件" (always-false conditions), opaque predicates, and instruction substitution patterns not identified or documented.

**14. Anomaly Indication:** Score: 2/10  
Justification: Broken functions not flagged (e.g., `param_fake_branch` returns `v0` without using input). No warnings about uncertain decompilation or potential errors.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_division_by_zero` should note `a0 != 0` for normal path). Error handling via `setjmp/longjmp` not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. Global variables have cryptic names (`g_412008`, `g_401050`). No call graph or dependency information.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output. No logical grouping by category (obfuscation vs. optimization vs. edge cases). Original's section organization lost.

**18. Build System Compatibility:** Score: 3/10  
Justification: Multiple compilation issues: undefined functions (`armg_calculate_condition`), type mismatches, missing headers. Would require significant patching to compile standalone.

**Overall Readability Score:** 3.5/10

### Critical Defects:
- **`param_fake_branch()` loses its `int x` parameter entirely** - function signature completely broken
- **All variables use generic names** (`v0`, `v1`, `v2`) with no semantic meaning
- **Extensive goto usage** instead of structured control flow (if/while/for)
- **No comments** explaining obfuscation patterns, dead branches, or algorithm intent
- **Magic numbers unexplained** (4294967295, 90, 0x77359400) without constant recovery

### Best Quality Aspect:
- **Function Naming Quality (5/10)** - Core function names like `param_fake_branch`, `call_opaque_predicate` were preserved from source, providing some semantic anchor points despite signature issues.

### Most Needs Improvement:
- **Semantic Transparency (2/10 average)** - Complete absence of comments, no pattern recognition, no anomaly flagging, and no contract documentation makes the code practically unusable for understanding original intent.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover function signatures from calling conventions** - `param_fake_branch` should have `int x` parameter based on how `call_fake_branch` invokes it with literal `10`
2. **Detect and document dead code patterns** - Flag `if (x * x < 0)` as always-false condition with explanatory comment
3. **Infer variable names from usage context** - Variables used in string operations should be named `buffer`, `ptr`, `decrypted` rather than `v0`, `cur`, `dst`
