# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `v2`, `arg_0`, `arg_1`) with no semantic meaning. Original names like `capture_by_value`, `instance_count`, `multiplier` are completely lost. Only occasional names like `i`, `index`, `result`, `ptr` provide minimal hints.

**2. Function Naming Quality:** Score: 5/10  
Justification: Functions retain mangled C++ names (`_Z20test_cpp_member_funcv`, `_ZN4Base12virtual_funcEi`) that are technically recoverable but not human-readable without demangling. Some runtime functions use generic `sub_401644` naming. Test function names are partially preserved.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation: `4`, `20`, `10`, `50`, `100`, `200`, `250`. Hardcoded struct offsets like `ptr[40]`, `ptr[80]`, `v2[24]` replace field access. String literals referenced via opaque globals (`&g_4025f9`, `&$d.72`).

**4. Noise Reduction:** Score: 4/10  
Justification: Frame pointer setup (`v0 = &v1`) repeated in nearly every function. Compiler runtime calls (`__cxa_atexit`, `__do_global_dtors_aux`) mixed with application logic. Redundant variable assignments (`v0 = v2; v1 = v3;`) throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Some loop structures recovered (do-while in `test_cpp_constructor`), but `goto` statements used instead of proper control flow (`goto g_413ff8`). Many test functions are empty stubs returning immediately without actual implementation.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized and class methods are properly separated. Thunk functions for virtual dispatch correctly identified (`non-virtual thunk to MultiDerived::funcB()`). Template instantiations created as separate functions.

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom with reasonable structure. Some `goto` statements disrupt flow. Exception handling creates non-linear paths with `_Unwind_Resume`. Generally traceable with effort.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Frame pointer setup pattern repeated identically across functions. Some duplicate variable assignments. Thunk functions show similar patterns. No obvious dead code removal but significant repetition exists.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Extensive use of `void*` instead of specific class types. `unsigned long long` used generically throughout. Original types like `size_t`, `int32_t` lost. Const correctness completely absent. Template type information only visible in mangled names.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs defined as `struct_0`, `struct_1` with generic `field_0` naming. No field names recovered from original classes. Class layouts not reconstructed—pointer arithmetic (`ptr[40]`, `ptr[80]`) replaces field access. Array sizes sometimes correct (`char v1[15]`) but semantics lost.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts: `(long long)`, `(char *)`, `(unsigned int)`, `(void**)`. Pointer arithmetic on void pointers throughout. Type confusion in virtual calls (`(*(v0))(&v0, 5)`). No const correctness preserved. High risk of undefined behavior.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no API documentation. Only function headers with addresses. Original source comments completely lost.

**13. Idiom Recognition:** Score: 4/10  
Justification: Virtual dispatch patterns partially recognized (thunk functions created). Exception handling visible (`__cxa_throw`, `__cxa_begin_catch`). Template instantiation recognized but not explained. No design pattern identification or naming.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags anywhere. Indirect calls presented confidently without uncertainty warnings. No indication of potential issues with vtable recovery or this-pointer adjustments. Code appears certain even when clearly degraded.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. Exception contracts not recovered. Original comments explaining expected values (e.g., "期望: 10 + 34 + 4660 = 4704") completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. Global variables referenced (`g_4139a8`, `g_4139e8`) without usage explanation. Virtual dispatch relationships partially visible through thunks. No call graph documentation.

**17. Module Organization:** Score: 3/10  
Justification: All functions in single output file with no logical grouping. Runtime functions (`_init`, `_start`, `_fini`) mixed with application code. No namespace structure. Class methods scattered without organization.

**18. Build System Compatibility:** Score: 3/10  
Justification: References external symbols (`operatornew[]`, `__cxa_atexit`, `strlen`) without definitions. Would not compile standalone. Many `extern` declarations without corresponding implementations. Dependencies not documented.
