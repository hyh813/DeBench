# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v1`, `v2`, `result`, `str`) with zero semantic meaning. For example, `int64_t v1` in `test_cpp_exception` should indicate exception object, and `str` in `test_cpp_member_func` is actually a char array for "Test".

**2. Function Naming Quality:** Score: 4/10  
Justification: Functions retain mangled C++ names (`_Z20test_cpp_member_funcv`) with demangling comments, but many don't reflect actual functionality since they return constants (e.g., `_Z21test_cpp_virtual_funcv` just returns 42).

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation (`0x125c`, `40`, `80`, `0x11d58`). String literals stored as hex-encoded globals (`g14 = "CPP-L3-01: %d (\xe6\x9c\x9f...)"`) instead of readable strings.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive compiler runtime artifacts (`register_tm_clones`, `deregister_tm_clones`, `frame_dummy`, `__do_global_dtors_aux`). Multiple wrapper functions for standard library calls (`function_d80` for `puts`, `function_ea0` for `printf`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Exception handling converted to broken switch statement instead of try/catch blocks. Most test functions return constant values rather than executing actual logic, completely losing the original control flow.

**6. Function Decomposition:** Score: 5/10  
Justification: Individual functions are appropriately sized, but wrapper functions for library calls create unnecessary fragmentation. Class methods are separated but implementations are incorrect.

**7. Code Linearization:** Score: 5/10  
Justification: Most functions read top-to-bottom, but `test_cpp_exception` has confusing switch-based control flow with fall-through cases that don't match the original nested try/catch structure.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Multiple wrapper functions for same library calls (`function_d80`, `function_d90`, `function_ea0` all wrap standard functions). Runtime initialization code duplicated across multiple helper functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `int64_t`, `int32_t`, or `void*`. Template functions lose all type information (`_Z12template_maxIdET_S0_S0_` uses `int128_t v1` uninitialized). Pointer arithmetic lacks proper typing.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Vtable structures partially recovered (`vtable_11d58_type`, `vtable_11d80_type`) but class layouts incomplete. Container class structures not properly reconstructed (offsets 40, 80 used without struct definition).

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe casts (`(int64_t *)v1`, `(float64_t)*(float128_t *)"Test"`). Template functions use uninitialized variables (`int64_t v1; return v1`). Pointer arithmetic without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated demangling comments present ("Demangled: test_cpp_member_func()"). No algorithm explanations, no semantic documentation, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: C++ idioms completely lost - RAII patterns, virtual dispatch, exception handling, template instantiation all unrecognized. No design pattern identification.

**14. Anomaly Indication:** Score: 2/10  
Justification: Broken code presented confidently without warnings. Functions returning constants instead of logic have no anomaly flags. Uninitialized variable usage not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Error conditions unexplained. Expected values from source comments completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function call relationships visible but virtual dispatch connections unclear. Global variable usage (`g4`, `g5` for vtables) not explained. Callback relationships not documented.

**17. Module Organization:** Score: 4/10  
Justification: Functions grouped by address rather than logical purpose. No header file structure. Related class methods scattered throughout file without namespace organization.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code references undefined symbols (`_ZNSt8ios_base4InitC1Ev`, `__cxa_*` functions). Template instantiations broken. Would require extensive stubbing to compile.

- **Exception Handling Broken**: Try/catch converted to switch statement that doesn't preserve exception semantics
- **Template Functions Unusable**: Use uninitialized variables (`int64_t v1; return v1`) making them non-functional
- **Variable Naming**: All variables generic (`v1`, `v2`, `result`) with zero semantic meaning

### Most Needs Improvement:

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover actual control flow instead of constant folding** - Functions should implement logic, not return expected constants
2. **Add anomaly flags for uncertain decompilation** - Mark switch-based exception handling and uninitialized variable usage as potentially incorrect
3. **Infer semantic variable names from usage patterns** - Variables used in size checks should be named `size`, pointer variables should indicate their target type
