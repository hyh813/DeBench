# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (v1, v2, v3, result, str) with no semantic meaning. In `_Z18test_cpp_exceptionv`, variables v1-v13 obscure the exception handling logic that should show `exception_value`, `catch_result`, etc.

**2. Function Naming Quality:** Score: 4/10  
Justification: Functions retain mangled C++ names (`_Z20test_cpp_member_funcv`) instead of readable names like `test_cpp_member_func`. Helper functions use mechanical names (`function_1670`, `function_1680`) with no semantic meaning.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals preserved (test output messages in g27-g39), but magic numbers like `42`, `100`, `200` in exception handling lack explanation. Vtable addresses shown as hex without context.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack check failures (`__stack_chk_fail()`) appear throughout as compiler artifacts. Redundant assignments visible (`v3 = v2`, `v5 = v2` patterns) but not severely polluted.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Extensive goto labels (`lab_0x1840`, `lab_0x1848`) replace natural control structures. Exception handling converted to explicit state machine with jumps. At -O0, should have much more natural if/try-catch flow.

**6. Function Decomposition:** Score: 5/10  
Justification: Many functions are stubs returning constants (`test_cpp_multiple_inheritance` returns 71 directly, `test_cpp_lambda` returns 85). Virtual function implementations mostly empty. Template logic lost.

**7. Code Linearization:** Score: 5/10  
Justification: Mixed linearity - some functions read top-to-bottom, but exception handling has complex jump patterns. Stack check failures create artificial branches disrupting flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant variable assignments visible throughout. Vtable setup code repeated across similar class functions. Some optimization artifacts visible despite -O0 compilation.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of int64_t/int32_t instead of original C++ types. Class `this` pointers are generic `int64_t *`. Template type information lost (Container<int> vs Container<double> distinguished only by mangled names). No const correctness visible.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Vtable structures defined but with generic field names (e0, e1, e2, e3). Class structures not recovered (no SimpleClass, LifecycleClass definitions). Inheritance relationships noted in comments but not in type system.

**11. Type Safety:** Score: 5/10  
Justification: Many implicit casts between int64_t and int32_t. Pointer arithmetic visible without type safety (`*(int32_t *)(v2 + 8)`). Some unsafe patterns but no obvious buffer overflow vulnerabilities.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Some demangled function names in comments ("Demangled: test_cpp_member_func()"). Line range information present. No algorithm explanations or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: C++ patterns (vtables, exceptions, RTTI) visible but not named or explained. Exception handling pattern visible through `__cxa_*` calls but not identified as such. No design pattern recognition.

**14. Anomaly Indication:** Score: 3/10  
Justification: No explicit anomaly warnings. Stack check failures present but not flagged as compiler artifacts. Functions returning constants without explanation. No confidence indicators for uncertain decompilation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented. No postconditions inferred. Exception contracts not recovered. No invariant recognition for class methods.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call graph visible through direct calls. Vtable relationships shown in global variables (g2-g12). Inheritance noted in comments ("base classes: RTTIBase"). Callback relationships not explicit.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped by source file. Global variables at top, function prototypes separated. But no logical grouping by C++ feature/test category.

**18. Build System Compatibility:** Score: 7/10  
Justification: Includes standard headers (stdint.h, stdio.h, stdlib.h, string.h). External function declarations present. Would need significant work to compile (missing C++ runtime, exception handling support).

**Overall Readability Score: 4.5/10**

### Critical Defects:
- **Variable names completely generic**: `v1`, `v2`, `result` throughout - should recover semantic names like `exception_value`, `instance_count`, `multiplier`
- **Mangled function names preserved**: `_Z20test_cpp_member_funcv` instead of `test_cpp_member_func` - decompiler should demangle C++ symbols
- **Control flow uses goto labels**: `lab_0x1840`, `lab_0x1848` instead of natural if/try-catch structures, especially problematic at -O0
- **Class structures not recovered**: No SimpleClass, LifecycleClass, Base, Derived definitions - only vtable structs with generic fields (e0, e1, e2, e3)
- **Many functions return constants**: `test_cpp_multiple_inheritance` returns 71 directly, `test_cpp_lambda` returns 85 - actual logic completely lost

### Best Quality Aspect:
- **Build System Compatibility (7/10)** - Standard headers included, external declarations present, structure is compilable with modifications despite missing C++ runtime

### Most Needs Improvement:
- **Assertion & Contract Recovery (2/10)** - No preconditions, postconditions, or invariants documented; exception contracts completely missing despite being central to the code

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement C++ name demangling** - Convert mangled names (`_Z20test_cpp_member_funcv`) to readable names automatically using Itanium C++ ABI demangling
2. **Recover class structures from vtable layouts** - Analyze vtable function pointers and this-pointer usage to reconstruct class definitions with meaningful field names
3. **Convert exception handling patterns to try-catch** - Recognize `__cxa_*` call sequences and recover high-level try-catch-finally structures instead of goto-based state machines
