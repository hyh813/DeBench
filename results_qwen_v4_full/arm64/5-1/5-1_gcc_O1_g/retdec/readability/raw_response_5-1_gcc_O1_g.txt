# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `v3`, `str`, `result`, `a1`, `a2`) with no semantic meaning. Original source had meaningful names like `capture_by_value`, `instance_count`, `multiplier` that are completely lost.

**2. Function Naming Quality:** Score: 4/10  
Justification: Functions use mangled C++ names (`_Z20test_cpp_member_funcv`) with demangled comments, but many helper functions are generically named (`function_1490`, `function_14a0`). Original source had clear, descriptive names.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound (`0x125c`, `0x13958`, `0x1dbc`) without explanation. String literals are present but Chinese characters appear as hex escapes. Some constants like `71` in `test_cpp_multiple_inheritance` are hardcoded returns.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts including redundant wrapper functions (`function_1490` just calls `_Znam`), scattered `__stack_chk_fail()` calls, and unnecessary type casts throughout the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Extensive use of `goto` labels (`lab_0x1a84`, `lab_0x1a8c`) instead of structured loops and conditionals. Exception handling converted to complex jump chains that obscure the original try-catch logic.

**6. Function Decomposition:** Score: 3/10  
Justification: Many functions are incomplete stubs that just return (`test_cpp_operator_overload`, `test_cpp_lambda`, `test_cpp_template_class` all just `return;`). Original class methods scattered as separate functions without cohesion.

**7. Code Linearization:** Score: 4/10  
Justification: Heavy goto usage disrupts top-to-bottom reading. Exception handling in `test_cpp_exception` creates particularly convoluted jump patterns that make execution flow difficult to trace.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Multiple wrapper functions that simply call other functions (e.g., `function_1510` â†’ `_Znwm`). Duplicate vtable structures (`g1`, `g2`, `g3`, `g4`) add verbosity without clarity.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Heavy degradation to `int64_t` and `int32_t` throughout. Original C++ class types completely lost. Pointer types often just `int64_t*` instead of proper class pointers. `typedef int64_t int128_t` is incorrect.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Some vtable structures recovered (`vtable_13958_type`, etc.) but class member layouts lost. No field names for class data members. Original classes like `SimpleClass` with `value` and `name[32]` not reconstructed.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(int64_t)(int32_t)`, `(int64_t *)v1`). Pointer arithmetic without type safety. The incorrect `int128_t` typedef could cause serious issues.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: Demangled function names and line ranges provided in comments (e.g., `// Line range: 41 - 50`), but no algorithm explanations or logic documentation. Comments are auto-generated metadata, not semantic guidance.

**13. Idiom Recognition:** Score: 2/10  
Justification: C++ idioms completely unrecognized. RAII patterns, smart pointer semantics (`unique_ptr`), lambda captures, and template instantiation patterns all lost. No design pattern identification.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomaly flags for uncertain decompilation. Code confidently presents potentially incorrect logic (e.g., `test_cpp_rtti` returns `result` which is a pointer, not the expected int sum).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition/postcondition documentation. Original function contracts (e.g., `getValue() const` returns `value`) not recovered. No invariant recognition for class state.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but relationships unclear. Virtual dispatch partially recovered through vtables (`g1`, `g2`), but inheritance relationships not documented. No call graph visualization.

**17. Module Organization:** Score: 4/10  
Justification: All code in single file with no logical grouping. Original source organized by C++ feature (constructors, virtual functions, templates) is lost. No header file separation.

**18. Build System Compatibility:** Score: 4/10  
Justification: Has includes but many external C++ runtime dependencies (`__cxa_*`, `_Znwm`, `_ZdlPvm`). Would require significant stubbing to compile standalone. No dependency documentation.
