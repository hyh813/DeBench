# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`) with no semantic meaning. Original source had meaningful names like `temp1`, `sum`, `count`, `result` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core test function names preserved well (`sequential_ops`, `single_if`, `recursion_factorial`), but helper functions use generic names (`sub_400954`, `sub_400a30`) and runtime functions are standard boilerplate.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive throughout: `4294967295` (0xFFFFFFFF), `1717986919` (division constant), `4294967294`. Original had clear literals (10, 5, 2) that are now obscured by optimization artifacts.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: `__stack_chk_guard` checks, `arm64g_calculate_condition` calls, `__ROL__` macros, frame pointer manipulation (`v0 = &v1`), and redundant type casts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic if-else preserved but loops converted to awkward do-while with complex conditions. `switch_fallthrough` uses goto with `LABEL_400e3c`. At -O0, should have near-perfect recovery but `arm64g_calculate_condition` obscures logic.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately sized and cohesive. Helper functions properly separated. Some wrapper functions exist (`frame_dummy`, `sub_400b08`) but no massive monolithic functions.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but forward jumps disrupt flow (`goto LABEL_400e3c` in switch_fallthrough). Most functions follow linear execution but some have awkward jump patterns.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Sign extension operations repeated throughout (`| a0 >> 31) & 4294967294`). No obvious dead code but optimization artifacts create redundant-looking patterns.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: original `int` becomes `unsigned long long` or `unsigned int`. `sequential_ops` parameters changed from `int` to `unsigned int`. No const-correctness preserved. Function pointer signatures degraded.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Arrays represented as pointer arithmetic (`v6 = &v1` in loop_break). Stack variables shown as `char` arrays with offsets. No struct recovery needed for this code but array semantics lost.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout. `call_func_ptr_array` has incorrect function pointer signature. Sign extension issues create potential undefined behavior. Pointer arithmetic without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: ZERO comments in decompiled code. Original had extensive documentation explaining each function's purpose, test cases, and expected behavior. No decompilation confidence indicators present.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. State machine functions (`state_machine`, `fsm_func_table`) exist but not recognized as such. Loop patterns not identified or named.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomaly warnings despite obvious issues. `nested_if_2` uses undefined variable `v0` but no warning issued. Confidently presents potentially incorrect decompilation without uncertainty markers.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original had test validation comments with expected inputs/outputs. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls present but not documented. Global variables referenced without explanation (`g_413f78`, `g_414018`). No call graph information. Callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with no logical grouping comments. Static functions properly marked. No header file structure or namespace organization.

**18. Build System Compatibility:** Score: 4/10  
Justification: Requires extensive stubbing: `arm64g_calculate_condition`, `__stack_chk_guard`, `__ROL__` macros. Platform-specific AARCH64 code. Many undefined global references.

**Critical Defects:**
- Variable names entirely generic (`v0`, `v1`, `a0`) losing all semantic meaning from original (`temp1`, `sum`, `count`)
- Magic numbers pervasive without explanation (`4294967295`, `1717986919`, `4294967294`)
- Compiler artifacts obscure logic (`arm64g_calculate_condition`, `__ROL__`, `__stack_chk_guard`)
- Type precision severely degraded (`int` → `unsigned long long` incorrectly)
- Zero comments despite original having extensive documentation for each function
- `nested_if_2` contains bug: uses undefined variable `v0` instead of parameter `a1`

**Best Quality Aspect:** Function Naming Quality (scored 7/10)  
- Core test function names preserved accurately from source (`sequential_ops`, `recursion_factorial`, `state_machine`)

**Most Needs Improvement:** Semantic Transparency (averaged 2.5/10)  
- Complete absence of comments, no pattern recognition, no anomaly warnings, no contract recovery

**Top 3 Recommendations for Decompiler Improvement:**
1. **Implement semantic variable naming** - Analyze usage patterns to infer meaningful names (e.g., variables used in loop counters should be named `i`, `count`, `index`)
2. **Detect and annotate compiler artifacts** - Mark `arm64g_calculate_condition`, `__ROL__`, sign-extension patterns as compiler artifacts with explanations of original intent
3. **Recover documentation from debug symbols** - When `-g` flag is present, extract and preserve original comments, function documentation, and test case information
