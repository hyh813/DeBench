# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`v0`, `v1`, `v2`, `a0`, `a1`) instead of semantic names like `temp1`, `sum`, `count`, `i`, `j` from original source. Example: `sequential_ops` uses `v2 = a0 + a1` instead of `temp1 = a + b`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core test functions preserved original names (`sequential_ops`, `single_if`, `nested_if_2`), but helper functions use mechanical names (`sub_400894`, `sub_400970`). Good recovery of intentional function names.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere: `4294967294` (UINT_MAX-1), `4294967295` (UINT_MAX), `305419896` (0x12345678) appear without explanation. Original had no such opaque constants.

**4. Noise Reduction:** Score: 5/10  
Justification: Excessive bit operations for signed/unsigned handling: `(v2 * 2 | v2 >> 31) & 4294967294` instead of simple `temp1 * 2`. Frame pointer manipulation (`v1 = &v2`) adds noise throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic if/else and loops recovered well (`loop_for_fixed`, `nested_if_2`), but switch statements converted to `goto &(&g_400d70)[$d.1[a0]]` jump tables instead of structured switch-case.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly; no inappropriate inlining or fragmentation. Each original function remains separate with reasonable size.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but switch functions use goto-based dispatch that disrupts linear flow. Loop structures generally well-linearized.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant signed/unsigned conversion patterns repeated in every function. Example: `__ROL__(a0, 1) & 4294967294` appears multiple times for simple `x * 2`.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Original `int` degraded to `unsigned int`/`unsigned long long` inconsistently. Function pointers lose signature precision (`unsigned long long *a0` instead of `int (*f)(int)`).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays lost completely—`int arr[] = {10, 20, 30, 40, 50}` becomes `v2 = 3169126501123975826038943907850` with pointer arithmetic `*((int *)((char *)&v2 + 4 * i))`.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe pointer casts throughout: `*((long long *)((char *)&v0 + 8 * a0))` in `call_func_ptr_array`. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**13. Idiom Recognition:** Score: 3/10  
Justification: Control flow patterns (loops, conditionals) recognized structurally but not labeled. No identification of patterns like "factorial calculation" or "digit counting."

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about problematic patterns (goto-based switches, magic numbers, type conversions). Code presented confidently despite significant transformations.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for factorial). Error conditions not flagged. Original test validation comments completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===
