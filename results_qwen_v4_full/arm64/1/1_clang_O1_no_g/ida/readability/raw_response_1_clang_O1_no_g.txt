# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a1`, `a2`, `a3`, `v1`, `v2`) with no semantic meaning. Original source had meaningful names like `temp1`, `temp2`, `sum`, `count` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most test functions preserved original names (`sequential_ops`, `nested_if_deep`, `loop_for_fixed`). However, helper functions like `sub_880`, `sub_174C`, `sub_1754` have generic address-based names.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0xFFFFFFFFLL` (for -1), `0x12` (for 18), `dword_1EE0` array references lack explanation. Some literals like `10`, `100`, `2` are inferable from context.

**4. Noise Reduction:** Score: 5/10  
Justification: Excessive casts (`(unsigned int)`, `(unsigned __int64)`, `LODWORD()`), compiler intrinsics (`__OFSUB__`), and temporary variables (`v1`, `v2`, `v3`) clutter the code throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: `sequential_ops` recovered cleanly, but `single_if` uses bit shift trick `(a1 << (a1 > 0))` instead of clear if. `switch_fallthrough` uses goto labels. `loop_for_fixed` replaced with formula instead of loop structure.

**6. Function Decomposition:** Score: 7/10  
Justification: Function granularity is appropriate; helper functions (`double_value`, `triple_value`) preserved separately. No excessive inlining, but some anonymous functions exist (`sub_174C`).

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `duffs_device` and `loop_multi_exit` have disruptive goto patterns (LABEL_5, LABEL_7, LABEL_8, LABEL_11) that break linear flow.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Redundant casts throughout. `loop_for_fixed` replaced iteration with closed-form formula (optimization artifact). `switch_small` uses array lookup `dword_1EE0[a1]` obscuring original switch structure.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Return types often wrong (`__int64` instead of `int`). `if_else` returns `bool` instead of `int` (semantic change!). `ternary_op` has wrong parameter name (`result` instead of `a`). Function pointers use generic `_QWORD` types.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Arrays referenced as `dword_18D8`, `dword_1EE0`, `unk_18F0`. Function pointer tables as `off_12D30`, `off_12D48`. No struct recovery; all composite types lost to raw memory references.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout. Function pointer calls use generic `_QWORD` losing type information (`call_func_ptr`, `call_func_ptr_array`). Signed/unsigned confusion in multiple functions.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**13. Idiom Recognition:** Score: 3/10

**14. Anomaly Indication:** Score: 4/10

**15. Assertion & Contract Recovery:** Score: 2/10

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===
