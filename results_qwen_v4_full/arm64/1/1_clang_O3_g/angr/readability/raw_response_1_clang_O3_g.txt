# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `v2`, `a0`, `a1`) with no semantic meaning. Original names like `temp1`, `sum`, `count`, `result` are completely lost, making it difficult to track variable purpose across functions like `loop_while` and `sequential_ops`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core test function names are preserved (`sequential_ops`, `single_if`, `nested_if_deep`, `switch_fallthrough`), which is excellent. However, helper functions use mechanical names (`sub_400894`, `sub_400970`, `sub_4009fc`) losing all semantic meaning.

**3. Literal Clarity:** Score: 2/10  
Justification: Pervasive magic numbers like `4294967295` (0xFFFFFFFF), `1717986919` (division by 10 multiplier), `4294967294` obscure logic. No named constants recovered; optimization artifacts like `__ROL__(a0 * 1717986919, 1)` replace simple `x / 10` operations.

**4. Noise Reduction:** Score: 4/10  
Justification: Contains compiler artifacts: `__ROL__` macros, `arm64g_calculate_condition()` calls, `[D] unsupported_Iop_ZeroHI96ofV128()` placeholders, and redundant bit masking (`& 4294967295`, `>> 31` for sign extension) throughout arithmetic operations.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple loops destroyed - `loop_while` and `loop_dowhile` use complex bit manipulation instead of `x = x / 10`. `loop_break` incorrectly converted to switch statement. `duffs_device` has chaotic goto labels. Well-recovered: `if_else`, `nested_if_deep`, `ternary_op`.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but test functions (`test_control_flow_l1/2/3`) are constant-folded - printf statements show hardcoded expected values instead of actual function calls, destroying test semantics entirely.

**7. Code Linearization:** Score: 5/10  
Justification: Mixed quality - `nested_if_deep` reads naturally top-to-bottom, but `duffs_device` has 6+ goto labels (`LABEL_4010c0` through `LABEL_4010e0`) disrupting flow. `call_func_ptr` uses `goto a0` instead of proper indirect call.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Optimization artifacts create apparent redundancy - repeated sign extension patterns (`| x >> 31`) and bit masking (`& 4294967294`) throughout. `sequential_ops` has redundant OR operations that obscure simple `(a+b)*2-c` logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Significant type degradation - `int` becomes `unsigned int`/`unsigned long long` inconsistently. `volatile` qualifiers lost (critical for `infinite_loop`, `loop_external_state`). Function pointers degraded to `unsigned long` (`call_func_ptr` parameters).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays completely lost - `loop_break`'s `int arr[] = {10,20,30,40,50}` converted to switch statement. `switch_small` references global `g_401a20[4]` instead of local array. No struct recovery attempted (none in source).

**11. Type Safety:** Score: 3/10  
Justification: Critical type safety violations: `call_func_ptr` uses `goto a0` for function pointer call (undefined behavior). Multiple implicit conversions between `int`/`unsigned`/`long long`. `conditional_func_ptr` uses `goto v4` for indirect call without proper typing.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation (difficulty levels, test cases, scene labels). No decompilation confidence indicators or uncertainty markers for problematic recoveries.

**13. Idiom Recognition:** Score: 3/10  
Justification: Basic if/else recognized but optimization idioms not: division-by-constant multiplication not identified, loop patterns converted to bit operations, `switch_fallthrough` partially recognized but nested if used instead of proper fall-through semantics.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some anomalies marked (`[D] unsupported_Iop_ZeroHI96ofV128()`, `/* do not return */`) but critical issues unflagged: `goto a0` in `call_func_ptr`, incorrect `loop_break` switch conversion, constant-folded test functions not warned about.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original had implicit contracts (e.g., `duffs_device` requires `n > 0`, `goto_backward` computes factorial) completely lost in decompilation.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Test functions don't actually call target functions - constant-folded to expected values. `test_control_flow_l1` prints `21` instead of calling `sequential_ops(5,7,3)`. Global variable references (`g_401a20`, `g_412ff8`) lack context.

**17. Module Organization:** Score: 4/10  
Justification: Functions ordered by address not logical grouping. Original's clear sections (L1/L2/L3 control flow features) lost. Related functions (`state_idle`, `state_processing`, `fsm_func_table`) scattered throughout output.

**18. Build System Compatibility:** Score: 3/10  
Justification: Does not compile standalone - contains `[D] unsupported_Iop_ZeroHI96ofV128()` placeholders, undefined `arm64g_calculate_condition()`, external globals without declarations. Requires significant patching.
