# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Internal variables use generic names like `v1`, `v2`, `result`, `v3` throughout (e.g., `loop_continue` has `v1` through `v14`). Parameter names are preserved (`a`, `b`, `c` in `sequential_ops`), but no semantic inference was attempted for temporaries.

**2. Function Naming Quality:** Score: 8/10  
Justification: Original function names are well-preserved (`sequential_ops`, `single_if`, `nested_if_deep`). Some runtime functions use generic names (`sub_880`, `_do_global_dtors_aux`), but user code retains meaningful names.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x12`, `0x65`, `0x1A00`, `0x19CC` appear without explanation. Some are inferable (`100` for loop bounds), but SIMD constants (`0x400000004LL`, `0x800000008LL`) are opaque.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe SIMD vectorization noise in loop functions (`int32x4_t`, `vaddq_s32`, `vmulq_s32`, `vbicq_s8`). Architecture-specific types (`__int64`, `__fastcall`, `w8`, `q0`) clutter the code extensively.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple loops converted to mathematical formulas (`loop_for_fixed` → formula, `loop_nested` → `n * m`). `switch_fallthrough` uses goto labels (`LABEL_5`). Some functions like `nested_if_deep` retain natural if-chain structure.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved correctly, but test functions print hardcoded values instead of calling actual functions (e.g., `test_control_flow_l1` prints "21" instead of calling `sequential_ops(5,7,3)`).

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `switch_fallthrough` has goto jumps, and SIMD loops have complex nested do-while structures that disrupt linear reading.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Aggressive loop unrolling and SIMD vectorization create massive redundancy (`loop_continue` is 60+ lines of vector operations for a simple odd-sum loop). Original simple logic is obscured.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `unsigned int`), but architecture-specific types dominate (`__int64`, `int32x4_t`, `int8x16_t`). Function pointer types are correct but verbose with `__fastcall` annotations.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Limited composite types in source. Array in `loop_break` converted to switch statement. No struct/union recovery needed for this codebase, but pointer arithmetic is minimal.

**11. Type Safety:** Score: 7/10  
Justification: No unsafe casts observed. Type conversions are explicit (`(unsigned int)`, `(unsigned __int64)`). Function pointer calls maintain correct signatures.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no comments beyond function headers. No algorithm explanations, no decompilation confidence indicators. Test functions lack explanation for hardcoded values.

**13. Idiom Recognition:** Score: 4/10  
Justification: Basic patterns (if-else chains, switches) recognized, but loop idioms lost to formula substitution. No pattern naming (e.g., "digit counting loop" for `loop_while`).

**14. Anomaly Indication:** Score: 4/10  
Justification: Failed decompilations noted at end ("FAILED to decompile: _setjmp"). However, semantic anomalies (test functions not calling actual functions, `call_func_ptr_array` logic errors) are not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or error handling documentation. Functions like `duffs_device` lack "requires: n > 0" annotations despite having such checks.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but test functions don't actually invoke tested functions. `call_func_ptr_array` has broken logic (switches on `x` instead of `idx`), breaking call graph semantics.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in source order. No logical grouping by feature. Runtime functions mixed with user code. No header file separation.

**18. Build System Compatibility:** Score: 5/10  
Justification: External functions (`printf`, `puts`, `longjmp`) declared but not defined. Architecture-specific types require ARM NEON headers. Would need significant patching to compile standalone.

**Overall Readability Score:** 5.1/10

### Critical Defects:
- **Test functions print hardcoded values instead of calling actual functions** - `test_control_flow_l1` prints "21" directly instead of `sequential_ops(5,7,3)`, defeating verification purpose
- **SIMD vectorization obscures simple loop logic** - `loop_continue` (simple odd-sum) becomes 60+ lines of NEON intrinsics (`vaddq_s32`, `vbicq_s8`, etc.)
- **`call_func_ptr_array` has broken semantics** - Switches on parameter `x` instead of `idx`, completely wrong logic from source
- **Loop-to-formula conversion loses original intent** - `loop_for_fixed` and `loop_nested` become mathematical expressions, hiding iterative nature
- **Zero semantic comments** - No explanation of algorithm purpose, optimization artifacts, or decompilation confidence

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Original function names preserved accurately (`sequential_ops`, `nested_if_deep`, `switch_fallthrough`), enabling quick identification

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Virtually no comments explaining logic, algorithms, or decompilation uncertainty; critical for understanding transformed code

### Top 3 Recommendations for Decompiler Improvement:
1. **Detect and flag test function anomalies** - When test functions contain hardcoded expected values instead of actual function calls, add warning comments indicating potential decompilation artifact
2. **Provide decompilation mode options** - Offer "readability mode" that prefers loop reconstruction over formula substitution and avoids aggressive SIMD recovery for simple scalar loops
3. **Add semantic confidence annotations** - Mark regions where decompilation is uncertain (e.g., "// DECOMPILER: SIMD recovery may not match original scalar logic") to guide manual review
