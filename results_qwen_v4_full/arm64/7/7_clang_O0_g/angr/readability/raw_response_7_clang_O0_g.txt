# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Most variables use generic names like `v0`, `v1`, `v2` instead of semantic names (e.g., `result`, `str`, `encrypted`). Some functions have better names like `flag1`, `cur`, `i`, `j`, but these are exceptions rather than the rule.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions preserved original names (`param_fake_branch`, `call_opaque_predicate`), but many helper functions use mechanical names (`sub_400774`, `sub_400870`). Function names don't indicate parameter types or purpose consistently.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `3735928559` (0xDEADBEEF) and `2863311530` (0xAAAAAAAA) appear without explanation. String literals like `"test"` are preserved, but encrypted data references (`&$d.2`) are opaque.

**4. Noise Reduction:** Score: 5/10  
Justification: Every function contains unnecessary frame pointer setup (`v0 = &v1; v2 = &v3`). Runtime initialization code (`_init`, `register_tm_clones`, `__do_global_dtors_aux`) clutters the output without clear purpose.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loop structures mostly preserved (for/while), but some conditionals are inverted awkwardly (e.g., `if (0 <= strlen(v0)) return v1; v1 += 1000;` instead of original `if (strlen(str) < 0)`). No excessive goto usage.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved correctly, but granularity is inconsistent. Some functions are trivial wrappers (`call_fake_branch` is 5 lines), while runtime helpers are fragmented unnecessarily.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with minimal forward jumps. Early returns are used appropriately. No spaghetti control flow observed.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Frame setup code repeated in every function. Some expressions duplicated (e.g., `register_tm_clones()` called twice in `frame_dummy`). No dead code elimination visible.
