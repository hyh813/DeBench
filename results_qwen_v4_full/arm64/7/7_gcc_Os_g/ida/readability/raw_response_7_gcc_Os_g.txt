# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`v0`, `v1`, `v2`, `v6`, `v7`) with no semantic meaning. Original source had meaningful names like `result`, `encrypted`, `decrypted`, `buffer` that are lost in decompilation.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most user-defined function names preserved accurately (`param_fake_branch`, `param_opaque_predicate`, `decrypt_string`). Only runtime/entry functions use generic names (`sub_860`, `start`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers appear without explanation (`0x20u`, `90`, `0x10u`, signal numbers `8` and `11`). Original source had comments explaining XOR key `0x5A` and signal constants.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack annotations (`[xsp+28h] [xbp+28h] BYREF`) and type casts (`(__sighandler_t)`, `(__int64)`) add visual noise. 14 external functions failed to decompile, cluttering output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops recovered as `while`/`for` constructs naturally. However, `param_tail_call_optimized` converted from recursion to iteration (optimization artifact), and dead branches eliminated entirely.

**6. Function Decomposition:** Score: 7/10  
Justification: Function granularity preserved well. However, `param_buffer_overflow_stack` reduced to empty `void` function (dead code elimination), losing original intent.

**7. Code Linearization:** Score: 8/10  
Justification: Code reads top-to-bottom with minimal forward jumps. Early returns used appropriately. No excessive `goto` usage disrupting flow.

**8. Redundancy Elimination:** Score: 8/10  
Justification: Dead code from fake branches properly eliminated. Constant folding applied (`call_fake_branch` returns `10` directly). No obvious redundant expressions remain.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Generic `int` used instead of `int32_t`. Some unsigned conversions visible (`(unsigned int)x`, `(unsigned __int8)`). Function pointer types degraded to generic casts.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Bitfield struct from `param_implementation_defined` completely optimized away (returns constant `48`). Array parameters preserved but no struct definitions visible in output.

**11. Type Safety:** Score: 6/10  
Justification: Unsafe casts present (`(__sighandler_t)`, `(__int64)`). Signal handler casts are necessary but type-unsafe. No obvious buffer overflow patterns in decompiled code.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**13. Idiom Recognition:** Score: 4/10

**14. Anomaly Indication:** Score: 4/10

**15. Assertion & Contract Recovery:** Score: 2/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===
