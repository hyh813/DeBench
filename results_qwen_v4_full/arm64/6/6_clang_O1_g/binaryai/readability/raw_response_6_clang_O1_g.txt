# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`param_1`, `param_2`, `local_30`, `iVar1`, `uVar2`) with no semantic meaning. Original source had meaningful names like `dst`, `src`, `buffer`, `fd`, `shm` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names are well-preserved from source (`param_strcpy`, `call_strcmp`, `param_fork_exec`). Action verbs and semantic purpose are clear, matching original intent closely.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive: `0x38`/`0x39` for syscall numbers, `0x2a` for success code (42), `0xffffffff` for -1. String comparison uses XOR with hex constants (`0x68636e65426e6942`) instead of readable `strcmp`.

**4. Noise Reduction:** Score: 7/10  
Justification: Minimal compiler artifacts; some unnecessary casts (`(undefined8 *)`, `(ulong)`) and stack variable declarations (`local_30`, `uStack_58`) add noise but don't severely impede reading.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Control flow uses standard if-else and do-while constructs. At `-O0` optimization, flow is reasonably natural, though some functions use awkward do-while patterns instead of for loops.

**6. Function Decomposition:** Score: 8/10  
Justification: Function granularity matches source well; each original function is recovered separately. No monolithic functions or over-fragmentation observed.

**7. Code Linearization:** Score: 7/10  
Justification: Code reads mostly top-to-bottom with minimal forward jumps. Some functions like `param_shmget_shmat` use goto labels (`LAB_00102434`) that disrupt flow slightly.

**8. Redundancy Elimination:** Score: 6/10  
Justification: `param_fread_fwrite` uses XOR comparison instead of `strcmp` for string verification—redundant complexity. Some constant folding in `call_*` functions loses original logic transparency.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `size_t`→`ulong`, `pid_t`→`__pid_t`/`int`, `FILE*` preserved but `char*` loses const-correctness. Original `const char *src` becomes `char *param_2`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Structs poorly recovered: `sockaddr` shows only `sa_family` and `sa_data[14]` array. `ThreadData` struct not recovered—fields accessed via array indexing (`param_1[0]`, `param_1[1]`, `param_1[2]`).

**11. Type Safety:** Score: 5/10  
Justification: Frequent unsafe casts: `(undefined8 *)`, `(pthread_t *)`, `(int *)((ulong)&local_70 | 8)`. Pointer arithmetic on generic types without bounds checking visible.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no comments beyond function headers. No explanation of magic numbers, syscall numbers, or complex logic like the XOR string comparison in `param_fread_fwrite`.

**13. Idiom Recognition:** Score: 3/10  
Justification: Standard library patterns (strcpy, strcmp) recognized through function calls, but no explicit pattern identification. Producer-consumer pattern in condition variable test not documented.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about suspicious patterns. XOR string comparison, direct TLS register access (`tpidr_el0`), and atomic intrinsic calls (`__aarch64_ldadd4_acq_rel`) presented without anomaly flags.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition documentation (e.g., "requires: param_1 != NULL"). Error return values (-1, -2, -3) not documented with their meanings.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 7/10  
Justification: Function call relationships clear through direct calls. External function stubs properly declared. Global variables (`shared_counter`, `atomic_counter`, `signal_received`) visible but usage not documented.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single flat namespace. No logical grouping by category (library functions vs. syscalls vs. threads). Original section comments lost.

**18. Build System Compatibility:** Score: 8/10  
Justification: Code compiles with external dependencies declared. External function stubs present for all libc/syscall functions. Would need minimal patching to compile standalone.
