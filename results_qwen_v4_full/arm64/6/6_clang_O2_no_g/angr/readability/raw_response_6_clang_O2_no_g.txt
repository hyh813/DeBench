# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (`v0`, `v1`, `v2`, `dst`, `len`) with no semantic meaning. Original source had meaningful names like `buffer`, `ptr`, `iterations`, `shared_counter` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Core test functions retain original names (`param_strcpy`, `call_strcmp`), but helper functions use mechanical names (`sub_4012f4`, `$x`). Some function names are preserved but parameters are often missing or incorrect.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere: `4294967295` (appears 50+ times as -1), `18446744073709551608`, `34359738360`. No named constants recovered. String literals preserved but numeric constants completely opaque.

**4. Noise Reduction:** Score: 4/10  
Justification: Stack frame setup code pervasive (`v0 = &v1` patterns in every function). Register spill variables clutter logic. Some functions have `[D] unsupported_Iop_ZeroHI96ofV128()` artifacts that should be eliminated.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `param_signal_handling` uses complex condition tracking (`cc_op`, `cc_dep1`, `cc_dep2`) instead of simple boolean checks. Some functions use `goto LABEL_401c54` instead of natural loop constructs.

**6. Function Decomposition:** Score: 6/10  
Justification: Functions are properly separated but some are nonsensical (e.g., `call_strcpy` calls `strlen(&v0)` twice instead of calling `param_strcpy`). Thread helper functions (`thread_compute`, `thread_sum`) properly extracted.

**7. Code Linearization:** Score: 5/10  
Justification: Most code reads top-to-bottom but complex functions like `param_malloc_free` and `param_thread_local_storage` have multiple goto labels disrupting flow. Deep nesting in signal handling code.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Many functions call the same operation twice (e.g., `call_strcpy` returns `strlen(&v0)` after already computing `len = strlen(&v0)`). `call_fread_fwrite` calls `param_fread_fwrite` twice unnecessarily.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types degraded to `unsigned long long` instead of `size_t`, `int`, `char*`. `param_strlen` has no parameters in decompiled version (should have `const char*`). Pointer types often become `void*` or `char*`.

**10. Composite Type Recovery:** Score: 5/10  
Justification: Some structs recovered (`struct sockaddr`, `struct FILE`) but with generic field names (`field_0`, `field_5`, `padding_0`). TLS structure partially recovered but loses semantic field names like `tls_counter`, `tls_thread_name`.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(char *)ptr + v7`, `(volatile int*)&atomic_counter`). Pointer arithmetic uses `char*` casting extensively. `param_scanf` missing input string parameter creates type confusion.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments explaining logic. Only type hints present (`Other Possible Types: unsigned int`). No algorithm explanations, API contracts, or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library functions recognized (`strcpy`, `strlen`, `printf`, `pthread_create`). Thread patterns partially identified but no design pattern documentation. Atomic operations recognized as `__aarch64_ldadd4_acq_rel`.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some functions marked "Failed to decompile" (`__aarch64_cas4_acq_rel`). `[D]` markers for unsupported operations present. No explicit warnings about uncertain decompilation or parameter mismatches.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions or postconditions documented. Error returns use magic numbers (4294967295 for -1) without explanation. No contract information about null checks or valid ranges recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. Global variables have generic names (`g_414ff8`, `g_4031a4`, `g_40349c`). No call graph or callback relationship documentation.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in roughly logical order (library functions, syscalls, threads). No header file structure. Test functions grouped but mixed with implementation functions.

**18. Build System Compatibility:** Score: 4/10  
Justification: Code has type definitions needing cleanup. External dependencies declared but some functions missing parameters. Would require significant modification to compile standalone (missing `signal_handler` declaration in some contexts).

**Overall Readability Score:** 4.0/10

### Critical Defects:
- **Parameter loss**: `param_strlen`, `param_scanf`, `param_fopen_fclose` missing required parameters in decompiled version
- **Magic number explosion**: `4294967295` appears 50+ times instead of `-1` constant
- **Variable naming**: All variables use `v0`, `v1`, `v2` pattern losing all semantic meaning
- **Redundant calls**: Functions like `call_strcpy` compute same value twice instead of calling intended function
- **Type degradation**: `size_t`, `int`, `char*` all collapsed to `unsigned long long`

### Best Quality Aspect:
- **Function Decomposition (6/10)** - Thread helper functions (`thread_compute`, `thread_sum`, `thread_increment`) are properly extracted as separate functions with reasonable cohesion

### Most Needs Improvement:
- **Variable Naming Quality (2/10)** - Complete loss of semantic variable names makes understanding data flow nearly impossible without cross-referencing original source

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement usage-based variable renaming** - Analyze how variables are used (e.g., passed to `strlen` → `string_ptr`, used in loop counter → `index`) to infer semantic names
2. **Recover constant definitions** - Detect repeated magic numbers (`4294967295`, `42`, `18446744073709551608`) and suggest named constants with comments
3. **Validate function signatures against call sites** - Cross-reference function declarations with actual call patterns to detect and fix missing parameters before output
