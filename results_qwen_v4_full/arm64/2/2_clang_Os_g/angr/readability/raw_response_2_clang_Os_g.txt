## Decompilation Readability Assessment

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `v2`, `a0`, `a1` with no semantic meaning. Only occasional loop counters like `i`, `ptr`, `cur` provide minimal context, making it extremely difficult to track variable purposes across functions.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core processing functions retained original names (`process_char`, `array_1d_stack`, `ptr_single`), but many helper functions degraded to `sub_400634`, `sub_400708`, and `$x`. Test functions preserved names but their logic was constant-folded.

**3. Literal Clarity:** Score: 4/10  
Justification: Pervasive magic numbers (4294967295, 18446744073709551614, 2251799813685247) without explanation. Test functions show hardcoded expected values (e.g., `printf("DT-L1-01...: %c\n", 97)`) instead of actual function calls, obscuring the computation flow.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: frame pointer setup (`v0 = &v1`), register annotations (`x0`, `x9`, `w8`), and architecture-specific intrinsics (`__ROL__`, `arm64g_calculate_condition`). ### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loops converted to do-while with manual counter management instead of natural for/while constructs. Function pointer calls use `goto a0` instead of proper call syntax (`ptr_func_simple`, `struct_func_ptr`). `enum_switch` uses array lookup instead of switch statement.

**6. Function Decomposition:** Score: 6/10  
Justification: Original function boundaries mostly preserved, but test functions collapsed into constant printing (losing actual computation). Runtime functions (`__addtf3`, `__multf3`) became monolithic 500+ line blocks that should be library calls.

**7. Code Linearization:** Score: 4/10  
Justification: Excessive goto statements and labels (`LABEL_400db0`, `LABEL_40143c`, `LABEL_401518`) fragment execution flow. The `__addtf3` function has deeply nested conditionals with forward jumps that make linear reading nearly impossible.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions show aggressive constant folding that hides actual computation logic. Repeated bit manipulation patterns in `__addtf3`/`__multf3` suggest missed common subexpression elimination. Some expressions appear multiple times without consolidation.

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation: `char` becomes `unsigned int`, `short` becomes `unsigned int`, `float`/`double` functions return `long long`. Pointer types degraded to `unsigned long` or `void*`. Signed/unsigned distinctions frequently incorrect (e.g., `process_int` returns `unsigned int` instead of `int`).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered as generic `struct_0`, `struct_1` with meaningless field names (`field_0`, `field_8`, `padding_4[4]`). Original semantic field names lost (e.g., `Point3D.x` becomes `field_0`). Unions not properly distinguished from structs in several cases.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe casts throughout: `(unsigned int)ptr`, `(char*)a0`, `(int)(long)handle`. Pointer arithmetic on generic `unsigned long` types creates undefined behavior risk. Function pointers invoked via `goto` bypass type checking entirely.

### === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments explaining algorithm intent or function purpose. Only compiler-generated stack offset annotations (`// [bp-0x10]`). No decompilation confidence indicators for uncertain reconstructions. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns not identified: linked list traversal appears as raw pointer chasing, array summation not recognized as reduction pattern, switch-on-enum converted to array lookup without pattern annotation. No design pattern detection present.

### === LEVEL 5: CONTEXTUAL COHERENCE ===

**14. Anomaly Indication:** Score: 2/10  
Justification: Broken functions like `process_float` and `process_double` return uninitialized values without any warning. `goto a0` for function pointer calls not flagged as suspicious. No indication of decompilation uncertainty or potential errors.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `ptr != NULL` requirements). No postconditions or return value guarantees. Error conditions not documented. Original source's implicit contracts completely lost.

Justification: Function call relationships visible through code but not documented. Global variables referenced as `g_413ff8`, `g_402f1f` without context or purpose. No call graph documentation or callback relationship identification.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output without logical grouping by category (data types, arrays, pointers, composites). No header file structure or API declarations. Static functions not properly scoped with visibility indicators.

**18. Build System Compatibility:** Score: 5/10
