# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are overwhelmingly generic (`param_1`, `local_14`, `iVar1`, `uVar2`) with no semantic meaning. For example, `test_data_types_l1` uses `local_18`, `local_14` instead of meaningful names like `value`, `vol_value` from the source.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user function names preserved accurately (`process_char`, `array_1d_stack`, `ptr_single`, `struct_simple`). However, some runtime functions remain generic (`FUN_00100680`, `FUN_001007fc`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers pervasive without explanation: `0x41` instead of `'A'`, `0x20` instead of `32`, `0x28` for struct offsets. Character comparisons use hex values obscuring intent.

**4. Noise Reduction:** Score: 7/10  
Justification: Minimal compiler artifacts for O0 build. Some unreachable block warnings present. Redundant type casts visible but not overwhelming.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops recovered as for/while constructs successfully. Test functions have awkward parameter passing but core logic flows naturally. No excessive goto usage.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries well-preserved with appropriate granularity. Original function structure maintained. Helper functions (`__addtf3`, `__multf3`) correctly identified as library calls.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Some forward references in test functions disrupt flow slightly but overall linear reading is possible.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No significant dead code. Some redundant type casts from promotions present but manageable. No loop unrolling artifacts obscuring logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `process_ll` returns `long` instead of `long long`, `process_double` returns `undefined1 [16]` instead of `double`, `process_ld` has void return. Pointer types often `undefined8*` or `long`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - all structs accessed via pointer arithmetic. `struct_simple` takes `int *` instead of `Point3D *`, `struct_nested` takes `int *` instead of `Rect *`. No struct/union definitions recovered.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe casts throughout. `ptr_void` uses `uint *` instead of `void *`. Function pointers degraded to `code *`. Pointer arithmetic on typed pointers loses safety guarantees.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only decompiler metadata comments present. No algorithm explanations, no parameter documentation. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. Linked list traversal, tree recursion, and graph traversal not recognized as idioms. Switch statements recovered but enum values not named.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**14. Anomaly Indication:** Score: 5/10  
Justification: Some warnings present ("WARNING: Type propagation algorithm not settling", "Removing unreachable block"). Anomalies flagged but not comprehensively explained.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Bounds checking in `array_complex_index` present but not documented as contract.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. No call graph information. Callback relationships (`ptr_func_simple`, `struct_func_ptr`) not clearly indicated.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output file. Original logical grouping (data types, arrays, pointers, composites) lost. No header file structure.

**18. Build System Compatibility:** Score: 3/10  
Justification: Does not compile cleanly - wrong function signatures, missing type definitions, undefined symbols (`__multf3`, `__addtf3`). Requires significant patching.

**Overall Readability Score: 5.0/10**

### Critical Defects:
- **Composite types completely lost** - All structs/unions accessed via raw pointer arithmetic (`param_1[1]`, `param_1[2]`) instead of field names
- **Type precision severely degraded** - `long long` becomes `long`, `double` becomes `undefined1 [16]`, `long double` becomes `void`
- **Variable names entirely generic** - `local_14`, `iVar1`, `uVar2` provide zero semantic information
- **Magic numbers everywhere** - `0x41`, `0x20`, `0x28` instead of `'A'`, `32`, struct offsets
- **No contract documentation** - Bounds checks exist but preconditions not documented

### Best Quality Aspect:
- **Function Decomposition (8/10)** - Original function boundaries and granularity well-preserved; each function maintains single responsibility

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions; all composite access reduced to pointer arithmetic losing all field semantics

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct layouts from access patterns** - Detect consistent offset patterns (`+0`, `+4`, `+8`) and propose struct definitions with field names
2. **Preserve original type information from debug symbols** - When `-g` flag present, use DWARF info to recover exact types instead of inferring from usage
3. **Infer variable names from usage context** - Variables used in size comparisons should be named `size`/`length`, loop counters should be `i`/`j`/`index`
