# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: ALL variables use generic names (`v1`, `v2`, `v3`, etc.) with zero semantic meaning. Even simple functions like `double_value` use `v1` instead of `x`, making it impossible to understand variable purposes without tracing every usage.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names ARE preserved (`process_char`, `array_1d_stack`, `struct_simple`), which is excellent. However, helper functions use mechanical names (`function_6f0`, `function_700`) and some signatures are incorrect.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere (`0xfffffffe`, `0x100000000`, `0x8000000000001`) with no explanations. String literals stored as opaque globals (`g5`, `g6`) instead of inline constants.

**4. Noise Reduction:** Score: 4/10  
Justification: Contains massive compiler runtime code (`__addtf3`, `__multf3`, `__sfp_handle_exceptions`) mixed with user code. Assembly intrinsics (`__asm_ubfx`, `__asm_mrs`) pollute the output without explanation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: All for-loops converted to while-loops with manual increments. Floating-point functions have HUNDREDS of goto labels (`lab_0x1bb4`, `lab_0x1900`) creating spaghetti code. Simple functions like `process_char` use bizarre modulo arithmetic instead of clear conditionals.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved, but signatures are broken (most show `void` parameters instead of actual parameters). Runtime functions mixed with user code without clear separation.

**7. Code Linearization:** Score: 4/10  
Justification: Heavy use of forward jumps disrupts reading flow. The `__addtf3` function spans 1500+ lines with complex label jumping. Simple functions are more readable but still use awkward patterns.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundant masking operations (`& 0xfffffffe`, `& 0xffffffff`) from optimization. No obvious dead code in user functions, but runtime code has significant redundancy.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Catastrophic type degradation - `char`, `short`, `int`, `long` ALL become `int64_t`. Float/double functions return `int64_t` instead of floating types. No const/volatile preservation (`volatile_access` loses volatile semantics).

**10. Composite Type Recovery:** Score: 1/10  
Justification: COMPLETE FAILURE - ALL structs (`Point3D`, `Rect`, `Widget`), unions (`UnionData`), and enums (`State`) are lost. Struct access becomes raw pointer arithmetic (`*(int32_t *)(v2 + 8)`). Bitfields completely unrecoverable.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout. Pointer arithmetic without type information. Function pointers lost (e.g., `ptr_func_simple` shows no function pointer parameter). Many operations would cause undefined behavior.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: ZERO comments in decompiled code. No algorithm explanations, no decompilation confidence indicators, no API documentation. Original source had extensive comments; all lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition whatsoever. Loop idioms not recognized (for→while). Switch statements sometimes recovered but often become complex if-else chains. No design pattern identification.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags despite obvious problems. `process_float` returns `int64_t` confidently instead of float. `process_double` returns a string pointer address. No warnings about unresolved calls or uncertain recovery.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. Error conditions not identified. NULL checks present but not explained as safety measures.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but with wrong signatures. Global variables named generically (`g1`, `g2`...`g11`) losing all semantic meaning. No call graph documentation.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file without logical grouping. Runtime functions mixed with user code. No header file separation. Function prototypes listed but not organized by category.

**18. Build System Compatibility:** Score: 3/10  
Justification: Will NOT compile - references undefined assembly functions (`__asm_movi`, `__asm_ubfx`, `__asm_bfi`). Custom typedefs (`int80_t`, `int128_t`) not standard. Requires extensive patching to compile.

**Overall Readability Score: 3.4/10**

### Critical Defects:
- **All variables named `v1`, `v2`, `v3`** - zero semantic information, makes code practically unusable for analysis
- **Complete loss of composite types** - structs, unions, enums all reduced to pointer arithmetic
- **Type system collapsed** - everything becomes `int64_t`, float/double operations broken
- **Zero comments** - no explanation of logic, algorithms, or decompilation confidence
- **Non-compilable output** - references undefined assembly intrinsics requiring extensive patching

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Original function names mostly preserved (`process_char`, `array_1d_stack`, `struct_simple`), providing some semantic anchors in otherwise degraded code.

### Most Needs Improvement:
- **Composite Type Recovery (1/10)** - Complete failure to recover ANY struct, union, or enum definitions, forcing all composite access into unreadable pointer arithmetic.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement variable name inference** from usage patterns, parameter positions, and string references (e.g., `v1` in `process_char` should be `c` based on char operations)
2. **Recover composite type definitions** by analyzing consistent memory access patterns and field offsets to reconstruct struct/union layouts with field names
3. **Add decompilation confidence annotations** to flag uncertain recoveries (e.g., "// UNCERTAIN: float operation may be incorrect") instead of presenting broken code confidently
