# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic type-hinted names (`iVar1`, `uVar5`, `piVar1`) with no semantic meaning; "Unresolved local var: int sum@[???]" comments indicate the decompiler couldn't recover original names like `sum` or `i`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user functions preserved original names (`process_char`, `array_1d_stack`, `struct_simple`), but runtime functions use generic names (`FUN_00100620`, `FUN_0010077c`).

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions show constant folding with hex literals (`0x61` instead of `'a'`, `0x400c000000000000` for float 3.5); magic numbers obscure intent.

**4. Noise Reduction:** Score: 5/10  
Justification: Frequent "Unresolved local var" comments, compiler runtime functions (`__do_global_dtors_aux`), and verbose floating-point helpers (`__addtf3` with 500+ lines) add significant noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions natural (`process_char`), but optimized loops heavily unrolled (`array_1d_stack` processes 8 elements per iteration); `binary_tree_sum` converted from recursion to do-while loop.

**6. Function Decomposition:** Score: 6/10  
Justification: Original function boundaries preserved, but test functions bloated with constant-folded code; floating-point helpers (`__multf3`) excessively large for simple operations.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom naturally, but loop unrolling creates repetition that disrupts flow; minimal forward jumps.

**8. Redundancy Elimination:** Score: 4/10  
Justification: `array_3d` has 60+ temporary variables (`uVar1` through `uVar60`); aggressive loop unrolling creates massive code duplication without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types accurate (`int`, `char`, `long`), but `const` qualifiers lost (`const_param` shows `int *p` not `const int *p`); function pointers use generic names (`_func_int_int`).

**10. Composite Type Recovery:** Score: 8/10  
Justification: Excellent struct recovery with field names (`Point3D`, `Rect`, `Widget`); nested access preserved (`w->bounds.position.z`); unions and enums correctly identified.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe casts (`(undefined8 *)`, `(ulong)`); function pointer calls have warnings ("Could not recover jumptable"); no buffer overflow patterns introduced.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: "Unresolved local var" and "WARNING" comments indicate uncertainty, but no algorithm explanations or API contract documentation; original source comments completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Data structure patterns (linked list, tree) recognizable from code structure, but no explicit pattern naming or algorithm identification.

**14. Anomaly Indication:** Score: 7/10  
Justification: Good uncertainty signaling ("WARNING: Removing unreachable block", "Could not recover jumptable", "Unresolved local var"); decompiler honestly indicates recovery limitations.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No precondition/postcondition documentation; null checks present in code but not documented as contracts; error conditions undocumented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible within decompiled code, but no call graph documentation; callback relationships not explicitly noted; global variable usage unclear.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping; runtime functions mixed with user code; original source section organization lost.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates compilation succeeds, but requires stubs for runtime functions; compiler-specific types (`long_double`, `undefined8`) reduce portability.

**Overall Readability Score: 5.5/10**

### Critical Defects
- **Variable names completely generic** (`iVar1`, `uVar5`, `piVar1`) obscuring semantic intent like `sum`, `index`, `array_ptr`
- **Test functions constant-folded** eliminating actual function call logic (e.g., `printf("DT-L1-01", 0x61)` instead of calling `process_char('A')`)
- **Aggressive loop unrolling** in `array_3d` creates 60+ temporary variables making code unmaintainable
- **Original source comments completely lost** - no documentation of function purposes or test scenarios
- **`const` qualifiers stripped** from pointer parameters losing important semantic information

### Best Quality Aspect
- **Composite Type Recovery (8/10)** - Struct/union definitions excellently recovered with accurate field names and nested access patterns preserved (`w->bounds.position.z`)

### Most Needs Improvement
- **Assertion & Contract Recovery (3/10)** - No precondition/postcondition documentation despite clear validation patterns in source code (null checks, bounds checking)

### Top 3 Recommendations for Decompiler Improvement
1. **Implement semantic variable name inference** - Analyze usage patterns to suggest names like `sum` for accumulators, `index` for loop counters, `array_ptr` for pointer parameters
2. **Preserve original source comments in debug builds** - When `-g` flag is present, extract and embed DWARF debug information comments
3. **Add optimization level awareness** - At `-O0`, avoid loop unrolling recovery; present code closer to source structure when debug symbols available
