# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `a1`, `result`, `v3`) with no semantic meaning. Original source had meaningful names like `sum`, `current`, `head`, `len` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most user functions retained original names (`process_char`, `array_1d_stack`, `struct_simple`), but infrastructure functions are generic (`sub_6D0`, `init_proc`). Good preservation of core function identities.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `65` (for 'A'), `0x19u` (range check), and `xmmword_3420` (long double constant) lack explanation. Test functions show pre-computed expected values instead of actual computations.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive compiler artifacts: register annotations (`// x0`, `// x1`), redundant casts (`(unsigned int)`, `(__int64)`), `LODWORD(result)` macros, and stack variable declarations with `BYREF` annotations clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple functions have natural flow, but `binary_tree_sum` transformed from clean recursion into 400+ lines of nested do-while loops with 40+ local variables. Loop structures often use do-while instead of for.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but `binary_tree_sum` became a 400+ line monstrosity. Test functions were over-optimized to print constants instead of computing values. Helper function `complex_callback` is missing.

**7. Code Linearization:** Score: 5/10  
Justification: Simple functions read linearly, but complex functions have excessive forward references and deeply nested structures. `binary_tree_sum` has 15+ levels of nesting that should be recursive.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated type casts throughout, register variable declarations create redundancy, and test functions show computed values that should be runtime calculations. Optimization artifacts create duplicate code paths.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Severe type degradation: `char` becomes `__int64` in `process_char`, pointer types are `_DWORD *` instead of specific types, function pointers use generic signatures (`__int64 (__fastcall *a1)(_QWORD)`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - structs accessed via array indexing (`a1[0]`, `a1[1]`, `a1[2]`) instead of field names (`p->x`, `p->y`, `p->z`). No struct definitions recovered. Bitfields shown as bit manipulation (`(*a1 & 1) + ((*a1 >> 1) & 3)`).

**11. Type Safety:** Score: 3/10  
Justification: Many unsafe casts throughout, const correctness lost (`const_param` shows `_DWORD *` not `const int *`), volatile semantics lost (`volatile_access` just shows `2 * *a1`), pointer arithmetic on generic types.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no useful comments - only register annotations (`// x0`). Original source had extensive documentation explaining each test case that is completely lost. No algorithm explanations.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition - linked list traversal, tree recursion, and string length calculation not identified. Common C idioms transformed into opaque pointer arithmetic.

**14. Anomaly Indication:** Score: 3/10  
Justification: Failed decompilations noted at end (`FAILED to decompile: __libc_start_main`) but no contextual warnings. Optimized test functions not flagged as suspicious (printing constants vs. computing).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original test expectations (e.g., "期望: 'a'") completely lost. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. No call graph information. Callback relationships unclear (function pointers show generic types). Global variable usage not explained.

**17. Module Organization:** Score: 3/10  
Justification: All functions in single output with no logical grouping. Original had clear section organization (Data Types, Arrays, Pointers, Composite Types) that's lost. Static functions not properly scoped.

**18. Build System Compatibility:** Score: 3/10  
Justification: 7 external functions marked "FAILED to decompile". Would need significant stubs to compile. Architecture-specific code (arm64 registers) present. Would not compile standalone without major work.

**Overall Readability Score: 3.9/10**

### Critical Defects:
- **Composite types completely lost** - `struct_simple` shows `*a1 + a1[1] + a1[2]` instead of `p->x + p->y + p->z`
- **Variable names are all generic** - `v1`, `v2`, `result` provide no semantic information
- **`binary_tree_sum` decompilation failure** - 400+ lines of nested loops instead of clean 5-line recursion
- **Test functions optimized away** - Print pre-computed constants instead of actually calling functions
- **No comments recovered** - Original extensive documentation completely lost

### Best Quality Aspect: **Function Naming Quality (7/10)**
- Most user-defined function names were preserved correctly from the original source, maintaining semantic identity.

### Most Needs Improvement: **Composite Type Recovery (2/10)**
- Complete failure to recover struct/union definitions - all composite types accessed via raw pointer arithmetic losing all field semantics.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct definitions from access patterns** - Detect consistent offset patterns (`+0`, `+4`, `+8`) and propose struct layouts with field names
2. **Preserve recursion in tree/graph functions** - Recognize recursive call patterns and restore recursive structure instead of iterative expansion
3. **Infer variable names from usage context** - Variables used in size checks should be named `size`/`count`, loop indices should be `i`/`j`/`k`
