# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `uVar2`, `lVar3`) with no semantic meaning. Original source had meaningful names like `sum`, `len`, `current`, `head` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately (`process_char`, `array_1d_stack`, `ptr_single`, `struct_simple`). Some runtime functions have generic names (`FUN_00100620`) but core logic functions match source.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere: `0x20` (should be `32` for char conversion), `0x41` (should be `'A'`), `0x2c`, `0x1b8` for array offsets. No named constants recovered from original code.

**4. Noise Reduction:** Score: 7/10

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: For loops converted to do-while patterns with manual decrement (`uVar2 = uVar2 - 1; ... while (uVar2 != 0)`). At `-O0` this restructuring is unexpected and reduces readability.

**6. Function Decomposition:** Score: 7/10  
Justification: Function granularity preserved well; test functions remain separate. No inappropriate inlining or over-fragmentation. Parameter lists are reasonable.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with minimal forward jumps. Some complex functions (`__addtf3`, `__multf3`) have excessive branching but core logic is linear.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Some redundant variable assignments present but no significant dead code. Loop unrolling not present (appropriate for `-O0`).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Integer types often degraded to generic `int`. Long double becomes `undefined8`. Float operations use `undefined4`. Bool preserved in `process_bool` but pointer types frequently become `void*` or `undefined*`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - All struct/union types lost. `Point3D` accessed as `param_1[0], param_1[1], param_1[2]`. `Widget`, `Node`, `Graph` all reduced to pointer arithmetic. No field names recovered.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts (`(undefined4*)`, `(int**)*param_1`). Function pointers degraded to `code*`. Pointer arithmetic without type safety throughout struct access functions.
