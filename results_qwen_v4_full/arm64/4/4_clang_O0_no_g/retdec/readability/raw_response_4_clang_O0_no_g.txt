# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`v1`, `v2`, `v3`, `temp`, `result`) with zero semantic meaning. For example, `varargs_func` uses `v1` through `v12` instead of meaningful names like `count`, `sum`, or `args_ptr`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`cdecl_func`, `varargs_func`) but many degraded to `function_6f0`, `function_700`. Critical issue: function signatures lost all parameters (e.g., `cdecl_func(void)` instead of `cdecl_func(int a, int b)`).

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers pervasive (`0xffffffff`, `0xfffffffe`, `0x100000000`) without explanation. String literals stored as unnamed globals (`g7`, `g8`, `g9`) instead of meaningful constants.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive goto labels (`lab_0xc74`, `lab_0xcc4`, `lab_0xcd8`), redundant type casts throughout, and compiler artifacts like `& 0xffffffff` masking operations clutter every function.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: All loops converted to goto chains instead of `for`/`while`. Example: `varargs_func` uses 6 goto labels for what should be a simple `for` loop. Switch statements partially recovered but with fall-through issues.

**6. Function Decomposition:** Score: 4/10  
Justification: Function boundaries preserved but logic fragmented. Many functions are essentially stubs (`call_cdecl` just calls `cdecl_func`), and parameter passing is broken making decomposition meaningless.

**7. Code Linearization:** Score: 3/10  
Justification: Heavy forward/backward jumps prevent top-to-bottom reading. `varargs_func` has 4 separate label blocks that jump non-sequentially, making execution flow impossible to trace linearly.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Repetitive patterns in variable initialization (`v1`, `v2`, `v3` declared identically across functions). Similar goto structures duplicated in `varargs_func` and `param_varargs` without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Everything degraded to `int64_t` or `int32_t`. Original types like `char*`, `double`, `LargeStruct` completely lost. Function pointers become `int64_t` with no signature information.

**10. Composite Type Recovery:** Score: 1/10  
Justification: **Complete failure** - all struct definitions (`LargeStruct`, `SmallStruct`, `SmallPoint`, `TestIntPair`) are missing. Struct access reduced to pointer arithmetic (`*(int32_t *)(v2 + 4)`).

**11. Type Safety:** Score: 2/10  
Justification: Numerous unsafe casts visible (`(int64_t)v1`, `(int32_t)*v2`). Pointer arithmetic without bounds checking. Function pointer calls have no type safety (`param_func_ptr()` has no callback parameter).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: **Zero meaningful comments**. Only auto-generated address comments (`// Address range: 0x6b0 - 0x6c8`). No algorithm explanations, no uncertainty indicators, no API documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: Common C idioms completely unrecognized. `va_list` handling in `varargs_func` becomes opaque pointer arithmetic. Struct passing conventions not identified or documented.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about lost parameters, incorrect logic, or uncertain decompilation. Incorrect code presented confidently (e.g., `cdecl_func` shows `2 * v1` instead of `a + b`).

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `func_struct_byptr` should note `p != NULL`). No postconditions, no error handling documentation, no return value contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Function calls visible but signatures mismatched. Global variables named `g1` through `g25` with no semantic meaning. No call graph or relationship documentation.

### Most Needs Improvement: Composite Type Recovery (scored 1/10)
- Complete failure to recover any struct/union definitions, forcing all composite type access through raw pointer arithmetic

1. **Recover function parameter information from calling convention analysis** - ARM64 AAPCS passes first 8 arguments in registers (x0-x7), which should be mapped to named parameters

3. **Replace goto chains with structured control flow** - Detect loop patterns (backward jumps with counter increments) and recover as `for`/`while` constructs with proper loop variables
