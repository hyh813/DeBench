# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (`v1`, `v2`, `v3`) with zero semantic meaning. Original source had descriptive names like `size`, `result`, `counter`, `exec_mem` that are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Main test function names preserved (`param_macro_constants`, `call_asm_basic`), but helper functions use generic naming (`function_7c0`, `function_7d0`). Many functions lost their parameters entirely.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers throughout (`0xdef0`, `512`, `64`, `0xdf22`) without explanation. String literals stored as hex-encoded globals (`g1`, `g2`) rather than readable text. Original macro constants (`MAX_SIZE`, `BUFFER_COUNT`) not recovered.

**4. Noise Reduction:** Score: 5/10  
Justification: Significant compiler artifacts present: initialization functions (`__do_global_dtors_aux`, `register_tm_clones`), architecture-specific intrinsics (`__aarch64_ldadd4_acq_rel`), and unnecessary casts (`& 0xffffffff`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Control flow present but expressed through convoluted expressions. Example: `param_macro_constants` uses complex conditional expression `v3 == 0 | v3 < 0 != ((v3 ^ v2) & (v2 ^ (int32_t)&g27)) < 0` instead of simple `if-else`.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved with reasonable granularity, but parameter information frequently lost. `param_macro_constants(int size)` becomes `param_macro_constants(void)`.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but contains unnecessary complexity. Some functions have awkward control flow that could be simplified.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repetitive masking operations (`& 0xffffffff`) appear throughout without justification. Similar redundant patterns in multiple functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 4/10  
Justification: Significant type degradation. Most variables default to generic `int64_t`/`int32_t`. Function signatures lose parameters entirely. Pointer types often degraded to `int64_t *`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: No struct recovery visible. Global variables named generically (`g1`, `g2`, `g3`...) with no structure information. Original code had no complex structs but this is still a failure.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts without warnings. Pointer arithmetic present (`*(int32_t *)v3`). No type safety annotations recovered.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled code. Original source had extensive documentation about preprocessing, macros, and assembly that is completely absent.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Macro semantics completely lost (no indication of `MAX_SIZE`, `SQUARE`, `ADD_16`). Architecture-specific code unexplained.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags. Architecture-specific intrinsics (`__aarch64_ldadd4_acq_rel`) appear without explanation of why or what they do.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No function contracts documented. Error handling paths present but unexplained. No precondition/postcondition information.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships visible in code but not documented. Global variable usage not explained. Call graph not provided.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped with prototypes listed upfront, but organization somewhat arbitrary. No logical grouping by feature (preprocessing vs. assembly).

**18. Build System Compatibility:** Score: 5/10  
Justification: Headers present but external dependencies (`__aarch64_ldadd4_acq_rel`) may not compile standalone. Architecture mismatch (original ARM64, decompiler shows ARM64 intrinsics).
