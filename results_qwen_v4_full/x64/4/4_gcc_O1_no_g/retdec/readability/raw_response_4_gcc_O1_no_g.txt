# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`a1`, `a2`, `v1`, `v2`, `temp_1`) with zero semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `str`, `len` that are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Main test functions preserved original names (`cdecl_func`, `varargs_func`, `test_calling_conventions`), but helper functions degraded to `function_1070`, `function_1080`. Better than typical but inconsistent.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere (`0xffffffff`, `0xfffffffe`, `0x100000000`) without explanation. String literals stored in unnamed globals (`g1`, `g2`...`g22`) instead of meaningful constant names.

**4. Noise Reduction:** Score: 3/10  
Justification: Assembly intrinsics pollute code (`__asm_movq`, `__asm_pxor`, `__asm_addsd`). Stack canary checks (`__readfsqword(40)`, `__stack_chk_fail()`) repeated in nearly every function. Redundant bit masking (`& 0xffffffff`) throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loops recovered as while constructs but with awkward variable tracking. Switch statement in `ret_multi_branch` recovered correctly. However, many functions constant-folded to return literals instead of computing.

**6. Function Decomposition:** Score: 4/10  
Justification: Critical failure: `call_cdecl()` returns `15` directly instead of calling `cdecl_func(5, 10)`. Same pattern for `call_stdcall()`, `call_fastcall()`, etc. - destroys call graph relationships entirely.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with reasonable flow. While loops present but with complex index tracking. No excessive gotos, which is positive for -O0 compilation.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Stack canary checks duplicated in ~30 functions. Bit masking operations (`& 0xffffffff`, `& 0xfffffffe`) appear after nearly every arithmetic operation. No dead code elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Almost everything is `int64_t` or `int32_t`. Original `double` parameter in `func_mixed_args` lost (now `int64_t a3`). `char*` sometimes preserved but often as `int64_t`. Function pointer types completely lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `LargeStruct` and `SmallStruct` definitions completely absent. `func_struct_byptr` uses `a1 + 4` pointer arithmetic instead of `p->y`. `func_struct_byval` has no struct parameter - uses raw stack access.

**11. Type Safety:** Score: 3/10  
Justification: Implicit casts throughout via int64_t coercion. Function pointer called via raw address (`param_func_ptr(0x11a9, 5)`). Pointer arithmetic without bounds checking. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation explaining each test case, expected values, and calling convention details. Complete loss of semantic documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: `va_list` pattern not recognized - `varargs_func` shows complex stack manipulation instead of clean `va_start`/`va_arg`/`va_end`. No design pattern identification whatsoever.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about constant-folded functions losing call relationships. No indication that struct types were lost. Decompiler presents incorrect code confidently without flagging uncertainty.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: NULL checks present (`if (a1 == 0)`) but no documentation of preconditions. No postcondition inference. Original's explicit test expectations completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but relationships not documented. Global variables `g1-g22` have no semantic meaning (should be format strings). Call graph destroyed by constant folding.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with prototypes at top. Original had clear section markers (`// === 4.1 调用约定 ===`) completely lost. No logical grouping comments.

**18. Build System Compatibility:** Score: 5/10  
Justification: Has includes and prototypes but assembly intrinsics (`__asm_movq`, `__asm_movaps`) may not compile on all platforms. Some function signatures mismatch original (missing parameters).

**Overall Readability Score:** 3.5/10

### Critical Defects:
- **Constant folding destroyed call relationships**: `call_cdecl()` returns `15` instead of calling `cdecl_func(5, 10)` - makes reverse engineering impossible
- **Complete struct type loss**: `LargeStruct`, `SmallStruct` definitions absent; pointer arithmetic used instead of field access
- **Zero semantic documentation**: No comments despite original having extensive test documentation
- **Assembly intrinsics pollute output**: `__asm_movq`, `__asm_pxor` blocks reduce portability and readability
- **Type precision collapsed**: `double` parameters become `int64_t`; function pointer types lost entirely

### Best Quality Aspect:
- **Function Naming Quality (6/10)** - Main test function names preserved from source, allowing some identification of purpose despite other degradations

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation makes understanding test purposes impossible without original source

### Top 3 Recommendations for Decompiler Improvement:
1. **Disable aggressive constant folding** for functions that call other user-defined functions - preserve call graph structure even when values are known
2. **Recover struct/union types** from stack access patterns and field offset usage (e.g., `a1 + 4` should suggest struct field access)
3. **Add decompilation confidence annotations** to flag uncertain reconstructions (constant-folded functions, lost types, assembly intrinsics)

---

**Note:** This decompilation is from RetDec (per file header), not Ghidra as metadata states. At `-O0` optimization level, expected readability should be 8-10/10. This 3.5/10 score indicates severe decompiler limitations rather than optimization artifacts.
