# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (`a1`, `a2`, `v0`, `v7`, `v8`) with no semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `ptr` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names preserved (`cdecl_func`, `varargs_func`, `test_calling_conventions`), but runtime helpers use generic names (`sub_1020`, `deregister_tm_clones`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x28u` (stack canary offset) unexplained; string literals referenced as `s`, `aCallL206` instead of showing content; constant-folded returns (15, 50, 6) hide actual computation.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: `__readfsqword(0x28u)` stack canary checks, redundant `(unsigned int)` casts, runtime init functions (`__do_global_dtors_aux`), and BYREF annotations clutter code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loops recovered as while-loops in `varargs_func`, but constant-folded test functions hide actual control flow; complex variable tracking in varargs handling disrupts readability.

**6. Function Decomposition:** Score: 5/10  
Justification: Original function boundaries preserved, but `func_struct_byval` has 7 `__int64` parameters instead of 1 struct; many `call_*` functions constant-folded to just return values, losing call relationships.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but `varargs_func` has complex variable shuffling (`v10`, `v11`, `v12`) that disrupts linear reading; forward references to globals present.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Constant folding creates redundancy (test functions just return constants); stack canary code repeated across functions; type casts redundant in many places.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Generic types dominate (`__int64`, `_DWORD`, `_QWORD`); lost original types like `size_t`, `va_list`; `call_varargs_param` incorrectly shows 7 parameters when original took none beyond count.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Critical failure - `LargeStruct` passed as 7 separate `__int64` parameters instead of 1 struct; `SmallStruct *` becomes `_DWORD *`; no struct field names recovered; `func_struct_byval` signature completely wrong.

**11. Type Safety:** Score: 4/10  
Justification: Many unsafe casts (`(unsigned int)`, `(__int64)`, `(_DWORD *)`); function pointer types generic (`unsigned int (__fastcall *)(_QWORD)`); no const correctness; pointer arithmetic without type safety.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments in decompiled code; no algorithm explanations; no decompilation confidence indicators; only function headers with addresses present.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified; variable argument handling recovered but not identified as stdarg pattern; no recognition of common C idioms like array decay or struct passing.

**14. Anomaly Indication:** Score: 5/10  
Justification: One warning present ("positive sp value has been detected"); 7 functions marked "FAILED to decompile"; but no specific warnings about uncertain recoveries within successfully decompiled functions.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented; null checks present in code (`if (a1)`) but not documented as contracts; no postconditions or invariants recognized.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented; no call graph provided; callback relationships visible (`param_func_ptr` → `callback_func`) but not explained.

**17. Module Organization:** Score: 4/10  
Justification: Functions appear in arbitrary order; no logical grouping by test category; runtime functions mixed with user code; no header file structure.

**18. Build System Compatibility:** Score: 4/10  
Justification: 7 external functions failed to decompile (`puts`, `strlen`, `__printf_chk`); contains compiler-specific constructs (`__readfsqword`, `__fastcall`); requires extensive stubs to compile.

**Overall Readability Score:** 4.5/10

### Critical Defects:
- **Struct type information completely lost** - `LargeStruct` decomposed into 7 separate `__int64` parameters instead of single struct parameter
- **Variable names entirely generic** - `a1`, `a2`, `v0`, `v7` provide no semantic information about purpose
- **Constant folding hides actual computation** - Test functions like `call_cdecl()` return `15` directly instead of showing `cdecl_func(5, 10)` call
- **No comments or documentation** - Zero algorithm explanations, confidence indicators, or contract documentation
- **Type precision severely degraded** - Everything becomes `__int64`, `_DWORD`, or `_QWORD` losing original type semantics

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Most original function names preserved (`cdecl_func`, `varargs_func`, `test_calling_conventions`), making it possible to correlate decompiled code with source.

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)** - Virtually no comments exist; adding decompilation confidence indicators and algorithm explanations would significantly improve usability.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover composite types from stack layouts** - Analyze parameter passing patterns to reconstruct struct definitions instead of decomposing into primitive parameters
2. **Infer variable names from usage patterns** - Use data flow analysis to name variables based on their role (e.g., loop counter → `i`, accumulator → `sum`)
3. **Add decompilation confidence annotations** - Mark uncertain recoveries with comments like `// UNCERTAIN: varargs handling may be incorrect` to guide manual review
