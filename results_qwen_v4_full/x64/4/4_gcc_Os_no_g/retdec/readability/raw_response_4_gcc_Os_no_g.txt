# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`a1`, `a2`, `v1`, `v2`, `v3`) with zero semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `str`, `len` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`cdecl_func`, `varargs_func`, `func_struct_byval`), but helper functions use mechanical names (`function_1070`, `function_1080`). Core test functions retain original semantics.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive (`0xffffffff`, `0x100000000`, `0x40091eb851eb851f`) without explanation. String literals stored as hex-encoded globals (`g1`, `g2`) instead of readable constants.

**4. Noise Reduction:** Score: 3/10  
Justification: Stack canary checks (`__readfsqword(40)`, `__stack_chk_fail()`) in nearly every function. Assembly intrinsics (`__asm_movsd`, `__asm_rep_movsd_memcpy`, `__asm_cvtsi2sd`) clutter logic. Redundant bit masking (`& 0xffffffff`) throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loops recovered as while constructs but with awkward variable tracking (`v2 = v2 + 1 & 0xffffffff`). Many functions constant-folded (`call_cdecl()` returns 15 directly) losing original call structure.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved but varargs functions have excessive parameters (`param_varargs` has 6 parameters for what should be 2). Some functions too small due to constant folding.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but stack canary checks at function end disrupt flow. Loop variables tracked through multiple temporaries (`v3`, `v5`, `v6`, `v7`, `v8`) creating non-linear reading.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Bit masking (`& 0xffffffff`, `& 0xfffffffe`) repeated in nearly every return statement. Stack canary pattern duplicated across 20+ functions without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `int64_t` or `int32_t`. Original had `int`, `char*`, `double`, `long long`, `void*` - all lost. No const correctness. Function pointers treated as integers.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct types completely lost (`LargeStruct`, `SmallStruct`, `TestIntPair`). All struct access through pointer arithmetic (`*(int32_t *)(a1 + 4)`). No field names recovered.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe casts throughout (`(int32_t)a1`, `(int64_t)&v2`). Pointer arithmetic without bounds checking. Function pointers passed as raw integers (`param_func_ptr(0x11d9, 5)`).

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled code. Original had extensive documentation explaining each test case. No decompiler confidence indicators or uncertainty warnings.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern not recognized (uses raw stack traversal). No design patterns identified. Assembly intrinsics for floating-point operations not explained as type conversions.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomalies flagged despite assembly intrinsics, stack canary checks, and type degradation. Uncertain decompilation sites presented confidently without warnings.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `str != NULL` checks not annotated). No postconditions inferred. Error conditions silent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but callback relationships unclear (`callback_func` address passed as integer `0x11d9`). Global variables referenced but purpose opaque.

**17. Module Organization:** Score: 4/10  
Justification: All 83 functions in single file with no logical grouping. Original had clear sections (calling conventions, parameter passing, return values) completely lost.

**18. Build System Compatibility:** Score: 5/10  
Justification: Function prototypes present but external dependencies unclear. Assembly intrinsics (`__asm_movsd`, `__asm_hlt`) not portable. Would require significant patching to compile.
