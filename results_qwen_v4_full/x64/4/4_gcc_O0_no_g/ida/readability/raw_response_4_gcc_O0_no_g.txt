# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are generic (`a1`, `a2`, `v1`, `v2`) with no semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `large`, `s` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Core function names preserved well (`cdecl_func`, `varargs_func`, `test_calling_conventions`). Some helper functions use generic names (`sub_1020`, `sub_1030`) but main test functions match source.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x28u` (stack canary offset), `0x2F` (varargs threshold) appear without explanation. String literals referenced as `s`, `aCallL206` instead of inline content.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__readfsqword(0x28u)`) repeated in most functions. Excessive type casts (`(unsigned int)`, `(__int64)`) and frame pointer setup create visual noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loop structures preserved as for-loops. Switch statements converted to if-else chains (`ret_multi_branch`), which is acceptable but less clear than original switch.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved, but struct-by-value functions have excessive parameters (`func_struct_byval` has 7+ `__int64` params instead of 1 struct).

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Stack variable declarations at function start with some forward references, but flow is traceable.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Stack frame setup repeated across functions. Some redundant type casts present but no obvious dead code or duplicate expressions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Original `int` types often become `__int64` with casts. Pointer types partially preserved but degraded (`_DWORD *`, `__int64 *`). Function pointer signatures generic.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Critical failure - `LargeStruct`, `SmallStruct` completely lost. `func_struct_byval` takes 7+ separate `__int64` parameters instead of 1 struct. No field names recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(__int64)`, `(_DWORD *)`, function pointer casts). Pointer arithmetic visible in varargs handling. Risk of undefined behavior.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**13. Idiom Recognition:** Score: 4/10

**14. Anomaly Indication:** Score: 5/10

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `p != NULL` checks not noted). No postconditions or invariants recovered from source assertions.

## === LEVEL 5: CONTEXTUAL COHERENCE ===
