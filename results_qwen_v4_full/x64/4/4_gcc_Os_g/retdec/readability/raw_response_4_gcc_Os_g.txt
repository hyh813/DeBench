# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names (`v1`, `v2`, `a1`, `a2`, `result`) with no semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `len` which are completely lost in decompilation.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved well (`cdecl_func`, `varargs_func`, `test_calling_conventions`). However, helper functions use mechanical names (`function_1070`, `function_1080`) for standard library wrappers.

**3. Literal Clarity:** Score: 4/10  
Justification: String literals stored as hex-encoded globals (`g1 = "=== \xe6\xb5\x8b..."`). Magic numbers pervasive (`0xffffffff`, `0xfffffffe`, `0x100000000`, `48`) without explanation or named constants.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: stack canary checks (`__readfsqword(40)`, `__stack_chk_fail()`), assembly intrinsics (`__asm_movsd`, `__asm_cvttsd2si`), and unnecessary bit masking (`& 0xffffffff`) throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loops converted to while constructs with manual counters. Many functions constant-folded to direct returns (`call_cdecl()` returns 15), losing original computation logic. Varargs uses complex pointer arithmetic instead of `va_list`.

**6. Function Decomposition:** Score: 6/10  
Justification: Original function boundaries mostly preserved, but some over-fragmented (library wrappers as separate functions). Constant-folded functions are trivially small, losing semantic value.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but forward variable declarations disrupt flow. Loop structures use while instead of for, requiring mental translation to understand iteration patterns.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Stack canary checks duplicated in every function with locals. Bit masking operations repeated throughout. Constant folding creates redundancy between computation and return values.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Original `int` types often degraded to `int64_t`. Pointer types frequently become `int64_t` instead of proper pointer types. Float/double handling uses assembly intrinsics instead of proper floating-point types.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Critical failure - `LargeStruct` and `SmallStruct` not recovered. `func_struct_byval` takes `(int64_t a1, int64_t a2, int64_t a3)` instead of `LargeStruct`. `func_struct_byptr` takes `int64_t` instead of `SmallStruct*`.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe implicit casts throughout via `int64_t` usage. Pointer arithmetic on integers instead of proper pointer types. No const correctness preserved. Function pointer signatures partially preserved but degraded.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated module/line range comments present. Original source's extensive documentation (calling convention explanations, test scenarios) completely lost. No algorithm explanations or decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: Varargs pattern not recognized (manual stack traversal instead of `va_list`). Struct passing conventions not identified. Calling convention differences not explained despite being the test's primary purpose.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about constant folding, type degradation, or assembly intrinsics. Silent failures where decompiler confidently presents incorrect or simplified information without flagging uncertainty.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks present but not documented as preconditions. No postconditions inferred. Original function contracts (parameter expectations, return guarantees) completely absent.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`g1`-`g22`) unexplained. Callback relationships partially visible in `param_func_ptr` but not explicitly documented.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with no logical grouping beyond original structure. Helper functions scattered. No namespace or package structure to organize related functionality.

**18. Build System Compatibility:** Score: 6/10  
Justification: Standard headers included. External function declarations present. However, assembly intrinsics (`__asm_movsd`, `__asm_rep_movsd_memcpy`) and custom typedefs (`int128_t`) may not compile on all platforms without modification.

**Overall Readability Score:** 4.3/10

### Critical Defects:
- **Struct types completely lost**: `LargeStruct` and `SmallStruct` degraded to raw `int64_t` parameters, destroying semantic meaning of struct passing tests
- **Variable names all generic**: `v1`, `v2`, `a1`, `a2` provide zero semantic information about purpose (should be `count`, `sum`, `args`, `len`)
- **String literals hex-encoded**: Global strings stored as byte arrays (`g1 = "=== \xe6\xb5\x8b..."`) instead of readable text
- **Varargs pattern not recognized**: Manual stack pointer arithmetic instead of `va_list`/`va_start`/`va_arg`/`va_end` idiom
- **Assembly intrinsics unexplained**: `__asm_movsd`, `__asm_cvttsd2si` used without comments explaining their purpose

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Core function names preserved accurately from source, maintaining semantic identity of test functions

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct definitions, converting structured data to raw integer parameters

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover composite types from stack access patterns** - Detect struct field access patterns (e.g., `*(int32_t *)(a1 + 4) * (int32_t)a1`) and reconstruct struct definitions with named fields
2. **Recognize standard library idioms** - Detect `va_list` patterns, `strlen` calls, and printf-style formatting to restore high-level C constructs instead of raw assembly intrinsics
3. **Preserve string literal readability** - Output string globals as readable C strings rather than hex-encoded byte arrays, especially for UTF-8 content
