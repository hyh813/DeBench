# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Mixed quality—parameter names preserved from debug info (`c`, `x`, `arr`, `n`) but local variables use generic `v1`, `v2`, `v3` pattern; complex functions like `binary_tree_sum` have 30+ variables named `v21`, `v20`, `v24` obscuring their roles.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most original function names preserved (`process_char`, `array_1d_stack`, `struct_simple`, `test_data_types_l1`); only startup helpers use generic names (`sub_1020`, `sub_1030`).

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers like `32` (char conversion), `65` ('A'), `0x1Au` (range) lack explanation; test functions show pre-computed constants (15, 300, 11) instead of original expressions, losing semantic meaning.

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__readfsqword(0x28u)`) in every test function; excessive type casts (`(_DWORD *)`, `(__int64)`); 6 external functions marked "FAILED to decompile" clutter the output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple loops recover well (`array_1d_stack`, `linked_list`) but `process_char` uses inverted condition `(c - 65) >= 0x1Au` instead of readable range check; `binary_tree_sum` transforms recursion into deeply nested loops.

**6. Function Decomposition:** Score: 7/10  
Justification: Individual functions maintain reasonable granularity; however, test functions inline constant results instead of calling actual functions, losing the test structure entirely.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally; `binary_tree_sum` has 15+ levels of nested loops disrupting flow; test functions maintain linear structure despite constant folding.

**8. Redundancy Elimination:** Score: 6/10  
Justification: `volatile_access` optimized from two reads to `2 * *p` losing volatile semantics; `array_3d` has repetitive pointer arithmetic patterns; test functions eliminate all actual computation through constant folding.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types preserved correctly (`char`, `int`, `float`, `double`, `long double`); some pointer types degrade to `_DWORD *` instead of `int *`; function pointer signatures mostly preserved.

**10. Composite Type Recovery:** Score: 8/10  
Justification: Excellent struct recovery—`Point3D`, `Rect`, `Widget`, `Node`, `DNode`, `TreeNode` all preserved with field names; bitfield struct `Flags` converted to manual bit operations losing semantic clarity.

**11. Type Safety:** Score: 7/10  
Justification: Explicit casts visible throughout (`(_DWORD *)`, `(char *)`) making type conversions apparent; `ptr_void` and `ptr_cast` show void pointer handling; no dangerous buffer patterns introduced.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments explaining logic; all original documentation (test case descriptions, scene markers, star ratings) completely lost; no decompiler confidence indicators or uncertainty markers.

**13. Idiom Recognition:** Score: 5/10  
Justification: Linked list traversal pattern recognizable but unlabeled; tree recursion transformed beyond recognition; enum switch uses jump table (`CSWTCH_61[s]`) without explanation; char case conversion idiom not identified.

**14. Anomaly Indication:** Score: 5/10  
Justification: Failed decompilations marked ("FAILED to decompile"); stack warning present ("positive sp value has been detected"); but `array_oob` undefined behavior not flagged; volatile optimization side effects not warned.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (null checks, bounds validation); `array_complex_index` bounds check preserved but not explained as contract; no postconditions or error condition documentation.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible in code; global string references use opaque names (`s`, `asc_33E1`, `asc_3454`); callback relationships (`ptr_func_simple`, `struct_func_ptr`) not documented.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in source-order sequence; no file/module structure; static functions not clearly scoped; test functions grouped but not separated from implementation.

**18. Build System Compatibility:** Score: 4/10  
Justification: 6 external functions failed decompilation requiring stubs; `__printf_chk`, `puts`, `__libc_start_main` need declarations; does not compile standalone without significant patching.

### Critical Defects:
- **Zero comments**—all original documentation (test descriptions, scene markers, explanations) completely lost
- **`binary_tree_sum` recursion transformed into 15+ level nested loops**, making tree traversal algorithm unrecognizable
- **Test functions constant-folded**—printf calls show computed results (15, 300, 11) instead of actual function calls, losing test structure

- **Bitfield struct `Flags` converted to manual bit operations** (`>> 6`, `& 7`, `& 3`) losing field name semantics

### Best Quality Aspect:
- **Composite Type Recovery (8/10)**—Struct names and field names (`Point3D`, `Rect`, `Widget`, `Node`) preserved excellently from debug info, enabling clear understanding of data layouts.

### Most Needs Improvement:
- **Comment Quality & Presence (2/10)**—Complete absence of any documentation; original source had extensive comments explaining each test case, scene, and expected behavior—all lost.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover and preserve source comments from debug sections**—DW_AT_description and line table comments should be extracted and inserted as function/variable documentation.
2. **Detect recursive patterns and preserve recursion**—`binary_tree_sum` should remain recursive; detect call-to-self patterns and avoid loop transformation for tree traversals.
3. **Flag optimization-induced semantic changes**—When volatile reads are combined or test computations are constant-folded, add warnings like `// OPTIMIZED: original had two volatile reads`.
