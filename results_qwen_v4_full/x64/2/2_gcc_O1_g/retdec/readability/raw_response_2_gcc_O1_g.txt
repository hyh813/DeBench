# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v1`, `v2`, `v3` with no semantic meaning (e.g., `int64_t v1; // 0x122c` in `const_param` should be `value`). Even in well-preserved functions like `double_value`, the parameter `x` is kept but local temporaries are all `v#`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names are preserved (`double_value`, `process_char`, `array_1d_stack`, `struct_simple`), which is excellent. However, helper functions are generically named (`function_1060`, `function_1070`) and some signatures don't match (e.g., `binary_tree` takes no parameters but should take `TreeNode*`).

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers abound without explanation: `0x3fffffffc`, `0x100000000`, `0x3fc0000000000000`. String literals are stored as hex-encoded globals (`g1 = "=== \xe6\xb5\x8b..."`). No constants recovered for values like array sizes (10, 5).

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: `__asm_addss`, `__asm_movsd`, `__readfsqword(40)` for stack canaries. Unnecessary casts everywhere (`(int64_t)(int32_t)v1`). Assembly intrinsics in `process_float`/`process_double` make them unreadable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple for-loops converted to complex while loops with manual pointer arithmetic (e.g., `array_1d_stack` uses `while (v2 != v1 + 4 + (4 * (int64_t)n + 0x3fffffffc & 0x3fffffffc))`). Boolean logic obfuscated (`a5 < a3 != (a4 < a2 == ...)` in `array_complex_index`).

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved correctly. Test functions are large but that matches source. However, `binary_tree` is incorrectly separated from `binary_tree_sum` when it should just call it. Some functions like `ptr_func_simple` return wrong values (`return (int64_t)f` instead of `f(x)`).

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but nested loops in `array_3d` and `graph_traverse` have confusing structure with multiple while loops that should be for-loops. Stack canary checks at function ends disrupt flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions are constant-folded (all `printf` calls have pre-computed values), losing the actual function call semantics. Loop bodies have redundant variable assignments (`v8 = result; v2++` patterns repeated).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 2/10  
Justification: Catastrophic type degradation. `process_float` and `process_double` return `int128_t` instead of `float`/`double`. `process_short` takes `int64_t` parameters instead of `int16_t`. Pointer types lost (`int*` becomes `int64_t`). `volatile` and `const` qualifiers completely missing.

**10. Composite Type Recovery:** Score: 1/10  
Justification: Complete failure. All structs (`Point3D`, `Rect`, `Widget`, `Node`) are reduced to pointer arithmetic with hardcoded offsets (`a1 + 8`, `a1 + 20`). No struct definitions recovered. Unions completely lost (`union_type` uses switch on type parameter instead of union access). Bitfields in `struct_bitfields` use division/modulo instead of bit operations.

**11. Type Safety:** Score: 2/10  
Justification: Numerous unsafe patterns: uninitialized variables used (`int64_t v1; return v1 + 10` in `const_param`), incorrect pointer casts everywhere, `ptr_func_simple` returns function pointer instead of calling it. Many functions would crash if executed.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated metadata comments present ("From module: /work/2.c", "Line range: 6-8"). No semantic explanations of algorithm logic. No indication of decompilation uncertainty despite obvious errors.

**13. Idiom Recognition:** Score: 2/10  
Justification: Common patterns not recognized: linked list traversal appears as pointer arithmetic, not identified as list iteration. Switch-on-enum in `enum_switch` preserved but not documented as state machine pattern.

**14. Anomaly Indication:** Score: 1/10  
Justification: Zero anomaly detection. Functions with uninitialized variables (`const_param`, `volatile_access`, `ptr_single`) presented confidently without warnings. Broken float/double functions have no uncertainty markers.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for array functions). NULL checks present in code but not documented as contracts. No postcondition inference.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call graph mostly intact (test functions call processing functions). However, callback relationships lost (`ptr_func_simple` doesn't actually call the function pointer). Global variable usage unclear (`g1`-`g4` are strings but not documented).

**17. Module Organization:** Score: 4/10  
Justification: All code in single file with no logical grouping. Helper functions (`function_1060`-`1090`) mixed with application logic. No separation between test code and implementation.

**18. Build System Compatibility:** Score: 3/10  
Justification: Has function prototypes but many are incorrect (wrong parameter types, missing parameters). Assembly intrinsics (`__asm_hlt`, `__asm_addss`) won't compile without special headers. Type mismatches would cause compilation errors.

**Overall Readability Score: 3.4/10**

### Critical Defects:
- **Composite types completely lost**: All structs/unions reduced to pointer arithmetic with magic offsets (`a1 + 8`, `a1 + 20`)
- **Type precision catastrophic**: `float`/`double` functions return `int128_t`, use assembly intrinsics incorrectly
- **Uninitialized variables used confidently**: `const_param`, `volatile_access`, `ptr_single` all use `v1` without initialization
- **Function semantics broken**: `ptr_func_simple` returns function pointer address instead of calling `f(x)`
- **Test functions constant-folded**: Actual function calls replaced with pre-computed values, losing call relationships

### Best Quality Aspect: **Function Naming Quality (7/10)**
- Most original function names preserved accurately (`double_value`, `process_char`, `array_1d_stack`), providing semantic anchors despite other issues.

### Most Needs Improvement: **Composite Type Recovery (1/10)**
- Complete failure to recover any struct/union definitions; all composite type access reduced to opaque pointer arithmetic with hardcoded offsets.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement struct/union pattern detection**: Analyze consistent offset patterns across functions to reconstruct composite type definitions with field names inferred from usage context.
2. **Add type propagation from function signatures**: Preserve original parameter/return types from debug symbols or infer from usage patterns (e.g., floating-point operations indicate float/double).
3. **Insert uncertainty markers for suspicious patterns**: Flag uninitialized variable usage, assembly intrinsics, and type mismatches with comments like `// DECOMPILER_UNCERTAIN: variable may be uninitialized`.
