## EVALUATION: Decompilation Readability Assessment

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all local variables use generic names (`v1`, `v2`, `v3`, `v4`, etc.) with no semantic meaning. For example, in `heap_basic`, variables `v1` through `v6` obscure their roles as array indices, loop counters, and result values. Global variables are named `g1`, `g2`, etc., losing all original meaning.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names are preserved (`double_value`, `local_vars`, `heap_basic`, `test_stack_memory`). However, helper functions for library calls use mechanical names (`function_1120`, `function_1130`, etc.) and some signatures are incorrect with extra parameters.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `0xfffffffe`, `0x100000000`, `0x4080` appear throughout without explanation. String literals are stored as hex-encoded globals (`g1 = "=== \xe6\xb5\x8b..."`) rather than readable strings. No constants are recovered.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__readfsqword(40)`, `__stack_chk_fail()`) appear in most functions. Unnecessary casts (`(int64_t)`, `(int32_t)`) clutter expressions. Compiler runtime functions (`__do_global_dtors_aux`, `register_tm_clones`) add noise.

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Original `for` loops are converted to `while` loops with manual counter management (e.g., `local_array` uses `v3 += 4; v4 = v4 + a1` instead of clean for-loop). Some functions use `goto` labels (`lab_0x19b8` in `linked_list_heap`). At `-O0`, this restructuring is excessive.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries are mostly preserved, but signatures are corrupted (e.g., `heap_basic` has 3 parameters instead of 1, `test_heap_memory` has 3 parameters instead of 0). Library wrappers are extracted as separate functions (`function_1160` for `memset`).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom, but `linked_list_heap` uses goto-based loop structure that disrupts flow. Some functions have early returns mixed with complex conditionals that could be simplified.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions are constant-folded (e.g., `test_stack_memory` just prints expected values like `20`, `10`, `15` instead of calling actual functions). This removes semantic information about what the code actually does.

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Almost everything is `int64_t` regardless of original type. `int`, `int32_t`, `size_t`, `char*` distinctions are lost. Function `double_value(int x)` becomes `double_value(int64_t a1)`. Pointer types are `int64_t *` instead of specific types.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct types (`Point`, `Node`, `TreeNode`) are completely lost. Access is via pointer arithmetic (`*(int32_t *)mem`, `*(int64_t *)(v1 + 8)`) instead of field access (`p->x`, `node->next`). No struct definitions appear in output.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout (`(int64_t)mem`, `(int32_t)v5`). Pointer arithmetic on generic `int64_t *` where specific struct types should be used. No const-correctness preserved.

**13. Idiom Recognition:** Score: 3/10

**14. Anomaly Indication:** Score: 2/10

**15. Assertion & Contract Recovery:** Score: 2/10
