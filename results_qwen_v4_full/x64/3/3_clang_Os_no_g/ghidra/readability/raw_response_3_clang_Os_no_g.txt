# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `lVar2`, `puVar2`, `local_818` with no semantic meaning. Original source had meaningful names like `arr`, `large_buf`, `head`, `current` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`local_vars`, `heap_basic`, `test_stack_memory`), but some are generic (`FUN_00101020`) and test functions show constant-folded behavior obscuring purpose.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers in hex obscure meaning (`0x2a`=42, `0x14`=20, `0x3c`=60, `0xf`=15). Many values constant-folded during optimization, losing original intent like `return 42` becoming `return 0x2a`.

**4. Noise Reduction:** Score: 6/10  
Justification: Unnecessary type casts abound (`(undefined4 *)`, `(long)`, `(ulong)`). Stack variables like `local_818`, `auStack_8` add confusion. No debug instrumentation but optimization artifacts present.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Original `for` loops transformed to `do-while` with manual indexing (e.g., `lVar2 = lVar2 + 1; } while (lVar2 != 10)`). Test functions heavily constant-folded, obscuring actual test logic flow.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved but optimization collapsed some functions to trivial returns (`tree_heap_traversal` just returns `0x3c`). Original multi-line logic lost to constant folding.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but do-while loops disrupt natural for-loop reading pattern. Some forward jumps in complex functions like `linked_list_heap`.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Loop unrolling visible in `large_stack_frame` (16 chars per iteration with `cVar2` through `cVar17`). Some redundancy from optimization but no obvious dead code.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Generic types dominate: `undefined4`, `undefined8`, `ulong` replace specific `int`, `char*`, `size_t`. Pointer types degraded to `void*` or `undefined*` throughout.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct `Point`, `Node`, `TreeNode` not recovered. Fields accessed via pointer arithmetic (`*(undefined4 **)(__ptr + 2)`) instead of `node->next`. Complete struct information lost.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts (`(undefined4 *)`, `(long)`, `(ulong)`). Pointer arithmetic without type safety. Original safer type usage completely degraded.
