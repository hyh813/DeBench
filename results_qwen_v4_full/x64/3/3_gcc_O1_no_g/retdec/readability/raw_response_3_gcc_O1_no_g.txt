# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v1, v2, v3, mem, ptr) with no semantic meaning. Original code had descriptive names like `arr`, `local`, `head`, `current`, `new_node` that clearly indicated purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved from source (double_value, local_vars, heap_basic, test_stack_memory). Some helper functions have generic names (function_1120, function_1130) for library wrappers.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers from type casting abound (0xfffffffe, 0xffffffff, 0x100000000). String literals stored as hex-encoded globals (g1-g8) instead of readable strings. Original had clear test expectations documented.

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (__readfsqword, __stack_chk_fail) and compiler runtime functions (__cxa_finalize, __gmon_start__) present throughout. At -O0 optimization, these artifacts should be minimal.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Some loops recovered but many use manual pointer arithmetic instead of array indexing. linked_list_heap uses goto labels (lab_0x1937, lab_0x1942) instead of while loops. At -O0, should be much cleaner.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions properly separated and match original structure well. Test functions and implementation functions are correctly distinguished. Some helper functions could be better organized.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but linked_list_heap has multiple forward jump labels disrupting flow. Some functions have complex branching that could be simplified.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant type casts and bit masking operations throughout (e.g., `& 0xfffffffe`, `& 0xffffffff`). Some operations like `v4 = v4 + a1 & 0xffffffff` are unnecessarily verbose.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything collapsed to int64_t or int32_t. Original types (Point, Node, TreeNode, size_t) completely lost. Pointer types are generic int64_t* instead of proper typed pointers.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct definitions (Point, Node, TreeNode) completely lost. All struct access through raw pointer arithmetic (e.g., `*(int64_t *)(v1 + 8)` for next pointer). No field names recovered.

**11. Type Safety:** Score: 3/10  
Justification: Many unsafe casts throughout, pointer arithmetic without bounds checking. Original had proper type safety with malloc return checks and proper struct member access.

**15. Assertion & Contract Recovery:** Score: 2/10
