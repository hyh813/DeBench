# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v1`, `v2`, `result`, `temp` with no semantic meaning (e.g., `int128_t v1` in `_Z20test_cpp_member_funcv` should be `obj` or `test_object`). No variable names reflect their C++ class or purpose.

**2. Function Naming Quality:** Score: 4/10  
Justification: Functions use mangled C++ names (`_Z20test_cpp_member_funcv`) with demangling comments, but names aren't human-readable. Original function names like `test_cpp_member_func` are recoverable but not used as actual identifiers.

**3. Literal Clarity:** Score: 4/10  
Justification: String literals stored as hex-encoded globals (`g1 = "CPP-L3-01: %d (\xe6\x9c\x9f\xe6\x9c\x9b: 4704)\n"`) with Chinese characters unreadable. Magic numbers like `0x125c` appear without explanation in return statements.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe assembly intrinsic pollution (`__asm_movups_5`, `__asm_maxsd`, `__asm_xorps`) throughout code. Compiler artifacts like `__asm_movups_5(0xffffffffffffffff * v1 >> 64, __asm_movups(0x74736554))` make code nearly unreadable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Many test functions return pre-computed constants instead of actual logic (e.g., `_Z21test_cpp_virtual_funcv` just `return 42;`). At `-O0` optimization, this constant folding should not occur, indicating decompiler failure to recover original control flow.

**6. Function Decomposition:** Score: 4/10  
Justification: Functions have incorrect signatures with extra parameters (e.g., `_Z20test_cpp_constructorv` takes 4 `int64_t` parameters but original takes none). Class methods not properly associated with their classes.

**7. Code Linearization:** Score: 5/10  
Justification: Code reads top-to-bottom but assembly intrinsics disrupt flow. Some functions have reasonable structure (e.g., `_ZN9ContainerIiE4pushEi` has clear if-branch), but many are just constant returns.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some redundancy from repeated assembly patterns, but not severe. However, semantic redundancy exists where original logic is replaced with constant returns, losing all computational meaning.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Heavy degradation to `int64_t` and `int32_t` where specific C++ types should exist. Class types (`SimpleClass`, `LifecycleClass`, `Base`, `Derived`) completely lost. Pointer types generic (`int64_t *` instead of `Base*`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: Classes not recovered at all - only generic vtable structs with `e0`, `e1`, `e2` fields. `Container<int>` and `Container<double>` templates not distinguished by type. Original class members (`value`, `name[32]`, `data`, `size`) completely lost.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout (`(int64_t *)v1`, `(int32_t *)v1`). Pointer arithmetic on class objects (`v1 + 40`, `v1 + 80`) without type safety. No const-correctness preserved despite original `const` methods.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only auto-generated demangling comments present (e.g., `// Demangled: test_cpp_member_func()`). No algorithm explanations, no documentation of C++ semantics, no recovery of original source comments.

**13. Idiom Recognition:** Score: 2/10  
Justification: No C++ idioms recognized - no class structures, no operator overloading recovery, no template instantiation patterns identified. Virtual function calls reduced to raw pointer dereferences.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomalies flagged despite massive semantic loss. Functions returning constants instead of logic should be marked as "OPTIMIZED AWAY" or "CONSTANT FOLDED". Assembly intrinsics not explained.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Original test expectations (e.g., "期望: 4704") lost. Exception handling contracts not recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls present but virtual dispatch not recovered (vtable calls appear as raw function pointers). Inheritance relationships noted in comments but not in type system. Global variable usage unclear.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file with some grouping (prototypes, globals, functions). However, no logical separation between test functions, class methods, and runtime support code.

**18. Build System Compatibility:** Score: 4/10  
Justification: Has includes and declarations but won't compile as-is due to assembly intrinsics (`__asm_movups_5`, `__asm_maxsd`). Missing C++ standard library headers. Runtime functions (`__cxa_*`) declared but not implemented.

**Overall Readability Score:** 3.3/10

### Critical Defects:
- **Class structures completely lost** - All C++ classes (`SimpleClass`, `Base`, `Derived`, `Container<T>`) reduced to raw pointer arithmetic with no type information
- **Functions return constants instead of logic** - At `-O0` optimization, functions like `test_cpp_virtual_func` should not be constant-folded to `return 42;`
- **Assembly intrinsics pollute output** - `__asm_movups_5`, `__asm_maxsd`, `__asm_xorps` make code unreadable and non-compilable
- **Variable names have zero semantic meaning** - All variables are `v1`, `v2`, `result` with no connection to original source (`obj`, `ptr1`, `capture_by_value`)
- **Template instantiations not distinguished** - `Container<int>` and `Container<double>` methods use same generic types, losing type safety

### Best Quality Aspect:
- **Code Linearization (5/10)** - Code generally reads top-to-bottom without excessive goto statements; basic block structure is preserved even if semantics are lost

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover C++ class structures, inheritance hierarchies, and member layouts; this is fundamental to understanding the original code

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover C++ class structures from vtable layouts** - Analyze vtable patterns to reconstruct class hierarchies, member functions, and inheritance relationships with proper type names
2. **Eliminate assembly intrinsics and recover high-level constructs** - Replace `__asm_*` calls with equivalent C operations; detect constant folding and recover original computation logic
3. **Infer semantic variable names from usage patterns** - Use data flow analysis to name variables based on their role (e.g., variables passed to `strlen` should be named `string` or `name`)
