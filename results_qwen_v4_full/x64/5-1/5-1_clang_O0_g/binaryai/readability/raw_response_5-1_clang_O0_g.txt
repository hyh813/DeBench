# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`iVar1`, `local_30`, `puVar1`, `local_20`) with no semantic meaning. Original source had meaningful names like `value`, `name`, `capture_by_value`, but decompiled code uses stack offsets and type-hinted generics.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions recovered well (`test_cpp_member_func`, `test_cpp_constructor`), and class methods partially recovered (`SimpleClass::getValue`). However, external stubs are circular (`operator_new__` calls itself) and helper functions have mangled names.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x1234`, `0x1f`, `0x32` appear without explanation. String literals replaced with `DAT_00105025` references. Some constants recovered (getClassID returns 0x1234 correctly).

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts including circular external function stubs (20+ functions that call themselves), duplicate destructor thunks, and initialization code (`_init`, `_start`, `__do_global_dtors_aux`) cluttering output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic test functions have natural flow, but complex inheritance cases use pointer arithmetic instead of clean method calls. Lambda functors recovered as separate classes but with awkward `operator__` naming.

**6. Function Decomposition:** Score: 5/10  
Justification: Test functions appropriately sized, but template instantiations create excessive fragmentation (20+ `std::unique_ptr` helper functions). Many thunk functions just forward to other functions.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably well. Exception handling creates some non-linear flow, and virtual dispatch through vtables adds indirection but remains traceable.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Multiple duplicate destructor thunks for same classes (`MultiDerived::_MultiDerived` appears 4+ times). Template instantiations create significant code duplication without consolidation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of `undefined4`, `undefined8`, `void*` instead of precise types. Some types recovered (`int`, `double`, `bool`), but smart pointer types heavily mangled (`unique_ptr_int_test_cpp_smart_ptr______2_`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Class structures use field offsets (`this + 4`, `this + 8`) instead of named fields. No struct definitions recovered - all composite types accessed via pointer arithmetic. Inheritance visible but not explicit.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(undefined4*)`, `(long*)`, `(code***)`). Pointer arithmetic replaces field access throughout. Virtual dispatch uses raw vtable pointer manipulation without type safety.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no API documentation. Original source had extensive comments explaining each test case.

**13. Idiom Recognition:** Score: 4/10  
Justification: C++ patterns partially recognized (constructors/destructors identified), virtual dispatch visible through vtable pointers. Lambda functors recovered but not identified as lambdas. Template instantiations not labeled.

**14. Anomaly Indication:** Score: 2/10  
Justification: Circular external function stubs not flagged as problematic. Incomplete exception handling in `test_cpp_exception` (only shows throw, not catch blocks). No uncertainty indicators anywhere.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. Original source had clear expected values in comments (e.g., "期望: 10 + 34 + 4660 = 4704"), all lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Virtual dispatch relationships visible through vtable setup but not explained. Global variables like `instance_count` used without declaration context.

**17. Module Organization:** Score: 4/10  
Justification: All code in single flat output. No logical grouping beyond function boundaries. Original source had clear section headers (`CPP-L3-01`, `CPP-L3-02`, etc.) all lost.

**18. Build System Compatibility:** Score: 2/10  
Justification: External function stubs are circular (call themselves) - will not compile. Dependencies not documented. Would require extensive manual fixing to produce working code.

### Critical Defects:
- **Circular external function stubs**: 20+ functions like `operator_new__` call themselves, making code non-compilable
- **Complete loss of field names**: All class members accessed via offsets (`this + 4`, `this + 8`) instead of named fields
- **Zero comments**: Original source had extensive documentation; decompiled output has none
- **Generic variable naming**: `iVar1`, `local_30`, `puVar1` obscure semantic purpose throughout
- **Type degradation**: Heavy use of `undefined4`, `undefined8` instead of precise C++ types

### Best Quality Aspect: **Function Naming Quality (6/10)**
- Test function names recovered accurately (`test_cpp_member_func`, `test_cpp_virtual_func`) matching original source intent

### Most Needs Improvement: **Comment Quality & Presence (1/10)**
- Complete absence of any documentation, confidence indicators, or semantic explanations despite complex C++ features

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover field names from debug symbols**: With `-g` flag present, struct/class field names should be recovered instead of raw offsets
2. **Flag unresolved external calls**: Circular stubs should be marked as `[EXTERNAL - NOT DECOMPILED]` rather than creating self-referential functions
3. **Add decompilation confidence annotations**: Mark uncertain reconstructions (virtual dispatch, exception handling) with comments indicating analysis confidence level
