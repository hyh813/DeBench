# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`local_30`, `local_10`, `iVar1`, `puVar1`, `auVar8`) with no semantic meaning. Original source had meaningful names like `capture_by_value`, `instance_count`, `multiplier` that are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test function names are preserved (`test_cpp_member_func`, `test_cpp_constructor`), and some class methods recovered (`Base::virtual_func`, `Derived::getName`), but many helpers are generic (`FUN_00102020`, `template_max_int_`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere without explanation (`0x125c`, `0x3e9`, `0x15`, `0x2bf`). Original constants like `0x1234` for `getClassID()` are buried in arithmetic expressions.

**4. Noise Reduction:** Score: 4/10  
Justification: Stack canary checks (`__stack_chk_fail()`) appear in every function, frame pointer manipulation (`in_FS_OFFSET`), and compiler artifacts (`auVar1._8_8_ = 0`) clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Basic control flow preserved but interrupted by stack checks. Some functions return constant values directly (`test_cpp_multiple_inheritance` returns `0x47`) losing the original logic structure.

**6. Function Decomposition:** Score: 5/10  
Justification: Multiple destructor variants for same class (`MiddleA::_MiddleA` appears 6+ times with different signatures) creates fragmentation. Template instantiations properly separated.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but stack canary checks at function ends disrupt flow. Some functions have awkward return patterns (`auVar1 << 0x40` in `main`).

**8. Redundancy Elimination:** Score: 4/10  
Justification: Excessive destructor duplication (diamond inheritance creates 8+ `~DiamondDerived` variants). Same logic repeated across multiple function versions without consolidation.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Heavy type erosion with `undefined4`, `undefined8`, `undefined [16]` throughout. Original `int`, `size_t`, `double` types replaced with generic placeholders. No const-correctness preserved.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Class structures partially recovered (`Container_int_`, `unique_ptr_int_`) but field access uses pointer arithmetic (`*(int *)(this + 0x28)`) instead of named fields. No struct definitions visible.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts (`(undefined4 *)__cxa_allocate_exception(4)`, `(Base *)&local_28`). Raw pointer arithmetic throughout. No type safety annotations recovered.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only decompiler header comment present. Zero documentation of algorithm logic, class relationships, or C++ semantics. Original source had extensive comments explaining each test case.

**13. Idiom Recognition:** Score: 4/10  
Justification: Vtable calls visible (`(***(code ***)param_1)()`) but not identified as virtual dispatch. C++ patterns (RAII, smart pointers) present but not named or explained.

**14. Anomaly Indication:** Score: 3/10  
Justification: No uncertainty flags for indirect calls or unresolved symbols. Complex C++ features (virtual inheritance, RTTI) handled silently without warning about potential inaccuracies.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Exception handling logic (`__cxa_throw`, `__cxa_begin_catch`) present but contracts not recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Virtual dispatch partially recovered (`Base::virtual_func`, `Derived::virtual_func`) but inheritance hierarchy not explicit.

**17. Module Organization:** Score: 4/10  
Justification: Functions listed sequentially without class grouping. Related methods scattered (6+ destructor variants for `DiamondDerived` appear separately). No namespace structure.

**18. Build System Compatibility:** Score: 3/10  
Justification: Would not compile standalone. Missing type definitions, class declarations, headers. External functions declared but not properly typed (`operator_new`, `__cxa_throw`).
