# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (v0, v1, v2, ptr, result) with no semantic meaning. For example, `test_cpp_member_func` uses `v0[31]` and `result` instead of `name` and `return_value`, completely obscuring the original intent.

**2. Function Naming Quality:** Score: 4/10  
Justification: Some C++ mangled names recovered (_Z20test_cpp_member_funcv → test_cpp_member_func), but many functions remain as generic "sub_XXXX" prefixes. Thunk functions are labeled but class method names are partially lost.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like 42, 100, 200 appear without explanation. String literals like "Test" are preserved, but numeric constants (31 for buffer size, 4 for exception size) lack semantic context.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler artifacts including 20+ thunk functions, multiple destructor variants (D0, D1, D2, D4), and stub functions (sub_4023c5, sub_402475) that add clutter without value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Excessive goto usage (e.g., `call_virtual_func` uses `goto *((long long *)*((long long *)&arg_0))`), and many functions are empty stubs returning immediately without logic.

**6. Function Decomposition:** Score: 3/10  
Justification: Critical test functions are empty stubs (test_cpp_constructor, test_cpp_multiple_inheritance, test_cpp_operator_overload, test_cpp_lambda return immediately). Template functions partially recovered but incomplete.

**7. Code Linearization:** Score: 4/10  
Justification: Forward jumps and goto statements disrupt flow. Exception handling in test_cpp_exception uses confusing conditional chains instead of try-catch blocks.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Multiple destructor variants for same class (D0, D1, D2, D4), redundant thunk functions for virtual dispatch, and duplicate struct definitions (struct_0 defined 4+ times with different layouts).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of void*, unsigned long long instead of precise types. Some class types recovered (Base, Derived, Container<int>) but pointer types degraded (this pointers as void* instead of proper class pointers).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs defined with generic field names (field_0, field_8, field_10). Class layouts partially recovered but field semantics lost (e.g., SimpleClass::value and name[32] not recovered).

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout (char* to long long, void* arithmetic). Pointer arithmetic like `((char *)a0 + (*(a0))[24])` in thunk functions is error-prone and obscures intent.

**14. Anomaly Indication:** Score: 3/10

**15. Assertion & Contract Recovery:** Score: 2/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===

### Critical Defects:
- **Empty stub functions**: test_cpp_constructor, test_cpp_multiple_inheritance, test_cpp_operator_overload, test_cpp_lambda return immediately without implementing original logic
- **Generic variable naming**: All variables use v0, v1, ptr, result instead of semantic names (value, name, size, data)

- **No documentation**: Zero comments explaining C++ mechanisms (vtable, RTTI, exception handling, templates)
- **Type degradation**: Class this-pointers as void*, struct fields as field_0/field_8 instead of meaningful names

### Best Quality Aspect:
- **Function Naming Quality (4/10)** - Some C++ mangled names were successfully demangled to readable test function names (test_cpp_member_func, test_cpp_virtual_func)

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of documentation makes understanding C++ mechanisms (virtual dispatch, exception handling, templates) nearly impossible

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover C++ control structures**: Convert __cxa_* exception calls back to try-catch blocks and recover class method call semantics from vtable lookups
2. **Implement semantic variable naming**: Use usage patterns to infer names (variables used in size comparisons → size/count, string operations → name/buffer)
3. **Add decompilation confidence annotations**: Mark incomplete functions, uncertain type recoveries, and unresolvable indirect calls to guide manual analysis
