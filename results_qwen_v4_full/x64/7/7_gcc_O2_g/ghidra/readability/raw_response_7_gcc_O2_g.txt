# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables use mechanical names like `iVar1`, `iVar2`, `pbVar2`, `bVar1` instead of semantic names like `result`, `encrypted`, `decrypted`. Original source had meaningful names (`result`, `str`, `p1`, `p2`, `p3`) that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately (`param_fake_branch`, `param_opaque_predicate`, `decrypt_string`). Some helper functions use generic names (`FUN_00101020`) but core test functions match source.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers appear as hex without explanation (`0x14`, `0x23`, `0xe1`, `0x7a314`, `0x5a`). Original source had comments explaining expected values (e.g., "期望: 20", "期望: 225").

**4. Noise Reduction:** Score: 6/10  
Justification: Stack canary checks (`__stack_chk_fail()`), frame offset operations (`in_FS_OFFSET`), and calling convention warnings present. These compiler artifacts add visual noise but don't obscure core logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Tail recursion converted to do-while loops (expected at -O2), but fake branch conditions completely eliminated. GCD loop preserved but uses `do-while` instead of original `while`. Some functions like `param_fake_branch` return immediately, losing original structure.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately sized; `test_obf_opt_edge` is large but cohesive. Helper functions like `decrypt_string` properly separated. Some functions optimized away entirely (buffer overflow checks removed).

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom with minimal forward jumps. Do-while loops could be while loops for better readability. No excessive goto usage observed.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Dead code from fake branches properly eliminated. `call_vectorized_loop` has verbose explicit array initialization. No significant duplicate expressions observed.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `char*`, `size_t`), but some precision lost (generic `int` vs `int32_t`). Unsigned/signed distinctions sometimes unclear (`(uint)x >> 1`). Pointer types generally accurate.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Arrays handled correctly (`int a[8]`, `char decrypted[32]`). No complex structs in this sample. Bitfield struct in `param_implementation_defined` optimized away to constant return.

**11. Type Safety:** Score: 5/10  
Justification: Unsafe casts present: `(undefined *)0x0`, `(__sighandler_t)0x0`, `(long)x`. Stack canary provides some safety. Original signal handler types partially preserved but with casts.

Justification: Some warnings present ("Removing unreachable block", "Subroutine does not return"). "Unresolved local var" indicates decompilation uncertainty. Could be more explicit about optimization artifacts vs. decompilation failures.

**15. Assertion & Contract Recovery:** Score: 2/10
