# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`v0`, `v1`, `v2`, `v3`, `v4`, `v5`, `v6`, `v7`) requiring significant effort to track purpose. Original source had semantic names like `result`, `p1`, `p2`, `p3`, `div2`, `mod16`, `mul15` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved accurately (`param_fake_branch`, `param_opaque_predicate`, `param_instruction_substitution`), matching original source. However, helper functions use generic names (`sub_1020`, `sub_1030`) and metadata inconsistency (claims Ghidra but header says IDA Pro).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound without explanation: signal constants `8` and `11` (should be `SIGFPE`, `SIGSEGV`), `0x28u` (stack canary offset), `90` (should be `'Z'` or `0x5A`), `0x20u` (buffer size). Original used character literals and named constants.

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler artifacts present: `__fastcall`/`__cdecl` calling conventions, `__readfsqword(0x28u)` stack canary checks, type casts like `(__sighandler_t)`. 13 library functions marked "FAILED to decompile" creating visual clutter.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops correctly recovered as `while`/`for` constructs (e.g., `param_opaque_predicate`, `param_vectorized_loop`). Tail recursion in `param_tail_call_optimized` properly converted to iteration. However, `decrypt_string` has awkward loop: `for ( i = v6; *i; *(i - 1) = v9 )` obscures the XOR operation.

**6. Function Decomposition:** Score: 6/10  
Justification: Function granularity preserved appropriately, but critical logic lost: `param_fake_branch` reduced to `return x;` (fake branch conditions eliminated), `param_buffer_overflow_stack` returns just `a1` (overflow logic removed). Constant folding in `call_*` functions hides original computation.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with appropriate early returns. No excessive `goto` usage. Minor disruption in `decrypt_string` where loop body assignment `*(i - 1) = v9` appears after increment, confusing the XOR decryption flow.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Aggressive constant folding: `call_fake_branch` returns `10` directly, `call_tail_call_optimized` returns `500500`, `call_link_time_optimization` returns `20`. While efficient, this obscures the original test logic and expected value verification.
