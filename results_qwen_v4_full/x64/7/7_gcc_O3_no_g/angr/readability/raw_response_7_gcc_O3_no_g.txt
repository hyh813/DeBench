# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names (`v0`, `v1`, `a0`, `a1`, `result`, `ptr`) with no semantic meaning recovered. Original variables like `x`, `result`, `encrypted`, `decrypted` are lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`param_fake_branch`, `call_fake_branch`) but many are generic (`sub_401xxx`). Preserved names are accurate but inconsistent coverage.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers abound (`4294967295`, `340282366683253975865241362417122803708`, `0x5A` as `90`) without explanation. Original constants like `0xDEADBEEF` are lost.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant noise from wrapper functions (`sub_401365`, `sub_401377`), unsupported jumpkind markers (`[D] Unsupported jumpkind Ijk_SigTRAP`), and compiler artifacts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Complex do-while loops replace simple if statements (e.g., `param_opaque_predicate`). Goto usage present (`goto g_403f80`). Original simple control flow is obscured.

**6. Function Decomposition:** Score: 3/10  
Justification: Many useless wrapper functions (`sub_401365` just calls `param_fake_branch`). Some functions constant-folded entirely (`call_fake_branch` returns literal `10`). Poor granularity.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but with many forward jumps and complex nesting in optimized functions like `param_tail_call_optimized`.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy from optimization artifacts. Loop unrolling in `param_tail_call_optimized` creates 8+ levels of nested if statements that weren't in original.

## Additional Observations

**Optimization Mismatch Concern:** The original was compiled with `-O0 -g` (no optimization, debug symbols), but the decompiled code shows heavy optimization artifacts (constant folding, loop unrolling, vectorization). This suggests either:
- The binary analyzed was compiled differently than documented
- The decompiler is introducing optimization-like transformations
- There's a fundamental mismatch affecting all structural scores

**Constant Folding Issue:** Functions like `call_fake_branch()` return literal `10` instead of calling `param_fake_branch(10)`. While semantically equivalent, this loses the call structure important for understanding program flow.

**Architecture Mismatch:** Original metadata says `arm64` but decompiled code header says `AMD64` - this indicates potential analysis of wrong binary or metadata error.
