# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All parameters use generic names (`a1`, `a2`, `a3`) and locals use `v1`, `v2`, `result` - completely obscuring semantic meaning. Original had `temp1`, `temp2`, `temp3` which at least indicated computation sequence.

**2. Function Naming Quality:** Score: 8/10  
Justification: Core test functions retained original names (`sequential_ops`, `single_if`, `nested_if_deep`) accurately reflecting purpose. Only helper/runtime functions use generic names (`sub_1030`, `sub_1090`).

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x65` (101), `0x12` (18) appear without explanation. Test functions have hardcoded expected values which aids verification but obscures actual computation logic.

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler artifacts present (`__cdecl`, `__usercall`, `@<eax>` register hints). Empty stub functions (`_x86_get_pc_thunk_bx`) add noise. However, no excessive dead code or debug instrumentation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Optimizations obscure original structure - `single_if` becomes `a1 << (a1 > 0)`, `loop_for_fixed` becomes closed-form formula. `duffs_device` uses goto labels (LABEL_3, LABEL_5) instead of structured switch. At O2 optimization, some transformation is expected but semantic clarity is lost.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity matching original source. Each test function is cohesive with single responsibility. No monolithic functions or over-fragmentation observed.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally. `duffs_device` and `goto_backward` use goto labels disrupting flow, but this reflects the original complexity. Test functions are very linear.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Minimal dead code observed. Some redundant assignments exist (`result = -1` then immediately returned in switch cases). Loop unrolling in `goto_backward` is optimization artifact but not excessive.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Generic `int` used instead of specific types. `_DWORD *` less precise than original pointer types. `_BOOL4` for booleans is reasonable. Some signed/unsigned distinctions preserved (`unsigned int a1` in switch functions).

**10. Composite Type Recovery:** Score: 7/10  
Justification: Function pointer arrays recovered (`funcs_1B57`, `funcs_20A7`, `funcs_212B`). Array access patterns visible but as pointer arithmetic. No complex structs in this test suite to evaluate.

**11. Type Safety:** Score: 6/10  
Justification: Unsafe casts present (`(unsigned int)(a1 - 1)`, `(unsigned __int64)`). Signed/unsigned mixing in comparisons. Function pointer calls maintain type safety. No obvious buffer overflow patterns.

**13. Idiom Recognition:** Score: 5/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `n > 0` for factorial). No postconditions inferred. Test functions have hardcoded expected values which partially serves as contract verification.

**16. Cross-Reference Clarity:** Score: 6/10

**17. Module Organization:** Score: 5/10

**18. Build System Compatibility:** Score: 5/10
