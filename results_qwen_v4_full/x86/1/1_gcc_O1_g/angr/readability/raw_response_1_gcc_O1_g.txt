# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`v0`, `v1`, `a0`, `a1`) losing semantic meaning from original names like `temp1`, `sum`, `count`. Loop counter `i` is occasionally preserved but most purpose-indicating names are lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions recovered with original names (`sequential_ops`, `recursion_factorial`, `state_machine`), but many runtime helpers remain as `sub_401xxx` addresses. Good recovery for user-defined functions, poor for compiler-generated code.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `4294967295` (unsigned -1) and `1717986919` (division optimization for /10) appear without explanation. Simple constants (10, 20, 30) are clear, but compiler optimizations obscure intent.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: `__x86.get_pc_thunk.*` functions, repeated `struct_0` definitions, stack canary checks (`__stack_chk_fail_local`), and runtime initialization code (`_init`, `__do_global_ctors_aux`) mixed with application logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions like `sequential_ops` recover well, but `if_elseif_long` and `switch_large` use `goto` with computed addresses instead of switch statements. Loop structures partially recovered but jump tables become opaque gotos.

**6. Function Decomposition:** Score: 5/10  
Justification: Original function boundaries mostly preserved, but test functions expanded significantly with inline printf calls. Many `sub_401xxx` helper functions fragment the code unnecessarily.

**7. Code Linearization:** Score: 5/10  
Justification: Functions like `sequential_ops` read top-to-bottom naturally, but `switch_fallthrough` uses `LABEL_4015a4` goto patterns. Jump table functions have disruptive forward jumps that break linear reading.

**8. Redundancy Elimination:** Score: 6/10  
Justification: `struct_0` type definition repeated 15+ times with different layouts. Stack canary checks duplicated across functions. No obvious dead code but type redundancy is significant.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Original `int` parameters often become `unsigned int`. Function `if_else` returns `char` instead of `int`. Signed/unsigned mismatches throughout (e.g., `single_if` takes `unsigned int` but checks `> 0`).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Struct definitions use generic names (`struct_0`, `field_0`, `padding_0[40]`). No semantic field names recovered. Array handling converted to pointer arithmetic losing original structure.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts present (`(char *)&v16[4 * v18]`). Function pointer calls use generic `void*` types. Stack canary failures indicate potential buffer issues. Parameter type mismatches in `call_func_ptr`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation (test validation, difficulty ratings, scene labels) that is completely lost. No decompilation confidence indicators.

**13. Idiom Recognition:** Score: 4/10  
Justification: Basic patterns (loops, conditionals) recognized but not named. State machine pattern in `state_machine` function structurally present but not identified. No algorithm pattern documentation.

**14. Anomaly Indication:** Score: 5/10  
Justification: One unsupported operation noted (`[D] Unsupported jumpkind Ijk_SigTRAP`). Stack canary failures indicated but not explained. No warnings for suspicious type conversions or potential undefined behavior.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for factorial). No postconditions inferred. Original test validation comments completely absent. Error conditions not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`g_403064`, `g_40316c`) referenced without context. Callback relationships partially visible but not explicit.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping. Runtime initialization code mixed with application functions. Original source's clear section organization (L1/L2/L3 control flow) completely lost.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates compilation succeeds, but requires stubs for runtime functions. Compiler-specific constructs (`__x86.get_pc_thunk.*`) present. External dependencies not clearly documented.
