# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`, `ptr`) with no semantic meaning. Original source had meaningful names like `temp1`, `sum`, `count`, `total` that are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Core functions like `sequential_ops`, `recursion_factorial`, `loop_for_fixed` retain original names, but many helpers degrade to `sub_401030`, `sub_401040` format. Mixed recovery quality.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic number `4294967295` (unsigned -1) appears repeatedly without explanation. Constants like `19207`, `12004` in `switch_small` pointer arithmetic are completely opaque.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler artifacts: `__stack_chk_fail_local()`, `__x86.get_pc_thunk.*` functions, opaque struct definitions with `padding_0[19939]`, and redundant stack variable initialization throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: `switch_fallthrough` uses goto with `LABEL_401538` instead of fall-through semantics. `if_else` has incorrect logic (`return *((int *)&v0) > 0`). Some functions like `sequential_ops` recover well.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved correctly, but scattered runtime helpers (`_init`, `deregister_tm_clones`) mixed with application logic. No inappropriate inlining detected.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps disrupt flow (`goto v1->field_28`, `goto LABEL_401538`). `computed_goto` and `call_func_ptr_array` use computed jumps that break linear reading.

**8. Redundancy Elimination:** Score: 4/10  
Justification: `duffs_device` has excessive temporary arrays (`v4[8]`, `v14[8]`, `ptr3[8]`, etc.). Multiple redundant variable assignments (`v3 = v6; v2 = v7;`) appear in nearly every function.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most variables degraded to generic `unsigned int` or `int`. Original `volatile int *flag` sometimes preserved but often lost. Function pointer signatures use generic `unsigned int` parameters.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct definitions are nearly useless: `struct_0 { char padding_0[19939]; unsigned int field_4de3; }`. No field names recovered, just arbitrary offsets and padding arrays.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe patterns: `goto a0` in `call_func_ptr`, computed jumps through `(&v0)[a0]`, generic pointer arithmetic without bounds checking. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation explaining each function's purpose, test cases, and difficulty levels—all completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. `duffs_device` not identified as Duff's Device optimization. State machine functions (`state_machine`, `fsm_func_table`) not labeled as such.

**14. Anomaly Indication:** Score: 2/10  
Justification: Only one anomaly flagged: `[D] Unsupported jumpkind Ijk_SigTRAP`. Incorrect decompilation in `if_else` and `nested_if_2` not flagged as uncertain.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `duffs_device` requires `n > 0`). No postconditions or error condition documentation. Original test validation comments lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Direct function calls clear, but function pointer arrays (`call_func_ptr_array`) show opaque address arithmetic. Global variable references (`g_4030e8`) lack semantic names.

**17. Module Organization:** Score: 3/10  
Justification: Runtime initialization code (`_init`, `_fini`, `__do_global_ctors_aux`) scattered throughout. No logical grouping of related test functions.

**18. Build System Compatibility:** Score: 3/10  
Justification: Contains undefined externals (`g_4030e8`, `_ccall`), platform-specific constructs (`__x86.get_pc_thunk.*`), and architecture mismatches (metadata says arm64, output says X86).
