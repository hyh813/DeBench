# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`local_10`, `local_18`, `iVar1`, `puVar1`) despite `-O0 -g` debug symbols being available. Original names like `capture_by_value`, `result`, `obj` are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Test function names preserved well (`test_cpp_member_func`, `test_cpp_virtual_func`). Class methods retain names (`SimpleClass::getValue`, `Base::virtual_func`). Template instantiations show type info (`template_max<int>`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `0x1234`, `0x32` (50), `0x1e` (30) appear without explanation. String literals referenced as `&DAT_0001503c` instead of showing actual content.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive compiler artifacts: `__x86.get_pc_thunk.ax` injections, stack canary checks in every function (`if (local_10 != *(int *)(in_GS_OFFSET + 0x14))`), 200+ helper functions for standard library templates.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Exception handling uses explicit try/catch comments with goto to catch handlers. Basic loops preserved but virtual dispatch shown as raw vtable calls (`(*(code *)**(undefined4 **)param_1)`).

**6. Function Decomposition:** Score: 5/10  
Justification: Severe over-fragmentation from template expansion (20+ functions for `unique_ptr` internals). Original test functions intact but standard library exploded into many small helpers.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom within functions. Exception handling introduces non-linear flow with `goto LAB_00011fcb` patterns disrupting readability.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Massive redundancy from template instantiations (`Container<int>` and `Container<double>` generate nearly identical code). Smart pointer helpers repeat similar patterns across types.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Class types preserved (`SimpleClass*`, `Base*`) but degraded to `undefined4`, `undefined*`, `void*` in many places. Stack canary uses generic `int` instead of `uintptr_t`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Classes recognized but field access via pointer arithmetic (`*(int *)(this + 4)`, `this[0x23]`) instead of named members. No struct definitions with field names recovered.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts: `(Base *)&local_34`, `(undefined4 *)this`, `(type *)param_1`. Pointer arithmetic on class objects without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: Structural comments present ("try from 00011642 to 00011659 has its CatchHandler") but no semantic documentation. No algorithm explanations or API contracts.

**13. Idiom Recognition:** Score: 5/10  
Justification: C++ patterns visible (constructors, destructors, virtual calls, lambdas) but not explicitly named. Lambda closures identified as `{lambda(int)#1}::operator()` without explaining capture semantics.

**14. Anomaly Indication:** Score: 6/10  
Justification: Warnings present: "WARNING: Function: __x86.get_pc_thunk.ax replaced with injection", "WARNING: Unknown calling convention". Some unreachable blocks flagged but not all anomalies documented.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition/postcondition documentation. Exception boundaries marked but no contract inference (e.g., "requires: ptr != NULL" missing).

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls clear with preserved names. Virtual dispatch visible through vtable access patterns. Inheritance relationships inferable but not explicitly documented.

**17. Module Organization:** Score: 3/10  
Justification: Flat organization with 211 functions in single output. No logical grouping beyond alphabetical/address order. Static initialization functions scattered throughout.

**18. Build System Compatibility:** Score: 3/10  
Justification: References external functions (`operator_new`, `__cxa_throw`, `strlen`) without declarations. Would need extensive stubs and standard library headers to compile.

**Overall Readability Score:** 4.5/10

### Critical Defects:
- **Variable names completely generic** (`local_10`, `iVar1`) despite `-g` debug symbols that should preserve original names
- **Class field access via pointer arithmetic** (`*(int *)(this + 4)`) instead of named member access (`this->value`)
- **Template expansion bloat** generates 200+ helper functions obscuring the 12 original test functions
- **String literals hidden** as `&DAT_0001503c` references instead of showing actual content
- **No contract documentation** for exception handling, null checks, or resource ownership

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Original function and method names are well-preserved, making it possible to identify test cases and class methods despite other degradation.

### Most Needs Improvement:
- **Composite Type Recovery (3/10)** - Class structures recognized but field names completely lost to offset arithmetic, making object layout unintelligible without source reference.

### Top 3 Recommendations for Decompiler Improvement:
1. **Leverage debug symbols for variable naming** - At `-O0 -g`, DWARF debug info contains original variable names that should be recovered instead of `local_N` generics.
2. **Recover struct/class field names from debug info** - Use DWARF type information to reconstruct named field access instead of `this + offset` pointer arithmetic.
3. **Collapse standard library template instantiations** - Recognize common STL patterns (`unique_ptr`, `tuple`, `vector`) and present them at higher abstraction level rather than expanding all internal helpers.
