# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are generically named (`param_1`, `iVar1`, `local_28`, `uVar3`) with no semantic meaning. Original source had meaningful names like `arr`, `head`, `current`, `new_node` that are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core functions retained original names (`local_vars`, `heap_basic`, `double_value`), but many helpers are generic (`FUN_00011030`, `FUN_00011eb5`). About 40% of functions have meaningless names.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation (`0x2a` for 42, `0x14` for 20, `0x32` for 50). Original had clear constants and expressions; decompiled uses hex literals obscuring intent.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: `get_pc_thunk_bx` warnings, `unaff_retaddr` variables, "Removing unreachable block" warnings. Redundant casts like `(undefined4 *)` clutter the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: For-loops converted to do-while patterns. Complex functions use goto labels (`joined_r0x00011872`, `LAB_000118a7`). `stack_alias` completely lost logic (returns constant `0x14`).

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes reasonable, but test functions heavily constant-folded (printing hardcoded values instead of calling functions). Some compiler helper functions unnecessarily exposed.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but forward jumps in `linked_list_heap` and `test_heap_memory` disrupt flow. Goto labels create non-linear reading experience.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Test functions show aggressive constant folding (e.g., `test_stack_memory` prints `0x14` instead of calling `local_vars(5)`). Some redundant type casts throughout.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Many `undefined4`, `undefined1`, `undefined8` types instead of proper C types. Pointer types degraded to generic `void*` or `int*`. Original `size_t`, `int32_t` precision lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: `Point` struct completely lost (functions do `param_1 * 3` arithmetic). `Node` struct recovered as `int[2]` pattern. `TreeNode` struct lost entirely. No struct definitions visible.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe casts pervasive (`(undefined4 *)`, `(int *)`). Void pointer arithmetic throughout. Functions return `undefined4` instead of proper types, risking undefined behavior.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only compiler-generated warnings as comments ("WARNING: Function...replaced"). No algorithm explanations, no API contracts, no semantic documentation whatsoever.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. Memory allocation patterns visible but unnamed. Linked list and tree traversal idioms not recognized or documented.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies flagged with "WARNING" comments and "Subroutine does not return". However, semantic losses not indicated (e.g., `stack_alias` returning constant instead of implementing alias logic).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. NULL checks present in code but not documented as contracts. Error conditions undocumented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables referenced but not clearly identified. Callback relationships partially visible (`global_func_ptr`).

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file. No file organization matching source. Static functions not clearly marked as file-scoped.

**18. Build System Compatibility:** Score: 7/10  
Justification: Metadata indicates compilation succeeds. External dependencies visible (`extern_function`, `malloc`, `free`). Platform-specific code (`fork`, `waitpid`) preserved.

**Overall Readability Score: 4.5/10**

### Critical Defects:
- **Variable `param_1`, `iVar1`, `local_28`** should be named based on semantic usage (e.g., `input_value`, `loop_index`, `stack_buffer`)
- **`stack_alias` function** completely lost its alias logic, returning constant `0x14` instead of implementing pointer aliasing
- **Composite types (`Point`, `Node`, `TreeNode`)** not recovered; all struct access reduced to raw pointer arithmetic
- **Test functions constant-folded** printing hardcoded values instead of calling actual functions, losing test semantics
- **`vla_stack` and `alloca_usage`** have extremely complex stack manipulation code that obscures the simple VLA/allocation semantics

### Best Quality Aspect:
- **Build System Compatibility (7/10)** - The decompiled code compiles successfully with minimal stubs needed, and external dependencies are properly declared.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Struct definitions completely lost; `Point`, `Node`, and `TreeNode` types reduced to raw pointer arithmetic, making data structure semantics invisible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct/union types from access patterns** - Analyze consistent offset patterns (e.g., `+0`, `+4`, `+8`) to reconstruct struct definitions with field names based on usage context.

2. **Preserve function call semantics in test code** - Avoid aggressive constant folding that replaces function calls with hardcoded return values; maintain call graph integrity for analysis.

3. **Infer variable names from usage context** - Use data flow analysis to name variables based on their role (e.g., loop counters, buffer pointers, size parameters) rather than generic `iVar1`, `param_1` patterns.
