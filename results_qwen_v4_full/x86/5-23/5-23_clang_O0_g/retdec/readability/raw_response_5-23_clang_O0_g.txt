# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Variables are mostly generic (`v1`, `v2`, `result`) obscuring semantic meaning; original source had meaningful names like `size`, `counter`, `increment` that are lost. Some functions preserve names like `len`, `arr`, but most are auto-generated.

**2. Function Naming Quality:** Score: 8/10  
Justification: Main function names are well preserved (`param_macro_constants`, `call_asm_basic`, `param_atomic_c11`) matching original source. However, helper functions have generic names (`function_1040`, `function_1050`) for library wrappers.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation (`0xdef0`, `0x1bbc`, `0x1bc8`, `282`); original macros like `MAX_SIZE`, `BUFFER_COUNT` are completely lost. String literals referenced via pointer arithmetic (`v1 + (int32_t)&g7 - 0x1fe2`) instead of direct strings.

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler artifacts present: `__x86_get_pc_thunk_*` functions, redundant assignments (`int32_t v2 = v1;`), and opaque global variables (`g1` through `g9`). However, no excessive debug instrumentation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Most control flow is natural with recognizable if/while structures. `param_asm_clobber` uses while loop instead of for, and conditional compilation branches are correctly resolved to single paths (expected at -O0).

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized and cohesive. Library call wrappers exist as separate functions (`function_1050` for `mprotect`), which adds verbosity but maintains clarity. No monolithic functions observed.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with recognizable loop structures. Some forward references to globals via pointer arithmetic disrupt flow slightly, but no excessive goto usage.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Minimal dead code observed. Some redundant variable assignments present (`int32_t v1; int32_t v2 = v1;` in `param_stringize`). No loop unrolling artifacts (appropriate for -O0 compilation).

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 6/10  
Justification: Uses `int32_t`, `uint32_t`, `int64_t` but degrades to generic `int32_t` frequently. Pointer types preserved but often generic (`int32_t *` vs specific types). Custom `int128_t` typedef for SIMD operations is reasonable.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Limited composite types in source code (mostly primitives). Arrays represented as pointers appropriately. No struct/union recovery needed for this codebase, so score reflects adequate handling of what exists.

**11. Type Safety:** Score: 5/10  
Justification: Unsafe pointer arithmetic prevalent (`*(int32_t *)(v1 + (int32_t)&g7 - 0x1fe2)`). Assembly intrinsics use custom types that may not compile portably. Some functions have incorrect return semantics (`param_variadic_macro` returns `y + 50` instead of proper calculation).

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Virtually no meaningful comments; only auto-generated line range markers (`// Line range: 37 - 43`). Original source had extensive documentation explaining macro behavior, expected values, and test purposes—all lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition for macro expansions, atomic operations, or assembly patterns. `param_macro_recursion` correctly shows `x + 16` but doesn't indicate this came from nested macro expansion. Assembly intrinsics (`__asm_movaps`, `__asm_paddd`) not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about semantic errors or uncertain decompilation. Critical bugs silent: `param_variadic_macro` returns wrong value, `param_dynamic_code` has incorrect logic (`result = v3 + 5` where `v3` is uninitialized). No anomaly flags.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or error documentation recovered. Original source had expected return values in comments (e.g., "期望: 64") completely absent. No indication of valid input ranges.

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function call relationships clear through preserved names. Global variable usage opaque (`g1` through `g9` with hex addresses). No call graph documentation or callback relationship identification.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with no logical grouping comments. Helper functions mixed with main test functions. Original source's section organization (5.2, 5.3) completely lost.

**18. Build System Compatibility:** Score: 4/10  
Justification: Includes standard headers but custom `int128_t` typedef may cause issues. Assembly intrinsics (`__asm_movaps`, `__asm_paddd`) are non-standard and won't compile without special handling. Some globals undefined.
