# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v1`, `v2`, `a1`, `a2`, `result`). The `binary_tree_sum` function has 125+ variables named `v1` through `v125` with no semantic meaning. Simple functions like `process_char` use `a1` instead of `c` or `character`.

**2. Function Naming Quality:** Score: 6/10  
Justification: Original function names largely preserved (`process_char`, `array_1d_stack`, `ptr_double`). However, helper functions have mechanical names (`function_1080`, `function_1090`, `function_112c`) and `binary_tree` wrapper is unclear.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation (`0x1c8`, `0x233f`, `0x4ee4`). String literals accessed via pointer arithmetic (`v1 + 0x1c5d`) instead of named constants. Offset values like `+4`, `+8`, `+12` for struct fields are unexplained.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler artifacts: `__x86_get_pc_thunk_*` functions (10+ variants), `__readgsdword(20)` stack canary checks, `__stack_chk_fail_local()`. Global variables `g1` through `g14` with hex values instead of meaningful names.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple functions have reasonable loops, but `binary_tree_sum` has deeply nested conditionals that should be recursive. Loop conditions use awkward patterns (`4 * a2 + a1 != v1` instead of `i < n`).

**6. Function Decomposition:** Score: 3/10  
Justification: `binary_tree_sum` is catastrophically large (500+ lines, 125+ local variables) - clearly failed recursion recovery. Should be 10 lines of recursive code. `test_data_types_l1` is one monolithic block instead of organized test cases.

**7. Code Linearization:** Score: 5/10  
Justification: Most functions read top-to-bottom reasonably. However, `binary_tree_sum` has extreme nesting (15+ levels) with forward references that disrupt flow. Stack canary checks at function ends add noise.

**8. Redundancy Elimination:** Score: 2/10  
Justification: `binary_tree_sum` contains massive code duplication - identical blocks repeated 50+ times from failed tail-call optimization recovery. The same pointer arithmetic pattern (`*(int32_t *)(v2 - 16)`) appears hundreds of times.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: All integers degraded to `int32_t` regardless of original type (`short`, `long`, `long long` all become `int32_t`). Pointer types lost (`int*` becomes `int32_t`). Float types partially preserved (`float32_t`, `float64_t`, `float80_t`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Complete failure** - no struct/union definitions recovered. `Point3D`, `Rect`, `Widget`, `TreeNode`, `Graph` all accessed via raw pointer arithmetic (`*(int32_t *)(a1 + 8)`). Field names completely lost.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts (`(int32_t *)a1`, `(char *)a1`). Pointer arithmetic on typed data without bounds checking. Function pointers degraded to `int32_t` losing signature information. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only compiler-generated metadata comments present ("Address range", "Line range"). Zero algorithm explanations, no decompilation confidence indicators, no API documentation. Original source comments completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition whatsoever. Tree traversal, linked list iteration, graph traversal all appear as raw pointer chasing. No identification of common algorithms or data structure patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomalies flagged despite massive decompilation uncertainty in `binary_tree_sum`. No warnings about indirect calls, stack layout issues, or potential undefined behavior. Code presented confidently despite obvious problems.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `arr != NULL`, `n > 0`). No postconditions or invariants. Error conditions not documented. Original test expectations completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. No call graph information. Global variable usage opaque (`g1` through `g14` with hex addresses). Callback relationships lost in function pointer degradation.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file with no logical grouping. Original source organization (Data Types, Arrays, Pointers, Composite Types sections) completely lost. No header file separation.

**18. Build System Compatibility:** Score: 6/10  
Justification: Includes necessary headers (`stdint.h`, `stdio.h`, `stdlib.h`). Function prototypes declared. However, requires stubs for `__libc_start_main`, `__printf_chk`, `__readgsdword`. Some architecture-specific code (`__x86_get_pc_thunk_*`) present despite ARM64 target.
