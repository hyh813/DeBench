# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are overwhelmingly generic (`v1`, `v2`, `v3`, `result`) instead of semantic names like `sum`, `i`, `n` from source. For example, `array_1d_stack` uses `v2` for loop index instead of `i`, and `result` instead of `sum`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved well (`process_char`, `array_1d_stack`, `struct_simple`), but helper functions have generic names (`function_1080`, `function_1090`, `function_10fc`).

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers like `32` (char conversion), `10` (enum multiplier), `40` (struct stride) appear without explanation. String literals are referenced indirectly via offset calculations.

**4. Noise Reduction:** Score: 3/10  
Justification: Every function contains `__x86_get_pc_thunk_*` calls that serve no semantic purpose. Stack canary checks (`__readgsdword(20)`) appear in most functions, adding significant noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loops recovered as `while` instead of `for` in many cases (e.g., `array_1d_stack`). Conditional logic is clear but uses nested if-else instead of ternary operators where source used them.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries match source well. However, runtime functions (`__do_global_ctors_aux`, `deregister_tm_clones`) are mixed with user code, creating confusion.

**7. Code Linearization:** Score: 5/10  
Justification: Stack canary checks at function ends disrupt reading flow. Some functions have forward references to variables before declaration (e.g., `test_array_types` with `vla_arr2 = {0, 0, 0}`).

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant variable assignments throughout (e.g., `int32_t v1; __x86_get_pc_thunk_ax(v1);` in every function). Some duplicate expressions in complex pointer arithmetic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int32_t`, `int16_t`, `int64_t`, `float32_t`), but `long` becomes `int32_t` (platform-dependent). Pointer types often degraded to `char*` or `int32_t*` (e.g., `ptr_void` parameter is `char*` not `void*`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: **Critical failure** - All struct/union definitions lost. Access via pointer arithmetic: `*(int32_t *)(v1 + 4)` instead of `p->y`. `struct_simple` shows `*(int32_t *)(v1 + 4) + *p + *(int32_t *)(v1 + 8)` instead of `p->x + p->y + p->z`.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout (`(int32_t)p`, `(char *)size`). Function pointer signatures sometimes degraded. No const-correctness preserved despite source using `const`.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated metadata comments (address ranges, line ranges). Zero semantic comments explaining algorithm purpose or non-obvious logic.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common C idioms not recognized (string length calculation, array sum, linked list traversal). No pattern identification for data structures.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about decompilation uncertainty. Stack canary checks present but unexplained. No indication of potential undefined behavior (e.g., `array_oob`).

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `arr != NULL`, `n > 0`). No postconditions or invariants recovered despite source having clear contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls clear, but global variables poorly named (`g1`, `g2`, `g3`, `g4`, `g5`). Callback relationships partially preserved but function pointer types degraded.

**17. Module Organization:** Score: 4/10  
Justification: Runtime/compiler functions mixed with user code. No logical grouping by functionality. Test functions and implementation functions interleaved.

**18. Build System Compatibility:** Score: 5/10  
Justification: Standard headers included but external functions (`__libc_start_main`, `__cxa_finalize`) undeclared. Would require significant stubbing to compile standalone.

**Overall Readability Score:** 4.6/10

### Critical Defects:
- **All struct/union type definitions lost** - accessed via raw pointer arithmetic (`*(int32_t *)(v1 + 4)`)
- **Every function polluted with `__x86_get_pc_thunk_*` calls** adding no semantic value
- **Variable names completely generic** (`v1`, `v2`, `result`) losing all semantic meaning
- **Zero documentation** - no comments explaining algorithm purpose or non-obvious logic
- **Type degradation** - `void*` becomes `char*`, `const` qualifiers lost, function pointer signatures simplified

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Core function names preserved from source (`process_char`, `array_1d_stack`, `struct_simple`), making basic navigation possible despite other issues.

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct/union definitions forces all field access through opaque pointer arithmetic, making complex data structure code nearly unreadable.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct/union definitions from access patterns** - Analyze consistent offset patterns (e.g., +0, +4, +8) to reconstruct `Point3D { x, y, z }` instead of `*(int32_t *)(v1 + 4)`
2. **Eliminate compiler thunk artifacts** - Filter out `__x86_get_pc_thunk_*` calls that serve no semantic purpose in decompiled output
3. **Infer variable names from usage context** - Variables used in loop bounds should be named `i`, `n`; accumulators should be `sum`; pointers should indicate target type
