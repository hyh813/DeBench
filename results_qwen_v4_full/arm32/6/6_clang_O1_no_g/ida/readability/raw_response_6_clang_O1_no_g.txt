# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables use generic names like `a1`, `a2`, `v1`, `v2`, `v3` throughout (e.g., `param_strcpy(char *a1, const char *a2)` should be `dst`, `src`). No semantic meaning is preserved from original source names like `buffer`, `fd`, `shm`.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core function names preserved (`param_strcpy`, `call_strcmp`, `param_fork_exec`) matching source intent. However, helper functions like `sub_C48` and 65 "FAILED to decompile" entries indicate incomplete recovery.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation: syscall numbers `5`, `6`; signal numbers `10`, `14` (should be `SIGUSR1`, `SIGALRM`); socket constants `2`, `1` (should be `AF_INET`, `SOCK_STREAM`); hex sizes `0x12u`, `0x1Fu`, `0x28u`.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present including register variable comments (`// r0`, `// r4`), BYREF annotations, and type casts like `(__sighandler_t)`. Some redundant temporaries (`v2 = v1; if (v1) v2 = -1;`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Loop structures mostly recovered as do-while patterns. Some awkward patterns like `for (i = 0; i != -10; --i)` instead of standard `for (i = 0; i < 10; i++)`. Conditional logic generally clear but verbose.

**6. Function Decomposition:** Score: 6/10  
Justification: Original function boundaries mostly preserved. However, `call_strcpy` and `call_strcmp` were constant-folded (returning literal values `8`, `0`), losing the original call structure. Thread functions properly separated.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with early returns. Some forward references disrupt flow (e.g., `LABEL_6` jumps in `param_mutex_lock`). Nested conditionals in `param_signal_handling` create moderate complexity.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Redundant variable assignments common (`v2 = v1; v3 = v2;`). The `thread_sum` function shows a complex arithmetic expression replacing a simple loop, indicating optimization artifacts not simplified.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (`int`, `char*`, `size_t`, `FILE*`). However, precision lost: `param_scanf(int a1)` should be `const char*`; `param_memset(unsigned __int8 *a1)` loses `void*` generality. Function pointers often cast to `void*`.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs degraded to raw arrays: `ThreadData` becomes `_DWORD v5[9]` with manual field offsets. `sockaddr` accessed via `addr.sa_data[2]` instead of proper struct fields. `struct stat` partially recovered but fields accessed inconsistently.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts present: `(_DWORD *)thread_return`, `(void *(*)(void *))thread_compute`. Pointer arithmetic visible (`v5[-3 * v2 / 4u + 2]`). No const-correctness preserved from original source.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated comments present (`// [sp+0h] [bp-28h] BYREF`, `// r0`). Zero semantic comments explaining algorithm intent, API contracts, or decompilation uncertainty. Original source comments completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (`strcpy`, `strlen`, `printf`, `pthread_create`). However, patterns not explicitly identified (no "producer-consumer pattern" comment for condition variables, no "mutex-protected critical section" annotation).

**14. Anomaly Indication:** Score: 4/10  
Justification: Failed decompilations listed at end (`FAILED to decompile: __imp_pthread_mutex_unlock`) but not integrated into call sites. One warning present (`positive sp value has been detected, the output may be wrong!`) but no inline uncertainty markers.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `requires: a1 != NULL`). Error paths visible but not labeled as contracts. Return value semantics not explained (e.g., what `42` vs `-1` means in context).

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`shared_counter`, `atomic_counter`, `signal_received`) used without declaration context. No call graph or dependency information provided.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in source-order grouping (library functions, syscalls, threads). However, no file/module boundaries indicated. Related functions not explicitly grouped with comments or namespaces.

**18. Build System Compatibility:** Score: 3/10  
Justification: 65 external functions marked "FAILED to decompile" requiring stubs. Architecture mismatch (source ARM64, binary ARM32). Would require significant modification including syscall number mapping and library function declarations to compile.

**Overall Readability Score:** 4.75/10

### Critical Defects
- **Variable names completely generic** (`a1`, `a2`, `v1`...) losing all semantic meaning from original `dst`, `src`, `buffer`, `fd`
- **Magic numbers throughout** without recovery of constants (syscall `5`/`6`, signals `10`/`14`, sockets `2`/`1`)
- **65 functions failed to decompile** leaving large portions of code as unresolved external references
- **Zero semantic comments** - only register/stack annotations, no algorithm or intent documentation
- **Struct types degraded** to raw arrays with manual offset calculations (`v5[-3 * v2 / 4u + 2]`)

### Best Quality Aspect
- **Function Naming Quality (6/10)** - Core function names like `param_strcpy`, `param_fork_exec`, `param_mutex_lock` were preserved from source, maintaining semantic identity for primary entry points.

### Most Needs Improvement
- **Comment Quality & Presence (2/10)** - Complete absence of semantic documentation; only mechanical register/stack annotations that add noise without explaining intent, algorithms, or decompilation confidence.

### Top 3 Recommendations for Decompiler Improvement
1. **Implement semantic variable naming** - Analyze usage patterns to infer names (e.g., parameters passed to `strcpy` → `dst`/`src`, file operations → `fd`/`fp`)
2. **Recover named constants from system headers** - Map syscall numbers, signal constants, and socket constants back to their symbolic names using platform-specific header databases
3. **Add decompilation confidence annotations** - Mark uncertain regions inline (e.g., `// UNCERTAIN: indirect call target`) rather than listing failures separately at end of file
