# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`a1`, `a2`, `v1`, `v2`, `v3`) obscuring semantic meaning. Original source had meaningful names like `dst`, `src`, `buffer`, `filename` that are completely lost. Some exceptions exist (`thread_return`, `newthread`, `pipedes`) but are rare.

**2. Function Naming Quality:** Score: 7/10  
Justification: Core function names preserved well (`param_strcpy`, `call_strcmp`, `param_mutex_lock`). However, helper functions use generic names (`sub_C48`, `thread_compute`) and 65 functions failed to decompile entirely, losing their semantic identity.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals preserved clearly ("/etc/passwd", "HelloLib"). However, magic numbers abound without explanation: syscall numbers (5, 6), signal numbers (10, 14), socket constants (2, 1), and hex values (0x12u, 0x1Fu) lack symbolic names.

**4. Noise Reduction:** Score: 5/10  
Justification: Significant compiler artifacts present: register annotations (`// r0`, `// r4`), stack frame details (`// [sp+0h] [bp-28h] BYREF`), and type casting noise (`_DWORD *`, `__int64`). These clutter every function and impede reading.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic control structures recovered (if/else, while, do-while). However, some patterns are awkward: `v2 = v1-- > 1` in signal handling loops, and nested conditionals in `param_fread_fwrite` could be flattened. Fork/exec logic is reasonably clear.

**6. Function Decomposition:** Score: 6/10  
Justification: Thread helper functions properly separated (`thread_compute`, `thread_sum`, `consumer_thread`). However, caller functions are constant-folded away (`call_strcpy` returns `strlen(v1)` directly), losing the test structure. Some functions like `param_thread_local_storage` are overly complex (100+ lines).

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with appropriate early returns. However, some functions have deep nesting (4+ levels in `param_fread_fwrite`, `param_signal_handling`) and backward-jump patterns converted to do-while loops disrupt flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Constant folding creates inconsistency (some callers return constants, others call functions). Redundant variable assignments common: `v2 = v1; if (v1) v2 = -1; if (v1 >= 1) return 1; return v2;` pattern repeats across multiple comparison functions.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 5/10  
Justification: Basic types present (`int`, `char*`, `size_t`, `FILE*`) but degraded. Generic `int` used where `int32_t` or `pid_t` appropriate. Atomic operations use `volatile int*` instead of `atomic_int*`. Pointer casts (`_DWORD *`, `void *`) lose type safety.

**10. Composite Type Recovery:** Score: 4/10  
Justification: `struct stat` partially recovered but fields generic. `ThreadData` struct flattened to `_DWORD` arrays (`v4[2]`, `v6[2]`, `v8[2]`). `sockaddr` visible but `sa_data` accessed via offset. TLS variables lost (`unk_141C4` instead of `tls_counter`).

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout: `(_DWORD *)thread_return`, `(void *(*)(void *))thread_compute`, `(volatile int*)&atomic_counter`. Pointer arithmetic on generic types. No const-correctness preserved. Atomic operations use compiler intrinsics instead of C11 atomics.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Almost no semantic comments. Only decompiler warnings present ("positive sp value has been detected", "attributes: thunk"). No algorithm explanations, no parameter documentation, no return value semantics. Original source had extensive comments explaining each test.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (`strcpy`, `strcmp`, `printf`, `pthread_create`). However, patterns not named: producer-consumer pattern in condition variables not identified, mutex-protected critical sections not annotated, atomic CAS patterns not explained.

**14. Anomaly Indication:** Score: 4/10  
Justification: 65 failed decompilations listed at end but not integrated into code flow. No inline warnings for indirect calls or unresolved symbols. Stack anomaly warning present but not actionable. No uncertainty indicators in recovered code.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: a1 != NULL"). No postconditions (e.g., "ensures: return value is heap-allocated"). Error returns visible (-1, -2, -3) but meaning not explained. Original source had clear test expectations.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables referenced (`shared_counter`, `atomic_counter`, `signal_received`) without usage context. No call graph, no callback relationship documentation. Thread entry points not clearly marked.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear roughly in source order but no logical grouping. All functions in single output file. No header file separation. Static variables not scoped appropriately. Test functions mixed with implementation.

**18. Build System Compatibility:** Score: 3/10  
Justification: 65 external functions failed to decompile (listed as `__imp_*`). Would require extensive stubbing to compile. Dependencies not documented. Platform-specific code (ARM syscalls, pthread) not abstracted. Not standalone compilable.
