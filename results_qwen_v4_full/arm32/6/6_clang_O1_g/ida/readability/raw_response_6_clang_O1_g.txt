# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names (v0, v1, v2, temp_1) with no semantic meaning. Original source had meaningful names like `dst`, `src`, `buffer`, `filename` that are completely lost.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most functions retain original names (param_strcpy, call_strcpy, param_mutex_lock). However, some helpers use generic names (sub_C48) and thread functions are adequately named.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive: `syscall(5, ...)` instead of SYS_open, signal numbers as literals (10, 14) instead of SIGUSR1/SIGALRM, hex constants (0x12u, 0x1Fu) without explanation.

**4. Noise Reduction:** Score: 6/10  
Justification: Some compiler artifacts present (register variable comments like `// r0`, `// r4`), unnecessary type casts (_DWORD *), but generally readable without excessive noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Control flow mostly uses if/else and while loops naturally. However, some loops use awkward patterns (e.g., `for (i = 0; i != -10; --i)` instead of standard forward iteration).

**6. Function Decomposition:** Score: 5/10  
Justification: Critical issue: call_* functions are constant-folded and don't call param_* functions (call_strcpy returns strlen directly instead of calling param_strcpy). This breaks the original call hierarchy.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with clear function boundaries. Loop structures are recognizable despite some backward iteration patterns from ARM compilation.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Significant redundancy: call_* functions duplicate param_* logic instead of calling them. test_standard_library_functions duplicates call_* function logic inline.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved (int, char*, size_t, FILE*), but significant erosion with void* casts and _DWORD types. Original const-correctness mostly lost.

**10. Composite Type Recovery:** Score: 5/10  
Justification: stat struct recovered but ThreadData struct lost (accessed via pointer arithmetic `*((_DWORD *)arg + 2)`). pthread_t and other structs partially recovered.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts throughout (_DWORD *, void *), pointer arithmetic obscures type boundaries, e.g., `*(_DWORD *)&s[-4 * j]` in call_memset.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Minimal auto-generated comments only (stack variable annotations). No semantic comments explaining algorithm purpose or non-obvious logic.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls recognized (strcpy, strlen, memcmp), but thread patterns, signal handling, and IPC patterns not explicitly identified or documented.

**14. Anomaly Indication:** Score: 5/10  
Justification: 65 functions marked "FAILED to decompile" at end. Some stack warnings present ("positive sp value has been detected"). However, no inline anomaly markers.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants documented. Error handling present but return value semantics not explained (e.g., what -1, -2, -3 mean).

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible in code, global variables (shared_counter, atomic_counter) identifiable. However, callback relationships not explicitly documented.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output without logical grouping. Original source's section organization (6.1, 6.2, 6.3) not preserved in decompiled output.

**18. Build System Compatibility:** Score: 3/10  
Justification: 65 external functions failed to decompile. Would require extensive stubs and type definitions to compile. Many platform-specific calls unresolved.

**Overall Readability Score:** 5.0/10

### Critical Defects:
- **Variable names completely generic** (v0, v1, v2) losing all semantic meaning from original source
- **Constant folding broke function hierarchy** - call_* functions don't call param_* functions as original source intended
- **65 external functions failed to decompile** - severely limits practical utility
- **Magic numbers everywhere** - syscall numbers (5, 6), signal numbers (10, 14), hex constants without explanation
- **Struct definitions lost** - ThreadData accessed via pointer arithmetic instead of named fields

### Best Quality Aspect: Function Naming Quality (scored 7/10)
- Most original function names preserved (param_strcpy, param_mutex_lock, param_signal_handling), maintaining semantic connection to source.

### Most Needs Improvement: Variable Naming Quality (scored 3/10)
- Complete loss of semantic variable names makes tracking data flow extremely difficult; every variable is v0, v1, v2 regardless of purpose.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable naming** - Infer names from usage patterns (e.g., variables used in size checks should be named `size`, `length`, `count`)
2. **Preserve call hierarchy over constant folding** - Don't inline constant-folded call sites; maintain original function call structure for readability
3. **Recover symbolic constants** - Map syscall numbers to SYS_* constants, signal numbers to SIG_* names, and detect common magic numbers for annotation
