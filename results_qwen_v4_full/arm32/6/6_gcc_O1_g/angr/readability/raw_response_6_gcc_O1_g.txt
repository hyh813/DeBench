## Decompilation Readability Assessment

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Most variables use generic names like `v0`, `v1`, `v2`, `ptr` obscuring semantic meaning. A few exceptions exist (`dst`, `len`, `fp`, `count`, `result`) but the majority require manual inference to understand purpose.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test functions retained original names (`param_strcpy`, `call_strcpy`, `thread_compute`, `signal_handler`), but many helper functions use mechanical names (`sub_10ca8`, `sub_10fc8`, `sub_112bf`). Thread functions recovered well but atomic operation helpers are unnamed.

Justification: Magic numbers pervasive throughout: `4294967295` (appears as -1 repeatedly), `42` (test success code), signal numbers as literals (10, 14 instead of SIGUSR1, SIGALRM), socket constants (2, 1, 0 instead of AF_INET, SOCK_STREAM). No named constants recovered.

**4. Noise Reduction:** Score: 3/10  
Justification: Significant compiler/decompiler artifacts: `__stack_chk_guard` checks in nearly every function, `armg_calculate_condition` calls obscuring simple conditionals, redundant register save/restore patterns (`v3 = v4; v2 = v5; v1 = v6; v0 = v7;`), and numerous `__sync_synchronize()` stub calls.

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Heavy reliance on `goto` statements with labels (LABEL_11008, LABEL_115c5, etc.) instead of structured loops/conditionals. `armg_calculate_condition` calls wrap simple comparisons, making logic harder to follow. Some do-while loops recovered but often with awkward termination conditions.

**6. Function Decomposition:** Score: 5/10

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps disrupt top-to-bottom reading flow. Deep nesting in functions like `param_thread_local_storage` with multiple goto targets. Loop structures often use backward jumps with manual counter management instead of clean for/while constructs.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Stack canary checks duplicated in nearly every function. Some caller functions call the same function twice unnecessarily (`call_strcpy` calls `param_strcpy` then returns the result of calling it again). Dead code patterns in atomic operation stubs with infinite loops.
