# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables use generic names like `a1`, `a2`, `v1`, `v2` instead of semantic names (`dst`, `src`, `buffer`, `len`). For example, `param_strcpy(char *a1, const char *a2)` loses the meaningful `dst`/`src` distinction from source.

**2. Function Naming Quality:** Score: 8/10  
Justification: User-defined function names are well-preserved (`param_strcpy`, `call_strcmp`, `param_signal_handling`). However, 61 library functions show "FAILED to decompile" which limits completeness.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound: `syscall(5, a1, 0)` hides `SYS_open`, `signal(10, signal_handler)` hides `SIGUSR1`, `0x14u` for 20 bytes. No constants recovered from original `#define` statements.

**4. Noise Reduction:** Score: 7/10  
Justification: Relatively clean for `-O0` compilation. Some redundant variable assignments (`v3 = v4--` patterns) but no severe compiler artifacts or debug instrumentation pollution.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Control structures mostly preserved (if-else, for loops). Some awkward constructions like `while(!signal_received) { v1 = v4--; if(v1 <= 0) break; }` instead of cleaner `for` loop.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries match source well. Thread functions (`thread_compute`, `consumer_thread`) properly separated. No monolithic functions or over-fragmentation observed.

**7. Code Linearization:** Score: 7/10  
Justification: Generally reads top-to-bottom. Some nested conditionals in `param_socket_create` create deep indentation but flow remains traceable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some redundant patterns from compiler output. The `call_memcpy` function has unnecessary zero-initialization of `v3`, `v4`, `v5` that aren't properly used in return calculation.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 6/10  
Justification: Basic types preserved but degraded. `param_strcpy` returns `size_t` (correct) but original returned `int`. Many parameters use generic `int` instead of specific types like `__pid_t`, `key_t`.

**10. Composite Type Recovery:** Score: 5/10  
Justification: `stat buf` struct recovered but fields accessed generically (`buf.st_size`). Thread data structures flattened to arrays (`_DWORD v6[3]`) losing struct semantics from `ThreadData`.

**11. Type Safety:** Score: 6/10  
Justification: Multiple unsafe casts present: `(_DWORD *)thread_return`, `(unsigned __int8 *)v2`. Function pointer casts like `(void *(*)(void *))thread_compute` lose type safety.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no comments in decompiled output vs. extensive documentation in source. No algorithm explanations, no parameter descriptions, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 5/10  
Justification: Standard library calls (`strcpy`, `memcpy`, `pthread_create`) recognized but not explicitly documented as patterns. Atomic operations decompiled to `__sync_*` helpers without explanation.

**14. Anomaly Indication:** Score: 6/10  
Justification: "FAILED to decompile" markers for 61 functions provide transparency. However, no warnings about potential semantic errors like the `call_memcpy` array access issue.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `ptr != NULL` checks not annotated). No postconditions or error condition documentation recovered from source comments.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls visible but relationships not documented. Global variables (`shared_counter`, `atomic_counter`, `signal_received`) used without context about their purpose.

**17. Module Organization:** Score: 6/10  
Justification: Functions appear in logical order matching source sections. However, no file organization info, no header declarations, no namespace grouping visible.

**18. Build System Compatibility:** Score: 4/10  
Justification: 61 external functions failed to decompile requiring stubs. Atomic helper functions (`__sync_fetch_and_add_4`, etc.) would need implementation. Cannot compile standalone.
