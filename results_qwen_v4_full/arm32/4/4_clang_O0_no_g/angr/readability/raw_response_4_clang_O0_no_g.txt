# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `a0`, `a1`, `ptr`) with no semantic meaning. For example, `func_mixed_args` uses `v0-v6` instead of meaningful names like `string_length` or `float_sum`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names match original source (`cdecl_func`, `call_cdecl`, `varargs_func`), but helper functions have cryptic names (`sub_40040c`, `$a.8`) and runtime functions are unclear.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound without explanation: `4294967295` (should be `-1`), `1374389535` (float representation), `1074339512`. No constants recovered from original source.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe compiler artifacts in floating-point helpers (`__adddf3` has 100+ lines of `armg_calculate_condition` calls). Redundant function calls like `return cdecl_func(5, 10);` after already storing result.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Excessive `goto` usage instead of structured loops (e.g., `call_weak_fn` uses `goto LABEL_4004d0`). Simple conditionals become nested goto chains in complex functions.

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes are reasonable, but some are fragmented (`call_cdecl` calls function twice unnecessarily). Runtime helpers are monolithic blobs.

**7. Code Linearization:** Score: 5/10  
Justification: Forward jumps disrupt reading flow. `test_calling_conventions` has 40+ local variables with scattered initialization, making top-to-bottom reading difficult.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Duplicate function calls throughout (e.g., `call_cdecl` returns `cdecl_func(5, 10)` twice). `call_ret_large_struct` calls function then accesses different variable than returned.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `unsigned int` or `char*`. `func_mixed_args` originally had `(int, char*, double, long long)` but decompiled as 6 `unsigned int` parameters. No const-correctness preserved.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure. `LargeStruct` (128 bytes) becomes 4 `unsigned int` parameters. `SmallStruct` pointer access becomes `*(a0) * a0[1]` with no struct definition. `TestIntPair` lost entirely.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe pointer arithmetic everywhere (`ptr[8 * i]`, `*((int *)&v0)`). Function pointer calls use wrong signatures (`v0(5, v0)` passes extra argument). No null checks recovered properly.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no API documentation.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern not recognized (manual pointer arithmetic instead of `va_arg`). Struct-by-value not identified. Function pointer callbacks not labeled.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings for problematic code. `param_func_ptr` has invalid call signature but no flag. Floating-point conversion errors silent.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `func_struct_byptr` should note `p != NULL`). No postconditions or error condition documentation.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. Callback `$a.8` not linked to `callback_func`. Global variables (`g_413008`) have no context.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in source order but no logical grouping. Runtime helpers mixed with application code. No header file or API separation.

**18. Build System Compatibility:** Score: 4/10  
Justification: Requires many external stubs (`__libc_start_main`, `__gmon_start__`, `armg_calculate_condition`). Floating-point runtime functions incomplete. Won't compile standalone.

**Overall Readability Score:** 3.7/10

### Critical Defects:
- **Complete type system collapse**: All types degraded to `unsigned int`, losing `double`, `long long`, struct distinctions
- **Zero composite type recovery**: `LargeStruct`, `SmallStruct`, `TestIntPair` all lost, replaced with raw parameter lists
- **No comments or documentation**: Zero semantic guidance for complex operations like varargs handling
- **Excessive compiler noise**: Floating-point helpers contain 100+ lines of `armg_calculate_condition` calls obscuring logic
- **Redundant function calls**: Many functions call themselves twice (`return cdecl_func(5, 10);` after already storing result)

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Most application function names preserved from source (`cdecl_func`, `varargs_func`, `test_calling_conventions`), providing basic semantic anchors

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any documentation, making complex operations like varargs iteration and floating-point conversion impossible to understand without original source

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover type information from calling conventions** - ARM AAPCS passes different types in different registers; use this to infer `double` vs `int` parameters
2. **Detect and label varargs patterns** - Recognize `va_start`/`va_arg`/`va_end` sequences and emit appropriate C constructs instead of raw pointer arithmetic
3. **Eliminate redundant function calls** - Detect when return statement duplicates already-computed value and emit cleaner `return v0;` instead of re-calling function
