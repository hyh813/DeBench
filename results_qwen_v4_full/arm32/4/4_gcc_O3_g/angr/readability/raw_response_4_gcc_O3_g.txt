# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v0, v1, a0, a1, ptr) with zero semantic meaning. No variable indicates purpose like `count`, `sum`, or `buffer_size` despite clear usage patterns in varargs functions.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (cdecl_func, stdcall_func) but many are generic (sub_103c4) or have optimization artifacts (.constprop.0 suffix). Function names don't indicate parameter types or calling conventions.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers abound (4294967295 for -1, 0x100000, 0x80000000) without explanation. Constant-folded results (15, 50, 42) are correct but hide original computation logic.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise from stack canary checks in every function, armg_calculate_condition calls obscuring simple logic, and compiler artifacts like `vvar_286{r64|1b}` that are completely unreadable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Excessive goto/label patterns instead of loops (LABEL_10ab4, LABEL_10fa5). The `__aeabi_dadd` function has hundreds of lines of nested conditionals that should be simple floating-point addition.

**6. Function Decomposition:** Score: 3/10  
Justification: Catastrophic failure: `func_struct_byval` has 31 parameters instead of 1 struct. Many functions constant-folded to single returns, losing all original logic structure.

**7. Code Linearization:** Score: 4/10  
Justification: Heavy forward jumps disrupt reading flow. Deep nesting in `__aeabi_dadd` with 20+ levels of conditional branches makes linear reading impossible.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repeated armg_calculate_condition calls with identical parameters. Some dead code from optimization artifacts, but constant folding did eliminate some redundancy.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Everything degraded to `unsigned int` or `char*`. Original types (int, double, long long, struct) completely lost. Function pointers become `unsigned int*`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure: LargeStruct (128 bytes) expanded to 31 individual parameters. SmallStruct becomes `unsigned int*`. No struct field names recovered.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe pointer arithmetic throughout (`ptr[1] * *(ptr)`). Float/int conversions use raw `__aeabi_*` calls without type safety. Double pointer handling in `param_double_ptr` loses type information.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments. Only auto-generated decompiler labels (LABEL_10fa5). No algorithm explanations or decompiler confidence indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs pattern completely obscured (manual pointer arithmetic instead of va_list). Function pointer callbacks not identified. No design pattern recognition.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some "do not return" annotations present. However, critical anomalies unflagged: 31-parameter struct, architecture mismatch, floating-point obfuscation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `ptr != NULL` checks not annotated). No postconditions or invariants recovered from original code.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. Global variables named generically (g_1143c, g_22008). No call graph documentation.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping. No header file structure. Static functions not properly scoped.

**18. Build System Compatibility:** Score: 3/10  
Justification: Unresolved external dependencies (GLIBC_2.4::__stack_chk_guard, armg_calculate_condition, __aeabi_*). Cannot compile standalone without significant stubbing.

**Overall Readability Score:** 3.4/10

### Critical Defects:
- **Struct parameter explosion**: `func_struct_byval` has 31 `unsigned int` parameters instead of 1 `LargeStruct` parameter, completely destroying semantic meaning
- **Floating-point obfuscation**: `__aeabi_dadd` function is 400+ lines of nested `armg_calculate_condition` calls instead of simple double addition
- **Complete type erasure**: All types degraded to `unsigned int`/`char*`, losing int/double/struct distinctions
- **Architecture mismatch artifacts**: ARMEL decompilation of x86/x64 source introduces `armg_calculate_condition` calls that obscure all logic
- **Constant folding over-aggression**: Functions like `call_cdecl()` return literal 15 instead of showing `cdecl_func(5, 10)` call, hiding program structure

### Best Quality Aspect:
- **Function Naming Quality (5/10)** - Some original function names preserved (cdecl_func, stdcall_func, varargs_func) providing minimal semantic anchors

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover struct types; 128-byte LargeStruct became 31 separate parameters, making the code semantically meaningless

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement struct parameter detection**: Analyze calling convention patterns to identify when multiple consecutive parameters represent a single struct passed by value
2. **Add floating-point operation recognition**: Map `__aeabi_*` and similar runtime calls back to high-level arithmetic operations with appropriate type annotations
3. **Preserve call structure over constant folding**: When decompiling for readability, prefer showing actual function calls over constant-folded results to maintain program structure visibility
