# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `uVar2`, `local_10` with no semantic meaning. Original source had meaningful names like `count`, `sum`, `args`, `x`, `y`, `s` that conveyed purpose.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source symbols (`cdecl_func`, `test_calling_conventions`, `func_struct_byptr`). However, compiler helper functions have cryptic names (`__aeabi_dadd`, `__floatsidf`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `0xf`, `0x32`, `0x2a` appear as constant-folded results, obscuring original intent (e.g., `call_cdecl` returns `0xf` instead of showing `cdecl_func(5, 10)`).

**4. Noise Reduction:** Score: 5/10  
Justification: Stack canary checks (`__stack_chk_fail`), frame pointer manipulation, and verbose floating-point helper functions (200+ lines each) create significant noise.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions have natural flow, but test functions are constant-folded losing original structure. Floating-point operations use excessive branching instead of idiomatic expressions.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries preserved, but call relationships lost (e.g., `call_cdecl` doesn't call `cdecl_func`, just returns 15). This destroys the semantic relationship between functions.

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom with reasonable do-while loops. However, floating-point helper functions have excessive branching that disrupts flow.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Massive redundancy in floating-point helpers (`__aeabi_dadd`, `__subdf3`, `__floatsidf` share 80% similar code). No consolidation or abstraction attempted.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Types degraded to generic `int`, `undefined4`, `undefined8`. Original had `size_t`, `double`, `long long`, `char*` - all lost. Function pointers become `code*`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - `LargeStruct` becomes `int param_1, undefined4 param_2, undefined4 param_3, undefined4 param_4`. `SmallStruct` becomes `int *param_1` with array indexing. No struct definitions recovered.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe casts throughout (`(int)uVar2`, `(ulonglong)uVar2 >> 0x20`). Pointer arithmetic on generic int pointers. Float/int conversions through opaque helper functions.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original had extensive documentation explaining each test case, calling convention, and expected behavior.

**13. Idiom Recognition:** Score: 2/10  
Justification: Varargs handling completely broken (doesn't use `va_list` at all - just counts iterations). No pattern recognition for common C idioms like string handling or struct access.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomalies flagged despite critical failures. `varargs_func` is completely broken but not indicated. Struct handling is wrong but silently degraded.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Null checks exist (`if (param_1 == (int *)0x0)`) but not documented as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but many constant-folded away. No call graph documentation. Global variable usage unclear (`handle1_1`, `handle2_0` appear without context).

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in somewhat logical order but no file organization. All code in single output with no header files or API declarations.

**18. Build System Compatibility:** Score: 4/10  
Justification: References external functions (`__printf_chk`, `strlen`, `memcpy`) without declarations. Floating-point helpers would need linking. Requires extensive stubs to compile standalone.

### Critical Defects:
- **Variable `param_1`, `param_2`** should be named based on semantic usage (e.g., `count`, `sum` in varargs functions)
- **Struct types completely lost** - `LargeStruct` and `SmallStruct` degraded to raw pointer arithmetic with no type information
- **Varargs handling broken** - `varargs_func` doesn't process variable arguments at all, just counts iterations
- **Constant folding destroys call relationships** - `call_cdecl` returns `0xf` instead of calling `cdecl_func(5, 10)`
- **Zero comments** despite complex floating-point operations and calling convention tests that need explanation

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Symbol names were preserved from debug information, maintaining function identity across decompilation

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of documentation makes understanding complex operations (especially floating-point helpers) nearly impossible

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover composite types from stack access patterns** - Detect struct access patterns (e.g., `param_1[0]`, `param_1[1]`) and propose struct definitions with field names
2. **Preserve call relationships over constant folding** - When debug symbols exist, maintain function call structure rather than folding to constants
3. **Add confidence annotations for uncertain recoveries** - Flag varargs handling, struct access, and type conversions with uncertainty indicators to guide manual review
