# Decompilation Readability Assessment

**1. Variable Naming Quality:** Score: 4/10

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Simple functions (`sequential_ops`, `single_if`) recover well. `switch_fallthrough` degrades to goto labels (`LABEL_7`, `LABEL_6`). `nested_if_deep` becomes sequential ifs losing nesting structure.

**6. Function Decomposition:** Score: 7/10  
Justification: Function boundaries preserved correctly. Helper functions (`op_add`, `double_value`) remain separate. Test functions are large but cohesive with printf sequences.

**7. Code Linearization:** Score: 5/10  
Justification: `duffs_device` and `switch_fallthrough` use extensive goto patterns disrupting flow. Most other functions read top-to-bottom naturally with appropriate loop constructs.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `loop_nested` has unnecessary complexity in inner loop (empty for-loop). `switch_fallthrough` has redundant label jumps. Some functions have excessive temporary variables.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 6/10

**10. Composite Type Recovery:** Score: 7/10  
Justification: Arrays recovered as stack variables (`v8[12]`, `v16[5]`). Function pointer arrays recovered but with generic naming. No complex structs in this code to evaluate.

**11. Type Safety:** Score: 5/10  
Justification: Unsafe function pointer casts in `call_func_ptr_array` (`((int (__fastcall *)(int))v4[a1 - 4])`). `computed_goto` has inline assembly. Pointer arithmetic in `loop_break` is suspicious.

**14. Anomaly Indication:** Score: 4/10

**15. Assertion & Contract Recovery:** Score: 2/10
