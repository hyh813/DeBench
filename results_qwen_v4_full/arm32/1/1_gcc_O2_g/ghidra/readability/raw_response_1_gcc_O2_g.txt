# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`iVar1`, `iVar2`, `uVar1`, `piVar2`) obscuring semantic meaning. Original source had meaningful names like `temp1`, `sum`, `count`, `result` which are lost. Many show "Unresolved local var" comments indicating recovery failure.

**2. Function Naming Quality:** Score: 9/10  
Justification: Excellent recovery - all function names preserved from original source (`sequential_ops`, `nested_if_deep`, `duffs_device`, etc.). Action-oriented names clearly reflect purpose and match source exactly.

**3. Literal Clarity:** Score: 6/10  
Justification: Inconsistent literal representation - some decimal (10, 5), others hex (0x14, 0x1e, 0x3e9). Magic numbers like `0x3e9` (1001) and `0x65` (101) not explained. Test function outputs show pre-computed values instead of actual function calls.

**4. Noise Reduction:** Score: 5/10  
Justification: Multiple "WARNING: Removing unreachable block" and "Unresolved local var" comments clutter code. Compiler runtime functions (`__aeabi_idiv`, `__gmon_start__`) included without abstraction. Test functions show duplicated array initialization code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic structures recovered (if-else, loops) but optimization artifacts present. `recursion_factorial` converted to iterative loop (expected at -O0 is suspicious). `duffs_device` shows complex goto patterns instead of clean switch fall-through. `switch_fallthrough` logic restructured awkwardly.

**6. Function Decomposition:** Score: 8/10  
Justification: Function boundaries preserved correctly with appropriate granularity. No excessive inlining or fragmentation. Helper functions (`double_value`, `op_add`, etc.) remain separate as in source.

**7. Code Linearization:** Score: 7/10  
Justification: Most functions read top-to-bottom naturally. Test functions (`test_control_flow_l1/2/3`) have forward jumps (`goto LAB_00010b8c`) disrupting flow. Loop constructs generally use do-while instead of for, reducing readability.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Test functions show significant redundancy - array initialization duplicated multiple times. `loop_modify_var` logic appears twice in test function. Some optimization-induced redundancy in division functions (`__aeabi_idiv` has 35+ boolean variables).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Basic `int` types preserved but no precision hints (int32_t, uint32_t). Function pointer types partially recovered (`_func_int_int *`). Some unsigned conversions implicit (`(uint)x < 5`). Pointer arithmetic visible in array access.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Arrays recovered correctly (`int arr[5]`, `int matrix[3][4]`). No complex structs in this codebase to evaluate. Local array initialization visible in decompiled code. Pointer-to-array distinctions maintained.

**11. Type Safety:** Score: 6/10  
Justification: Multiple "WARNING: Could not recover jumptable" and "Treating indirect jump as call" warnings indicate type uncertainty. Function pointer calls lack full signature verification. Some implicit casts (`(uint)(event == 1)`).
