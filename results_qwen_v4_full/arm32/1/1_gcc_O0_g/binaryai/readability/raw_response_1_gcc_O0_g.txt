# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `local_c`, `local_10` with no semantic meaning. Original source had `temp1`, `temp2`, `temp3` indicating computation sequence, but decompiled version loses even this minimal clarity.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are perfectly preserved from source (`sequential_ops`, `single_if`, `nested_if_deep`, etc.). This is exceptional recovery that immediately identifies each function's purpose.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x66666667` (compiler optimization for division), `0x3e9` (1001), `0x65` (101) obscure intent. Original used clear decimals (10, 5, 100). Hex `0xffffffff` for -1 is readable but inconsistent.

**4. Noise Reduction:** Score: 6/10  
Justification: Contains runtime stubs (`_init`, `__do_global_dtors_aux`), stack canary checks (`__stack_chk_fail`), and external function wrappers that clutter the output. Core logic functions are relatively clean.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most loops recovered as for/while/do-while correctly. However, `switch_fallthrough` uses goto labels (`LAB_00010a64`), and `duffs_device` has complex goto chains that obscure the original fall-through pattern.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are properly separated with appropriate granularity. However, some return types are wrong (`call_func_ptr_array` returns `code*` instead of `int`), indicating decomposition issues.

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom naturally. `switch_fallthrough` and `duffs_device` use goto statements that disrupt linear reading. `switch_default` converts switch to if-chain unnecessarily.

**8. Redundancy Elimination:** Score: 6/10  
Justification: `duffs_device` has significant redundancy with 8 nearly-identical case labels. Compiler optimization artifacts (like `0x66666667` multiplier) are not simplified or explained.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Critical type errors: `if_else` returns `bool` instead of `int`, `loop_while` returns `undefined8` with `CONCAT44`, many functions return `undefined4`. Function pointers use `code*` instead of proper signatures.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Arrays are recovered as stack variables (`local_20[4]`). No structs needed for this code (mostly primitives), so limited opportunity to demonstrate this capability.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe patterns: `CONCAT44` indicates type confusion, function pointer calls use generic `code*` type, `__aeabi_idiv` calls lack proper type signatures. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. Original source had extensive documentation explaining each function's purpose, test cases, and difficulty level. Complete loss of semantic context.

**13. Idiom Recognition:** Score: 3/10  
Justification: Duff's device not identified as such, state machine patterns unlabeled, no recognition of common algorithms (factorial, digit counting). Patterns are present but unnamed.

**14. Anomaly Indication:** Score: 3/10  
Justification: No warnings about type mismatches, potential issues, or decompilation uncertainty. `CONCAT44` return type should be flagged as suspicious but isn't.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `duffs_device` requires n>0), no postconditions, no error condition documentation. Original had explicit test validation comments.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are visible but relationships aren't documented. No call graph, no indication of which functions are test helpers vs. core logic. Callback relationships unclear.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output with no logical grouping. Runtime startup code mixed with application logic. No separation between test functions and tested functions.

**18. Build System Compatibility:** Score: 4/10  
Justification: Contains external dependencies (`__aeabi_idiv`, `__stack_chk_fail`) without clear documentation. Would require significant stubbing to compile standalone. ARM-specific intrinsics present.
