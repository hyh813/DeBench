# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: All parameters use generic names (`a1`, `a2`, `a3` instead of `a`, `b`, `c` or `x`); locals are `v1`, `v2`, `v3` with no semantic meaning. For example, `sequential_ops(int a1, int a2, int a3)` loses the original `a, b, c` clarity.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most original function names preserved (`sequential_ops`, `single_if`, `nested_if_deep`), but some internal references use generic labels (`sub_4DC`, `dword_1E70`). Good preservation of test function names.

**3. Literal Clarity:** Score: 6/10  
Justification: Test functions show constant folding (`printf("CF-L1-01 (sequential_ops): %d\n", 21)`) obscuring actual computation. Magic numbers like `0x12EBC` for function pointer arrays lack explanation.

**4. Noise Reduction:** Score: 5/10  
Justification: `__fastcall` calling convention markers clutter signatures; frequent type casts (`(__int64)`, `(unsigned int)`); `duffs_device` contains inline assembly `__asm { ADD PC, LR, R3 }` that wasn't decompiled.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: `single_if` optimized to `a1 << (a1 > 0)` instead of clear if-statement; `if_else` returns `bool` instead of `int` 1/0; `loop_for_fixed` converted to mathematical formula losing loop semantics; `goto_forward` logic significantly altered from original.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately sized with good separation; helper functions (`double_value`, `op_add`, etc.) properly extracted; test functions separate from implementation logic.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom, but `switch_fallthrough` uses awkward `LABEL_5` goto pattern; `loop_multi_exit` has confusing `while(2)` infinite loop with break pattern.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Test functions show aggressive constant folding eliminating computation visibility; some redundant variable assignments (`v1 = a1; v2 = v1;`); no obvious dead code but optimization artifacts present.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Generic `int` used throughout instead of specific types; pointers shown as `_DWORD *`; `__int64` appears inconsistently on ARM 32-bit; function pointer types partially recovered (`int (__fastcall *)(int)`).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Arrays shown as pointer arithmetic (`dword_1848[result]`); no struct definitions visible in output; `jmp_buf` shown as `struct __jmp_buf_tag *` without field details; matrix in `loop_multi_exit` loses 2D structure.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts throughout (`(__int64)`, `(unsigned int)`); `duffs_device` has inline assembly bypassing type system; function pointer calls with generic types; no bounds checking visible on array access.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**13. Idiom Recognition:** Score: 5/10  
Justification: Basic loop patterns (for/while/do-while) recognized; switch statements mostly recovered; `duffs_device` not recognized (shows as assembly); state machine patterns in `state_machine` not explicitly identified.

**14. Anomaly Indication:** Score: 4/10

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `n > 0` for factorial); no postcondition inference; error conditions not documented; return value semantics unexplained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**17. Module Organization:** Score: 6/10
