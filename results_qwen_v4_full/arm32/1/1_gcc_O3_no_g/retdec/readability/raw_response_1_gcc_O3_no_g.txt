# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `v1`, `v2`, `v3` through `v19` with no semantic meaning. For example, `duffs_device` has 19 temporaries (`v1`-`v19`) that should represent `dst`, `src`, `count` pointers based on the original source.

**2. Function Naming Quality:** Score: 4/10  
Justification: Some original names preserved (`duffs_device`, `non_local_jump`, `computed_goto`) but 15+ functions completely missing. Test functions renamed to `test_control_flow_l1/2/3` losing individual function identity.

**3. Literal Clarity:** Score: 5/10  
Justification: String literals preserved as UTF-8 encoded globals (`g1`, `g2`, `g3`), but magic numbers like `0x116c8` in `computed_goto` and case values `-2`, `-3`, `-4` in `duffs_device` switch are unexplained.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive compiler artifacts including redundant variable assignments (`v4 = v1`, `v5 = v2` repeated 19 times in `duffs_device`), complex struct initialization for `setjmp`, and unreachable code paths.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: `duffs_device` uses explicit `goto lab_0x10dbc` instead of loop construct. `computed_goto` returns address `0x116c8` instead of proper label dispatch. Original for/while loops completely lost.

**6. Function Decomposition:** Score: 2/10  
Justification: 15+ original functions completely missing from decompilation output. Test functions constant-folded to print expected values instead of calling actual functions. Only 8 functions detected vs 20+ in source.

**7. Code Linearization:** Score: 4/10  
Justification: `duffs_device` has backward goto creating loop effect but not recognized as such. `non_local_jump` has reasonable top-to-bottom flow but `computed_goto` returns raw address breaking linearity.

**8. Redundancy Elimination:** Score: 2/10  
Justification: `duffs_device` has 19 nearly identical variable assignments (`v4 = v1`, `v5 = v2`, etc.) that are compiler register-spilling artifacts. No dead code elimination performed.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses `int32_t` consistently but loses original `int` semantics. `computed_goto` parameter `a2` is `uint32_t` but should be `int`. Function pointer types completely lost (no recovery of callback signatures).

**10. Composite Type Recovery:** Score: 3/10  
Justification: `setjmp` buffer represented as anonymous struct with `e0[8]`, `e1`, `e2.e0[1]` fields instead of `jmp_buf`. No struct recovery for any original data structures.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts like `(int32_t)a2`, `(int32_t)a1` in `duffs_device`. Pointer arithmetic uses `(int32_t *)v5` without proper type context. `computed_goto` returns address as int.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled code. No explanation of Duff's Device pattern, setjmp/longjmp semantics, or computed goto mechanism. Original source had extensive documentation.

**13. Idiom Recognition:** Score: 1/10  
Justification: Duff's Device completely unrecognizable (switch with fall-through cases -2,-3,-4). State machine patterns, loop idioms, and recursion patterns not identified or documented.

**14. Anomaly Indication:** Score: 2/10  
Justification: `computed_goto` returns raw address `0x116c8` without warning this is abnormal. No indication that 15+ functions are missing. Setjmp struct initialization not flagged as compiler-specific.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions documented (e.g., `duffs_device` requires `n > 0`). No postconditions or error condition documentation. Original test validation comments completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**17. Module Organization:** Score: 4/10
