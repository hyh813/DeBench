# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `a0`, `a1`, `temp_1`) with no semantic meaning. Original source had descriptive names like `temp1`, `sum`, `count`, `total` that are completely lost.

**2. Function Naming Quality:** Score: 6/10  
Justification: Core test function names preserved (`sequential_ops`, `recursion_factorial`, `double_value`), but helper functions use generic names (`sub_10454`, `sub_10518`) and runtime calls are opaque (`armg_calculate_condition`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers abound: `4294967295` (UINT_MAX/-1) appears repeatedly without explanation, `1717986919` in division operations, and raw addresses like `0x23050` lack context.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: pervasive `armg_calculate_condition` calls, stack canary checks (`__stack_chk_guard`), and register spill variables clutter the code without adding semantic value.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Critical failures: `if_elseif_long`, `switch_small`, `switch_large` all return constant -1 (completely broken). `loop_while`/`loop_dowhile` use bizarre multiplication patterns instead of division. Goto labels (`LABEL_1053c`) replace structured constructs.

**6. Function Decomposition:** Score: 5/10  
Justification: Function boundaries mostly preserved, but parameter lists corrupted (`nested_if_deep` shows 4 params instead of 5, `goto_forward` has 4 params instead of 1). Some functions like `double_value` remain clean.

**7. Code Linearization:** Score: 4/10  
Justification: Excessive forward jumps (`goto LABEL_1053c`, `goto LABEL_10558`) disrupt top-to-bottom reading. `armg_calculate_condition` creates convoluted branching that obscures simple if-else logic.

**8. Redundancy Elimination:** Score: 4/10  
Justification: `armg_calculate_condition` calls repeat with identical arguments throughout. Stack canary checks duplicate across functions. Some functions over-simplified to point of losing functionality.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: All types degraded to generic `unsigned int` or `int`. Original `int` parameters sometimes incorrectly become `unsigned int`. Function pointers typed as `unsigned int *` instead of proper function pointer signatures.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Arrays completely lost—`int arr[] = {10, 20, 30, 40, 50}` becomes individual variables (`v0 = 10; v1 = 20;`). No struct recovery evident. Matrix in `loop_multi_exit` becomes scattered local variables.

**11. Type Safety:** Score: 4/10  
Justification: Function pointers incorrectly typed as `unsigned int *`. Pointer arithmetic on arrays uses `v6 += 1` instead of proper indexing. Multiple unsafe implicit conversions between signed/unsigned.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompilation confidence indicators, no documentation of non-obvious logic. Completely bare code.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition. State machines not identified, recursion not labeled, switch-case fallthrough in `switch_fallthrough` partially recovered but not explained as intentional fall-through.

**14. Anomaly Indication:** Score: 2/10  
Justification: Broken functions (`if_elseif_long` returning -1) presented confidently without warning. `armg_calculate_condition` calls unexplained. No uncertainty indicators for problematic decompilation.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., `n > 0` for factorial). Error conditions unexplained. Return value semantics not documented anywhere.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls preserved but relationships not documented. Global variable usage unclear. Callback relationships in `process_with_callback` visible but not explained.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single flat file. No logical grouping by control flow level. Static functions not properly scoped. No header file structure.

**18. Build System Compatibility:** Score: 3/10  
Justification: References external ARM runtime (`__aeabi_idiv`, `__libc_start_main`). Would require significant stubbing and platform-specific work to compile standalone.

**Overall Readability Score:** 3.7/10

### Critical Defects:
- **Functions completely broken**: `if_elseif_long`, `switch_small`, `switch_large` all return constant `4294967295` (-1) instead of implementing logic
- **Parameter corruption**: `nested_if_deep` shows 4 parameters instead of original 5; `goto_forward` has 4 parameters instead of 1
- **Pervasive `armg_calculate_condition`**: Obscures simple comparisons with opaque runtime calls throughout
- **Zero comments**: No documentation, no confidence indicators, no explanation of non-obvious logic
- **Array/struct loss**: All composite types flattened to individual variables, losing semantic structure

### Best Quality Aspect:
- **Function Naming Quality (6/10)** - Core test function names were preserved from debug symbols (`sequential_ops`, `recursion_factorial`, `double_value`), providing some semantic anchor points.

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any comments makes understanding decompiler confidence, algorithm intent, or problematic areas impossible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Detect and flag broken decompilation** - Functions returning constant values should be marked as "UNRELIABLE" with confidence indicators
2. **Recover array/struct patterns** - Detect sequential stack variables with regular access patterns and reconstruct as arrays
3. **Replace `armg_calculate_condition` with readable comparisons** - Map ARM condition codes back to standard C comparison operators (`>`, `<`, `==`, etc.)
