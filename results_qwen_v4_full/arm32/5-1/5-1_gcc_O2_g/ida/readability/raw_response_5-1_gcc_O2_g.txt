# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are predominantly generic (`v0`, `v1`, `v2`, `exception`) with no semantic meaning. Only `this`, `obj`, `a`, `b` provide minimal context; names like `capture_by_value` from source are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: C++ mangled names recovered (`_Z20test_cpp_member_funcv`) but not human-readable. Some class methods properly named (`Base::virtual_func`), but 25 functions failed entirely and helpers use `sub_1092C` pattern.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive without explanation (`4700`, `0x1Fu`, `4u`, `0x10u`, `0x18u`). String literals like `"Test"` preserved but numeric constants lack symbolic names.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts: thunk functions (`_ZThn8_N12MultiDerived5funcBEv`), frame pointer warnings ("positive sp value has been detected"), register spilling variables (`v4`, `va`), and 25 failed external function stubs.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Most test functions collapsed to constant returns losing all original control flow. Exception handling completely lost (`test_cpp_exception` just throws). Template functions (`template_max`, `template_swap`) show natural if/return patterns.

**6. Function Decomposition:** Score: 5/10  
Justification: Test functions over-collapsed (single return statements). Class methods properly separated but excessive thunk functions for virtual dispatch create fragmentation. No logical grouping of related functionality.

**7. Code Linearization:** Score: 6/10  
Justification: Code that exists reads top-to-bottom reasonably well. No excessive gotos in recovered code. However, constant-folded functions provide no flow to evaluate.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Thunk functions create significant redundancy (virtual/non-virtual thunks for same method). Pointer arithmetic repeated across similar class methods without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Mix of `int`, `size_t`, `void*`, `__int64` with inconsistent precision. Return type of `test_cpp_member_func` changed from `int` to `size_t`. Many pointer types degraded to `void*`.

**10. Composite Type Recovery:** Score: 4/10  
Justification: `SimpleClass` struct partially recovered (`name[32]` visible) but most class layouts accessed via pointer arithmetic (`this->_vptr.Base`, `this[-1].dataB`). Vtable structures not reconstructed as named types.

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts visible: `(void *)operator new(4u)`, `*(_DWORD *)v0`, `(char *)&this->dataA`. Raw pointer arithmetic throughout class member access without bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Almost no comments beyond auto-generated function signatures. No algorithm explanations, no decompilation confidence indicators. Original source comments completely lost.

**13. Idiom Recognition:** Score: 4/10  
Justification: C++ patterns visible (vtable calls, RTTI checks, dynamic_cast) but not named or explained. No recognition of RAII, smart pointer patterns, or lambda capture semantics.

**14. Anomaly Indication:** Score: 6/10  
Justification: Some anomalies flagged ("FAILED to decompile", "positive sp value has been detected", "__noreturn" attributes). However, many semantic losses (exception handling, lambda captures) go unnoted.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants documented. Exception contracts completely lost. No null checks or bounds validation comments despite visible pointer arithmetic.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible within decompiled code but 25 external functions failed. Virtual dispatch partially recovered (vtable calls visible) but callback relationships unclear.

**17. Module Organization:** Score: 4/10  
Justification: Functions listed sequentially without logical grouping. No header file structure, no namespace organization. Related class methods scattered throughout output.

**18. Build System Compatibility:** Score: 3/10  
Justification: 25 functions failed to decompile requiring stubs. External dependencies (`_cxa_throw`, `operator new`, `__dynamic_cast`) not properly declared. Will not compile without significant modification.

**Overall Readability Score:** 4.5/10

### Critical Defects:
- **Constant folding destroyed semantics**: Test functions like `test_cpp_virtual_func()` return literal `42` instead of showing actual computation logic
- **Exception handling completely lost**: `test_cpp_exception` shows only throw, no catch blocks or stack unwinding logic
- **25 functions failed to decompile**: Critical runtime functions (`__cxa_throw`, `__dynamic_cast`, `operator new`) unavailable
- **Generic variable naming**: `v0`, `v1`, `v2` throughout obscures variable purpose (e.g., `exception` should be `int_exception_value`)
- **Pointer arithmetic instead of field access**: Class members accessed via `*((_DWORD *)this->_vptr.MiddleA - 3)` instead of named fields

### Best Quality Aspect:
- **Code Linearization (scored 6/10)** - Where code exists, it reads top-to-bottom without excessive jumps or goto statements

### Most Needs Improvement:
- **Comment Quality & Presence (scored 3/10)** - Virtually no documentation explaining non-trivial logic, algorithms, or decompilation uncertainty

### Top 3 Recommendations for Decompiler Improvement:
1. **Preserve control flow for test functions**: Avoid aggressive constant folding that eliminates semantic logic; emit original computation even when result is known
2. **Recover C++ exception handling structures**: Reconstruct try/catch blocks from `.gcc_except_table` and landing pad information
3. **Infer semantic variable names from usage patterns**: Variables used in string operations should be named `buffer`/`dest`, loop counters should be `i`/`idx`, exception objects should indicate type
