# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Generic names dominate (`iVar1`, `iVar2`, `this_local`, `param_1`), though test functions preserve some original names (`capture_by_value`, `obj`, `result`). Lambda capture fields use semi-descriptive names (`__capture_by_value`, `__capture_by_ref`) but most temporaries are meaningless.

**2. Function Naming Quality:** Score: 7/10  
Justification: Test functions retain original names (`test_cpp_member_func`, `test_cpp_constructor`) and class methods preserve names (`SimpleClass::getValue`). However, compiler-generated helpers get generic names (`FUN_00010a68`) and template instantiations are verbose but accurate.

**3. Literal Clarity:** Score: 6/10  
Justification: Magic numbers present without explanation (`0x1234`, `0x1f`, `0x32`), though some are recognizable (0x1234 = class ID). String literals referenced as `&DAT_0001426c` instead of showing actual content inline.

**4. Noise Reduction:** Score: 4/10  
Justification: 203 functions from 540 lines of source (7x expansion) with massive template instantiation bloat. WARNING comments add noise ("Removing unreachable block", "Unknown calling convention") and compiler helpers (`__do_global_dtors_aux`, `register_tm_clones`) clutter output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Test functions maintain reasonable flow, but exception handling becomes convoluted (`__cxa_allocate_exception`, `__cxa_throw`). RTTI code uses goto labels (`LAB_00011954`) and virtual function calls are clear but indirect (`(**obj->_vptr_Base)(obj,x)`).

**6. Function Decomposition:** Score: 5/10  
Justification: Template expansions create excessive fragmentation with many single-purpose helpers (`_M_ptr`, `_M_deleter`, `get<0,...>`). Test functions are appropriately sized but destructor chains are verbose with multiple variants (with/without `__in_chrg`).

**7. Code Linearization:** Score: 6/10  
Justification: Most code reads top-to-bottom, but exception handling introduces jumps and virtual dispatch has indirect calls. Some forward references in template code disrupt flow, though basic test functions remain linear.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Template instantiations create massive duplication (`unique_ptr<int>` vs `unique_ptr<int[]>` have nearly identical code). Multiple destructor variants for same class and getter/setter patterns repeated across template specializations.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 6/10  
Justification: Class types mostly preserved (`SimpleClass`, `LifecycleClass`, `Base`, `Derived`) and template types verbose but accurate (`Container<int>`, `unique_ptr<int,_std::default_delete<int>_>`). However, some fields degrade to generic types (`undefined4`, `void**`, `pointer`).

**10. Composite Type Recovery:** Score: 6/10  
Justification: Class structures recognized with field names partially preserved, but some become generic (`field_0x10`, `super_BaseA`, `super_BaseB`). Virtual inheritance handling shows issues ("WARNING: Struct ignoring overlapping field") though vtable pointers recovered (`_vptr_Base`).

**11. Type Safety:** Score: 5/10  
Justification: Many casts present, especially in virtual function calls and pointer arithmetic in virtual inheritance code. Questionable operations like `(int)((ulonglong)uVar1 >> 0x20)` and template code has many type-erased operations.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 5/10  
Justification: DWARF prototype comments present but auto-generated. WARNING comments indicate issues but don't explain them. No algorithm or logic explanations, though some useful notes exist ("non-virtual thunk", "virtual thunk").

**13. Idiom Recognition:** Score: 4/10  
Justification: C++ patterns visible but not explicitly identified. RAII pattern in smart pointers not named, virtual dispatch structure clear but not labeled. Lambda becomes anonymous struct (pattern visible but not named as "lambda closure").

**14. Anomaly Indication:** Score: 6/10  
Justification: Warnings flag some issues ("Struct ignoring overlapping field", "Unknown calling convention", "Removing unreachable block"). However, some issues not flagged (e.g., exception handling incompleteness in `test_cpp_exception`).

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No pre/post conditions documented, exception contracts not clear, and no invariant documentation. Function purposes inferable from names but not explicitly stated as contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but virtual dispatch relationships unclear. Template instantiation connections obscured, though inheritance relationships visible through `super_` prefixes. No call graph documentation.

**17. Module Organization:** Score: 5/10  
Justification: All code in one output file with no logical grouping beyond function order. Class methods grouped but not explicitly, and no namespace or file structure recovery from original source.

**18. Build System Compatibility:** Score: 3/10  
Justification: Won't compile standalone (missing headers, external dependencies). Compiler-specific constructs (`__thiscall`, `__aeabi_*`) and external function references (`operator_new`, `__cxa_throw`) with no dependency documentation.

**Overall Readability Score: 5.2/10**

### Critical Defects:
- **Variable `iVar1`, `iVar2`, `iVar3`** should be named based on usage (e.g., `r1`, `r2`, `r3` as in source)
- **Template instantiation bloat** creates 7x code expansion with near-duplicate functions
- **Exception handling incomplete** - `test_cpp_exception` decompilation cuts off mid-function
- **Magic numbers unexplained** - `0x1234`, `0x1f`, `0x32` used without constant definitions
- **Virtual inheritance fields generic** - `field_0x10`, `field_0x8` lose semantic meaning

### Best Quality Aspect:
- **Function Naming Quality (7/10)** - Test functions and class methods retain original semantic names accurately, making core functionality identifiable despite other issues.

### Most Needs Improvement:
- **Assertion & Contract Recovery (3/10)** - No preconditions, postconditions, or exception contracts documented, making it difficult to understand function guarantees and error handling expectations.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement C++ template consolidation** - Detect and collapse near-identical template instantiations with parameterized comments rather than emitting duplicate code
2. **Recover variable names from debug symbols** - At -O0 with `-g`, DWARF contains original variable names that should be prioritized over generic `iVar#` patterns
3. **Add C++ idiom annotations** - Label RAII patterns, lambda closures, and virtual dispatch explicitly with comments explaining the high-level construct being represented
