# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `v2`, `ptr` with no semantic meaning. Original variables like `capture_by_value`, `instance_count`, `multiplier` are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some test functions recovered (`test_cpp_member_func`, `test_cpp_virtual_func`), but most are mangled (`_Z20test_cpp_member_funcv`) or generic (`sub_10a68`). Class methods partially recovered (`SimpleClass::getValue`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere: vtable addresses (83440, 83416, 83396), array sizes as literals (31, 35), and unexplained constants (0x1461c, 0x14600). No named constants recovered.

**4. Noise Reduction:** Score: 4/10  
Justification: Stack canary checks (`__stack_chk_guard`) in every function, C++ runtime calls (`__cxa_*`, `__aeabi_*`), and compiler artifacts (`/* do not return */`) clutter the code significantly.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Uses `goto` labels (LABEL_11035, LABEL_11036) instead of structured conditionals. Exception handling uses raw jumps instead of try/catch blocks. At -O0, this is unacceptable.

**6. Function Decomposition:** Score: 4/10  
Justification: C++ template internals fully exposed as separate functions (`std::_Tuple_impl`, `std::__get_helper`, `std::_Head_base`). 2536 LOC vs 540 original shows severe over-fragmentation.

**7. Code Linearization:** Score: 5/10  
Justification: Forward jumps to labels disrupt top-to-bottom reading. Exception handling paths jump around significantly. Some functions read linearly but many don't.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Template instantiations create massive code duplication. Each smart pointer operation spawns 5-10 helper functions. No common subexpression elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types degraded to `unsigned int`, `void*`, `char*`. Original types like `size_t`, `int32_t`, class references lost. Function signatures often wrong (missing `this` pointer explicitly).

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs named generically (`struct_0`, `struct_1`) with fields `field_0`, `field_4`. Class layouts partially recovered but field names lost (`padding_0`, `field_28`).

**11. Type Safety:** Score: 4/10  
Justification: Numerous unsafe casts (`(unsigned int)`, `(char *)`), pointer arithmetic on class objects, and type-erased function calls. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero meaningful comments. Only decompiler annotations like `/* do not return */`. No algorithm explanations, no API contracts, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 3/10  
Justification: C++ idioms completely lost: RAII patterns appear as raw new/delete, virtual dispatch shown as raw vtable calls, lambdas as awkward functor structs. No pattern recognition.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomalies flagged despite significant uncertainty. Indirect calls, vtable layouts, and this-pointer adjustments presented confidently without warnings.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. Exception contracts lost. No null checks documented even where present in source.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships unclear. Virtual dispatch partially recovered but inheritance relationships not documented. No call graph provided.

**17. Module Organization:** Score: 4/10  
Justification: All 2536 lines in single flat file. No logical grouping by class or feature. Runtime helpers mixed with application code. No namespace structure.

**18. Build System Compatibility:** Score: 3/10  
Justification: Will not compile standalone. Missing headers, incorrect function signatures, undefined external symbols (`g_25008`, `g_145f0`). Requires extensive patching.

### Critical Defects:
- **Variable names completely generic** (`v0`, `v1`, `v2`) losing all semantic meaning from original (`capture_by_value`, `instance_count`, `multiplier`)
- **Stack canary checks in every function** adding noise that wasn't in original source logic
- **C++ template internals fully exposed** creating 5x code bloat (2536 vs 540 LOC)
- **Zero comments or documentation** despite complex C++ features (virtual inheritance, smart pointers, lambdas)
- **Magic numbers everywhere** (vtable addresses 83440, 83416, type_info addresses 0x1461c) with no explanation

### Best Quality Aspect: **Function Naming Quality (5/10)**
- Some test function names were recovered (`test_cpp_member_func`, `test_cpp_virtual_func`) providing minimal semantic anchors in an otherwise opaque codebase.

### Most Needs Improvement: **Comment Quality & Presence (2/10)**
- Complete absence of any explanatory comments makes understanding complex C++ features (diamond inheritance, smart pointer custom deleters, lambda captures) nearly impossible without original source.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable name inference** from usage patterns (e.g., variables used in size comparisons should be named `size`, `count`, `length`)
2. **Recover high-level C++ constructs** by recognizing patterns (try/catch from `__cxa_*` calls, loops from backward jumps, smart pointers from RAII patterns)
3. **Add decompilation confidence annotations** flagging uncertain reconstructions (indirect calls, vtable layouts, this-pointer adjustments) with warnings for manual review
