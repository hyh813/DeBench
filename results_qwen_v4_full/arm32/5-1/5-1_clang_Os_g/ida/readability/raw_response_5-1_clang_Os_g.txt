# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are almost entirely generic (`v0`, `v1`, `v2`, `temp`, `result`) with no semantic meaning. For example, `test_cpp_member_func` uses `v1[4]` instead of `name[32]` for the class's name buffer.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions retain readable names (`test_cpp_member_func`, `test_cpp_virtual_func`), but many have mangled C++ names (`_Z20test_cpp_member_funcv`). Some match source intent but require demangling.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `4700`, `1001`, `130`, `14` appear without explanation. These are constant-folded results but provide no insight into original computation logic.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: `HIBYTE`, `LODWORD`, stack variables initialized to 0 (`v2=0, v3=0, v4=0...`), and `BYREF` annotations clutter the code without adding clarity.

**10. Composite Type Recovery:** Score: 5/10

**Overall Readability Score:** 4.8/10

### Critical Defects:
- **Constant folding obscures logic**: Most test functions return pre-computed constants (42, 71, 650) instead of showing actual computation, making verification impossible
- **Generic variable naming**: All variables use `v0`, `v1`, `v2` pattern with no semantic meaning (e.g., `v1[4]` should be `name[32]`)
- **22 failed decompilations**: Critical runtime functions (`_cxa_throw`, `operator new`, `printf`) not recovered, breaking build compatibility
- **No C++ semantic recovery**: Exception handling, RTTI, and virtual dispatch shown as raw pointer operations without C++ context
- **Missing class structure**: Class fields not properly named or typed (e.g., `Container` data array lacks type specification)

### Best Quality Aspect:
- **Code Linearization (scored 8/10)** - Code reads top-to-bottom naturally with minimal jumps; Container methods show clean sequential flow without confusing control transfers.

### Most Needs Improvement:
- **Assertion & Contract Recovery (scored 2/10)** - No preconditions, postconditions, or exception contracts documented; critical for understanding C++ semantics like exception safety guarantees.

### Top 3 Recommendations for Decompiler Improvement:
1. **Implement semantic variable naming**: Analyze usage patterns to infer names (e.g., variables used with `strlen` should be named `string` or `buffer`, not `v1`)
2. **Recover C++ high-level constructs**: Recognize exception handling patterns (`_cxa_throw`/`_cxa_begin_catch`), RTTI operations, and vtable dispatch as C++ semantics rather than raw pointer operations
3. **Document constant folding**: When compiler has pre-computed values, annotate with original computation (e.g., `// Original: r1 + r2 + r3 = 10 + 34 + 4660`) to preserve semantic intent
