# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Nearly all variables use generic names like `v1`, `v2`, `v3`, `a1`, `a2` instead of semantic names (e.g., `local_array` uses `v1`, `v2` instead of `arr`, `i`). Only parameter names like `nmemb`, `size` in calloc wrapper retain meaning.

**2. Function Naming Quality:** Score: 5/10  
Justification: Mixed resultsŌĆöoriginal names preserved for user functions (`local_vars`, `heap_basic`, `double_value`) but compiler helpers use mechanical names (`function_588`, `function_594`, `unknown_ffff0fc0`). Sync functions are overly verbose (`__sync_fetch_and_add_4`).

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers throughout without explanation (e.g., `2048`, `40`, `8` for struct sizes). String literals stored as hex-encoded globals (`g15`, `g17`) instead of readable strings. No named constants recovered.

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive `__sync_*` function implementations (30+ functions) bloat the output. Compiler startup code (`_init`, `_start`, `__do_global_dtors_aux`) included unnecessarily. Many redundant type casts like `(int32_t *)`.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: For-loops transformed to while-loops with manual counters (e.g., `local_array` uses `i > -10` backward iteration). `linked_list_heap` uses goto labels (`lab_0xd20`). Loop conditions are inverted and harder to follow.

**6. Function Decomposition:** Score: 5/10  
Justification: User functions have reasonable granularity, but `test_stack_memory` (0x92c-0xa34) inlines all test logic into one 260-line function. `__sync_*` functions are over-fragmented with near-identical implementations.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but `linked_list_heap` has goto jumps disrupting flow. `vla_stack` and `alloca_usage` have complex stack pointer arithmetic that breaks linear reading.

**8. Redundancy Elimination:** Score: 4/10  
Justification: 30+ `__sync_*` functions with nearly identical logic (bit manipulation patterns repeated). Loop initialization code duplicated across `heap_basic`, `heap_array`, `memory_leak`. No common subexpression elimination visible.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 4/10  
Justification: Everything degraded to `int32_t`ŌĆöno `size_t`, `ptrdiff_t`, or proper signedness. Pointers typed as `int32_t *` instead of specific types. `global_array_access` takes `uint32_t` but source used `int`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failureŌĆö`Point` struct lost (fields accessed via `+4` offset), `Node` struct lost (linked list uses raw pointer arithmetic `*(int32_t *)(v4 + 4)`). `TreeNode` not recovered at all.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe casts everywhere (`(int32_t *)a1`, `(char *)&g6`). Pointer arithmetic on `int32_t *` without bounds checking. `dangling_pointer` returns use-after-free value without warning.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables (`g1`-`g25`) have no semantic connection to source names (`global_counter`, `file_scope_static`). Callback mechanism (`global_func_ptr`) obscured.

**17. Module Organization:** Score: 4/10  
Justification: Functions ordered by address, not logical grouping. Test functions mixed with implementation. No header file separation. Static functions not properly scoped (all appear global).

**18. Build System Compatibility:** Score: 6/10  
Justification: Includes necessary headers (`stdio.h`, `stdlib.h`, `string.h`). Has all function prototypes. Would likely compile with minor fixes, but `unknown_ffff0fc0` is unresolved and `extern_function` needs linking.
