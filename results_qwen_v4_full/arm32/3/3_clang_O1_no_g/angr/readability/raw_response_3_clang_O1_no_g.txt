# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are overwhelmingly generic (`v0`, `v1`, `v2`, `ptr`, `idx`) with no semantic meaning. For example, `local_array` uses `v0`, `v1`, `v3` instead of `arr`, `i`, `result`. Only occasional names like `result` and `index` provide hints.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`local_vars`, `heap_basic`, `vla_stack`), but many are mechanical (`sub_400568`, `$a.30`). Function `double_value` became `$a.30` losing all semantic meaning.

**3. Literal Clarity:** Score: 3/10  
Justification: Pervasive magic numbers like `4294967295` (unsigned -1), `4294967286`, `0x800` without explanation. No constants recovered; even simple values like `10` in loops lack context.

**4. Noise Reduction:** Score: 3/10  
Justification: Heavy decompiler artifacts: `armg_calculate_condition()` calls throughout, redundant variable assignments (`v1 = v3; v0 = v4`), and spurious goto labels (`LABEL_4006b0`, `LABEL_40076c`).

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Loops converted to `do-while` with goto instead of natural `for`/`while`. Example: `local_array` uses `i -= 1` with `i != 4294967286` instead of `i < 10`. Excessive goto disrupts flow.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved from source, but test functions became monolithic. `test_stack_memory` inlines all test logic instead of calling individual functions properly.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps (`goto LABEL_4006b0`) and backward jumps for loops disrupt top-to-bottom reading. `vla_stack` has complex jump patterns that obscure the simple loop logic.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Duplicate condition checks appear (e.g., `armg_calculate_condition` called twice in same branch). Register spilling creates redundant assignments (`v1 = v4; v0 = v5`).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Everything becomes `unsigned int` regardless of original type. `int` parameters become `unsigned int`, losing sign semantics. Pointer types degraded to `void*` or `unsigned int*`. No const-correctness preserved.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered with generic names (`struct_0`, `struct_1`) and fields (`field_0`, `field_4`, `padding_0`). `Point` struct lost; `Node` struct has `field_0` instead of `data`, `field_4` instead of `next`.

**11. Type Safety:** Score: 3/10  
Justification: Unsafe pointer arithmetic throughout (`(char *)&v0 + 2 * idx`). Function pointers treated as `unsigned int`. `dangling_pointer` returns `struct_0 **` but accesses freed memory without warning.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero comments in decompiled output. No algorithm explanations, no decompiler confidence indicators, no API documentation. Completely bare code.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns (malloc/free, array iteration) not identified. `armg_calculate_condition` obscures simple comparisons. No recognition of linked list traversal or tree patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: Dangerous patterns like `dangling_pointer` (use-after-free) and `double_free` not flagged. `memory_leak` has no warning about missing free. Decompiler presents undefined behavior confidently.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `n > 0` for `vla_stack`). Null checks exist but aren't marked as contracts. Return value semantics not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but globals use opaque names (`g_414008`, `g_402f62`). `extern_function` recovered but `extern_global_var` relationship unclear. Callback through `global_func_ptr` partially recovered.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in source order but no logical grouping. Runtime initialization functions (`_init`, `_start`, `__do_global_dtors_aux`) mixed with application code. No header file structure.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code contains `armg_calculate_condition` calls that won't compile without Angr runtime. Incomplete function bodies (`return ...;`). Many external dependencies (`__libc_start_main`, `__gmon_start__`) undeclared.

**Overall Readability Score:** 3.6/10

### Critical Defects:
- **`armg_calculate_condition()` artifacts** throughout code make logic unreadable and non-compilable
- **Variable names** like `v0`, `v1`, `v2` provide zero semantic information about purpose
- **Control flow** uses goto/labels instead of natural loop constructs (for/while)
- **Struct fields** all generic (`field_0`, `field_4`) losing original member names (`data`, `next`, `x`, `y`)
- **Dangerous patterns** (use-after-free, memory leaks) not flagged with any warnings

### Best Quality Aspect:
- **Function Decomposition (6/10)** - Function boundaries mostly preserved from source; individual test functions remain separate rather than being inlined into monolithic code.

### Most Needs Improvement:
- **Comment Quality & Presence (1/10)** - Complete absence of any comments, documentation, or decompiler confidence indicators makes understanding intent nearly impossible.

### Top 3 Recommendations for Decompiler Improvement:
1. **Replace `armg_calculate_condition()` with native C comparisons** - This Angr-specific artifact obscures simple boolean logic and prevents compilation
2. **Infer variable names from usage patterns** - Variables used in loop bounds should be named `i`, `count`, `size`; pointers should indicate target type
3. **Flag undefined behavior patterns** - Use-after-free (`dangling_pointer`), double-free, and memory leaks should be annotated with warnings for analyst awareness
