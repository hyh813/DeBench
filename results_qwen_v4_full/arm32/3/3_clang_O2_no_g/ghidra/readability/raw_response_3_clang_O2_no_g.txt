## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `uVar2`, `local_808`) with zero semantic meaning. For example, `local_808` in `large_stack_frame` should be `large_buf`, and `param_1` in `heap_basic` should be `n` or `element_count`.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are excellently preserved from source (`local_vars`, `heap_basic`, `linked_list_heap`, `test_stack_memory`). Action-oriented names clearly indicate purpose.

**3. Literal Clarity:** Score: 3/10  
Justification: Pervasive magic numbers without explanation: `0x2a` (42), `0x14` (20), `0x3c` (60), `0x41` (65/'A'). In `address_of_local`, `0x2a` should be `42` or `LOCAL_VALUE`.

1. **Implement struct pattern recognition**: Detect consistent memory access patterns (e.g., `ptr[0]`, `ptr[1]` with fixed offsets) and propose struct definitions with field names based on usage context.

3. **Recover constant definitions from repeated literals**: Detect frequently-used magic numbers (`0x2a`=42, `0x14`=20) and suggest named constants with comments explaining their semantic meaning.
