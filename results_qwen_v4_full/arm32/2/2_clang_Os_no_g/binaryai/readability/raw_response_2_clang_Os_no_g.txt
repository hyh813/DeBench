# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `uVar2` with zero semantic meaning. Original source had meaningful names like `sum`, `len`, `c`, `x` that indicated purpose.

**2. Function Naming Quality:** Score: 8/10  
Justification: Function names are well-preserved (`process_char`, `array_1d_stack`, `struct_simple`) matching original source. Only helper functions like `FUN_00011594` are generic.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere without explanation (`0xbf`, `0x1a`, `0x10000`, `0x3fc00000`). Original used clear expressions like `c + 32` for ASCII conversion.

**4. Noise Reduction:** Score: 5/10  
Justification: Includes extensive compiler runtime functions (`__muldf3`, `__adddf3`, `__floatsidf`) that bloat output. Test functions have redundant printf patterns with hardcoded values.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Original `for(i=0; i<n; i++)` converted to awkward `do{... param_2 = param_2 + -1; ...} while(param_2 != 0)` patterns. Backward-counting loops are harder to read.

**6. Function Decomposition:** Score: 6/10  
Justification: Function sizes are reasonable, but test functions lost actual computation logic—many just print hardcoded values instead of calling functions.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but complex functions like `__muldf3` (200+ lines) have deeply nested conditionals disrupting flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Floating-point helper functions contain massive redundant bit-manipulation code. Test functions repeat identical printf patterns without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Critical type failures: `process_float` returns `void` instead of `float`, `process_ll` uses `(uint,int)` instead of `long long`, all structs degraded to `int*`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure—no struct definitions recovered. `Point3D` accessed as `param_1[0], param_1[1], param_1[2]`. Unions, bitfields, enums all lost.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe pointer arithmetic throughout (`*(int *)(param_1 + 0x14)`). Function pointers degraded to `code*`. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Zero meaningful comments. Only one anomaly marker (`UNRECOVERED_JUMPTABLE`). No algorithm explanations or API contracts.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns (array sum, linked list traversal, string length) not identified or named. No pattern hints provided.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies marked (`UNRECOVERED_JUMPTABLE`) but most type degradation issues silently accepted without warning.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions documented (e.g., `param_1 != NULL`). No error condition explanations for functions like `array_complex_index`.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls exist but test functions have hardcoded return values instead of actual calls, breaking call graph clarity.

**17. Module Organization:** Score: 5/10  
Justification: All code in single file with no logical grouping. Runtime helpers mixed with application code without separation.

**18. Build System Compatibility:** Score: 6/10  
Justification: Metadata indicates compilation succeeds, but requires many external runtime functions (`__muldf3`, `__adddf3`) that need linking.

**Critical Defects:**
- All struct/union/enum types completely lost—accessed via raw pointer arithmetic (`param_1[0]`, `param_1 + 0x14`)
- Function return types wrong: `process_float`, `process_double`, `process_ld` return `void` instead of floating-point types
- Test functions print hardcoded values instead of calling actual functions, breaking semantic meaning
- Variable names entirely generic (`param_1`, `iVar1`) with zero semantic information
- Floating-point operations expanded into 200+ line helper functions instead of recognizable `*`, `/`, `+` operations

**Best Quality Aspect:** Function Naming Quality (scored 8/10)  
- Function names preserved accurately from source, making high-level purpose immediately identifiable

**Most Needs Improvement:** Composite Type Recovery (scored 2/10)  
- Complete failure to recover any struct, union, or enum definitions—all composite types reduced to pointer arithmetic

**Top 3 Recommendations for Decompiler Improvement:**
1. Implement struct/union type inference from access patterns (e.g., `param_1[0], param_1[1], param_1[2]` → `struct {int x,y,z;}`)
2. Recover floating-point operations from runtime helper calls (`__mulsf3` → `*`, `__adddf3` → `+`)
3. Preserve function return types from calling convention analysis instead of defaulting to `void`
