# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Overwhelming use of generic names (`v0`, `v1`, `a0`, `a1`, `vvar_22{r64|4b}`) obscures semantic meaning. Only occasional meaningful names like `idx`, `ptr`, `i` appear. Original variables like `sum`, `len`, `current` are completely lost.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original names preserved (`double_value`, `process_int`, `process_char`, `array_1d_stack`), but many critical functions are `sub_103e8`, `sub_115a8`. Test functions like `test_data_types_l1` are preserved but have incorrect signatures.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers everywhere without explanation: `65` (should be `'A'`), `32` (lowercase offset), `0x80000000`, `2047`, `255`. Floating-point constants appear as raw integers (`0x3f000000`, `2576980378`).

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise from ARM condition code calculations (`armg_calculate_condition`, `armg_calculate_flag_c`, `armg_calculate_flag_v`). Excessive temporary variables and register spill artifacts throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Extensive goto usage (`LABEL_107f0`, `LABEL_10814`, `LABEL_113dd`) instead of natural loops and conditionals. While-loops recovered in some cases but with awkward `while(true)` + goto patterns.

**6. Function Decomposition:** Score: 5/10  
Justification: Some functions reasonable size, but floating-point library functions (`__muldf3`, `__aeabi_dadd`) are enormous (300+ lines). Test functions have incorrect parameter lists exposing internal variables.

**7. Code Linearization:** Score: 4/10  
Justification: Frequent forward jumps disrupt top-to-bottom reading. Deep nesting in floating-point functions with multiple label targets. Some functions like `array_string` read reasonably linear.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy in condition code handling - same `armg_calculate_condition` patterns repeated dozens of times. Dead code from register allocation visible throughout.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation: `char` becomes `unsigned int`, `short` becomes `unsigned short` then masked, `float`/`double`/`long double` all become `unsigned int` with library calls. Pointer types often just `unsigned int`.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered as `struct_0`, `struct_1` with `field_0`, `field_4`, `field_8` - no semantic field names. Original structs like `Point3D` (x,y,z), `Rect` (position, size) completely lost. Union types not distinguished.

**11. Type Safety:** Score: 3/10  
Justification: Multiple unsafe patterns: void* casts without explanation, function pointers as `unsigned int`, pointer arithmetic on integer types. `ptr_func_simple` uses `goto a0` for indirect call - dangerous pattern.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero meaningful comments. Only auto-generated labels and `/* do not return */` annotations. No algorithm explanations, no parameter documentation, no decompilation confidence indicators.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Character conversion (`c >= 'A' && c <= 'Z'`) becomes arithmetic comparisons. Floating-point operations completely obscured as integer bit manipulation. No design patterns identified.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags despite obvious issues: incorrect function signatures, unresolved indirect calls, potential undefined behavior in `array_oob`. Code presented confidently even when semantically wrong.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. NULL checks present but not explained. No error condition documentation for functions that can fail.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but callback relationships unclear. `ptr_func_simple(double_value, 5)` preserved but function pointer semantics lost. Global variable usage not documented.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single output with no logical grouping. No header file structure. Static functions not properly scoped. Test functions mixed with implementation.

**18. Build System Compatibility:** Score: 3/10  
Justification: Would require extensive stubs for `armg_calculate_condition`, `__printf_chk`, `__stack_chk_guard`. Missing type definitions, incorrect function signatures prevent direct compilation.

**Overall Readability Score:** 3.5/10

### Critical Defects:
- **Variable `v0`, `v1`, `a0`** should be named based on usage (e.g., `sum`, `index`, `input_value`)
- **Floating-point functions** (`process_float`, `process_double`, `process_ld`) completely unreadable - reduced to integer bit manipulation with `armg_calculate_condition` calls
- **Struct field names** all generic (`field_0`, `field_4`) losing semantic meaning (should be `x`, `y`, `z`, `position`, `size`, etc.)
- **Control flow** uses goto labels instead of natural loop/conditional constructs throughout
- **Function signatures** incorrect - test functions have extra parameters exposing internal stack variables

### Best Quality Aspect: **Function Naming Quality (5/10)**
- Some original function names preserved (`double_value`, `process_int`, `array_1d_stack`, `linked_list`) providing anchor points for understanding

### Most Needs Improvement: **Comment Quality & Presence (1/10)**
- Zero meaningful comments despite complex floating-point emulation and ARM condition code handling that desperately needs explanation

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover high-level types from debug symbols** - The `-g` flag should enable recovery of original type names (`char`, `short`, `float`) instead of generic `unsigned int`
2. **Implement ARM condition code abstraction** - Replace repetitive `armg_calculate_condition` patterns with natural C conditionals (`if`, `while`, `switch`)
3. **Add semantic annotation pass** - Detect common patterns (character conversion, array indexing, struct field access) and add explanatory comments with confidence indicators
