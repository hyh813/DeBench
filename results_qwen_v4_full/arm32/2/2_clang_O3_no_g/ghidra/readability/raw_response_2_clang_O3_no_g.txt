# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30% weight) ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `param_1`, `iVar1`, `local_1fc` with zero semantic meaning. Original source had meaningful names like `sum`, `len`, `current`, `head` that are completely lost.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names preserved accurately (`process_char`, `array_1d_stack`, `struct_simple`). This is the decompiler's strongest aspect, maintaining source-level naming.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere: `0xbfU`, `0x3fc00000` (float constants), `0x10000`. Test functions print pre-computed constants instead of actual computations, obscuring logic.

**4. Noise Reduction:** Score: 5/10  
Justification: Compiler artifacts present: massive stack variable lists in test functions (`local_c` through `local_1fc`), verbose floating-point helper functions (`__muldf3`, `__adddf3`) fully decompiled with 200+ lines each.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight) ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: For-loops converted to do-while patterns (`param_2 = param_2 + -1; ... while (param_2 != 0)`). Floating-point functions have extremely complex branching that obscures simple arithmetic operations.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries preserved but test functions are monolithic with 100+ local variables. Floating-point helpers should be library calls, not decompiled inline.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom but complex functions have forward jumps disrupting flow. Test functions have sequential printf calls that are reasonably linear.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Test functions show massive redundancy: 80+ local variables all initialized to `1` for stack space. This compiler optimization artifact was not cleaned up.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20% weight) ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation: `process_short` returns `int` not `short`, `process_long` takes `int` not `long`, float/double functions return `void` instead of proper types. Many `undefined4` types.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Complete failure - all struct/union types lost. `struct_simple` takes `int*` instead of `Point3D*`, field access via array indexing (`param_1[1]`, `param_1[2]`). No struct definitions recovered.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts present, pointer arithmetic without type safety, function pointers degraded to `code*`. Warning: "Could not recover jumptable" for `ptr_func_simple`.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15% weight) ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Only compiler warnings present ("WARNING: Removing unreachable block"). No algorithm explanations, no function purpose documentation, no parameter/return descriptions.

**13. Idiom Recognition:** Score: 3/10  
Justification: No pattern recognition - linked list traversal, array summation, tree recursion all appear as raw pointer arithmetic without idiom identification.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some warnings present for unreachable blocks and jumptables, but critical issues unflagged: test functions printing constants instead of computing, type mismatches, missing return values in float functions.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL`), no postconditions inferred, no error condition documentation despite many functions having implicit contracts.

## === LEVEL 5: CONTEXTUAL COHERENCE (10% weight) ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but relationships not documented. Global variables shown as `DAT_000124fb` without meaning. No call graph information provided.

**17. Module Organization:** Score: 4/10  
Justification: All 80 functions in single output with no logical grouping. Static functions not properly scoped, helper functions mixed with application code.

**18. Build System Compatibility:** Score: 5/10  
Justification: Requires significant patching to compile: external dependencies (printf, puts), floating-point helpers need linking, type mismatches need fixing.

**Overall Readability Score: 4.22/10**

### Critical Defects:
- **All variable names generic** (`param_1`, `iVar1`, `local_1fc`) - zero semantic meaning recovered
- **Complete struct/union type loss** - all composite types reduced to pointer arithmetic (`param_1[1]`, `param_1[2]`)
- **Type precision degraded** - `short`, `long`, `float`, `double` all collapsed to `int` or `undefined4`
- **Test functions optimized away** - print constant values instead of actual computation, losing test logic
- **Floating-point functions return `void`** - `process_float`, `process_double` don't return values despite source returning floats

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Most function names preserved accurately from source, making basic identification possible

### Most Needs Improvement:
- **Composite Type Recovery (2/10)** - Complete failure to recover any struct/union definitions, all accessed via raw pointer arithmetic losing all semantic structure

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct/union types from access patterns** - Detect consistent offset patterns (`[0]`, `[1]`, `[2]`) and propose struct definitions with field names
2. **Preserve original type information from debug symbols** - Since compiled with `-g`, type information should be recoverable for `short`, `long`, `float`, `double`
3. **Infer variable names from usage context** - Variables used in loop counters should be named `i`, `count`; accumulators should be `sum`; pointers should indicate target type
