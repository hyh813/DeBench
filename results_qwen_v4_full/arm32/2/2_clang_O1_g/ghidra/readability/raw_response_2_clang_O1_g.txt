# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 4/10  
Justification: Variables are generically named (`iVar1`, `iVar2`, `uVar1`, `pEVar2`, `bVar3`) with no semantic meaning. Original source had meaningful names like `sum`, `len`, `current` that are completely lost.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are excellently preserved from source (`process_char`, `array_1d_stack`, `ptr_single`, `struct_simple`, `binary_tree_sum`). Action verbs and semantic purpose are clear.

**3. Literal Clarity:** Score: 5/10  
Justification: Test functions show constant-folded magic numbers (`0x61`, `0x62`, `0xb`, `0xf`, `0x14`) instead of actual function calls. Original computations like `process_char('A')` are replaced with hardcoded results.

**4. Noise Reduction:** Score: 6/10  
Justification: Compiler runtime functions (`__muldf3`, `__adddf3`, `__mulsf3`) are present and verbose. "Unresolved local var" comments and "WARNING: Unknown calling convention" warnings add noise throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: For-loops converted to do-while with manual counter management (`n = n + -1; arr = arr + 1`). `enum_switch` uses lookup table instead of switch statement. `array_complex_index` has convoluted `SBORROW4` conditionals.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions are appropriately sized (most under 50 lines) with single responsibility. No excessive inlining or fragmentation. Test functions are large but expected for their purpose.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably well. Complex functions like `array_3d` and floating point helpers have forward jumps disrupting flow. Loop structures are recognizable despite do-while conversion.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No obvious dead code in user functions. Test functions have constant-folded results which loses original computation logic. Floating point runtime functions have expected redundancy.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 7/10  
Justification: Basic types preserved (`char`, `short`, `int`, `long`, `longlong`, `float`, `double`). `volatile` qualifier lost in `volatile_access`. Function pointer types degraded to generic `_func_int_int *` instead of proper signatures.

**10. Composite Type Recovery:** Score: 8/10  
Justification: Struct definitions preserved with field names (`Point3D`, `Rect`, `Widget`, `Node`, `TreeNode`, `Graph`). Field access correct (`p->x`, `r->position.x`). Bitfield struct `Flags` recovered but access through bit manipulation instead of field names.

**11. Type Safety:** Score: 7/10  
Justification: Some unsafe casts present: `(int)handle`, `(void *)size`, `(uint)*(byte *)p`. Pointer arithmetic generally correct. No buffer overflow patterns introduced by decompiler.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Comments are auto-generated warnings ("Unresolved local var", "WARNING: Unknown calling convention"). No algorithm explanations, API contracts, or useful documentation. Original source comments completely lost.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. Linked list traversal, binary tree recursion, and switch-on-enum patterns present but not named or documented. Common idioms completely unrecognized.

**14. Anomaly Indication:** Score: 5/10  
Justification: "Unresolved local var" and "WARNING: Removing unreachable block" indicate some issues. No indication of potential undefined behavior in `array_oob`. Anomaly flagging is inconsistent.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition documentation (e.g., "requires: p != NULL"). No postcondition or invariant recognition. Error conditions not documented. Zero contract information recovered.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are clear but no call graph documentation. Callback relationships not explicit (`ptr_func_simple` doesn't document `f` is a callback). Global variable usage not documented.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single file (expected for decompilation). No header file separation. Static functions preserved (`double_value`, `complex_callback`). No namespace structure.

**18. Build System Compatibility:** Score: 4/10  
Justification: Requires compiler runtime functions (`__muldf3`, `__adddf3`, etc.) that need linking. Some type definitions would need to be inferred. Metadata claims "Compilation Result: Yes" but requires significant stubbing.
