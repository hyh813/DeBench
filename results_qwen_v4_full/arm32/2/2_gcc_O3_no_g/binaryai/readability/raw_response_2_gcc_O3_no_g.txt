# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (`param_1`, `iVar1`, `uVar1`, `local_64`) with zero semantic meaning. For example, `process_char` uses `param_1` instead of `character` or `input_char`, making it impossible to understand variable purpose without tracing usage.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names preserved from source (`process_char`, `array_1d_stack`, `struct_simple`), accurately reflecting purpose. However, helper functions like `FUN_00011a18` and floating-point routines (`__muldf3`) are generic or external.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers pervasive throughout: `0x41`, `0x1a`, `0x20` in `process_char` instead of `'A'`, `26`, `32`. Test functions have hardcoded expected values (`0x61`, `0x62`) instead of actual function call results, obscuring test logic.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler runtime artifacts included (`__muldf3`, `__addsf3`, `__aeabi_dadd` - 400+ lines of floating-point emulation). External function stubs call themselves (`__libc_start_main`, `puts`), creating confusing circular references.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions retain natural flow, but `binary_tree_sum` explodes into 8+ levels of nested do-while loops instead of clean recursion. Loop constructs recovered as `do-while` with manual counters instead of idiomatic `for` loops.

**6. Function Decomposition:** Score: 6/10  
Justification: Function boundaries mostly preserved, but `binary_tree` and `binary_tree_sum` show severe degradation with excessive local variables (16+ locals). Some functions incorrectly typed (`ptr_func_simple` returns `void` instead of `int`).

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but `array_3d` unrolls 125 array accesses into a single return statement (100+ lines). `test_array_types` has awkward stack variable initialization disrupting flow.

**8. Redundancy Elimination:** Score: 3/10  
Justification: `array_3d` completely unrolled (125 individual `param_1[index]` additions). `pointer_array` has 10 nearly identical if-blocks for each array index instead of a loop. Severe redundancy obscures original loop intent.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Struct types completely lost (all become `int*`). Floating-point functions use `undefined4`/`undefined8` instead of `float`/`double`. `process_ll` takes `(uint, int)` instead of `long long`. Pointer indirection levels often wrong (`ptr_triple` takes `undefined4*` not `int***`).

**10. Composite Type Recovery:** Score: 2/10  
Justification: Zero struct/union definitions recovered. All struct access becomes pointer arithmetic (`param_1[0]`, `param_1[1]`, `param_1[2]` for `Point3D`). `struct_bitfields` becomes `ushort*` with manual bit extraction. Enum types lost (become `int`).

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts: `(int *)(param_1 + -4)`, `(undefined4 *)*param_1`. Function pointers degraded to `code*`. `ptr_void` takes `uint*` instead of `void*`, losing type-agnostic semantics. No const-correctness preserved.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only decompiler header comment present. Zero documentation of algorithm intent, parameter meanings, or return values. Complex functions like `binary_tree_sum` have no explanation of the recursion pattern.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Linked list traversal (`linked_list`, `doubly_linked_list`) not identified as such. Tree recursion not labeled. Switch-on-enum (`enum_switch`) becomes indirect table lookup without explanation.

**14. Anomaly Indication:** Score: 4/10  
Justification: One anomaly noted (`UNRECOVERED_JUMPTABLE` in `ptr_func_simple`). Most issues silently presented as correct code. Stack canary checks (`__stack_chk_fail`) present but not explained as security artifacts.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., `param_1 != NULL` for pointer functions). No postconditions. Null checks present in code but not documented as contracts. Error returns (`0xffffffff`) unexplained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls present but relationships not documented. No call graph information. `test_data_types_l1` calls hardcoded values instead of actual functions, breaking traceability to tested functions.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output with no logical grouping. Runtime functions mixed with application code. No separation between test functions and implementation functions.

**18. Build System Compatibility:** Score: 4/10  
Justification: External dependencies undeclared (`__printf_chk`, `__aeabi_dadd`, `puts`). Would not compile standalone without significant stubbing. Architecture-specific code (`arm64` calling conventions) embedded.

**Overall Readability Score:** 3.8/10

### Critical Defects:
- **All struct/union types lost** - `Point3D`, `Rect`, `Widget` become `int*` with manual offset arithmetic
- **Variable names completely generic** - `param_1`, `iVar1`, `uVar1` provide zero semantic information
- **Loop unrolling without recovery** - `array_3d` has 125 explicit array accesses instead of nested loops
- **Floating-point operations opaque** - 400+ lines of `__muldf3`, `__addsf3` emulation instead of `d * 2.0 + 0.1`
- **Test functions broken** - Expected values hardcoded instead of actual function calls, breaking verification

### Best Quality Aspect: **Function Naming Quality (7/10)**
- Most original function names preserved (`process_char`, `array_1d_stack`, `struct_simple`), providing immediate semantic context for function purpose.

### Most Needs Improvement: **Composite Type Recovery (2/10)**
- Complete failure to recover any struct, union, or enum definitions; all composite types degraded to pointer arithmetic, making data structure intent completely opaque.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover struct definitions from access patterns** - Detect consistent offset patterns (`[0]`, `[1]`, `[2]` for 3-int struct) and propose named struct types with field names.
2. **Reconstruct loops from unrolled code** - Detect repetitive access patterns (125 array accesses with sequential indices) and recover original loop constructs with comments.
3. **Infer variable semantics from usage** - Analyze how variables are used (comparison with `'A'`/`'Z'`, multiplication by 2) to suggest meaningful names like `input_char`, `doubled_value`.
