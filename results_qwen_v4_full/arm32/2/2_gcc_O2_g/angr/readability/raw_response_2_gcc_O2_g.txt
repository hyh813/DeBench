# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Nearly all variables use generic names like `v0`, `v1`, `a0`, `a1`, `ptr`, `i` with no semantic meaning. For example, in `process_char`, `v0` and `v1` should be `char_offset` and `result` based on their usage patterns.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some original function names preserved (`process_char`, `ptr_single`, `struct_simple`), but many are generic (`sub_103a0`, `sub_10450`). Test functions like `test_data_types_l1` retained names but lost actual function call logic.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation: `65`, `32`, `255`, `0x80000000`, `4294967295`. Test functions have hardcoded expected values (e.g., `__printf_chk(1, "DT-L1-01...: %c\n", 97)`) instead of actual function calls.

**4. Noise Reduction:** Score: 3/10  
Justification: Excessive `armg_calculate_condition()` calls obscuring simple comparisons. Compiler artifacts like `vvar_26{r64|4b}`, `cc_op`, `cc_dep1` variables pollute the code throughout.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Extensive `goto LABEL_XXXX` patterns instead of natural loops and conditionals. Simple `if` statements become multi-label branching structures. For example, `process_bool` uses 4 labels for a simple boolean check.

**6. Function Decomposition:** Score: 4/10  
Justification: `binary_tree_sum` exploded from ~10 lines to 300+ lines with excessive nesting. Helper functions like `__muldf3` are 400+ lines of incomprehensible floating-point emulation instead of library calls.

**7. Code Linearization:** Score: 3/10  
Justification: Frequent forward jumps into loop bodies disrupt reading flow. Deep nesting in `binary_tree_sum` exceeds 10 levels. Multiple exit points scattered throughout functions.

**8. Redundancy Elimination:** Score: 2/10  
Justification: Massive redundancy in floating-point functions with repeated `armg_calculate_condition` patterns. Test functions print the same values twice (e.g., `test_data_types_l1` returns duplicate `__printf_chk` calls).

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most types degraded to `unsigned int` regardless of original type. `process_char` takes `unsigned int` instead of `char`. `process_short` uses `unsigned short` but returns `unsigned int`. Pointer types often become `unsigned int`.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Structs recovered but with generic field names (`field_0`, `field_4`, `field_8`). `Point3D` becomes `unsigned int *a0` with array indexing instead of `p->x + p->y + p->z`. Bitfield struct `Flags` lost all field semantics.

**11. Type Safety:** Score: 3/10  
Justification: Numerous unsafe casts throughout. `ptr_void` uses `void*` but casts to `int*` and `char*` without safety checks. Function pointers become `unsigned int` with `goto a0` instead of proper call syntax.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: Zero useful comments. No algorithm explanations, no decompilation confidence indicators, no API documentation. Only auto-generated function headers present.

**13. Idiom Recognition:** Score: 2/10  
Justification: No pattern recognition. Linked list traversal in `linked_list` not identified as such. Tree recursion in `binary_tree_sum` completely obscured by nested conditionals.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly warnings despite obvious issues. Floating-point functions completely broken without any indication. Indirect calls (`goto a0` for function pointers) not flagged as uncertain.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No precondition/postcondition documentation. Null pointer checks exist but aren't documented as contracts. Array bounds checks in `array_complex_index` not explained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Function call relationships obscured. Test functions don't actually call the functions they test (hardcoded values). Global variables named `g_12044`, `g_23008` with no semantic meaning.

**17. Module Organization:** Score: 4/10  
Justification: All functions in single file without logical grouping. No header file separation. Runtime initialization functions (`_init`, `_start`, `__do_global_dtors_aux`) mixed with application code.

**18. Build System Compatibility:** Score: 5/10  
Justification: Code references external symbols (`GLIBC_2.4::__stack_chk_guard`, `__printf_chk`) without declarations. Floating-point library functions (`__muldf3`, `__aeabi_dadd`) need proper linking. Would require significant patching to compile.

**Critical Defects:**

- Control flow uses `goto LABEL_XXXX` instead of natural `if/while/for` constructs
- Floating-point functions (`process_float`, `process_double`, `process_ld`) completely broken - don't accept parameters
- Test functions contain hardcoded expected values instead of actual function calls, losing all test logic
- Struct fields all generic (`field_0`, `field_4`) losing semantic meaning from original `x`, `y`, `z`, `width`, `height`

**Best Quality Aspect:** Function Naming Quality (scored 5/10)

**Most Needs Improvement:** Comment Quality & Presence (scored 1/10)

**Top 3 Recommendations for Decompiler Improvement:**
1. Implement pattern-based variable renaming using usage context (e.g., variables used in size comparisons should be named `*_size`, `*_len`, `*_count`)
2. Recover high-level control structures from ARM condition codes instead of emitting `armg_calculate_condition()` calls and `goto` labels
3. Add decompilation confidence annotations for uncertain constructs (function pointers, floating-point operations, indirect calls) with suggested manual review
