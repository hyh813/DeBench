# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY (30%) ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables are generically named `v1`, `v2`, `v3`, `result` throughout; original semantic names like `sum`, `i`, `current`, `head` are completely lost, making data flow tracking difficult.

**2. Function Naming Quality:** Score: 5/10  
Justification: Some function names preserved (`array_1d_stack`, `struct_simple`) but critical functions missing entirely (`process_char`, `process_int`, `process_double`, `process_float`); test function names retained but implementations broken.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers pervasive (`4`, `8`, `12`, `20`, `40`, `100`, `400`) representing struct offsets and array strides without explanation; string literals stored as hex-encoded globals (`g1`, `g2`, `g3`, `g4`).

**4. Noise Reduction:** Score: 4/10  
Justification: Excessive temporary variables (`v1` through `v16`); unnecessary casts like `(int32_t)arr`, `(int32_t)p` throughout; compiler optimization artifacts visible despite -O0 compilation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY (25%) ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Original `for` loops converted to `while` loops with confusing increment patterns (`v1 += 11` in `array_2d_stack`); recursive `binary_tree_sum` converted to iterative while loop breaking semantic clarity.

**6. Function Decomposition:** Score: 4/10  
Justification: Function boundaries exist but implementations semantically broken (`ptr_triple` returns `(int32_t)p + 1` instead of `***p + 1`); test functions print pre-computed values instead of calling actual functions.

**7. Code Linearization:** Score: 4/10  
Justification: Nested while loops in `array_3d` and `graph_traverse` obscure original 3-level for loop structure; forward jumps and complex loop conditions disrupt top-to-bottom reading.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Repetitive pointer arithmetic patterns (`4 * v1 + (int32_t)arr`) throughout; loop bodies duplicate similar operations without abstraction.

## === LEVEL 3: TYPE SYSTEM FIDELITY (20%) ===

**9. Type Precision:** Score: 2/10  
Justification: Catastrophic type degradation—all types become `int32_t`, `int32_t*`, or `uint32_t`; original `char`, `short`, `long`, `float`, `double`, `long double` completely lost; no `const`/`volatile` preservation.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Zero struct definitions recovered (`Point3D`, `Rect`, `Widget`, `Node`, `Graph` all lost); all struct access via raw pointer arithmetic (`*(int32_t *)(v1 + 4)`); unions and enums not recovered.

**11. Type Safety:** Score: 3/10  
Justification: Pervasive unsafe casts (`(int32_t)arr`, `(int32_t)p`, `(int32_t *)*(int32_t *)`); function pointer signatures degraded; pointer arithmetic without type safety throughout.

## === LEVEL 4: SEMANTIC TRANSPARENCY (15%) ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated metadata comments (`// From module`, `// Address range`, `// Line range`); zero comments explaining logic, algorithms, or non-obvious operations.

**13. Idiom Recognition:** Score: 3/10  
Justification: Common patterns (linked list traversal, tree recursion, array summation) not identified or named; no pattern hints in output.

**14. Anomaly Indication:** Score: 2/10  
Justification: Broken functions (`ptr_triple`, `ptr_func_simple`, `struct_func_ptr`) not flagged; semantic loss unmentioned; decompiler presents incorrect code confidently.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: NULL checks present but no preconditions/postconditions documented; no contract information for function parameters or return values.

## === LEVEL 5: CONTEXTUAL COHERENCE (10%) ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls visible but many semantically broken; global variables (`g1-g4`) purpose unclear; callback relationships not documented.

**17. Module Organization:** Score: 5/10  
Justification: Functions grouped in single file but original section organization lost; no header file structure; static functions not properly scoped.

**18. Build System Compatibility:** Score: 5/10  
Justification: Standard includes present but `__printf_chk` requires stubs; code may compile but won't execute correctly due to semantic errors.

**Overall Readability Score: 3.5/10**

### Critical Defects:
- **Type system completely collapsed**: All `float`, `double`, `long double`, struct types lost; everything is `int32_t*`
- **Function pointer calls eliminated**: `ptr_func_simple`, `ptr_func_complex`, `struct_func_ptr` return values instead of calling functions
- **Test functions optimized away**: Print pre-computed constants instead of calling actual functions (e.g., `test_data_types_l1` prints `97` instead of calling `process_char('A')`)
- **Recursive algorithms broken**: `binary_tree_sum` converted from recursion to while loop, losing tree traversal semantics
- **Zero struct/union definitions**: All composite type access via raw pointer arithmetic (`*(int32_t *)(v1 + 4)`)

### Best Quality Aspect:
- **Module Organization (5/10)**: Functions are at least grouped in a single coherent file with prototype declarations, providing minimal structural coherence despite semantic failures.

### Most Needs Improvement:
- **Type System Fidelity (2.33/10 average)**: Complete loss of type information makes the decompiled code fundamentally unreliable for understanding program semantics.

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover composite type definitions**: Analyze memory access patterns to reconstruct struct layouts with field names based on access offsets and usage patterns.
2. **Preserve function call semantics**: Detect and preserve indirect calls through function pointers instead of replacing them with value returns.
3. **Add anomaly flags for uncertain recovery**: Mark functions where semantic recovery failed (e.g., `// DECOMPILER_WARNING: Function pointer call not resolved`) to alert analysts to potential issues.
