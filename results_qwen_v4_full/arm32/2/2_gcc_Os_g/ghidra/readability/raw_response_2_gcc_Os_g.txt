# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 5/10  
Justification: Function parameters preserve original names (x, p, arr, n), but local variables degrade to generic names (iVar1, uVar1, iVar2). Multiple "Unresolved local var" comments indicate the decompiler couldn't recover variable semantics (e.g., `sum`, `i`, `len` all show as unresolved).

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match original source exactly (process_char, array_1d_stack, struct_simple, linked_list). Helper functions use generic names (FUN_000115a8, __mulsf3), but core test functions are well-preserved.

**3. Literal Clarity:** Score: 4/10  
Justification: Floating-point operations use unexplained magic numbers (0x3fc00000, 0x3f000000 for 1.5f and 0.5f). Test functions show hardcoded expected values (0x61, 0x62) instead of actual function calls, obscuring the computation flow.

**4. Noise Reduction:** Score: 5/10  
Justification: Multiple "WARNING: Removing unreachable block" comments throughout. "Unresolved local var" annotations appear in nearly every function. Library function calls (__mulsf3, __addsf3, __aeabi_dadd) replace simple arithmetic operations.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions have natural flow, but critical errors exist: `binary_tree_sum` incorrectly uses a for-loop instead of recursion, `enum_switch` uses a lookup table instead of switch statement, and `graph_traverse` has overly complex nested loops that don't match source structure.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately sized and cohesive. Floating-point helper functions are appropriately separated. No major inlining issues, though some test functions could be better organized.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom reasonably well. Some do-while loops could be clearer as for-loops. Forward jumps in test functions disrupt flow slightly.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Test functions contain hardcoded expected values that duplicate actual function computations. Floating-point helper functions show repetitive bit-manipulation patterns that could be abstracted.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Basic types mostly preserved (int, char, short), but `longlong` and `longdouble` used instead of proper C syntax (`long long`, `long double`). Function pointer types are generic (_func_int_int). The `volatile` qualifier is completely lost in `volatile_access`.

**10. Composite Type Recovery:** Score: 7/10  
Justification: Struct names preserved excellently (Point3D, Rect, Widget, Node, TreeNode). Field names mostly preserved in simple structs. Bitfield struct (Flags) loses field names, showing raw bit manipulation instead. Union types preserved but field access unclear in some cases.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe casts visible (void* to int*, char* conversions in ptr_cast). Pointer arithmetic without bounds checking. Const correctness lost in several functions. No restrict or alignment annotations recovered.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Comments are primarily decompiler warnings ("Unresolved local var", "WARNING: Removing unreachable block"). No algorithm explanations, no API contracts (@param, @return), no documentation of non-obvious logic.

**13. Idiom Recognition:** Score: 4/10  
Justification: Linked list traversal recognized correctly. Tree recursion NOT recognized (incorrectly converted to loop). Switch-on-enum recognized as lookup table but not named. No pattern naming or explanation provided.

**14. Anomaly Indication:** Score: 6/10  
Justification: "WARNING: Could not recover jumptable" for function pointers. "Unresolved local var" indicates uncertainty. Some anomalies flagged but coverage is inconsistent across functions.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: ptr != NULL"). No postconditions. No invariants. No error condition documentation. Complete absence of contract information.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls visible but not documented. No call graph information. Global variable usage not clearly documented. Callback relationships not explicitly marked.

**17. Module Organization:** Score: 5/10  
Justification: All functions in single output file. No header file separation. Static functions not clearly marked. No namespace or package structure recovered.

**18. Build System Compatibility:** Score: 4/10  
Justification: Contains library function calls (__mulsf3, __addsf3, etc.) that need linking. Some type definitions may need adjustment (longlong → long long). Would require significant work to compile standalone.

**Overall Readability Score: 5.2/10**

### Critical Defects:
- **`binary_tree_sum` and `binary_tree`**: Recursion incorrectly converted to for-loop, fundamentally changing algorithm semantics
- **`volatile_access`**: Lost volatile qualifier, showing `*p + *p` instead of two separate volatile reads
- **Variable names**: Local variables all generic (iVar1, uVar1) with "Unresolved local var" comments throughout
- **Magic numbers**: Floating-point constants shown as hex (0x3fc00000) without explanation
- **Test functions**: Hardcoded expected values instead of actual function calls, obscuring computation flow

### Best Quality Aspect:
- **Function Naming Quality (8/10)** - Most function names match original source exactly, making it easy to identify purpose

### Most Needs Improvement:
- **Assertion & Contract Recovery (2/10)** - Complete absence of preconditions, postconditions, or any contract documentation

### Top 3 Recommendations for Decompiler Improvement:
1. **Recover recursion patterns**: Detect backward calls to same function and preserve recursive structure instead of converting to loops
2. **Preserve type qualifiers**: Maintain volatile, const, and restrict qualifiers from debug information when available
3. **Infer variable names from usage**: Analyze variable usage patterns to suggest semantic names (e.g., variables used in loop bounds → `i`, `count`; accumulators → `sum`, `total`)
