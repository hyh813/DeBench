- **Return type mismatch**: Function declared as `void` but should return `int` (the `strlen` result is computed but discarded)

- **No anomaly flagging**: Decompiler does not indicate the semantic error of computing `strlen` but not returning it
- **Generic parameter names**: `param_1` and `param_2` should be `dst` and `src` based on `strcpy` usage pattern
- **No contract documentation**: Missing preconditions (buffer size) and postconditions (return value meaning)

### Top 3 Recommendations for Decompiler Improvement

1. **Recover return values from functions**: When a function computes a value (like `strlen`) and the original source returns it, the decompiler should preserve this. The return type should be `int`, not `void`.

2. **Add const-correctness inference**: Analyze library function calls (like `strcpy(dst, src)`) to infer that the second parameter should be `const char*` to prevent modification of source data.

3. **Flag semantic anomalies**: When a computed value is not used (like `strlen(param_1)` result), the decompiler should add a comment like "WARNING: Return value not used" or "ANOMALY: strlen result discarded" to alert analysts to potential issues.
