# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables are named with generic patterns like `iVar1`, `uVar1`, `plVar1`, `local_10`, `auStack_18`. The original source used meaningful names like `arr`, `n`, `i`, `p1`, `p2` which are completely lost. This makes tracking variable purpose nearly impossible without extensive cross-referencing.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names closely match the original source: `double_value`, `local_vars`, `local_array`, `heap_basic`, `heap_calloc`, `global_var_access`, `memop_memset`, etc. The decompiler correctly preserved the semantic naming from the source.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers appear without explanation: `0x2a` (42), `0x14` (20), `0x800` (2048), `0x28` (40), `0xf` (15). The original source had clear constants like `42`, `10`, `2048` that were meaningful in context. No named constants are recovered.

**4. Noise Reduction:** Score: 4/10  
Justification: Significant compiler artifacts present: `__stack_chk_fail()` calls, `in_FS_OFFSET + 0x28` stack canary checks, `/* Unresolved local var: int i@[???] */` comments, `/* WARNING: Unknown calling convention */` warnings. These clutter the code significantly.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Control flow structures are mostly natural - loops use `do-while` patterns correctly, conditionals are clear. At `-O0` optimization, the flow should be nearly identical to source. Minor awkwardness in some complex functions like `vla_stack` due to stack frame manipulation.

**6. Function Decomposition:** Score: 9/10  
Justification: Each function from the original source is preserved as a separate function. No inlining or outlining artifacts observed. Function granularity matches the source exactly - small, focused functions like `double_value`, `local_vars`, etc.

**7. Code Linearization:** Score: 8/10  
Justification: Code generally reads top-to-bottom. Early returns are used appropriately. No excessive goto statements or confusing jumps. The flow is easy to follow in most functions.

**8. Redundancy Elimination:** Score: 7/10  
Justification: Some loop unrolling present in `local_array` and `heap_basic` but it's minimal. No significant dead code. The decompiler handled common subexpression elimination reasonably well.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Uses generic `int` throughout instead of specific types. Parameters like `size_t` are lost, becoming `int` or `ulong`. The original had `size_t` parameters in memory operations, now generic. Signed/unsigned distinction is often lost.

**10. Composite Type Recovery:** Score: 9/10  
Justification: Struct types are well recovered: `Node`, `TreeNode`, `Point` all have correct field names (`data`, `next`, `left`, `right`, `x`, `y`). The struct layouts match the original source exactly.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe patterns: `(int *)((long)__ptr + uVar1 * 4)` pointer arithmetic, `(void *)0x0` NULL casts, `(int **)(__ptr + 2)` accessing struct fields via raw offsets. The original used proper field access (`node->data`), but decompiler uses raw pointer arithmetic.

No meaningful comments explaining algorithm intent or recovered logic. The original had no comments in test functions either, but some explanation would help.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies are flagged: "Unknown calling convention", "Unresolved local var", "Subroutine does not return". However, these are generic warnings without specific explanations of what couldn't be resolved or why.

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants are documented. The original source had implicit contracts (e.g., "requires: n > 0" for heap functions) that are completely lost. Error return values are present but not explained.

### Critical Defects (Bullet List)
- Variable naming is completely generic (`iVar1`, `uVar1`, `local_10`) - original names like `arr`, `n`, `i`, `p1` are lost
- Type precision severely degraded - all `size_t` parameters become `int` or `ulong`, losing signed/unsigned distinction
- Unsafe pointer arithmetic replaces struct field access: `*(int **)(__ptr + 2)` instead of `node->next`
- Magic numbers everywhere without explanation: `0x2a`, `0x14`, `0x800` instead of named constants
- Heavy compiler artifact noise: `__stack_chk_fail`, stack canary checks, unresolved var warnings

### Best Quality Aspect: Function Decomposition (Score: 9/10)
- Each function from the original source is preserved as a separate entity with correct boundaries. No inlining or outlining artifacts. Function names match exactly.

### Most Needs Improvement: Variable Naming Quality (Score: 2/10)
- Generic variable names like `iVar1`, `uVar1`, `plVar1`, `local_10`, `auStack_18` make it nearly impossible to track variable purpose without extensive cross-referencing. The original source had clear, semantic names that are completely lost.

### Top 3 Recommendations for Decompiler Improvement
1. **Infer variable names from usage patterns**: When a variable is used as array index, name it `index` or `i`. When it's used to store size, name it `size` or `n`. Use data flow analysis to recover semantic names.
2. **Recover struct field access**: Instead of `(int **)(__ptr + 2)`, emit `node->next`. The decompiler knows the struct type (`Node`) but uses raw pointer arithmetic.
3. **Replace magic numbers with named constants**: Detect repeated literal values and suggest meaningful names based on context (e.g., `0x2a` → `42` → could be `DEFAULT_VALUE` based on usage).
