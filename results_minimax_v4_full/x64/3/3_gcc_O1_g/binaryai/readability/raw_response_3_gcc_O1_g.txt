# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 3/10  
Justification: Variables use generic names like `param_1`, `local_10`, `iVar1`, `uVar1`. The source uses meaningful names like `x`, `arr`, `n`, `p1`, `p2`, `head`, `temp` but these are lost in decompilation. Even stack buffer `local_818` is just a memory address with no semantic meaning.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match source exactly: `double_value`, `local_vars`, `heap_basic`, `malloc`, `free`. Minor issues with internal functions like `FUN_00101020` but overall good recovery.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x2a` (42), `0x14` (20), `0x800` (2048) appear without explanation. String literals use cryptic references like `&DAT_00103004`. Some constants like `10` are preserved but semantic meaning is often lost.

**4. Noise Reduction:** Score: 5/10  
Justification: Contains stack canary checks (`__stack_chk_fail`), FS register offset references (`in_FS_OFFSET`), and type-prefixed variables (`iVar1`, `uVar1`, `lVar1`). These compiler artifacts add noise but are manageable.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 7/10  
Justification: Loops and conditionals are correctly recovered. The `for` loops in source appear as `do-while` constructs in decompiled code, which is typical for -O0. Control flow is readable despite minor restructuring.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain original boundaries and granularity. Each test function (`test_stack_memory`, `test_heap_memory`) is properly separated. No over-fragmentation or monoliths observed.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns in error handling are present. Some forward jumps exist in complex functions like `vla_stack` but flow is traceable.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code observed. Loop unrolling is minimal at -O0. Code is clean without duplicate expressions.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Mixed types - `int`, `undefined4`, `uint`, `long` all used. Function parameters use `undefined4` for int, `undefined8` for pointers. Some signed/unsigned confusion (e.g., `param_1` as `uint` when source uses `int`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Struct `Point` is completely lost - `heap_struct` uses raw `malloc(8)` with manual field access. Linked list nodes are `int*` with manual offset calculations (`*(int **)(__ptr + 2)`). No struct field names recovered.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe casts present. Pointer arithmetic is manual without safety. Function pointers use `undefined8` type. No `restrict` or alignment annotations. Buffer operations lack bounds checking.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No comments explaining algorithm logic. Only auto-generated function signatures. No documentation of preconditions, postconditions, or non-obvious behavior. Source comments like "// MEM-L1-01" are completely lost.

**13. Idiom Recognition:** Score: 5/10  
Justification: Some patterns recognizable (linked list traversal, tree traversal) but not explicitly named. Memory operations (malloc/free pairs) are identifiable but not labeled as patterns.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about suspicious code. The `dangling_pointer` function accesses freed memory but no anomaly flagged. `double_free` has no warning. `memory_leak` not flagged.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: n > 0"). No postconditions (e.g., "ensures: return is heap-allocated"). Error handling is present but contracts are implicit.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 6/10  
Justification: Function calls are clear. External functions marked with `<EXTERNAL>`. But no documentation of call relationships, callback setups, or global variable usage patterns.

**17. Module Organization:** Score: 6/10  
Justification: All functions in single file. Related functions grouped logically (stack, heap, static, memop sections). No header file structure recovered.

**18. Build System Compatibility:** Score: 5/10  
Justification: Code has compilation issues - undefined types like `undefined4`, `EVP_PKEY_CTX`. External dependencies not documented. Requires significant patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 5.1/10

### Critical Defects:
- **Variable naming**: `param_1`, `local_10`, `iVar1` are meaningless; should infer semantic names from usage (e.g., `param_1` is `n` in `local_array(int n)`)
- **Struct recovery failure**: `Point` struct completely lost; `heap_struct` uses raw `malloc(8)` with manual field offsets instead of `Point* p`
- **No comments**: All source code comments lost; no explanation of test scenarios or expected values
- **Type precision**: Uses `undefined4`/`undefined8` instead of proper `int`/`void*`; loses const-correctness
- **Anomaly silent**: `dangling_pointer`, `double_free`, `memory_leak` functions have no warnings about undefined behavior

### Best Quality Aspect: Function Naming (8/10)
Function names like `double_value`, `heap_basic`, `malloc`, `free` accurately reflect source intent. Most test functions are correctly identified.

### Most Needs Improvement: Semantic Transparency (2.75/10)
No comments, no anomaly detection, no contract recovery. The decompiled code provides zero context about what it's doing or what could go wrong.

### Top 3 Recommendations:
1. **Infer variable names from usage patterns**: `param_1` in `local_array(param_1)` should be named `n` based on loop bounds `i < 10`
2. **Recover struct types**: Detect `malloc(8)` + two field assignments as `Point*` struct allocation; use field names `x`, `y`
3. **Add anomaly warnings**: Flag `dangling_pointer` as "use-after-free", `double_free` as "double-free vulnerability", `memory_leak` as "potential memory leak"
