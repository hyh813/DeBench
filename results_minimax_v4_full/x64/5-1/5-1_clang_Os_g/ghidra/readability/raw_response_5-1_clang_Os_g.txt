# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 1/10  
Justification: Extensive "Unresolved local var" comments throughout (e.g., `SimpleClass obj@[???]`, `int r1@[???]`, `int r2@[???]`). Variables that are named use generic identifiers like `local_24`, `puVar1`, `lVar2`, `iVar4` - completely obscuring semantic purpose. This is essentially unusable for analysis.

**2. Function Naming Quality:** Score: 4/10  
Justification: Some functions correctly named (test_cpp_member_func, test_cpp_constructor, call_virtual_func, template_max<int>), but critical methods like SimpleClass::getValue, SimpleClass::setValue are completely missing. Many functions just return constants without implementing actual logic.

**3. Literal Clarity:** Score: 2/10  
Justification: Functions return magic numbers directly (0x2a=42, 0x47=71, 0x28a=650, 0x16=22) without explaining computation. The decompiled code shows `return (int)sVar1 + 0x125c` but the actual formula (value * x + strlen(name)) is lost.

**4. Noise Reduction:** Score: 5/10  
Justification: Moderate noise from "WARNING" comments about unresolved variables and unknown calling conventions. However, the more critical issue is that most functions don't contain real logic - they're just constant returns, making the noise assessment somewhat moot.

**10. Composite Type Recovery:** Score: 1/10  
Justification: No struct/union recovery is evident. The decompiled code doesn't show struct definitions for SimpleClass, LifecycleClass, Base, Derived, etc. Everything is handled as raw memory with pointer arithmetic (e.g., puVar1[2] accessing what should be struct members).

**11. Type Safety:** Score: 4/10

### Critical Defects (Bullet List)

- **Missing member function implementations**: SimpleClass::getValue(), setValue(), compute() are never decompiled - functions just return constants instead of actual computed values
- **Unusable variable names**: Extensive "Unresolved local var" comments with `[???]` placeholders make code analysis impossible
- **No struct/class recovery**: Composite types like SimpleClass, LifecycleClass, Base, Derived are not reconstructed - all accessed as raw memory
- **Magic numbers without explanation**: Functions return hex values (0x2a, 0x47, 0x28a) without documenting the computation that produces them
- **Incomplete virtual function handling**: call_virtual_func shows vtable access but the actual polymorphic behavior isn't properly recovered

### Best Quality Aspect: Anomaly Indication (7/10)
- Ghidra properly marks unresolved variables with `[???]` and provides warnings about unknown calling conventions and indirect jumps. The DWARF original prototypes are preserved which helps with type information.

### Most Needs Improvement: Composite Type Recovery (1/10)
- The decompiler failed to recover any C++ class structures. All class members are accessed via raw pointer arithmetic losing all semantic meaning. This is critical for C++ code analysis.

### Top 3 Recommendations for Decompiler Improvement

1. **Recover class/struct definitions**: Parse vtable symbols and constructor/destructor to reconstruct class layouts with proper field names instead of treating everything as raw memory

2. **Infer variable names from usage patterns**: When local variables can't be resolved from debug info, analyze how they're used (arithmetic, indexing, function calls) to suggest meaningful names

3. **Flag incomplete decompilation**: When a function just returns a constant that matches expected output, flag this as a potential decompilation failure rather than presenting it as the final result
