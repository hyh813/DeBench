# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables are almost exclusively generic (`v0`, `v1`, `v2`, `ptr`, `result`, `v4`, `v5`, `v6`, `v7`, `v8`, `v9`, `v10`). No semantic meaning is recovered - e.g., `v3 = 50` should indicate `shared_data = 50`. This is severely degraded compared to source where variables like `capture_by_value`, `capture_by_ref` were clearly named.

**2. Function Naming Quality:** Score: 5/10  
Justification: Mixed results - C++ functions like `test_cpp_exception`, `test_cpp_member_func`, `template_max<int>`, `Container<int>::push` are correctly demangled, but many utility functions remain as meaningless `sub_402020`, `sub_40202d` patterns. The original source had clearly named functions like `getValue`, `call_virtual_func`.

**3. Literal Clarity:** Score: 4/10  
Justification: Multiple unexplained magic numbers throughout - e.g., `31` in `strncpy(&v0, "Test", 31)` should be `sizeof(name)-1`, `8589934593` for array initialization is opaque. String literals are accessed via global pointers (`&g_403016`) without recovery of actual content.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise pollution - dozens of small stub functions (sub_402020 through sub_40215f) that are just register initialization wrappers. The decompiled output contains 1582 LOC vs 540 source LOC, indicating massive artifact injection.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: Exception handling code uses goto jumps creating spaghetti control flow - the `test_cpp_exception()` function has multiple conditional jumps to `_Unwind_Resume` that obscure the try-catch-throw semantics. Loop constructs are not properly recovered.

**6. Function Decomposition:** Score: 6/10  
Justification: C++ member functions are correctly identified and separated (e.g., `Base::virtual_func`, `Derived::virtual_func`, `template_max<int>`). However, some functions like `call_virtual_func` have incorrect implementations - it uses `goto *((long long *)*((long long *)&arg_0))` which is a broken vtable dispatch.

**7. Code Linearization:** Score: 4/10  
Justification: The code frequently disrupts linear reading with goto statements and computed jumps. The `test_cpp_exception` function is particularly non-linear with multiple exception handling paths interleaved. Forward jumps interrupt natural flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Some template instantiations appear multiple times (e.g., `template_max<int>` and `template_max<double>`), but this is expected from compilation. Dead code is minimal, but the overall redundancy is moderate due to the noise from stub functions.

---

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Mixed precision - basic types like `int`, `double`, `unsigned int` are used correctly, but many pointers are generic `void*` or `unsigned long long`. The `arg_0`, `arg_1` parameters lack proper type information. Const correctness is not preserved.

**10. Composite Type Recovery:** Score: 4/10  
Justification: Partial struct recovery with generic names - `struct_0`, `struct_2`, `struct_3` are defined but fields are generic (`field_0`, `field_8`, `field_10`). The original `SimpleClass` with `int value; char name[32];` is not recovered as a named struct.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe operations - extensive pointer arithmetic (e.g., `(char *)ptr + 4 * index`, `(char *)a0 + a0->field_0[32]`), raw memory manipulation in `test_cpp_smart_ptr` constructing objects via `operator new`. Many casts between pointer types without validation.

---

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Almost no useful comments. The only comments are auto-generated headers like "Angr Decompilation" and "Platform: AMD64". No explanation of algorithm logic, no recovery of original comments from debug symbols.

**13. Idiom Recognition:** Score: 4/10  
Justification: Some C++ idioms partially recognized - virtual function thunks (`_ZThn16_N12MultiDerived5funcBEv`) are identified, template instantiation is correct. However, RAII patterns (unique_ptr lifecycle) are not recognized as such, and exception handling semantics are lost.

**14. Anomaly Indication:** Score: 3/10  
Justification: Minimal anomaly indication. The broken `call_virtual_func` implementation using computed goto is presented as normal code. No warnings about indirect calls that couldn't be resolved or potential undefined behavior.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No contract recovery. Preconditions, postconditions, and invariants are not documented. The original source had clear semantics (e.g., "requires: ptr != NULL") that are completely lost.

---

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Some function call relationships are clear (e.g., `test_cpp_oo_features` calls other test functions), but global variable usage is opaque (`g_4058a8`, `g_4058d8`). Virtual dispatch relationships are not clearly documented.

**17. Module Organization:** Score: 5/10  
Justification: Functions are somewhat grouped by type (test functions together, template functions together), but the original class-based organization is lost. All classes are flattened into standalone functions rather than grouped by class.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code cannot compile standalone - many external symbols are undefined (`operator new`, `operator delete`, `__cxa_throw`, `__cxa_begin_catch`). Requires significant patching to produce compilable code.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.7/10

### Critical Defects:
- **Variable naming**: All variables use generic names (`v0`, `v1`, `ptr`, `result`) with no semantic recovery
- **Broken virtual dispatch**: `call_virtual_func` uses `goto *((long long *)*...)` which is incorrect and misleading
- **No struct recovery**: Original classes like `SimpleClass`, `LifecycleClass` are not reconstructed as named structs
- **Exception handling**: Completely obfuscated - try-catch-throw semantics lost, replaced with raw `__cxa_*` calls with goto
- **No comments**: Zero useful comments explaining algorithm intent or recovered semantics

### Best Quality Aspect: Function Decomposition (6/10)
- C++ member functions are correctly separated and some are properly demangled (e.g., `template_max<int>`, `Container<int>::push`)

### Most Needs Improvement: Variable Naming (2/10)
- At -O0 optimization, variables should have near-perfect naming recovery. The complete failure to infer semantic names like `capture_by_value`, `shared_data`, `size` from usage patterns is the primary readability blocker.

### Top 3 Recommendations:
1. **Infer variable names from usage**: Analyze how variables are used (arithmetic, indexing, comparison) to suggest semantic names like `index`, `size`, `multiplier`
2. **Recover class structures**: Group related functions and reconstruct struct layouts with meaningful field names based on member access patterns
3. **Add semantic comments**: Document recovered high-level constructs (virtual dispatch, RAII, exception handling) rather than showing raw ABI calls
