# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names (`a1`, `a2`, `a3`) instead of source names (`a`, `b`, `c`, `x`). Local variables are `v2`, `v3`, `i`, `j` - completely non-descriptive. Compare source `int temp1 = a + b;` vs decompiled which computes `(a2 + a1)` without naming the intermediate result.

**2. Function Naming Quality:** Score: 9/10  
Justification: Nearly all function names match source exactly: `sequential_ops`, `single_if`, `if_else`, `nested_if_2`, `loop_for_fixed`, `loop_while`, `recursion_factorial`. Excellent recovery of function-level semantics.

**3. Literal Clarity:** Score: 4/10  
Justification: Multiple unexplained magic numbers: `switch_small` returns `15` for case 0 (should be `a + b = 15`), `loop_break` uses cryptic hex `0x140000000ALL` for array data. No constants or enum recovery. The literal `1000` in `infinite_loop` is unexplained.

**4. Noise Reduction:** Score: 7/10  
Justification: At -O0, minimal compiler artifacts expected. Some unusual syntax in `loop_break` with malformed hex literals (`0x140000000ALL`), but overall clean without excessive casts or debug instrumentation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control flow structures well-preserved: `for` loops, `while`, `do-while`, `switch` statements all correctly identified. `goto_forward` reasonably converted to if-else, `goto_backward` to for loop. Some switch statements use if-else chains but this is acceptable.

**6. Function Decomposition:** Score: 8/10  
Justification: Each source function decompiled as separate function with appropriate granularity. No over-fragmentation or monoliths. Parameter counts match source (e.g., `sequential_ops(int a1, int a2, int a3)` has 3 params).

**7. Code Linearization:** Score: 8/10  
Justification: Code reads top-to-bottom with minimal disruption. Early returns used appropriately. No excessive forward jumps. `duffs_device` has complex control flow but that's inherent to the pattern.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code observed. No significant duplicate expressions. At -O0, loop unrolling shouldn't appear anyway. Clean code in this regard.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Major type issues: All functions return `__int64` instead of `int` (e.g., `sequential_ops` returns `__int64` but source returns `int`). `ternary_op` uses `unsigned int` parameters but source uses `int`. Function pointer types are imprecise (e.g., `__int64 (__fastcall *)(_QWORD)` instead of proper signatures).

**10. Composite Type Recovery:** Score: 7/10  
Justification: Source uses minimal composite types (mostly arrays like `int arr[]`). The `loop_break` array is partially recovered as `_QWORD v3[2]` with hex data. No struct recovery needed as source has none.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe operations: function pointer casting in `call_func_ptr_array` uses complex indirect calls with multiple casts. The `duffs_device` has pointer arithmetic that could be safer. No buffer overflow protection. Minor issues overall.

**13. Idiom Recognition:** Score: 5/10  
Justification: `duffs_device` partially recognized as switch-do-while pattern but not named or explained. `fsm_func_table` shows state machine pattern but not labeled. Some patterns recognizable but not explicitly identified.

**14. Anomaly Indication:** Score: 3/10  
Justification: No anomaly flags. The malformed hex literals in `loop_break` (`0x140000000ALL`) should be flagged as suspicious data. No indication of indirect call resolution issues or unusual stack layouts.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No precondition/postcondition recovery. No error condition documentation. Functions like `duffs_device` with `if (n <= 0) return -1;` don't have "requires: n > 0" style documentation.
