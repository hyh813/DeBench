# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

### 1. Variable Naming Quality
**Score: 3/10**

Justification: Variables use generic names like `a0`, `a1`, `a2` for parameters and `v0`, `v1`, `v2` for locals. While some like `idx`, `cur`, `ptr`, `len` are semi-descriptive, the vast majority lack semantic meaning. For example, in `varargs_func`, variables `v10`, `v11`, `v12` obscure their roles as loop counter and accumulator.

### 2. Function Naming Quality
**Score: 8/10**

Justification: Most function names match the original source exactly: `cdecl_func`, `call_cdecl`, `stdcall_func`, `fastcall_func`, `varargs_func`, `func_no_args`, `param_by_value_int`, `ret_pointer`, etc. Some initialization functions have generic names like `sub_401020`, but these are startup code not in the original test suite.

### 3. Literal Clarity
**Score: 5/10**

Justification: Many magic numbers appear without explanation (e.g., `15`, `50`, `6` in test functions). The mask `& 4294967295` (0xFFFFFFFF) appears frequently but isn't explained. However, string format pointers like `&g_402008` are used consistently for printf format strings.

### 4. Noise Reduction
**Score: 3/10**

Justification: Significant noise includes: excessive casts like `& 4294967295` and `(char)v0`; redundant variable declarations (`char v0[8]`, `char v1[4]`); incomplete function `param_struct_byval` that returns nothing; unnecessary `void* v0[1]` in `call_ptr_array`. The decompiled output is 1044 LOC vs 665 LOC source - 57% bloat.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

### 5. Control Flow Naturalness
**Score: 7/10**

Justification: Control flow is generally natural - loops use `do-while`, conditionals use `if-else`, switch statements converted to if-else chains. However, `varargs_func` uses awkward pointer arithmetic in its loop (`ptr = &(&v3)[v12 & 4294967295]`) that obscures the simple iteration pattern.

### 6. Function Decomposition
**Score: 8/10**

Justification: Function decomposition closely matches the original source. Each test function (`call_cdecl`, `call_stdcall`, etc.) and helper function (`cdecl_func`, `varargs_func`, etc.) is properly separated. The large `test_calling_conventions` function reflects the original structure.

### 7. Code Linearization
**Score: 5/10**

Justification: Code generally reads top-to-bottom, but `call_varargs_param` has an unusual signature with extra unused parameters (`unsigned long a0, unsigned long a1, unsigned long a2, unsigned long a3, unsigned long a4, unsigned long long a5`) that don't appear in the source, disrupting linear understanding.

### 8. Redundancy Elimination
**Score: 4/10**

Justification: Significant redundancy exists - `varargs_func` and `param_varargs` are nearly identical implementations. In `test_calling_conventions`, there's repetitive `memcpy` pattern for struct initialization that could be abstracted. The loop in `test_calling_conventions` initializing array elements is manually unrolled in the source but appears as explicit memcpys in decompiled.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

### 9. Type Precision
**Score: 5/10**

Justification: Types are imprecise - original `int` becomes `unsigned int` or `unsigned long long` throughout. The parameter `double d` in `func_mixed_args` becomes `unsigned long a2` with `int v2; // xmm0` for the float register. Signed/unsigned confusion is widespread.

### 10. Composite Type Recovery
**Score: 4/10**

Justification: Some struct recovery exists (`struct_0` for `param_ptr_array`), but fields are generic (`field_0`). Original types like `SmallStruct` with fields `x`, `y` and `LargeStruct` with `data[16]` are lost - the decompiled code uses raw pointer arithmetic instead (e.g., `a0[1] * *(a0)` for struct access).

### 11. Type Safety
**Score: 3/10**

Justification: Multiple type safety issues: `param_func_ptr` incorrectly treats function pointer as `unsigned long long *a0` then calls it as `a0(a1)`; `ret_pointer` returns `a0 + 4` assuming 4-byte ints; `param_ptr_array` uses `struct_0 **` incorrectly for `char *arr[]`. The mask `& 4294967295` appears frequently suggesting type confusion.

1. **Variable naming**: Parameters `a0`, `a1`, `a2` and locals `v0`, `v1`, `v2` are completely generic with no semantic meaning
2. **Type precision**: Original `int` becomes `unsigned int` throughout; `double` parameter becomes `unsigned long` with separate float register variable
3. **No comments**: Zero explanatory comments despite complex calling convention tests

- **Function Naming Quality (8/10)**
- **Why?** Most function names match the original source exactly (`cdecl_func`, `call_cdecl`, `varargs_func`, `ret_pointer`), making it easy to identify the purpose of each function.

### Most Needs Improvement (Single Dimension)
- **Comment Quality & Presence (2/10)**
- **Why?** The decompiled code has almost no comments despite the source having extensive documentation explaining each test case, expected values, and calling convention semantics. This makes it difficult to understand the purpose of complex patterns like varargs handling.

---

### Top 3 Recommendations for Decompiler Improvement

1. **Infer variable names from usage patterns**: When `a0` is used in arithmetic with `a1` and the function is `cdecl_func`, infer `a0` ŌåÆ `param_a`, `a1` ŌåÆ `param_b`. When `v0` accumulates sum in a loop, name it `sum` or `accumulator`.

2. **Recover stdarg patterns**: The varargs handling should be recognized and labeled as such, with `va_list`, `va_arg` patterns reconstructed rather than showing raw pointer arithmetic.

3. **Preserve type signedness**: The original uses `int` but decompiled shows `unsigned int` throughout. The decompiler should preserve signedness when the original type can be inferred from context (e.g., return values used in arithmetic).
