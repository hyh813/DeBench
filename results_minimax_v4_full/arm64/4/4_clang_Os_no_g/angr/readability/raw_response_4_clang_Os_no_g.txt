# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All local variables use cryptic sequential names (`v0` through `v35`), and parameters use mechanical register names (`a0`, `a1`, `a2`). No semantic inference from usage patterns (e.g., loop counter `i` is the only exception). Compare to source: original uses meaningful names like `count`, `args`, `sum` in varargs functions.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions correctly named (`cdecl_func`, `call_cdecl`, `test_calling_conventions`), but many internal functions use mechanical addresses (`sub_400694`, `sub_400770`). The main public API functions match source, while helper functions lack semantic naming.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `4294967295` (0xFFFFFFFF) appear without explanation; `18446743528248704968` is unexplained. However, simple constants like `15`, `50`, `6` in return statements are correct values from source. String literals use external references (`&g_40157a`) which obscures content.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise pollution: unused variables `v23`-`v30` assigned from registers but never used meaningfully; redundant `memcpy` calls in `sub_400694`; excessive register spill variables in varargs functions. Compare to source's clean 8-line varargs implementation vs. 50+ line decompiled version.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Basic control flow preserved (if/else, while loops). However, `register_tm_clones` uses convoluted boolean expression instead of simple conditional. Loop structures in `varargs_func` are recognizable but overly complex compared to source's clean `for` loop.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions appropriately separated matching source granularity. No 500-line monoliths. Test wrapper functions (`call_cdecl`, `call_stdcall`) correctly kept separate from implementation functions. Minor issue: some small functions could be inlined.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom, but `varargs_func` has complex nested conditions disrupting flow. Forward jumps present in loop handling. Some `do-while` constructs could be clearer as `for` loops.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy in varargs functions: unused vector register variables (`v0`-`v7`) clutter code; loop unrolling not recognized or commented; duplicate register saves. Source's clean 8-line function becomes 50+ lines of decompiled noise.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Parameters correctly use `unsigned int` for ARM64 calling convention. However, function pointers use `unsigned long long *` instead of proper function pointer types (see `param_func_ptr`). Return types inconsistent: some functions return `unsigned long long` when `int` would be correct. `void*` often becomes `unsigned long`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: No meaningful struct recovery. Source's `SmallStruct` with fields `x`, `y` becomes raw pointer arithmetic in `func_struct_byptr`. Fabricated structs like `struct_0 { char field_0; }` in `param_ptr_array` lack semantic meaning. Large struct handling shows as unsupported operation.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe operations present: `param_func_ptr` treats `unsigned long long *` as function pointer; `ret_opaque_handle` returns `"d"` (string literal) as void* handle; pointer arithmetic not type-safe. However, basic pointer dereferences are structurally correct.
