# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names like `v0`, `v1`, `v2`, `a0`, `a1`, `a2`, `ptr`, `dst`. The source uses meaningful names like `MAX_SIZE`, `BUFFER_COUNT`, `x`, `y`, `result`, but these are completely lost. Even stack variables like `char v1; char *v0;` provide no semantic information.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions like `param_macro_constants`, `call_macro_constants`, `test_preprocessing_features` correctly match source names. However, many initialization functions use generic names like `sub_400824`, `sub_400a88`, `frame_dummy`. The function `sub_400824` is clearly `memcpy` but not named as such.

**3. Literal Clarity:** Score: 5/10  
Justification: Some constants are correctly computed: `0x400` (1024), `64` (16*4), `0x200` (512). However, `57072` (0xDEF0) and `57122` (0xDF22) are unexplained magic numbers. The decompiler correctly computed the values but didn't explain their origin from the source macros.

**4. Noise Reduction:** Score: 9/10  
Justification: Clean decompilation with minimal compiler artifacts. No debug symbols, redundant casts, or frame pointer manipulation clutter. The code reads naturally without unnecessary scaffolding from the decompilation process.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control flow is natural with `if` statements, `while` loops, and `return` statements matching source logic. The `param_asm_clobber` function correctly shows a while loop. At `-O0`, the control flow should closely match source, which it does.

**6. Function Decomposition:** Score: 9/10  
Justification: Function boundaries match source exactly. Each test function (`param_macro_constants`, `param_asm_basic`, etc.) is correctly separated. No inlining or outlining artifacts from optimization since this is `-O0` compilation.

**7. Code Linearization:** Score: 9/10  
Justification: Code reads top-to-bottom without goto statements or confusing jumps. Early returns are used appropriately. The flow is straightforward and matches the source's linear structure.

**8. Redundancy Elimination:** Score: 9/10  
Justification: No dead code, redundant expressions, or unnecessary duplications. The decompiled code is clean and focused on the actual logic.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Parameters use generic types like `unsigned int`, `unsigned long`, `unsigned long long` instead of specific types like `int`, `size_t`. The return types are mostly `int` or `long long`. Some type information is preserved but much is generic. For example, `param_asm_clobber` takes `unsigned long ptr` instead of `int* arr`.

**10. Composite Type Recovery:** Score: 1/10  
Justification: No struct or union types are recovered. The source has no complex structs in the test functions, but the decompiler doesn't attempt to reconstruct any composite types. Arrays are accessed via pointer arithmetic: `*((int *)(ptr + v2 * 4))`.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe operations present: pointer arithmetic without bounds (`ptr + v2 * 4`), casts between integers and pointers (`unsigned long ptr`), and raw memory access. However, these are typical for decompiled low-level code and some are unavoidable.

**13. Idiom Recognition:** Score: 1/10

**15. Assertion & Contract Recovery:** Score: 1/10

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Function calls are present but relationships aren't documented. External functions like `__printf_chk`, `mmap`, `mprotect` are declared as `extern`. However, there's no indication of which functions call which, or the purpose of callbacks.

**17. Module Organization:** Score: 2/10  
Justification: All functions appear in a single flat listing without logical grouping. The source has clear sections (preprocessing features, asm features), but the decompiled output is an arbitrary list sorted by address.

**18. Build System Compatibility:** Score: 5/10  
Justification: Code uses `extern` declarations for external functions and appears structurally compilable. However, some functions like `param_asm_simd` have incorrect signatures (takes 3 params but source takes 3 pointers). The `param_memory_protection` function has a broken `mprotect()` call without arguments.
