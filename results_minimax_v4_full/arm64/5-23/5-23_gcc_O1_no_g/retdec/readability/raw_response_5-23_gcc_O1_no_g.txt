# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All variables use generic names (v1, v2, v3, a1, a2, i, result) with no semantic meaning. In `param_macro_constants`, variable `v1` should be named `size` based on the comparison `v1 > 1024`. The decompiler makes no effort to infer purpose from usage patterns.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names match the original source exactly (param_macro_constants, call_asm_basic, param_asm_clobber). Minor issues: memcpy is named `function_830` and printf is `function_860`, but core business logic functions are well-named.

**3. Literal Clarity:** Score: 6/10  
Justification: Call wrapper functions correctly return hardcoded expected values (64, 30, 32). However, implementation functions contain unexplained magic numbers like `0xdef0`, `0xdf22`, and `512` without constants. The value `&g27` used for comparison is opaque.

**4. Noise Reduction:** Score: 8/10  
Justification: Minimal compiler artifacts; no debug symbols, register spills, or frame pointer manipulations visible. The code is relatively clean for a decompiled output.

**6. Function Decomposition:** Score: 7/10

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns are used appropriately in error handling (e.g., param_memory_protection). No excessive goto or backward jumps.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No obvious dead code. Some loop unrolling in param_asm_simd but it's recognized as a loop structure. Common subexpressions are reasonably handled.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 6/10  
Justification: Uses int64_t and int32_t consistently, but loses precision of original types. The original used `int` and `size_t`, but decompiler uses 64-bit types. Function parameters like `int *arr` become `int64_t v1` losing pointer semantics.

**10. Composite Type Recovery:** Score: 4/10  
Justification: No struct recovery. The original had no complex structs, but the decompiler uses raw pointer arithmetic (e.g., `*(int32_t *)(4 * i + v1)`) instead of array notation. The global variable `g5` is used for `global_var` with offset calculations.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe operations: pointer arithmetic like `4 * i + v1` without bounds, casting between pointer and int types (e.g., `(int64_t)&g28`). The param_asm_multi_insn incorrectly uses memcpy with same source/dest.
