# Decompilation Readability Assessment

## LEVEL 1: LEXICAL CLARITY

### 1. Variable Naming Quality
**Score: 2/10**

The decompiled code uses extremely generic variable names throughout. Examples include:
- `iVar1`, `iVar2`, `iVar3` for integer variables
- `local_8`, `local_10`, `local_20` for stack variables
- `aSStack_30`, `aDStack_18` for object instances
- `puVar1`, `ptVar4` for pointers

These names provide no semantic meaning and make tracking data flow extremely difficult.

### 2. Function Naming Quality  
**Score: 5/10**

Test functions like `test_cpp_member_func`, `test_cpp_constructor` are well-named and match the original. However:
- Many internal functions use cryptic patterns: `FUN_00101840`, `call_weak_fn`
- Member functions are named correctly (e.g., `SimpleClass::getValue`)
- But operator overloads become: `Point::operator_`, `Point::operator__`

### 3. Literal Clarity
**Score: 6/10**

Some literals are clear:
- `0x1234` for class ID is understandable
- String constants like `"Test"` preserved

However, many numeric values lack context, and the magic numbers in complex expressions (especially in virtual function tables and RTTI code) are not explained.

### 4. Noise Reduction
**Score: 3/10**

Significant noise from:
- Extensive STL boilerplate (unique_ptr, tuple, forward implementations)
- Duplicate function definitions
- Many external function stubs that add clutter
- Stack guard checks (`___stack_chk_guard`) add noise

## LEVEL 2: STRUCTURAL INTELLIGIBILITY

### 5. Control Flow Naturalness
**Score: 5/10**

The control flow is generally understandable but has some awkward patterns:
- Virtual function calls use complex indirect patterns: `(***(code **)param_1)(param_1,param_2)`
- Exception handling is fragmented (the `test_cpp_exception` function is incomplete in output)
- Some goto-like patterns in error handling

### 6. Function Decomposition
**Score: 4/10**

The test functions are properly separated, but:
- STL implementation details create many tiny functions
- Some functions are over-fragmented (e.g., many `std::*` helper functions)
- The main test functions are appropriately sized

### 7. Code Linearization
**Score: 6/10**

Generally reads top-to-bottom with:
- Early returns in some places
- Some nested conditionals in RTTI code
- Acceptable linearity for most test functions

### 8. Redundancy Elimination
**Score: 5/10**

Some redundancy in:
- Repeated type checking patterns in RTTI code
- Similar STL wrapper functions
- But overall acceptable for the complexity

## LEVEL 3: TYPE SYSTEM FIDELITY

### 9. Type Precision
**Score: 6/10**

Types are partially preserved:
- Class types are recognized (`SimpleClass`, `LifecycleClass`)
- Basic types like `int`, `char*` are used

- Parameter types sometimes unclear

### 10. Composite Type Recovery
**Score: 5/10**

Structures are partially recovered:
- Class member functions are properly associated
- But field names are lost (replaced with offsets)
- vtable structures are recognized but not fully explained

### 11. Type Safety
**Score: 5/10**

Some type safety issues:
- Many `void*` casts

- Some unsafe conversions in STL code
