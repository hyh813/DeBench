# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names (`a1`, `a2`, `a3`) and local variables are cryptic (`v1`, `v2`, `v3`, `result`, `i`, `j`). No semantic meaning is conveyed - e.g., `ptr_single(_DWORD *a1)` should indicate `p` or `ptr` for a pointer parameter.

**2. Function Naming Quality:** Score: 9/10  
Justification: Nearly all test functions retain their original names: `process_char`, `process_short`, `array_1d_stack`, `ptr_single`, `struct_simple`, etc. Only a few startup functions have generic names (`sub_1030`, `sub_1070`). The function naming is excellent.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers are unexplained throughout - e.g., `65`, `32`, `0x1A` in `process_char`; `1074528256`, `1073794252` in float/double printing. Test functions hardcode expected values (`printf("DT-L1-01: %d\n", 15)`) without explaining what these represent. No named constants recovered.

**4. Noise Reduction:** Score: 8/10  
Justification: Minimal compiler artifacts present. No debug instrumentation, register spills, or frame pointer manipulation visible. The code is relatively clean except for some loop unrolling in array functions. "FAILED to decompile" messages for external functions are appropriate.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most control flow is natural (if/else, while loops). However, array functions like `array_1d_stack` and `array_3d` use aggressive loop unrolling with manual index calculations (`v2 += 8`, `while ((a2 & 0x7FFFFFF8) != v2)`), making the original loop logic obscure. At -O0 this is unexpected.

**6. Function Decomposition:** Score: 8/10  
Justification: Functions maintain appropriate granularity (20-80 lines each). Each test function calls individual test cases. No monolithic functions. Parameter counts match original (e.g., `ptr_func_complex(int (*a1)(int, _DWORD *), int a2)` correctly shows 2 parameters).

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately in functions like `array_1d_stack` and `array_complex_index`. Some minor disruption from loop unrolling in array functions, but overall flow is reasonable.

**8. Redundancy Elimination:** Score: 5/10  
Justification: Significant loop unrolling creates redundant code - `array_1d_stack` manually adds 8 elements per iteration with 8 separate memory accesses. This obscures the simple `sum += arr[i]` pattern from the original. No comments explaining the unrolling.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Mixed results - `bool` correctly used for `process_bool`, `__int16` for `process_short`, `__int64` for `process_ll`. However, most pointers use generic `_DWORD *` instead of proper types like `int *`, `const int *`. `volatile` is lost in `volatile_access` (shows as `_DWORD *` instead of `volatile int *`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: All struct/union information lost. `struct_simple(Point3D *p)` becomes `int struct_simple(_DWORD *a1)` with field access as `a1[2] + *a1 + a1[1]`. No field names (`x`, `y`, `z`) recovered. Nested structs like `Rect` and `Widget` are flat pointer arithmetic. Union types not distinguished.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe patterns: raw pointer arithmetic (`*(_DWORD *)(a1 + 4 * a2)`), void pointer casts without comments, incorrect function pointer signatures (e.g., `ptr_func_complex` uses `_DWORD *` instead of `char **` for second parameter). No `const` correctness preserved.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function call relationships are visible (test functions call individual test targets), but no documentation of who calls what. Callback relationships not explained - e.g., `ptr_func_simple` takes a function pointer but no comment about the callback pattern. Global variable usage not documented.

**17. Module Organization:** Score: 4/10  
Justification: All functions dumped into single file with no logical grouping. Original had clear sections: data types, array types, pointer types, composite types. No static function visibility recovered. No header/API separation.
