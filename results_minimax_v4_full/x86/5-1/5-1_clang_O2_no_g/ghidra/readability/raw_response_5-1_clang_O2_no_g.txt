# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `local_28`, `sVar1`, `puVar1`, `piVar2`, `iVar3`, `iVar4`, `iVar5`, `uVar2`, `auStack_4`. These are completely uninformative compared to source variables like `value`, `name`, `data`, `size`, `instance_count`. The decompiler made no attempt to infer semantic meaning from usage patterns.

**2. Function Naming Quality:** Score: 3/10  
Justification: Test functions are correctly named (`test_cpp_member_func`, `test_cpp_constructor`, etc.), but utility functions use generic names like `FUN_00011030`, `FUN_00011385`. Template functions are partially named (`template_max<int>`, `Container<int>::push`) but member functions lack proper class association in some cases.

**3. Literal Clarity:** Score: 2/10  
Justification: Magic numbers throughout: `0x125c` (should be 4704), `0x3e9` (1000), `0x2a` (42), `0x47` (71), `0x28a` (650), `0x16` (22), `0x27` (39), `0x10` (16), `0x55` (85), `0x2bf` (703). No constants or meaningful names assigned. The string "Test" appears as literal without explanation.

**4. Noise Reduction:** Score: 5/10  
Justification: Many WARNING comments about unknown calling conventions, inlined functions, unreachable blocks create visual clutter. However, the actual decompiled C code itself is relatively clean without excessive compiler artifacts like register spills or debug instrumentation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Control flow is generally readable - simple sequential code with basic conditionals. However, some functions like `test_cpp_member_func` show awkward manual string initialization patterns (setting each byte individually) that obscure the original `strncpy` call. At -O0, this manual byte-by-byte approach is unexpected.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions maintain appropriate granularity. Template instantiations are properly separated (`template_max<int>`, `template_max<double>`). Member functions correctly extracted as separate functions. No excessive fragmentation or monoliths observed.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom with minimal forward jumps. The `test_cpp_oo_features` function is linear despite being long. Early returns are used appropriately in some functions like `template_max`.

**8. Redundancy Elimination:** Score: 7/10  
Justification: No obvious dead code. Some functions like `test_cpp_member_func` show repetitive initialization patterns (setting buffer bytes to null) that could be better optimized, but overall redundancy is manageable.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Uses generic `undefined4` instead of `int`, `undefined1` instead of `char`. Function parameters correctly typed as `int`, `double`, `int*`. However, `size_t` is sometimes `undefined4`, sometimes `size_t`. Return types often `undefined4` instead of proper types. Template types partially recovered (`Container<int>`, `Container<double>`).

**10. Composite Type Recovery:** Score: 5/10  
Justification: Some class names recovered: `Base`, `Derived`, `LifecycleClass`, `Container<int>`, `Container<double>`, `RTTIBase`, `RTTIDerivedA`, `RTTIDerivedB`. However, member fields accessed via raw offsets: `*(undefined4 *)(this + 0x28)` instead of `this->size`. No field names recovered.

**11. Type Safety:** Score: 6/10  
Justification: Generally type-safe code with correct pointer usage. Some unsafe patterns: raw memory access with offsets, but no obvious buffer overflows or type-confused function pointers. Minor issues don't make code dangerous.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 4/10  
Justification: Function signatures include parameter names but no descriptions. WARNING comments are present but generic ("Unknown calling convention"). No explanation of what functions do, algorithm reasoning, or recovery confidence. The decompilation lacks semantic context.

**13. Idiom Recognition:** Score: 5/10  
Justification: Some C++ idioms recognized: vtable calls shown as `(*(code *)**(undefined4 **)param_1)(param_1,param_2)`, virtual dispatch in `call_virtual_func`. However, no explicit pattern naming (e.g., "vtable dispatch pattern"). Template instantiation recognized but not explained.

**14. Anomaly Indication:** Score: 5/10  
Justification: Some anomalies flagged: "Unknown calling convention", "This is an inlined function", "Removing unreachable block". However, indirect calls that couldn't be resolved are not clearly flagged. Some functions return constants without explanation of why (e.g., `test_cpp_virtual_func` returns `0x2a` directly).

**15. Assertion & Contract Recovery:** Score: 3/10  
Justification: No preconditions, postconditions, or invariants documented. No recovery of assert statements or validation logic. The decompilation provides no contract information about what functions expect or guarantee.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are visible in code. Static functions not clearly marked. No documentation of who calls whom, callback relationships, or global variable usage. Relationships must be inferred from code.

**17. Module Organization:** Score: 6/10  
Justification: Functions grouped logically - test functions together, template instantiations together, member functions grouped by class. However, all in single file with no clear module boundaries. Related functionality like RTTI classes scattered.

**18. Build System Compatibility:** Score: 5/10  
Justification: Uses non-standard function names: `operator_new`, `operator_delete`, `builtin_strncpy` instead of `new`, `delete`, `strncpy`. Would require significant patching to compile. Dependencies not documented.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.4/10

### Critical Defects:
- Variable naming completely generic (`local_28`, `sVar1`, `puVar1`) with no semantic inference from usage
- Magic numbers everywhere (`0x125c`, `0x3e9`, `0x2a`) without constant names or explanations
- Composite type field recovery absent - member access shows raw offsets (`this + 0x28`) instead of named fields
- Some functions return hardcoded constants (`test_cpp_virtual_func` returns `0x2a`) losing all algorithmic structure
- No contract/precondition documentation - impossible to understand function requirements

### Best Quality Aspect: Function Decomposition (7/10)
- Template instantiations properly separated, member functions correctly extracted as separate functions with appropriate granularity

### Most Needs Improvement: Variable Naming Quality (2/10)
- Completely uninformative generic names make code practically unusable for analysis; decompiler made zero attempt to infer semantic meaning from usage patterns

### Top 3 Recommendations:
1. **Infer variable names from usage patterns**: `local_28` used with `strlen()` should be named `name` or `buffer`; `sVar1` storing `strlen()` result should be `name_len`
2. **Recover struct field names**: Analyze vtable offsets and constructor patterns to recover field names like `size`, `data`, `instance_count` instead of `this + 0x28`
3. **Replace magic numbers with constants**: Detect computed return values and replace with meaningful constants (e.g., `0x3e9` → `1000`, `0x125c` → `4704`)
