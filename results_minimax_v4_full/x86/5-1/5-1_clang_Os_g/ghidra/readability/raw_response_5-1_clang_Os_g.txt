# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Extensive "Unresolved local var" annotations like `SimpleClass obj@[???]`, `int r1@[???]`, `int result@[???]` provide zero semantic information. Every local variable in test functions is unresolved, making it impossible to track purpose.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions correctly named (`test_cpp_member_func`, `test_cpp_constructor`), but many utility functions are mechanical (`FUN_00011030`). Template instantiations properly named (`template_max<int>`). Mixed results between semantic accuracy and address-based naming.

**3. Literal Clarity:** Score: 3/10  
Justification: Magic numbers everywhere without explanation: `0x1f` (31 for string), `0x14` (20 for object size), `0x32` (50 for loop), `0x2a` (42 expected result). No constants or enum values recovered; all are raw hex/integers.

**4. Noise Reduction:** Score: 4/10  
Justification: Heavy noise pollution: "Unknown calling convention", "Unresolved local var", "Removing unreachable block", "Function: __x86.get_pc_thunk.bx replaced with injection". These WARNING comments clutter every function and obscure actual logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 6/10  
Justification: Most control flow is natural (loops, conditionals), but several test functions are reduced to single return statements with computed constants (e.g., `test_cpp_virtual_func` returns `0x2a` directly). The `call_virtual_func` properly shows vtable indirect call pattern.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately separated - test functions, class methods, template instantiations all distinct. Template functions properly extracted as separate instances. Minor issues with some utility functions being address-based.

**7. Code Linearization:** Score: 7/10  
Justification: Code generally reads top-to-bottom. Early returns used appropriately. No excessive goto or backward jumps disrupting flow. Some forward jumps in exception handling but manageable.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Dead code removal noted in warnings ("Removing unreachable block"). Some loop unrolling visible in `test_cpp_constructor` but not excessive. Minor redundancy from repeated type casts.

### Most Needs Improvement: Variable Naming Quality (2/10)

### Top 3 Recommendations for Decompiler Improvement
1. **Infer variable names from usage patterns**: Analyze how variables are used (arithmetic, indexing, comparison) to suggest semantic names like `packet_size`, `index`, `result`
2. **Recover computed constants**: When a function returns a constant, add comment showing the expected computation (e.g., `// Expected: 10 + 34 + 4660 = 4704`)
3. **Eliminate WARNING noise**: Move decompilation confidence notes to separate metadata rather than cluttering every function with "Unknown calling convention" warnings
