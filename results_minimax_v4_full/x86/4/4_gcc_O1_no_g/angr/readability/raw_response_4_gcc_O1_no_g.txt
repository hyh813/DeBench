# Decompilation Readability Assessment

## LEVEL 1: LEXICAL CLARITY

**1. Variable Naming Quality:** Score: 3/10
Justification: Variables use generic names like `a0`, `a1`, `v0`, `v1`, `ptr`, `i` - no semantic meaning. Compare to original: `cdecl_func(int a, int b)` becomes `cdecl_func(unsigned int a0, unsigned int a1)` losing parameter names entirely.

**2. Function Naming Quality:** Score: 4/10
Justification: Some functions correctly named (`callback_func`, `func_a`, `func_b`, `cdecl_func`) but most use address-based names like `sub_401030`, `sub_401040`. The original function `call_cdecl` is correctly recovered but wrapper functions lose semantic names.

**3. Literal Clarity:** Score: 6/10
Justification: Return values like `15`, `50`, `42` are inlined correctly. However, magic numbers like `4294967295` (for -1) appear without explanation. Some literals like `15959`, `16007` in struct offsets are unexplained.

**4. Noise Reduction:** Score: 6/10
Justification: Relatively clean output but contains some decompilation artifacts like `[D]` markers and incomplete expressions. The `__stack_chk_fail_local` and `__x86.get_pc_thunk` functions add noise but are necessary compiler artifacts.

**7. Code Linearization:** Score: 5/10
