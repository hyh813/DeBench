# Decompilation Readability Assessment

## Function Under Evaluation: `param_fake_branch`

Let me analyze this specific function which demonstrates fake/always-false branches - an obfuscation technique.

### Original Source (param_fake_branch)
```c
int param_fake_branch(int x) {
    int result = x;
    
    // 永假条件：x*x < 0 对于实数x恒不成立
    if (x * x < 0) {
        result = result * 2 + 0xDEADBEEF;
    }
    
    // 另一个永假条件：字符串长度不可能为负
    const char *str = "test";
    if (strlen(str) < 0) {
        result += 1000;
    }
    
    return result;
}
```

## LEVEL 1: LEXICAL CLARITY (30% weight)

### 1. Variable Naming Quality (10%)

**Score: 2/10**

The decompilation uses completely meaningless variable names (`v1`, `v2`, `v3`, `v4`) that provide no semantic information. The original variable `x` is lost, and `result` is replaced with `v2`. The computation `0x100000000 * v1 >> 32` appears to be extracting the upper 32 bits of a 64-bit multiplication - this should be recognized as a sign-extension or high-word operation, not left as cryptic arithmetic. The variable `v4` is used only for an address calculation for `strlen` but serves no actual purpose in the function's logic.

### 2. Function Naming Quality (10%)

**Score: 8/10**

The function name `param_fake_branch` is correctly preserved from the original source. This is excellent recovery - the decompiler correctly identified the function name. The naming accurately reflects the obfuscation technique being demonstrated (fake/always-false branches).

### 3. Literal Clarity (5%)

**Score: 3/10**

The magic number `0xDEADBEEF` from the original source has been transformed into `0x21524111` (which is 0xDEADBEEF * 2 = 0x21524112 minus 1 due to the `2 * v2 -` operation). This transformation obscures the original constant. The string "test" is completely lost - it's accessed via `strlen((char *)(v4 + 0x1d7c))` with an arbitrary offset. The comparison `x * x < 0` is rendered as `(v3 * v3) >= 0` (inverted logic), losing the semantic meaning of "always false for real numbers."

### 4. Noise Reduction (5%)

**Score: 4/10**

The code contains significant noise from the decompilation process. There are extraneous address comments (`// 0x1280`) that add clutter. The variable `v4` is declared but only used in a dead `strlen` call that has no effect on the return value - this is clearly a decompilation artifact where the original string "test" was partially recovered but incorrectly processed. The computation `0x100000000 * v1 >> 32` is low-level noise that should have been abstracted.
