# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables like `local_28`, `uStack_24`, `puVar1`, `iVar3` are generic stack variable names that completely obscure their semantic purpose. The original source had meaningful names like `value`, `name`, `data`, `size`, `multiplier` - all lost in decompilation.

**2. Function Naming Quality:** Score: 9/10  
Justification: Nearly all function names perfectly match the original source: `test_cpp_member_func`, `test_cpp_constructor`, `call_virtual_func`, `test_cpp_virtual_func`, `test_cpp_rtti`, etc. The decompiler correctly recovered the test function names.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic numbers like `0x74736554` (raw "Test" string as integer), `0x125c` (4704), `0x2a` (42) appear without explanation. The intermediate computations (e.g., `obj.getValue()`, `obj.compute(3)`) are completely missing - only final results are shown.

**4. Noise Reduction:** Score: 5/10  
Justification: Multiple WARNING comments clutter the output: "WARNING: Unknown calling convention", "WARNING: Could not recover jumptable", "WARNING: Treating indirect jump as call". While these flag decompilation issues, they add significant noise to the code.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 3/10  
Justification: The decompilation shows only final computed return values, completely losing the actual control flow logic. For example, `test_cpp_member_func` should show object creation, method calls, and arithmetic - instead it just shows `strlen(...) + 0x125c`. The real logic is entirely missing.

**6. Function Decomposition:** Score: 8/10  
Justification: Template instantiations are properly extracted as separate functions: `template_max<int>`, `template_max<double>`, `template_swap<int>`, `Container<int>::push`, `Container<double>::get`, etc. Good decomposition of template code.

**7. Code Linearization:** Score: 8/10  
Justification: The simple test functions are highly linear - mostly just return statements. No complex jumps or nesting. Reads straightforwardly top-to-bottom.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No obvious dead code or redundant expressions. The functions are clean single-return statements without unnecessary duplication.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Heavy use of `undefined4`, `undefined2`, `undefined1`, `undefined8` instead of concrete types like `int`, `double`, `size_t`. While some types like `size_t` and `int` are used correctly, the undefined types degrade type precision significantly.

**10. Composite Type Recovery:** Score: 1/10  
Justification: No class or struct recovery whatsoever. The original `SimpleClass`, `LifecycleClass`, `Base`, `Derived`, `Container<T>` are completely lost. We only see raw stack variables (`local_28`, etc.) instead of reconstructed objects with proper fields.

**11. Type Safety:** Score: 4/10  
Justification: Unsafe patterns present: `(code *)**(undefined4 **)param_1` for vtable calls, pointer arithmetic like `this + 0x28` and `param_1 * 4` without bounds checking. Type-confused function pointers and raw memory operations.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments are almost entirely auto-generated warnings about decompilation issues ("Unknown calling convention", "Could not recover jumptable"). No comments explaining the actual algorithm, class relationships, or expected behavior.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or C++ idioms recognized. The virtual function mechanism, RAII patterns, smart pointers, and exception handling are not identified or named. Just raw computed values.

**14. Anomaly Indication:** Score: 7/10  
Justification: Good use of WARNING comments to flag decompilation problems: indirect jump handling, unreachable blocks, unknown calling conventions. These help analysts recognize uncertain areas.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. No indication of what the functions require or guarantee. The contracts are completely lost.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function names indicate purpose but there's no documentation of call relationships, virtual dispatch targets, or global variable usage. The vtable mechanism is not explained.

**17. Module Organization:** Score: 5/10  
Justification: Functions appear in somewhat arbitrary order in the output. Template instantiations are grouped together which is good, but there's no logical ordering matching the original source file structure.

**18. Build System Compatibility:** Score: 2/10  
Justification: The code uses `undefined4`, `undefined8` types and many undeclared functions (`operator_new`, `operator_delete`, `__dynamic_cast`, `__cxa_throw`). Would not compile without significant patching and type fixes.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.1/10

**Critical Defects:**
- Variable naming is entirely generic (`local_28`, `puVar1`, `iVar3`) - completely loses semantic meaning
- Composite type recovery is absent - no classes or structs reconstructed from original source
- Control flow logic is entirely missing - only final computed results shown, not the actual operations
- Type precision is poor - heavy use of `undefined4`/`undefined8` instead of concrete types
- No comments explaining algorithm logic - only decompilation warnings present

**Best Quality Aspect:** Function Naming Quality (9/10)  
The decompiler correctly recovered all test function names from the original source, making it easy to identify which test case each function implements.

**Most Needs Improvement:** Composite Type Recovery (1/10)  
The decompiler failed to reconstruct any C++ classes or structs. Original classes like `SimpleClass`, `LifecycleClass`, `Base`, `Derived`, `Container<T>` are completely lost, replaced with raw stack variables. This is the most severe readability issue.

**Top 3 Recommendations:**
1. **Recover class/struct layouts** - Infer object structure from stack frame analysis and vtable references; reconstruct fields with meaningful names based on usage patterns
2. **Infer variable names from usage** - Use data flow analysis to rename `local_28` to `name` when used with `strncpy`, or `size` when used in bounds checks
3. **Recover intermediate computations** - Instead of showing only final results like `strlen(...) + 0x125c`, reconstruct the actual operations (object creation, method calls, arithmetic) that produce those results
