# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables are almost exclusively generic names like `a0`, `a1`, `v0`, `v1`, `v2`, `ptr`, `i`. The decompiler uses register-based naming (r0, r1, r2, r3) and stack positions ([bp-0x4], [bp+0x0]) rather than semantic names. Compare source's meaningful names like `count`, `args`, `sum` to decompiled's `i`, `v7`, `v6`.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are correctly recovered: `cdecl_func`, `stdcall_func`, `fastcall_func`, `varargs_func`, `func_no_args`, `func_many_args`, `func_mixed_args`, `ret_basic_type`, `ret_pointer`. Some internal functions have generic names like `sub_103e8`, but the test functions are well-named.

**3. Literal Clarity:** Score: 6/10  
Justification: Return values like `15`, `50`, `6`, `100`, `21` are computed results (not magic numbers). However, string format addresses like `0x114cc`, `0x11590` are used without recovering the actual strings. Stack check guard `GLIBC_2.4::__stack_chk_guard` is correctly identified as external.

**4. Noise Reduction:** Score: 6/10  
Justification: Some noise present: register spill comments (`// r0`, `// r1`), stack frame annotations (`// [bp-0x4]`, `// [bp+0x0]`), and some redundant constructs like calling `param_varargs` twice in `call_varargs_param`. However, no debug instrumentation or excessive artifacts.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Excessive use of `armg_calculate_condition()` for simple comparisons creates unreadable nested ternary expressions. Example: `if (!armg_calculate_condition(5, v5, v6, v7))` instead of simple `if (v2 != v4)`. Some goto labels present (`LABEL_104d0`). The `__aeabi_dadd` function has extremely convoluted control flow with 100+ variables.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions are appropriately separated. The main test functions (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) are correctly extracted. Some inlining occurred as expected with -O0 (e.g., `call_cdecl()` returns constant 15). Function sizes are reasonable.

**7. Code Linearization:** Score: 6/10  
Justification: Generally reads top-to-bottom with minimal forward jumps. However, some functions have complex nested conditions that disrupt linear flow. The `param_complex_cast` function has confusing goto patterns. Most functions are reasonably linear.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `call_varargs_param` calls `param_varargs(4, 10, 20, 30)` twice redundantly. Some duplicate stack check patterns. Loop constructs are sometimes unrolled awkwardly. However, no severe redundancy that makes code unmaintainable.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Almost all integers are `unsigned int`, losing sign information from source (`int` vs `unsigned int`). Function parameters like `int a, int b` become `unsigned int a0, unsigned int a1`. String parameters correctly identified as `char *a1`. Return types sometimes wrong (e.g., `int` vs `unsigned int`).

**10. Composite Type Recovery:** Score: 3/10  
Justification: Some struct typedefs created with generic names: `typedef struct struct_0 { char field_0; } struct_0;`. The `param_ptr_array` function has a strange struct_0** type. No meaningful struct field names recovered. The source's `SmallStruct` with `x, y` fields is not recovered in `func_struct_byptr`.

**11. Type Safety:** Score: 5/10  
Justification: Some unsafe patterns present: `*((int *)&v0)` type punning, pointer arithmetic like `ptr + 1`, `ptr[1]`. Function pointer calls like `a0(a1)` lose type safety. However, no severe buffer overflow patterns or completely dangerous code.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality:** Score: 2/10  
Justification: Almost no useful comments. Only register location annotations (`// r0`, `// [bp-0x4]`) which are noise. No explanation of algorithm logic, no recovery of original variable purposes, no uncertainty indicators. The decompiled code provides no guidance on what functions do.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns or algorithms explicitly identified. Some hints: varargs handling is recognizable as loop over arguments, struct by-value has characteristic copy patterns. But nothing explicitly named or documented.

**14. Anomaly Indication:** Score: 1/10  
Justification: No anomaly flags. The `param_complex_cast` function has completely broken logic with `if ((vvar_1 != 0x1<32>)) { Goto None }` but no indication this is problematic. Indirect calls not resolved are not flagged. Code presents incorrect information confidently.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. No error condition explanations. The source's NULL checks in functions like `func_struct_byptr` are present but not documented as requirements.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Some external declarations present (`extern unsigned int GLIBC_2.4::__stack_chk_guard`). Function call relationships not documented. Global variable usage not explained. Callback relationships (like `param_func_ptr` taking function pointer) not documented.

**17. Module Organization:** Score: 6/10  
Justification: Functions are grouped reasonably - test functions together, helper functions separate. Static functions properly scoped. However, the massive `__aeabi_dadd` library function is included inline rather than recognized as external.

**18. Build System Compatibility:** Score: 5/10  
Justification: Has `extern` declarations for library functions. Would require linking against libgcc or equivalent. Some function signatures may not compile directly (the broken `param_complex_cast`). Moderate patching needed.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 4.3/10

**Critical Defects:**
- Variable names are entirely generic (`a0`, `a1`, `v0`, `v1`) with no semantic recovery
- The `param_complex_cast` function has completely broken logic with nonsensical conditions
- No comments explaining any function's purpose or algorithm
- Type precision severely degraded (all `int` became `unsigned int`)
- No struct field names recovered (all `field_0`, `field_1`)
- No anomaly indication for suspicious or broken code
- Excessive use of `armg_calculate_condition()` makes control flow unreadable

**Best Quality Aspect:** Function Naming Quality (8/10)  
The decompiler correctly recovered most function names from the source, including test function names like `cdecl_func`, `varargs_func`, `ret_pointer`, etc.

**Most Needs Improvement:** Variable Naming Quality (2/10)  
Variables are named after registers and stack positions rather than semantic purposes. The original source's meaningful names like `count`, `args`, `sum`, `ptr`, `val` are completely lost.

**Top 3 Recommendations:**
1. **Infer variable names from usage patterns**: Track how variables are used (e.g., `ptr` used in pointer arithmetic, `sum` accumulated in loops) to assign semantic names
2. **Recognize and flag anomalies**: The `param_complex_cast` function has broken logic - add uncertainty indicators for unresolvable indirect calls
3. **Recover struct field names**: Analyze struct access patterns to recover meaningful field names like `x`, `y`, `data` instead of `field_0`, `field_1`
