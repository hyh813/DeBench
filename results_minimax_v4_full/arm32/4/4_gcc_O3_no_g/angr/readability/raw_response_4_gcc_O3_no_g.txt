# Decompilation Readability Assessment

## Original Source Analysis

The source code is a comprehensive test suite for ARM64 calling conventions, parameter passing patterns, and return value handling. It contains 665 lines of well-organized C code testing:
- Multiple calling conventions (cdecl, stdcall, fastcall, ARM AAPCS, etc.)
- Parameter passing modes (by value, by pointer, arrays, varargs, function pointers)
- Return value handling (basic types, pointers, small/large structs)

The binary was compiled with `-O0 -g` (debug mode), so we should expect near-perfect readability.

---

## LEVEL 1: LEXICAL CLARITY (30% weight)

### 1. Variable Naming Quality (10%)

**Score: 2/10**

The decompiled code uses exclusively generic register and stack slot names. Examples:
- `v0`, `v1`, `v2`, `v3`, `v4` for local variables
- `a0`, `a1`, `a2`, `a3` for function parameters
- `ptr` used generically even when it represents specific data structures

Compare to source where variables like `count`, `sum`, `args` (for va_list) are meaningful. The decompiler makes no attempt to infer semantic meaning from usage patterns.

### 2. Function Naming Quality (10%)

**Score: 6/10**

Mixed results. Some functions are correctly named:
- `cdecl_func`, `call_cdecl`, `func_a`, `func_b`, `func_no_args`, `varargs_func`

But many functions use address-based naming:
- `sub_103c4`, `sub_1047c`, `sub_103c4`
- `param_varargs.constprop.0` (obfuscated variant naming)

The test functions (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) are correctly named, showing the decompiler can recover some names from debug symbols.

### 3. Literal Clarity (5%)

**Score: 4/10**

The decompiler uses raw numeric addresses for string literals:
- `puts(0x1143c)` instead of `puts("=== 测试基础调用约定 ===")`
- `__printf_chk(1, 0x11500, ...)` instead of meaningful format strings

Magic numbers like `4294967295` (0xFFFFFFFF) appear without explanation. The constant `15` appears in multiple places (e.g., `call_cdecl()`) but without context that it represents `5 + 10`.

### 4. Noise Reduction (5%)

**Score: 3/10**

Significant noise pollution:
- Stack canary checks: `GLIBC_2.4::__stack_chk_guard` and `__stack_chk_fail()` appear in many functions
- Complex condition calculation functions: `armg_calculate_condition()` with cryptic parameters
- Fake struct definitions: `typedef struct struct_0 { char field_0; } struct_0;` inserted inline
- Incomplete function stubs: `__aeabi_dadd`, `__aeabi_drsub` with empty implementations

## LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight)

### 5. Control Flow Naturalness (8%)

**Score: 5/10**

Simple functions have natural control flow:
```c
int cdecl_func(unsigned int a0, unsigned int a1) {
    return a0 + a1;
}
```

But more complex functions use goto-based flow that obscures logic:
```c
if (a0 <= 2) {
    v0 = a0 + 1;
    if (!armg_calculate_condition(146, a0, 2, 0))
        goto LABEL_10e8a;
```

The `armg_calculate_condition` calls are decompiler artifacts representing ARM condition flags, not meaningful control flow.

### 6. Function Decomposition (7%)

**Score: 7/10**

Function granularity is appropriate. Each test function is properly separated:
- `test_calling_conventions()`, `test_parameter_passing()`, `test_return_values()`
- Individual testee functions like `cdecl_func`, `varargs_func` are properly isolated

The decompiler correctly kept functions separate rather than inlining everything.

### 7. Code Linearization (5%)

**Score: 6/10**

Most functions read top-to-bottom. However, some functions have forward jumps:
```c
if (i > 0) {
    v6 = 0;
    v7 = 0;
    do { ... } while (i != v6);
} else {
    v7 = 0;
}
```

The varargs loop is correctly recognized as a do-while, but the overall flow is somewhat disrupted by the if-else structure.

### 8. Redundancy Elimination (5%)

**Score: 5/10**

Some redundancy from optimization artifacts:
- `param_varargs` and `varargs_func.constprop.0` are duplicate functions
- `call_varargs_param` calls the constprop.0 version twice unnecessarily

The loop in `varargs_func` is unrolled but still recognizable as iteration.

## LEVEL 3: TYPE SYSTEM FIDELITY (20% weight)

### 9. Type Precision (8%)

**Score: 2/10**

Severe type degradation. All parameters and returns are `unsigned int`:
- Source: `int cdecl_func(int a, int b)` → Decompiled: `int cdecl_func(unsigned int a0, unsigned int a1)`
- Source: `char *param_string(const char *str)` → Decompiled: `int param_string(char *a0)`
- Source: `double` parameters → Decompiled: `unsigned int` with library calls

The decompiler lost:
- Signed/unsigned distinction
- Floating-point types
- Size_t, int32_t specificity

### 10. Composite Type Recovery (7%)

**Score: 2/10**

The decompiler created fake struct types with no semantic meaning:
```c
typedef struct struct_0 {
    char field_0;
} struct_0;
```

Original structs like `SmallStruct { int x, y; }` and `LargeStruct { long long data[16]; }` are completely lost. The decompiler treats struct parameters as individual integers:
```c
int func_struct_byval(unsigned int a0, unsigned int a1, ..., unsigned int a30)  // 31 parameters!
```

### 11. Type Safety (5%)

**Score: 4/10**

Several type safety issues:
- Unsafe pointer arithmetic in `varargs_func`: `v8 = ptr + 1;`
- Incorrect return type for `ret_pointer`: returns `int` instead of `int*`
- Function pointer type loss: `param_func_ptr(unsigned int *a0, unsigned int a1)` where `a0` is actually a function pointer
- The `param_complex_cast` function has completely broken type handling

## LEVEL 4: SEMANTIC TRANSPARENCY (15% weight)

### 12. Comment Quality & Presence (5%)

**Score: 1/10**

No comments present in the decompiled output. Compare to source which has extensive comments explaining:
- Test scenarios (e.g., "CALL-L1-01: cdecl调用约定")
- Expected values
- ABI-specific behaviors

The decompiled code provides zero context about what functions do or why.

### 13. Idiom Recognition (3%)

**Score: 3/10**

Some patterns are recognizable:
- Varargs handling is partially recognized (va_list simulation)
- Function pointer callbacks partially work (`a0(a1)` in param_func_ptr)

But design patterns like "calling convention testing" are completely lost.

### 14. Anomaly Indication (3%)

**Score: 2/10**

No anomaly indication. The decompiler silently produces incorrect code:
- `param_complex_cast` has completely broken logic with unreachable code
- `ret_opaque_handle` returns `"d"` (a string literal!) instead of proper handle
- The condition calculation functions are unexplained artifacts

### 15. Assertion & Contract Recovery (4%)

**Score: 1/10**

No contract information recovered. The source has implicit contracts:
- `func_struct_byptr` requires non-null pointer
- Varargs functions have count parameter semantics

These are not documented in the decompiled output.

---

## LEVEL 5: CONTEXTUAL COHERENCE (10% weight)

### 16. Cross-Reference Clarity (4%)

**Score: 3/10**

Some relationships are clear:
- `call_cdecl()` calls `cdecl_func()`
- `test_calling_conventions()` calls various test functions

But callback relationships are lost:
- `callback_func` is defined but its usage through `param_func_ptr` is unclear

### 17. Module Organization (3%)

**Score: 5/10**

Functions are roughly grouped by address but not logically organized. The output is a flat list of functions in address order, not organized by:
- Test categories (calling conventions, parameters, returns)
- Related functionality

### 18. Build System Compatibility (3%)

**Score: 2/10**

The code would not compile:
- Fake struct typedefs are incomplete
- `armg_calculate_condition` is undefined
- String literals are replaced with numeric addresses
- `handle1.1`, `handle2.0` are undefined externs

---

### Critical Defects

1. **Variable naming is unusable**: All variables use generic names (v0, v1, a0, a1, ptr) with no semantic meaning derived from usage patterns

2. **Complete type precision loss**: All types are `unsigned int` - the original had int, char*, double, long long, size_t, etc.

3. **Struct recovery failure**: Original structs like `LargeStruct { long long data[16] }` are flattened into 31 individual integer parameters

4. **No comments or documentation**: Zero context about what functions do, their contracts, or expected behavior

5. **String literals replaced with addresses**: `puts(0x1143c)` instead of `puts("=== 测试基础调用约定 ===")` - makes output completely unreadable

---

### Best Quality Aspect

**Function Decomposition (7/10)** - The decompiler correctly kept functions separate and at appropriate granularity. Test functions like `test_calling_conventions()` are properly isolated from the functions they test.

---

### Most Needs Improvement

**Type System Fidelity (2.7/10)** - The decompiler completely lost type information. Every parameter and return value is `unsigned int`, structs are flattened into individual parameters, and floating-point operations are replaced with complex library calls. This is the most severe degradation.

---

### Top 3 Recommendations

1. **Infer variable names from usage patterns**: When a variable is used as a loop counter, name it `i` or `index`. When used for string length, name it `len`. The current generic names make code analysis extremely difficult.

2. **Recover string literals from data section**: The format strings and output messages exist in the binary - the decompiler should extract them and use `puts("message")` instead of `puts(0x1143c)`.

3. **Preserve type information from debug symbols**: With `-g` flag, DWARF debug info contains type definitions. The decompiler should recover `int`, `double`, `char*`, struct layouts rather than defaulting to `unsigned int`.
