# Decompilation Readability Assessment

## Analysis Overview

This is a comprehensive test suite for calling conventions, parameter passing, and return value handling compiled with `-O0` on ARM64. The decompiled output is 1934 LOC vs 665 LOC original - significant expansion due to type imprecision and structural artifacts.

---

### LEVEL 1: LEXICAL CLARITY (30% weight)

#### 1. Variable Naming Quality
**Score: 3/10**

Justification: Variables use generic positional names (`param_1`, `param_2`, `local_28`, `local_24`) that obscure semantic meaning. For example, in `varargs_func`, `local_28` should be named `args` based on its usage as a `va_list`, and `local_24` should be `sum`. The decompiler makes no attempt to infer purpose from usage patterns.

#### 2. Function Naming Quality
**Score: 9/10**

Justification: Function names are accurately recovered and match the original source exactly (`cdecl_func`, `call_cdecl`, `varargs_func`, `param_by_value_int`). The decompiler correctly preserved the semantic intent in function names, which is critical for readability.

#### 3. Literal Clarity
**Score: 5/10**

Justification: Mixed results - some literals are clear (e.g., `0x2a` for 42 in `func_no_args`, `0x14` for 20), but many remain as unexplained magic numbers. In `call_mips`, parameters are passed as hex (`0x14`, `0x1e`, `0x28`) rather than decimal, making the values harder to verify against source expectations.

#### 4. Noise Reduction
**Score: 6/10**

Justification: Moderate noise present. Stack canary checks (`__stack_chk_fail`) are legitimate but clutter logic. Many functions have unused local variables (`local_c`) used only for canary checking. Type declarations include `undefined4`, `undefined8` which add noise but are unavoidable without debug symbols.

---

### LEVEL 2: STRUCTURAL INTELLIGIBILITY (25% weight)

#### 5. Control Flow Naturalness
**Score: 7/10**

Justification: Most control flow is natural and matches source. Loops are properly recovered as `for` constructs. However, some awkward patterns exist - in `func_struct_byval`, the loop uses `local_2c * 2` indexing which doesn't cleanly represent the original array access pattern. At `-O0`, this level of fidelity is expected.

#### 6. Function Decomposition
**Score: 8/10**

Justification: Functions are appropriately decomposed. The decompiler correctly kept the test wrapper functions (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) separate from the functions they test. No over-fragmentation or inlining issues at this optimization level.

#### 7. Code Linearization
**Score: 7/10**

Justification: Code generally reads top-to-bottom. Early returns are preserved. Some forward jumps exist in complex functions like `test_calling_conventions` but are manageable. The main test function has many sequential calls which read naturally.

#### 8. Redundancy Elimination
**Score: 6/10**

Justification: Some redundancy from loop unrolling in `func_struct_byval` where the original simple loop becomes a complex indexed access pattern. The decompiler didn't abstract this into a cleaner loop construct. However, no severe duplicate code blocks observed.

---

### LEVEL 3: TYPE SYSTEM FIDELITY (20% weight)

#### 9. Type Precision
**Score: 4/10**

Justification: Significant type degradation. Integer types are generic `int` instead of specific widths. Pointers are often `int *` instead of proper types. In `param_string`, the parameter is `byte *` instead of `char *`. The `double` parameter in `func_mixed_args` is broken into multiple `undefined4` parameters. Function pointers use generic `code *` type.

#### 10. Composite Type Recovery
**Score: 2/10**

Justification: Severe degradation. The original source defines `SmallStruct` with fields `x` and `y`, but `func_struct_byptr` receives `int *param_1` and accesses `param_1[1]` and `*param_1` - losing all struct semantics. `LargeStruct` (128 bytes) is completely lost, passed as separate parameters. No struct type recovery whatsoever.

#### 11. Type Safety
**Score: 5/10**

Justification: Moderate type safety issues. In `param_ptr_array`, the decompiler uses `**(byte **)(param_1 + local_c * 4)` which is unsafe pointer arithmetic. The `func_mixed_args` function has broken floating-point handling with multiple undefined parameters. Some unsafe casts present but not catastrophic.

---

### LEVEL 4: SEMANTIC TRANSPARENCY (15% weight)

#### 12. Comment Quality & Presence
**Score: 2/10**

Justification: No comments present in the decompiled output. The original source has extensive comments explaining each test case (e.g., "CALL-L1-01: cdecl调用约定"). The decompiler provides zero documentation about the calling convention tests, parameter passing patterns, or expected behaviors.

#### 13. Idiom Recognition
**Score: 4/10**

Justification: Some basic patterns recognized - the varargs functions are identified as loops, but the decompiler doesn't recognize the `va_list`/`va_start`/`va_arg` pattern. No recognition of calling convention idioms (cdecl vs stdcall differences are not highlighted).

#### 14. Anomaly Indication
**Score: 3/10**

Justification: No anomaly flags. The decompiler silently produces incorrect-looking code (like the broken `func_mixed_args` with multiple undefined parameters) without any indication that the floating-point handling may be incorrect. No warnings about unresolved indirect calls or unusual stack layouts.

#### 15. Assertion & Contract Recovery
**Score: 2/10**

Justification: No contract information recovered. The original source has clear semantics (e.g., `func_struct_byptr` requires non-null pointer, returns -1 on null) but the decompiled version shows the null check without explaining the contract. Preconditions/postconditions not documented.

---

### LEVEL 5: CONTEXTUAL COHERENCE (10% weight)

#### 16. Cross-Reference Clarity
**Score: 5/10**

Justification: Function call relationships are clear from the code structure - you can see `call_cdecl` calling `cdecl_func`. However, no explicit documentation of call graphs or relationships. The callback relationship in `param_func_ptr` → `callback_func` is visible but not annotated.

#### 17. Module Organization
**Score: 7/10**

Justification: Functions are grouped logically - all calling convention tests together, parameter tests together, return tests together. The three main test functions (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) are properly separated. Some library functions interspersed but that's expected.

#### 18. Build System Compatibility
**Score: 5/10**

Justification: The code would not compile as-is due to `undefined4`, `undefined8`, `code *` types. External dependencies (`printf`, `strlen`, `memcpy`) are declared but some incorrectly (e.g., `printf` with `int` return). Requires significant patching to compile.

### Critical Defects

- **Struct type recovery completely failed**: `SmallStruct` and `LargeStruct` are reduced to raw pointer arithmetic (`param_1[1] * *param_1`), losing all semantic meaning of struct fields
- **Variable naming is universally generic**: All parameters use `param_N` pattern; locals use `local_NN` pattern with no semantic inference
- **No comments or documentation**: The extensive test case documentation from source (e.g., "CALL-L1-01: cdecl调用约定") is completely lost
- **Type precision severely degraded**: Floating-point parameters in `func_mixed_args` are broken into multiple undefined4 parameters; function pointers use generic `code *` type
- **No anomaly indication**: The broken floating-point handling in `func_mixed_args` is presented without any warning about potential decompilation issues

---

### Best Quality Aspect

**Function Naming Quality (9/10)** - The decompiler accurately recovered all function names from the original source, making it easy to identify which test case each function implements.

---

### Most Needs Improvement

**Comment Quality & Presence (2/10)** - The decompiler provides zero comments, losing all the semantic context from the original source's extensive documentation. Combined with the low scores on variable naming and type recovery, this makes the decompiled code extremely difficult to understand without reference to the original.

---

### Top 3 Recommendations

1. **Infer variable names from usage patterns**: When `param_1` is used as a `va_list` (passed to va_start/va_arg), name it `args`. When `local_24` accumulates a sum in a loop, name it `sum`. Usage analysis can dramatically improve naming.

2. **Recover struct types from stack layout analysis**: When consecutive stack parameters or local variables are accessed with fixed offsets (like `param_1[1]`), attempt to reconstruct the underlying struct type with field names.

3. **Add decompilation confidence comments**: When type recovery is uncertain (like the broken floating-point parameters), add comments indicating "DECOMPILER UNCERTAIN: floating-point handling may be incorrect" to help analysts verify critical logic.
