# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic positional names (`a0`, `a1`, `a2`) and local variables use cryptic names (`v0`, `v1`, `v2`, `v3`, `v5`, `v6`, `v7`). The original source uses meaningful names like `a`, `b`, `count`, `ptr` but these are completely lost in decompilation. Register hints like `// r0` add noise without semantic value.

**2. Function Naming Quality:** Score: 7/10  
Justification: Application-level functions like `cdecl_func`, `call_cdecl`, `test_calling_conventions`, `test_parameter_passing`, `test_return_values` are correctly named matching the source. However, internal/runtime functions use generic names like `sub_4003f0`, `_init`, `_start`. The core test functions are well-recovered.

**3. Literal Clarity:** Score: 3/10  
Justification: String literals are replaced with external references like `&g_40152c`, `&g_401554`, `&g_4017d6` with no indication of their content. Magic numbers like `4294967295` (0xFFFFFFFF) appear without explanation. The original source had clear string outputs like `"CALL-L1-01: %d\n"` but these are now opaque addresses.

**4. Noise Reduction:** Score: 2/10  
Justification: Severe noise pollution throughout. Register annotations (`// r0`, `// r1`, `// [bp-0x4]`, `// [bp-0x10]`) clutter every variable. The `__adddf3`, `__floatsidf`, `armg_calculate_condition` functions are decompiler artifacts. Variable declarations like `unsigned int v0; // r0` mix source and tool output.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Simple functions like `cdecl_func` have natural flow: `return a1 + a0`. However, complex functions like `param_double_ptr` use goto statements (`LABEL_400b15`, `LABEL_400b16`) and the massive `__adddf3` function (400+ lines) has extremely convoluted control flow with nested condition calculations. At -O0, this level of complexity is unexpected.

**6. Function Decomposition:** Score: 8/10  
Justification: The decompiler correctly maintains the function boundaries from the source. Each test function (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`) is separate. Helper functions like `cdecl_func`, `varargs_func` are properly extracted. The granularity matches the original source well.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom without excessive jumps. However, `varargs_func` has a backward jump in its do-while loop that could be clearer. The `param_double_ptr` function has multiple goto labels disrupting linear flow. Generally readable but with some interruptions.

**8. Redundancy Elimination:** Score: 4/10  
Justification: Significant redundancy found. In `call_varargs_param`: `v1 = param_varargs(4, 10, 20, 30); return param_varargs(4, 10, 20, 30);` - the function is called twice unnecessarily. Similar issue in `test_parameter_passing` with `v0 = 40;` followed by unused variable. Dead code and redundant calls present.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: All parameters use generic `unsigned int` type. The original source uses `int` for most parameters, but decompiler outputs `unsigned int a0`, `unsigned int a1`. No `const` correctness preserved. Function pointers like `int (*callback)(int)` become `unsigned int *a0`. Pointers lose their reference semantics.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct recovery visible. The original source defines `SmallStruct`, `LargeStruct`, `BigStruct`, `TestIntPair` but the decompiled code flattens all struct parameters into individual unsigned integers (`func_struct_byval` has 16 parameters `a0` through `a15`). The struct concept is completely lost.

**11. Type Safety:** Score: 5/10  
Justification: Moderate type safety issues. Pointer arithmetic is exposed directly (e.g., `ptr + 1` in `ret_pointer`). Array indexing like `a0[1] + *(a0)` in `param_string` works but loses array semantics. No bounds checking. The `param_func_ptr` incorrectly treats function pointer as `unsigned int *` rather than function signature.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No meaningful comments explaining algorithm or intent. Only register annotations like `// r0`, `// [bp-0x4]` which are decompiler artifacts, not useful documentation. The original source had extensive comments explaining test scenarios (e.g., "CALL-L1-01: cdecl调用约定"). Zero semantic comments in decompiled output.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns or algorithms identified. The varargs pattern (`va_list`, `va_start`, `va_arg`, `va_end`) is partially visible in `varargs_func` but not named or explained. The original source's test scenarios (calling conventions, parameter passing patterns) are not recognized as idioms.

**14. Anomaly Indication:** Score: 2/10  
Justification: No anomaly flags. The `ret_opaque_handle` function has suspicious code (`return "d";` for an int function) but no warnings. The complex `armg_calculate_condition` calls in `__adddf3` could indicate decompilation uncertainty but aren't flagged. Silent on potential issues.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants documented. The original source has clear expectations (e.g., "期望: 15", "期望: 42") but these are lost. No documentation of what parameters require or what functions guarantee.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are present but relationships aren't documented. No indication of which functions call which. The original source clearly shows `call_cdecl()` calls `cdecl_func(5, 10)` but this caller-callee relationship isn't explicitly documented in decompiled output.

**17. Module Organization:** Score: 4/10  
Justification: Functions are grouped by address (0x4005c0, 0x4005c8, etc.) rather than logically. The original source groups by test category (calling conventions, parameter passing, return values) but decompiled output is just address-ordered. No logical grouping visible.

**18. Build System Compatibility:** Score: 3/10  
Justification: Extensive `extern` declarations required. Many undefined references (`g_412008`, `g_40152c`, etc.). The code would not compile standalone. Requires linking against libc and significant patching. No dependency documentation provided.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.9/10

**Critical Defects:**
- Variable naming uses generic positional names (`a0`, `a1`, `v0`, `v1`) losing all semantic meaning
- No struct/union recovery - all composite types flattened to individual parameters
- String literals replaced with opaque addresses (`&g_40152c`) losing all content
- Severe noise from register annotations (`// r0`, `// [bp-0x4]`) cluttering every declaration
- No comments explaining algorithm intent or test scenarios
- Redundant function calls in `call_varargs_param` (calls same function twice)
- Type precision lost - all `int` becomes `unsigned int`, no `const` preserved

**Best Quality Aspect:** Function Decomposition (8/10)  
The decompiler correctly preserved function boundaries matching the original source. Each test function and helper function remains separate with proper granularity.

**Most Needs Improvement:** Variable Naming Quality (2/10)  
All parameters and local variables use cryptic positional names. At -O0, the decompiler should be able to infer meaningful names from usage patterns. This is the most impactful degradation.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage**: Analyze how parameters are used (arithmetic, indexing, dereferencing) to suggest names like `packet_size`, `index`, `buffer` instead of `a0`, `v1`
2. **Recover string literals**: Instead of external references, extract and display the actual string content for printf-style functions
3. **Reconstruct struct types**: Identify consecutive parameter groups that form structures and reconstruct the composite type with field names
