# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Parameters use generic names like `a0`, `a1`, `a2` throughout. In `func_struct_byval`, there are 31 parameters named `a0` through `a30`. Local variables use register names (`v0`, `v1`, `r0`, `r1`) or stack offsets (`[bp-0x4]`), providing no semantic meaning about their purpose.

**2. Function Naming Quality:** Score: 6/10  
Justification: Test functions correctly named (`test_calling_conventions`, `test_parameter_passing`, `test_return_values`), and target functions like `cdecl_func`, `stdcall_func`, `varargs_func` match source. However, some utility functions have auto-generated names like `sub_4003f0`, `sub_400490`.

**3. Literal Clarity:** Score: 5/10  
Justification: Return values like `15`, `50`, `6` are hardcoded without explaining they're computed results. String literals referenced as `&g_4017d6` are external. Magic number `4294967295` (0xFFFFFFFF) appears without constant name. Mixed - some values clear, others cryptic.

**4. Noise Reduction:** Score: 3/10  
Justification: Severe noise pollution from register annotations (`// r0`, `// lr`), stack frame comments (`// [bp-0x10]`), and ARM condition calculations (`armg_calculate_condition`, `armg_calculate_flag_c`). The `__adddf3` function contains hundreds of lines of incomprehensible flag manipulation code that obscures the actual floating-point addition logic.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 4/10  
Justification: Simple functions like `cdecl_func` have natural flow, but `varargs_func` uses goto-based loops (`goto LABEL_4004b4`). The `__adddf3` function has spaghetti control flow with nested conditionals and forward jumps, making execution nearly impossible to trace.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions maintain appropriate granularity - test functions call multiple targets, library functions are separate. No 500-line monoliths. Minor issue: some functions like `func_struct_byval` expose too many parameters (31) revealing implementation details.

**7. Code Linearization:** Score: 4/10  
Justification: Simple functions read top-to-bottom, but complex functions like `__adddf3` have extensive forward jumps and computed gotos that disrupt linear reading. The `param_double_ptr` function has multiple labeled blocks interrupting flow.

**8. Redundancy Elimination:** Score: 5/10  
Justification: `call_varargs_param` redundantly calls `param_varargs` twice with identical results. Loop constructs present but uncommented. Some dead code from register spilling visible. Moderate redundancy from optimization artifacts.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 5/10  
Justification: Parameters use generic `unsigned int` instead of `int32_t` or `size_t`. Pointers correctly typed (`int *`, `char *`). Function pointers like in `param_func_ptr` use `unsigned int *` instead of proper function signature. Mixed precision - basic types okay, but loses signed/unsigned distinction and size_t.

**10. Composite Type Recovery:** Score: 3/10  
Justification: `param_ptr_array` defines a fake struct `struct_0 { char field_0; }` instead of recovering `char*`. `func_struct_byval` expands all 16 `long long` array elements as separate parameters (`a0` through `a30`), losing the struct entirely. No struct/union recovery visible.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe operations: pointer arithmetic in `ret_pointer` returns `a0 + 4` without cast. Function pointer called directly: `a0(a1)` in `param_func_ptr`. The `func_struct_byptr` has complex conditional logic that could lead to undefined behavior. Significant type safety concerns.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 3/10  
Justification: Comments only show register assignments (`// r0`, `// lr`) and stack offsets (`// [bp-0x4]`), which are noise rather than useful documentation. No algorithm explanations, no decompilation confidence indicators, no API contracts. Auto-generated noise dominates.

**13. Idiom Recognition:** Score: 3/10  
Justification: No design patterns identified. The varargs handling in `varargs_func` uses goto loops instead of being recognized as stdarg pattern. No algorithm patterns (loop, recursion) named. Completely missing pattern recognition.

**14. Anomaly Indication:** Score: 2/10  
Justification: No warnings about indirect calls that couldn't be resolved. No flags for unusual stack layouts. The `ret_opaque_handle` returns `"d"` (a string literal) which seems wrong but no anomaly indicated. Silent failures throughout.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions documented (e.g., "requires: ptr != NULL"). No postconditions (e.g., "ensures: return is heap-allocated"). No invariant recognition. The `func_struct_byptr` null-check logic has implicit contracts not documented.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 5/10  
Justification: Function calls are clear (`cdecl_func(5, 10)` becomes direct calls). Global variables referenced as `extern`. But no documentation of call graph relationships, no indication of who calls whom. Some relationships inferable but not explicit.

**17. Module Organization:** Score: 6/10  
Justification: Functions logically grouped: test functions, target functions, library functions (`__adddf3`, `__fixdfsi`). Static functions properly scoped. Minor misplacement - some utility functions interleaved with test code. Reasonable organization.

**18. Build System Compatibility:** Score: 4/10  
Justification: Code has `extern` declarations for external functions but won't compile standalone. Dependencies on libc functions not documented. The `__adddf3` function has incomplete code (calls `__adddf3()` with no parameters). Requires extensive patching to compile.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.9/10

**Critical Defects:**
- Variable naming uses generic `a0`-`a31`, `v0`-`v100` with register annotations instead of semantic names
- `func_struct_byval` expands 128-byte struct as 31 individual parameters, completely losing composite type
- `__adddf3` contains ~400 lines of incomprehensible ARM condition flag calculations that obscure floating-point logic
- No comments explaining algorithm intent, only register/stack annotations
- `ret_opaque_handle` returns string literal `"d"` instead of proper pointer - anomaly not flagged

**Best Quality Aspect:** Function Decomposition (scored 7/10)  
Functions maintain appropriate granularity with clear separation between test functions, target functions, and library utilities.

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
Parameters use `a0`, `a1` etc. and locals use register names (`v0`, `r0`, `lr`) with stack offsets, providing zero semantic information about variable purpose.

**Top 3 Recommendations for Decompiler Improvement:**
1. Infer variable names from usage patterns - `a0` used in size check should be named `count` or `size`
2. Recover struct types from parameter expansion patterns - 31 consecutive parameters with pattern suggest a struct
3. Simplify or abstract ARM condition flag calculations in floating-point functions - current output is machine-generated noise
4. Add anomaly indicators for suspicious code patterns like returning string literals where pointers expected
5. Replace register annotations with meaningful comments about parameter roles and calling convention implications
