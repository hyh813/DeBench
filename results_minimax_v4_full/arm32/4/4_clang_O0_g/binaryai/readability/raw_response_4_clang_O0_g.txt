# Decompilation Readability Assessment

## LEVEL 1: LEXICAL CLARITY

### 1. Variable Naming Quality
**Score: 2/10**
Justification: Variables use completely generic names like `param_1`, `param_2`, `local_1c`, `local_18`, `local_c`. These provide no semantic meaning - for example, `cdecl_func(int param_1, int param_2)` should use `a` and `b` to match source. The decompiler makes no attempt to infer meaningful names from usage patterns.

### 2. Function Naming Quality
**Score: 9/10**
Justification: Function names accurately match the original source - `cdecl_func`, `call_cdecl`, `varargs_func`, `func_no_args`, etc. External library functions are correctly marked as `<EXTERNAL>`. Minor deduction for some internal functions like `call_weak_fn` being generic.

### 3. Literal Clarity
**Score: 4/10**
Justification: Magic numbers appear without explanation - e.g., `0x14` (20), `0x1e` (30), `0x28` (40) in `call_mips()`. String literals are shown as `&DAT_000122b3` instead of actual strings. Constants like `0x10` (16) used in loops without explanation.

### 4. Noise Reduction
**Score: 3/10**
Justification: Significant decompiler artifacts present. Unused variables like `local_c [3]` in `varargs_func`. Complex stack frame manipulations visible. Internal compiler helper functions (`__adddf3`, `__subdf3`) clutter the output. The decompilation includes ~800 lines of floating-point library functions that aren't part of the original source.

## LEVEL 2: STRUCTURAL INTELLIGIBILITY

### 5. Control Flow Naturalness
**Score: 7/10**
Justification: At -O0, control flow closely matches source. Loops use `for` constructs appropriately. Conditional expressions are clear. Minor issues include some awkward variable initialization patterns, but overall flow is readable.

### 6. Function Decomposition
**Score: 7/10**
Justification: Functions are appropriately separated. The large `test_calling_conventions` function (original also large) is kept as one unit. Static functions properly identified. Some internal functions like `call_weak_fn` could be better organized.

### 7. Code Linearization
**Score: 6/10**
Justification: Generally reads top-to-bottom. Some forward jumps in complex functions like `test_calling_conventions`. Early returns used appropriately in functions like `func_struct_byptr`. Moderate disruption from stack frame setup code.

### 8. Redundancy Elimination
**Score: 6/10**
Justification: At -O0, minimal optimization artifacts expected and delivered. Some redundant variable declarations visible. No significant dead code. Loop constructs are clean.

## LEVEL 3: TYPE SYSTEM FIDELITY

### 9. Type Precision
**Score: 5/10**
Justification: Uses generic `int` and `undefined4` instead of specific types. Function parameters correctly identified as `int` but with `undefined4` for uncertain types. Pointers are generally correct. No `const` correctness recovered. The type precision is adequate but not precise.

### 10. Composite Type Recovery
**Score: 3/10**
Justification: Struct recovery is poor. In `func_struct_byptr`, the decompiler shows `param_1[1]` instead of proper struct field access like `p->x * p->y`. LargeStruct is completely lost - `func_struct_byval` shows raw array access `local_10[local_1c * 2]` instead of `s.data[i]`. No struct type definitions recovered.

### 11. Type Safety
**Score: 5/10**
Justification: Some unsafe operations present. In `param_ptr_array`, the cast `**(byte **)(param_1 + local_10 * 4)` is complex and error-prone. Pointer arithmetic is technically correct but not type-safe. No bounds checking visible. Mixed `undefined4` and `int` types create potential issues.

## LEVEL 4: SEMANTIC TRANSPARENCY

### 12. Comment Quality & Presence
**Score: 1/10**
Justification: No comments present in the decompiled output whatsoever. No explanation of function purpose, parameter meanings, return value expectations, or algorithm descriptions. The original source had extensive comments explaining calling conventions that are completely lost.

### 13. Idiom Recognition
**Score: 3/10**
Justification: No design patterns or algorithm patterns identified. The varargs handling in `varargs_func` is shown as raw pointer iteration without recognizing the `va_list`/`va_arg` pattern. No recognition of calling convention patterns being tested.

### 14. Anomaly Indication
**Score: 2/10**
Justification: No warnings about suspicious code. The `call_order_dep()` function calls `param_order_dep()` with no arguments despite the function expecting 2 parameters - this is a critical semantic error that should be flagged. No indication of potential undefined behavior from the original source's intentional sequence point issues.

### 15. Assertion & Contract Recovery
**Score: 2/10**
Justification: No preconditions, postconditions, or invariants documented. The original source's assumptions (e.g., "ptr != NULL" checks, "requires: count > 0") are not recovered. No error condition documentation.

---

## LEVEL 5: CONTEXTUAL COHERENCE

### 16. Cross-Reference Clarity
**Score: 5/10**
Justification: Function call relationships are technically visible through direct calls. However, no documentation of who calls whom, no callback relationships documented, no global variable usage transparency. The relationship between `call_cdecl` and `cdecl_func` is implicit, not explicit.

### 17. Module Organization
**Score: 7/10**
Justification: Functions are grouped logically - all calling convention functions together, all parameter passing functions together, all return value functions together. This matches the original source organization. Static functions properly scoped.

### 18. Build System Compatibility
**Score: 3/10**
Justification: Code would not compile as-is. Uses undefined types like `undefined4`, `undefined8`, `undefined`. External function declarations are circular (`printf` calls itself). String literals shown as `&DAT_...` addresses. Would require significant patching to compile.

### Critical Defects (Bullet List)

1. **Variable naming is completely generic**: `param_1`, `param_2`, `local_1c`, `local_18` provide zero semantic information about their purpose (e.g., should be `a`, `b`, `count`, `sum`)

2. **No struct type recovery**: `LargeStruct` and `SmallStruct` are completely lost - code shows raw pointer arithmetic (`local_10[local_1c * 2]`, `param_1[1]`) instead of struct field access

3. **No comments whatsoever**: The original source had extensive documentation about calling conventions, parameter semantics, and test purposes - all completely lost

4. **Critical semantic error undetected**: `call_order_dep()` calls `param_order_dep()` with zero arguments when the function expects 2 parameters - this should be flagged as a decompilation failure

5. **Type precision severely degraded**: Uses `undefined4`, `undefined8` instead of proper types; no `const` correctness; generic `int` everywhere

---

### Best Quality Aspect: Function Naming Quality (9/10)
Function names accurately match the original source - `cdecl_func`, `call_cdecl`, `varargs_func`, etc. External library functions are correctly identified. This is the only area where the decompiler performs well.

---

### Most Needs Improvement: Comment Quality & Presence (1/10)
The complete absence of comments is devastating for readability. The original source had rich documentation explaining calling conventions, parameter purposes, expected return values, and test scenarios. A decompiler should recover at least basic documentation from debug symbols or infer semantics from patterns.

---

### Top 3 Recommendations for Decompiler Improvement

1. **Infer variable names from usage patterns**: When `param_1 + param_2` appears, name them `a` and `b`. When `local_18 = 0` followed by accumulation loop appears, name it `sum`. Use semantic patterns to recover meaningful names.

2. **Recover struct types from stack layout**: When consecutive stack variables are accessed with fixed offsets (e.g., `[local_1c * 2]`), recognize this as array/struct access and reconstruct the composite type with field names.

3. **Add semantic comments automatically**: Generate basic comments like "Sums varargs parameters" for `varargs_func`, "Tests calling conventions" for `test_calling_conventions`. Flag uncertain decompilations with "UNCERTAIN: indirect call" or "POTENTIAL ERROR: missing arguments".
