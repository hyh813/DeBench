# Decompilation Readability Assessment

## Analysis Overview

This is a comprehensive test suite for calling conventions and parameter passing patterns in C, compiled with gcc -O0 for arm64. The decompiled code shows significant degradation in several areas, particularly variable naming, type recovery, and semantic documentation.

---

### === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: All parameters use generic names like `param_1`, `param_2`, `local_28`, `local_24`. The original source uses meaningful names like `a`, `b`, `count`, `args`. For example, in `cdecl_func(int a, int b)`, the decompiler produces `int cdecl_func(int param_1,int param_2)` - completely losing semantic meaning.

**2. Function Naming Quality:** Score: 9/10  
Justification: Function names are perfectly preserved - `cdecl_func`, `call_cdecl`, `stdcall_func`, `varargs_func`, etc. All match the original source exactly. Minor deduction for some wrapper functions that could be more descriptive.

**3. Literal Clarity:** Score: 5/10  
Justification: Mixed results - some literals like `0x2a` (42 in `func_no_args`) and `0x14` (20) are shown as hex without explanation. However, string format strings like `"CALL-L1-01: %d\n"` are preserved. The magic number `0x51eb851f` (representing 3.14 as double) is completely opaque.

**4. Noise Reduction:** Score: 7/10  
Justification: Relatively clean output with minimal compiler artifacts. Some unnecessary stack checking code (`__stack_chk_fail`) is present but expected at -O0. No significant dead code or redundant casts observed in the main functions.

---

### === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 8/10  
Justification: Control flow is natural and readable. Loops use proper `for` constructs, conditionals are clear. At -O0, no goto abuse or spaghetti code. Example: `for (local_20 = 0; local_20 < param_1; local_20 = local_20 + 1)` is readable.

**6. Function Decomposition:** Score: 9/10  
Justification: Function granularity is excellent - each function has a single responsibility matching the original. No monolithic functions. Parameter lists are reasonable. The decomposition exactly matches the source code structure.

**7. Code Linearization:** Score: 8/10  
Justification: Code reads top-to-bottom with minimal disruption. Early returns used appropriately. No forward jumps into loop bodies. The `varargs_func` loop structure is clear and linear.

**8. Redundancy Elimination:** Score: 8/10  
Justification: No dead code or significant redundancy. At -O0, there's no aggressive optimization artifacts like loop unrolling to clean up. The code is straightforward.

---

### === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation. Original `int` becomes `undefined4` in many places. Function pointers use `code *` instead of proper function signatures. The `func_mixed_args` shows `undefined4` for what should be `double` and `long long`. The original `int x, char *s, double d, long long ll` becomes `int param_1, char *param_2, undefined4 param_3, undefined4 param_4, undefined4 param_5, undefined4 param_6`.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Struct recovery is poor. The original `LargeStruct` (128-byte with `long long data[16]`) is completely lost. In `func_struct_byval`, it becomes raw parameters `int param_1, undefined4 param_2, undefined4 param_3, undefined4 param_4` with manual array reconstruction. The `SmallStruct` {int x, int y} becomes just `int *param_1` with manual field access `param_1[1] * *param_1`.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe operations. In `param_ptr_array`: `(uint)**(byte **)(param_1 + local_c * 4)` is a chain of unsafe casts. The `func_mixed_args` uses raw float/double manipulation functions (`__floatsidf`, `__aeabi_dadd`, `__aeabi_d2iz`) without proper type safety. Pointer arithmetic like `param_1 + local_c * 4` lacks bounds checking.

---

### === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: No comments in decompiled output. The original source has extensive comments explaining each test case (e.g., "// CALL-L1-01: cdecl调用约定"). The decompiler provides zero documentation about what each function tests or why certain patterns exist.

**13. Idiom Recognition:** Score: 4/10  
Justification: Some patterns are recognizable (varargs using va_list style loop), but not explicitly named or documented. The original source explicitly tests calling conventions, but this semantic information is completely lost in decompilation.

**14. Anomaly Indication:** Score: 3/10  
Justification: No indication of decompilation uncertainties. The `call_thiscall` returns hardcoded `0xf` (15) but there's no note that this is a C++ method stub. The indirect function pointer calls in `param_func_ptr` have no uncertainty markers.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No preconditions, postconditions, or invariants documented. The original has implicit contracts (e.g., `func_struct_byptr` requires non-null pointer, `varargs_func` requires count > 0) but these are not recovered.

---

### === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 3/10  
Justification: Function call relationships are syntactically visible but not documented. The caller functions (e.g., `call_cdecl()`) invoke the tested functions, but there's no documentation of the call graph or which functions test which calling conventions.

**17. Module Organization:** Score: 7/10  
Justification: Functions are grouped logically in the output, roughly matching the source file structure (calling conventions, parameter passing, return values sections). However, all functions are in a single flat namespace without any grouping markers.

**18. Build System Compatibility:** Score: 5/10  
Justification: The code would not compile as-is. It uses undefined types like `undefined4`, external function declarations are missing (e.g., `printf`), and the `code *` type is non-standard. Requires significant patching to compile.

---

### === FINAL SUMMARY ===

**Overall Readability Score:** 4.6/10

**Critical Defects:**
- Variable naming uses generic `param_1`, `param_2`, `local_XX` completely losing semantic meaning from original `a`, `b`, `count`, `args`
- Composite types (structs) are completely lost - `LargeStruct` with 128 bytes becomes 4 raw int parameters with manual array reconstruction
- Type precision severely degraded - `double` becomes `undefined4`, function pointers become `code *`
- No comments or documentation explaining what each test case validates
- No indication of calling convention semantics being tested (cdecl, stdcall, fastcall, etc.)

**Best Quality Aspect:** Function Decomposition (scored 9/10)  
Function granularity is excellent - each function has single responsibility matching the original source exactly. No monoliths, proper parameter lists.

**Most Needs Improvement:** Variable Naming Quality (scored 2/10)  
All parameters and locals use generic numbered names. The original uses meaningful names like `a`, `b`, `count`, `args`, `sum` that are completely lost.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns**: When `param_1 + param_2` is used in an arithmetic context, infer `a` and `b`. When `local_28` is used as a pointer advancing through an array, name it `ptr` or `iter`.
2. **Recover struct types from stack layout analysis**: When multiple consecutive parameters or local variables are accessed as fields (e.g., `local_10[0]`, `local_10[1]`), reconstruct the original struct definition.
3. **Add semantic annotations for calling convention tests**: When a function clearly tests a specific pattern (like varargs with va_list), add a comment explaining the pattern being tested.
