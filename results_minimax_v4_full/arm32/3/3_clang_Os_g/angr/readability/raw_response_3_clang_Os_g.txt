# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use generic names like `v0`, `v1`, `ptr`, `idx`, `result` throughout. The original source had meaningful names like `arr`, `i`, `n`, `local`, `out` but these are completely lost. For example, in `local_array`, the decompiler uses `v1`, `i`, `v3`, `v0` instead of `arr`, `n`, `i`.

**2. Function Naming Quality:** Score: 5/10  
Justification: Mixed results - some functions correctly recovered like `local_vars`, `heap_basic`, `test_stack_memory`, but many use address-based naming like `sub_400568`, `sub_40068c`. The helper function `double_value` was recovered as `$a.30` at address 0x401484.

**3. Literal Clarity:** Score: 4/10  
Justification: Magic number `4294967295` (0xFFFFFFFF) appears frequently for error returns without explanation. Constants like `20`, `42`, `10` are hardcoded. The original source had clear intent (e.g., `arr[5]`, `local = 42`) but these become opaque in decompiled form.

**4. Noise Reduction:** Score: 6/10  
Justification: Some compiler artifacts present like complex pointer arithmetic (`*((unsigned int *)((char *)&v8 - (7 + i * 4 & 0xfffffff8) + 4 * idx))`). However, no excessive register spilling or debug instrumentation. The VLA and alloca handling shows confusing stack frame manipulation.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: Loops are present but use unusual constructs - the `local_array` function uses `do-while` with negative index manipulation (`i -= 1` then `i != 4294967286`). Some goto statements present in error handling paths. At -O0, control flow should be more natural.

**6. Function Decomposition:** Score: 7/10  
Justification: Function granularity is preserved - many small functions exist matching the source. However, parameter lists expose implementation details (e.g., `unsigned int a0` instead of meaningful parameter names). Functions like `vla_stack` and `alloca_usage` are correctly separated.

**7. Code Linearization:** Score: 5/10  
Justification: Generally reads top-to-bottom but has forward jumps and complex indexing expressions that disrupt flow. The `vla_stack` function has particularly convoluted expression: `*((unsigned int *)&(&v0)[2 * idx + -1 * (7 + a0 * 4 & 0xfffffff8)])`.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some loop unrolling present but not excessive. Dead code is minimal. The `test_stack_memory` function hardcodes expected values (e.g., `printf("MEM-L1-01 (local_vars): %d\n", 20);`) which is redundant with calling the actual function.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Most integers are `unsigned int` when original used `int`. Pointers are `void*` or `unsigned int*` rather than specific types. The original `int *arr` became `unsigned int *ptr`. Const correctness is lost - `const int const_global` becomes regular access.

**10. Composite Type Recovery:** Score: 3/10  
Justification: Some struct definitions recovered with generic names like `struct_0`, `struct_1`, `struct_2`. Field names are `field_0`, `field_4` instead of meaningful names like `data`, `next`. The original `Point` struct and `Node` struct are not properly recovered.

**11. Type Safety:** Score: 5/10  
Justification: Several unsafe operations present. The `dangling_pointer` function correctly shows `free(ptr)` followed by `return *(ptr);` - this is undefined behavior but shown accurately. Some pointer arithmetic could lead to issues. No `restrict` or alignment annotations.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 1/10  
Justification: No comments present in the decompiled code. The original source had extensive comments explaining test scenarios (e.g., `// MEM-L1-01: 局部变量（基础类型）⭐`). None of this context is preserved.

**13. Idiom Recognition:** Score: 4/10  
Justification: Some patterns recognizable - linked list traversal in `linked_list_heap`, tree structure in `tree_heap_traversal`. However, no explicit naming of patterns (e.g., "Linked List", "Binary Tree"). The alloca/VLA patterns are obscured by complex stack manipulation.

**14. Anomaly Indication:** Score: 1/10  
Justification: No flags for suspicious code. The `dangling_pointer` function shows undefined behavior without warning. The `double_free` and `heap_overflow` dangerous patterns are not flagged. No indication of indirect call resolution failures.

**15. Assertion & Contract Recovery:** Score: 1/10  
Justification: No pre/postcondition recovery. The original had explicit checks like `if (n <= 0 || n > 1000) return -1;` but these become `armg_calculate_condition` calls without clear documentation of what conditions are being checked.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are visible (`malloc`, `free`, `printf`) but relationships unclear. The callback pattern in `set_global_callback` / `call_global_callback` is partially visible but the function pointer assignment `global_func_ptr = $a.30` is cryptic.

**17. Module Organization:** Score: 5/10  
Justification: Functions are grouped by type (stack, heap, static, memory ops) which roughly matches source organization. However, all in single file with no logical section headers. Static functions not properly scoped in output.

**18. Build System Compatibility:** Score: 3/10  
Justification: Code has compilation issues - type mismatches (e.g., `unsigned int` vs `int*`), missing struct definitions, undefined `armg_calculate_condition`. The `extern` declarations for `g_413008`, `g_402cfe` etc. are not resolved.

---

## === FINAL SUMMARY ===

**Overall Readability Score:** 3.9/10

**Critical Defects:**
- Variable naming is entirely generic (`v0`, `v1`, `ptr`, `idx`) losing all semantic meaning from source
- No comments or documentation - original test scenario labels completely lost
- Composite types recovered with generic names (`struct_0`, `field_0`) instead of meaningful names (`Point`, `Node`, `data`, `next`)
- Complex pointer arithmetic in VLA/alloca functions makes stack operations unreadable
- No anomaly indication for dangerous patterns like dangling pointers, double-free, heap overflow

**Best Quality Aspect:** Function Decomposition (scored 7/10)  
Function granularity is preserved - many small, focused functions exist matching the source structure. The test wrapper functions are correctly separated.

**Most Needs Improvement:** Semantic Transparency (scored 1.75/10)  
The decompiled code lacks comments, pattern recognition, anomaly indication, and contract recovery. This dimension scored lowest because the original source had extensive context (test scenarios, expected values, comments) that is completely absent.

**Top 3 Recommendations for Decompiler Improvement:**
1. **Infer variable names from usage patterns** - `ptr` used with `malloc` should suggest `heap_ptr`, loop index `i` should be named `idx` or `index`, variables assigned constant values like `42` should consider that as a hint
2. **Recover struct field names from context** - If a struct is used for linked lists (`node->data`, `node->next`), these field names should be inferred rather than generic `field_0`, `field_4`
3. **Add semantic annotations for dangerous patterns** - Flag undefined behavior like dangling pointers, double-free, buffer overflows with comments like `// WARNING: use-after-free` or `// ANOMALY: indirect call could not be resolved`
