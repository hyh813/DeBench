# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: The decompiled code uses extremely generic names like `param_1`, `param_2`, `iVar1`, `iVar2`, `uVar1`, `uVar2`, `local_808`, `puVar1`, `piVar3`. These provide zero semantic meaning - e.g., `param_1` should be `x` or `n` based on usage, and `local_808` should be `large_buf` or `arr`.

**2. Function Naming Quality:** Score: 7/10  
Justification: Most function names correctly match the original source (e.g., `local_vars`, `heap_basic`, `global_var_access`, `double_value`). However, external functions like `<EXTERNAL>::calloc` are just stubs, and some internal functions have slightly off names.

**3. Literal Clarity:** Score: 3/10  
Justification: The code is littered with unexplained magic numbers: `0x2a` (42), `0x14` (20), `0xf` (15), `0x800` (2048), `0xffffffff` (-1). No named constants or explanatory comments help understand these values.

**4. Noise Reduction:** Score: 7/10  
Justification: Relatively clean output with minimal compiler artifacts. Some unnecessary casts present but not overwhelming. The code is mostly readable without debug instrumentation clutter.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 4/10  
Justification: Extensive use of imprecise types: `undefined4`, `undefined`, `undefined4 *`, `byte`, `uint`. Should be `int`, `int*`, `char`, `size_t`. The type precision is severely degraded - e.g., `int param_1` should be `int x`.

**10. Composite Type Recovery:** Score: 3/10  
Justification: The `Point` struct is completely lost - accessed as raw memory. The `Node` and `TreeNode` structures are not recovered, using `undefined4*` instead of proper `Node*`. No struct field names recovered.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe operations present. Extensive pointer arithmetic without type safety (e.g., `*(int *)((int)__ptr + iVar1 * 2)`). Function pointer type is wrong (`code *` instead of proper function signature). Some buffer overflow patterns visible.
