# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use extremely generic names like `param_1`, `iVar1`, `local_28`, `local_808` that provide no semantic meaning. For example, in `local_array`, the array `local_28[10]` should be named `arr` based on the source, and loop indices `iVar1`, `iVar2` are completely opaque.

**2. Function Naming Quality:** Score: 8/10  
Justification: Most function names are correctly preserved from source: `local_vars`, `heap_basic`, `global_var_access`, `memop_memset`. Minor issues like `call_static_func` being slightly different from source but still recognizable.

**3. Literal Clarity:** Score: 5/10  
Justification: Magic numbers like `0x2a` (42), `0x14` (20), `0x800` (2048) appear without explanation. Some constants like `0xffffffff` for error returns are consistent but not documented. The literal `0x3e9` (1000) in VLA bounds check is unexplained.

**4. Noise Reduction:** Score: 7/10  
Justification: Minimal compiler artifacts; some auto-generated WARNING comments about unreachable blocks but these don't impede reading. No excessive register spilling artifacts or debug instrumentation visible in main decompiled functions.

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

**5. Control Flow Naturalness:** Score: 5/10  
Justification: The `local_array` function uses backwards loop indexing (`local_28[-iVar2]`), which is confusing. At `-O0` this should be a normal forward loop. However, most other functions have natural control flow with proper if/else structures.

**6. Function Decomposition:** Score: 7/10  
Justification: Functions maintain reasonable granularity - each test function is separate. Some inlining occurred (e.g., `double_value` is called directly in `call_global_callback`), but overall decomposition is acceptable for debug build.

**7. Code Linearization:** Score: 6/10  
Justification: Most functions read top-to-bottom with minimal forward jumps. The `linked_list_heap` function has complex goto-style jumps that disrupt linear reading. Early returns are used appropriately in many functions.

**8. Redundancy Elimination:** Score: 6/10  
Justification: Some loop unrolling present (e.g., in `heap_basic` using stride-2 access pattern) but not excessive. No obvious dead code. Some common subexpression patterns but manageable.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

**9. Type Precision:** Score: 3/10  
Justification: Severe type degradation - uses `undefined4`, `undefined1`, `undefined4*` instead of proper types like `int`, `char`, `int*`. The `param_1` should be `int` but is shown as `undefined4`. No `size_t`, no `uint32_t`, just generic undefined types.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct recovery at all. The `Point` struct is completely lost - `local_struct` just returns `param_1 * 3` instead of using struct fields. The `Node` and `TreeNode` structures are reduced to raw pointer arithmetic with no field names.

**11. Type Safety:** Score: 4/10  
Justification: Multiple unsafe operations: pointer arithmetic with incorrect strides (using `*2` for int arrays instead of `*4`), casting through `undefined4*`, and in `heap_overflow` the loop uses negative indexing `__ptr[-iVar2]` which is confusing and error-prone.

## === LEVEL 4: SEMANTIC TRANSPARENCY ===

**12. Comment Quality & Presence:** Score: 2/10  
Justification: Only auto-generated header comments ("Decompiled by Ghidra 12.0") and WARNING comments about unreachable blocks. No useful comments explaining algorithm logic, preconditions, or non-obvious operations. The source had detailed comments completely lost.

**13. Idiom Recognition:** Score: 2/10  
Justification: No design patterns or algorithms identified. Memory management patterns (malloc/free pairs, linked list traversal) are not recognized or named. The source's explicit test scenarios (MEM-L1-01, HEAP-L2-01, etc.) are completely lost.

**14. Anomaly Indication:** Score: 4/10  
Justification: Some WARNING comments about unreachable blocks and indirect jumps, but these are generic Ghidra warnings, not specific to the code's actual anomalies like the dangling pointer access in `dangling_pointer` or the double-free pattern.

**15. Assertion & Contract Recovery:** Score: 2/10  
Justification: No preconditions, postconditions, or invariants recovered. The safety checks in source (e.g., `if (arr == NULL) return -1;`) are present but not documented as contracts. The VLA bounds validation is present but unexplained.

## === LEVEL 5: CONTEXTUAL COHERENCE ===

**16. Cross-Reference Clarity:** Score: 4/10  
Justification: Function calls are visible but relationships are not documented. The callback pattern in `set_global_callback`/`call_global_callback` is present but not explained. No indication which functions call which others.

**17. Module Organization:** Score: 4/10  
Justification: Functions appear in arbitrary order (by address), not matching the logical grouping in source (stack memory, heap memory, static memory, memory operations). Static functions like `static_helper` not marked as static.

**18. Build System Compatibility:** Score: 2/10  
Justification: Code will not compile - uses `undefined4`, `undefined1` types which don't exist. Missing includes for standard library functions. The `extern` declarations for `extern_global_var` and `extern_function` are incomplete.

### Critical Defects (Bullet List)
- **Variable naming**: All variables use generic names (`param_1`, `iVar1`, `local_28`) providing zero semantic meaning - should infer names from usage patterns
- **Type precision**: Uses `undefined4`, `undefined1` throughout instead of `int`, `char`, `size_t` - compilation impossible
- **Struct recovery**: Complete failure - `Point`, `Node`, `TreeNode` structs all lost, reduced to raw pointer arithmetic
- **Control flow**: `local_array` uses backwards negative indexing (`local_28[-iVar2]`) which is confusing and doesn't match source's forward loop
- **Comments**: Only auto-generated headers, all source comments (MEM-L1-01, test scenarios) completely lost

### Best Quality Aspect (Single Dimension)
- **Function Naming Quality (8/10)** - Most function names correctly preserved from source (`local_vars`, `heap_basic`, `global_var_access`), making it easy to identify which function is being analyzed.

### Most Needs Improvement (Single Dimension)
- **Type System Fidelity (3.0/10)** - The combination of type precision (3/10), composite type recovery (2/10), and type safety (4/10) shows severe degradation. Using `undefined4` everywhere makes the code uncompilable and obscures data structures entirely.

### Top 3 Recommendations for Decompiler Improvement
1. **Infer variable names from usage patterns**: When `param_1` is used in arithmetic like `param_1 * 2 + 10`, infer it as `x` or the original parameter name. When `local_28[i]` is indexed sequentially, name it `arr` or `buffer`.
2. **Recover struct types from allocation patterns**: When seeing `malloc(8)` followed by field accesses at offsets 0 and 4, recognize this as a 2-field struct and attempt recovery (like `Node` with `data` and `next`).
3. **Preserve or reconstruct type information**: At `-O0` debug level, types should be nearly perfect. Map `undefined4` to `int`, `undefined1` to `char`, and preserve `size_t` for memory operations.
