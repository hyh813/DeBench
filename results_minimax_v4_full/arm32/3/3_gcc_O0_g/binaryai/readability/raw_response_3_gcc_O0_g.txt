Justification: The decompiler uses imprecise types: `undefined4` for int, `undefined` for char, `undefined4 *` for various pointers. Function parameters often use `undefined4` instead of `int`. Some type inference exists but much is lost.

**10. Composite Type Recovery:** Score: 2/10  
Justification: Critical failure - the `Point` struct is not recovered in `heap_struct` or `global_struct_access`. The `Node` struct in linked list functions is treated as raw `int*` with manual field offsets (`local_18[1]` for next pointer). The `TreeNode` struct is also not recovered. All composite types lost.

**11. Type Safety:** Score: 5/10  
Justification: Multiple unsafe operations: raw pointer arithmetic in VLA/alloca functions (`*(int *)((int)aiStack_30 + ...)`), manual struct field access via offsets, no bounds checking on array accesses. The code is functional but not type-safe.
