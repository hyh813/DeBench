## === LEVEL 1: LEXICAL CLARITY ===

**1. Variable Naming Quality:** Score: 2/10  
Justification: Variables use extremely generic names like `param_1`, `iVar1`, `piVar1`, `uVar1`, `pvVar1`, `local_34`, `iStack_38`. The original parameter `x` becomes `param_1`, loop counters become `iVar2`, and pointers become `piVar1`. These names provide no semantic meaning and make tracking variable purpose extremely difficult.

Justification: Function names are accurately preserved from source: `double_value`, `local_vars`, `local_array`, `heap_basic`, `heap_calloc`, `linked_list_heap`, `tree_heap_traversal`. The decompiler correctly recovered the original function names with only minor signature differences.

**3. Literal Clarity:** Score: 5/10

**4. Noise Reduction:** Score: 8/10

Justification: Uses `undefined4` instead of `int`, `undefined` for uninitialized data, and generic types throughout. The original `int` types are not recovered. `size_t` is used in some places (e.g., `heap_calloc`). Overall significant type degradation.

**10. Composite Type Recovery:** Score: 2/10  
Justification: No struct recovery visible. The `Point` struct (used in `local_struct`, `heap_struct`, `global_struct_access`) is completely lost. `Node` and `TreeNode` structures are not recovered - all accessed via raw pointer arithmetic with indices like `__ptr[1]`, `__ptr[2]`.

**11. Type Safety:** Score: 6/10  
Justification: Some unsafe pointer operations present, particularly in struct access via raw offsets. Pointer arithmetic is used extensively (e.g., `(int)__ptr + (param_1 / 2) * 4`). Function pointers are typed as `code *` which is non-standard. Moderate risk of undefined behavior.
