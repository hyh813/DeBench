# Decompilation Readability Assessment

## === LEVEL 1: LEXICAL CLARITY ===

### 1. Variable Naming Quality
**Score: 2/10**

The decompiled code uses exclusively generic variable names: `v0`, `v1`, `v2`, `v3`, `v4`, `v5`, `v6`, `v7`, `v8`, `v9` for local variables, and `a0`, `a1`, `a2`, `a3` for parameters. The original source used semantically meaningful names like `temp1`, `temp2`, `temp3`, `sum`, `i`, `count`, `result`. Additionally, register names like `r0`, `r1`, `r2`, `r3`, `lr` are exposed as variable names, and stack frame offsets like `[bp-0x4]`, `[bp-0x8]` clutter the code. This makes it nearly impossible to track variable purpose without extensive analysis.

### 2. Function Naming Quality
**Score: 5/10**

Some functions retain their original names: `sequential_ops`, `single_if`, `if_else`, `nested_if_2`, `loop_for_fixed`, `loop_while`, `recursion_factorial`, `double_value`, `triple_value`. However, many functions have generic names like `sub_10454`, `sub_10530`, `sub_11ac0`, and internal labels like `LABEL_10600`, `LABEL_106bc` are exposed. The test functions are mostly preserved (`test_control_flow_l1`, `test_control_flow_l2`, `test_control_flow_l3`), but the inconsistency between well-named and auto-generated names creates confusion.

### 3. Literal Clarity
**Score: 2/10**

The code is riddled with unexplained magic numbers: `4294967295` (0xFFFFFFFF, used as -1), `1717986919` (magic constant for division), `74072` (jump table address), `71368`, `71352`, `71316` (computed goto addresses). The string literals like `"CF-L1-01 (sequential_ops): %d\n"` are hardcoded in the binary and exposed directly. The constant `1001` appears without explanation (loop iteration limit). No named constants or enums are recovered from the original source.

### 4. Noise Reduction
**Score: 2/10**

The decompiled code contains severe noise artifacts:
- Excessive goto statements with labels like `LABEL_10570`, `LABEL_1057c`, `LABEL_10600`
- Complex conditional expressions using `armg_calculate_condition(194, a0, 0, 0)` instead of simple comparisons
- Unnecessary variable declarations for register saves like `unsigned int v1; // lr`
- Stack frame offset comments like `// [bp-0x4]`, `// [bp-0x8]` exposed as variable annotations
- Redundant stack canary checks like `if (!(v3 ^ GLIBC_2.4::__stack_chk_guard))` that add no semantic value

## === LEVEL 2: STRUCTURAL INTELLIGIBILITY ===

### 5. Control Flow Naturalness
**Score: 3/10**

The control flow is severely degraded. The decompiler uses goto statements extensively instead of natural loop constructs. For example, `loop_for_fixed` uses a do-while with goto labels instead of a clean for loop. The `if_elseif_chain` function has convoluted conditional logic with nested if statements and goto jumps. The `loop_complex_cond` function is completely unrolled into a series of if-else chains instead of a while loop. Even simple functions like `single_if` use inverted conditionals with goto instead of clean if-else. The original source had idiomatic C control flow; the decompiled version is nearly unreadable.

### 6. Function Decomposition
**Score: 7/10**

The function boundaries are correctly preserved. Each function from the original source exists as a separate function in the decompiled output. The parameter counts generally match the original (though types are degraded). Functions are not over-fragmented or merged inappropriately. However, some helper functions like `__aeabi_idiv`, `.divsi3_skip_div0_test` are compiler-generated and exposed, which adds noise but doesn't represent decomposition failure.

### 7. Code Linearization
**Score: 2/10**

The code has extremely poor linearization. Forward jumps are everywhere: `goto LABEL_10600`, `goto LABEL_107d2`, `goto LABEL_107d6`, etc. The `if_elseif_long` function has 6+ goto labels disrupting flow. The `loop_complex_cond` is completely unrolled with 10+ labeled sections. The `nested_if_deep` function has multiple forward jumps. Reading any non-trivial function requires constant jumping between non-adjacent code sections. The original source had clean, linear code; the decompiled version is spaghetti.

### 8. Redundancy Elimination
**Score: 3/10**

Significant redundancy exists throughout. The `loop_complex_cond` function is unrolled into repetitive if-else chains that could be a simple loop. The `if_elseif_long` function has duplicate conditional checks. The `switch_large` function has redundant range checks and goto patterns. The `loop_modify_var` function has dead code paths and unreachable labels. The decompiler did not recognize and eliminate these redundant patterns.

## === LEVEL 3: TYPE SYSTEM FIDELITY ===

### 9. Type Precision
**Score: 3/10**

All parameters and most local variables are typed as `unsigned int`. The original source used `int` for most parameters (signed), but the decompiler consistently uses `unsigned int`. Function pointers are typed as `unsigned int` instead of proper function pointer types. The `loop_while` and `loop_dowhile` functions incorrectly use `unsigned int` for what should be signed integers (the original divides by 10). No `const` correctness is preserved. The original had proper signed/unsigned distinctions; these are lost.

### 10. Composite Type Recovery
**Score: 1/10**

No struct or union types are recovered. The original source had arrays like `int arr[] = {10, 20, 30, 40, 50}` and `int matrix[3][4]`, but these are not reconstructed. The `loop_break` function uses a switch on the target value instead of the original array lookup. The `duffs_device` function has completely degraded pointer types. The `large_jump_table` function uses raw function pointer arrays without proper typing. All composite types are lost.

### 11. Type Safety
**Score: 4/10**

Multiple type safety issues exist:
- Function pointer calls like `goto a0` in `call_func_ptr` are dangerous
- Pointer arithmetic in `process_with_callback` uses raw offsets without bounds checking
- The `opaque_predicate` function uses complex flag calculation functions that could cause undefined behavior
- The `overlapped_code` function has cryptic flag manipulation with `armg_calculate_flag_c` and `armg_calculate_flag_v`
- No bounds checking on array accesses in switch statements

3. **No type precision**: All parameters and most variables are `unsigned int`, losing the original signed/unsigned distinctions. No struct types are recovered; all composite types are lost.

The decompiler should track how variables are used (array index, accumulator, size, etc.) and assign semantic names.

2. **Recover natural control flow constructs**: Replace goto-based control flow with natural loop constructs (for, while, do-while). Recognize loop patterns from backward jumps and convert them to idiomatic C. The current output has spaghetti code that is nearly unreadable.

3. **Add comprehensive comments and pattern recognition**: Identify and label algorithms (state machine, jump table, recursion, etc.). Add comments explaining parameter purposes and return value semantics. Flag anomalies and uncertain decompilations. The current output provides zero semantic context.
